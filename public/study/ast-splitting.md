# AST（抽象構文木）分割

> 学習記録 ｜ 分野: cs（コンパイラ / プログラム解析） ｜ 筆者: 自分

## 概要

**AST（Abstract Syntax Tree, 抽象構文木）** はソースコードの構造をツリー状に表現したもの。この AST を、処理しやすい単位やノード（**ブロック・文・式・変数**など）に切り分ける技術が **AST 分割（AST splitting / chunking）** である。丸ごとの巨大な木を扱うより、意味のある単位に割ることで、コンパイラの並列解析・コードの AI 生成（LLM）・リファクタリング最適化などで扱いやすくなり、構文の構造を正確に掴める。

## 背景 / きっかけ

コードを機械的に扱うとき、単なる文字列（テキスト）として見ると「どこが関数で、どこが if で、どこが変数か」という**構造**が失われる。一方 AST は構造を保持できるが、木が丸ごと大きすぎると、

- 深く長い依存で学習・解析モデルが構造を捉えきれない
- 一括処理だと並列化できず遅い
- LLM に渡すとき関数やクラスの途中で分断され、文脈が壊れる

といった問題が出る。そこで「AST を意味のあるまとまりに割る」発想が生まれた。

## 学んだこと（本体）

### 1. AST とは（前提）

ソースコードをパーサに通すと、トークン列 → 構文木へと変換される。この木の各ノードが「関数定義」「if 文」「二項演算」「識別子（変数）」などに対応する。具体を先に見ると分かりやすい。

```text
式: a = b + 2

        =(代入)
       /      \
   a(変数)    +(加算)
             /     \
         b(変数)  2(リテラル)
```

このように、**式・文・宣言といった構文要素が親子関係のツリー**になっている。コンパイラの「意味解析」「最適化」「コード生成」はすべてこの木の上で行われる。

### 2. AST 分割とは

AST 分割は、この木を**より小さな部分木（サブツリー）や単位に切り分ける**こと。切り分けの粒度にはいくつかある。

| 粒度 | 単位 | 例 |
|---|---|---|
| 粗い | ブロック / 関数 / クラス | メソッド1つ、ループ本体まるごと |
| 中間 | 文（statement） | `x = f(y);` の1文 |
| 細かい | 式 / ノード | 変数参照、二項演算、リテラル |

「どこで切るか」を**構文の境界**に合わせるのが肝で、これにより各断片が**構文的に妥当（syntactically valid）**なまま残る。関数の途中や括弧の途中で切らない、という点が単純な固定長分割との決定的な違い。

### 3. 何に使われるか（用途）

#### (a) コンパイラの並列解析・最適化

木を独立した部分木に割れれば、**部分ごとに並列で解析・最適化**できる。関数単位・ブロック単位に切って別スレッドで処理する、といった高速化に効く。最適化パスも部分木を局所的に書き換える形で進む。

#### (b) ソースコードの AI 生成・要約（LLM / RAG）

LLM でコードを扱うとき、リポジトリを丸ごとは渡せないので**チャンク（chunk）に分割**して埋め込み（embedding）→ ベクタDB に入れ、質問時に関連チャンクを検索して文脈として渡す（**RAG**, Retrieval-Augmented Generation）。ここで固定長で切ると関数が分断されるが、**AST 境界で切ると論理的なまとまりが保たれ**、検索精度・生成品質が上がる。

#### (c) リファクタリングの最適化

リファクタリングは「意味を変えずに構造を変える」操作。AST 上で部分木を単位に**抽出・移動・置換**するので、分割された単位がそのまま操作対象になる。IDE の「メソッド抽出」などがこれ。

### 4. 代表的な手法

#### BASTS（Block-wise AST Splitting）

**コード要約（code summarization）**の精度を上げるために提案された手法（ICPC 2021, XMUDM）。ポイントは切り方にある。

- **制御フローグラフ（CFG）の支配木（dominator tree）のブロック**に基づいてメソッドのコードを分割する
- 各分割ごとに **split AST（分割された部分木）** を生成する
- 各 split AST を **Tree-LSTM** で（事前学習を使って）モデル化し、局所的な非線形の構文エンコードを捉える

要は「制御構造の切れ目で AST を割り、部分木ごとに構造を学習させる」ことで、木が大きくても構文構造を正確に掴めるようにした、というのが core idea。ユーザの理解にあった「ブロック単位などに分割することで構文の構造を正確に掴む」がまさにこれ。

#### cAST（structural Chunking via AST）

より新しい、コード RAG 向けの構造的チャンキング手法（2025）。**tree-sitter** で AST にパースし、**サイズ上限を超えない範囲で兄弟ノード（sibling nodes）をマージして可能な限り大きなチャンクを作る**。大きすぎるノードは再帰的に小さく割る。設計目標として「構文的整合性」「高い情報密度」「言語非依存」「プラグアンドプレイ」を掲げる。

#### ASTNN

AST を小さな**文レベルの部分木（statement subtrees）の列**に分解してコード表現を学ぶニューラル手法。木が大きいことによる勾配消失や長距離依存の問題を、分割で緩和する狙い。

> ⚠️ 補足（正確を期すため）: ユーザが挙げた「並列解析・AI 生成・リファクタリング」は AST 分割の**一般的な用途**。一方 BASTS は厳密には**コード要約**のための研究手法で、そのために「ブロック単位で AST を割る」。両者は目的がやや違うが、"構文構造を正確に掴むために木を意味単位で割る" という発想は共通している。

### 5. 実装の勘所（tree-sitter を例に）

実務での AST 分割は tree-sitter などのパーサライブラリを使い、木を再帰的に辿ってノード種別を見ながら切る。擬似コードで書くと分割の骨格はこうなる。

```python
# サイズ上限を守りつつ AST を意味単位でチャンク化する（cAST 的な骨格）
def chunk(node, max_size):
    text = node.text
    if size(text) <= max_size:
        return [text]                 # 収まるならそのまま1チャンク
    chunks, buf = [], []
    for child in node.children:       # 子（兄弟ノード）を順に見る
        if size("".join(buf)) + size(child.text) > max_size:
            chunks.append("".join(buf))
            buf = []
        if size(child.text) > max_size:
            chunks += chunk(child, max_size)   # 大きすぎる子は再帰で割る
        else:
            buf.append(child.text)             # 収まる子はマージ
    if buf:
        chunks.append("".join(buf))
    return chunks
```

ここでの要点は、**(1) 構文境界（ノード）で切る、(2) 収まる兄弟はまとめる、(3) 超えるものは再帰で割る** の3点。

## つまずき / 誤解していた点

- **「AST 分割 = 固定長で切る」ではない。** むしろ逆で、固定長分割の「関数の途中でぶった切る」問題を避けるために構文境界で切るのが AST 分割の眼目。
- **BASTS の "Block" は行ブロックではなく制御フロー（CFG の支配木）のブロック。** 単純な見た目の {} ブロックとは限らない。
- **用途によって最適な粒度が違う。** 並列コンパイルなら関数/ブロック粒度、リファクタリングなら式〜文粒度、RAG なら「上限サイズに収まる最大の構文単位」。目的から粒度を決める。

## まとめ・次に学ぶこと

AST 分割は「コードの構造を保ったまま、扱いやすいサイズ・単位に割る」技術。丸ごとの木では重い/文脈が壊れる問題を、**構文境界での分割**で解決する。並列解析・コード生成（RAG/LLM）・リファクタリングと応用は広く、BASTS（ブロック単位で割って要約精度を上げる）や cAST（RAG 向けの構造的チャンキング）が代表例。

次に学ぶこと:

- 制御フローグラフ（CFG）と支配木（dominator tree）— BASTS の切り方の前提
- tree-sitter を実際に動かして自分のコードを AST 分割してみる
- Tree-LSTM / ASTNN の部分木エンコードの仕組み
