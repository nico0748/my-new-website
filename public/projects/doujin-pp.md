

# 同人活動支援アプリケーション 「同人++」 をつくってみた話

4 ヶ月。ずっと書きたいと思っていた、初の個人開発プロダクトについての記事です。先日学内ハッカソンで発表する機会があり、自分の中で一区切りついたので、設計のあれこれをまとめておきます。

## 1. システム概要

「同人++（doujin++）」は、同人即売会（コミケ・コミティア・文学フリマ等）の参加者を対象にした、**蔵書管理 + 買い物リスト + 会場 MAP ナビ + POS レジ**を一体化した PWA です。

買い手モードでは、サークルと頒布物の事前登録、巡回順のドラッグ並び替え、ナビモードでの「次の目的地」案内、購入後の本棚管理までをひとつのアプリで完結できます。売り手モードでは自サークルの頒布物登録と POS レジ（紙幣・硬貨ボタンでお釣り自動計算）まで対応しています。

## 2. 導入決定までの背景

### 2-1. 即売会当日の現状フロー

僕自身が同人イベントに通っていて、当日のフローはだいたいこんな感じでした。

1. 公式サイトから紙のマップを DL・印刷
2. 行きたいサークルに手書きでマーカー
3. 前日にスプレッドシートで購入リストを整理
4. 当日、紙とスマホを両手に会場を歩き回る
5. 立ち寄ったサークルに手書きチェック

混沌と化した会場でペンと紙を出してメモメモ……果たして本当に出来るか？というのが出発点でした。

### 2-2. 紙とスプレッドシートの限界

| 知りたいこと | 紙だと | 同人++ なら |
|---|---|---|
| 買えた？ / 売り切れ？ | 判別不能 | ステータスで一発管理 |
| いくら使った？ | 暗算が必要 | リアルタイム集計 |
| 次はどこへ行く？ | 地図を広げて判断 | 最短ルートを自動案内 |
| 買った本をあとで探せる？ | 記憶頼り | 本棚機能で完全管理 |

スマホは常に手元にあるのに、紙・ペンの用意、「買えた」「売り切れ」の区別、当日〜帰宅後の蔵書管理 — この間で文明の利器がほとんど活かされていませんでした。

### 2-3. 市場としても十分大きいことを確認した

ニッチに見えて、調べてみると想像以上に広いマーケットでした。

| 指標 | 値 |
|---|---|
| コミケ 1 回あたりの動員数 | 15〜20 万人 |
| 全国で年間に開催される即売会の数 | 数百〜千以上 |
| ユーザーの利用頻度 | 週 1 〜 複数回（高継続率） |
| 直接競合 | ゼロ（先行者優位） |

「購入後の蔵書管理」に特化したアプリは既に存在していますが、**購入前〜購入後を一気通貫でカバーするものはほぼ存在しない**。ここが差別化の出発点になりました。

## 3. 技術的な構成

### 3-1. フロントエンド

React 19 + TypeScript 5 + Vite 7 + Tailwind CSS 4 で開発しました。

- **React 19**：並行レンダリングと Transitions、エコシステムの成熟度
- **TypeScript 5**：API レスポンス型から UI の Props 型まで一貫した型安全
- **Vite 7**：webpack と比べてビルドが 10 倍以上高速、HMR が即時反映
- **Tailwind CSS 4**：ユーティリティファーストで設計の一貫性、テーマも CSS 変数で完結

#### サーバー状態管理：TanStack Query

会場ではモバイル回線が混雑して API が不安定になりがちなので、サーバー状態管理は TanStack Query を採用しました。

1. 30 秒キャッシュ → 不要な API リクエストの削減
2. `mutate + invalidate` → データ更新後の自動再取得
3. 楽観的更新 → UI が即時反応
4. エラーリトライ → ネットワーク不安定な会場でも安定動作

```ts
const queryClient = new QueryClient({
  defaultOptions: {
    queries: { staleTime: 1000 * 30, retry: 1 },
  },
});
```

#### なぜ PWA にしたか — ドメインから逆算した技術選定

ネイティブアプリではなく PWA を選んだのは、**「即売会」というドメインの特性から逆算した結果**です。

| 観点 | PWA を選んだ理由 |
|---|---|
| ⚡ 即時デプロイ | 即売会はイベント直前にニーズが急増する。App Store のレビューを 1 週間待っていたら開催日が終わってしまう。「**審査を待つコスト**」がドメインに対して致命的すぎた |
| 🌐 インストール不要 | ストアアカウントすら不要で、参入障壁がゼロ |
| 📱 ネイティブ同等 UX | ホーム画面追加・オフライン対応・プッシュ通知まで実現可能 |
| 🔄 マルチプラットフォーム | iOS / Android / Windows をひとつのコードでカバー |

「流行ってるから React Native」ではなく、「ドメインの制約上、審査の待ち時間が許容できないから PWA」というふうに、選定理由を**ドメイン側の言葉で説明できる**形に持っていけたのが、自分の中では大きな前進でした。

### 3-2. バックエンド

Node.js + Express 5 + TypeScript + Prisma ORM + PostgreSQL 16 で開発しました。

#### Prisma ORM の何が嬉しかったか

スキーマから DB 定義と TypeScript 型を**同時生成**してくれるので、こういうことが起きます。

```prisma
model Book {
  id       String  @id
  title    String
  isbn     String?
  coverUrl String?  // ← この行を追加した瞬間に型が更新される
}
```

`coverUrl` を追加した瞬間、TypeScript が全参照箇所をチェックしてくれる。「DB 変更を忘れてどこかでサービスが落ちる」というよくあるパターンを構造的に防げます。マイグレーション履歴もコード管理されるので、本番反映も `prisma migrate deploy` だけで済みます。

### 3-3. 認証 — Firebase Authentication

Email / パスワード + Google OAuth に対応しています。フロントで取得した ID トークンを Express 側のミドルウェアで検証する構成です。

```ts
export async function authenticate(req, res, next) {
  const token = req.headers.authorization?.replace('Bearer ', '');
  if (!token) return res.status(401).json({ error: 'Unauthorized' });
  try {
    const decoded = await admin.auth().verifyIdToken(token);
    (req as any).uid = decoded.uid;
    next();
  } catch {
    res.status(401).json({ error: 'Invalid token' });
  }
}
```

将来 LINE ログインを追加するときも、バックエンド側の変更はほぼゼロで済む設計にしています。

### 3-4. ストレージ — Cloudflare R2 を選んだ「金額」の理由

会場 MAP 画像や書影など、画像系データの置き場が必要になりました。最初は AWS S3 を検討していたのですが、結論として Cloudflare R2 を採用しました。理由は単純で、**金額**です。

| サービス | データ転送料 |
|---|---|
| AWS S3 | **$0.09 / GB** |
| Cloudflare R2 | **$0.00** |

会場 MAP は 1 ファイル数 MB で、画像を頻繁に取得する性質のアプリなので、転送料が真っ先に効いてきます。R2 にしておけば「ユーザーが増えても転送料の心配がない」という構造になりました。

加えて、Presigned URL を発行することで、**ブラウザから R2 へ直接 PUT** できます。バックエンドのメモリ・CPU を一切経由しない画像アップロードが実現できて、これも嬉しい副産物でした。

### 3-5. インフラ全体像

```
ブラウザ
  │  HTTPS
  ▼
Cloudflare CDN  ← DDoS 保護 / HTTP/3 / 静的アセット 1 年 immutable キャッシュ
  │
  ▼
ConoHa VPS (Ubuntu 22.04)
  │
  ├─ Nginx (host)  ← Let's Encrypt + HTTP→HTTPS リダイレクト
  │   │
  │   ▼
  └─ Docker Compose
      ├─ frontend (Nginx + React build)
      ├─ backend  (Express + Prisma)
      └─ db       (PostgreSQL 16)

  + Cloudflare R2  ← 画像（presigned PUT）
```

frontend の Nginx が `/api/*` を backend へリバースプロキシしているので、ブラウザからは全て同一オリジンに見えて、CORS が発生しないようにしています。

## 4. 設計思想

ここが一番、自分なりに考えたところでした。

### 4-1. 「既存のやり方を全部取っ替え」ではなく、「既存をデジタルに置き換える」

既存の紙のマップ + スプレッドシート運用は、確かに不便ではあるけれど、ユーザーが**長年蓄積してきた資産**でもあります。「全部新しくしたほうが便利ですよ！」というアプリは、便利でも移行コストが高すぎて結局誰も使ってくれません。

なので、設計のスタンスを **「既存の運用フローをそのままデジタルに置き換える」** に振り切りました。

具体的には、**CSV / Excel インポート機能を最優先で実装**しています。長年スプレッドシートで管理してきたデータをゼロから再入力する必要はない、というメッセージです。これは「機能要件の優先順位は、ドメイン理解から決まる」という体験そのものでした。

### 4-2. オフラインファーストからの軌道修正

実は最初の要件定義書では、「クラウドを使用しない、ローカル DB のみ、運営費 0 円」というオフラインファーストのアーキテクチャを目指していました。実際 `feat: Firebase 依存を完全に除去` というコミットも履歴に残っています。

ただ、開発を進めるうちに「マルチユーザー対応」「サークル間でのデータ共有」「ランディングページからの集客」を視野に入れるようになり、サーバー構成への転換を決めました。

オフラインの良さ（会場で API が落ちてもアプリが真っ白にならない）は捨てたくなかったので、Service Worker + IndexedDB 書き込みキュー → オンライン復帰時に Last-Write-Wins で自動同期、という Phase 2 をロードマップに残しています。CRDT は同人即売会のドメインに対して過剰設計だと判断しました。

### 4-3. ドメイン駆動設計（DDD）

「即売会」というドメインをシステムの中心に据えて、サークル・頒布物・会場 MAP・買い物リストといったドメインオブジェクトを Prisma スキーマに直接対応させています。

- 機能追加が容易：新機能はドメインに追加するだけ
- テスト可能：ドメインロジックを独立してテストできる
- スケーリング：DB スケールアップをアプリロジックに影響なく実施

## 5. 開発プロセスの工夫 — AI エージェントを「PdM 的存在」として運用ループに組み込む

ここは技術スタックではなく、**開発プロセスそのものの工夫**です。個人的に一番効いたところでもあるので、独立して書きます。

個人開発で一番つらかったのが「ユーザーフィードバックを受けてから実装するまでの導線」でした。フィードバックが Twitter の DM や Form の回答に散乱して、気付いたら忘れていた、というのを最初の方で何度かやってしまいました。

そこで、Claude を使って **フィードバック → タスク化 → 実装** を半自動化するループを組みました。

```
[ユーザー]
     │
     ▼
[Google Form]   ← フィードバックを投稿してもらう
     │
     ▼
[Notion DB]     ← Form の回答を自動蓄積
     │
     ▼
[Claude Code]   ← Notion コネクトで FB を取得・分析
     │
     ▼
[タスク DB]     ← 優先度・実装方針付きでタスク化
     │
     ▼
[コーディング]  ← Claude Code がそのまま実装
```

ポイントは、Claude が**「フィードバックを読み取ってタスクに落とす」工程を担っている**ことです。「同じ要望が複数件あるか」「既存機能の延長か新規実装か」「セキュリティへの影響はあるか」を判断してタスク DB に整理してくれるので、僕はタスク DB の優先順だけを眺めればよくなりました。

「個人開発なのに PdM と PG が分業しているような状態」が実現できて、これは想像以上に効いた工夫でした。

## 6. セキュリティ

OWASP Top 10 に沿って [`docs/security-requirements.md`](https://github.com/) を作成し、優先度付きで実装しました。

実装済みのもの：

- Firebase ID トークン検証ミドルウェアで全 `/api` を保護
- `app.disable('x-powered-by')` でサーバー情報の露出を抑制
- `CORS_ORIGIN` 環境変数で本番オリジンを明示（ワイルドカード回避）
- Prisma 経由のパラメタライズドクエリで SQL インジェクション対策
- DB ポートを外部公開しない（Docker 内部ネットワークのみ）
- `.env.prod` を gitignore、Firebase Service Account は Base64 で環境変数注入
- Cloudflare 経由の `trust proxy` 設定で正しいクライアント IP を取得
- Let's Encrypt + HTTP→HTTPS リダイレクト、証明書の自動更新

ロードマップ化された未対応項目（helmet・express-rate-limit・Zod バリデーション・HSTS など）も優先度付きで明文化しています。「全部やるぞ」ではなく「優先度を決めて段階的に対応するぞ」という設計を、セキュリティの領域でもやっておきたかったところです。

### 7-2. ハマったところ

- **スマホ実機でカメラ API が動かなかった**：ISBN バーコードスキャンに使っている `getUserMedia()` は HTTPS でしか動かない。`mkcert + vite-plugin-mkcert` でローカル CA を作ってユーザーキーチェーンに登録する手順までドキュメント化したのが地味に大変でした。
- **会場 MAP 画像が `express.json` のデフォルト 100kb 制限を超える**：`limit: '50mb'` まで上げて、ホスト側 Nginx でも `client_max_body_size 60m` を設定。最終的には R2 への直接 PUT に切り替えました。
- **Phase 2 のオフライン書き込みキューでの競合**：複数端末から同じサークルを更新する可能性があるので、Last-Write-Wins（`updatedAt` 比較）を採用。CRDT は同人即売会のドメインに対して明らかに過剰でした。

## 8. さいごに

最初の提案から 4 ヶ月。214 コミット、67 PR、学内ハッカソンでの発表で一区切りになりました。

「紙とペンを両手に会場を歩き回る」という現状の不便さを、PWA という形でなんとかしたかっただけだったのですが、結果として React 19・Prisma・Cloudflare R2 まで含む実運用のスタックを一通り触ることになりました。

技術選定は「流行っているから」で決めるよりも、**ドメインの制約から逆算したほうが自分の言葉で説明できる**ということ — 「審査を待つコストが致命的だから PWA」「転送料が効くから R2」「即売会の競合更新パターン的に Last-Write-Wins で十分」— これが、4 ヶ月で得た一番大きな収穫だったと思います。

「同人活動のすべてをポケットの中に。」 — まだまだ未完成ですが、引き続き育てていきたいプロダクトです。ここまで読んでくださってありがとうございました。
