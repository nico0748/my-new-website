# 1. システム概要

私たちが作ったのは **ALiS (Arduino Light Signal System)** という、Wi-FiもBluetoothも一切使わずに、**光のみで文字を伝える4段リレー型の通信システム**です。

仕組みをざっくり言うとこんな流れです。

```text
文字列
  ↓ 符号化テーブル (8×8)
数値ペア (行, 列)
  ↓ ステッピングモータで距離を作る
赤外線距離 ── (段1→2)
  ↓ レーザーモジュールでON/OFF
モールス符号(光) ── (段2→3)
  ↓ フルカラーLEDで2色ペアに変換
色(RGB) ── (段3→4)
  ↓ カラーセンサで読み取り
数値ペアに復元 → 文字列
```

たとえば `"ALiS"` という4文字を送ると、まず符号化テーブルで `(0,0)(1,3)(4,2)(2,2)` という数値の列に変換され、それが「赤外線で読み取る距離」「レーザーの点滅(モールス)」「LEDの色の組み合わせ」と**3回も物理媒体を変えながら**運ばれていって、最後にもう一度文字列に戻る、というシステムです。

各段にArduinoが1台ずつ立っていて、計4台がバケツリレーのように情報を運びます。

[^1]: 実際、できあがったものは伝言ゲームそのものです。中継するたびに情報が劣化していく可能性があるところまで含めて。


# 2. なぜ電波を使わない通信を作ったのか

この制作物は、大学2年前期の「**ハッカソン**」という科目の制作課題として作ったものです。チームは4人、期間は約10週間、テーマは**「無線通信による伝言ゲーム」**。

「無線通信」とだけ指定されていて、媒体は自由でした。私たちは「光」を選んだのですが、他のチームでは**磁力**や**音**、**衝撃を検知する加速度センサ**など、それぞれかなり攻めた媒体で実装に挑んでいました。「無線」という単語の解釈の幅をこんなに見せつけられる場面はそうそうないので、講評会のときの他チームのデモはどれも興味深かったです。

私たちが「光」を選んだ理由は、教科書通りに言うとこうです。

最近のIoT機器の普及はすさまじく、Wi-FiやBluetoothによる通信はもはや **「何も考えなくても使えるインフラ」**になっています。一方で、電波は空間中に全方位へ放射される性質があるので、**スニッフィング(盗聴)のリスクが原理的につきまとう**。実際、2024年にはMediaTek製Wi-Fiチップにゼロクリック脆弱性 (CVE-2024-20017) が見つかって、家庭用ルータが一斉アップデートを迫られる事態がありました[^2]。

[^2]: 暗号化されていても、ファームウェアのバグを突かれるとそもそも復号されてしまう、というやつです。「暗号化してれば安全でしょ」という素朴な信頼が崩れた事件として印象に残っています。

つまり、**電波に依存しない代替的な通信手段**を考えてみる価値はある。じゃあ何で運ぶか?——光、距離、色。**物理的に「見える」もの**なら、少なくとも盗聴は物理的にその場にいないとできない。

……と、提案資料には立派な背景が書かれているわけですが、本音を言うと、**「光だけで文字って送れるの? 面白そう」**というのがチームの最初のモチベーションだったと思います。社会的意義は後付けな部分もあります。でも、それで動き始めたものが最終的にちゃんと社会的文脈の中に位置づけられたので、結果オーライでした。

[^3]: 後述しますが、この担当決めの時点で、私はまだ「フォトトランジスタって何?」のレベルでした。

# 4. 設計思想 ── 4段リレーという狂気

技術構成やら裏話を説明する前に、設計思想について先に話しておこうと思います。

最初のチームミーティングで設計の話をしたとき、「光通信、面白そうだけど、ただ光らせるだけだと薄くない?」という話になりました。それで出てきたのが、

- **赤外線(距離)** で運ぶ
- **レーザー(モールス)** で運ぶ
- **可視光(色)** で運ぶ

を**全部やる**、というアイデアです。冷静に考えると無謀ですが、当時のテンションでは「全部できたらすごいじゃん」という流れになっていました。

ただ、無謀なだけのアイデアではなくて、設計上のメリットもあります。

**1. モジュール化しやすい**
各段の通信方式は完全に独立しているので、誰かの担当部分が遅れても他の人は自分の担当を進められます。これは結果的にチーム開発で本当に効きました(なお、進められるとは言ってない)。

**2. 共通の符号化テーブルで対称性を保てる**
入口(段1)と出口(段4)が同じ8×8テーブルを参照することで、間で何があってもとにかく **(行, 列)のペアが0-7の数字として正しく運べれば**復号できる。中の段にとっては「数字を運ぶ」だけが仕事になります。

**3. 通信路自体が"見える"**
これは副次的なメリットですが、人間が目で見て「いま何の数字が運ばれているか」をある程度追えます。後述しますが、これがデバッグでものすごく効いてきました。

設計思想は雑に言えば、**「中継の各段は数字さえ正しく運べばよい。意味は両端だけが知っている」** という構造です。後でわかったことですが、これは普通のネットワークの階層モデルと同じ発想で、**自分たちが知らないうちに似たような設計に辿り着いていた**のが面白かったです。

# 5. 技術的な構成

各段の中身を簡単にまとめておきます。私の担当は段2と段3なので、そこに重きを置いて書きます。

## 5-1. 段1: 文字列 → 距離

8×8の符号化テーブルで文字列を `(row, col)` のペアの数列に変換し、**ステッピングモータで物理的にmm単位の距離を作り出す**段です。L6470というモータドライバをSPIで叩いて、ベルトコンベア上の障害物を移動させます。

```cpp
int mm[] = {200, 250, 300, 350, 400, 450, 500, 600, 700};
```

数字 0〜8 のラベルがそれぞれ200mm〜700mmに対応していて、たとえば「3」を送りたければ障害物を350mmの位置に動かす、という仕組みです。

## 5-2. 段2: 距離 → モールス符号 (← 私の担当の半分)

シャープ GP2Y0A02YK という赤外線距離センサで段1の障害物までの距離を測り、**10段階の閾値テーブル**で 0〜7 の数字に分類します。両端の閾値はヘッダー(開始)とフッター(終了)のシグナルとして使います。

```cpp
const int th[10] = {550, 525, 480, 415, 350, 290, 225, 194, 173, 173};
```

その後、得られた数字を**独自モールス符号**に変換してレーザーモジュールから送出します。一般的なモールスは符号長が1〜5で可変ですが、こちらは扱う対象を 0〜7 の8種類に絞れるので、**3要素固定**にしてしまいました。

```text
0: ---     1: --.     2: -.-     3: -..
4: .--     5: .-.     6: ..-     7: ...
ヘッダー: ..---..    フッター: .-.-. (AR)
```

## 5-3. 段3: モールス符号 → 色 (← 私の担当のもう半分)

フォトトランジスタでレーザーの点滅を読み取って数字に復元し、それをフルカラーLEDの**2色ペア**で再送信します。

```text
0: 赤+薄青       1: 赤+薄緑       2: 赤+シアン
3: 薄青+赤       4: 薄青+薄緑     5: 薄青+シアン
6: 薄緑+赤       7: 薄緑+薄青
```

ヘッダーは青単発、フッターは緑単発、データペアの区切りは薄シアン、と制御色を分けてあります。

## 5-4. 段4: 色 → 文字列

TCS34725 というI2C接続のカラーセンサで色を読み取り、ペアから数字を復元、最後に符号化テーブルを逆引きして文字に戻します。明示的な状態機械(`WAIT_FOR_HEADER → DATA1 → DATA2 → SEPARATOR → COMPLETE`)を組んで、プロトコルの整合性を担保しました。

# 6. 苦労したこと、いろいろ

ここからが本題です。

## 6-1. Gitが定着していなかった問題

技術スタック以前の話として、まず**チームのGit運用が成立していませんでした**。

当時、メンバーの中でGitHubを普段使いしていたのは私だけでした。ハッカソンではソフトウェア開発の経験を必ずしも前提にしていないので、「リポジトリを作ってプルリクで……」というワークフロー自体が初めてのメンバーがほとんどでした。私自身は高校の頃からGitHubを触っていたので、**チームの「Git反映係」**になりました。

実際の運用はこんな感じでした。

1. メンバーがコードを書く
2. **Discordにコードを直接貼る**(あるいは `.ino` ファイルを添付する)
3. 私がそれを受け取って、ローカルで動作確認して、リポジトリにcommit & push
4. バージョン管理(笑)

最初はgit運用を布教しようと思っていたんですが、メンバー全員にgit使い方講座を開いている時間で本体の実装をしたほうが早い、という判断になり、**「もう私が反映係でいいや」**で落ち着きました。今思えば布教しておけばよかった気もしますが、10週間という期間を考えるとあれが正解だった気もします。

ただ、副次的に良かった点もあって、**全員のコード変更が私の手元を通る**ので、誰がどの段の何を変更したか自然と把握できました。結合の段階でこれがけっこう効きました。**反映係 = レビュアー**を兼ねていた、という見方もできます。

## 6-2. 要件定義とマネジメントの苦労

これが一番きつかったです。

今回は**システム開発自体が初めて**というメンバーも多い中で、要件定義もプロジェクトマネジメントも自分たちで回す必要がありました。私もチーム制作の経験はあったものの、4人を相手にWBSやガントチャートを引くのは初めてで、最初から手探りでした。

ワークフローとしては、

1. 制作するシステムの**機能要件を洗い出す**
2. それを **WBS (Work Breakdown Structure)** に起こす
3. WBSから**ガントチャート**を作って、誰が何をいつまでにやるかを決める
4. 開発を始める

という、一見ちゃんとしたプロセスを踏んでいました。問題は、**いざ開発を始めると「あれ、この機能必要じゃん」が多発した**ことです。

具体的に出てきたものを思い出せる範囲で挙げると、

- 受信側で**ヘッダー/フッターの判定**が必要(最初の要件には入っていなかった)
- データ列の**区切り**をどう表現するか(最初の要件には入っていなかった)
- **タイムアウト処理**(送信が途中で止まったときどうするか)
- **デバッグ用のシリアル出力**(初期の要件にはモニタへの出力すら明記されていなかった)
- **エラー時のリカバリ**(最初の要件には正常系しか書かれていなかった)

このうちのいくつかは「実装しながら気づくのが普通でしょ」と言われればそうなんですが、**気づいたらWBSがどんどん増築されており**、ガントチャートが現実とどんどんズレていきました。チャートを引き直すたびに後ろに線が伸びていく感覚は、今思い出してもちょっと胃が痛いです。また、開発期間にバッファを設けるなどの対策を講じていなかったので、今思えば、ほんとに危なかったです...

機材まわりもなかなかに大変でした。**気がついたら機材購入で一万円以上費やしていました。**

- ステッピングモータドライバ(L6470)
- ステッピングモータ本体
- タイミングプーリ・タイミングベルト
- 各種抵抗・コンデンサ
- フォトトランジスタ(NJL7502L)
- レーザーモジュール
- フルカラーLED(HV-5RGB60)
- カラーセンサ(TCS34725)
- ジャンパワイヤ大量

ハードウェアを甘く見ていました。**ソフトウェアプロジェクトと違って、`npm install` でモジュールが降ってこない**んですよね。買って、届くのを待って、組んで、初めてスタートラインです。

そして実装後の検証フェーズでもまた苦労しました。**テストケースの準備が甘く、評価指標も曖昧**だったので、「これって動いてると言えるんだっけ?」という根本的な問いが何度も湧いてきました。最終的にはCER (Character Error Rate) を採用して定量評価する方針で落ち着きましたが、成果報告フェーズまでてんやわんやしてました(笑)

このへんの苦労を通じて思ったのは、**最初の要件定義と工程見積もりを甘くすると、後続のすべてに利息がついて返ってくる**ということです。これはこの記事で一番伝えたいことなので、最後にもう一度まとめます。

## 6-3. 環境条件という強敵

設計時に完全に見落としていたのが、**環境光・周囲温度・床面の反射といった環境条件が、光通信の精度に直撃する**という事実です。

私たちは普段、決まった作業場所を持っていなくて、

- 学科の**実験室**
- たまに**学食**(空いている時間)
- **図書館**の作業スペース


など、いろんな場所で集まって、その場でハードを組み立ててテストを回していました。**「動くじゃん!」**と思って次の集まりで違う場所で動かすと、**動かない**。

最初は「コードのバグでは?」と疑って、シリアルプロッタを眺めて、デバウンスを変えて、閾値を弄って、何時間も溶かしました。あるとき「**今日は実験室が暑くて、窓から夕方の光が斜めに差し込んでいる**」ということに気づいて、**「ちょっと待って、これ環境光の差じゃない?」**となりました。

実際にADCの生値を比較してみると、

- 実験室(蛍光灯のみ): ベース値だいたい一定
- 食堂(窓からの自然光あり): ベース値が時間帯で大きく変動
- 図書館(LED天井灯、間接照明多): また違う特性

完全に環境ごとに別物になっていました。**「光通信を選んだのにまさか光に負けるとは」**と全員で頭を抱えました。いや、本来真っ先に想定すべきだったんですが。

対策はけっこう泥臭くて、**フォトトランジスタとカラーセンサの周りを黒い厚紙で物理的に覆う**ということをやりました。覆いの内側に送信側の光だけが届くように、紙を切って組んで貼って。突貫工事のわりにはかなり効いて、CERが目に見えて改善しました。

これも要件定義の話に絡みますが、**「動作環境」を要件として明示しておかないと、こういうところで詰まる**んだなと身をもって学びました。「教室程度の環境光で動作する」と書いておけば、最初から遮光を設計に含められたはずです。

## 6-4. UNIT_TIMEの最適化と、教科書には載っていない山

私の担当範囲(光モールス)で最大の技術的山場は、**モールス符号の単位時間 (UNIT_TIME) をいくつにすればいいか**でした。

理屈の上では、UNIT_TIMEを短くすればするほど通信は速くなります。光は速いし、Arduinoのループも速いし、何が問題なんだ?と最初は思っていました。

実際に色んな値で試してみたところ、こうなりました。

| UNIT_TIME | 平均CER | 平均送信時間 |
|---|---|---|
| 50 ms | 43% | 1.42 s |
| 60 ms | 65% | 1.68 s |
| 70 ms | 45% | 1.78 s |
| **80 ms** | **0%** | **2.27 s** |
| 100 ms | 0% | 2.84 s |
| 300 ms | 0% | 8.35 s |

**60msあたりがなぜか一番悪い**という、教科書には載っていない山が出ました。最初これを見たときは「えっ、単調じゃないの?」と本気で混乱しました。

原因の見立ては、

- **フォトトランジスタの応答速度の限界**(光ON/OFFの立ち上がり/立ち下がりに鈍さがある)
- 受信側の `loop()` 内で `millis()` を毎ループ呼んでいるので、実は1ループあたり数ms使っている
- デバウンス処理で10msを消費している
- 状態遷移ロジックがギリギリで間に合っていない可能性

このあたりが複合的に絡んで、特定の周期で**ちょうど共振するように**誤判定が起きていたのだと思います。仮説の検証はしきれていません。

ともあれ、**「正確性が崩れない最速」が80ms**だと実験で確定し、ここを以後のシステム全体の基準にしました。**実装より計測の方が重い**という、ハードウェアを触ってみて初めて実感したやつです。

## 6-5. 「動かない」の原因の8割はソフトじゃなかった

ソフトウェア出身の人間として一番カルチャーショックだったのが、**バグの大半がソフトの外にある**という事実でした。

- 受信できない → ジャンパワイヤが浮いていた
- 受信できない → ブレッドボードの接触不良
- 受信できない → レーザーの照射軸がずれていた
- 受信できない → フォトトランジスタの向きが微妙
- 受信できない → 環境光が窓から差し込んでいた(前述)
- 受信できない → ステッピングモータがそもそも回っていない(後述)

「コードを直す前にハードを疑う」という発想に切り替えるまで、たぶん2週間くらいかかったと思います。**いつもの `console.log` デバッグ感覚で `Serial.println` を撒きまくっても、原因がコードにないので何も出てこない**んですよね。

これに気づいてからは、まず物理的な接続を全チェックして、それから受信側のADCの生値をシリアルプロッタに出して観測する、というワークフローに変えました。**目で見て分かるところまで戻ってから疑う**という、当たり前のことが当たり前にできるようになるまで時間がかかりました。

## 6-6. ステッピングモータが普通に壊れた

これは私の担当ではないんですが、チーム全体への影響が大きかったので書いておきます。

段1のステッピングモータが、プロジェクトの中盤で**普通に故障しました**。回らない。SPIの設定を見直しても、L6470のレジスタを叩き直しても、回らない。物理的に駄目だったようです。

代替品を発注して届くまで作業が止まる、というハードウェアならではの遅延が発生しました。ソフトウェアなら `git revert` で済む話が、ハードウェアでは**注文と発送のリードタイムを待つしかない**という、これも体感としては新鮮でした。

このリスクを甘く見ていて、結果として赤外線センサ担当の二人のスケジュールが後ろにずれ込み、最終週の結合作業が想定より圧迫されました。**ハードウェアプロジェクトでは予備を持て**、というのが大きな学びです。これも要件定義の段階で「主要部品の冗長化」を入れておくべきでした。

## 6-7. 生成AIと一緒にコードを書いていた話

私たちが開発していたのは2025年の5〜6月頃で、ちょうど**ChatGPTやGemini、Copilotといった生成AIをコーディングに使うのがメイン**になり始めていた時期でした。Arduinoまわりの試作コードは、たいてい生成AIに「こういうことがやりたい」と投げて、出てきたコードを叩き台にして手で直していく、という流れで書いていました。

ただ、当時の生成AIは今(2026年4月時点)から振り返るとまだ素朴で、

- **ハルシネーションが多かった**: 存在しないライブラリ名や、実在しない関数のシグネチャを平然と書いてくる
- **プロジェクト全体の文脈を保持できなかった**: 一つ前のターンで決めた仕様を次のターンで忘れる
- **整合性のとれないコード**: 段2と段3の接続を何度説明しても、データフォーマットが噛み合わないコードを生成する

このあたりに悩まされました。結局、**生成AIが出してきたコードを鵜呑みにせず、ネットの技術記事や公式ドキュメントを読み合わせながら、自分たちの手で部分部分を修正する**というワークフローに落ち着きました。生成AIが書く速度より、**生成AIの間違いを見つける速度のほうが大事**だった時期です。

執筆している現在(2026年4月)では、状況がかなり変わっています。**AIエージェント**と呼ばれる、プロジェクト全体を理解しながら自律的にコーディングを進めてくれるツールが主流になっていて、出力されるコードの精度も飛躍的に向上しました。手元のリポジトリを丸ごと読んで、整合性のあるパッチを返してくれる。1年足らずでこんなに変わるのか、というのが正直な感想です。

**今だったら一人で4人分の作業を回せる自信があります**[^4]。これは半分冗談で半分本気です。

[^4]: いつも助けてくれるClaudeくんに感謝。

ただ、これは後で詳しく書きますが、**「AIが速くなったから要件定義はテキトーでいい」**には絶対にならない、というのが**この経験を経て今思っていること**です。むしろ逆で、**AIが速くなったからこそ要件定義の質がアウトプットの質を決めるようになった**と感じています。

# 7. 評価結果

最後に、できあがったものがどのくらいの精度で動いたかを書いておきます。

評価指標は **CER (Character Error Rate, 文字誤り率)** を使いました。音声認識の世界で「人間が書き起こしとして意味をなす最低ライン」が認識率75%程度と言われていて、これを参考に**目標CERは25%以下**に設定。

結果はこちら:

| 区間 | 平均CER | ビットレート |
|---|---|---|
| **システム全体** | **4.73%** | 0.15 bit/s |
| 段1-2: 赤外線距離 | 4.73% | 1.09 bit/s |
| 段2-3: 光モールス | **0.00%** | 12.5 bit/s |
| 段3-4: フルカラーLED | **0.00%** | 0.98 bit/s |

私が担当した光モールスの段は、テストデータすべてでCER 0%でした[^5]。誤りはほぼ段1-2(赤外線距離)で発生していて、これは「障害物を移動させている最中にセンサが読んでしまう」という物理的な問題が支配的でした。

[^5]: もちろん、これは「動作確認できた範囲で」の数字です。受信が完全に失敗したケース(ヘッダー検出に至らない、など)は分母に入っていません。そこを含めるとまた別の話になります。

ビットレートで言うと、システム全体で0.15 bit/s、つまり**1文字運ぶのに大体1分**かかります。Wi-Fiが100 Mbpsとか出るのと比べると、笑ってしまうくらい遅い。でも、**目で見える速度で、目で見える光が、文字を運んでいる**というのは、初めて完走したときに本当に感動的でした。

> 受信側のシリアルモニタに、送ったはずの文字列がぽつぽつと出てきた瞬間。あれは一生忘れません。マジで泣きそうになった。

# 8. オープンキャンパスに展示された話

苦労が絶えない3か月でしたが、**ALiSは最終的に学科のオープンキャンパスで展示されることになりました**。

高校生が学科ブースに入ってきて、チームメンバーや自分がが「ここで入力した文字列が、4台のArduinoを経由して、最後にここに表示されるんですよ」と言いながら実際に動かす。**赤外線で物体が動いて、レーザーが点滅して、LEDが2色のペアで光って、最後に文字列が出てくる**。普段の通信では見えないものが目で追えるので、説明としてのウケは思った以上に良かったらしいです。

**「光通信で文字を伝える」**というキャッチーな見せ方ができたこと、各段の動作を**目で追える**設計にしていたことが効きました。設計思想のところで書いた「**通信路自体が"見える"**」というメリットが、ここで予想外の形で活きました。

# 10. さいごに ── 要件定義の話を、もう一度

長くなりましたが、最後にこの記事で**一番伝えたかったこと**を書きます。

それは、**「最初の要件定義と開発計画はしっかりと詰めておくこと**です。

身も蓋もなく当たり前の話なんですが、当たり前のことを当たり前にやれていなかったのが私たちでした。WBSの増築、機材費の膨張、評価指標の曖昧さ、環境条件の見落とし、結合タスクの宙吊り、エラー処理の後付け……これらの苦労は**全部、最初の要件定義と工程見積もりが甘かったことに由来している**と、振り返ってみて思います。

「いやでも、最初に全部詰めるのは無理でしょ。やりながら見えてくる部分もあるし」というのは、もちろんその通りです。**最初に完璧な要件定義を作るのは不可能**だし、それを目指すのは別の意味で危険です。でも、**「最初に詰められる部分は詰めておく」と「やりながら詰める」は別物**で、前者をサボって後者だけでやろうとすると、私たちのようなことになります。

そして、これは**生成AI、特にAIエージェントが普及した今こそ重要**だと思っています。

「AIが速くコードを書いてくれるなら、要件定義に時間をかけるよりさっさと書き始めたほうが早いのでは?」と思うかもしれません。1年前の私もそう思っていた節があります。**でも、実際にAIエージェントを使ってみると、逆だということがわかります**。

第一に、**AIエージェントは速く書ける分、間違った方向に走ったときのリカバリも速くは済まない**んです。要件が曖昧なまま「とりあえず作って」と投げると、エージェントは何かを作ってくれます。作ってくれるんですが、それが**意図していたものとはちょっとずつ違う**。差分が小さいうちはいいんですが、エージェントは止めない限り作り続けるので、気づいたときには**全体がちょっとずつズレた状態で結合されている**ことがあります。後から要件を詰め直すと、結局その大半のコードを捨てることになる。**人間が手で書くなら自然にブレーキがかかる場面で、エージェントはブレーキがかからない**んです。

第二に、**エージェントは前提を勝手に埋めて動く**という性質があります。これは長所でもあり短所でもあって、要件で明文化されていない部分はエージェントが「こうだろう」と推測して埋めます（AIの冪等性）。たいていの場合は妥当な推測なんですが、**「妥当な推測」と「自分が意図していたもの」は別物**です。要件が固まっていないと、**エージェントの推測がそのまま自分の意図として固定化される**ことが起こりえます。これは、自分の意図が明確であればあるほど避けたい事態です。

第三に、**「何を作るべきか」の判断はAIには任せられない**。これはたぶんしばらく変わらない。AIは「**どう作るか**」をどんどん肩代わりしてくれるようになりますが、「**何を作るか**」「**何を作らないか**」は人間が決めるしかない。要件定義というのは「何を作るか」を言語化する作業なので、**AIが速くなればなるほど、相対的に要件定義の重みが増していく**ことになります。

私たちのプロジェクトで言えば、

- **「どう実装するか」**(モールスの符号化、ステートマシン、色判定の閾値): AIに聞きながら進められた
- **「何を実装するか」**(4段リレーで、各段が0-7の整数を運び、両端で文字に戻す): これは人間が決めないと始まらなかった

の境界がはっきりしていて、後者を最初にちゃんと固められなかったために、**AIに聞いても答えが返ってこない問い**(「結合のタスクは誰のWBSに入れる?」「動作環境はどう決める?」)で時間を溶かしました。

要件定義と開発計画というのは、「**AIに何を任せ、何を任せないか**」を決める作業でもあります。**この境界線が引けていないチームは、AIエージェントが速くなったぶんだけ、速く迷子になる**。私はそう感じています。

- ハードウェアは**物理的に壊れる**という当たり前の事実

- インターフェイスを先に切る設計は、ハードが絡むときほど効く
- 計測と観測なしには何もデバッグできない
- **システムの精度は定量評価で決定する**
- **要件定義と開発計画の質が、最終成果物の質を決める**
