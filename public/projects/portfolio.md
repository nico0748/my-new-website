# このポートフォリオサイト自体を、設計図のように作った話

普段は他人のシステム設計について書くことが多いのですが、今回は **このサイトそのもの** を題材にします。「ポートフォリオを作る」という、世のWeb系学生の通過儀礼みたいなテーマを、自分なりに設計思想とUI/UXで突き詰めた結果のまとめです。

技術的な部分を評価してもらいたいので、見栄えよりも **「何をどう選んで、なぜそうしたか」** に重きを置いて書きます。コードが読みたい人は GitHub を見てもらうとして、この記事では **設計判断とその裏側** を中心に整理します。

:::message
本記事は個人開発の振り返りです。すべての判断は「個人プロジェクトで運用負荷をかけずに学びを最大化する」という方針に基づいています。チーム開発や大規模プロダクトでは別の最適解があります。
:::

## 1. プロジェクト概要

このサイトは、**React 19 + TypeScript + Vite** で構築した個人ポートフォリオです。デザイン上のコンセプトは **「方眼紙のようなブループリント (設計図)」** で、エンジニアの作業机にある方眼紙ノートをそのまま Web に持ち込んだような世界観を目指しています。

主なスペックはこんな感じです。

| 項目 | 内容 |
|---|---|
| フレームワーク | React 19.1 / TypeScript 5.8 |
| ビルドツール | Vite 7.1 + `@vitejs/plugin-react-swc` |
| スタイリング | Tailwind CSS 3.4 + CSS Variables (テーマ切替) |
| アニメーション | Framer Motion 12 |
| ルーティング | React Router 7 |
| グラフ | Recharts 3 (スキルレーダーチャート) |
| Markdown | react-markdown 10 + remark-gfm |
| データソース | **Google Sheets API**（バックエンド・DB なし） |
| PWA | `vite-plugin-pwa` (Workbox) |
| ホスティング | Vercel |

特徴をひとことで言うと、**「自前のサーバーを一切持たない、けれど更新できる」** ポートフォリオです。Google Sheets を「外部DB」に見立てることで、バックエンド構築・DB運用・APIサーバーの全部をスキップしています。

## 2. 開発の背景

このサイトの最初のコミットは **2025年9月3日** で、本記事を書いている **2026年4月時点で約8か月** 経ちました。途中、就活や授業の波で何度も止まりましたが、合計17本のPRを積み上げて今の形になっています。

開発のきっかけは「ポートフォリオがあれば便利」という素朴な動機でした。ただ、テンプレを買って組み立てるだけでは技術的な学びがほぼ得られないので、**「自分の使いたい技術スタックでフルに作り直す」** という方針を最初に決めました。

ベースの考え方として、以下を軸にしています。

- **運用にお金と手間をかけない**: 個人プロジェクトなので、サーバー代もDBの保守も払いたくない。
- **更新が手元で完結する**: ブログ記事や受賞歴を追記するたびに `git push` するのは現実的ではない。Google Sheets で済ませたい。
- **見せ方そのものを設計する**: 「ポートフォリオ」という静的な印象のサイトを、**触っていて気持ちのいい体験** に作り変える。
- **自分の技術力の "見本市" にする**: 使っている技術が、そのまま作品の説得力になるように。

最後の項目は地味ですが大事で、たとえばここで「`useState` だけで全部書きました」と言うと、それはそれで自分の技術評価につながってしまいます。逆に、必要以上に Redux や状態管理ライブラリを足しても **やりすぎ** に見える。**規模に見合った道具を選ぶ判断** そのものを、サイトに残しておきたかったというのが本音です。

## 3. 技術的な構成

各レイヤーで何を選んでいるかと、その選定理由を整理します。

### 3-1. フロントエンド (React 19 + TypeScript)

React は **19系** を採用しています。これは執筆時点で最新の安定版で、新しい `use()` フックや Actions 周りの恩恵を受けるためというよりは、**「最新版で詰まったときの調査経験を作る」** という勉強目的の比重が大きいです。

TypeScript は **strict 設定** で運用していて、データ層には専用の `interface` を切っています。たとえばプロジェクトデータはこんな型です。

```ts
export interface PortfolioItem {
  id: string;
  title: string;
  description: string;
  thumbnail: string;
  tags: string[];
  liveUrl?: string;
  repoUrl?: string;
  markdownFile?: string;
}
```

Google Sheets から返ってくる値は基本すべて `string` なので、`mapPortfolioData()` の段階で **型付きオブジェクトに正規化** しています。**「外の世界の `any` をなるべく早く捕まえて型に閉じ込める」** という、よくあるパターンです。

### 3-2. ビルド・ランタイム (Vite + SWC + PWA)

ビルドは **Vite 7** で、React プラグインは **`@vitejs/plugin-react-swc`** を選んでいます。SWC は Babel と比べてビルド・HMR が体感で明確に速く、開発サーバーを `localhost:5177` で立ち上げるのが軽快なので、開発体験としてかなり効きます。

PWA 化には **`vite-plugin-pwa`** を使っていて、内部的には Workbox で Service Worker を生成します。`vite.config.ts` の `runtimeCaching` で、**取得対象ごとにキャッシュ戦略を切り分けている** のがポイントです。

```ts
runtimeCaching: [
  // Google Fonts: 一度取れれば長期間使い回す
  { urlPattern: /fonts\.googleapis\.com/, handler: 'CacheFirst', ... },
  // Google Sheets API: 内容が変わるので Network を優先
  { urlPattern: /sheets\.googleapis\.com/, handler: 'NetworkFirst', ... },
  // プロジェクト Markdown: NetworkFirst だが TTL は1日
  { urlPattern: /\/projects\/.*\.md$/, handler: 'NetworkFirst', ... },
]
```

「**変わらない静的アセットは CacheFirst**、**動的なAPIは NetworkFirst**」というのが基本方針です。これだけでオフライン時もある程度動くPWAになります。

### 3-3. スタイリング (Tailwind + CSS Variables + Framer Motion)

スタイリングは **Tailwind CSS** をベースにしていて、テーマカラー（背景・テキスト・アクセント）は **CSS Variables** に切り出しています。`src/index.css` に以下のように書いていて、

```css
:root {
  --bg: #0a0a0f;
  --text-primary: #e2e8f0;
  --accent: #3b82f6;
  /* ...30個ほど */
}
[data-theme="light"] {
  --bg: #f4f6fb;
  --text-primary: #1e293b;
  --accent: #2563eb;
  /* ... */
}
```

ダーク/ライトの切替は **`document.documentElement.setAttribute("data-theme", theme)`** で属性を切り替えるだけ。コンポーネント側は `style={{ color: 'var(--text-primary)' }}` のように変数を参照しているので、**全コンポーネントが両テーマに自動追従** します。

最初は色をハードコードしていたのですが、ダークモード対応を入れたタイミングで **既存コンポーネント約15ファイルを CSS 変数に置換するリファクタ** を実施しました。一度この基盤に乗せてしまえば、以後のテーマ追加 (たとえば「セピア風」) もCSSだけで完結します。

アニメーションは **Framer Motion** で、`motion.div` の `whileInView` や `useScroll` をフルに使っています。後述しますが、タイムラインのスクロール連動グラデーションラインがその代表例です。

### 3-4. データ層 (Google Sheets API + Markdown)

このサイトの **一番割り切ったところ** がこの層です。バックエンドはありません。データはすべて Google Sheets の各シート (Profile, Projects, Skills, Timeline) に置いていて、`fetchSheetData()` で REST API 経由で取得しています。

```ts
const url = `${BASE_URL}/${SPREADSHEET_ID}/values/${sheetName}?key=${API_KEY}`;
```

返ってくるのは2次元配列なので、1行目をヘッダー扱いしてオブジェクト化し、**`dataMapper.ts` の `mapXxxData()` 関数で型付きオブジェクトに変換** しています。データが増えるたびに Google Sheets の行を追加するだけで反映されるので、**個人運用としてはこれが一番楽** です。

プロジェクトの詳細記事 (この記事もそう) は **Markdown ファイル** として `public/projects/<id>.md` に置く運用にしています。Google Sheets に長文を入れるのは現実的ではないので、長文は Markdown に逃がしている形です。表示は `react-markdown` + `remark-gfm` (テーブル・タスクリスト等のGFM対応)。

### 3-5. デプロイ (Vercel)

ホスティングは Vercel で、`vercel.json` に SPA 用のリライト設定を入れています。React Router で `/projects/:id` のような動的ルートを直接叩いたときに 404 にならないよう、すべてのリクエストを `/index.html` にフォールバックさせています。

```json
{ "rewrites": [{ "source": "/(.*)", "destination": "/index.html" }] }
```

PR を出すたびに Preview Deploy が立ち上がるので、**マージ前にPRごとに本番相当の環境で確認できる** のが地味に効いています。スタックPRで開発するときに特に有用でした。

## 4. 設計思想 ──「ポートフォリオは設計図である」

技術構成の話だけだと「ふーん、よくあるReactサイトね」で終わってしまうので、**デザイン上の設計思想** をここでまとめておきます。

このサイトのデザインコンセプトは **「方眼紙 (graph paper) のブループリント」** です。背景には常に細かい方眼パターンが敷かれていて、各セクションのカードは **設計図のドラフト** のような佇まいになっています。

なぜ方眼紙なのかというと、**「自分の作品を並べる場所」というポートフォリオの本質を、"設計の作業場" として表現したかった** からです。製品の完成形を派手に見せるサイトは世の中に山ほどありますが、「どう考えて、どう測って、どう組み立てたか」を見せるサイトは案外少ない。だったら、土台そのものを **方眼紙＋ブループリント** にしてしまおう、というのが出発点です。

具体的にこの思想がどう表れているかを挙げると、以下のとおりです。

**1. 方眼紙の "目盛り" が常に背景にある**
`GraphPaperBackground` コンポーネントが画面全体に固定背景として敷かれていて、スクロールしても方眼が動きません。「ここは設計の作業場である」というメッセージが、ページ遷移しても消えない。

**2. カードに "コーナーマーク" がついている**
`CornerMarks` という小さなコンポーネントを入れていて、各カードの四隅に **設計図風の括弧 `「 」` のようなマーク** が描かれています。トリミング枠やトンボのメタファーで、「ここから内側がコンテンツ」という設計指示書のニュアンスです。

**3. セクションタイトルの下にアクセントライン**
すべてのセクションタイトルの下に、**短いライン + ドット + 短いライン** のアクセント装飾が入っています。これも図面の中央線・センターマークのモチーフです。

**4. Glassmorphism のカード**
カードは半透明 + `backdrop-blur` で、**下の方眼が透けて見える** デザインにしています。これにより「カード上のコンテンツも方眼の上にある」ことが視覚的にわかる。情報が浮遊しているのではなく、**測れる場所に置かれている** という感覚を狙っています。

設計思想は雑に言えば、**「すべてのコンテンツは方眼の上に置かれた一時的なメモであり、いつでも測り直せる」** という構造です。後述しますが、ライト/ダーク切替やテーマカラーが CSS 変数で抽象化されているのも、**「この方眼の上で色だけが変わる」** というメンタルモデルに沿っています。

## 5. UI/UX で工夫したポイント

設計思想を踏まえた上で、実装レベルで工夫した点を順に書きます。技術評価の対象として一番熱を入れたい部分です。

### 5-1. 方眼紙背景は「2層のグリッド」で奥行きを出している

`GraphPaperBackground` の内部は、4枚の `linear-gradient` を重ねた **2層のグリッド** で構成されています。

```css
background-image:
  linear-gradient(var(--grid-major) 1px, transparent 1px),
  linear-gradient(90deg, var(--grid-major) 1px, transparent 1px),
  linear-gradient(var(--grid-minor) 1px, transparent 1px),
  linear-gradient(90deg, var(--grid-minor) 1px, transparent 1px);
background-size: 40px 40px, 40px 40px, 8px 8px, 8px 8px;
```

**40px の主グリッド** と、その内側に **8px の細グリッド** が重なっていて、近くで見ると細かいマス目が、引いて見ると大きなマス目が見える、**「ノートの方眼そのもの」** の質感になっています。色も `var(--grid-major)` と `var(--grid-minor)` で2段階に分けていて、ライト/ダーク両テーマで読みやすい濃度を別々に調整できます。

### 5-2. CSS 変数で実装したテーマシステム

ダーク/ライト切替は、**ContextでReactステートを持ち、`<html>` の `data-theme` 属性を切り替える** という、ややクラシックな実装にしています。

```tsx
// ThemeContext.tsx
useEffect(() => {
  document.documentElement.setAttribute("data-theme", theme);
  localStorage.setItem("theme", theme);
}, [theme]);
```

ポイントは、**コンポーネント側のスタイルが Context を直接参照していない** ことです。色はすべて CSS 変数経由なので、コンポーネントは `var(--text-primary)` を読むだけ。Context が更新するのは `<html data-theme="...">` のひとつだけで、**再レンダリングが Context Consumer に閉じています**。

その結果、テーマ切替の処理がほぼゼロコストで、**Framer Motion のアニメーション中にテーマを切り替えても破綻しません**。Reactで全コンポーネントを再描画させるのではなく、**ブラウザのCSSエンジンに切替を委ねる** という選択です。

### 5-3. スクロール連動のグラデーションタイムライン

これは個人的に一番気に入っているUIです。タイムラインセクションの縦ラインが、**ページのスクロール位置に応じて、グレーから青→紫→シアンのグラデーションに塗り変わっていく** 仕掛けになっています。

実装は Framer Motion の `useScroll` + `useTransform` で、**スクロール進捗 0〜1 を縦ラインの `scaleY` にマップ** しています。

```tsx
const { scrollYProgress } = useScroll({
  target: ref,
  offset: ["start center", "end end"],
});
const scaleY = useTransform(scrollYProgress, [0, 1], [0, 1]);

<motion.div
  style={{
    scaleY,
    background: 'linear-gradient(to bottom, #2563eb, #8b5cf6, #06b6d4)',
  }}
  className="absolute bottom-0 left-[7px] top-[12px] w-px origin-top"
/>
```

`origin-top` で上端を基準にスケールするので、スクロールに連れて線が下に伸びていきます。**「自分の歴史を辿るほどに線が伸びていく」** という時間的なメタファーがそのまま視覚に乗ります。

各タイムラインアイテムは、**同じ年の最初のアイテムにのみ年ラベルを大きく表示** する仕組みにしていて、`isFirstOfYear` フラグを `Set<string>` で計算しています。年ラベルの繰り返しを防ぐ地味な工夫ですが、リスト全体の情報密度がだいぶ整います。

### 5-4. Recharts によるスキルレーダーチャート

スキルセクションでは、各カテゴリ (フロントエンド・バックエンド・インフラなど) を **レーダーチャート** で見せています。Recharts の `RadarChart` を使い、テーマ変更に追従するように **`stroke` や `fill` も CSS 変数で指定** しています。

```tsx
<PolarGrid stroke="var(--border-color)" />
<PolarAngleAxis tick={{ fill: 'var(--text-secondary)' }} />
<Radar stroke="var(--accent)" fill="#3b82f6" fillOpacity={0.25} />
```

レベル表示はチャートだけだと粒度が粗いので、**各スキルに 0/5 の段階バーを別途併記** しています。チャートで全体感、バーで個別の解像度、という二段構えです。

### 5-5. CornerMarks ──「設計指示書」のメタファー

`CornerMarks` は地味ながら、このサイトのトーンを決定づけているコンポーネントです。SVG で描かれた **L字型の小さなマーク** をカードの四隅に配置するだけのものですが、これがあるかないかで **「ただのカード」と「設計図のカット枠」** くらいの差が出ます。

```tsx
<CornerMarks color={color} size={size} offset={offset} strokeWidth={1.25} />
```

色は `color` プロパティで上書きできるようにしていて、**デフォルトでは `var(--corner-marks-color)`** を読むようにしてあります。テーマに連動するようになっており、ライトでは濃いめの青、ダークではトーンを落とした青に切り替わります。

### 5-6. Active Section の追従にIntersectionObserver

ヘッダーの現在地ハイライトは、**`IntersectionObserver`** で各セクションが画面中央付近に入ったかを検出する方式です。

```ts
const observer = new IntersectionObserver(
  (entries) => entries.forEach((e) => {
    if (e.isIntersecting) setActiveSection(e.target.id);
  }),
  { rootMargin: "-50% 0px -50% 0px" }
);
```

`rootMargin: "-50% 0px -50% 0px"` で、**ビューポートの中央線** が交差判定領域になるようにしています。スクロールイベント＋座標計算で書くより圧倒的に軽量で、**スクロール中も60fps を維持** できます。

### 5-7. トップページの "情報の密度" を絞る判断

つい先日のPRで、**トップページに表示するプロジェクトとスキルカテゴリを最大3件に制限** しました (それ以前は6件)。「すべて見る →」リンクから一覧ページに遷移できるので、トップページは **"ハイライトを見せる場所"** に特化させた形です。

これは技術というより設計判断ですが、**「情報を全部見せる ≠ 情報をよく伝える」** という Web デザインの基本原則に立ち戻った変更です。スクロール量が減り、ファーストビュー直下で各セクションの存在が把握できるようになりました。

## 6. 「サーバーレス」を徹底するためのデータ設計

このサイトで一番尖っているのが、**バックエンドを置かずに Google Sheets を "DB" として使っている** ことだと思います。

### 6-1. なぜ Google Sheets なのか

理由はシンプルで、**「個人ポートフォリオに DB の運用負荷を払いたくない」** からです。Firestore も Supabase も無料枠はありますが、

- 認証情報やデータ移行の手間がかかる
- 更新するたびに管理画面を開くか、自前のCMSを書かないといけない
- それでも結局、**スプレッドシートをコピペして整形する** 作業に近いなら、最初からスプレッドシートでよくないか?

という判断です。Google Sheets なら、**スマホからでも更新できる**、**他人と共有もできる**、**変更履歴も残る**、**API は無料枠で十分**、と個人プロジェクトの要件をほぼ満たします。

### 6-2. 取得後すぐに型付けする

Google Sheets API のレスポンスは2次元配列なので、そのままでは `string[][]` で扱いづらい。`fetchSheetData()` の中で **1行目をヘッダー扱いしてオブジェクト化** しておき、`mapXxxData()` でドメインの型に変換する、という2段階を踏んでいます。

```ts
fetchSheetData<SheetRow>("Projects").then(rows => {
  setPortfolio(mapPortfolioData(rows));
});
```

この設計のおかげで、**Sheets の列順を変えてもアプリの型は壊れない**。ヘッダー名さえ合っていればアプリ側はそのまま動きます。スプレッドシートを「テーブル」ではなく「列名つきの行コレクション」として扱う、という小さな抽象化です。

### 6-3. 長文は Markdown に逃がす

プロジェクト詳細のような長文を Sheets に入れるのは、編集体験的に厳しい (改行・コードブロック・テーブルなど)。なので、**詳細記事は `public/projects/<id>.md` に Markdown で配置** し、`react-markdown` + `remark-gfm` でレンダリングしています。

```tsx
<ReactMarkdown remarkPlugins={[remarkGfm]}>{markdown}</ReactMarkdown>
```

Markdown 用のスタイルは `src/index.css` の `.prose-custom` クラスにまとめていて、**見出し・テーブル・コードブロック・引用・水平線** がブループリント調で描画されます。コードブロックの背景色も CSS 変数で、ライト/ダーク両対応です。

ちなみに **この記事自体が `public/projects/portfolio.md` に置かれている** のは、ある種のドッグフーディングです。

## 7. PWA 化と細かなパフォーマンス対応

### 7-1. オフラインでも開ける

`vite-plugin-pwa` で Service Worker を自動生成しています。ビルド時に `precaching` が効くので、**初回アクセス後は HTML/CSS/JS/フォントが端末にキャッシュされ、オフラインでも開ける** ようになります。

`runtimeCaching` の戦略は前述のとおりで、

- **Google Fonts**: `CacheFirst` + 1年TTL (フォントは変わらない)
- **Google Sheets API**: `NetworkFirst` + 1時間TTL (内容が変わる)
- **Markdown**: `NetworkFirst` + 1日TTL (内容が変わるが頻度は低い)

それぞれの「変化頻度」に合わせて使い分けています。

### 7-2. Manifest でホーム画面に追加できる

`manifest` も `vite.config.ts` で指定していて、**iOS/Android のホーム画面に追加するとスタンドアロン表示** されます。テーマカラー (`#2563eb`) と背景色 (`#f4f6fb`) を指定しているので、起動時のスプラッシュもポートフォリオの世界観で統一されます。

### 7-3. CLS / LCPを意識した実装

派手なところではないですが、

- フォントは `link rel="preconnect"` で接続を先行
- `M PLUS Rounded 1c` (本文) と `Concert One` (装飾) と `JetBrains Mono` (コード) を **役割ごとに分けて使い分け**
- 画像はサムネイルだけ即時、本文はスクロールで遅延 (`whileInView`)

このあたりを地道にやっています。Vite の SWC ビルドは生成物が小さく、PWA キャッシュも効くので、**初回以降のアクセスはほぼ瞬時** で開きます。

### 一気に作らなかったことのメリット

このプロジェクトは **意図的に少しずつ進めて** います。8か月という期間自体が長いというより、

- **コンセプトが途中で変わる前提で作る**
- **作っては寝かせ、別の視点で見直す**
- **新しい技術 (PWA、CSS変数テーマ、Framer Motion `useScroll` 等) を後から取り込む余地を残しておく**

ということを重視しました。期限を切らない代わりに、**「気になったら手を入れる」** を続けた結果が今の姿です。個人プロジェクトの強みを活かす意味でも、これは正解だったと思っています。

## 9. AI エージェントとの協働について

このプロジェクトの後半 (Phase 4 以降) は、ほぼすべての変更を **AI エージェント (Claude Code) と協働** で進めています。1年前の生成AIと比べて、エージェントは **プロジェクト全体を読み取りながら整合性のあるパッチを返してくる** ので、リファクタや横断的な変更で特に強いです。

たとえば「ハードコードしている色を CSS 変数に置換して」という指示で、**15ファイル分の差分を一度に整える** のはエージェントの得意分野でした。一方で、

- **どんなテーマにするか**
- **どこまで方眼紙のメタファーを徹底するか**
- **トップページに何件出すべきか**

といった **「何を作るべきか」の判断は、人間が決めるしかありません**。AI が速く書ける時代だからこそ、**設計判断と要件定義の重みが相対的に増している** という実感があります。

このサイトのGitログには、**`Co-Authored-By: Claude` のクレジットがついたコミットが多数残っています**。これも一種のドキュメントで、**「どこは AI と一緒に書いて、どこは自分が決めたか」** を後から辿れるようにしてあります。

## 10. さいごに ── ポートフォリオを "作品" として作るということ

長くなりましたが、最後にこの記事で **一番伝えたかったこと** を書いておきます。

ポートフォリオはよく「自分の作品の一覧」として捉えられますが、**ポートフォリオ自体が一番手前にある作品** だと考えています。訪れた人が最初に触れるのはサイトのUIで、そこで感じた印象が、奥に並んだ各プロジェクトの読み解かれ方を決めてしまいます。

なので、

- **見えない場所のアーキテクチャ** (Google Sheets による疑似DB、CSS 変数によるテーマ抽象化、PWA キャッシュ戦略)
- **見える場所のディテール** (方眼紙の二層グリッド、コーナーマーク、スクロール連動タイムライン)
- **設計思想の一貫性** (「ポートフォリオは設計図である」という比喩を全部の判断に通す)

の3つを、**全部同じ重みで** 揃えるよう意識しました。どれかひとつだけ尖っていても、たぶん作品としては成立しません。

8か月かかってようやくこの形になりましたが、まだやりたいことはあります。たとえば、

- スキルレーダーチャートをカテゴリ間で重ねて比較する機能
- プロジェクト詳細の Markdown に **目次・読了時間** を自動で付与する
- 検索機能 (Google Sheets ではなく Pagefind 等の静的全文検索を組み込む案も)
- アクセシビリティ ── **今はキーボードナビゲーションが弱い** ので、`focus-visible` リングを整える

このあたりは Phase 6 以降の宿題として残しておこうと思います。
