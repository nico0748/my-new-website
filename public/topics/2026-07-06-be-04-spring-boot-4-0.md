# Spring Boot 4.0 GA — モジュール化・JSpecify による null 安全・API バージョニングで新世代へ

> トピック: Spring Boot 4.0 正式版｜出典: Spring 公式ブログ（spring.io）｜種別: メジャーリリース（GA）

## 一行サマリー

2025-11-20 に GA した Spring Boot 4.0 は、コードベースの全面モジュール化による小さく焦点の絞られた JAR、JSpecify による全体的な null 安全、REST の API バージョニングと HTTP Service Client、そして Spring Framework 7・Java 25 対応（Java 17 互換維持）を柱に、エンタープライズ Java の新しい基盤世代を始めた。

## 🔰 初学者向け（何が・なぜ重要か）

Spring Boot は、Java で業務システムやWeb APIを作るときの定番フレームワークです。その大型アップデート 4.0 が正式リリースされました。

大きな変化のひとつは「モジュール化」です。これまで Spring Boot は大きな塊で、使わない機能まで一緒についてきがちでした。4.0 では中身が細かい部品（小さな JAR）に分割され、「必要な部品だけ入れる」ができるようになりました。結果としてビルドや起動が軽くなります。

もうひとつは「null（値が無い状態）による事故を減らす」改善です。Java で最も多いバグの一つが「値が無いのに使おうとして落ちる」ことです。4.0 は JSpecify という共通の目印を使い、「ここは null になりうる／ならない」をフレームワーク全体で一貫して扱えるようにしました。加えて、API のバージョン管理（v1・v2 の切り替え）や、Java のインターフェースからHTTPクライアントを自動生成する仕組みも入り、実務の面倒が減ります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **全面モジュール化**: Spring Boot のコードベースを小さく焦点化した JAR 群に再編。ビルド時間・起動速度の改善と、必要依存のみの取り込みを可能に。
- **JSpecify による null 安全**: ポートフォリオ全体で JSpecify アノテーションを採用。エンドポイントの任意パラメータを `@Nullable` で記述でき、Spring が入力バインド時にそれを解釈。
- **API バージョニング / HTTP Service Client**: REST での複数バージョン管理を支援。Java インターフェースを HTTP クライアントへ変換する仕組みを提供。
- **基盤アップグレード**: Spring Framework 7 上に構築。Jakarta EE 11（Tomcat 11 / Hibernate ORM 7 / Hibernate Validator 9）、Kotlin 2、Jackson 3、JUnit 6 へ更新。Java 25 を一級サポートしつつ Java 17 互換を維持。
- リリース 2025-11-20（GA）。数値・対応表は公式ブログに基づく。

### 仕組み / 背景（なぜこう設計されたか）

モジュール化の狙いは、モノリシックな依存が招く「使わない機能まで読み込む・起動が重い・ネイティブ化しにくい」問題の解消だ。小さな JAR に割ることで、GraalVM ネイティブイメージやサーバーレスのコールドスタートに効くほか、依存グラフが明確になり脆弱性管理もしやすくなる。トレードオフは、依存の粒度が細かくなることで初期の BOM（依存版管理）や移行時のパッケージ再構成の手間が増える点。

JSpecify 採用は、Java エコシステムに乱立していた null 注釈（`javax.annotation`、Spring 独自、JetBrains 系など）を、業界横断の標準へ収斂させる動きだ。フレームワークが一貫した意味論で null 可能性を表現できると、静的解析ツールや Kotlin との相互運用で「どこが null になりうるか」を機械的に検査でき、実行時 NPE を設計段階で減らせる。Framework 7 / Jakarta EE 11 への土台更新は、`javax.*` から `jakarta.*` への完全移行と最新ランタイム前提を固める世代交代であり、これが 4.0 を「新基盤の起点」と位置づける理由になっている。

### 実務での使いどころ・移行の勘所

新規プロジェクトは 4.0 系から始めるのが自然。既存の Boot 3.x からの移行では、(1) `jakarta.*` 名前空間・依存の版ずれ、(2) Jackson 3 / JUnit 6 など基盤ライブラリの非互換、(3) モジュール分割に伴う依存宣言の見直し、が主な作業になる。null 安全は既存コードに `@Nullable` を段階導入し、静的解析（NullAway 等）と併用すると効果が高い。API バージョニングは、破壊的変更を伴う公開 API を運用しているサービスで移行負債を平準化できる。Java 17 互換は維持されるため、ランタイム更新は段階的に進められる。

## 📖 用語解説（このトピックとのつながり）

- **Spring Framework 7**: Boot 4.0 の土台となるコアフレームワーク世代。
- **JSpecify**: null 可能性を表す業界横断の標準アノテーション。4.0 の null 安全の要。
- **Jakarta EE 11**: `jakarta.*` 名前空間の企業向け Java 標準群。移行の主対象。
- **API バージョニング**: v1/v2 など公開APIの版を管理する仕組み。4.0 で標準支援。
- **HTTP Service Client**: Java インターフェースから HTTP クライアントを生成する機能。

## 影響範囲 / 推奨アクション

- 対象: Java/Spring でバックエンドを構築する組織。
- 推奨: 新規は 4.0 系で開始。既存 3.x は jakarta 名前空間と基盤ライブラリ非互換を洗って計画移行。null 安全は段階導入＋静的解析併用。
- 注意: モジュール分割に伴う依存宣言の再構成、Jackson 3 / JUnit 6 の非互換に留意。
