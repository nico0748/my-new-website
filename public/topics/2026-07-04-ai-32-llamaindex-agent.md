# LlamaIndex がエージェント基本機能を標準装備 — RAG 中心のエージェント基盤へ

> トピック: 開発トピック｜出典: medium（atnoforgenai）｜種別: ai

## 一行サマリー
RAG（検索拡張生成）ライブラリの代表格 LlamaIndex が、ツール呼び出しやマルチステップ推論といったエージェント基本機能を標準装備し、単なる検索ライブラリから RAG 中心のエージェント基盤へと進化している。

## 🔰 初学者向け（何が・なぜ重要か）
LLM は学習時点の知識しか持たず、社内文書や最新情報は知りません。そこで「質問に関係する文書をまず検索し、その内容を LLM に渡して答えさせる」手法が RAG（Retrieval-Augmented Generation）です。LlamaIndex はこの RAG を作るための定番ライブラリで、文書の取り込み・分割・ベクトル化・検索を一手に引き受けてくれます。

これまで LlamaIndex は「検索して答える」までが得意分野でしたが、いまや「検索した内容をもとに次の行動を判断し、道具を使い、複数ステップで作業を進める」というエージェント的な振る舞いも標準で持つようになりました。図書館の司書が本を探すだけでなく、調べた内容をもとに次に何をすべきか自分で段取りするイメージです。

これにより「まず社内知識を調べてから動くエージェント」を、別のフレームワークと組み合わせずに一つで作れるようになります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
出典によれば、LlamaIndex はツール呼び出し・エージェントループ・複数ステップ推論といった機能をコアに取り込んだ。従来は検索インデックス（VectorStoreIndex 等）とクエリエンジンが中心だったが、これに AgentWorkflow やイベント駆動のオーケストレーション層が加わった構成とされる。具体的な API 名・バージョンは要確認。

### 仕組み / 背景（なぜこう設計されたか）
RAG の精度は「どの文書をどのタイミングで検索するか」に大きく左右される。単純な一発検索では、複雑な質問に対応できない。そこでエージェント的に「検索→評価→再検索→統合」を回すことで、より的確な回答を生成できる。LlamaIndex は自らの強みである検索基盤の上にエージェント機能を乗せることで、RAG に最適化された差別化を図っている。トレードオフは、汎用エージェント用途では LangGraph などより機能が薄い場合がある点。

### 実務での使いどころ・移行の勘所
社内ドキュメント検索、ナレッジベース Q&A、レポート生成など「知識検索が主役」のエージェントに最適。ベクトル DB 連携が豊富で、既存の RAG パイプラインをそのままエージェント化できる。汎用マルチエージェント協調が主目的なら LangGraph や CrewAI（項目28）との比較を推奨。

## 📖 用語解説（このトピックとのつながり）
- **RAG（検索拡張生成）**: 関連文書を検索して LLM に渡す手法。LlamaIndex の中核用途。
- **ベクトル検索**: 文章を数値ベクトル化し意味的に近いものを探す仕組み。RAG の検索エンジン。
- **エージェントループ**: 観察→判断→行動を繰り返す実行サイクル。今回標準装備された機能。
- **ツール呼び出し**: LLM が外部関数を実行する仕組み。エージェントの手足。
- **オーケストレーション**: 複数ステップ・複数コンポーネントの実行制御。エージェント基盤の中枢。

## 影響範囲 / 推奨アクション
RAG を軸に据える開発者・チームに影響。既に LlamaIndex で RAG を組んでいるなら、別フレームワークを追加せずエージェント化できるため移行コストが低い。RAG 中心か汎用エージェント中心かで基盤を選び分けるとよい。
