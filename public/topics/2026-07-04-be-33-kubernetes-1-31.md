# Kubernetes 1.31 — Gateway API GA・ネイティブsidecar・GPUスケジューリング

> トピック: 開発トピック｜出典: loginline｜種別: backend

## 一行サマリー
Kubernetes 1.31 では Gateway API の GA 化、ネイティブ sidecar コンテナ、AI ワークロード向けの GPU スケジューリング強化が進み、コンテナオーケストレーションが AI/エッジ時代の要求に追随している。

## 🔰 初学者向け（何が・なぜ重要か）
Kubernetes（k8s）は、たくさんのコンテナ（アプリの入れ物）を自動で配置・再起動・スケールしてくれる「コンテナの管制塔」です。1.31 はその管制塔の機能アップデートで、3つの改善が目立ちます。

1つ目の Gateway API は、外部からアプリへの入口（ルーティング）を扱う新しい標準の仕組みが「正式版」になったこと。旧来の Ingress より柔軟で、複数チームでの入口共有がしやすくなります。2つ目の sidecar は、本体アプリに寄り添って動く補助コンテナ（ログ収集やプロキシなど）を、より安全に起動・終了できるようになったこと。3つ目の GPU スケジューリングは、AI の学習・推論で欲しい GPU をうまく割り当てる改善です。

要するに「入口・お供・AI用ハードの扱い」が、より実務に耐える形に整った、というアップデートです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
- **Gateway API GA**：L4/L7 ルーティングを役割分担（インフラ担当と開発者）で扱える API が安定版へ。Ingress の後継的位置づけ。
- **ネイティブ sidecar**：`initContainers` に restartPolicy を付与する形で sidecar のライフサイクルを本体と協調させる仕組み。起動順・終了順の問題を緩和。
- **GPU/デバイススケジューリング**：AI ワークロード向けにアクセラレータ割り当てを改善（DRA=Dynamic Resource Allocation 等の周辺機能を含む。細かな安定度・バージョン対応は要確認）。

### 仕組み / 背景（なぜこう設計されたか）
Ingress は表現力が限られ、実装ごとの独自アノテーションが乱立していました。Gateway API はロールベースの拡張可能なモデルでこれを整理します。sidecar は従来「通常コンテナ」として運用され終了順の制御が難しかったため、専用ライフサイクルが導入されました。GPU 対応は AI ワークロードの急増という需要に応えるものです。

### 実務での使いどころ・移行の勘所
Ingress を多用してきた環境は Gateway API への段階移行を検討する段階に入りました。ただし既存の Ingress コントローラ資産や運用スクリプトへの影響があるため、並行運用しつつ移すのが安全です。sidecar パターン（サービスメッシュのプロキシ等）を使う構成は、ネイティブ sidecar 対応で運用が安定します。

## 📖 用語解説（このトピックとのつながり）
- **Gateway API**：k8s の新しいルーティング標準。Ingress の後継で 1.31 で GA。
- **sidecar**：本体に寄り添う補助コンテナ。ネイティブ対応でライフサイクルが改善。
- **DRA（動的リソース割り当て）**：GPU 等の特殊資源を柔軟に割り当てる仕組み。AI ワークロードで重要。
- **オーケストレーション**：多数コンテナの配置・回復・拡縮を自動化する営み。k8s の本業。

## 影響範囲 / 推奨アクション
プラットフォームエンジニア・SRE・AI基盤担当に影響します。Ingress 中心の運用者は Gateway API の学習と検証を開始し、AI ワークロードを載せるクラスタは GPU スケジューリング機能の対応状況を確認するのが推奨アクションです。
