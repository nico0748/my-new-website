# F5 NGINX のHTTP/2プロキシ・gRPCモジュールに CVSS 9.2 のヒープオーバーフロー（CVE-2026-42055）— 未認証でDoS/RCE

> トピック: 脆弱性ニュース ｜ 出典: F5 Security Advisory / SecurityWeek / Security Affairs ｜ 種別: ヒープベースのバッファオーバーフロー（CWE-122）に起因する未認証DoS／条件付きRCE

## 一行サマリー

全Webサイトの約3割を支えるWebサーバー／リバースプロキシ NGINX の **HTTP/2プロキシ（ngx_http_proxy_v2_module）およびgRPCモジュール** に、未認証の攻撃者がワーカープロセスのヒープを破壊できる CVSS 9.2 の脆弱性 CVE-2026-42055 が見つかり、F5が定例外（out-of-band）でパッチを公開した。同時公開のHTTP/3 use-after-free（別CVE、当vaultでは6/20に記事化済み）とは**別モジュール・別メカニズム**であり、リバースプロキシ／APIゲートウェイ構成では即時更新が必要。

## 🔰 初学者向け（何が・なぜ問題か）

**NGINX（エンジンエックス）** は、世界中のWebサイトの「玄関係」を務めているソフトです。利用者からのアクセスをいったん受け止め、必要に応じて裏側のアプリへ取り次ぐ（これを**リバースプロキシ**と言います）役割を担っています。マイクロサービスやAPIの世界では、この取り次ぎに **gRPC** や **HTTP/2** という高速な通信方式がよく使われ、NGINXがその中継役になります。

今回の問題は、その中継処理の中で、受け取ったデータを置くための「作業机（ヒープというメモリ領域）」のサイズ計算を誤り、**机からはみ出して隣の書類を上書きしてしまう**点にあります。攻撃者がログインせずに、わざと細工したHTTP/2やgRPCの通信を送りつけると、玄関係（NGINXのワーカープロセス）の作業机が壊れ、最低でもサービス停止（玄関が閉まる＝サイトダウン）、条件が揃えば玄関係そのものの乗っ取り（任意コード実行）まで起こりえます。

たとえるなら、受付が書類を所定の机に整理して置くとき、規格外に分厚い束を渡されて机からあふれさせ、隣に置いてあった「次にやることリスト」を攻撃者の指示で上書きされてしまうイメージです。NGINXは多くのサービスの正面玄関なので、ここが落ちる・乗っ取られると、その裏にあるサービス全体に影響します。**ログイン不要**で攻撃できる点が特に危険です。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **CVSS**: 9.2（Critical）。**未認証・ネットワーク経由**で悪用可能。
- **CWE分類**: CWE-122（Heap-based Buffer Overflow）。`ngx_http_proxy_v2_module`（HTTP/2プロキシ）および `ngx_http_grpc_module`（gRPC中継）における境界処理の不備。
- **前提条件**: NGINXが**リバースプロキシ／gRPC中継**として構成され、HTTP/2またはgRPCのアップストリーム連携を処理していること。RCEは **ASLRが無効、または回避可能な環境**という条件下で成立しうる（多くの環境では実害はまずDoS）。
- **影響/修正版**:
  - NGINX Open Source → mainline **1.31.2** または stable **1.30.3**（本CVE対応）。
  - NGINX Plus → **R37 P2.1** または **R36 P6**。
  - NGINX Gateway Fabric → **2.6.4**（本件含む複数CVEに対応）。
  - 正確な対象は製品系統ごとにF5公式で要確認。
- **悪用状況**: 本稿時点で実環境悪用は**要確認**。ただしF5は「前回同等深刻度のNGINX Criticalは公開3日後に悪用が始まった」と注意喚起しており、緊急度は高い。
- **同時公開の別CVE**: HTTP/3モジュール（ngx_http_v3_module）のuse-after-free（CVE-2026-42530、CVSS 9.2）は**別の脆弱性**で、当vaultでは2026-06-20に別記事化済み。混同しないこと。

### メカニズム（攻撃が成立する仕組み）

以下は概念レベルの説明であり、武器化可能な手順・PoC・正確なオフセット等は含めない。

1. **信頼境界の所在**: リバースプロキシのNGINXは、「外部クライアントからの未認証リクエスト」を最前線で受け、HTTP/2フレームやgRPCメッセージを解析してアップストリームへ橋渡しする。この**認証前のプロトコル解析**が攻撃面となる。
2. **長さ計算の不備（ヒープ過書き）**: HTTP/2のフレーム長やgRPCのメッセージ長を扱う際、確保したヒープバッファのサイズと実際にコピーするデータ量の突き合わせが不十分なため、**ヒープ領域の範囲外に書き込み（overflow）**が発生する。攻撃者はフレーム/メッセージの長さフィールドを操作してこのプリミティブを引き出す。
3. **DoSから実行制御への幅**: 範囲外書き込みは、まずワーカープロセスのクラッシュ（**DoS＝サービス停止**）を引き起こす。さらに、ヒープ上の関数ポインタやメタデータを攻撃者が制御できる位置に配置できれば、制御フロー奪取による**任意コード実行（RCE）**へ発展しうる。RCE成立はASLR等の保護機構の有無に依存する。
4. **正面玄関としての波及**: NGINXは多数サービスの入口を兼ねるため、ワーカー停止はサイト全体の可用性喪失に、ワーカー乗っ取りは背後サービスへの侵入・トラフィック傍受の足場になる。

脆弱性クラスとしては、「認証前にバイナリプロトコル（HTTP/2・gRPC）を解析するコンポーネントの長さ検証欠如によるヒープオーバーフロー」という、高性能プロキシで再発しやすい型に該当する。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: NGINXワーカープロセスの**異常終了・コアダンプ・worker再起動の頻発**、HTTP/2/gRPCトラフィックでの不正なフレーム長・プロトコルエラー急増、プロキシ／gRPCエンドポイントへの不審な接続パターン。`error_log` のワーカークラッシュ記録とアップストリーム異常を監視。
- **暫定緩和（パッチ前）**: 即時更新が難しい場合、不要であれば**HTTP/2やgRPC中継の無効化**を検討（`listen ... http2;` やgRPC location の停止）。WAF/L7前段で異常なHTTP/2フレームをフィルタ。攻撃元の到達経路を限定。
- **恒久対策**: 製品系統に応じた固定版（OSS 1.31.2/1.30.3、Plus R37 P2.1/R36 P6、Gateway Fabric 2.6.4）へ即時更新。更新後はワーカークラッシュ履歴を確認し、侵害兆候をハント。
- **運用示唆**: リバースプロキシ／APIゲートウェイは「未認証のバイナリプロトコルを最前線で解析する」高リスク地点。プロトコルモジュールの有効範囲を必要最小限にし、F5のOOB含む緊急パッチを即時運用に組み込む。

## 📖 用語解説（このニュースとのつながり）

- **NGINX / リバースプロキシ**: アクセスを受け止め裏のアプリへ取り次ぐWebサーバー。意味＝サービスの「正面玄関」。本件＝この玄関のプロキシ処理が破壊される。
- **HTTP/2 / gRPC**: 高速・多重化された通信方式と、その上のRPCフレームワーク。本件＝これらを中継するモジュールが攻撃面。
- **ヒープオーバーフロー（CWE-122）**: 動的メモリ（ヒープ）の確保枠を超えて書き込む欠陥。本件＝DoS〜RCEの根本原因。
- **ASLR**: メモリ配置をランダム化する保護機構。本件＝無効/回避時にRCEが現実化する条件。
- **out-of-band patch（定例外パッチ）**: 通常サイクル外で緊急配布される修正。本件＝F5が緊急公開した点が重大性の証左。

## 影響範囲 / 推奨対応

影響を受けるのは、NGINX（OSS / Plus / Gateway Fabric）を**HTTP/2プロキシ・gRPC中継**として運用する全組織。多数サービスの入口を兼ねるため、被害は可用性喪失（DoS）から、条件次第で背後サービスへの侵入（RCE）まで及ぶ。推奨対応は、(1) 製品系統に応じた固定版（OSS 1.31.2/1.30.3、Plus R37 P2.1/R36 P6、Gateway Fabric 2.6.4）への即時更新、(2) 不要なHTTP/2・gRPC機能の暫定無効化、(3) ワーカークラッシュ・コアダンプの監視と侵害ハント、(4) 同時公開のHTTP/3 UAF（CVE-2026-42530）対応漏れがないかの確認。
