# vLLM — 投機的デコーディングと Model Runner V2 で LLM 推論をさらに高速・低コスト化

> トピック: vLLM 推論最適化（投機的デコーディング / Model Runner V2）｜出典: vLLM 公式ブログ｜種別: 推論基盤（MLOps）更新

## 一行サマリー

LLM 推論エンジンの定番 vLLM は、小さなドラフトモデルが提案したトークンを本体モデルが並列検証する「投機的デコーディング」で受理率0.7以上のとき1.3〜2倍高速化し、Qwen3 密モデルで Model Runner V2 を既定化、DeepSeek V4 の MTP 投機的デコード対応など、推論の速度とコストを継続的に改善している。

## 🔰 初学者向け（何が・なぜ重要か）

LLM（生成 AI）に文章を書かせるとき、モデルは「1単語（トークン）ずつ順番に」生成します。1個作って、それを踏まえて次の1個、また次の1個…と直列に進むので、長い文章ほど時間がかかります。これが「LLM の返事が遅い・GPU 代が高い」原因のひとつです。

**vLLM** は、この推論（AI に答えを出させる処理）を効率よく回すための人気エンジンです。今回話題の **投機的デコーディング** は、待ち時間を縮める賢いトリックです。イメージは「下書き係と清書係の分業」。まず小さくて速い『下書きモデル』が「次はたぶんこの数単語だろう」と先読みして一気に提案し、本体の『清書モデル』がそれを一括でチェックします。当たっていれば複数単語をまとめて確定できるので、1個ずつ作るより速くなります。外れたら本体が正しく作り直すので、答えの質は落ちません。

つまり「品質はそのまま、速さと GPU コストだけ改善する」――そういう美味しい最適化を、vLLM が使いやすく提供しているわけです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- vLLM は prefix caching、paged attention、投機的デコーディングをネイティブ支援。
- **投機的デコーディング**: 小さなドラフトモデルがトークンを提案 → 本体が並列検証。ドラフトの受理率 ≥ 0.7 で **1.3〜2倍** の高速化。設定は `--speculative-model` と `--num-speculative-tokens`。コード中心ワークロードで出力100万トークンあたりコストを約19.4%削減との報告。
- **DeepSeek V4** に MTP（Multi-Token Prediction）投機的デコード対応を追加（ハードニング）。
- **Model Runner V2** が Qwen3 密モデルで既定化。
- Red Hat の記事では gpt-oss で投機的デコードによる性能改善も報告。

### 仕組み / 背景（なぜこう設計されたか）

自己回帰生成は本質的に直列だ。各トークンの生成には、それまでの全トークンを条件とした1回のフォワードパスが要る。ところが GPU は「大量の並列計算は得意だが、直列の依存には弱い」ため、デコード段は演算器を遊ばせがち（メモリ帯域律速）になる。投機的デコーディングはこの遊びを埋める。安価なドラフトモデルが k 個先のトークンを推測し、本体モデルはその k 個を「1回のフォワードパスでまとめて検証」する。検証は各位置の確率分布を比べ、ドラフトと一致する限り受理、外れた位置以降を棄却して本体の分布から引き直す。これにより出力分布は本体単独と数学的に等価（＝品質不変）を保ちつつ、1パスで複数トークンを確定できる。効果は「受理率」に強く依存し、ドラフトが本体をよく近似するほど（受理率が高いほど）加速する。だから受理率0.7前後が損益分岐の目安になる。

paged attention は別方向の最適化で、KV キャッシュ（過去トークンの中間状態）を OS の仮想メモリのように「ページ」で管理し、断片化を防いでバッチ内の多数リクエストを高密度に詰める。prefix caching は共通プレフィックス（システムプロンプト等）の KV を再利用して再計算を省く。Model Runner V2 はモデル実行経路の再設計で、これらの機構を効率よく束ねる基盤更新にあたる（詳細は要確認）。トレードオフは、投機的デコードは追加のドラフトモデル分の VRAM とチューニング（k 値・ドラフト選択）を要し、受理率が低いワークロードでは逆に遅くなりうる点。

### 実務での使いどころ・移行の勘所

チャット、コード補完、エージェントの反復推論など「レイテンシが支配的なデコードバウンド」用途で投機的デコードが効く。特にコード生成は定型パターンが多くドラフトの受理率が上がりやすい。導入は (1) ドラフトモデルの選定（本体を良く近似する小型モデル）、(2) `--num-speculative-tokens`（k）のチューニング、(3) 実ワークロードで受理率・スループット・p99 レイテンシを実測、が要点。バッチが大きくスループット律速の状況では効果が薄れるため、レイテンシ重視かスループット重視かで採否を判断する。SGLang や TensorRT-LLM との比較では、vLLM は機能網羅性とコミュニティの広さが強み。

## 📖 用語解説（このトピックとのつながり）

- **推論エンジン**: 学習済みモデルに実際に答えを出させる実行基盤。vLLM が代表格。
- **自己回帰生成**: トークンを直列に1つずつ生成する方式。デコードが遅い根本原因。
- **投機的デコーディング**: ドラフトモデルの先読みを本体が並列検証して加速する手法。品質は不変。
- **KV キャッシュ / paged attention**: 過去トークンの中間状態と、その断片化を防ぐ管理法。高密度バッチの鍵。
- **受理率（acceptance rate）**: ドラフト提案が採用される割合。投機的デコードの加速度を左右する。

## 影響範囲 / 推奨アクション

- 影響: LLM を自前ホストする全チーム。推論レイテンシと GPU コストに直結。
- 推奨: レイテンシ支配の用途で投機的デコードを試験。ドラフトモデルと k 値をチューニングし、受理率・スループット・p99 を実測。スループット律速時は採否を再検討。
