# DBOS — Postgres だけで実現する Durable Execution（耐障害ワークフロー）

> トピック: DBOS Transact（Postgres バックエンドの Durable Execution ライブラリ）と Go SDK 公開｜出典: DBOS 公式 / GitHub｜種別: バックエンド基盤（OSS ライブラリ）

## 一行サマリー

DBOS Transact は、専用のオーケストレーションサーバーを立てずに Postgres 互換 DB へ状態を保存するだけで「途中で落ちても正確に再開できる」耐障害ワークフロー（Durable Execution）を実現する軽量ライブラリ。2026年6月18日に Go SDK が公開され、Python / TypeScript / Java / Go の4言語が揃った。ワークフローのメタデータとアプリのデータを同一 DB トランザクションで更新できるため、冪等性・原子性の扱いが単純になる。

## 🔰 初学者向け（何が・なぜ重要か）

長い処理——たとえば「注文を受けて、在庫を引いて、決済して、メールを送る」——の途中でサーバーが落ちたら、どこまで進んだか分からなくなり、二重決済や送信漏れが起きかねません。これを防ぐ「途中で落ちても、続きから正確にやり直せる」仕組みが Durable Execution（耐障害実行）です。

従来この仕組みは、Temporal のような専用サーバーを別に立てて使うのが定番でした。DBOS はそれを、**普段使っている Postgres データベースに状態を書くだけ**で実現します。たとえるなら、進捗を書き留める専用の管理室を新設する代わりに、もともとある帳簿（DB）に一緒に記録してしまう発想。管理する部品が減り、構成がシンプルになります。

2026年に Go 版も出て主要言語が出そろい、AI エージェントのように「複数ステップを確実にこなす」処理の土台としても注目されています。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **アーキテクチャ**: DBOS Transact はライブラリとしてアプリに組み込み、Postgres 互換 DB に接続してワークフロー/ステップの状態（チェックポイント）を保存する。重量級のオーケストレーションサーバーは不要。
- **正確な実行保証**: ステップが DB 操作を伴う場合、ステップ本体とチェックポイント記録を**単一トランザクション**で実行（piggyback）することで exactly-once を保証する。障害後はチェックポイントから未完のワークフローを再開する。
- **多言語対応**: Python / TypeScript に続き、2026年6月18日に Go パッケージを公開（Go SDK は Python/TS とのパリティを向上）。Java 版も登場。
- **エコシステム**: 2026年4月に Databricks Lakebase 連携を発表。観測性（DBOS Conductor の集約ビュー、メトリクス連携、RBAC）を強化。

### 仕組み / 背景（なぜこう設計されたか）

耐障害実行の核心は「どこまで実行したか（progress）を、失われない形で記録する」こと。専用オーケストレータ方式（Temporal 等）は、この記録を外部サービスが持つため、アプリ DB との間で二相的な整合（アプリの更新とワークフロー状態の更新がずれない保証）を別途工夫する必要があった。

DBOS の設計思想は「ワークフロー状態もアプリと同じ Postgres に置けば、両者を同一トランザクションで更新でき、整合の悩みが消える」というもの。トランザクショナル DB の ACID 性をそのまま冪等性・原子性の基盤に流用する点が要諦だ。トレードオフは、Postgres の可用性・スループットにワークフロー基盤が乗る（DB がボトルネック/単一障害点になり得る）こと、そして超大規模・超長時間ワークフローでは専用オーケストレータの水平分散が有利な場面が残ること。逆に、既に Postgres を中心に据えたアプリでは、新規コンポーネントを増やさずに耐障害性を得られる利点が大きい。

### 実務での使いどころ・移行の勘所

決済・注文・データパイプライン・AI エージェントの多段処理など「途中失敗を許さない」バックエンド処理に適する。既存アプリが Postgres を使っているなら、ワークフロー/ステップにアノテーションを付けるだけで段階導入でき、インフラ追加が最小で済む。移行の勘所は、ステップ境界の切り方（冪等にしたい単位でステップ化する）と、外部副作用（メール送信等、DB 外の操作）の扱い——外部呼び出しは冪等キーやチェックポイントと併せて設計する。Temporal 等と比較検討する場合は、DB 集約による単純さと、専用オーケストレータのスケール特性を要件に照らして選ぶ。

## 📖 用語解説（このトピックとのつながり）

- **Durable Execution（耐障害実行）**: 障害後も未完処理を正確に再開できる実行モデル。本トピックの主題。
- **チェックポイント**: ワークフローの進捗を保存した記録。DBOS は Postgres に保存する。
- **exactly-once**: 各ステップがちょうど一度だけ実行される保証。DB トランザクションで担保。
- **冪等性（idempotency）**: 同じ操作を複数回行っても結果が変わらない性質。再実行の安全性の鍵。
- **Temporal**: 代表的な専用オーケストレータ型の耐障害実行基盤。DBOS の比較対象。

## 影響範囲 / 推奨アクション

Postgres 中心のバックエンドは、インフラを増やさずに耐障害ワークフローを導入できる選択肢を得た。多段処理や AI エージェントの信頼性向上に有効。導入時はステップ境界の設計と外部副作用の冪等化を重視し、Temporal 等とはスケール要件で比較すること。
