# OpenTelemetry for Cloudflare Workers — エッジ関数を OTLP JSON で計測する

> トピック: 開発トピック｜出典: oneuptime｜種別: backend

## 一行サマリー
Cloudflare Workers のようなエッジ実行環境でも OpenTelemetry を使い、OTLP over HTTP-JSON でトレースを外部バックエンドに送って可観測性を確保できるようになってきた。

## 🔰 初学者向け（何が・なぜ重要か）
「エッジ関数」は、ユーザーに近い世界中のデータセンターで動く小さなサーバー処理だ。反応が速い反面、実行環境が特殊で、普通のサーバー用の監視ツールがそのままでは動かないことが多い。世界中に散らばった無数の小さな出張所を、本社からどう見張るか、という問題に近い。

Cloudflare Workers は Node.js ではなく独自のランタイム（V8 isolate ベース）で動くため、gRPC のような重い通信手段が使いにくい。そこで OTel の HTTP-JSON 版（OTLP/HTTP + JSON エンコーディング）を使い、`fetch` で計測データを送る方法が現実解になる。

これにより「どのリクエストが、どのエッジ拠点で、どれだけ時間を食ったか」を、いつものダッシュボードで見られるようになる。散らばった出張所からの日報を、本社の統一フォーマットで集約するイメージだ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Workers ランタイムは Node 互換 API が限定的で、OTLP/gRPC エクスポータは動かしにくい。そこで OTLP/HTTP + JSON を使い、Workers の `fetch` API 経由でスパンを送信する構成が採られる。`tail` handler や `ExecutionContext.waitUntil` を活用し、レスポンス返却をブロックせずにテレメトリを非同期送出するのがポイント。対応ライブラリ・SDK バージョンの詳細は要確認。

### 仕組み / 背景（なぜこう設計されたか）
エッジ環境は起動が極短命で、長寿命のバックグラウンド・エクスポート・ループを前提にできない。従来の OTel SDK は常駐プロセスでのバッチ送信を想定していたため、そのままでは相性が悪かった。JSON エンコーディングは gRPC より軽量で、制約された isolate でも扱いやすく、`waitUntil` でリクエスト完了後にフラッシュすることで短命性の問題を回避する。

### 実務での使いどころ・移行の勘所
Cloudflare Workers / Python Workers / エッジ FaaS を本番運用するなら、trace_id をリクエスト先の下流サービスへ伝播させ、エッジ〜オリジンを貫く1本のトレースにするのが肝。送信のたびに `fetch` が走るのでコストとレイテンシに注意し、サンプリングは必須。OneUptime など OTLP を受けられるバックエンドを選べば移行は容易。

## 📖 用語解説（このトピックとのつながり）
- **OTLP/HTTP + JSON**: gRPC を使わない軽量な送信方式。エッジ制約下での計測を可能にする。
- **isolate**: Workers が使う軽量な V8 実行単位。コンテナより速く起動するがランタイムが限定される。
- **waitUntil**: レスポンス後も処理を継続させる Workers API。非同期テレメトリ送出の要。
- **context propagation**: trace_id を下流へ引き継ぐ仕組み。エッジ〜オリジンの一貫トレースに必須。

## 影響範囲 / 推奨アクション
エッジ / サーバーレスでバックエンドを組む開発者・SRE が対象。エッジ関数を「監視の死角」にしないため、OTLP/HTTP-JSON での計測を早期に導入すべき。
