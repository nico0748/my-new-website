# Rust 1.96 — Copy な新 Range 型が 10 年来の設計負債を清算、assert_matches! も安定化

> トピック: Rust 1.96.0 の安定化機能｜出典: Rust 公式ブログ｜種別: 言語定期リリース（6週間サイクル）

## 一行サマリー

Rust 1.96.0（2026年5月28日）は、`Copy` を実装しスライス添字にも使える新しい Range 型ファミリー（`core::range`）を安定化して長年のエルゴノミクス問題を解消し、パターン照合アサーション `assert_matches!` / `debug_assert_matches!` の追加、Wasm リンクの厳格化、cargo の脆弱性2件（CVE-2026-5223 / CVE-2026-5222)の修正を含む。

## 🔰 初学者向け（何が・なぜ重要か）

Rust はメモリ安全性で知られるプログラミング言語で、6週間ごとに新版が出ます。1.96 の目玉は「範囲（Range）」の作り直しです。

Rust では `0..10` のように書くと「0から9までの範囲」を表す値が作れます。ところがこの範囲、初期の設計で「数え上げの進行状況も自分で覚える」作りにしてしまったため、「コピーして使い回す」ことができない不便な型になっていました。ノートに「1〜10ページを読む」とメモしたはずが、メモ自体に「いま3ページ目」という状態が書き込まれてしまい、同じメモを2人で共有できない——そんな感じの不便さです。

新しい Range 型は「範囲というただのデータ」と「数え上げの進行状況」を別々の型に分けました。これで範囲を気軽にコピーしたり、構造体に持たせたりできるようになります。10年越しの「あるある不満」の解消で、地味ながら Rust プログラマ全員に効く変更です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **`core::range` の新 Range 型ファミリーを安定化**: 新 `Range` / `RangeFrom` / `RangeToInclusive` などが `Copy` を実装し、スライスのインデックスとしても使用可能。イテレーションは分離された専用イテレータ型（`RangeIter`、`RangeFromIter`、`RangeToInclusiveIter` 等）が担う。
- **`assert_matches!` / `debug_assert_matches!` 安定化**: 値がパターンに一致するかを検査し、失敗時は値の `Debug` 表現つきでパニックする。
- **`From<T>` 実装の追加**: `AssertUnwindSafe<T>`、`LazyCell<T, F>`、`LazyLock<T, F>` 向け。
- **Wasm リンクの変更**: `--allow-undefined` を既定でリンカに渡さなくなり、未定義シンボルは「env モジュールからの import に変換」ではなくリンクエラーになる。
- **cargo の脆弱性修正**: CVE-2026-5223（シンボリックリンクを含む crate tarball 展開、中深刻度）、CVE-2026-5222（正規化 URL での認証、低深刻度）。

### 仕組み / 背景（なぜこう設計されたか）

旧 `std::ops::Range` が `Copy` でないのは、`Range` 自身が `Iterator` を実装しているため。イテレータは `next()` のたびに内部状態（現在位置）を進める可変オブジェクトであり、これが暗黙コピーできると「コピーの片方だけ進めたのに両方進んだつもりになる」型のバグを量産する——という判断で `Copy` が外された。しかしその代償として、「範囲」を純粋なデータとして構造体に格納したり関数へ渡したりする普通の用途が borrow/clone の儀式まみれになった。

新設計は「データとしての範囲（Copy 可能な純粋な値）」と「プロセスとしてのイテレーション（状態を持つ専用イテレータ）」を型で分離する。これは C++ の range/iterator 分離や、Rust 自身の `String`/`Chars` の関係と同じ「値と走査の分離」原則への回帰であり、初期設計が『便利さのために2責務を1型に混ぜた』ことの是正といえる。エディション互換のため旧型は残り、新旧の共存期間が続く（標準ライブラリ API の移行スケジュールは要確認）。

Wasm の未定義シンボル厳格化は「暗黙に env import へ変換されて実行時に初めて壊れる」挙動を「リンク時に確実に失敗」へ前倒しするもので、Rust の「早期にエラーを」哲学の一貫。既存の「意図的に env import に頼っていた」ビルドには非互換となる。

### 実務での使いどころ・移行の勘所

- 範囲を設定値・ドメインモデルとして持ち回るコード(例: バッファのウィンドウ、ID 帯域、時間区間)は新 Range 型への置き換えでライフタイム/clone ノイズが消える。新規コードから `core::range` を優先採用し、境界で旧型と変換するのが現実的。
- `assert_matches!` はテストで `if let ... else panic!` や `matches!` + `assert!` の複合を置き換え、失敗時の診断情報（Debug 出力）が標準で付く分だけ調査効率が上がる。
- Wasm ターゲット（wasm32-unknown-unknown 等）の CI があるプロジェクトは、1.96 更新でリンクエラーが出ないか要確認。意図的な env import はリンカフラグの明示で復元する。
- cargo の CVE 対応として、私設レジストリや git 依存を多用する環境は早めのツールチェーン更新を。

## 📖 用語解説（このトピックとのつながり）

- **Copy トレイト**: 代入で暗黙にビットコピーされる型の印。旧 Range はイテレータ状態を持つため付けられなかった。
- **IntoIterator**: 「値からイテレータを作る」変換。新 Range はこれ経由で専用イテレータを生成し、自身は純データに徹する。
- **`assert_matches!`**: パターン照合版 assert。enum の特定バリアントであることの検査などテストの定番イディオムを一行にする。
- **未定義シンボル（Wasm）**: リンク時に実体が見つからない関数。従来は宿主側 import に化けて実行時まで露見しなかった。
- **crate tarball**: crates.io で配布されるパッケージのアーカイブ。CVE-2026-5223 はその展開処理の問題。

## 影響範囲 / 推奨アクション

- 全 Rust プロジェクト: `rustup update` で 1.96 系へ（cargo CVE 修正のため、ビルド環境の更新は特に推奨）。
- Wasm ビルドを持つプロジェクト: リンク挙動の変更をリリースノートで確認しテストを一周。
- ライブラリ作者: 公開 API での新 Range 型採用方針（MSRV との兼ね合い）を検討。
