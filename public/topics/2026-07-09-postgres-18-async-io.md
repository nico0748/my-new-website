# PostgreSQL 18 の非同期 I/O（AIO）が本番安定 — シーケンシャルスキャン等で最大3倍

> トピック: PostgreSQL 18 の非同期 I/O サブシステム｜出典: PostgreSQL 公式｜種別: DB のメジャー機能の成熟

## 一行サマリー

PostgreSQL 18（2025年9月リリース）が導入した非同期 I/O（AIO）サブシステムは、ポイントリリース 18.3 以降で本番運用に耐える安定機能として定着した。複数の I/O 要求を逐次待ちせず並行発行することで、シーケンシャルスキャン・ビットマップヒープスキャン・VACUUM などで最大3倍の性能向上が報告されている。`io_method` で `worker` / `io_uring` / `sync` を選べる。

## 🔰 初学者向け（何が・なぜ重要か）

データベースの仕事の多くは「ディスクからデータを読む」ことです。従来の PostgreSQL は、この読み込みを「1つ頼んで、返ってくるのを待って、次を頼む」という順番待ち方式でやっていました。ディスクは応答に少し時間がかかるので、この「待ち時間」の積み重ねが遅さの一因でした。

非同期 I/O（AIO）は、この待ち方を変えます。「これとこれとこれ、まとめて読んでおいて」と複数の読み込みを一度に頼み、返ってきたものから順に処理します。飲食店で注文を1品ずつ運ぶのではなく、まとめて厨房に通すイメージです。これにより、大量のデータを読む処理（全件スキャンや掃除作業＝VACUUM）が最大3倍ほど速くなることがあります。PostgreSQL 18 で入ったこの機能が、その後の細かな修正（18.3 以降）を経て「本番でも安心して使える」段階になった、というのが今回の位置づけです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- PostgreSQL 18（2025年9月）で **非同期 I/O（AIO）サブシステム**を導入。18.3 以降で本番レディと整理（2026年時点の最新ポイントは 18 系が継続）。
- 対応操作: シーケンシャルスキャン、ビットマップヒープスキャン、VACUUM。ベンチマークで**最大3倍**の性能向上。
- `io_method` で挙動を選択: `worker`（I/O 用ワーカープロセス）、`io_uring`（Linux ネイティブの非同期 I/O）、`sync`（従来の同期挙動を維持）。
- `io_combine_limit` / `io_max_combine_limit` で I/O のまとめ具合を制御。新ビュー `pg_aios` で使用中のファイルハンドルを監視可能。
- 18 系ではほかに UUIDv7、OAuth、論理レプリケーション強化なども導入。数値は環境依存のため自環境で実測を推奨。

### 仕組み / 背景（なぜこう設計されたか）

従来 PostgreSQL の読み込みは同期ブロッキングだった。1ブロックの読み取りを OS に要求し、返るまでそのバックエンドプロセスは待つ。ストレージが高速化（NVMe SSD、クラウドのネットワークストレージ）しても、「1要求ずつ待つ」構造では、ストレージが持つ**並行処理能力（I/O 並列度）**を使い切れない。特に全件スキャンや VACUUM のように大量ブロックを順に舐める処理では、待ち時間が積み上がってスループットが頭打ちになる。

AIO はこの構造的ボトルネックを解く。read-ahead（先読み）で必要そうなブロック群を先回りして複数同時に発行し、ストレージの並列性を引き出す。`io_uring` は Linux カーネルが提供する高効率な非同期 I/O インターフェースで、システムコールのオーバーヘッドを抑えつつ多数の I/O をキューイングできる。`worker` 方式は io_uring が使えない環境向けに、専用プロセスが非同期 I/O を代行する。トレードオフは複雑性と可搬性——io_uring はカーネルバージョンやセキュリティ設定（seccomp 制限で無効化される場合）に依存し、環境によっては `worker` や `sync` が無難。ワークロードによっては先読みが空振りしオーバーヘッドになることもあるため、`sync` へ戻せる設計が安全弁として用意されている。

### 実務での使いどころ・移行の勘所

分析系クエリ（大規模スキャン）、データウェアハウス的用途、巨大テーブルの VACUUM/メンテナンスが重いシステムで恩恵が大きい。逆に、小さなインデックス参照が中心の OLTP では効果が限定的なこともある。移行は 18 系へアップグレード後、`io_method` を既定（多くは環境で自動選択）から検証し、`io_uring` が使えるカーネル・権限かを確認。`pg_aios` と既存のスキャン系メトリクスで効果を実測し、想定外の悪化があれば `sync` へ切り戻す。クラウドマネージド（RDS/Cloud SQL 等）では設定可否がベンダー依存のため、提供状況を確認する。数値効果はワークロード次第なので自環境ベンチが必須。

## 📖 用語解説（このトピックとのつながり）

- **非同期 I/O（AIO）**: 複数の読み込みを並行発行する方式。本トピックの中核。
- **io_uring**: Linux の高効率な非同期 I/O インターフェース。`io_method` の高性能オプション。
- **VACUUM**: 不要になった行領域を回収する PostgreSQL の保守処理。AIO の恩恵対象。
- **io_combine_limit**: 隣接 I/O をまとめる上限設定。チューニングの要点。
- **pg_aios**: 実行中の非同期 I/O を可視化する新システムビュー。監視に利用。

## 影響範囲 / 推奨アクション

- 影響: PostgreSQL を使う全システム、特に大規模スキャン/メンテナンスが重い用途。
- 推奨: 18 系へ更新し `io_method` を検証（io_uring 可否をカーネル/権限で確認）。`pg_aios` で効果測定、悪化時は `sync` へ切戻し。マネージド DB は提供状況を確認。
