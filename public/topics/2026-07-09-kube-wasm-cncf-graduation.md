# Kube-Wasm が CNCF Graduated へ前進 — WASM をコンテナ並みに扱う、コールド起動2.3ms

> トピック: Kube-Wasm の CNCF Graduated 承認（暫定）｜出典: byteiota / CNCF｜種別: クラウドネイティブ基盤の成熟

## 一行サマリー

WebAssembly モジュールを Kubernetes クラスタ上でコンテナと同格の「第一級コンピュート」としてスケジューリングする Kube-Wasm が、CNCF 技術監督委員会により最高成熟度の Graduated へ暫定承認された（セキュリティ監査待ち）。runc コンテナのコールドスタート約620msに対し、WASM は平均2.3msと桁違いに速く、コンテナと WASM の共存が現実的な運用フェーズへ入った。

## 🔰 初学者向け（何が・なぜ重要か）

いまのサーバー運用は「コンテナ」という仕組みが主役です。アプリを箱に詰めて、Kubernetes（クバネティス）という管理者がその箱を何百台ものサーバーに配って動かします。ただコンテナには弱点があり、「起動に少し時間がかかる（数百ミリ秒）」ため、リクエストが来た瞬間だけ立ち上げる用途では待たされます。

WebAssembly（WASM）は、この「箱」をずっと軽く・速くしたものと考えてください。起動がミリ秒単位と圧倒的に速く、安全に隔離できます。Kube-Wasm は、この WASM を「コンテナと同じように Kubernetes で扱える」ようにする仕組みで、今回それが CNCF（クラウド技術の中立な標準団体）から最高ランクの成熟度として認められる方向になりました。つまり「コンテナと WASM を同じ土俵で混ぜて使う」運用が、実験ではなく本番の選択肢になりつつある、という節目です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- CNCF 技術監督委員会が Kube-Wasm を **Graduated（最高成熟度）へ暫定承認**。セキュリティ監査は Q2 2026 完了予定（承認はその完了が前提）。
- 性能: WASM モジュールの平均コールドスタート **2.3ms** に対し、標準 runc コンテナは **約620ms**。
- WASM ワークロードを従来コンテナと**同一ノード上で並走**させる「共存（coexistence）」戦略。Krustlet や、runwasi 経由で WasmEdge/Wasmtime を CRI（Container Runtime Interface）へ統合する方式が中核。
- 前提として WebAssembly 3.0 が2026年6月に正式化、WASI 0.3.0 でネイティブ async、コンポーネントモデルが安定——WASM を共有メモリなしで合成できる土台が整った。
- 数値・承認状況は監査結果で変動しうるため一次情報での再確認を推奨（要確認）。

### 仕組み / 背景（なぜこう設計されたか）

コンテナのコールドスタートが遅い根本理由は、Linux 名前空間・cgroup・ファイルシステムレイヤ・プロセス起動という「OS レベルの隔離環境」を毎回構築するからだ。これは強力な互換性（どんな Linux バイナリも動く）と引き換えに、起動コストと1インスタンスあたりのメモリを重くする。WASM は逆に、隔離を「言語ランタイム内のサンドボックス」で実現する。OS の重い機構を立てず、検証済みのバイトコードを軽量 VM 上で走らせるため、起動が桁違いに速く、密度も高い。

Kube-Wasm の設計の妙は、この WASM を**既存の Kubernetes 抽象（Pod・スケジューラ・CRI）にそのまま載せる**点にある。runwasi が CRI を実装することで、kubelet から見れば WASM も「ただのコンテナ」として扱え、既存の運用ツール（監視・スケジューリング・ネットワーク）が流用できる。トレードオフは互換性——WASM は任意のネイティブバイナリを動かせず、対応言語・ライブラリのエコシステムに縛られる。ゆえに「全面置換」ではなく「起動速度と密度が効く部分（イベント駆動・関数・プラグイン）だけ WASM、残りはコンテナ」という共存戦略が現実解になっている。CNCF の Graduated 承認は、この共存モデルが production レディに近づいた指標だ。

### 実務での使いどころ・移行の勘所

スケールゼロからの即応が要るイベント駆動処理、マルチテナントの関数実行、プラグイン/拡張の安全なサンドボックス、超高密度なマイクロサービスで効果が大きい。導入は既存 Kubernetes に runwasi ベースのランタイムクラスを追加し、対象ワークロードだけ WASM 化する形が現実的で、クラスタ全体を作り替える必要はない。注意点は、WASM 対応言語・ライブラリの制約、WASI 0.2/0.3 のまだらなツール成熟度、そしてセキュリティ監査完了前は本番採用を慎重にすべき点。まずは非クリティカルな関数系ワークロードで密度・起動速度の効果を実測し、コンテナと共存させながら適用範囲を広げるのがよい。

## 📖 用語解説（このトピックとのつながり）

- **Kube-Wasm**: WASM を Kubernetes の第一級コンピュートとして扱うプロジェクト。本トピックの主役。
- **CNCF / Graduated**: クラウドネイティブの中立団体と、その最高成熟度ランク。今回暫定承認。
- **CRI（Container Runtime Interface）**: kubelet とランタイムの標準境界。runwasi がここに WASM を接続。
- **runwasi / Krustlet / WasmEdge / Wasmtime**: WASM を k8s に載せる実装群。共存戦略の中核。
- **コールドスタート**: 起動遅延。WASM(2.3ms) と runc(620ms) の差が採用動機。

## 影響範囲 / 推奨アクション

- 影響: Kubernetes 運用チーム、サーバーレス/エッジ基盤。起動速度・密度・コストに波及。
- 推奨: runwasi ベースのランタイムクラスを追加し非クリティカルな関数系から検証。セキュリティ監査完了状況を確認のうえ本番判断。
