# Go 1.25 の実験的 Green Tea GC — GCのメモリ局所性を作り直す

> トピック: 開発トピック｜出典: Go 公式リリースノート｜種別: backend

## 一行サマリー
Go 1.25（2025-08）は実験的な新ガベージコレクタ「Green Tea GC」を導入した。メモリを「span」単位でまとめて走査することでキャッシュ効率を高め、GC のオーバーヘッドを削減する設計で、マイクロサービスや高スループットなサーバーの尾レイテンシ改善が期待される。

## 🔰 初学者向け（何が・なぜ重要か）
プログラムは実行中にメモリ（作業机）を借りて使いますが、使い終わった分を返さないと机がすぐ埋まってしまいます。Go には「ガベージコレクタ（GC）」という後片付け係がいて、もう使われていないメモリを自動で回収してくれます。おかげで開発者は解放を手作業でやらずに済みます。

ただ、この後片付けには代償があります。片付け係が机を見回っている間、プログラム本体は少し待たされたり、遅くなったりします。特にメモリのあちこちに散らばったデータを一つずつ確認する方式だと、机の引き出しを開けたり閉じたりする往復（メモリアクセス）が増え、効率が落ちます。

Green Tea GC は、この「見回り方」を作り直した新方式です。バラバラに確認するのではなく、近くにあるメモリをまとめて一気に処理することで、往復を減らして片付けを効率化します。結果として、サーバーがカクつく瞬間（遅延のスパイク）が減りやすくなる、というのが狙いです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Go 1.25 は 2025 年 8 月に Go 1.24 の 6 か月後としてリリースされ、実験的な Green Tea ガベージコレクタを搭載した（`GOEXPERIMENT=greenteagc` で有効化。既定は従来 GC）。Green Tea はオブジェクト単位ではなくメモリの「span（同サイズオブジェクトの連続領域）」単位でマーキングを行い、空間的局所性（spatial locality）を活かして走査中のキャッシュミスを削減する。Go チームの報告では GC の CPU オーバーヘッドを一定程度削減し得るとされるが、効果はワークロード依存で、正式採用時期や既定化は今後のリリースで要確認。

### 仕組み / 背景（なぜこう設計されたか）
Go の GC は並行マーク&スイープ（アプリと同時に動く）で低レイテンシを重視してきたが、マーク段階はヒープ上のポインタを辿るためメモリアクセスが分散し、現代CPUのキャッシュ階層と相性が悪い場面があった。Green Tea は「どのオブジェクトを辿るか」ではなく「どのメモリ領域（span）をまとめて調べるか」へ視点を変え、隣接オブジェクトの一括処理でメモリ帯域とキャッシュを有効活用する。これは GC 研究でいう局所性最適化の一種で、トレードオフとして span 管理のための追加ロジックと、割り当てパターンによっては効果が薄いケースが存在する。だからこそ既定ではなく実験フラグでの提供から始まっている。

### 実務での使いどころ・移行の勘所
GC 由来の p99 レイテンシや CPU コストが課題の高負荷サービス（API ゲートウェイ、ストリーム処理など）で試す価値がある。導入は `GOEXPERIMENT=greenteagc` でビルドし、本番相当の負荷で before/after を計測するのが鉄則。アロケーション削減（オブジェクト再利用、`sync.Pool`）といった従来のチューニングは引き続き有効で、Green Tea はそれを置き換えるものではなく補完する。実験機能ゆえ、本番投入は計測とロールバック手順を用意したうえで段階的に。

## 📖 用語解説（このトピックとのつながり）
- **ガベージコレクタ（GC）**: 不要になったメモリを自動回収する仕組み。Go は並行マーク&スイープ方式。
- **Green Tea GC**: span 単位で走査し局所性を高める Go 1.25 の実験的 GC。キャッシュ効率を狙う。
- **span**: 同サイズオブジェクトが並ぶ連続メモリ領域。Green Tea のマーキング単位。
- **空間的局所性（spatial locality）**: 近接データへ連続アクセスするとキャッシュが効きやすい性質。最適化の核。
- **尾レイテンシ（p99）**: 遅い側の応答時間。GC のスパイクが悪化させやすく、改善対象。

## 影響範囲 / 推奨アクション
Go で高スループット/低レイテンシのサーバーを運用するチームに関係。実験フラグで有効化し、本番相当負荷で計測して効果を判断。既定 GC のままでも問題ないため、恩恵が見込める GC 敏感なワークロードから検証するのが推奨。
