# Code with Claude 2026 — 数百の並列サブエージェントと Agent SDK の従量課金分離

> トピック: Claude Code / Agent SDK の 2026 更新｜出典: Anthropic 公式（発表・ドキュメント）｜種別: 製品/エージェント機能更新

## 一行サマリー

Anthropic は Code with Claude 2026 で、1セッション内で数百のサブエージェントを並列に走らせ結果を検証してから報告する多エージェント制御、Agent Skills オープン標準による「一度作ればアプリ間で使える」スキル、そして 2026-06-15 からの Agent SDK / GitHub Actions 利用の従量課金分離を打ち出し、エージェント運用のコストを予測可能にしつつ、より大規模な自律作業へ踏み込んだ。

## 🔰 初学者向け（何が・なぜ重要か）

Claude Code は、ターミナルやアプリから Claude に開発作業を任せられる道具です。2026 年の更新で、その「任せ方」が大きくなりました。

いちばん大きいのは「サブエージェント（部下 AI）の大量並列」です。これまで1体の AI が順番に作業していたのを、親 AI が仕事を分解し、数百体の部下 AI に同時にやらせ、結果を確認してから報告する、という進め方ができます。大量のファイルを跨ぐ調査や、コードベース規模の移行が現実的な時間で終わります。

もうひとつは「Agent Skills のオープン標準化」です。スキル（特定作業のやり方をまとめた部品）を一度作れば、Claude アプリでも Claude Code でも API でも使い回せます。

そしてお金の話。2026-06-15 から、Agent SDK（自作エージェントの土台）や GitHub Actions での利用が、対話的な Claude Code とは別建ての従量課金になりました。自動で大量に動くぶんのコストが分けて見えるようになり、予算管理がしやすくなります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **多エージェント制御（subagents）**: 親エージェントが計画を立て、単一セッション内で数百の並列サブエージェントを起動。各出力を検証してからユーザーへ報告する。
- **Agent Skills（オープン標準）**: クロスプラットフォーム移植を狙うオープン標準として公開。「一度作れば Claude アプリ / Claude Code / API で共通利用」を標榜。
- **MCP 統合**: MCP のサーバー実装が 200 超に到達。Playwright・Slack・GitHub 等を設定1行で接続。
- **課金の分離**: 2026-06-15 から Agent SDK と Claude Code GitHub Actions の利用を対話的 Claude Code と別に計量。従量（per-token）課金へ寄り、コストを予測可能化。
- **大規模移行**: 上位モデル（Opus 系）と組み合わせ、数十万行規模のコードベース移行をキックオフからマージまで遂行しうる。
- 発表・日付は Anthropic の公表に基づき、機能詳細は要確認。

### 仕組み / 背景（なぜこう設計されたか）

サブエージェント並列化の核心は「コンテキスト分離とファンアウト」だ。1体のエージェントに全部を詰め込むと、コンテキストが汚れ、長い作業ほど注意が散り精度が落ちる。親が仕事を独立サブタスクに分解し、それぞれをクリーンな文脈の子に委譲すれば、各子は狭い問題に集中でき、親は結果を検証して統合する。これは分散処理の map-reduce に近い発想で、探索的な調査（多数ファイルの横断把握）や独立に並列化できる変換（同型の修正を多数箇所へ）で特に効く。トレードオフは、並列度に比例してトークン消費と API コールが増える点で、だからこそ課金分離が同時に導入された——自動で大量に動く負荷を、対話利用と分けて可視化・予算化する必要があるからだ。

Agent Skills のオープン標準化は、MCP が「エージェントと外部ツールの接続」を標準化したのに続く「作業手順（スキル）の可搬化」の試みだ。スキルを特定製品に閉じず標準フォーマットで書ければ、アプリ・CLI・API・他ベンダー間で再利用でき、ロックインを避けられる。トレードオフは、標準が広く使われるかどうかにエコシステムの価値が依存する点（採用が伴わなければ「もう一つの標準」で終わる）。従量課金分離は、エージェント利用が「人が対話する時間課金」から「機械が自動で回すトークン課金」へ移る構造変化を反映しており、予測可能性と引き換えに、設計の不備（無駄な再検索・過剰な並列）が直接コストに跳ねる規律を開発者に課す。

### 実務での使いどころ・移行の勘所

大規模な調査・リファクタ・移行（依存更新、API 置換、命名規約の一括変更）はサブエージェント並列が効く典型。ただし並列度はコストに直結するため、タスク分解の粒度と上限を設計し、危険操作（削除・外部書き込み・デプロイ）には人間承認境界を必ず置く。Agent SDK で自作エージェントを本番運用するなら、2026-06-15 の課金分離を前提に、トークン消費をタスク単位で計測・予算化する。Agent Skills は横断的な定型作業（テスト生成、レビュー観点、社内規約適用）を標準スキル化して再利用性を上げる。MCP 接続は便利だが、接続するサーバーの権限範囲を最小化し、監査ログを残す運用が前提。

## 📖 用語解説（このトピックとのつながり）

- **Claude Code**: ターミナル/アプリから開発を任せるエージェント。本トピックの舞台。
- **サブエージェント**: 親が委譲する子エージェント。並列ファンアウトの単位。
- **Agent Skills**: 作業手順をまとめた再利用部品。オープン標準として可搬化。
- **MCP**: エージェントと外部ツールを繋ぐ標準プロトコル。実装 200 超。
- **従量課金（per-token）**: 使用トークンに応じた課金。Agent SDK 利用が別建てに。

## 影響範囲 / 推奨アクション

- 対象: Claude Code / Agent SDK でエージェント開発・自動化を行うチーム。
- 推奨: 大規模調査/移行はサブエージェント並列を活用。定型作業は Agent Skills 化。2026-06-15 の課金分離を前提に予算設計。
- 注意: 並列度はコスト直結。危険操作に人間承認境界。MCP 接続は権限最小化＋監査ログ。
