# HTMX — ビルド不要のサーバー駆動 UI（クライアント状態を持たない設計）

> トピック: 開発トピック｜出典: dev.to｜種別: frontend

## 一行サマリー
HTMX は HTML 属性だけで部分更新やインタラクションを実現し、ビルド工程やクライアント状態を持たないサーバー駆動 UI を可能にするライブラリで、SPA の複雑さへの反動として2026年も存在感を示している。

## 🔰 初学者向け（何が・なぜ重要か）
React に代表される SPA（シングルページアプリ）は強力ですが、状態管理・ビルド設定・大量の JavaScript など、覚えることが多く重くなりがちです。すべての画面が複雑な JavaScript を必要とするわけではないのに、です。

HTMX は「原点回帰」の発想です。サーバーが HTML の断片を返し、HTMX はそれを画面の一部に差し込むだけ。ボタンに `hx-get="/items"` のような属性を付けるだけで、ページ全体を再読み込みせずに一部だけ更新できます。JavaScript をほとんど書かず、状態はサーバーが持ちます。

これは「クライアントに頭脳を持たせず、サーバーに任せる」設計です。ビルド工程も要らず、HTML を書ける人ならすぐ扱えます。すべてを置き換えるものではありませんが、管理画面や CRUD 中心のアプリでは、SPA より圧倒的にシンプルに済むことが多いのです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
HTMX は `hx-get` / `hx-post` / `hx-swap` / `hx-target` などの HTML 属性で AJAX・部分更新・イベント駆動を宣言的に記述する。レスポンスは JSON ではなく HTML フラグメントで、クライアント状態を持たない（hypermedia-driven）。ビルドツール不要で単一スクリプトの読み込みで動く。バージョン詳細は要確認。

### 仕組み / 背景（なぜこう設計されたか）
SPA はクライアントとサーバーで状態が二重化し、複雑さとバンドル肥大を招いた。HTMX は HATEOAS の考えに立ち返り「サーバーが返す HTML こそがアプリの状態」とすることで、この二重化を消す。「単純さへの回帰」という潮流の象徴。

### 実務での使いどころ・移行の勘所
管理画面、フォーム中心、CRUD、コンテンツサイトで威力を発揮。逆にリッチなクライアント状態やオフライン、複雑なリアルタイム UI では React 等が適する。既存サーバーサイド（Django/Rails/Laravel 等）に段階的に足せるのが利点。「いつ HTMX で足りて、いつ React が要るか」の見極めが肝。

## 📖 用語解説（このトピックとのつながり）
- **ハイパーメディア駆動**: HTML 自体がアプリの状態と遷移を表す設計。HTMX の思想的基盤。
- **HTML フラグメント**: 画面の一部だけの HTML。HTMX が差し込む単位。
- **部分更新（swap）**: ページ全体でなく対象要素だけを差し替える動作。`hx-swap` で制御。
- **SPA**: クライアント側で状態と描画を担う方式。HTMX が対極として位置づけられる。

## 影響範囲 / 推奨アクション
サーバーサイド中心のチームや、SPA の複雑さに疲れた開発者に有効。CRUD 中心の新規画面で試し、要件に応じて SPA と使い分けるのがよい。
