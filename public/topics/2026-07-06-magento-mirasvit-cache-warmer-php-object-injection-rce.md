# Magento用Mirasvit Cache WarmerのPHPオブジェクトインジェクションで未認証RCE（CVE-2026-45247）

> トピック: 脆弱性ニュース ｜ 出典: Sansec / Imperva / The Hacker News / CISA KEV ｜ 種別: 信頼できないデータのデシリアライズ（PHPオブジェクトインジェクション → RCE）

## 一行サマリー

Magento 2 の人気キャッシュ拡張「Mirasvit Full Page Cache Warmer」に、Cookie経由で送られた攻撃者データを無防備に `unserialize()` へ渡す欠陥（CVSS 9.8）が発覚。認証なしでPHPコードを実行でき、CISA は実環境での悪用を確認してKEVに登録済み。1.11.12 以降への更新が必須。

## 🔰 初学者向け（何が・なぜ問題か）

Magento はネットショップを作るための代表的なソフトで、Mirasvit Cache Warmer はその表示を速くする“キャッシュ温め係”の追加部品（拡張機能）です。ECサイトの裏側で静かに働くこの部品に、深刻な穴が見つかりました。

ポイントは「Cookie（クッキー）」の扱いです。Cookie は、ブラウザとサイトの間でやり取りされる小さなメモのようなもの。通常はサイト側が発行し、ブラウザが持ち歩きます。ところがこの拡張は、`CacheWarmer` という名前のCookieの中身を、送られてきたままの状態で「PHPの“文字列を部品（オブジェクト）に組み立て直す”機能」に流し込んでいました。

たとえるなら、「宅配便の伝票（Cookie）に書かれた指示を、中身を検めずにそのまま倉庫の自動組み立てラインに投入してしまう」ようなものです。攻撃者は伝票に“細工した設計図”を書いておきます。倉庫のライン（PHP）は、それを言われるがまま組み立てるうちに、既存の部品（Magentoや周辺ライブラリに元から入っている部品＝「ガジェット」）を意図しない順序でつなぎ合わせ、最終的に「サーバ上で任意の命令を実行する」という危険な組み合わせを完成させてしまいます。

しかもこのCookieは、ログインしていない普通の買い物客のアクセスでも受け付けられます。つまり誰でも、認証なしで、ネットショップのサーバを乗っ取れる可能性があるということです。実際に世界中で悪用が観測されており、ゲーム系・ビジネス系サイトが特に狙われています。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **CVSS**: 9.8 Critical（未認証・ネットワーク経由）。
- **CWE**: CWE-502（信頼できないデータのデシリアライズ）＝PHPオブジェクトインジェクション。
- **前提条件**: 該当拡張が有効なMagento 2ストアフロントへHTTPリクエストを送れること。認証・管理者権限・特別な設定は不要。
- **影響製品/バージョン**: Mirasvit Full Page Cache Warmer for Magento 2、1.11.12 未満の全バージョン。
- **修正版**: 1.11.12（パッチ公開 2026-05-25）。
- **悪用状況**: CISA が 2026-06-03 にKEV登録（BOD 22-01により連邦機関へ期限付き是正指示）。Imperva/Sansec が実際の攻撃トラフィック（base64化シリアライズオブジェクトを含むペイロード）を観測。米・英・仏・豪が主要標的、ゲーム/ビジネスサイト中心。

### メカニズム（攻撃が成立する仕組み）

本件は「外部入力を検証なく `unserialize()` に渡す」という、PHPで最も危険なアンチパターンの一つ。

1. **信頼境界の破れ（入力の無検証デシリアライズ）**: 拡張はクライアント提供の `CacheWarmer` Cookie を受け取り、その値をPHPネイティブの `unserialize()` に**どのクラスを復元してよいか制限せずに**渡す。攻撃者が完全に制御できるCookieが、そのままオブジェクト生成の材料になる時点で信頼境界が崩れている。

2. **プリミティブ＝オブジェクトインジェクション**: `unserialize()` は、シリアライズ文字列の指示に従って任意クラスのインスタンスを復元し、その過程で `__wakeup()` `__destruct()` などのマジックメソッドを発火させる。攻撃者は「復元されるクラスとそのプロパティ値」を握れる。これがPHPオブジェクトインジェクション（POI）というプリミティブ。

3. **ガジェットチェーンによるRCEへの橋渡し**: POI単体では即コード実行にならないが、Magento本体やその依存ライブラリに既存する“ガジェット”（マジックメソッド発火時に危険な処理へ連鎖するクラス群）を数珠つなぎにすることで、ファイル書き込み・関数呼び出し・最終的な任意コード実行へと段階的に昇格する。攻撃者は自前でコードを持ち込む必要はなく、“アプリに元からある部品”を悪用して武器化する点が本質。

4. **クラスの一般原理**: `unserialize()` に未検証入力を渡す限り、対象アプリのクラス群次第でPOI→RCEが成立し得る。対策の王道は、ユーザ入力にネイティブ `unserialize()` を使わないこと（JSON等の安全な形式へ移行、または `allowed_classes` の厳格指定）。本件の修正もこの方向。

※武器化可能なシリアライズ構文・具体的ガジェット連鎖・PoCは記載しない（概念レベル）。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: ストアフロントのリクエストで `CacheWarmer` Cookie を持つものを監視。シリアライズPHPオブジェクトはbase64化すると先頭が `Tz` `Qz` `YT` になりやすいため、`CacheWarmer:(Tz|Qz|YT)` に一致するCookie値は悪用試行の強い指標。WAFログ・アクセスログで該当パターンと異常な子プロセス生成を相関。
- **暫定緩和**: パッチ適用が即時に難しい場合、WAFで `CacheWarmer` Cookie 内のシリアライズ様パターンをブロック。可能なら該当拡張を一時無効化。管理面・ストアフロントのアクセス監視を強化。
- **恒久対策**: Mirasvit Full Page Cache Warmer を 1.11.12 以降へ更新。更新後も侵害痕跡（不審なPHPファイル・Web shell・cron/設定改ざん）の遡及調査を実施。
- **運用示唆**: ECプラットフォームはサードパーティ拡張が攻撃面になりやすい。拡張のSBOM把握とパッチ運用、Magento系のガジェット悪用（過去のTrojanOrder等）を踏まえた監視を。

## 📖 用語解説（このニュースとのつながり）

- **デシリアライズ / `unserialize()`（CWE-502）**: 文字列をプログラム内のオブジェクトへ復元する処理。未検証の外部入力を渡すと攻撃者にオブジェクト生成を乗っ取られる。本件の根本原因。
- **PHPオブジェクトインジェクション（POI）**: 攻撃者が復元されるオブジェクトを制御し、マジックメソッド経由で不正処理を誘発する手法。
- **ガジェットチェーン**: アプリに既存するクラス群を連鎖させ、POIをRCE等へ昇格させる“部品の数珠つなぎ”。攻撃者は新規コードを持ち込まずに武器化できる。
- **CacheWarmer Cookie**: 本拡張が扱うCookie。中身が無検証でデシリアライズされたため攻撃の入口になった。
- **CISA KEV / BOD 22-01**: 実際に悪用中の脆弱性カタログと、連邦機関に期限内是正を義務づける指令。登録は“今攻撃されている”強い証左。

## 影響範囲 / 推奨対応

Mirasvit Full Page Cache Warmer for Magento 2 を導入し 1.11.12 未満で運用中の全ECサイトが対象。未認証・世界規模で悪用が進行中のため即時対応が必要。拡張を 1.11.12 以降へ更新し、更新まではWAFで `CacheWarmer` Cookie のシリアライズ様ペイロードを遮断、または拡張を一時停止する。既に侵害された可能性を前提に、Web shell・不審ファイル・改ざんの調査を行うこと。
