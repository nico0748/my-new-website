# Argo CD repo-server に未修正の認証不要RCE — Kustomizeオプション悪用でKubernetesクラスタ乗っ取り（Synacktiv公開）

> トピック: 脆弱性ニュース ｜ 出典: The Hacker News / CSO Online / Synacktiv ｜ 種別: 認証不要のgRPCエンドポイント悪用によるコマンド実行（RCE）→ Kubernetesクラスタ掌握

## 一行サマリー

Synacktiv が 2026-07-01、Argo CD の repo-server に未修正の攻撃チェーンを公開。**認証不要の GenerateManifest gRPC エンドポイント**に到達できる攻撃者が、マニフェスト生成リクエストで Kustomize の Helm 関連ビルドオプションを悪用し、攻撃者制御のコマンドを実行できる。Argo CD は GitOps の「Tier-0」（制御プレーン）で、リポジトリ読取・クラスタへの同期書込・デプロイ秘密を握るため、成功すれば**Kubernetesクラスタ全体の乗っ取り**に至る。CVE未採番・パッチ未提供で、緩和はネットワークポリシーによる repo-server / Redis の隔離。

## 🔰 初学者向け（何が・なぜ問題か）

近年、多くの企業は「GitOps」という仕組みでシステムを運用しています。ざっくり言うと、**「あるべき状態」をGit（ソースコード置き場）に書いておくと、それを自動でサーバー群（Kubernetes）に反映してくれる**仕組みです。その代表的な“反映エンジン”が **Argo CD** です。

Argo CD の中に「repo-server」という部品があります。これは Git から設定を取ってきて、Kubernetes に配る“レシピ（マニフェスト）”を組み立てる係です。今回の問題は、**この repo-server の一部の受付窓口が、ログイン（認証）なしで話しかけられる状態**になっていたことです。

攻撃者は、その窓口に細工したレシピ生成の依頼を送り込みます。レシピを組み立てる過程で使われる補助ツール（Kustomize／Helm）の設定には「外部コマンドを呼び出す」機能があり、そこを悪用すると**攻撃者の好きなコマンドをサーバー上で実行**できてしまいます。

なぜ致命的か。Argo CD は運用の“中枢神経”だからです。設計上、Argo CD は「非公開リポジトリを読む権限」「本番クラスタに変更を書き込む権限」「デプロイ用の秘密情報（パスワードや鍵）」をまとめて持っています。ここを奪われると、攻撃者は特権的な処理をクラスタに流し込み、管理ノード（master）まで乗っ取って**クラスタ全体を支配**できます。たとえるなら、ビル全体の設備を一括制御する中央制御室のドアが施錠されておらず、入った人が全フロアの空調・電気・鍵を操作できてしまう状態です。

やっかいなのは、**まだ公式の修正（パッチ）が出ていない**こと。Synacktiv は2025年1月に開発元へ報告しましたが約18か月未修正のため、利用者に警告する目的で詳細を公開しました。当面はネットワークで repo-server を隔離するしかありません。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **対象コンポーネント**: Argo CD の **repo-server**（Gitからマニフェストを生成する係）。
- **CVE / パッチ**: 本記事時点で **CVE未採番・公式パッチなし（要確認）**。Synacktiv が 2026-07-01 に技術詳細を公開（責任ある開示後、長期未修正のため公表）。
- **種別**: 認証を要しない内部 gRPC エンドポイント（**GenerateManifest**）の露出＋マニフェスト生成時の外部コマンド実行（Kustomize の Helm 関連ビルドオプション悪用）。分類上は「認証欠如＋危険な機能（コマンド実行）への到達」の複合。
- **前提条件**: 攻撃者が repo-server の gRPC エンドポイント（および関連する Redis）へ**ネットワーク到達可能**であること。既定ではクラスタ内部サービスだが、ネットワークポリシー不備・信頼できない Pod の同居・誤った露出があると悪用可能。
- **到達点（影響）**: repo-server 上でのコマンド実行。Argo CD の保有権限（リポジトリ読取・クラスタ同期書込・秘密情報）を踏み台に、特権ワークロードの master ノード配置などを経て**クラスタ全体の root 相当掌握**へ。
- **悪用状況**: 公開PoC/攻撃ツールの整備状況は流動的（**要確認**）。パッチ不在のため、露出環境は即時の緩和が必要。

### メカニズム（攻撃が成立する仕組み）

概念レベルで分解する（武器化手順・PoCは扱わない）。

1. **信頼境界**: 本来の境界は「Argo CD API/UI の認証済み利用者」と「内部コンポーネント（repo-server, Redis）」の間。repo-server の GenerateManifest gRPC は内部サービス前提で**認証を要求しない**設計になっており、ここへ到達できること自体が境界崩壊の起点。
2. **危険な機能への到達**: マニフェスト生成では Kustomize が使われる。Kustomize の Helm 連携ビルドオプションには、ビルド過程で外部プロセス/コマンドを起動しうる機能がある。攻撃者は生成リクエストに Kustomize オプションを仕込み、この「正規だが危険な機能」を呼び出す。
3. **攻撃者が握るプリミティブ**: 認証なしで到達できる生成エンドポイント＋外部コマンド起動機能の組み合わせにより、攻撃者は『repo-server コンテキストでの任意コマンド実行』プリミティブを得る。ここが RCE の核心。
4. **権限の橋渡し（Tier-0 の増幅）**: repo-server を握った攻撃者は、Argo CD が保持する強力な権限（クラスタへの同期書込・秘密情報）を利用できる。これにより、特権 Pod を master ノードへスケジュールする等の操作を通じ、コンテナ境界を越えてノード・クラスタ制御プレーンへ昇格する。
5. **最終被害**: Kubernetes クラスタ全体の掌握（全 namespace のワークロード操作、秘密情報の窃取、永続化）。GitOps エンジンは source code・構成管理・稼働インフラの結節点にあるため、単一コンポーネントの侵害が“全部の鍵を渡す”ことに直結する。
6. **クラスタとしての位置づけ**: 「内部前提で認証を省いたサービス」＋「テンプレート/ビルド機能に潜む外部コマンド実行」という2クラスの合流。後者は Helm/Kustomize/テンプレートエンジン全般に共通する“設定からのコマンド実行”リスクであり、ネットワーク到達性の管理が唯一の砦になる典型。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: repo-server / Redis への想定外の内部通信（信頼できない Pod や namespace からの接続）、GenerateManifest 呼び出しの異常、repo-server コンテナからの予期しない子プロセス生成・外向き通信、Argo CD 経由での不審な特権ワークロード（master ノード指定・hostPath/権限昇格）作成。
- **暫定緩和（最優先）**: **Kubernetes ネットワークポリシー**で repo-server と Redis への到達を Argo CD の正規コンポーネントのみに限定し、信頼できない Pod からの到達を遮断。Argo CD を専用 namespace/ノードに隔離。Redis の露出排除。Kustomize の Helm ビルド等、危険なビルドオプションの無効化・制限を検討。
- **恒久対策**: 公式修正の提供を注視し、提供され次第適用。それまでは緩和策を継続。GitOps 制御プレーンを **Tier-0 資産**として扱い、最小権限・分離・監査を強化。
- **運用示唆**: 「内部サービスだから認証不要でよい」という前提はネットワーク境界が崩れた瞬間に破綻する。ゼロトラスト的にコンポーネント間認証・到達制御を敷く。

## 📖 用語解説（このニュースとのつながり）

- **Argo CD / GitOps**: Gitに書いた「あるべき状態」を自動でKubernetesへ反映する仕組みとそのエンジン。運用の中枢で、今回の脆弱性の舞台。
- **repo-server**: Argo CD の部品で、Gitから設定を取得しマニフェストを生成する係。今回の脆弱性コンポーネント。
- **GenerateManifest（gRPC エンドポイント）**: マニフェスト生成を担う内部API。認証なしで到達できる点が問題の起点。
- **Kustomize / Helm**: Kubernetesマニフェストを組み立てるツール。ビルドオプションに外部コマンド起動機能があり、悪用されるとRCEに化ける。
- **Tier-0（制御プレーン）**: 侵害されると全体を制御できる最上位資産。GitOpsエンジンはこの階層に属するため被害が甚大。
- **ネットワークポリシー**: Pod間通信を制御するKubernetesの仕組み。パッチ不在の今、repo-server隔離のための主要な緩和策。
- **認証欠如（内部サービスの露出）**: 内部前提で認証を省いた窓口が到達可能になると即座に悪用される。ゼロトラストが必要な理由。

## 影響範囲 / 推奨対応

Argo CD を用いて Kubernetes を運用する組織全般が対象。特に repo-server / Redis がネットワーク的に十分隔離されていない環境、信頼できないワークロードが同居するクラスタは高リスク。パッチ未提供のため、(1) ネットワークポリシーで repo-server・Redis への到達を正規コンポーネントに限定、(2) Argo CD の namespace/ノード分離、(3) Kustomize の危険なビルドオプション制限、(4) 特権ワークロード作成や repo-server の異常子プロセスの監視、を直ちに実施する。GitOps 制御プレーンを Tier-0 として扱い、公式修正の提供を注視して速やかに適用する。
