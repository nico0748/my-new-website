# MCP 2026-07-28 リリース候補：ステートレス・コア＋MCP Apps＋Tasks 拡張で「MCP 2.0」へ

> トピック: Model Context Protocol 2026-07-28｜出典: MCP 公式ブログ｜種別: プロトコル仕様更新

## 一行サマリー

Model Context Protocol の 2026-07-28 リリース候補が公開され、ローンチ以来最大の改訂となった。通常の HTTP インフラでスケールする「ステートレス・コア」、サーバレンダリング UI の「MCP Apps」、長時間処理の「Tasks 拡張」、OAuth/OpenID に整合させた認可強化、そして正式な非推奨ポリシーを盛り込む。最終仕様は 2026年7月28日公開予定。

## 🔰 初学者向け（何が・なぜ重要か）

MCP は「AI エージェントと外部ツール（データベース・API・社内システム）をつなぐ共通プラグ規格」だ。USB のように、一度対応すればどんな AI クライアントからも同じ形でツールを呼べる。2024〜2025年で急速に普及し、いまや Claude/OpenAI/Google などが対応する事実上の標準になった。

普及すると、初期設計では想定しなかった課題が噴き出す。今回の 2026-07-28 改訂は「MCP 2.0」と呼ばれるほど大きく、その課題への回答が詰まっている。最大の変更は「ステートレス化」。従来の MCP はサーバがセッション状態を持つ前提だったが、これだとサーバを増やして負荷分散する（水平スケール）のが難しい。ステートレスにすれば、ごく普通の Web サーバのように何台でも並べて捌ける。

もう1つ大きいのが「MCP Apps」と「Tasks」。前者はツールが単なるテキストだけでなく UI（画面）を返せるようにするもの、後者は「1時間かかる処理」のような長時間ジョブを扱えるようにするものだ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **ステートレス・コア**: 通常の HTTP インフラ上でスケールする。従来のセッション保持前提を外し、水平スケールとロードバランスを容易にする。
- **MCP Apps**: サーバレンダリングの UI を返せる拡張。ツール結果を画面として提示できる。
- **Tasks 拡張**: 長時間実行の処理を扱う拡張。即時応答しないジョブを表現できる。
- **認可強化（6つの SEP）**: OAuth 2.0 / OpenID Connect の実運用に近づける。
  - **iss 検証 (SEP-2468)**: 認可レスポンスの `iss` パラメータを RFC 9207 に従い検証。MCP に多い「1クライアント×多サーバ」構成で起きやすい mix-up 攻撃への低コスト対策。
  - **application_type 宣言 (SEP-837)**: 動的クライアント登録時に OpenID Connect の `application_type` を宣言。デスクトップ/CLI クライアントが "web" と誤判定され localhost リダイレクトを拒否される問題を回避。
  - **クレデンシャルバインディング (SEP-2352)**: 登録クレデンシャルを発行元認可サーバの issuer に束縛し、リソース移行時に再登録。
- **正式な非推奨ポリシー**: プロトコルが既存実装を壊さず進化できるよう、廃止の手続きを明文化。
- **タイムライン**: RC 公開済み、最終仕様は 2026年7月28日公開予定。

### 仕組み / 背景（なぜこう設計されたか）

ステートレス化の狙いは、MCP を「特別なサーバ」から「普通の HTTP サービス」に近づけることにある。セッション状態をサーバに持たせると、リクエストが必ず同じサーバに来るよう固定（スティッキーセッション）する必要があり、これが水平スケールの足かせになる。ステートレスにすればロードバランサが任意のサーバへ振れ、オートスケールもクラッシュ耐性も普通の Web と同じ道具で扱える。急増する MCP サーバの本番運用負荷に応える設計だ。

認可強化は「MCP 特有の攻撃面」への対応が面白い。一般的な OAuth は「1つの Web アプリ×1つの認可サーバ」を想定するが、MCP は「1つのエージェント（クライアント）が多数の MCP サーバに接続する」パターンが標準だ。この構成では、悪意ある1サーバが別サーバの認可レスポンスにすり替わる「mix-up 攻撃」が起きやすい。RFC 9207 の `iss` 検証は、レスポンスがどの認可サーバから来たかを必ず確認することでこれを塞ぐ。MCP の使われ方に即したセキュリティ設計と言える。

MCP Apps と Tasks は「ツールはテキストを同期で返す」という初期の前提を緩める拡張だ。実務では「グラフを描いて見せたい」「バッチ処理を投げて後で結果を取りに来たい」といった要求が多く、それを標準の枠組みで表現できるようにした。

### 実務での使いどころ・移行の勘所

自作 MCP サーバを本番運用しているなら、ステートレス・コアへの移行で運用コストが下がる可能性が高い。ただしステートフル前提のサーバは設計変更が要る。認可を実装しているサーバは 6 SEP（特に `iss` 検証と `application_type` 宣言）への追従が実質必須で、これらは既存実装の破壊的変更を含みうるため RC 段階で検証すべき。MCP Apps / Tasks は拡張なので必要なサーバだけ実装すればよい。正式仕様は 7月28日なので、それまでに RC で自サーバの互換性を確認しておくのが安全。

## 📖 用語解説（このトピックとのつながり）

- **MCP (Model Context Protocol)**: AI エージェントと外部ツールをつなぐ共通規格。2026-07-28 が最大の改訂。
- **ステートレス**: サーバがセッション状態を持たない設計。水平スケール・負荷分散を容易にする。
- **SEP**: MCP の仕様変更提案（Spec Enhancement Proposal）。今回6件が認可強化に充てられた。
- **mix-up 攻撃**: 悪意あるサーバが別サーバの認可レスポンスにすり替わる攻撃。多サーバ構成で顕在化しやすい。
- **RFC 9207 (iss 検証)**: 認可レスポンスの発行元を検証する仕様。mix-up 対策の要。
- **MCP Apps / Tasks**: サーバレンダリング UI と長時間処理を扱う拡張。同期テキスト前提の緩和。

## 影響範囲 / 推奨アクション

- 自作 MCP サーバ運用者は RC 段階でステートレス・コアと認可強化への互換性を検証する。
- 認可実装があるサーバは `iss` 検証・`application_type` 宣言（SEP-2468 / SEP-837）への追従を計画。破壊的変更に注意。
- UI 提示や長時間ジョブが必要なら MCP Apps / Tasks 拡張の採用を検討（拡張なので任意）。
- 最終仕様は 2026年7月28日公開。それまでに移行影響を洗い出しておく。
