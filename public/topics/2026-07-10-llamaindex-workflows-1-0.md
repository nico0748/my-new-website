# LlamaIndex Workflows 1.0 — イベント駆動でエージェントの制御フローを設計する軽量フレームワーク

> トピック: LlamaIndex Workflows 1.0（2026年6月22日）｜出典: LlamaIndex 公式ブログ / GitHub｜種別: エージェント フレームワーク メジャーリリース

## 一行サマリー

LlamaIndex が、エージェント的システムの制御フローをイベント駆動・非同期・ステップ単位で組み立てる軽量フレームワーク「Workflows 1.0」を 2026年6月22日にリリースした。従来 `llama_index` 本体に同梱されていた機能を独立パッケージ（Python: `llama-index-workflows` / TypeScript: `@llamaindex/workflow-core`）として切り出し、RAG に限らない汎用のエージェント基盤を目指す。

## 🔰 初学者向け（何が・なぜ重要か）

LLM（大規模言語モデル）を使ったアプリは、「AI に一回質問して答えをもらう」だけでは済まないことが多い。実際には「まず検索して → 結果を見て次に何をするか決めて → ツールを呼んで → 足りなければもう一度検索」といった、複数ステップの流れ（ワークフロー）を組む必要がある。この「AI にどういう順序で仕事をさせるか」を設計するのがエージェント開発の肝だ。

Workflows 1.0 は、この流れを「イベントが起きたら次のステップが動く」という考え方（イベント駆動）で書けるようにする道具。各ステップは独立した部品で、「このイベントを受け取ったら、この処理をして、次のイベントを出す」と定義する。分岐・並列・ループ・状態の持ち回りが素直に書けるので、複雑なエージェントの挙動を「魔法の全自動」に丸投げせず、開発者が見通しよくコントロールできる。RAG（検索して答える仕組み）専用ではなく、汎用の「AI の段取り記述ツール」になったのがポイント。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

LlamaIndex 公式によれば、Workflows 1.0 は Workflows の初の独立（standalone）版で、独自リポジトリ・パッケージ・開発ラインを持つ。従来は `llama_index` / LlamaIndexTS の一部だったが、Python は `pip install llama-index-workflows`、TypeScript は `npm i @llamaindex/workflow-core` で単体導入できる。設計は async-first・イベント駆動で、ステップ（step）がイベントを購読し新たなイベントを発行することで、ルーティング、並列処理、ループ、複数ステップ横断の状態管理を表現する。RAG グラウンディングを伴うエージェントや、文書取り込み→解析→要約→ルーティングといった文書処理パイプラインが代表的ユースケース。発表は LlamaParse の「エージェント的文書処理」への進化などと同時期で、LlamaIndex のエージェント/文書スタック全体の再編の一環。

### 仕組み / 背景（なぜこう設計されたか）

エージェント設計には大きく2系統ある。ひとつは「グラフ（ノードとエッジ）で状態遷移を明示する」型（例：LangGraph）、もうひとつは「イベントの発行/購読で疎結合にステップを連結する」型。Workflows は後者を採り、各ステップを「入力イベント→出力イベント」の純粋な変換単位として扱うことで、フローの一部を差し替え・再利用しやすくする。イベント駆動＋async-first にした狙いは、外部 API 待ちや並列検索といった I/O 律速の処理を効率よくさばき、長時間・多段のエージェントでもスループットを保つこと。トレードオフは、明示的なグラフ図がない分、全体の制御フローがイベントの連鎖として分散し、大規模化すると「どのイベントが何を起動するか」の把握に可視化・トレースが要ること。LlamaIndex はこれを可観測性ツールと組み合わせる前提で設計している。

### 実務での使いどころ・移行の勘所

RAG を核にした業務エージェント、文書取り込み・分類・要約のパイプライン、複数ツールを条件分岐で使い分けるアシスタントに向く。既存 LlamaIndex ユーザーは、モノリシックな `llama_index` 依存から Workflows を単体パッケージとして切り出せるため、依存の軽量化とバージョン管理の独立が利点。移行の勘所は、独立版の API に合わせてインポート/依存を更新すること、Python/TypeScript 両対応のため言語横断でフロー設計を共有できる一方、可観測性（イベントのトレース）を早期に組み込んで大規模化に備えること。フレームワーク選定としては、状態遷移をグラフで厳密管理したいなら LangGraph、疎結合なイベント連結と RAG 親和性を重視するなら Workflows、という住み分けで比較するとよい。

## 📖 用語解説（このトピックとのつながり）

- **イベント駆動アーキテクチャ**: イベントの発行/購読で処理を連結する方式。Workflows のステップ間連携の基盤。
- **RAG（検索拡張生成）**: 外部知識を検索して LLM の回答を根拠づける手法。Workflows の主要ユースケース。
- **async-first**: 非同期処理を前提にした設計。I/O 待ちの多いエージェントでスループットを稼ぐ。
- **グラフ型 vs イベント型オーケストレーション**: 状態遷移を明示するか、疎結合にイベント連結するかの設計思想の違い。

## 影響範囲 / 推奨アクション

- エージェント/RAG 基盤の選択肢が拡充。LlamaIndex ユーザーは依存軽量化と汎用フロー記述の観点で移行を検討。
- 大規模化を見据え、イベントのトレース・可観測性を初期から組み込む。
- LangGraph 等のグラフ型と、要件（厳密な状態管理か疎結合な連結か）で比較評価する。
