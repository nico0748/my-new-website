# Apache Tomcat の認証バイパス（CVE-2026-55957）— JNDIRealm×GSSAPIでパスワード不要ログイン

> トピック：Java Webサーバの認証すり抜け ｜ 出典：HeroDevs / Apache Tomcat Security / Oligo ｜ 種別：認証バイパス（CWE-287 系）

## 一行サマリー

Javaの定番Webサーバ Apache Tomcat が2026年6月29日に計7件のCVEを公表し、うち CVE-2026-55957 は JNDIRealm を GSSAPI で使う構成において「正しいパスワードなしでも認証を通せる」認証バイパス（Important）で、9.0.101 / 10.1.37 / 11.0.5 以降で修正済み。

## 🔰 初学者向け（何が・なぜ問題か）

Apache Tomcat（トムキャット）は、Javaで作られたWebアプリを動かすための土台となるサーバソフトで、日本でも社内システム・業務Webアプリの多くがこの上で動いています。とても“縁の下の力持ち”的な存在です。

今回の中心的な問題（CVE-2026-55957）を例えると、建物の入館ゲートに「社員証をかざせば開く」仕組み（＝GSSAPIを使ったログイン認証）があるのに、ある設定で使うと、ゲートが「本当に正しい社員証かどうか」を最後まで確かめずに開いてしまう、というものです。つまり、正しいパスワードを持っていない人でも“ログインできてしまう”状態です。

認証は多くのシステムで「最初の関門」です。ここを突破されると、その先の機能や管理画面に正規ユーザーとしてアクセスされ、データ閲覧・改ざん・場合によっては更なる侵害の起点になります。今回はさらに、静的リソースのアクセス制御が効かなくなる別の欠陥（CVE-2026-55956）も同時公表されました。実際の悪用報告は確認できていませんが（要確認）、利用範囲が広いためパッチ適用の優先度は高いです。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **CVE**：CVE-2026-55957（認証バイパス、Important）／ CVE-2026-55956（既定サーブレットのセキュリティ制約バイパス、Moderate）。同日公表の計7件のうちの中核2件。残り5件は Low。
- **CWE**：CWE-287（不適切な認証）系。CVE-2026-55956 は CWE-863/CWE-284（不適切な認可・アクセス制御）系。
- **前提条件**：CVE-2026-55957 は **JNDIRealm を GSSAPI でのバインド認証に構成している**環境で成立。この構成でない場合は影響を受けない（＝設定依存の脆弱性）。CVE-2026-55956 は既定サーブレットに対しHTTPメソッド指定を伴うセキュリティ制約を設定している環境が対象。
- **影響/修正版**：9.0.101 / 10.1.37 / 11.0.5 以降で修正。9.0系利用者は 9.0.119 へ上げれば同日公表の7件をまとめて解消可能（9.0.101の修正が以降のリリースに取り込まれているため）。
- **悪用状況**：現時点で実環境悪用の公開情報は確認できず（**要確認**）。ただし利用母数が大きく、認証境界を破る性質上、優先対応が妥当。

### メカニズム（攻撃が成立する仕組み）

**CVE-2026-55957（認証バイパス）** — 認証ロジックの検証欠落という脆弱性クラスの典型です（概念レベル。武器化手順やPoCは記載しません）。

1. **認証の責務**：JNDIRealm は、ユーザーの資格情報をLDAP等のディレクトリに照合してログイン可否を判定するTomcatのコンポーネント。GSSAPI（Kerberos等の汎用認証枠組み）を使うバインド認証では、「提示された資格情報でディレクトリに正しくバインド（認証接続）できたか」を成功条件にすべき。
2. **検証条件の取り違え**：この構成で、**バインドが本当に正しいパスワードで成立したか**の確認が不十分／論理が誤っていた。結果として、正しい資格情報を伴わないバインド（匿名バインドや不完全な認証状態）でも「認証成功」と誤判定される余地が生まれる。
3. **プリミティブ→被害**：攻撃者は「パスワードを知らないユーザーとしてログイン成功状態を得る」プリミティブを握る。これにより本来保護された機能・画面へ正規セッションでアクセスでき、その先の権限・データ露出、他脆弱性との連鎖侵害の起点になる。
4. **クラス一般原理**：認証バイパスは、(a) 成功/失敗の判定条件が実際の検証結果と乖離している、(b) “認証済み”状態の確立をアプリが過信する、といった認証ロジックの欠陥から生じる。特に外部認証枠組み（GSSAPI/LDAP/SSO）との接続点は、成功条件の取り扱いを誤りやすい。

**CVE-2026-55956（アクセス制御バイパス）** — 既定サーブレットに対して「特定HTTPメソッドに適用する」セキュリティ制約を設定していても、そのメソッド指定（またはメソッド省略）が無視され、書いたアクセス制御が意図どおり強制されない。設定した保護ルールが“素通り”するアクセス制御欠落型。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**：JNDIRealm/GSSAPI認証の成功ログで、資格情報が伴わない・匿名バインド由来と思われる認証成功の痕跡。想定外アカウントでのログイン成功。静的リソースに対し、制約したはずのメソッド（PUT/DELETE等）が通っていないかのアクセスログ確認。
- **暫定緩和**：パッチ前は、JNDIRealm+GSSAPIバインド構成の見直し（一時的に別認証方式へ切替、または管理面のネットワーク制限）。既定サーブレットに依存したアクセス制御を、より上位（フィルタ/認可レイヤ）で二重化。
- **恒久対策**：9.0.119 / 10.1.37 / 11.0.5 以降へアップデート（バージョン系列に応じた最新へ）。設定依存脆弱性のため、自環境が該当構成か棚卸ししたうえで適用可否と優先度を判断。
- **運用示唆**：Tomcatは“土台”ゆえ更新が後回しになりやすい。同日7件のまとめパッチである点を活かし、一括更新で技術的負債を解消するのが効率的。

## 📖 用語解説（このニュースとのつながり）

- **Apache Tomcat**：Java Webアプリを動かす定番サーバ。社内業務システムの土台に多く、今回の認証欠陥は広範囲に波及し得る。
- **JNDIRealm**：Tomcatがユーザー認証をLDAP等のディレクトリに委ねる仕組み。本件はこのコンポーネントのGSSAPI構成での欠陥。
- **GSSAPI**：Kerberosなどを抽象化した汎用認証インタフェース。外部認証との接続点で、成功条件の取り扱いを誤ると認証バイパスにつながる。
- **認証バイパス**：正しい資格情報なしにログインを成立させる脆弱性。最初の関門を破られるため影響が大きい。
- **既定サーブレット／セキュリティ制約**：静的ファイル配信を担う標準機能と、そのアクセス制御ルール。CVE-2026-55956では制約が意図どおり効かない。

## 影響範囲 / 推奨対応

Tomcatを利用する組織は、まず **JNDIRealm＋GSSAPIバインド構成の有無**を確認。該当する場合はCVE-2026-55957の影響大として最優先で更新。非該当環境でも、同日公表の他6件を含めまとめて 9.0.119 / 10.1.37 / 11.0.5 以降へ更新するのが効率的。既定サーブレットのアクセス制御に頼っている場合はCVE-2026-55956の影響を確認する。
