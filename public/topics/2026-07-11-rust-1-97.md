# Rust 1.97.0 リリース — v0 シンボルマングリング既定化・Cargo の警告制御・リンカ出力の可視化

> トピック: Rust 1.97.0（2026-07-09）｜出典: Rust Blog・releases.rs・LWN｜種別: 言語 / ツールチェーン

## 一行サマリー

2026 年 7 月 9 日にリリースされた Rust 1.97.0 は、新しい v0 シンボルマングリング方式を安定版で既定化し、Cargo が警告を「許可 / 表示 / 拒否」で制御できるようになり、これまで隠されていたリンカ出力を既定で表示するようになった。整数プリミティブに新 API が加わり、`pin!()` マクロの暗黙 deref 強制が廃止される破壊的変更も含む。

## 🔰 初学者向け（何が・なぜ重要か）

Rust は「速くて安全な」システムプログラミング言語。今回の 1.97 は派手な新文法よりも、**開発時の体験を細かく整える**アップデートが中心。

たとえば「シンボルマングリング」は、コンパイラが関数名などを内部的に符号化する仕組み。デバッガやプロファイラがその符号を人間に読める名前へ戻す（デマングル）が、古い方式は情報が欠けやすかった。1.97 は新方式 v0 を標準にして、より正確に名前を復元できるようにした（ただし古いツールは一時的にうまく戻せないことがある）。

「Cargo の警告制御」は、ビルド時の警告を「無視する / 表示するがビルドは通す / 警告があれば失敗させる」と切り替えられる機能。CI では厳しく、手元では緩く、といった使い分けがしやすくなる。地味だが日々の開発が快適になる改善だ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **v0 シンボルマングリング既定化**: 2025 年 11 月から nightly で既定だった v0 方式を安定版で有効化。古いデバッガ/プロファイラではデマングルに失敗したり、バックトレースの整形が変わる場合がある。
- **Cargo の警告制御**: 警告を `allow`（抑制）/ `warn`（表示するが失敗させない・既定）/ `deny`（拒否＝失敗）で制御。Cargo 設定側で挙動が決まるため、この機能を使ってもビルドキャッシュは無効化されず、一時的な opt-in が容易。
- **リンカ出力の可視化**: これまで rustc はリンク成功時にリンカ出力を既定で握りつぶしていたが、1.97 では既定で表示。
- **新しい整数 API**: `u8`〜`u128` および `usize` に `isolate_highest_one` / `isolate_lowest_one` / `highest_one` / `lowest_one` / `bit_width` を追加。
- **ARM v8.6-A 機能**: `div32`・`lamcas` 等を nightly 無しで利用可能に。
- **破壊的変更**: `pin!()` マクロが deref 強制を許可しなくなり、`pin!(x)`（`x: &mut T`）は常に `Pin<&mut &mut T>` を生成する（従来は時に `Pin<&mut T>` へ強制されていた）。

### 仕組み / 背景（なぜこう設計されたか）

シンボルマングリングは「同じ名前でも型パラメータやモジュールが違えばリンク時に区別する」ために必要な符号化だ。旧来の "legacy" 方式はジェネリクスや `impl` の情報表現が曖昧で、デバッグ時に名前が衝突・欠落しやすかった。v0 は Rust の型システムをより正確に反映する設計で、デバッガやプロファイラでの識別性が上がる。代償として、v0 を理解しない古いツールは一時的にデマングルできず、移行期の摩擦が生じる——そのため長い nightly 期間を経てから安定化された。

Cargo の警告制御は「ビルドの成否」と「警告の扱い」を分離する思想。従来は `#![deny(warnings)]` などコード側に書くと、環境をまたいで硬直しがちだった。設定ファイル側で `allow/warn/deny` を切り替え、しかもキャッシュを壊さない設計にしたことで、「CI では deny、ローカルでは warn」を軽量に両立できる。リンカ出力の既定表示も同じ方向性で、「暗黙に隠す」より「見えるようにして開発者が判断する」透明性重視への転換だ。

### 実務での使いどころ・移行の勘所

日常のビルド/デバッグ体験の底上げが主眼。移行の勘所は、(1) デバッガ/プロファイラを最新化しておく（v0 デマングル対応のため）、(2) CI の警告方針を Cargo 設定の `deny` に寄せ、ローカルは `warn` にするなど環境別に整理、(3) `pin!()` を使うコードは型が `Pin<&mut &mut T>` になる変更に注意し、必要なら明示的に借用を調整、(4) ビット演算の多いコードは新整数 API で可読性・意図明確化を図る。総じて安全に上げられるが、周辺ツールのバージョンだけ先に揃えるとよい。

## 📖 用語解説（このトピックとのつながり）

- **シンボルマングリング**: 関数名等をリンク用に符号化する仕組み。1.97 で新方式 v0 が既定。
- **デマングル**: 符号化された名前を人間可読へ戻すこと。デバッガ/プロファイラが行う。
- **Cargo**: Rust のビルド/依存管理ツール。警告の扱いを allow/warn/deny で制御可能に。
- **pin!() / Pin**: 値を「移動させない」ことを型で保証する仕組み。1.97 で deref 強制が廃止。
- **ビルドキャッシュ**: 再ビルドを省く仕組み。警告制御はキャッシュを無効化しない設計。

## 影響範囲 / 推奨アクション

- デバッグ/プロファイル体験が向上。v0 対応のためデバッガ/プロファイラを更新しておく。
- CI と手元で警告方針を allow/warn/deny により分離すると運用が楽になる。
- `pin!()` 利用箇所は型変更（`Pin<&mut &mut T>`）に注意して移行。
