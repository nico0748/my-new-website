# Transformers.js — サーバー不要でブラウザ内 ML を動かす

> トピック: 開発トピック｜出典: startbit（blog）｜種別: frontend

## 一行サマリー
Hugging Face の Transformers.js を使うと、ONNX Runtime（および WebGPU/Wasm）経由で機械学習モデルをブラウザ内で直接実行できる。サーバーへデータを送らずに推論できるため、プライバシーと低レイテンシを両立する。

## 🔰 初学者向け（何が・なぜ重要か）
AI 機能というと「入力をサーバーに送り、AI が返事をくれる」という形が普通でした。しかしこの方式は、通信が発生し、個人情報がサーバーへ渡り、サーバー費用もかかります。

Transformers.js は「AI モデルそのものをブラウザにダウンロードして、その場で動かす」ライブラリです。たとえるなら、辞書アプリを毎回オンラインで引くのではなく、辞書データを端末に落として手元で引くようなもの。文章の感情分析、翻訳、画像分類、音声認識といった処理が、ネットに投げずにユーザーの端末内で完結します。データが外に出ないのでプライバシーに優しく、通信待ちがないので反応も速い。前述の WebAssembly や WebGPU の進化が、この「ブラウザ内 AI」を実用レベルに押し上げました。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Transformers.js は Python 版 Transformers の JS 移植で、モデルを ONNX 形式に変換して onnxruntime-web 上で実行する。バックエンドとして WebAssembly（CPU）と WebGPU（GPU アクセラレーション）を選択でき、対応環境では GPU により推論が大幅に高速化する。テキスト分類・埋め込み生成・翻訳・音声/画像処理など幅広いタスクに対応し、`pipeline()` API で数行から使える。モデルは初回にフェッチしキャッシュされる。

### 仕組み / 背景（なぜこう設計されたか）
サーバー推論はスケールコストとプライバシー・レイテンシの課題を抱える。エッジ（端末）で完結させれば、送信データゼロ・オフライン動作・サーバー費用削減が得られる。これを支えるのが Wasm/WebGPU の性能向上と、量子化などによるモデル軽量化。トレードオフは、モデルのダウンロードサイズ・初回ロード時間・端末性能への依存で、巨大モデルには不向き。

### 実務での使いどころ・移行の勘所
機微データを扱う入力補助（要約・分類・翻訳）、オフライン対応が欲しい PWA、リアルタイム性が要る UI 補助に好適。大規模生成モデルはサイズ的に厳しいため、小〜中規模の特化タスク向けモデルを選ぶ。WebGPU 非対応端末では Wasm にフォールバックする設計と、初回ロード UX（プログレス表示）を用意する。

## 📖 用語解説（このトピックとのつながり）
- **ONNX Runtime**: モデルを標準形式で高速実行する推論エンジン。Transformers.js の実行基盤。
- **WebGPU**: ブラウザから GPU を使う新 API。ブラウザ内推論を高速化する鍵。
- **pipeline API**: タスク指定だけで推論を呼べる Transformers.js の高水準インターフェース。
- **量子化**: モデルを軽量化し端末で動かしやすくする手法。ロードサイズ削減に寄与。
- **エッジ推論**: サーバーでなく端末側で AI を動かす方式。プライバシー・低遅延の源泉。

## 影響範囲 / 推奨アクション
プライバシー要件が強い、あるいはオフライン/低遅延が欲しいフロント機能を作るチームが対象。まず小さな特化モデルで PoC し、対象端末での推論速度とロードサイズを実測。WebGPU/Wasm のフォールバック戦略とロード中 UX を必ず設計する。
