# TypeScript の Go 移植（tsgo）— 型チェックを最大10倍高速化する次世代コンパイラ

> トピック: 開発トピック｜出典: Dev Newsletter｜種別: frontend

## 一行サマリー
TypeScript コンパイラを Go 言語へ全面移植した次世代版（通称 tsgo）は、型チェックとエディタ応答を従来比で最大 10 倍高速化することを目標に開発されており、巨大リポジトリでの「保存のたびに待たされる」問題を根本から改善しようとしている（倍率は目標値で要確認）。

## 🔰 初学者向け（何が・なぜ重要か）
TypeScript の型チェックは、コードが増えるほど重くなります。大規模プロジェクトでは、ファイルを保存してからエディタが赤線を出すまで、あるいは CI で型チェックが終わるまでに何秒も、時には数十秒もかかることがあります。この「待ち時間」は集中を切らし、開発の体感速度を大きく下げます。

原因の一つは、コンパイラ本体が JavaScript で書かれていて、基本的に一度に一つの処理しか進められない（並行実行が苦手）ことでした。そこで開発チームは、複数の CPU コアを同時に使える Go 言語へコンパイラを丸ごと書き直しました。

たとえるなら、これまで一人のレジ係が長蛇の列をさばいていたのを、複数レジを同時に開けるようにしたようなものです。同じ型チェックという仕事を、複数のコアで手分けして一気に片付ける。結果として、大規模コードでもエディタがきびきび反応し、CI の待ち時間も縮む——これが tsgo が目指す世界です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
tsgo は TypeScript コンパイラ（型チェッカー・言語サービス含む）を Go で再実装したもの。ネイティブバイナリとして動作し、マルチコア並列化とネイティブ実行により型チェック・ビルド・エディタ補完のレイテンシを大幅に削減する。掲げられる「最大 10 倍」は代表的ベンチでの目標値で、実測は規模・構成で変動するため要確認。言語仕様・型システムの互換は維持しつつ、実装言語だけを差し替える方針。

### 仕組み / 背景（なぜこう設計されたか）
JavaScript ランタイムはシングルスレッド前提で、GC 停止や JIT ウォームアップのコストも型チェックのような CPU 集約タスクには不利だった。Go は goroutine による軽量並行、AOT ネイティブコンパイル、予測可能なメモリ挙動を持ち、コンパイラ実装に適する。C++ や Rust ではなく Go を選んだ背景には、既存 TypeScript コードベースからの移植容易性（構造的な近さ）と、Microsoft 内の知見・保守性のバランスがあるとされる。

### 実務での使いどころ・移行の勘所
最大の恩恵は「大規模モノレポの型チェック時間短縮」と「エディタ補完の即応性向上」。導入時は、既存の `tsc` を前提にしたビルドパイプラインやプラグイン（型情報を内部 API 経由で読むツール）が tsgo でそのまま動くかの検証が要点。初期版は稀な型推論の挙動差やエラーメッセージ差が出る可能性があるため、CI に旧 `tsc` を残して差分監視しながら段階移行するのが安全。DX 改善効果が大きいため、ビルド時間に悩むチームは早期評価の価値が高い。

## 📖 用語解説（このトピックとのつながり）
- **tsgo**: Go で書き直された次世代 TypeScript コンパイラの通称。本記事の主題。
- **言語サービス（Language Service）**: エディタの補完・エラー表示を担う部分。高速化の主な恩恵先。
- **goroutine**: Go の軽量並行処理単位。マルチコア活用の中核。
- **モノレポ**: 多数のパッケージを一つのリポジトリに集約する構成。型チェック負荷が大きく恩恵が顕著。

## 影響範囲 / 推奨アクション
型チェック時間に悩む大規模プロジェクトほど恩恵が大きい。まずは開発環境で tsgo を試し、`tsc` を CI に残して挙動差を監視。ビルドツール・型生成系ライブラリの対応状況を確認してから段階移行することを推奨。
