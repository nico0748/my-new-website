# Java 25 LTS — 仮想スレッドの落とし穴を解消、Project Loom が実務段階へ

> トピック: 開発トピック｜出典: Java Code Geeks（要一次確認）｜種別: backend

## 一行サマリー
Java 25 LTS（2025-09）は、仮想スレッドが `synchronized` ブロックで OS スレッドに固定（pinning）されてしまう長年の落とし穴を解消し（JEP 491）、Project Loom の仮想スレッドがようやく安心して本番採用できる段階に到達した。続く Java 26（2026-03）では構造化並行性のプレビューも進む。

## 🔰 初学者向け（何が・なぜ重要か）
サーバーは大量のリクエストを同時にさばきます。従来の Java では、1リクエストに1つの「OS スレッド」を割り当てていましたが、OS スレッドは重く、数千〜数万を同時に立てるとメモリを食い尽くしてしまいます。そこでスレッドを使い回す複雑な仕組み（スレッドプール）が必要でした。

仮想スレッド（Project Loom）は、この重い OS スレッドの上に「軽いスレッド」をたくさん載せる仕組みです。待ち時間（DB や API の応答待ち）の間は OS スレッドを手放し、他の仕事に回すので、少ない OS スレッドで膨大な同時処理をこなせます。「普通のスレッドを書くだけで、勝手に効率よくさばいてくれる」のが理想でした。

ところが初期には落とし穴があり、`synchronized`（排他制御）を使うと軽いスレッドが重いスレッドに貼り付いてしまい（pinning）、効率が台無しになる場面がありました。Java 25 でこの問題が直り、仮想スレッドを「安心して本番で使える」段階になった、というのが今回の要点です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Java 25 は 2025 年 9 月の LTS リリース。JEP 491 により、`synchronized` ブロック内でブロッキング処理を行っても仮想スレッドが carrier（OS）スレッドに pinning されない仕組みが入り（JDK 24 で導入、Java 25 LTS に取り込み）、Project Loom の仮想スレッドが実務レベルで安定した。続く Java 26（2026-03）では構造化並行性（structured concurrency）のプレビューが進む。LTS のため長期サポートが付き、企業採用の基準版になる（各 JEP の最終状態・サポート期間は OpenJDK/ベンダの一次情報で要確認）。

### 仕組み / 背景（なぜこう設計されたか）
仮想スレッドは JVM がスケジュールする軽量スレッドで、ブロッキング I/O 時に carrier スレッドを解放して他の仮想スレッドを走らせる（マウント/アンマウント）。これにより「スレッド＝希少資源」という前提が崩れ、同期的で読みやすいコードのまま高い同時実行性が得られる。初期の pinning 問題は、`synchronized` のモニタが carrier スレッドに紐づく実装に起因し、ブロック中にアンマウントできず OS スレッドを占有してしまうものだった。JEP 491 はモニタ実装を見直してアンマウント可能にした。トレードオフとして、CPU バウンドな処理は仮想スレッドでは速くならない（Loom は I/O 待ちの効率化が主眼）点、`ThreadLocal` の大量使用がメモリ圧になりうる点に注意。構造化並行性は「並行タスクの生存期間を構文で束ねる」ことでリークやキャンセル漏れを防ぐ設計思想だ。

### 実務での使いどころ・移行の勘所
I/O 待ちが支配的な Web サービス（外部 API 呼び出し、DB アクセス多用）で効果が大きい。移行では (1) スレッドプールを仮想スレッドの `newVirtualThreadPerTaskExecutor` に置換、(2) `synchronized` の多用箇所を `ReentrantLock` に見直す（Java 25 で pinning は解消したが、ロック粒度の見直しは有益）、(3) `ThreadLocal` 依存を減らす、が勘所。Spring Boot など主要フレームワークは仮想スレッド対応が進んでおり、設定で有効化できることが多い。CPU バウンド処理には従来のプールや並列ストリームを併用する。

## 📖 用語解説（このトピックとのつながり）
- **仮想スレッド（Project Loom）**: JVM が管理する軽量スレッド。少ない OS スレッドで大量の同時処理を実現。
- **pinning（固定）**: 仮想スレッドが OS スレッドに貼り付き解放できない状態。JEP 491 で `synchronized` 起因の pinning を解消。
- **carrier スレッド**: 仮想スレッドを実際に走らせる OS スレッド。I/O 待ちで解放され再利用される。
- **構造化並行性（structured concurrency）**: 並行タスクの生存期間を構文で束ね、キャンセル/エラー伝播を整理する仕組み。Java 26 でプレビュー進行。
- **LTS（長期サポート）**: 長期の保守が保証される版。Java 25 が該当し企業採用の基準になる。

## 影響範囲 / 推奨アクション
I/O 集約型の Java バックエンドに大きく関係。Java 25 LTS への更新で仮想スレッドを本番採用しやすくなる。スレッドプールの置換とロック実装の見直しを段階的に進め、CPU バウンド処理は別手段を併用。LTS ゆえ長期運用の基準版として移行計画に組み込むとよい。
