# BerriAI LiteLLM CVE-2026-42271 — AIゲートウェイのテスト用APIからOSコマンド実行、認証回避と連鎖で未認証RCEへ

> トピック: 脆弱性ニュース ｜ 出典: The Hacker News / Help Net Security / CISA KEV ｜ 種別: コマンドインジェクション（CWE-78）

## 一行サマリー

社内のAIモデルを束ねる「LiteLLM」ゲートウェイに、テスト用APIへ送るリクエストの中身がそのままOSコマンドとして実行される欠陥があり、別の認証回避バグ（CVE-2026-48710）と連鎖すると認証なしのリモートコード実行（RCE）に至る。CISAが悪用を確認しKEVへ登録済みで、早急なアップデートが必要。

## 🔰 初学者向け（何が・なぜ問題か）

LiteLLM は、社内のアプリが「ChatGPT」「Claude」など複数のAIサービスを使うときに、窓口を一本化してくれる「受付係（プロキシ）」のようなソフトです。アプリは個々のAI会社と直接やり取りせず、この受付係に頼めば代わりに取り次いでくれます。便利な反面、この受付係はたくさんのAPIキー（各AIサービスの“合鍵”）を預かっているため、乗っ取られると一気に被害が広がります。

問題は、この受付係に付いている「接続テスト窓口」（/mcp-rest/test/connection などのテスト用API）です。本来は「ちゃんと外部サーバーにつながるか確認する」だけの機能ですが、テストの指定欄に「実行するコマンド」を書いて送ると、受付係がそのコマンドを自分のサーバー上でそのまま実行してしまいます。たとえるなら、受付に「電話がつながるか試して」と頼む紙に、こっそり「ついでに金庫を開けて」と書いておくと、受付係が疑わずに金庫まで開けてしまうようなものです。

さらに悪いことに、別の「本人確認をすり抜ける裏口」（CVE-2026-48710）と組み合わせると、合鍵（APIキー）を持っていない外部の攻撃者でも、この金庫破りを成立させられます。これが「未認証RCE（誰でもサーバーを乗っ取れる）」と呼ばれる最悪のパターンで、CISA（米国の政府機関）も「実際に攻撃が起きている」と警告しています。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **CVE/CVSS**: CVE-2026-42271、CVSS 8.7（High）。種別は OSコマンドインジェクション（CWE-78 系。LiteLLMが受け取った設定値を検証せずサブプロセスに渡す入力検証不備）。
- **影響箇所**: MCP（Model Context Protocol）サーバーのテスト用エンドポイント `/mcp-rest/test/connection` および `/mcp-rest/test/tools/list`。リクエストボディの `command` / `args` / `env` フィールドを受け取り、stdio トランスポートで子プロセスとして起動する。
- **前提条件（単体）**: 有効なプロキシAPIキーを持つ認証済みユーザー。実行されるコマンドは LiteLLM プロセスの権限を継承する。
- **前提条件（連鎖時）**: CVE-2026-48710（Starlette の Host ヘッダ検証バイパス）と連鎖すると認証自体を回避でき、**未認証RCE** に格上げされる。
- **影響バージョン / 修正版**: LiteLLM 1.74.2〜1.83.6 が影響。**1.83.7** で修正。連鎖緩和のため Starlette も 1.0.1 以降へ。
- **悪用状況**: CISA KEV 登録済み（2026-06-08 追加）、実環境での悪用が確認されている（要・最優先対応）。

### メカニズム（攻撃が成立する仕組み）

1. **信頼境界の取り違え**: テスト用エンドポイントは「外部MCPサーバーへの接続を試す」目的で、リクエスト送信者から `command`/`args`/`env` を受け取る設計になっている。ここで「クライアントが指定した値」を「サーバー側で実行してよい安全な設定」と暗黙に信頼しているのが根本原因。
2. **入力がプリミティブに直結**: stdio トランスポートは指定された `command` を子プロセスとして spawn する。検証・許可リスト・サンドボックスが無いため、攻撃者は「任意コマンド実行」というプリミティブを直接握る。
3. **権限の継承**: 起動された子プロセスは LiteLLM プロキシ本体の実行権限を引き継ぐ。AIゲートウェイは各種APIキーや内部ネットワークへの到達性を持つことが多く、ここを取られると横展開・鍵窃取の足場になる。
4. **認証境界の崩壊（連鎖）**: 単体では「APIキー保有者」という前提があるが、CVE-2026-48710 の Host ヘッダ検証バイパスを噛ませると、認証ミドルウェアの判定を欺いて未認証状態でテストAPIへ到達できる。結果、外部の攻撃者がそのまま手順1〜3を実行可能になる。

一般原理として、これは「管理・テスト目的の便利機能が、設計上のコマンド実行プリミティブを外部入力に開いてしまう」典型例（コマンドインジェクション）であり、さらに「複数の弱い欠陥（入力検証不備＋認証バイパス）が連鎖して致命的被害に至る」exploit chain の好例。なお、武器化の具体的手順・PoC・ペイロードはここでは扱わない（概念レベル）。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: `/mcp-rest/test/connection` `/mcp-rest/test/tools/list` への想定外のPOST、リクエストボディに `command`/`args`/`env` を含むもの、LiteLLM プロセスから派生する見慣れない子プロセス（シェル起動・ネットワーク発信）、Host ヘッダの不審な値。
- **暫定緩和**: 当該テストエンドポイントを WAF/リバースプロキシで遮断、LiteLLM 管理APIをインターネット非公開化（VPN/内部ネットワーク限定）、プロキシを最小権限の専用アカウントで実行、APIキーのローテーション。
- **恒久対策**: LiteLLM を **1.83.7 以降**、Starlette を **1.0.1 以降** へ更新。MCPサーバー設定を信頼境界の内側からのみ受け付けるよう構成し、外部入力由来の `command` を実行しない運用へ。

## 📖 用語解説（このニュースとのつながり）

- **AIゲートウェイ / LiteLLM プロキシ**: 複数のLLM APIを一元的に取り次ぐ中継基盤。本件では、その中継基盤自体が乗っ取りの標的になった。
- **MCP（Model Context Protocol）**: AIモデルに外部ツール／データ源を接続するためのプロトコル。本件の脆弱な「テスト用エンドポイント」はこのMCP接続を検証する機能だった。
- **コマンドインジェクション（CWE-78）**: 外部入力がOSコマンドの一部として実行されてしまう欠陥。本件では設定欄の `command` がそのまま実行された。
- **exploit chain（脆弱性の連鎖）**: 単体では限定的な複数の欠陥を組み合わせ、致命的な被害に引き上げる手法。本件は「認証回避＋コマンド実行」で未認証RCEに到達。
- **CISA KEV**: 米国CISAが「実際に悪用が確認された脆弱性」をまとめたカタログ。登録は「今まさに狙われている」ことを意味し、対応の最優先指標になる。

## 影響範囲 / 推奨対応

LiteLLM 1.74.2〜1.83.6 を運用する全環境（特に管理/テストAPIが到達可能な構成）。AIゲートウェイは多数のAPIキーと内部到達性を抱えるため、侵害時の影響は単一サーバーに留まらない。**最優先で 1.83.7 以降へ更新**し、Starlette も更新。更新までは当該エンドポイントの遮断と管理APIの非公開化、APIキーのローテーションを実施。KEV登録済みのため、悪用前提で侵害有無の調査（子プロセス/外部通信ログ）も行う。
