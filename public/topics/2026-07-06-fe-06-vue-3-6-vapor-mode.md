# Vue 3.6 Vapor Mode — 仮想DOMを捨てる新レンダリング

> トピック: Vue 3.6 Vapor Mode｜出典: Vue.js 公式ブログ / Vue School / Evan You（Vue.js Nation 講演）｜種別: メジャー機能（新レンダリング戦略）

## 一行サマリー

Vue 3.6 の Vapor Mode は、コンポーネントを仮想DOMを介さず直接DOM操作へコンパイルする新モードで、対象コンポーネントのバンドルを20〜50%小さくし、極端なケースで最大約97%高速なレンダリングを実現する。既存アプリに部分的・段階的に導入できるのが特徴。

## 🔰 初学者向け（何が・なぜ重要か）

これまで Vue は「仮想DOM（Virtual DOM）」という仕組みで画面を更新してきました。画面の設計図をメモリ上にコピーとして持っておき、データが変わるたびに「新しい設計図」と「古い設計図」を見比べて、違う部分だけを本物の画面に反映する——という方式です。差分を毎回計算するので安全ですが、その計算自体にコストがかかります。

Vapor Mode は、この「設計図を毎回見比べる」工程をやめます。コンパイル（=事前の変換）の段階で「このデータが変わったら、この場所を直接書き換えればいい」という命令をあらかじめ焼き込んでおくのです。料理でいえば、注文が来てからレシピを毎回読み直すのではなく、「この客が来たらこの一皿」と手が覚えている状態に近い。結果として画面更新が速くなり、メモリ使用も減ります。

嬉しいのは、これがオール・オア・ナッシングではない点です。アプリ全部を一気に書き換える必要はなく、「重い一覧画面だけ Vapor にする」といった部分導入ができます。React が Compiler で似た方向へ進んでいるのと同様、Vue も「実行時に頑張る」から「コンパイル時に賢くしておく」へ舵を切った、という流れの一部です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

Vapor Mode は Vue 3.6 で導入された、仮想DOMを使わない代替コンパイル戦略。単一ファイルコンポーネント（SFC）を VNode 生成ではなく命令的なDOM操作へ直接コンパイルする。公表されている特性は、Vapor 対象コンポーネントでバンドルサイズが約20〜50%減、極端なベンチマークで最大約97%高速なレンダリング、10万コンポーネントのマウントが約100ミリ秒、複雑な再レンダリング時のピークメモリ約22%減など（数値はベンチマーク条件依存のため実アプリでの効果は要検証）。Evan You は Vue.js Nation 等で Vapor を予告し、2026年4月に feature-complete に到達したと報じられている（正式な安定版の位置づけは公式リリースノートで要確認）。

### 仕組み / 背景（なぜこう設計されたか）

従来の仮想DOMは「宣言的に書ける安全網」だが、更新のたびに VNode ツリーを生成し diff する固定コストが乗る。Solid.js や Svelte 5 が示したのは、リアクティビティ（依存追跡）とコンパイルを組み合わせれば、diff を介さずに「変わった値 → 変わるDOMノード」を直接結線でき、VNode 生成そのものを不要にできるという設計だ。Vapor Mode はこの潮流を Vue の Composition API・リアクティビティシステムの上で実現したもの。VNode オブジェクトを割り当てないため GC 圧が下がり、メモリのピークも減る。

トレードオフもある。細粒度の結線を成立させるにはコンパイラがテンプレートの依存関係を静的に解析できる必要があり、動的な描画（自由な `h()` 手書きなど、仮想DOMの柔軟さに頼る書き方）とは相性が悪い。ゆえに Vapor は「テンプレート中心・opt-in」で提供され、vdom ベースの既存コードと共存する二層構成になっている。

### 実務での使いどころ・移行の勘所

効果が大きいのは、大量の要素を描画・頻繁に更新するデータ集約型UI（テーブル、ダッシュボード、リアルタイム表示）。まずはホットパスの一覧・グリッドを Vapor 化し、計測（レンダリング時間・メモリ）で効果を確認するのが安全。アプリ全体の書き換えは不要で、コンポーネント/ページ単位で段階導入できるため、リスクを局所化できる。一方、`render` 関数を手書きしている箇所や、仮想DOMの動的さに依存するライブラリ連携は Vapor 化の対象外になりやすいので、境界を意識して設計すること。Laravel + Inertia のような統合構成でも部分導入は可能だが、SSR/ハイドレーションとの組み合わせ挙動は自分のスタックで検証を。

## 📖 用語解説（このトピックとのつながり）

- **仮想DOM（Virtual DOM）**: UIの状態をメモリ上のオブジェクトツリーとして表し、差分計算で実DOMを更新する仕組み。Vapor Mode はこれを介さない点が新しい。
- **リアクティビティ（依存追跡）**: どの値がどの表示に影響するかを自動追跡する仕組み。Vapor は追跡結果をDOM操作へ直接結線する。
- **コンパイル時最適化**: 実行前にテンプレートを解析し、更新命令を事前生成する手法。Solid/Svelte と共通の思想。
- **feature-complete**: 予定機能が出揃った状態。安定版（stable）とは別概念で、本番採用可否はリリース表記で確認する。

## 影響範囲 / 推奨アクション

- 対象: Vue 3 を使うフロントエンド全般。特に描画量の多い画面を抱えるアプリ。
- アクション: まず重い一覧/表を1つ Vapor 化し、レンダリング時間とメモリを計測。効果と既存ライブラリ互換を確認してから範囲を広げる。数値は公式ベンチ依存のため自環境で裏取りする。
