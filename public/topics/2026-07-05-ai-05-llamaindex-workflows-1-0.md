# LlamaIndex Workflows 1.0 — イベント駆動でエージェントを組む軽量フレームワークが独立

> トピック: LlamaIndex Workflows 1.0｜出典: LlamaIndex 公式ブログ｜種別: フレームワーク（メジャーリリース）

## 一行サマリー

LlamaIndex が、多段のエージェント的 AI アプリを組むための軽量フレームワーク Workflows 1.0 を公開。独立したリポジトリ・パッケージ・開発ラインを持つ初の単独版で、llama_index に依存せずに導入でき、型付きの Workflow State・リソース注入・可観測性を備え、Python/TypeScript でイベント駆動の多段ロジックを表現できる。

## 🔰 初学者向け（何が・なぜ重要か）

AI エージェントに複雑な仕事をさせるには、「検索する→要約する→判断する→必要なら別の手順に分岐する」といった**多段の流れ**を組む必要があります。これを場当たり的なコードで書くと、途中で止めたり再開したり、途中経過を見たりするのが難しくなります。

LlamaIndex Workflows 1.0 は、この「多段の流れ」を整理して書くための軽量な骨組みです。特徴は「イベント駆動」という考え方。各処理ステップが「イベント（出来事）」を出し合い、それを受けて次のステップが動く、という形で流れを表現します。分岐やループ、並列、一時停止と再開もきれいに書けます。

たとえるなら、料理のレシピを「①下ごしらえ→②炒める→③味見して足りなければ調味」と、各工程を札（イベント）でつなぐようにして、途中で止めても続きから再開できるようにした段取り表。1.0 では、この骨組みが LlamaIndex 本体から独立し、他のアプリにも気軽に組み込めるようになりました。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

Workflows 1.0 は、多段のエージェント的システムを Python/TypeScript で構築する軽量フレームワーク。主な点:

- **独立版**: 専用リポジトリ・パッケージ・開発ラインを持つ初のスタンドアロン版。`llama_index` を使っていない環境にも導入できる。
- **型付き Workflow State**: 型安全性を高めた状態管理。リソース注入・可観測性（observability）と併せ、多段ロジックをオーケストレーションできる。
- **表現力**: ステップの連結、ループ、並列パス、async ワークフローによる高速実行、FastAPI など Python アプリとの統合、start/pause/resume のステートフルなワークフロー。
- **完全 OSS**: 商用・本番利用に制約なし。

用途は AI エージェント、文書処理パイプライン、マルチモーダル、リサーチアシスタント、ロールベースのマルチエージェント等。これにより LlamaIndex は「インデックス（検索）ライブラリ」から「イベント駆動のエージェント基盤＋検索」へと位置づけを広げた。

### 仕組み / 背景（なぜこう設計されたか）

エージェントの制御フローを表す方式には、大きく「グラフ（ノードとエッジで状態遷移を定義）」型と「イベント駆動」型がある。LlamaIndex Workflows は後者を採る。各ステップは特定のイベント型を入力に取り、別のイベント型を出力する関数として書かれ、フレームワークがイベントの流れに従ってステップを起動する。この設計の利点は、分岐・並列・ループが「どのイベントを出すか」という局所的な記述に還元され、全体のフロー図を中央集権的に定義しなくてよいこと。async ファーストなので、複数の検索・LLM 呼び出しを並行実行して待ち時間を隠せる。

型付き State と start/pause/resume は、実運用で重要な「長時間・中断可能なワークフロー」を支える。人間の承認待ちや外部イベント待ちで一旦止め、後で状態から再開できることは、エージェントを本番業務に載せる際の要件になる。1.0 で本体依存を切り離したのは、「RAG（検索拡張生成）に縛られない汎用オーケストレーション基盤」として使えるようにするためで、実際の本番構成では「LlamaIndex（取り込み・検索）＋ Workflows/LangGraph（オーケストレーション）＋ 観測ツール」という補完的な組み合わせが一般化している。トレードオフは、イベント駆動は柔軟な反面、フローが暗黙的になりやすく、全体像の把握には可観測性ツールが欠かせない点だ。

### 実務での使いどころ・移行の勘所

RAG に留まらない多段エージェント（調査→抽出→検証→生成のような複合タスク）を、型安全かつ中断可能に組みたいケースに向く。既存の FastAPI サービスへ組み込みやすいのも利点。導入の勘所は、(1) イベント設計を先に固めること（イベント型が API の骨格になる）、(2) 可観測性（トレース）を最初から有効にし、暗黙的なフローを可視化すること、(3) 長時間ワークフローでは pause/resume の状態永続化先（DB 等）を設計すること。オーケストレーションは Workflows か LangGraph か、検索は LlamaIndex か、と役割を分けて選ぶと、ロックインを避けつつ強みを組み合わせられる。

## 📖 用語解説（このトピックとのつながり）

- **Workflows 1.0**: イベント駆動で多段エージェントを組む LlamaIndex の軽量フレームワーク。本トピックの主役。
- **イベント駆動**: 各ステップがイベントを出し合い連鎖する制御フロー方式。分岐・並列を局所的に書ける。
- **型付き Workflow State**: 型安全な状態管理。多段処理の堅牢性を高める。
- **start/pause/resume**: ワークフローを途中停止・再開できる仕組み。人間承認待ち等に対応。
- **RAG（検索拡張生成）**: 検索で得た情報を LLM に与えて回答生成する手法。LlamaIndex の従来の主戦場。

## 影響範囲 / 推奨アクション

多段・中断可能なエージェントを型安全に組みたいチームに有効。イベント設計を先に固め、可観測性を初期から有効化し、長時間ワークフローは状態永続化を設計する。検索・オーケストレーション・観測を役割分担で選ぶとロックインを避けられる。
