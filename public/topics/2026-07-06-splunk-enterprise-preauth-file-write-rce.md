# Splunk Enterprise の未認証・任意ファイル作成/切り詰めでRCEへ（CVE-2026-20253）

> トピック: 脆弱性ニュース ｜ 出典: Splunk SVD-2026-0603 / CISA / 各社解析 ｜ 種別: 認証欠如による任意ファイル操作（→RCE）

## 一行サマリー

ログ分析基盤 Splunk Enterprise に、認証情報なしで任意のファイルを作成・切り詰めできる致命的脆弱性（CVSS 9.8）が見つかった。内蔵の PostgreSQL サイドカー用エンドポイントに認証が掛かっておらず、ここを踏み台に最終的にリモートコード実行へつながる。Splunk PSIRT は限定的な悪用を確認済みで、10.2.4 / 10.0.7 以降への更新が急務。

## 🔰 初学者向け（何が・なぜ問題か）

Splunk は、社内のサーバやアプリが吐き出す大量のログを集めて検索・監視する「ログの司令塔」です。セキュリティ監視（SIEM）の中心に据える組織も多く、ここが乗っ取られると監視網そのものを攻撃者に握られてしまいます。

今回の穴を例えるなら、「本館（Splunk本体）の隣に建っている“データ倉庫”（内蔵のPostgreSQLデータベースを世話する補助係）に、鍵のかかっていない裏口があった」というものです。この裏口（サイドカー用のエンドポイント）は本来、内部の部品どうしが連絡し合うためのもので、外から誰でも触れる想定ではありませんでした。ところが認証（合言葉の確認）がまったく無かったため、ネットワークが届く相手なら誰でも、ログインせずにこの裏口を叩けてしまいます。

裏口からできてしまうのが「ファイルを新しく作る」「既存ファイルの中身を空っぽにする（切り詰め）」という操作です。一見地味ですが、攻撃者が“中身を自由に置ける新しいファイル”をサーバの好きな場所に作れると、そのファイルが後から自動で実行される設定ファイルやスクリプトだった場合、結果として攻撃者の書いた命令がサーバ上で動きます。つまり「ファイルを置ける」という小さな力が、「サーバを乗っ取る（コード実行）」という大きな被害に化けるのです。ログインすら不要という点で、リスクは最大級です。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **CVSS**: v3.1 で 9.8 Critical（`AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H`）。
- **CWE**: CWE-306（重要機能に対する認証の欠如）。
- **脆弱性の性質**: 未認証の任意ファイル作成・切り詰め（arbitrary file creation & truncation）。悪用の連鎖で未認証RCEに至ると各社が評価。
- **前提条件**: 該当エンドポイント（PostgreSQL サイドカーサービス）へのネットワーク到達性のみ。認証情報不要。
- **影響バージョン**: Splunk Enterprise 10.2.0〜10.2.3、および 10.0.0〜10.0.6。9.4系以前は影響なし（要確認：構成依存の可能性あり）。
- **修正版**: 10.2.4 / 10.0.7 以降（または 10.4.0 以降）へアップグレード。
- **悪用状況**: 2026年6月に Splunk PSIRT が限定的な悪用（limited exploitation）を認識。CISA も期限付きパッチ適用を連邦機関へ指示。

### メカニズム（攻撃が成立する仕組み）

本件は「内部前提のコンポーネントが、実際にはネットワーク到達可能かつ無認証で公開されていた」という、信頼境界の設計ミスに根差す。

1. **信頼境界の誤り（内部前提の露出）**: Splunk は内蔵 PostgreSQL を“サイドカー”として同梱し、本体プロセスと連携させる。そのサイドカー制御用エンドポイントは「同一ホスト内・信頼済みの呼び出し元」だけが使う前提で作られていたが、実際にはネットワークから到達でき、かつ認証チェック（CWE-306）が実装されていなかった。ここで「誰でも特権機能を呼べる」というプリミティブが成立する。

2. **獲得プリミティブ＝任意ファイル書き込み/切り詰め**: 攻撃者はこのエンドポイント経由で、サーバ上の任意パスに対しファイルの新規作成・内容の切り詰めを行える。これは「任意書き込み」に相当する強力なプリミティブで、機密性・完全性・可用性すべてに影響する。

3. **RCEへの橋渡し**: 任意ファイル書き込みが単体でRCEへ昇格する経路は、一般に「①後から読み込まれる設定/起動スクリプト/信頼されたパスへの上書き」「②Web到達可能ディレクトリへの実行可能ファイル配置」「③既存ファイルの切り詰めによる保護機構の無効化」などが知られる。攻撃者が中身を制御できるファイルを“実行される場所”に置ければ、任意コード実行に直結する。これが「ファイル操作 → RCE」の橋渡しの型である。

4. **クラスの一般原理**: 認証欠如（CWE-306）＋任意ファイル書き込み（CWE-73/CWE-22系）の組み合わせは、Web/アプライアンス製品で繰り返し観測される高危険パターン。サイドカーや管理用内部APIを“ネットワーク公開してしまう”構成ミスが引き金になりやすい。

※武器化可能な具体的リクエスト・書き込み先パス・PoCは記載しない（概念レベル）。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: Splunk ホストで、PostgreSQL サイドカー関連ポート/エンドポイントへの外部からの想定外アクセスを監視。Splunk のディレクトリ配下で予期しないファイル新規作成・タイムスタンプ更新・既存ファイルのサイズ0化（切り詰め痕跡）を検出。プロセス生成の異常（Splunk配下からの子プロセス）を相関。
- **暫定緩和**: 管理・内部通信ポートをインターネットや広い社内セグメントから遮断し、到達元をローカルホスト/管理セグメントに限定。ネットワークACL・ホストFWで該当エンドポイントを保護。監視系サーバは特に露出面の最小化を。
- **恒久対策**: Splunk Enterprise を 10.2.4 / 10.0.7 以降（または 10.4.0 以降）へアップグレード。適用前に露出構成の棚卸しと侵害痕跡（IoC）確認を行う。
- **運用示唆**: SIEM/監視基盤の侵害は「監視の無効化」と「横展開の起点」を同時に招く。KEV/PSIRT情報に基づき最優先で対処する。

## 📖 用語解説（このニュースとのつながり）

- **Splunk Enterprise / SIEM**: ログを集約・検索・監視するプラットフォーム。多くの組織でセキュリティ監視の中核。ゆえに侵害時の影響が大きい。
- **サイドカー（sidecar）**: 主サービスの隣で補助機能を担う付随プロセス。本件では内蔵PostgreSQLの制御役。内部前提のはずが外部到達可能だった。
- **CWE-306（認証の欠如）**: 重要な機能に認証を課していない欠陥。今回の根本原因で、誰でも特権操作を呼べた。
- **任意ファイル作成/切り詰め**: 攻撃者が任意パスにファイルを作る・中身を空にする操作。RCEや保護無効化の足場になる。
- **未認証RCE**: 認証情報なしにリモートで任意コードを実行できる最も危険な状態。本件は複数解析が“RCEに至る”と評価。

## 影響範囲 / 推奨対応

Splunk Enterprise 10.2.0〜10.2.3 / 10.0.0〜10.0.6 が対象。監視基盤という性質上、被害は自組織のセキュリティ可視性の喪失に直結する。直ちに 10.2.4 / 10.0.7 以降へアップグレードし、適用までは PostgreSQL サイドカー関連の内部エンドポイントを外部・広域セグメントから遮断すること。適用後も侵害痕跡（不審ファイル・切り詰め・子プロセス）の遡及調査を推奨。
