# Gogs に CVSS 9.4 の認証後RCE（CVE-2026-52806）— 不正なブランチ名で git rebase に --exec を注入

> トピック: 脆弱性ニュース ｜ 出典: Rapid7 / The Hacker News / SecurityWeek ｜ 種別: 引数インジェクション（CWE-88）→ リモートコード実行

## 一行サマリー
セルフホスト型Gitサービス Gogs に、悪意あるブランチ名を使って `git rebase` の `--exec` オプションを注入し任意コマンドを実行できる脆弱性（CVE-2026-52806・CVSSv4 9.4）が見つかり、PoC/Metasploitモジュールも公開済み。既定設定では誰でもアカウントを作れるため実質「ほぼ未認証」で悪用され得る。バージョン 0.14.3（2026-06-07）で修正された。

## 🔰 初学者向け（何が・なぜ問題か）
**Gogs（ゴグス）** は、社内やサーバーに自分で立てて使える小さな「GitHubのような」ソースコード保管サービスです。多くのチームが手軽さからこれを自前のサーバーに置いて使っています。

今回の問題を料理にたとえると――レストランの注文票（ブランチ名）に「料理名」を書く欄があり、本来はそこに料理名しか入らないはずでした。ところが、お客（攻撃者）がその欄に「料理名」ではなく「厨房のコンロを全部つけろ」という"指示文"を書き込むと、システムがそれを命令だと勘違いして実行してしまう、という状態です。

Gogs には「マージ前にリベース（rebase）する」という、変更を整える機能があります。ここで内部的に `git` というコマンドに「ブランチ名」を渡すのですが、その名前が"ただの名前"ではなく"オプション（命令の追加指示）"として解釈されてしまいました。攻撃者が `--exec=（好きなコマンド）` という名前のブランチを作ってマージ操作を行うと、サーバー上でそのコマンドが実行されてしまいます。

さらに厄介なのは、Gogs は初期設定のままだと**誰でも自由に新規登録できる**点です。つまり攻撃者は「まず適当にアカウントを作る → 自分のリポジトリで仕掛ける」だけで攻撃が完結します。管理者でなくてもよく、他のユーザーを騙す必要もありません。結果として、悪用されるとサーバーを乗っ取られ、コードの窃取・改ざん（サプライチェーン汚染）に直結します。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）
- **CVE**: CVE-2026-52806
- **脆弱性種別**: 引数インジェクション / Argument Injection（CWE-88: Improper Neutralization of Argument Delimiters in a Command）
- **CVSS**: CVSSv4 9.4（Critical）※Rapid7評価
- **影響バージョン**: Gogs 0.14.2 および 0.15.0+dev（commit b53d3162）までで再現確認。これ以前も影響の可能性。
- **修正版**: Gogs 0.14.3（2026-06-07 リリース、PR #8301）
- **悪用状況**: PoC/エクスプロイトモジュールが公開済み。報道では実環境での悪用・スキャンが観測されており、関連するGogsの別欠陥（パストラバーサル系）はCISA KEV登録歴あり。本CVE自体のKEV登録は要確認。
- **前提条件**: 認証ユーザーであること。ただし既定設定（`DISABLE_REGISTRATION = false`、`MAX_CREATION_LIMIT = -1`）では誰でも登録＆リポジトリ作成が可能なため、実質的に未認証攻撃者でも成立し得る。「Rebase before merging」マージ操作のトリガーが必要。

### メカニズム（攻撃が成立する仕組み）
1. **信頼境界の取り違え**: ユーザーが制御できる「ブランチ名」という*データ*が、サーバー側で `git rebase` を起動する際に*コマンドライン引数*としてそのまま連結される。ここでデータとコードの境界が曖昧になる。
2. **オプション区切りの欠落**: 安全に外部コマンドを呼ぶには、引数列の末尾で `--`（オプション終端記号）を置き「これ以降は値であってオプションではない」と明示する必要がある。本件ではこの `--` デリミタが欠落していた。
3. **`--` で始まる名前がオプションに化ける**: `git` をはじめ多くのCLIは、`--` で始まるトークンを自動的に「オプション」と解釈する。攻撃者がブランチ名を `--exec=<コマンド>` という形にすると、`git rebase` はこれを"リベース中に各コミットで実行するコマンド"を指定する正規オプション `--exec` として受理する。
4. **プリミティブ→最終被害の橋渡し**: 攻撃者が握ったプリミティブは「`git` のサブコマンドに任意オプションを差し込める」という限定的なもの。しかし `--exec` は設計上"任意コマンド実行"を許す強力なオプションであるため、これが直接 RCE（Gogsプロセス権限での任意OSコマンド実行）に橋渡しされる。
5. **トリガー**: 攻撃者は悪意あるブランチ名（例: `--exec=touch${IFS}/tmp/rce_proof` のように空白回避に `${IFS}` を用いる概念例）でプルリクエストを作り、「Rebase before merging」でマージするだけ。自分のアカウント・自分のリポジトリ内で完結し、他者の操作や承認を要しない。

> ※ 一般原理として、本クラス（引数インジェクション）は「外部コマンド呼び出しでユーザー入力を引数列に渡す際、`--` 終端やallowlist検証を怠ると、`-`/`--` 始まりの入力がオプションに昇格する」ことで生じる。OSコマンドインジェクション（シェルメタ文字）とは異なり、シェルを経由せずとも成立する点が特徴。武器化可能な完全手順・正確なペイロードはここでは記載しない（概念レベル）。

### 検知・暫定緩和・恒久対策
- **恒久対策**: Gogs を **0.14.3 以降**へアップデート（最優先）。
- **暫定緩和**: パッチ適用までは `app.ini` で **`DISABLE_REGISTRATION = true`**（自由登録の無効化）。攻撃の自己完結性を断つ最も効果的な緩和。信頼できないユーザーにリポジトリ作成権を与えない。インターネット直結のGogsはアクセス制限/リバースプロキシ＋認証で隔離。
- **検知の着眼点**: `--` で始まる異常なブランチ名（特に `--exec` を含む）、`git rebase` 子プロセスからの予期しないプロセス生成（webサーバ→`git`→`sh`/任意バイナリ）、新規作成直後アカウントによるPR作成→即マージの不審な連鎖。WAF/監査ログでブランチ名のパターン監視。
- **運用示唆**: 開発基盤（Git/CI）は侵害時の影響が大きい高価値資産。アセット棚卸し（公開Gogsの洗い出し）と最小権限・登録制限をベースラインに。

## 📖 用語解説（このニュースとのつながり）
- **引数インジェクション（Argument Injection / CWE-88）**: 外部コマンドへ渡す引数に攻撃者入力が混じり、値であるべきものが「オプション」として解釈される脆弱性。本件では"ブランチ名"が `--exec` オプションに化けた。
- **オプション終端記号 `--`**: 「ここから先はオプションではなく値」とCLIに伝える区切り。これを置かなかったことが根本原因。
- **`git rebase --exec`**: リベース中に各コミットで指定コマンドを走らせる正規機能。攻撃者に握られると任意コマンド実行の足場になる。
- **PoC / Metasploitモジュール**: 攻撃の実証コード。公開されると悪用のハードルが急落するため、パッチ適用の緊急度が上がる。本件では公開済み。
- **`DISABLE_REGISTRATION`**: 自由なユーザー登録を許すか否かの設定。既定で許可のため「実質未認証」化していた。

## 影響範囲 / 推奨対応
インターネット公開・社内問わず Gogs を運用する全組織が対象。特に自由登録を有効にしたままの公開インスタンスは即時リスク。**0.14.3 への更新**を最優先で実施し、当面は自由登録の無効化とアクセス制限で攻撃面を縮小する。侵害有無は `git` 由来の不審プロセス生成痕跡で確認する。
