# WordPress「Everest Forms Pro」CVSS 9.8 の未認証RCE（CVE-2026-3300）が悪用中 — eval()への入力混入でサイト乗っ取り

> トピック: 脆弱性ニュース ｜ 出典: The Hacker News / BleepingComputer / SecurityWeek / Wordfence ｜ 種別: PHPコードインジェクション（未認証RCE）

## 一行サマリー

WordPress用フォーム作成プラグイン Everest Forms Pro の「複雑な計算（Complex Calculation）」機能に、ユーザー入力を eval() へそのまま渡す CVSS 9.8 の致命的欠陥（CVE-2026-3300）があり、未認証の攻撃者が任意PHP実行→管理者アカウント作成でサイトを乗っ取り中。1.9.13 への即時更新が必須。

## 🔰 初学者向け（何が・なぜ問題か）

Everest Forms は、WordPressサイトに「お問い合わせフォーム」や「申込フォーム」を簡単に作れる人気プラグイン（拡張機能）です。その有料版 Pro には、フォームに入れた数値を使って自動計算する「複雑な計算」機能があります。たとえば「個数 × 単価 = 合計」を自動表示する、といった便利機能です。

問題は、この計算機能の作りにあります。利用者がフォームに入力した値を、サイトが内部で「計算式の文章」として組み立て、それを eval() という“書いてある文字列をそのままプログラムとして実行する”仕組みに渡していました。料理にたとえると、お客さんが注文票に書いた文字を、確認せずにそのまま厨房への指示書として実行してしまうようなものです。

普通のお客さんは「2個」と書きますが、悪意ある人は注文票に“厨房を乗っ取る命令文”を書き込めます。サイトはそれを命令として実行してしまうため、攻撃者は自分専用の管理者アカウントを勝手に作り、サイトを丸ごと乗っ取れてしまいます。しかもログイン不要（未認証）で、フォームを開けば誰でも送れる入口なので、被害が広がりやすいのです。

実際この欠陥はすでに世界中で悪用されており、あるセキュリティ企業は2万9千件以上の攻撃試行をブロックしたと報告しています。フォーム一つの油断が、サイト全体の乗っ取りにつながる典型例です。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **種別**: PHPコードインジェクション（CWE-94: Improper Control of Generation of Code / eval injection の類型）。
- **CVSS**: 9.8（Critical）。攻撃元はネットワーク、認証不要、ユーザー操作不要。
- **前提条件**: 対象フォームが Calculation Addon の「Complex Calculation（複雑な計算）」機能を使用していること。攻撃者は当該フォームに値を送信するだけ。
- **根本原因**: Calculation Addon の `process_filter()` がユーザー送信のフィールド値をPHPコード文字列へ連結し、十分なエスケープなく `eval()` に渡す。入力に適用される `sanitize_text_field()` はシングルクォート等のPHPコード文脈文字をエスケープしないため、コード文脈への脱出を防げない。
- **影響バージョン**: Everest Forms Pro 1.9.12 以前すべて。
- **修正版**: 1.9.13。
- **悪用状況**: 実環境で**活発に悪用中**。初観測は 2026-04-13。Wordfence のテレメトリで 29,300 件超の試行をブロック。主たるペイロードは不正な管理者アカウント作成。

### メカニズム（攻撃が成立する仕組み）

以下は概念レベルの説明であり、武器化可能な具体ペイロード・手順は含めない。

1. **信頼境界の所在**: フォーム入力は「外部由来＝非信頼データ」である。だが計算機能はその値を「自サイトのコードの一部」として扱ってしまう設計で、データ／コードの境界が崩れている。
2. **コード生成への混入**: `process_filter()` がフィールド値をPHPの式文字列へ文字列連結する。ここで入力は本来「数値というデータ」であるべきだが、コード文字列を構築する材料に格上げされてしまう。
3. **不十分なサニタイズ**: 適用される `sanitize_text_field()` はHTML/制御文字向けの整形であり、PHPコード文脈のメタ文字（クォート等）を中和しない。よって攻撃者は文字列リテラルを閉じてコード文脈へ脱出するプリミティブを握る。
4. **eval() による実行**: 構築された文字列が `eval()` に渡され、注入されたPHPがサーバ権限（WebサーバのPHP実行権限）で実行される。これが任意コード実行（RCE）の核心。
5. **被害の橋渡し**: 得た実行権限でWordPressのDBへ管理者ユーザーを挿入、またはWebシェルを設置。以後は正規の管理機能を通じて永続的・全面的な制御（サイト乗っ取り、改ざん、別マルウェア配布）へ拡大する。

脆弱性クラスとしては、**動的コード評価（eval injection）への非信頼入力の混入**という、最も危険なインジェクションの一類型。「入力をコードとして評価しない」「どうしても評価が必要なら厳格なホワイトリスト式パーサで処理する」が原則的な防止策。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: フォーム送信ログ中の異常に長い／PHP的メタ文字を含む計算フィールド値、Webサーバからの予期しないアウトバウンド、`wp_users`/`wp_usermails`への不審な管理者追加、新規の不審PHPファイル（Webシェル）出現、想定外の管理者ログイン。
- **暫定緩和（パッチ前）**: 該当の「複雑な計算」機能を使うフォームを一時停止、またはプラグインを無効化。WAFで計算フィールドへのコード様入力を遮断。
- **恒久対策**: Everest Forms Pro を 1.9.13 へ即時更新。更新後は全管理者アカウントの棚卸し（不正作成の有無確認）、不審ファイル・スケジュールタスクの除去、必要に応じ資格情報リセット。
- **運用示唆**: 「入力→eval()」という設計を持つ拡張機能は高リスク。CMSプラグインは導入数を絞り、更新適用とログ監視を運用に組み込む。

## 📖 用語解説（このニュースとのつながり）

- **WordPress プラグイン**: WordPressに機能を追加する拡張部品。本件＝フォームプラグインの一機能が侵入口になった。
- **eval()**: 文字列をそのままコードとして実行するPHP関数。本件＝ここに非信頼入力が届いたことが致命傷。
- **コードインジェクション（CWE-94）**: 攻撃者の入力がプログラムコードとして実行される欠陥。本件の本質。
- **sanitize_text_field()**: WordPressの入力整形関数。意味＝HTML/制御文字向けの無害化。本件＝PHPコード文脈は守れず、防御として不十分だった。
- **未認証RCE**: ログインなしで任意コード実行に至ること。本件＝フォーム送信だけで成立し被害が拡大しやすい。
- **Webシェル**: 攻撃者がサーバを遠隔操作するために設置する不正スクリプト。本件＝乗っ取り後の永続化手段の一つ。

## 影響範囲 / 推奨対応

影響を受けるのは Everest Forms Pro 1.9.12 以前を使い、かつ「複雑な計算」機能を有するフォームを公開している全WordPressサイト。未認証・CVSS 9.8・実環境悪用中という三拍子で緊急度は最高。推奨対応は、(1) 1.9.13 への即時更新、(2) 更新まで該当機能の停止／プラグイン無効化、(3) 不正管理者アカウント・Webシェルの有無調査と除去、(4) WAF・ログ監視の強化。
