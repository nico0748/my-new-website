# Vue 3.6 Vapor Mode がフィーチャーコンプリート — 仮想DOMを外す選択肢

> トピック: 開発トピック｜出典: Vue.js 公式（X）｜種別: frontend

## 一行サマリー
Vue 3.6 の Vapor Mode が Suspense を除く安定機能で仮想DOMモードと機能同等（feature parity）に到達し、beta が公開された。コンポーネント単位で仮想DOMを外して実DOMを直接更新でき、初回JSやメモリ使用量が大幅に削減される。

## 🔰 初学者向け（何が・なぜ重要か）
Vue はこれまで「仮想DOM」という下書き用紙を使ってきました。画面を更新するとき、まず頭の中（メモリ上）に新しい下書きを作り、前の下書きと見比べて、違うところだけ本番の紙（実DOM）に清書する、という二段構えです。安全ですが、下書きを作って見比べる手間が毎回かかります。

Vapor Mode は、この下書き工程をビルド時（コードを準備する段階）に済ませてしまい、実行時は「どの値が変わったら、画面のどこを直すか」をあらかじめ配線しておく方式です。実行時は下書きを作らず、変わった値から直接その場所を書き換えます。結果として、持ち歩く道具（JSの量）が減り、頭の中の作業机（メモリ）も広く使えます。

重要なのは、これが「Vue を全部書き換えろ」という話ではない点です。既存の書き方を保ったまま、速度が欲しい箇所だけ Vapor に切り替えられる——段階的に速くできる、というのが 3.6 の嬉しさです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Vue 公式は 3.6.0-beta.1 を公開し、Vapor Mode が「Suspense を除く仮想DOMモードの全安定機能と機能同等」に達したと告知した。各種解説記事によれば、Vapor 採用で初回ロードのJSが約2/3減、実行時メモリがほぼ半減という測定例が報告される（数値は構成依存で、公式ベンチと自環境での検証が前提。要確認）。Vapor はコンパイル時に差分計算を解決し、実行時は細粒度リアクティビティで実DOMを直接更新する。既存の仮想DOMコンポーネントと混在でき、ホットパスだけの部分適用が可能。

### 仕組み / 背景（なぜこう設計されたか）
仮想DOMは「宣言的UI」を実現する汎用解だが、更新のたびに VNode ツリーを生成・diff する定常コストを伴う。Vue の Reactivity（`ref` / `computed` / `effect`）はもともと依存を追跡できるため、その依存グラフを直接DOM更新に結びつければ diff は不要になる——これが Vapor の発想で、Solid の「コンパイル + シグナル」路線に近い。トレードオフは、テンプレートを静的解析してDOM操作を生成する都合上、極端に動的な描画や一部の実行時トリックが制約を受ける点、および Suspense など一部機能が後追いになる点。エコシステム（ライブラリ）が Vapor 前提の挙動に追随するまでの過渡期も考慮が要る。

### 実務での使いどころ・移行の勘所
リスト描画やダッシュボードなど、コンポーネント数と再描画が多い「重い画面」から部分導入すると費用対効果が高い。Composition API（`<script setup>`）で書かれた素直なコンポーネントほど Vapor に載せやすい。Laravel + Inertia のような構成でも、負荷の高いページから試すのが定石。全面移行を急がず、計測 → ホットパス特定 → Vapor 化 → 再計測のループで進めるのが安全。

## 📖 用語解説（このトピックとのつながり）
- **仮想DOM（Virtual DOM）**: 更新前後を差分比較するための中間表現。Vapor はこれを実行時に持たない。
- **Vapor Mode**: 仮想DOMを迂回し、実DOMを直接更新する Vue のコンパイル戦略。3.6 で機能同等に到達。
- **細粒度リアクティビティ**: 変わった値の依存先だけを更新する仕組み。Vue の Reactivity を DOM 更新へ直結する。
- **feature parity（機能同等）**: 新方式が既存方式と同じ機能を提供できる状態。3.6 では Suspense を除き達成。
- **Suspense**: 非同期依存の解決中に待機UIを出す機能。Vapor では後追い対応となる。

## 影響範囲 / 推奨アクション
Vue を使う全チームに関係。既存アプリは書き換え不要で、重い画面から段階導入を検討。新規は Composition API を前提に、性能要件の高い部分で Vapor を選択肢に入れる。beta 段階のため本番採用は計測とライブラリ互換の確認を経てから。
