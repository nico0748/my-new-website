# MCP がほぼ普遍的に採用 — 月間SDK DL 9,700万が示すエージェント標準の定着

> トピック: 開発トピック｜出典: SitePoint｜種別: ai

## 一行サマリー
Model Context Protocol（MCP）は、月間 SDK ダウンロードが 9,700万件に達し（要確認）、AI エージェントが外部ツールやデータへ接続する事実上の共通規格として定着した。もはや「使うかどうか」ではなく「どのサーバに繋ぐか」を検討する段階に入っている。

## 🔰 初学者向け（何が・なぜ重要か）
MCP は「AI と外部世界をつなぐ USB-C」に例えられる。以前は、ChatGPT に社内 DB を読ませたい、Claude に GitHub を操作させたい、といった要望のたびに各社が独自の繋ぎ込みを実装していた。プラグごとに形が違うコンセントを毎回自作していたようなものだ。

MCP は、この繋ぎ口を一つの規格に統一した。サーバ（ツール提供側）が MCP に沿って作られていれば、どの AI クライアントからでも同じ手順で繋げる。月間 9,700万という SDK ダウンロード数（要確認）は、この「共通コンセント」が世界中の開発現場に配られたことを意味する。標準が普及すると、開発者は接続の苦労から解放され、AI にやらせたい仕事そのものに集中できるようになる。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
MCP は JSON-RPC ベースのプロトコルで、クライアント（AI アプリ）とサーバ（ツール・データ提供側）の間で「tools」「resources」「prompts」といった能力を交換する。近年はトランスポート層が stdio 中心から HTTP（Streamable HTTP / SSE）へ拡張され、リモートサーバやマルチテナントでの利用が現実的になった。月間 SDK DL 9,700万（要確認）という規模は、TypeScript / Python の公式 SDK が幅広いエコシステムに取り込まれた結果である。

### 仕組み / 背景（なぜこう設計されたか）
従来は N 個の AI モデル × M 個のツールで N×M の個別統合が必要だった。MCP は間に共通インターフェースを挟むことで N+M に落とす。これは言語サーバの LSP が「エディタ×言語」の組合せ爆発を解いた発想の再来である。プロトコルとして中立に保つことで、特定ベンダーのロックインを避け、エコシステム全体で採用が進みやすくなった。

### 実務での使いどころ・移行の勘所
自社ツールを AI から使わせたい場合、独自 Function Calling を書くより MCP サーバとして公開する方が将来性が高い。ただしリモート公開時は認証・認可・レート制限・監査ログが必須で、任意サーバを無条件に信頼するとプロンプトインジェクションや権限昇格のリスクがある。既存の Function Calling 資産は MCP ラッパーで包む移行パスが現実的。

## 📖 用語解説（このトピックとのつながり）
- **MCP（Model Context Protocol）**: AI と外部ツール/データを繋ぐオープン標準。本トピックの主役。
- **JSON-RPC**: MCP の下敷きとなる軽量なリモート手続き呼び出し方式。
- **SDK**: 開発キット。DL 数が採用度合いの代理指標になる。
- **トランスポート**: 通信路。stdio（ローカル）と HTTP（リモート）を選べる。
- **Function Calling**: モデルにツールを使わせる従来手法。MCP はこれを標準化・再利用可能にした位置づけ。

## 影響範囲 / 推奨アクション
AI エージェントや LLM 連携を扱う全開発者に影響する。新規のツール連携は独自実装より MCP サーバ化を第一候補にすべき。既に Function Calling を実装済みなら、MCP ラッパー化で他クライアントからの再利用性を高められる。リモート公開時は認証・監査を最初から設計に組み込むこと。
