# 偽PoCの依存パッケージにRATを仕込む「ChocoPoC」——脆弱性研究者を狙うサプライチェーン攻撃

> トピック: 脆弱性ニュース ｜ 出典: Sekoia / YesWeHack 共同レポート(7月1日) / The Hacker News / BleepingComputer ｜ 種別: ソフトウェアサプライチェーン汚染（悪意ある依存パッケージ）

## 一行サマリー

GitHub 上の偽の脆弱性実証コード（PoC）が、脆弱性研究者やペンテスターを狙って RAT（遠隔操作型マルウェア）「ChocoPoC」を配布していた。悪意はエクスプロイト本体ではなく `requirements.txt` の依存パッケージ（`frint`/`skytext`）に隠され、`pip install` した瞬間に感染。ブラウザの保存パスワード・Cookie を窃取し任意コマンドを実行される。サーバは現在も稼働中で、PoC を試す前の“出所確認”が必須。

## 🔰 初学者向け（何が・なぜ問題か）

セキュリティの世界では、新しい脆弱性が公開されると研究者たちが「本当に危険か」を確かめるため、PoC（Proof of Concept＝実証コード）という“実験用のサンプル攻撃コード”を GitHub などで探して手元で動かします。今回の攻撃は、この「みんなが急いで試したくなる心理」を逆手に取ったものです。

例えるなら、話題の新料理のレシピ（PoC）が配られているのですが、レシピ本体は本物なのに、「材料リスト」（依存パッケージ）にこっそり毒の材料が1つ混ぜてあった、という手口です。料理を作ろうと材料をそろえる（`pip install` で必要な部品を自動ダウンロードする）だけで、毒＝マルウェアが台所（PC）に入り込みます。コードを実行する前、準備段階でもう感染しているのが厄介な点です。

仕込まれた「ChocoPoC」は、いわば台所に潜り込んだ泥棒です。Chrome や Edge、Firefox に保存されたパスワード・Cookie・入力履歴を盗み、メモやファイル、コマンド履歴まで持ち出し、攻撃者が遠隔から好きな命令を実行できます。狙われたのがセキュリティ研究者という「攻撃技術や社内情報に詳しい人たち」である点も深刻で、盗まれた資格情報が次の侵入の踏み台になりえます。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **CWE**: CWE-506（埋め込み型の悪意コード）／CWE-1357・CWE-829（信頼できない依存物の取り込み）に相当するサプライチェーン汚染。
- **配布経路**: GitHub 上に少なくとも7つの偽 PoC リポジトリ。FortiWeb(CVE-2025-64446)、React2Shell(CVE-2025-55182)、MongoBleed(CVE-2025-14847)、PAN-OS(CVE-2026-0257)、Ivanti Sentry(CVE-2026-10520)、Check Point VPN(CVE-2026-50751)、Joomla SP Page Builder(CVE-2026-48908) などの“注目脆弱性”を餌にしていた（各CVEの真偽・詳細は要確認）。
- **悪性コンポーネント**: 悪意は PoC 本体ではなく `requirements.txt` に追加された悪性 PyPI 系パッケージ（`frint`／`skytext`）。`skytext` は約2,400回ダウンロードされ、大半が Linux 環境。
- **ペイロード**: Python 製 RAT「ChocoPoC」。ネイティブ拡張として読み込まれる形をとる。
- **前提条件**: 被害者が偽リポジトリを clone し、`pip install -r requirements.txt` 等で依存を解決すること。管理者権限は不要（ユーザ権限で成立）。
- **稼働状況**: Sekoia/YesWeHack が7月1日に共同公表。レポート時点でマルウェアと C2 サーバは稼働中。2025年と2026年の2波が確認され、単一の攻撃者による高確度の関与と評価。C2/PyPI/Mapbox アカウントを使い回しでローテーションして検知回避。

### メカニズム（攻撃が成立する仕組み）

本件は「実行フェーズではなく“環境構築フェーズ”に信頼の穴を作る」点が肝で、コードレビューの盲点を突く。

1. **信頼の錯覚（正規に見えるPoC）**: 攻撃者は本物らしい PoC リポジトリを用意し、注目 CVE の公開直後に露出させる。研究者は「脆弱性を早く検証したい」動機からリポジトリを信頼し、中身のレビューを PoC 本体（`.py` のエクスプロイト部分）に集中させがち。

2. **攻撃面の移動（本体→依存物）**: 悪意はエクスプロイト本体に埋めず、`requirements.txt` の依存リストに 1 行足すだけ。ここは目視レビューの優先度が低く、パッケージ名も正規物に似せてある（typosquatting/名前の紛れ込み）。これにより“実行前の準備”そのものを攻撃トリガーに変える。

3. **獲得プリミティブ＝任意コード実行（インストール時）**: `pip install` はパッケージのビルド/ロード時にコードを走らせうる。コンパイル済みネイティブ Python 拡張として読み込ませることで、依存解決の瞬間に攻撃者コードが実行される。ここで“被害者ユーザ権限での任意コード実行”というプリミティブが確立する。

4. **RAT展開と目的達成**: 実行後 ChocoPoC は常駐 RAT として、Chromium系/Firefox の資格情報・Cookie・オートフィル・履歴、ローカルDBやシェル履歴、ネットワーク設定、プロセス一覧を収集。攻撃者は任意シェル/任意 Python 実行、フォルダ丸ごと窃取が可能で、動作を“遅く”して静音化する回避機能も持つ。窃取した研究者の資格情報は次の標的への初期アクセスに転用されうる。

5. **クラスの一般原理**: 「信頼される配布経路（GitHub/PyPI）× 依存関係の自動解決 × レビューの盲点」を突くソフトウェアサプライチェーン攻撃の典型。単一の脆弱パッケージが多数の下流環境へ一斉に波及し、SBOM やロックファイル検証を欠く運用ほど被害が拡大する。

※具体的な悪性パッケージの導入手順・C2 プロトコル詳細等は記載しない（概念レベル）。

### 検知・暫定緩和・恒久対策

- **検知の着眼点（IoC/挙動）**: `requirements.txt` 等に含まれる不審パッケージ名（`frint`／`skytext` 等）の混入をCIやローカルで検査。`pip install` 直後に発生する予期しない子プロセス生成・外部通信（C2 ビーコン）・ブラウザプロファイル（`Login Data`/`Cookies`）への読み取りアクセスを監視。Linux 環境での想定外ネイティブ拡張ロードを相関。
- **暫定緩和**: PoC 検証は必ず使い捨ての隔離環境（VM/コンテナ、ネットワーク制限）で実施。`pip install` は仮想環境かつ最小権限で。未知リポジトリの依存は事前にハッシュ/出所を確認。
- **恒久対策**: 依存の固定（ロックファイル＋ハッシュ検証）、内部プロキシ/許可リストによるパッケージ取得の統制、SBOM 運用と依存監査、ブラウザ資格情報の保存最小化とMFA徹底、被害疑い時は保存パスワードの一斉ローテーション。
- **運用示唆**: 「注目CVE公開直後」は便乗攻撃の最盛期。PoC は“信頼できる出所か”を検証してから、隔離環境で。研究者・SOC 自身が高価値標的である前提で運用する。

## 📖 用語解説（このニュースとのつながり）

- **PoC（実証コード）**: 脆弱性が悪用可能かを示すサンプル攻撃コード。本件では“餌”として偽装された。
- **サプライチェーン攻撃**: 正規の配布・依存経路に悪意を混入させ下流へ波及させる手口。今回は依存パッケージ経由。
- **`requirements.txt` / 依存パッケージ**: Python が自動取得する部品リスト。レビューの盲点で、感染トリガーにされた。
- **RAT（遠隔操作型トロイの木馬）**: 攻撃者が遠隔で操作・情報窃取するマルウェア。ChocoPoC はブラウザ資格情報を狙う。
- **C2（Command and Control）**: マルウェアが指示を受ける司令サーバ。本件は稼働中でアカウントを使い回し回避。

## 影響範囲 / 推奨対応

GitHub 上の PoC を検証する脆弱性研究者・ペンテスター・SOC 要員が主な標的（Linux 環境が中心）。国内でも PoC 検証は日常的に行われるため他人事ではない。PoC は使い捨ての隔離環境でのみ実行し、依存パッケージの出所・ハッシュを検証すること。感染が疑われる場合はブラウザ保存資格情報の全ローテーションと C2 通信の遡及調査を行う。
