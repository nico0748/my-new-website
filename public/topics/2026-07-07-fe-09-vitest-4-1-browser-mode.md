# Vitest 4.1 — Browser Mode 安定化とビジュアル回帰、実ブラウザで書くフロントテスト

> トピック: Vitest 4.0（Browser Mode 安定化）/ 4.1｜出典: Vitest 公式ブログ（VoidZero）｜種別: メジャー／マイナーリリース

## 一行サマリー

Vite ネイティブのテストランナー Vitest が 4.0 で Browser Mode を安定版に昇格し、JSDOM 等の擬似 DOM ではなく実ブラウザで UI コンポーネントを検証できるようになった。ビジュアル回帰テストや `toBeInViewport` マッチャも追加。2026年3月12日の 4.1 では VSCode 拡張の常駐プロセス削減や GitHub Actions のジョブサマリー自動生成など DX を強化。

## 🔰 初学者向け（何が・なぜ重要か）

テストとは「コードが期待どおり動くかを自動で確かめる仕組み」です。フロントエンドでは「ボタンを押したら数字が増える」などを機械的にチェックします。Vitest はその実行役で、ビルドツール Vite とそのまま噛み合うので設定が軽く、動作も速いのが人気の理由です。

これまで多くのテストは、本物のブラウザではなく **JSDOM という「ブラウザのふりをする擬似環境」** の中で動いていました。速い代わりに、本物のブラウザとは微妙に挙動が違い「テストは通ったのに実機で崩れる」ことがありました。4.0 で正式版になった **Browser Mode** は、テストを本物のブラウザの中で走らせます。つまり「見た目や操作が実際のユーザーと同じ条件で確認できる」わけです。さらに「画面の見た目が前回と変わっていないか」を画像で比べるビジュアル回帰テストも入り、意図しないデザイン崩れを検知できます。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **Browser Mode が安定版に**（4.0）。Playwright/WebDriver 等のプロバイダ経由で実ブラウザ上でテストを実行。
- **ビジュアル回帰テスト**を Browser Mode でサポート。スクリーンショット比較で見た目のデグレを検出。
- **`toBeInViewport` マッチャ**追加。IntersectionObserver を用い、要素が実際にビューポート内にあるかを検証。
- 4.1（2026年3月12日）: VSCode 拡張が明示的に continuous run を有効化しない限りバックグラウンドプロセスを常駐させず、メモリ削減。`github-actions` レポーターがジョブサマリーを自動生成。`doMock` から disposable を返せる、chai スタイルアサーション、タイマ制御更新、matcher 型の公開など。
- Storybook 10 の新しいモジュールモック（`sb.mock`）は Vitest チームとの協業成果で、4 系と親和。

### 仕組み / 背景（なぜこう設計されたか）

従来のフロントテストは「速さ」を優先して JSDOM で DOM を模倣してきた。だが JSDOM はレイアウト・描画・多くのブラウザ API を完全再現しないため、`getBoundingClientRect` が実質ゼロ、CSS レイアウトが評価されない、IntersectionObserver/ResizeObserver 等が不完全、といった乖離が「テスト緑・本番赤」を生む温床だった。Browser Mode はテストを実ブラウザプロセスへ委譲し、Vite の変換パイプラインをそのまま使ってテストコードを配信することで、**開発時と同じレンダリング条件**を再現する。これにより `toBeInViewport` のような「実際の可視性」に依存するアサーションや、ピクセル単位のビジュアル回帰が成立する。

トレードオフは、実ブラウザ起動のコスト（起動時間・CI リソース・並列度管理）と、環境差（フォント・OS レンダリング差）によるスクリーンショット比較の揺らぎ。Vitest は unit（Node/JSDOM）と browser を workspace で使い分け、速い層と忠実な層を両立させる設計思想を採る。4.1 の常駐プロセス削減や CI サマリーは、この二層構成を日常運用に載せるための地味だが効く整備である。

### 実務での使いどころ・移行の勘所

- レイアウト・可視性・実 DOM API に依存するコンポーネントテストは Browser Mode へ寄せ、純粋ロジックは従来どおり Node/JSDOM で高速に回す二層戦略が有効。
- ビジュアル回帰はデザインシステム／共通 UI コンポーネントの守りに有効。ただしフォント・レンダリング差でフレーキーになりやすいため、CI 環境を固定しベースライン管理を徹底。
- Jest からの移行は watch モードの高速性（実測で数倍）とゼロコンフィグ寄りの設定が動機。既存 Jest 資産は互換 API で段階移行可能。
- Storybook 10 や Nuxt Test Utils v4 は Vitest 4 系を要求・前提化しており、ツールチェーン全体を 4 系へ揃えると噛み合わせが良い。

## 📖 用語解説（このトピックとのつながり）

- **Browser Mode**: テストを実ブラウザで実行する Vitest の機能。4.0 で安定化し、本トピックの中心。
- **JSDOM**: Node 上でブラウザ DOM を模倣する擬似環境。速いが実機との乖離があり、Browser Mode がその弱点を補う。
- **ビジュアル回帰テスト**: 画面の見た目をスクリーンショットで比較し変化を検知する手法。Browser Mode 上で提供。
- **`toBeInViewport`**: 要素がビューポート内にあるか検証するマッチャ。IntersectionObserver を用い、実描画前提で成立する。

## 影響範囲 / 推奨アクション

- フロントチーム: 可視性・レイアウト依存のテストを Browser Mode へ移行し、ビジュアル回帰を主要 UI に導入。CI 環境の固定を先に整える。
- 既存 Vitest 利用者: 4.1 へ更新し、VSCode 拡張のメモリ改善と CI ジョブサマリーを享受。
- Storybook/Nuxt 利用者: Vitest 4 系へ揃える。
