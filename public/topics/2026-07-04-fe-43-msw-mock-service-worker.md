# MSW（Mock Service Worker）— API モックのテスト標準

> トピック: 開発トピック｜出典: thetshaped.dev｜種別: frontend

## 一行サマリー
MSW はネットワーク層でリクエストを横取りして偽のレスポンスを返すモック手法で、テスト・開発・Storybook を横断する API モックの標準として2026年のフロントエンドスタックに組み込まれている。

## 🔰 初学者向け（何が・なぜ重要か）
フロントエンドをテストするとき、本物のサーバーに繋ぐと「サーバーが落ちている」「データが毎回変わる」「遅い」といった問題でテストが不安定になります。そこで「偽のサーバー役」を用意し、決まった答えを返させるのが「モック」です。

従来のモックは、コード内の関数を差し替える方式が多く、「実際の通信の流れ」とはズレていました。MSW は発想が違い、ブラウザの Service Worker（通信を横取りできる仕組み）やサーバー側のインターセプタを使って、ネットワークそのもののレベルで偽の応答を返します。アプリ側のコードは「本物のサーバーと話しているつもり」のまま動くのがポイントです。

これは「配達員を差し替える」のではなく「郵便局で手紙をすり替える」ようなものです。アプリのコードを一切いじらずに応答を制御できるため、より本番に近い状態でテストできます。同じモック定義をテスト・ブラウザ開発・Storybook で使い回せるのも大きな利点です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
MSW はブラウザでは Service Worker、Node/Vitest 環境では独自インターセプタでリクエストをフックする。REST と GraphQL の両方をハンドラで定義でき、`http.get()` / `graphql.query()` のような宣言的 API で応答を記述する。バージョンや API 詳細は要確認だが、Vitest・Playwright・Storybook との統合が定番構成。

### 仕組み / 背景（なぜこう設計されたか）
関数モックはアプリ実装に密結合し、リファクタで壊れやすい。MSW はネットワーク境界でモックすることで、実装詳細から独立し、実通信に近い挙動を再現できる。「モックを実装に依存させない」という設計思想が採用理由。

### 実務での使いどころ・移行の勘所
単体〜統合テスト、バックエンド未完成時の並行開発、Storybook でのデータ状態再現に有効。導入時は Service Worker ファイルの配置と、テスト環境でのサーバーセットアップ（`setupServer`）が必要。過剰なモックは本番との乖離を生むため、契約テストや E2E と役割分担すること。

## 📖 用語解説（このトピックとのつながり）
- **Service Worker**: ブラウザで通信を仲介できる常駐スクリプト。MSW のブラウザ側の要。
- **インターセプタ**: リクエストを横取りして加工・応答する仕組み。MSW の核心。
- **ハンドラ**: どのリクエストにどう応答するかの定義。REST/GraphQL 両対応。
- **契約テスト**: API の仕様一致を検証する手法。MSW モックと補完関係にある。

## 影響範囲 / 推奨アクション
テストや Storybook を運用する全チームに有効。API モックを関数差し替えで行っている場合、ネットワーク層モックへの移行で保守性が向上する。まずテスト環境から導入するとよい。
