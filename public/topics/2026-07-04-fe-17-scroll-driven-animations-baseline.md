# スクロール駆動アニメーションが Baseline 入り ― animation-timeline が prefix 不要に

> トピック: 開発トピック｜出典: modern-css｜種別: frontend

## 一行サマリー
スクロール量や要素の可視位置に連動してアニメーションを進める **スクロール駆動アニメーション（`animation-timeline` / `scroll()` / `view()`）**が Baseline に到達（時期は要確認）。ベンダプレフィクスなしで、JS のスクロールイベント監視なしにパララックスや進捗表現を実装できる。

## 🔰 初学者向け（何が・なぜ重要か）
「スクロールするとヘッダーの下に読書の進捗バーが伸びる」「下にスクロールすると画像がフワッと現れる」——こうした演出はサイトで頻繁に見かけます。従来これらは、JavaScript でスクロール位置を絶えず監視し、その値からアニメの進み具合を手計算するのが定番でした。しかしスクロールは非常に高頻度で発生するため、計算が重いとカクつき（ジャンク）の原因になります。

スクロール駆動アニメーションは、この「スクロール量」をアニメの時計（タイムライン）として CSS に直接与えられる仕組みです。「時間ではなくスクロール量で 0% → 100% を進める」と宣言するだけ。ブラウザが最適化して滑らかに描画してくれるので、JS の監視コードが不要になり、カクつきも減ります。時間で動くアニメを、スクロールで動くアニメに置き換えるイメージです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
- `animation-timeline` に `scroll()`（スクロールコンテナの進捗）や `view()`（要素のビューポート内可視進捗）を指定可能。
- 名前付きタイムライン（`scroll-timeline-name` / `view-timeline-name`）で任意要素のスクロールに紐づけられる。
- 従来必要だったベンダプレフィクスや実験フラグが不要に（Baseline 化、要確認）。

### 仕組み / 背景（なぜこう設計されたか）
通常の CSS アニメーションは「時間」を進行軸にしますが、この機能は進行軸を「スクロール位置」や「要素の可視率」に差し替えます。ブラウザはこれをコンポジタスレッド（描画専用スレッド）で処理できるため、メインスレッドの JS がブロックされていても滑らかに動きます。従来の JS スクロールイベントはメインスレッドで動き、重い処理と競合してジャンクを生んでいました。宣言的にブラウザへ委ねることでパフォーマンスと保守性を両立する設計です。

### 実務での使いどころ・移行の勘所
- スクロール進捗バー、パララックス、フェードイン、画像シーケンスなどに最適。
- 既存の JS（Intersection Observer やスクロールリスナ）ベースの演出を CSS へ置換でき、依存を削減。
- 過度な演出はアクセシビリティを損なうため、`prefers-reduced-motion` での無効化を併用すること。

## 📖 用語解説（このトピックとのつながり）
- **animation-timeline**: アニメの進行軸を指定するプロパティ。スクロール駆動化の入口。
- **scroll() / view()**: それぞれスクロール進捗、要素の可視進捗をタイムライン化する関数。
- **コンポジタスレッド**: 描画専用スレッド。ここで動くため滑らか。
- **Intersection Observer**: 従来この用途で使われた JS API。置換対象。
- **prefers-reduced-motion**: 動きを抑えたいユーザ向けの設定。演出の必須配慮。

## 影響範囲 / 推奨アクション
演出重視のサイトを作る開発者に影響大。新規のスクロール演出は CSS のスクロール駆動アニメを第一候補に。既存の JS スクロール演出は置換でパフォーマンス改善が見込める。導入時は `prefers-reduced-motion` 対応を忘れずに。
