# Drizzle ORM v1.0（Beta/RC）— Relational Queries v2 と defineRelations() で型安全 ORM が大台へ

> トピック: Drizzle ORM v1.0 Beta/RC｜出典: Drizzle 公式ドキュメント｜種別: メジャーバージョン Beta/RC

## 一行サマリー

TypeScript の型安全 ORM である Drizzle が v1.0 へ向けた Beta/RC を重ねている（2026年4月時点で 1.0.0-beta.22、以後 RC へ）。旧 Relational Queries（RQBv1）を廃し、`defineRelations()` を用いる Relational Queries v2 に刷新。パッケージ構成を整理し、`drizzle-kit push/pull` が既定で全スキーマを扱うようになるなど、1.0 に向けた API 固めが進む。本番向けの安定版は現状 0.45.2。

## 🔰 初学者向け（何が・なぜ重要か）

ORM とは「データベース（表形式のデータ倉庫）を、プログラムの言葉で扱えるようにする翻訳層」です。SQL を直接書く代わりに、コードでテーブルや検索条件を組み立てられます。Drizzle はその中でも「TypeScript の型と密着していて、SQL に近い薄さ」で人気です。存在しない列を書いたら即エラー、返ってくるデータの形も型でわかる——「間違いを実行前に潰せる」のが売りです。

v1.0 の焦点は、**テーブル同士の関係（リレーション）の書き方の作り直し**です。「ユーザーには複数の投稿がある」「投稿には1人の著者がいる」といった関係を、新しい `defineRelations()` という一箇所の定義にまとめる方式（Relational Queries v2）に変わりました。関係の宣言が整理され、関連データをまとめて取ってくる書き方がより型安全・一貫的になります。バージョン 1.0 という節目は「API がだいたい固まり、長く使える形になった」という区切りの合図です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **Relational Queries v2**: 旧 RQBv1 を削除し、`defineRelations()` API で関係を定義する方式へ刷新。関連データ取得（ネストした `with`）の型と挙動を再設計。
- **パッケージレイアウトの整理**: 1.0 は package 構成を整理する前向きなリリースと位置付け。
- **drizzle-kit の既定変更**: `drizzle-kit push` / `pull` が既定で public 以外も含む全スキーマを管理。
- **エコシステム連携**: `1.0.0-rc.1` は Effect v4 のネイティブサポートを同梱し、RC 段階入りを示唆。
- 2026年4月16日時点で `1.0.0-beta.22`。本番の保守的選択は現状 `0.45.2`（1.0 安定版が出るまで）。

### 仕組み / 背景（なぜこう設計されたか）

Drizzle の設計思想は「ORM の魔法を薄くし、SQL に近い明示性と TypeScript の型を両立する」こと。だが初代 Relational Queries（RQBv1）は、関係定義がテーブル定義に散在しがちで、複雑なネスト取得の型推論とクエリ生成の一貫性に限界があった。v2 は関係を `defineRelations()` に集約することで、(1) スキーマとリレーションの関心を分離し、(2) 関連取得時の型を単一の真実源から導出できるようにする。これは Prisma のスキーマ中心モデルの明示性と、Drizzle の「SQL に近い薄さ」の折衷を狙った再設計といえる。

トレードオフは、1.0 が破壊的変更を含み、RQBv1 依存コードの移行が要る点。だからこそ 0.45.x を安定版として残し、1.0 は beta/rc を長く回して現場のフィードバックで API を固める、という慎重な出し方をしている。`drizzle-kit` が既定で全スキーマを対象化する変更は、マルチスキーマ運用（テナント分割・拡張スキーマ）を素直に扱えるようにする一方、既存の public 前提のマイグレーション運用には影響が出るため、移行時の確認事項になる。Effect v4 ネイティブ対応は、エフェクトシステムで副作用（DB アクセス）を型で管理する層との統合需要に応えるもの。

### 実務での使いどころ・移行の勘所

- 型安全・軽量・SQL 近接を重視する TypeScript バックエンド（エッジ／サーバーレス含む）で、Prisma の重さを避けたいケースの第一候補。
- 本番は当面 0.45.x が無難。1.0 は検証環境で `defineRelations()` への移行コストと `drizzle-kit` の全スキーマ既定化の影響を評価してから。
- マルチスキーマ運用のプロジェクトは `drizzle-kit push/pull` の既定変更を要確認（意図しないスキーマを触らないようフィルタ設定）。
- Prisma との比較軸は「スキーマ言語＋生成クライアントの手厚さ vs SQL 近接＋薄い型安全」。Effect ベースの構成なら Drizzle 1.0 の Effect v4 対応が効く。

## 📖 用語解説（このトピックとのつながり）

- **ORM**: DB をプログラムの型・オブジェクトで扱う層。Drizzle は薄く型安全な立ち位置。
- **Relational Queries（RQB）**: 関連データをネストして取得する Drizzle の機能。v2 で `defineRelations()` 方式に刷新された。
- **defineRelations()**: 1.0 の中核 API。テーブル間の関係を一箇所に定義し、型と取得挙動の一貫性を担保。
- **drizzle-kit**: マイグレーション／スキーマ同期の CLI。1.0 で既定の対象スキーマ範囲が変わる。

## 影響範囲 / 推奨アクション

- 既存 Drizzle 利用: 本番は 0.45.x 維持。1.0 の `defineRelations()` 移行を検証環境で試算。
- マルチスキーマ運用: `drizzle-kit` 既定変更の影響を確認し、フィルタを明示。
- 新規: 型安全・軽量志向なら評価対象に。安定版 1.0 の GA を注視。
