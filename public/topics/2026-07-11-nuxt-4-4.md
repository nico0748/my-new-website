# Nuxt 4.4 — カスタム useFetch ファクトリと vue-router v5

> トピック: Nuxt 4.4 リリース ｜ 出典: Nuxt 公式ブログ ｜ 種別: マイナーバージョンアップ

## 一行サマリー

Vue のメタフレームワーク Nuxt が 4.4 を公開し、自前の `useFetch`/`useAsyncData` を量産できるファクトリ関数、内部ルーターの vue-router v5 化、アクセシビリティ告知機能、ビルドプロファイリングなどデータ取得と開発体験まわりを底上げした。

## 🔰 初学者向け（何が・なぜ重要か）

Nuxt は「Vue で Web サイトを作るときの土台一式」を用意してくれるフレームワークだ。ページのルーティング、サーバー側での事前レンダリング（SSR）、データの取り寄せなどを、いちいち自分で配線しなくても済むようにしてくれる。Next.js の Vue 版、と考えると近い。

今回の 4.4 は「大きな作り替え」ではなく「日々の使い勝手の改善」がテーマだ。目玉のひとつが **カスタム `useFetch` ファクトリ**。`useFetch` は「サーバーからデータを取ってきて、画面に出す」ための定番の道具だが、実プロジェクトでは「毎回このヘッダを付けたい」「このベース URL を使いたい」といった共通の決まりごとがある。従来はページごとにその設定を書き写していた。ファクトリを使うと、その決まりごとを一度だけ設定した"自分専用の useFetch"を作れる。料理でいえば、毎回だしを取り直すのではなく「自家製のだしパック」を作り置きするようなものだ。

もうひとつ地味だが大切なのが **アクセシビリティ告知（announcer）**。SPA（画面を切り替えても再読み込みしないサイト）はページ遷移が視覚的にしか起きないため、目の見えないユーザーがスクリーンリーダーで使うと「ページが変わったこと」に気づけないことがある。4.4 はページが切り替わったことを支援技術に読み上げさせる仕組みを標準で持つようになった。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

Nuxt 4.4（2026-03-12）の主な変更点は次のとおり。

- **カスタム `useFetch` / `useAsyncData` ファクトリ**: 既定オプション（baseURL、ヘッダ、`transform`、キャッシュ挙動など）を束ねた専用コンポーザブルを生成できる。API クライアントごとに `useApiFetch` のような関数を定義し、アプリ全体で一貫した取得ロジックを共有できる。
- **vue-router v5 への更新**: 内部ルーターを最新世代に更新。ルーティングの型付けとパフォーマンスが改善。
- **アクセシビリティ告知機能（route announcer）**: ルート遷移時に `aria-live` 領域へページタイトルを流し込み、スクリーンリーダーに遷移を伝える。
- **typed layout props**: レイアウトに渡すプロパティが型付けされ、レイアウト間の受け渡しが型安全に。
- **build profiling**: ビルド処理のボトルネックを可視化するプロファイリング。
- **smarter payload handling**: SSR で受け渡すペイロード（サーバーが計算した状態）の扱いを最適化。

これらは Nuxt 4 系（2025年7月に安定化、`app/` ディレクトリ構造への移行が目玉）の路線上にある継続的改善で、破壊的変更を最小化しつつ実務の摩擦を減らす方向。

### 仕組み / 背景（なぜこう設計されたか）

**なぜファクトリなのか**。`useFetch` は SSR とクライアントの両方で動く「アイソモーフィック」なデータ取得を担う。サーバーで取得した結果をペイロードに載せてクライアントへ渡し、ハイドレーション時に再フェッチを避ける——この一連の流れをフレームワークが面倒みてくれるのが価値だ。しかし共通設定の重複は DRY 原則に反し、認証ヘッダの付け忘れなどのバグ源になる。ファクトリは「設定済みのクロージャを生成する」古典的パターンで、共通軸（baseURL・認証・変換）を一箇所に閉じ込める。TanStack Query の `queryOptions` ヘルパや、axios のインスタンス生成と同じ発想だ。トレードオフとして、抽象を一枚増やすため「どの useFetch がどの設定を持つか」が暗黙化しやすく、命名規約とドキュメント整備が要る。

**なぜ告知機能が標準化されたか**。SSR/SPA ハイブリッドのフレームワークでは、クライアント側ナビゲーションが `history.pushState` で行われ、DOM のごく一部だけが差し替わる。ブラウザのフルロードが起きないため、支援技術が「新しいページ」と認識できない。`aria-live="polite"` な領域にタイトルを書き込むことで、視覚に頼らないユーザーにも遷移を伝える。アクセシビリティを「後付け」ではなくフレームワーク既定に組み込む潮流の一例。

### 実務での使いどころ・移行の勘所

複数の外部 API を叩く中〜大規模 Nuxt アプリでは、カスタムファクトリで API クライアント層を整理すると保守性が上がる。既存の 4.x からのアップグレードは基本的に非破壊だが、vue-router v5 化に伴う型やナビゲーションガードの挙動差は変更履歴で確認したい。告知機能はデフォルトで有効化される場合があるため、既存の独自アクセシビリティ実装と二重にならないか点検を。Nuxt 3 系からの移行は `app/` ディレクトリ構造という大きめの変更を伴うため、まず 4.0 の移行ガイドを通してから 4.4 に上げるのが安全。

## 📖 用語解説（このトピックとのつながり）

- **メタフレームワーク**: ライブラリ（Vue/React）の上に、ルーティング・SSR・ビルド設定を統合した上位の枠組み。Nuxt は Vue の、Next.js は React のメタフレームワーク。本トピックの主役。
- **useFetch / useAsyncData**: Nuxt のデータ取得コンポーザブル。SSR とクライアントで一貫して動く。今回ファクトリ化の対象。
- **コンポーザブル**: Vue Composition API で状態やロジックを関数として再利用する単位（`useXxx`）。ファクトリはコンポーザブルを生成する関数。
- **ハイドレーション**: サーバーが吐いた静的 HTML に、クライアントで JS を"注いで"操作可能にする処理。ペイロード最適化はこの効率化に効く。
- **aria-live**: 内容が動的に変わる領域を支援技術に通知する ARIA 属性。ルート告知の基盤。

## 影響範囲 / 推奨アクション

- Nuxt 4.x 利用者は非破壊アップグレードとして取り込み可。データ取得ロジックが散らばっているならカスタムファクトリへの集約を検討。
- アクセシビリティ要件のあるプロジェクトは、標準告知機能を有効化し独自実装と重複しないか確認。
- Nuxt 3 からの移行組は、まず 4.0 移行ガイド（`app/` 構造）を経てから 4.4 へ。
