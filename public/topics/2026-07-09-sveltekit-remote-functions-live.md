# SvelteKit — リモート関数に .live() が登場、サーバーのリアルタイムデータ取得が簡単に

> トピック: SvelteKit リモート関数の進化（.live / フォーム強化）｜出典: Svelte 公式ブログ｜種別: フレームワーク更新

## 一行サマリー

SvelteKit は「リモート関数」を継続強化し、6月にサーバーのリアルタイムデータへ簡単にアクセスできる新 query 関数 `.live(...)` を追加、7月には `form.error` の型を `any` から `App.Error | undefined` に厳格化し、アダプタが追加 Vite プラグインを提供可能にするなどの破壊的変更を含む更新を行った。

## 🔰 初学者向け（何が・なぜ重要か）

Web アプリは「ブラウザ側（フロント）」と「サーバー側（バック）」に分かれています。この2つのやり取り（サーバーからデータを取ってくる、フォームを送信する）を書くのは、実は初心者がつまずきやすい部分です。URL を決めて、通信の口（API）を作って、型を合わせて…と手間が多いからです。

SvelteKit の **リモート関数** は、この面倒を大きく減らす仕組みです。サーバー側の関数を、まるで手元の関数を呼ぶかのようにフロントから呼び出せます。裏側の通信は SvelteKit が自動で面倒を見てくれます。

今回の目玉 `.live()` は、そこに「リアルタイム」を足すもの。たとえばチャットや在庫数のように「サーバー側で値が変わったら自動で画面も更新してほしい」場面で、複雑な通信コードを書かずに済むようになります。加えてフォーム周りの型が厳しくなり、エラーの扱いがより安全になりました（＝バグに早く気づける）。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

Svelte/SvelteKit の 2026 年の主な進展（公式ブログの月次まとめより）：

- **5月**: SvelteKit のリモート関数改善、TypeScript 6.0 対応、Svelte CLI のコミュニティプラグインを実験公開。
- **6月**: フォームとリモート関数の多数の改善、リアルタイムデータ取得を容易にする新 query 関数 `.live(...)` を追加。
- **7月（〜7/6 時点）**: 破壊的変更を含む更新。`form.error` の型を `any` → `App.Error | undefined` に変更、アダプタが追加の Vite プラグインを提供可能に、トレーシングを experimental 名前空間から昇格。ほか `$app/stores`、リモート関数のファイル規則、param ファイルに関する変更、query refresh・環境変数・Service Worker・Vitest・アイコン等の修正。

（各バージョン番号の対応は GitHub の kit リリースで要確認。）

### 仕組み / 背景（なぜこう設計されたか）

リモート関数は「型付き RPC を Svelte のリアクティビティに統合する」試みだ。従来の SvelteKit では `+page.server.ts` の load 関数や `form actions`、あるいは手書きの `+server.ts` エンドポイントでフロント↔サーバーを繋いでいた。これは強力だが、データの読み・書き・リアルタイム更新でそれぞれ別の書き方が必要で、型の一貫性も自分で担保する必要があった。リモート関数はこれを `query`（読み）・`command`/`form`（書き）などの単位に整理し、クライアントからは通常の非同期呼び出しに見せつつ、内部でシリアライズ・転送・型共有を自動化する。

`.live()` はこの `query` に「サーバー主導の更新」を加える。通常の query が「呼んだ時点のスナップショット」なのに対し、live query はサーバー側データソースの変化を購読し、変化を検知したらクライアントのリアクティブな値へプッシュする。実装的にはストリーミング（SSE/WS 相当）の上に、Svelte の runes（`$state`/派生値）と結線される。トレードオフは、常時接続・購読管理のコスト（コネクション数、スケール時の負荷）と、破壊的変更に伴う既存コードの追随。型厳格化（`form.error`）は「実行時にしか分からなかったエラー形状をコンパイル時に固定する」方向で、安全性を上げる代わりに型注釈の修正を強いる。

### 実務での使いどころ・移行の勘所

チャット、通知、ダッシュボード、在庫・価格表示など「サーバー側の変化を即時反映したい」機能で `.live()` が刺さる。一方、静的〜低頻度更新のページで安易に live 化すると接続コストが無駄になるため、更新頻度で使い分ける。移行では 7 月の破壊的変更（`form.error` 型・トレーシング名前空間・`$app/stores`・リモート関数のファイル規則）に注意し、まずは型エラーを潰してから機能追加を進めたい。リモート関数はまだ発展途上の API のため、公式の安定度表記（experimental か否か）を都度確認するのが安全。

## 📖 用語解説（このトピックとのつながり）

- **リモート関数（remote functions）**: サーバー関数をフロントから型付きで呼べる SvelteKit の仕組み。本トピックの土台。
- **query / command / form**: 読み・書き・フォーム送信に対応するリモート関数の種別。`.live()` は query の拡張。
- **runes**: Svelte 5 の新リアクティビティ（`$state` 等）。live query の値と結線される。
- **SSE / WebSocket**: サーバー主導の更新を運ぶ通信方式。`.live()` の裏側で使われる。
- **App.Error**: SvelteKit のアプリ定義エラー型。`form.error` がこの型に厳格化された。

## 影響範囲 / 推奨アクション

- 影響: SvelteKit を使うアプリ。リアルタイム機能の実装コスト減と、7月破壊的変更への追随が必要。
- 推奨: リアルタイム要件のある画面で `.live()` を試す。更新頻度が低い箇所は通常 query のまま。アップデート時は `form.error` 型・トレーシング・ファイル規則の変更に対応。
