# OpenAI Agents SDK が MCP 複数トランスポート対応 — 既存サーバをそのまま再利用

> トピック: 開発トピック｜出典: OpenAI Agents（公式ドキュメント）｜種別: ai

## 一行サマリー
OpenAI Agents SDK が MCP の複数トランスポート（stdio / HTTP 系）に対応し、既存の MCP サーバをコード変更なしにそのまま接続・再利用できるようになった。ベンダーの垣根を越えて MCP エコシステムを活用する動きが一段と加速している。

## 🔰 初学者向け（何が・なぜ重要か）
「共通規格に対応する」と言っても、対応の仕方には濃淡がある。コンセントの形だけ合わせても、電圧や周波数が違えば実際には使えない。逆に、ローカル接続でもネット越しでも同じ機器が繋がるなら、それは本当に「使える共通規格」だ。

OpenAI の Agents SDK が MCP の複数の通信方式に対応したことは、「MCP サーバとして作られた部品なら、OpenAI 製のエージェントからでも、繋ぎ方を問わずそのまま使える」ことを意味する。MCP は Anthropic 発の規格だが、競合の OpenAI もしっかり対応した。これは MCP が「一社のもの」ではなく「業界共通の土台」になったことの実務的な裏付けだ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
OpenAI Agents SDK（Python）が MCP サーバを stdio と HTTP 系（Streamable HTTP / SSE 等、要確認）の複数トランスポートで接続できるようになった。これによりローカル実行のサーバもリモートホストのサーバも、同一の抽象で扱える。既存の MCP サーバ資産を書き換えずに OpenAI エージェントのツールとして組み込める。

### 仕組み / 背景（なぜこう設計されたか）
トランスポートを抽象化しておくと、開発時はローカル stdio、本番はリモート HTTP、といった使い分けが同じコードで可能になる。ベンダーが自社 SDK に MCP を取り込むほど、サーバ提供側は「一度作れば多クライアントで使える」利点を得る。相互運用が進むことで MCP のネットワーク効果がさらに強まる。

### 実務での使いどころ・移行の勘所
既に MCP サーバを持っているなら、OpenAI エージェントからの再利用が容易になり、マルチベンダー構成のハードルが下がる。ただしリモート接続時は認証・TLS・レート制限の扱いがトランスポートごとに異なる点に注意。開発（stdio）と本番（HTTP）で挙動差が出ないよう、両トランスポートでのテストを推奨。

## 📖 用語解説（このトピックとのつながり）
- **OpenAI Agents SDK**: OpenAI のエージェント構築キット。MCP を取り込んだ側。
- **トランスポート**: 通信路。stdio（ローカル）と HTTP 系（リモート）。
- **Streamable HTTP / SSE**: リモート MCP で使う HTTP ベースの通信方式（要確認）。
- **相互運用性**: 異なるベンダー間で部品を共有できる性質。本件の意義。
- **MCP サーバ資産**: 既存の MCP サーバ群。書き換えなしで再利用可能に。

## 影響範囲 / 推奨アクション
OpenAI と MCP を併用する開発者に直接影響する。既存 MCP サーバがあれば OpenAI エージェントへの組み込みを検討でき、ベンダー横断の構成が現実的になる。導入時はローカル/リモート両トランスポートでの動作確認と、リモート時の認証・TLS 設計を忘れないこと。
