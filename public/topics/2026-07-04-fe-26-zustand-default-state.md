# Zustand が既定のグローバル状態管理へ — 軽量・低ボイラープレートの定番化

> トピック: 開発トピック｜出典: OpenPR（State of React 2026）｜種別: frontend

## 一行サマリー
React のグローバル状態管理で、軽量ライブラリ Zustand が事実上の既定選択肢になりつつあり（採用率 35% 前後とされる。数値は要確認）、かつての Redux 的な冗長さを避けたい現場で広く採用されている。

## 🔰 初学者向け（何が・なぜ重要か）
React でアプリを作ると、「ログイン中のユーザー情報」や「テーマ設定」など、画面のあちこちから使いたいデータが出てきます。これらを各コンポーネントにバケツリレーで渡すのは大変なので、「グローバル状態管理」という共有の置き場を使います。

長年その定番は Redux でしたが、Redux は「アクション」「リデューサー」「ディスパッチ」といった専門用語と決まりごとが多く、小さな状態を一つ足すだけでもコードがかさむのが難点でした。初心者には特に敷居が高かったのです。

Zustand は、この置き場を「ただの箱」くらいシンプルにしたライブラリです。箱を一つ作り、その中に値と、それを変える関数を書くだけ。使う側は必要な値をつまむだけで済みます。学ぶことが少なく、書く量も少ない。だから「まず Zustand で十分では」という空気が広がり、既定の選択肢になってきた、というのが今回の話です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Zustand は `create` でストアを定義し、フック（`useStore`）経由で状態とアクションを購読する軽量ライブラリ。プロバイダで木全体をラップする必要がなく、セレクタで必要な部分だけ購読して再レンダリングを最小化できる。バンドルサイズが小さく（数 KB 級）、報告される「35% 採用」「35ms」等の数値は二次情報のため要確認。ミドルウェアで persist（永続化）・immer・devtools 連携も可能。

### 仕組み / 背景（なぜこう設計されたか）
Redux は「単方向データフローと予測可能性」を厳格に保証する代わりにボイラープレートが多く、Redux Toolkit で緩和されてもなお学習コストが残った。一方で React 自体が Context・`useReducer`・そしてサーバーコンポーネントやサーバー状態ライブラリ（TanStack Query 等）を得たことで、「クライアント側のちょっとしたグローバル状態」に求められる要件は軽くなった。Zustand は Provider 不要・購読ベースという設計でこの隙間に最適化し、必要十分な機能に絞ったことが普及要因。

### 実務での使いどころ・移行の勘所
用途は「UI 状態・モーダル開閉・テーマ・軽量なドメイン状態」など。サーバー由来データのキャッシュ・同期は TanStack Query 等に任せ、Zustand はクライアント状態に絞る役割分担が定石。Redux からの移行は、まず新規機能を Zustand で書き既存 Redux と併存させる漸進移行が現実的。ただし厳格な監査ログや複雑なミドルウェアパイプラインが必要な大規模ドメインでは Redux Toolkit の方が向く場合もあり、規模と要件で選ぶ。

## 📖 用語解説（このトピックとのつながり）
- **グローバル状態管理**: 複数コンポーネントで共有する状態の置き場。Zustand の守備範囲。
- **セレクタ**: ストアから必要な部分だけ取り出す関数。無駄な再レンダリングを防ぐ鍵。
- **ボイラープレート**: 定型的で冗長な決まり文句コード。Redux の課題、Zustand が削減。
- **サーバー状態**: API 由来のデータ。TanStack Query 等が担当し、Zustand と役割分担する。

## 影響範囲 / 推奨アクション
新規 React プロジェクトのチームに影響。クライアント状態は Zustand、サーバー状態は TanStack Query という役割分担を既定案として検討。Redux 資産は無理に一括移行せず、新規機能から漸進的に置き換えるのが安全。
