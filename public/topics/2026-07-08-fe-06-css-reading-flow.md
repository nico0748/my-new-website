# CSS `reading-flow` / `reading-order`：視覚順とフォーカス順を一致させるアクセシビリティ機能

> トピック: CSS `reading-flow` / `reading-order`｜出典: Chrome for Developers ブログ / MDN / chromestatus｜種別: CSS 新機能（アクセシビリティ・Chromium 先行）

## 一行サマリー

`reading-flow` は Flex / Grid / ブロックレイアウト内の子要素が「アクセシビリティツールに露出する順序」と「Tab キーでフォーカスされる順序」を制御する CSS プロパティ。`reading-order` は個別要素の順序を手動で上書きする。視覚上の並びと DOM 順がズレたときに、フォーカス順を視覚順に合わせられ、WCAG 2.2 の要件（フォーカス順序の意味的一致）に応えやすくなる。現状は Baseline 未達（Chromium 先行）。

## 🔰 初学者向け（何が・なぜ重要か）

CSS の Flexbox や Grid は、`order` プロパティや `flex-direction: row-reverse` などで「見た目の並び順」を DOM の記述順と関係なく変えられる。便利な反面、大きな落とし穴がある。**キーボードで Tab を押したときのフォーカス移動や、スクリーンリーダーが読み上げる順序は、あくまで DOM の順序に従う**という点だ。

つまり「見た目は左→右の順に並んでいるのに、Tab を押すと右→左に飛ぶ」といった、視覚順とフォーカス順の食い違いが起きる。目で見えている人には自然でも、キーボードだけで操作する人・スクリーンリーダー利用者には混乱の元になる。これはアクセシビリティ指針 WCAG の「フォーカス順序」に反する典型的な問題だった。

`reading-flow` は、この食い違いを CSS 側で直せるようにする。「見た目の並び（visual）に合わせてフォーカス順も並べ替える」と宣言できるので、`order` でレイアウトを組み替えても、フォーカスと読み上げが視覚順に追従する。`reading-order` を使えば、特定の要素だけ順番を微調整することもできる。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **`reading-flow`**: Flex / Grid / ブロックコンテナに指定し、子要素のアクセシビリティ露出順・シーケンシャルフォーカス順を制御。
  - Flex コンテナ: `flex-visual`（視覚順に従う）/ `flex-flow`（フローに従う）。
  - Grid コンテナ: `grid-rows` / `grid-columns` / `grid-order`。
- **`reading-order`**: reading-flow コンテナ内で、個別アイテムの順序を手動上書き（数値指定）。
- MDN 上は現状**実験的（experimental）**扱い、**Baseline 未達**（主要ブラウザの一部で未対応）。Chromium が先行実装（chromestatus / Intent to Ship 経由・出典で要確認）。

### 仕組み / 背景（なぜこう設計されたか）

問題の根は「CSS の視覚順序変更（`order` / `row-reverse` / Grid の自動配置）は DOM 順を変えない」という仕様上の分離にある。この分離自体は正しい。DOM 順はドキュメントの意味構造であり、レイアウトの都合で勝手に変えるべきではないからだ。しかし結果として、視覚順とフォーカス/読み上げ順が乖離し、アクセシビリティを損なう。

従来の回避策は「DOM の順序自体を視覚順に合わせて書き直す」か「JavaScript で `tabindex` を細かく制御する」だったが、前者は意味構造を歪め、後者は保守が重い。`reading-flow` は「意味構造としての DOM 順はそのまま、フォーカス/読み上げ順だけをレイアウトに合わせる」という第三の道を提供する。CSS で宣言的に解決するため、レイアウト変更に自動追従し、JavaScript の手続き的な `tabindex` 管理を不要にできる。

トレードオフは、まだ Baseline 未達で対応ブラウザが限られる点だ。未対応ブラウザでは従来どおり DOM 順でフォーカスするため、「reading-flow に依存しないと壊れる」設計は避け、DOM 順自体もできるだけ妥当に保つのが安全策になる。

### 実務での使いどころ・移行の勘所

- Grid / Flex で `order` を多用してビジュアルを組み替えているレイアウト（レスポンシブで並びが変わるダッシュボード等）で、フォーカス順の乖離を是正するのに有効。
- WCAG 2.2 の「意味のあるシーケンス / フォーカス順序」への対応手段として位置づけられる。
- Baseline 未達のため、未対応ブラウザでも致命的にならないよう DOM 順も妥当に設計し、reading-flow は「上乗せの改善」として使う。
- キーボード操作・スクリーンリーダーでの実機テストは必須（宣言だけで満足せず挙動を確認）。

## 📖 用語解説（このトピックとのつながり）

- **`order`（Flex/Grid）**: 視覚上の並び順を変える CSS プロパティ。DOM 順・フォーカス順は変えないため、reading-flow が必要になる原因。
- **シーケンシャルフォーカスナビゲーション**: Tab キーによる順送りのフォーカス移動。reading-flow が制御対象とする。
- **WCAG 2.2**: Web アクセシビリティ指針。フォーカス順序が意味的順序と一致することを求める。reading-flow はこれへの対応手段。
- **アクセシビリティツリー**: 支援技術に露出する要素の構造。reading-flow はこの露出順にも影響する。
- **Baseline**: 主要ブラウザでの安定利用指標。reading-flow は現状未達。

## 影響範囲 / 推奨アクション

- `order` を多用するレイアウトはフォーカス順の乖離を点検し、`reading-flow` で是正する。
- WCAG 2.2 準拠を目指すプロジェクトの対応候補に加える。
- Baseline 未達のため DOM 順も妥当に保ち、reading-flow は上乗せ改善として導入。
- キーボード / スクリーンリーダーで実機検証し、宣言だけに頼らない。
