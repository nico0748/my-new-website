# FastAPI が Pydantic v1 を完全終了 — Rust コアの v2 前提へ、Python API の地層が動く

> トピック: FastAPI の Pydantic v1 サポート終了／Pydantic v2 系の現在地｜出典: FastAPI／Pydantic 公式｜種別: 依存メジャー整理・エコシステム動向

## 一行サマリー

FastAPI 0.126.0（2025年12月）が Pydantic v1 サポートを完全に打ち切り、`pydantic >= 2.7.0` を最低要件とした。Pydantic は検証エンジンを Rust（pydantic-core）で実装した v2 系が本流で、2026年時点の安定版は v2.13 系。噂される v3 は「v2 のような大規模 API 刷新ではなく、壊れたエッジケースの修正」に留まる見込みで、当面の新機能は v2.x で出続ける。

## 🔰 初学者向け（何が・なぜ重要か）

Pydantic は Python で「入ってきたデータが正しい形か（この項目は数値、これは必須、など）を検証し、型付きのオブジェクトに変換する」ライブラリです。Web フレームワークの FastAPI は、リクエストの検証やレスポンスの整形に Pydantic を全面的に使っています。つまり Pydantic は FastAPI の土台の一部です。

かつての Pydantic v1 は Python で書かれていて、大量データの検証は遅めでした。v2 では**検証の中核を Rust（高速な言語）で作り直し（pydantic-core）**、桁違いに速くなりました。今回のニュースは、FastAPI が「もう古い v1 は面倒を見ません、v2 前提で行きます」と正式に線引きしたこと（0.126.0 で v1 サポート終了）。地味ですが、Python の Web バックエンドの"地層"が v2＝Rust コア前提に完全移行した、という節目です。まだ v1 に留まっているプロジェクトは、この機会に v2 へ上げる必要があります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **FastAPI 0.126.0（2025年12月）が Pydantic v1 サポートを完全終了**。最低要件 `pydantic >= 2.7.0`。
- **Pydantic v2 のアーキテクチャ**: 検証ロジックを Rust バイナリ **pydantic-core** に分離し、Python 側は薄いラッパ。バインディング経由で高速検証を呼ぶ。pydantic-core リポジトリは本体に統合済み。
- **現行安定版は v2.13 系**（2026年時点、例: 2.13.3）。新機能は v2.x で継続提供。
- **v3 の位置付け**: 大規模 API 刷新（v1→v2 のような）ではなく、v2 で「壊れている」と見なされるエッジケースの修正が主眼（GitHub issue #10033 での方針）。破壊は限定的の見込み（要確認）。

### 仕組み / 背景（なぜこう設計されたか）

v1→v2 の最大の設計転換は「検証エンジンの言語分離」。Python は柔軟だが CPU バウンドな逐次検証は遅く、大量の入力を型検証する API サーバーではボトルネックになりやすい。v2 は検証の熱い経路（フィールド走査・型強制・エラー収集）を Rust の pydantic-core に押し込み、Python 側はモデル定義とインターフェースに徹する。これにより検証は大幅高速化し、スキーマから JSON Schema/シリアライズまで一貫した高速パスを得た。トレードオフは、v1 の一部挙動（暗黙の型強制の緩さ、バリデータの書き方）と非互換になり、移行に手当てが要ること。FastAPI が v1 を切ったのは、この二重サポートの維持コスト（v1/v2 両対応の分岐）を断ち、v2 の性能・機能に一本化するため。

v3 を「破壊的刷新にしない」という宣言は、v2 移行で疲弊したエコシステムへの配慮であり、成熟したライブラリが取る安定化戦略でもある。大きな API 変更を避け、エッジケース修正と v2.x の継続進化に軸を置くことで、「アップグレード＝儀式」ではなく「日常の保守」に近づける。これは同時期の TypeScript（6.0→7.0 で挙動互換を守る）や React Router（future フラグで段階移行）と同じ、「破壊を最小化して信頼を積む」現代的な進化観の一例といえる。

### 実務での使いどころ・移行の勘所

- FastAPI を使う Python バックエンドは、まだ v1 依存なら **Pydantic v2 への移行が必須**。0.126 以降にアップデートするなら前提条件になる。
- 移行の主戦場は、v1→v2 の非互換（`@validator`→`@field_validator`/`@model_validator`、`Config`→`model_config`、暗黙型強制の厳格化など）。公式移行ガイドとコード変換ツール（bump-pydantic 等）で機械的に潰す。
- 性能面の恩恵は大きいが、v2 は型強制が厳格化する箇所があり「今まで通っていた緩い入力」が弾かれ得る。バリデーション挙動の差分をテストで確認。
- v3 を過度に待つ必要はない。新機能は v2.x で出続けるため、v2 系の最新に追随するのが実務的。

## 📖 用語解説（このトピックとのつながり）

- **Pydantic**: Python のデータ検証・変換ライブラリ。FastAPI の基盤で、v2 で Rust コア化された。
- **pydantic-core**: v2 の検証エンジン（Rust 実装）。高速化の核で、Python 側はこれを呼ぶ薄いラッパ。
- **型強制（coercion）**: 入力を宣言型へ変換する処理。v2 で厳格化され、v1 で通った緩い入力が弾かれ得る。
- **FastAPI 0.126.0**: Pydantic v1 サポートを終了した版。以降は v2 前提。

## 影響範囲 / 推奨アクション

- FastAPI + Pydantic v1 のプロジェクト: v2 へ移行してから 0.126 以降へ更新。バリデーション差分をテストで検証。
- ライブラリ作者: v1 対応コードを整理し v2 前提へ。
- v3 は破壊的でない見込み。v2.x 最新への追随を基本方針に。
