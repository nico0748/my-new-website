# OpenTelemetry が CNCF 卒業 — 観測性標準が AI 時代へ

> トピック: OpenTelemetry CNCF 卒業｜出典: The New Stack / CNCF｜種別: プロジェクト成熟イベント（2026年）

## 一行サマリー

クラウドネイティブの観測性（オブザーバビリティ）フレームワーク OpenTelemetry が CNCF を卒業（graduation）し、ログ・メトリクス・トレースの計装標準としての地位を確立。ClickHouse を高性能バックエンドに据える ClickStack や、Vercel AI SDK v7 の OTel ネイティブ連携など、AI エージェントの観測性へと適用領域を広げている。

## 🔰 初学者向け（何が・なぜ重要か）

システムが正しく動いているかを知るには、内部の様子を「見える化」する必要があります。これを観測性（オブザーバビリティ）と呼び、主に3種類のデータで支えます——ログ（出来事の記録）、メトリクス（数値の推移）、トレース（1つのリクエストがどの部品をどう通ったかの追跡）。問題は、これらを集める道具がベンダーごとにバラバラで、監視ツールを乗り換えるたびに計装（データを出す仕込み）をやり直す羽目になっていたこと。

OpenTelemetry（略して OTel）は、この計装のやり方を1つの共通規格に統一するプロジェクトです。今回 CNCF（クラウドネイティブ技術の中立的な母体）を「卒業」しました。卒業とは、CNCF の中で最も成熟・信頼された段階に達したという“お墨付き”で、Kubernetes などと同格の扱いです。つまり「もう業界の標準として安心して土台にできる」という節目です。

さらに近年は、AI エージェント（自律的に考えて動くプログラム）の中身を追跡する用途にも広がっています。AI が「なぜその判断をしたか」「どこで時間がかかったか」を見える化するのに、OTel の共通規格が使われ始めているのです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **CNCF 卒業**: OpenTelemetry がクラウドネイティブ計算で最も広く採用されるインフラの一つとして CNCF を卒業（Kubernetes 等と同じ最高成熟度段階）。
- **AI インフラ時代への適用**: ログ・メトリクス・トレースの計装標準として、AI エージェントの観測性領域へ拡大。
- **バックエンド連携**: ClickHouse を高性能な OTel バックエンドとして使う構成（ClickStack）が普及。OTel Collector を managed OTLP エンドポイントへ向けるだけでログ/メトリクス/トレースを探索できる。
- **フレームワーク連携**: Vercel AI SDK v7 が OTLP を通じてエージェント観測性を OTel ネイティブに統合（Langfuse 等経由）。

### 仕組み / 背景（なぜこう設計されたか）

観測性の根本課題は「計装（instrumentation）とバックエンド（保存・可視化）の密結合」だった。ベンダー独自エージェントで計装すると、監視 SaaS を替えるたびにアプリ側の仕込みを書き直すロックインが生じる。OpenTelemetry は、計装 API/SDK と OTLP（OpenTelemetry Protocol）という共通のデータ形式・伝送規約を定め、「アプリはOTelで一度計装し、送り先は自由に差し替える」という疎結合を実現する。これがベンダー中立の標準として広く採られ、卒業に至った。テレメトリを OTLP で吐ければ、バックエンドは Prometheus 系でも ClickHouse でも SaaS でも選べる——列指向で大量トレースを高速に扱える ClickHouse が OTel バックエンドとして伸びるのはこのためだ。AI エージェントは「複数ツール呼び出し・多段推論・非決定的な分岐」を含み従来より観測が難しいが、トレースの因果を OTel で標準化できれば、エージェントの実行過程を既存の観測基盤に載せられる。トレードオフは、標準の網羅性ゆえに計装の初期設定・サンプリング・コスト管理（大量トレースの保存費）に運用設計が要る点だ。

### 実務での使いどころ・移行の勘所

新規サービスは最初から OTel で計装しておくと、監視バックエンドの選択・乗り換えが自由になり、ベンダーロックインを避けられる。既存のベンダー独自計装があるシステムは、OTel Collector を挟んで段階的に OTLP へ寄せる移行が現実的。大量のトレースを扱う場合は、ClickHouse 系の列指向バックエンドや、サンプリング戦略でコストを制御する。AI エージェントを本番運用するチームは、エージェントの各ステップ（ツール呼び出し、推論、リトライ）を OTel トレースとして出すことで、レイテンシのボトルネックや失敗パターンを既存の観測ダッシュボードで追える。Vercel AI SDK v7 のような OTel ネイティブ連携を使うと計装の手間を減らせる。

## 📖 用語解説（このトピックとのつながり）

- **観測性（オブザーバビリティ）**: システム内部の状態を外部シグナルから把握する能力。OTel が計装を標準化。
- **ログ/メトリクス/トレース**: 観測性の3本柱。出来事の記録、数値推移、リクエスト経路の追跡。
- **OTLP（OpenTelemetry Protocol）**: OTel の共通データ形式・伝送規約。送り先バックエンドを差し替え可能にする。
- **OTel Collector**: テレメトリを受信・加工・転送する中継。バックエンド差し替えの要。
- **CNCF 卒業（graduation）**: CNCF プロジェクトの最高成熟度段階。標準としての信頼の証。

## 影響範囲 / 推奨アクション

- 対象: マイクロサービス/クラウドネイティブ運用、AI エージェントを本番運用するチーム。
- アクション: 新規は OTel で計装しバックエンドは差し替え可能に保つ。既存はCollector 経由で段階移行。大量トレースはサンプリングと列指向バックエンドでコスト管理。AI エージェントは各ステップをトレース化して観測。
