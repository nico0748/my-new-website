# React Router v8 — ESM専用化と Middleware 既定化、「年1メジャー」への移行

> トピック: React Router v8 リリース｜出典: Remix 公式ブログ（remix.run）｜種別: メジャーバージョンリリース

## 一行サマリー

React Router v8 が公開され、v7 の future フラグ（Middleware・Vite Environment API・Split Route Modules 等）がすべて既定動作になり、パッケージは ESM 専用・Node 22+/React 19+/Vite 7+ が新ベースラインとなった。今後は「年1回のメジャー＋壊さないアップグレード」を掲げる。

## 🔰 初学者向け（何が・なぜ重要か）

React Router は「React アプリの中でページの切り替え（URL とページ内容の対応付け）」を担う、最も広く使われているライブラリの一つです。地図でいえば「この住所（URL）ならこの建物（画面）を表示する」という案内板の役割です。

v8 の最大のポイントは、新機能の追加よりも「約束の仕方」が変わったことです。これまで実験的な機能は future フラグ（先行スイッチ）としてオプトインで提供されてきましたが、v8 ではそれらが標準動作になりました。つまり「v7 でフラグを全部オンにしていた人は、ほぼ何も壊れずに v8 に上がれる」設計です。引っ越しでいえば、新居に前もって家具を運び込んでおき、当日は鍵を受け取るだけ、というイメージです。

もう一つは Middleware（ミドルウェア）の標準化です。ページを表示する前後に「ログイン確認」「ログ記録」などの共通処理を挟める仕組みで、これまで各ページにコピペしがちだった処理を1か所にまとめられます。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- v7 の future フラグが既定化: `v8_middleware`（ルートレベルのミドルウェアパイプライン）、Vite Environment API、Split Route Modules など。
- **ESM 専用**: CommonJS ビルドの提供を終了。`tsconfig` の target/lib は ES2022 相当に更新。
- **新ベースライン**: Node 22+（Active LTS 全対応・Maintenance LTS は最新マイナーのみ）、React 19+、Vite 7+。
- 破壊的変更は最小限で、いずれも「v7 のうちに先行適用できる変更」に限定。公式は v7→v8 を実質非破壊アップグレードと位置付けている。
- 今後は**年次メジャーリリース**サイクルへ移行（正確なリリース日は要確認。2026年6月に公式アナウンス）。

### 仕組み / 背景（なぜこう設計されたか）

React Router は v6→v7 で Remix と統合され、「ライブラリ」と「フレームワーク」の2モードを持つ大型プロジェクトになった。その規模でメジャーバージョンごとに破壊的変更を積むと、エコシステム（チュートリアル、社内規約、周辺ライブラリ）が追従できない。そこで採られたのが future フラグ方式で、「次のメジャーの動作を現行版でオプトインさせ、メジャー本体は既定値の切り替えだけにする」という段階的移行モデルである。これは Ember の svelte deprecations や React 自身の StrictMode 的な「先に警告・先に体験」戦略の系譜にある。

トレードオフとしては、フラグ期間中は同一バージョンで動作が分岐するためドキュメントやトラブルシュートが複雑になる。一方で得られるのは「メジャー＝儀式ではなく日常のマイナー並みの作業」という予測可能性で、年1メジャー宣言はこの成果を制度化したものといえる。

ESM 専用化は、Vite 7 以降のエコシステム（Rolldown 化を含む）が ESM を前提に最適化されている流れと一致する。CJS 互換レイヤーの維持コスト（デュアルパッケージ問題、`require` 経由の初期化順の罠）を切り捨て、ツールチェーン全体の単純化を優先した判断。

### 実務での使いどころ・移行の勘所

- v7 利用者はまず **v7 のまま future フラグを全て有効化**し、テストが通る状態を作ってから v8 に上げるのが公式推奨ルート。
- CJS で React Router を `require` しているレガシー構成（古い Jest 設定、CJS のサーバーコード）は先に ESM 化が必要。Node 22 未満のランタイムも同時に更新する。
- Middleware 既定化により、認証ガードやリクエストスコープのコンテキスト注入を loader 内に書く自作パターンから公式パイプラインへ寄せられる。散在する認可チェックの一元化はセキュリティ監査面でも有効。
- 新規プロジェクトで Next.js を選ばない（RSC 不要・SPA/BFF 構成）場合の第一候補としての地位は継続。TanStack Router/Start（型安全ルーティング重視）との比較検討軸は「エコシステムの厚み vs 型推論の深さ」。

## 📖 用語解説（このトピックとのつながり）

- **future フラグ**: 次期メジャーの動作を現行版で先行有効化するオプション。v8 はこれらを既定化しただけ、という点が「壊れないメジャー」の根拠。
- **ESM（ECMAScript Modules）**: JS 標準のモジュール形式。v8 はこれ専用となり、旧式の CommonJS からは直接読み込めなくなった。
- **Middleware**: ルート処理の前後に共通処理を挟むパイプライン。v8 で標準機能となり、認証・ログ・エラー処理を一元化できる。
- **Vite Environment API**: クライアント/サーバー等の複数環境を Vite が一貫して扱う API。React Router のフレームワークモードの土台。

## 影響範囲 / 推奨アクション

- v7 プロジェクト: future フラグを順次有効化 → 全通過後に v8 へ。作業量は小さい見込み。
- v6 以前: v7 への移行（Remix 統合後の構成変更）が先。v8 直行は非推奨。
- ランタイム条件（Node 22+/React 19+/Vite 7+）を CI マトリクスに反映する。
