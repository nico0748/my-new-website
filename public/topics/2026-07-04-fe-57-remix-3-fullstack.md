# Remix 3（beta）― React 依存を捨てた「ゼロ依存フルスタックJSフレームワーク」への転身

> トピック: 開発トピック｜出典: remix.run｜種別: frontend

## 一行サマリー
Remix はこれまでの成果を React Router 8 の Framework Mode に還流させたうえで、Remix 3（beta）として React 依存を外し、ゼロ依存のフルスタック JavaScript フレームワークへと役割を転換した。React 前提ではない Web の作り方を志向する新路線で、React Router とは明確に役割分担する。

## 🔰 初学者向け（何が・なぜ重要か）
Web アプリを作るとき、多くのフレームワークは「React が土台にある」ことを前提にしています。Remix もかつてはそうでした。今回の Remix 3 は、その土台（React）を外し、「特定のライブラリに縛られない、素の JavaScript で組める土台」に作り直そうとしています。

料理に例えると、これまでは「必ずこの鍋（React）を使う前提のレシピ集」でした。Remix 3 は「鍋の種類に依存しない、火加減と手順そのものを提供するレシピ」に変えるイメージです。依存（外部の道具）を極力減らすことで、動作が軽く・読みやすく・壊れにくくなることを狙っています。すでに React で作られた膨大なサイトは React Router 8 が引き続き面倒を見るので、既存資産を捨てる話ではなく「新しい選択肢が増える」話です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
- Remix v0.x〜2.x の成熟した API は **React Router 8 の Framework Mode** に統合済み（Remix と React Router は実質同根）。
- **Remix 3 は beta** 段階で、**React 依存を外した**「ゼロ依存のフルスタック JS フレームワーク」を標榜（正確な GA 時期・API は要確認、公式ブログ参照）。
- React 専用だった従来路線から、より広い Web プラットフォーム志向へ転換。React Router（React 向け）とは並存し、役割を分ける。

### 仕組み / 背景（なぜこう設計されたか）
Remix と React Router の統合により、「React 上のフルスタック機能」は React Router 8 側に集約されました。すると Remix ブランド側は、React に縛られないフレームワークとして再定義する余地が生まれます。ゼロ依存を掲げる背景には、依存グラフの肥大化がビルド時間・セキュリティ面（サプライチェーン）・保守性を悪化させるという近年の反省があります。依存を減らせば、フレームワーク挙動の予測可能性が上がり、Web 標準（Fetch/Request/Response 等）に近い形でアプリを書けます。

トレードオフとして、React の巨大なエコシステム（コンポーネントライブラリ、既存知見）から距離を置くことになり、初期の採用者は「使えるライブラリが少ない・情報が少ない」状況を受け入れる必要があります。beta 段階である点も、本番採用にはまだ慎重さが求められる理由です。

### 実務での使いどころ・移行の勘所
- 既存の React ベースのアプリは **React Router 8（Framework Mode）** へ、が基本線。Remix 3 は「React に依存したくない新規プロジェクト」の実験的選択肢。
- ゼロ依存・Web 標準志向は、エッジ実行や軽量デプロイと相性が良い。ただし beta のため API 破壊的変更を織り込む。
- チームの既存スキル（React 中心か否か）で採否が分かれる。当面は PoC/検証用途が現実的。

## 📖 用語解説（このトピックとのつながり）
- **Remix 3**: React 依存を外したフルスタック JS フレームワークの新路線。本トピックの主役。
- **ゼロ依存 (zero-dependency)**: 外部ライブラリ依存を極力持たない設計。保守性とセキュリティの利点。
- **フルスタックフレームワーク**: フロント描画とサーバー処理を一体で扱う枠組み。
- **Framework Mode**: React 向けフルスタック機能を担う React Router 8 側の使い方。Remix 3 と役割分担。
- **Web 標準 API**: Fetch/Request/Response など。ゼロ依存志向が寄りかかる土台。

## 影響範囲 / 推奨アクション
React エコシステムの外側にフルスタックの選択肢が増える点で、フロントエンド全体の設計思想に影響しうる。既存プロジェクトは急いで移る必要はなく、React Router 8 を継続利用が無難。Remix 3 は beta の API 動向を追いつつ、非 React 志向の新規案件で小さく試すのが妥当。
