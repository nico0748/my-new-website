# Go 1.25 — コンテナ対応 GOMAXPROCS・encoding/json v2 実験版・Green Tea GC 実験で「本番既定」を強化

> トピック: Go 1.25 の実行環境改善｜出典: Go 公式リリースノート（go.dev）｜種別: 言語/ランタイム更新

## 一行サマリー

2025-08-12 リリースの Go 1.25 は、cgroup の CPU 上限を見て自動調整するコンテナ対応 GOMAXPROCS を既定化し、大幅に速い encoding/json v2 と局所性を改善する Green Tea GC を実験フラグで提供、コンテナ本番でのスロットリングや過剰並列という長年の「そのままだと踏む地雷」を潰した。

## 🔰 初学者向け（何が・なぜ重要か）

Go はサーバー開発で人気の言語です。1.25 では「コンテナ（Docker/Kubernetes）で動かしたときの落とし穴」が1つ埋まりました。

これまで Go は「このマシンには CPU が何個あるか」を、マシン全体の数で数えていました。ところがコンテナでは「あなたは4コアぶんだけ使ってよい」と制限をかけて動かすのが普通です。すると Go は「128コアある！」と勘違いして張り切り、実際は4コア分しか許されないため、カーネルに強制的に一時停止（スロットリング）をかけられ、応答がガクッと遅くなることがありました。

1.25 の GOMAXPROCS は、コンテナに設定された CPU 上限をちゃんと読み取り、そこに合わせて並列度を自動調整します。しかも上限が途中で変わっても追随します。特別な設定なしで、コンテナ本番の性能が素直に出るようになった、という改善です。加えて、JSON 処理の高速版や新しいゴミ集め（メモリ回収）方式も試せます。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **コンテナ対応 GOMAXPROCS（既定）**: Linux では所属 cgroup の CPU 帯域制限を考慮。制限が論理CPU数より低ければ、その低い値を GOMAXPROCS の既定にする。オーケストレータが上限を動的変更した場合も、定期的に再チェックして自動調整。
- **encoding/json v2（実験）**: `GOEXPERIMENT=jsonv2` で有効化。`encoding/json/v2` と低レベルの `encoding/json/jsontext` を提供。エンコードは概ね同等、デコードが大幅に高速。
- **Green Tea GC（実験）**: `GOEXPERIMENT=greenteagc` で有効化。小オブジェクトのマーク/スキャンを局所性と CPU スケーラビリティ改善で高速化。GC 多用プログラムで概ね 10〜40% の GC オーバーヘッド削減を見込む（ベンチ依存）。
- リリース 2025-08-12。数値は公式リリースノートに基づき、実験機能の結果はワークロード依存で要検証。

### 仕組み / 背景（なぜこう設計されたか）

GOMAXPROCS 問題の根は、Go 1.5（2015 年）以来「GOMAXPROCS = ホストの論理CPU数」を既定にしてきた歴史にある。ベアメタル時代は妥当だったが、コンテナ時代には破綻する。Linux の CFS は cgroup に「一定期間で使ってよい CPU 時間」を割り当て、使い切ったプロセスを最大 100ms 単位で停止させる。Go が実体より多いコア数を前提に大量の OS スレッドを走らせると、この帯域を一気に食い潰し、周期的な停止（＝テールレイテンシ悪化）を招く。1.25 は cgroup の quota を読み、並列度をその実効上限に合わせることでスロットリングを避ける設計だ。動的変更への追随は、オーケストレータが実行中に上限を変える現実に対応するための定期ポーリングによる。

encoding/json v2 は、旧 `encoding/json` がリフレクション主体で遅く、ストリーミングや厳密性の制御が弱かった課題への再設計だ。低レベル `jsontext`（構文処理）と高レベル `v2`（値マッピング）を分離し、デコード経路を最適化する。Green Tea GC は、既存 GC の弱点である「小オブジェクトのマーク時にメモリ上を飛び回って局所性が悪い」点を、走査対象の配置を作り直して改善する試みだ。いずれも実験段階で、既定化は将来版での安定性検証を経てから。トレードオフは、実験フラグ機能は本番投入すると想定外挙動のリスクがある点。

### 実務での使いどころ・移行の勘所

Kubernetes 等で CPU limit を設定して Go サービスを運用しているなら、1.25 へ上げるだけでテールレイテンシ改善が期待できる（多くの場合、手動 `GOMAXPROCS` 設定や uber-go/automaxprocs の外部依存を外せる）。json v2 と Green Tea GC はまずステージングで、実ワークロードのベンチとメモリ挙動を計測してから。特に JSON が支配的な API や、GC 圧の高いキャッシュ/バッチ処理では効果が出やすい。実験フラグはビルド時指定のため、CI/本番で有効化の有無を明示管理する。

## 📖 用語解説（このトピックとのつながり）

- **GOMAXPROCS**: Go が同時に使う OS スレッド（P）の上限。1.25 でコンテナ上限に自動追随。
- **cgroup / CPU quota**: Linux がプロセス群に課す資源制限。スロットリングの根拠。
- **スロットリング**: 割り当て超過時にカーネルがプロセスを一時停止する挙動。テール遅延の原因。
- **encoding/json v2**: 再設計された高速 JSON 実装（実験）。デコードが速い。
- **Green Tea GC**: 局所性重視の新 GC 実験。小オブジェクト走査を高速化。

## 影響範囲 / 推奨アクション

- 対象: コンテナ/K8s 上で動く Go バックエンド。
- 推奨: 1.25 へ更新しコンテナ既定の恩恵を取得。automaxprocs 依存の見直し。json v2 / Green Tea GC はステージングで計測。
- 注意: 実験フラグ機能は本番前に実ワークロードで検証。
