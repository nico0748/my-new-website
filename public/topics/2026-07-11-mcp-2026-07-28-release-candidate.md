# MCP 2026-07-28 リリース候補 — ステートレス化・MCP Apps・Tasks で「普通の HTTP」に載るエージェント基盤へ

> トピック: Model Context Protocol 2026-07-28（リリース候補）｜出典: Model Context Protocol Blog｜種別: プロトコル / エージェント

## 一行サマリー

AI エージェントに外部ツールを繋ぐ標準 MCP の次期仕様 2026-07-28 が、リリース候補（RC）として公開された。7 月 28 日に確定予定で、ローンチ以来最大の改訂。プロトコル層をステートレス化して普通の HTTP 基盤（ラウンドロビンな負荷分散）に載せられるようにし、サーバー描画 UI を扱う「MCP Apps」と長時間処理の「Tasks」拡張、そして OAuth/OIDC により整合した認可を導入する。

## 🔰 初学者向け（何が・なぜ重要か）

MCP は「AI エージェントと、外部のツールやデータ（社内 DB、SaaS、ファイル…）をつなぐ共通の差し込み口」。USB のように、一度この規格に対応すれば、いろいろな AI とツールを繋ぎ替えられる。

今回の 2026-07-28 は、その規格の**過去最大のアップデート**の最終確認版（リリース候補）だ。最大の変更は「ステートレス化」。これまで MCP サーバーは、接続ごとに『会話の途中経過（セッション）』を覚えておく必要があり、同じサーバーに繋ぎ続ける工夫（sticky session）や共有の記憶置き場が要って、大規模運用がしにくかった。

ステートレス化すると、サーバーは各リクエストを独立に処理できるので、ごく普通の負荷分散（どのサーバーに振っても OK）で動く。たとえるなら、これまで「同じ担当者にずっと付いてもらう必要があった窓口」が、「どの窓口に並んでも同じように用が足せる」ようになった。運用がぐっと楽になる。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **位置づけ**: 2026-07-28 仕様の**リリース候補が公開**、最終版は 7 月 28 日出荷予定。ローンチ以来最大の改訂。
- **ステートレスな中核**: プロトコル層をステートレス化。従来 sticky session・共有セッションストア・ゲートウェイでの深いパケット検査を要したリモートサーバーが、素のラウンドロビン負荷分散の背後で動かせる。
- **MCP Apps 拡張**: サーバー側で描画した UI（server-rendered UI）を提供する仕組み。
- **Tasks 拡張**: 長時間かかる処理を扱う拡張。
- **認可の強化**: OAuth / OpenID Connect のデプロイに、より整合。クライアントは RFC 9207 に従い認可応答の `iss`（発行者）パラメータを検証する必要があり、MCP に多い「1 クライアント・多サーバー」構成で起きやすい mix-up 攻撃を低コストに緩和。
- **エコシステム（参考）**: 2026 年 3 月時点で Glama レジストリは 19,831+ サーバー、MCP.so は 16,000+ を掲載。`.well-known` で能力を発見できるメタデータ形式も整備。

### 仕組み / 背景（なぜこう設計されたか）

初期の MCP は、クライアントとサーバーが接続をまたいで状態を共有する前提の設計で、リモート運用時に「同一サーバーへ貼り付ける」必要があった。これはスケール・可用性・ゲートウェイ運用の足枷になっていた。ステートレス化は、「各リクエストが自己完結し、どのサーバーインスタンスでも処理できる」HTTP らしい設計へ寄せることで、水平スケールと一般的なクラウドインフラへの適合を狙う。代償として、真に状態が要る対話や長時間処理は別の仕組みで支える必要があり、それが **Tasks 拡張**（長時間ジョブ）だ。UI を伴うツールのために **MCP Apps**（サーバー描画 UI）を分離拡張として持ち込み、中核は薄く保ちつつ表現力を拡張で足す構成にしている。

認可面は、MCP 特有の脅威モデルへの対応だ。1 つのクライアントが多数のサーバーに繋ぐ構成では、認可レスポンスの取り違え（mix-up 攻撃）が起きやすい。RFC 9207 の `iss` 検証を必須化することで、「どの認可サーバーからの応答か」をクライアントが確認でき、低コストで一群の攻撃を防げる。トレードオフは、既存の MCP サーバー/クライアントに改修が要ること、そしてステートレス前提に合わせてサーバー実装の状態管理を作り替える移行コストだ。

### 実務での使いどころ・移行の勘所

社内ツール・SaaS をエージェントに繋ぐ基盤や、多数のリモート MCP サーバーを運用する組織に効く。移行の勘所は、(1) 既存サーバーの状態依存を洗い出し、リクエスト自己完結型に作り替える、(2) 長時間処理は Tasks 拡張へ、UI 付きツールは MCP Apps へ寄せる、(3) 認可は OAuth/OIDC に整え、クライアント側で `iss` 検証（RFC 9207）を実装、(4) RC 段階のため仕様確定（7/28）を待って本番採用し、それまでは互換性検証にとどめる。ステートレス化で運用は楽になるが、既存実装の改修は避けられない前提で計画を。

## 📖 用語解説（このトピックとのつながり）

- **MCP（Model Context Protocol）**: AI エージェントに外部ツール/データを繋ぐ標準。今回過去最大の改訂。
- **ステートレス / sticky session**: 各リクエスト自己完結の設計と、同一サーバーへ貼り付ける従来方式。RC で前者へ。
- **MCP Apps / Tasks**: サーバー描画 UI を扱う拡張と、長時間処理を扱う拡張。中核を薄く保つ分離設計。
- **RFC 9207 / iss 検証**: 認可応答の発行者確認。多サーバー構成の mix-up 攻撃を緩和。
- **.well-known**: サーバー能力を接続前に発見するためのメタデータ公開場所。

## 影響範囲 / 推奨アクション

- リモート MCP を多数運用する組織は、ステートレス化で水平スケール・運用が容易に。
- 既存サーバーは状態依存を解消し、長時間処理は Tasks、UI は MCP Apps へ再設計。
- 認可を OAuth/OIDC に整え `iss` 検証を実装。RC のため確定（7/28）まで本番採用は保留し互換検証を。
