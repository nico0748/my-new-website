# Postgres 18 の uuidv7() と仮想生成列：インデックス効率とクエリ時計算

> トピック: 開発トピック｜出典: postgresql.org｜種別: backend

## 一行サマリー
PostgreSQL 18 は時系列順に並ぶ `uuidv7()` 組み込み関数と、値を保存せずクエリ時に計算する仮想生成列を追加。ID のインデックス効率と、ストレージを使わない派生列の実現手段が標準化された。

## 🔰 初学者向け（何が・なぜ重要か）
データベースの各行には「ID（背番号）」を振ります。従来よく使う UUID（ランダムな長い ID）は重複しにくい利点がある一方、値が完全にバラバラなため、索引（インデックス）に追加するたびにあちこちに挿入され、内部が断片化して遅くなる欠点がありました。

`uuidv7()` は「先頭に時刻を埋め込んで、だいたい時間順に並ぶ UUID」です。新しい行は索引の末尾付近にまとまって入るため、書き込みが素直で速くなります。もう一つの「仮想生成列」は、「税込価格 = 税抜 × 1.1」のような計算列を、実際には保存せず問い合わせのたびに計算する仕組みです。保存領域を食わずに便利な列を持てます。地味ですが、日々の設計を楽にする実用的な追加です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
18 で `uuidv7()` が組み込み関数として提供され、時刻ソート可能な UUID（RFC 9562 準拠 ※詳細は要確認）を DB 側で生成できる。仮想生成列（`GENERATED ... VIRTUAL`）は、格納型（STORED）と異なり値を物理保存せず読み取り時に評価する。従来は STORED 生成列やアプリ側計算で代替していた領域を標準化。

### 仕組み / 背景（なぜこう設計されたか）
ランダム UUID（v4）は B-tree インデックスでランダム挿入となり、ページ分割とキャッシュミスを増やす。v7 は上位ビットに時刻を持つため単調増加に近く、挿入局所性が高くインデックス肥大を抑える。仮想生成列は、更新頻度の低い派生値や滅多に参照しない列でストレージと書き込みコストを節約するトレードオフを提供する。

### 実務での使いどころ・移行の勘所
高スループットの挿入テーブルで主キーに v7 を採用すると書き込み性能とインデックス効率が改善しやすい。ただし v7 は生成時刻が ID から推測可能なため、機密性が要る識別子には向かない場合がある。仮想生成列は読み取り時計算コストが乗るため、頻繁に検索・ソートする列には STORED を選ぶなど使い分けが必要。

## 📖 用語解説（このトピックとのつながり）
- **UUIDv7**：時刻順に並ぶ UUID。挿入局所性でインデックスを効率化。
- **B-tree インデックス**：範囲検索向けの標準索引。ランダム UUID で断片化しやすい。
- **仮想生成列（VIRTUAL）**：保存せずクエリ時に計算する派生列。ストレージ節約。
- **STORED 生成列**：値を物理保存する派生列。頻繁検索に有利な対比。

## 影響範囲 / 推奨アクション
新規テーブル設計を行う開発者・DBA に影響。挿入多発テーブルは主キーへの `uuidv7()` 採用を検討。ただし ID からの時刻推測リスクを評価し、派生列は VIRTUAL/STORED を用途で選び分けること。
