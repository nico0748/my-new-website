# 6月のオープンウェイト大波 — GLM 5.2・Kimi K2.7・MiniMax M3 がクローズド最前線に肉薄

> トピック: 2026年6月のオープンウェイト三連発｜出典: 各ラボ（Z.ai / Moonshot AI / MiniMax）｜種別: 新モデルリリース群（オープンウェイト）

## 一行サマリー

2026年6月、中国の3ラボが相次いで強力なオープンウェイトモデルを公開した。MiniMax M3（6月1日）、Kimi K2.7（Moonshot AI、6月12日）、GLM 5.2（Z.ai、6月13日）。GLM 5.2 は1Mコンテキスト・MIT ライセンスで SWE-Bench Pro 62.1%（Claude Opus 4.8 に7点差）、Kimi K2.7 は1兆パラメータ MoE（活性320億）でトークン効率30%改善、MiniMax M3 は「フロンティア級コーディング＋1M文脈＋ネイティブ多モーダル」を初めて1モデルに統合し、疎注意で高速化した。

## 🔰 初学者向け（何が・なぜ重要か）

ほんの1年前まで「本当に賢いAI」はほぼクローズド（API 越しでしか使えない）勢の独占でした。ところが2026年6月、中国の3つの研究所が「重みを配って誰でも自前運用できるのに、クローズド最前線に迫る」モデルを立て続けに出しました。オープンウェイトAIが"追いついた月"として記憶される、という評価が出るほどの出来事です。

3つは個性が違います。**GLM 5.2** はライセンスが MIT（商用も自由）で、プログラミング課題を解く力がトップクラス。**Kimi K2.7** は「エージェント（自分で手順を考えて道具を使うAI）」向けに設計され、同じ答えを出すのに使うトークン（＝コスト）を約3割節約。**MiniMax M3** は「コードも書ける・長い文章も読める・画像や動画も理解する」を1つのモデルにまとめ、しかも独自の高速化で速い。共通するのは「高性能なのに自前で持てる」こと。開発者にとっては、クラウド頼みでなく手元でフロンティア級を動かす選択肢が一気に増えた、という意味を持ちます。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **リリース**: MiniMax M3（6月1日）、Kimi K2.7（6月12日）、GLM 5.2（6月13日）。いずれもオープンウェイト。
- **GLM 5.2（Z.ai）**: 1Mトークンコンテキスト、MIT ライセンス。SWE-Bench Pro 62.1%（Claude Opus 4.8 に約7点差、多くのプロプライエタリを上回る主張）。低価格を強調（GPT-5.5 の約1/7との比較記事）。
- **Kimi K2.7（Moonshot AI）**: 1兆パラメータ MoE、活性320億／トークン、256Kコンテキスト。前世代比でトークン効率30%改善・コーディングベンチ21.8%向上を標榜。エージェント志向。
- **MiniMax M3**: フロンティア級コーディング＋1M文脈＋ネイティブ多モーダル（画像・動画入力）を統合。**MiniMax Sparse Attention（MSA）** で per-token 計算を前世代の1/20に削減、prefill 9倍・decode 15倍高速化、出力 約100 tok/s。
- 数値・ベンチは各ラボ／二次記事由来のものを含み、一次ソースで要確認。

### 仕組み / 背景（なぜこう設計されたか）

3モデルに共通する土台は MoE と長文脈・疎注意の効率化。MoE で「大容量だが1トークンあたりの計算は軽い」を実現し、オープン配布でも現実的なコストで動かせるサイズ感に収める（Kimi K2.7 の1兆/活性320億はその典型）。**MiniMax の MSA（疎注意）** は、長文脈で支配的になる注意計算を「全トークン対全トークン」から必要な部分だけに絞ることで、prefill（入力読み込み）と decode（生成）の双方を大幅高速化する設計。長文脈を"持てる"だけでなく"速く扱える"ことを狙う点が新しい。

なぜ中国ラボが6月に集中したか——オープンウェイトは「エコシステム占有」の戦略手段である。重みを MIT 等で配れば、微調整・蒸留・ツール統合・推論プロバイダ採用が世界規模で進み、事実上の標準の座を狙える。クローズド勢が精度で先行しても、オープン側が3〜6か月遅れで同水準に達し、かつ自前運用・低価格・データ主権という別軸の価値を積み上げれば、用途によって天秤は傾く。トレードオフは、ベンチ上の数値と実タスクの体感差、各ラボのベンチ選定バイアス、そして巨大 MoE のサービング難度（GPU メモリ・ルーティング最適化）。「動かせるか」はモデルの賢さと別に、推論スタック（SGLang/vLLM 等）と GPU 資源に依存する。

### 実務での使いどころ・移行の勘所

- **コーディング／エージェント**: GLM 5.2（MIT・高コーディング精度）、Kimi K2.7（エージェント志向・トークン効率）が候補。自前運用でフロンティア級のコード支援を組みたい場合に。
- **多モーダル＋長文脈**: MiniMax M3。画像・動画理解と1M文脈を1モデルで扱いたいパイプラインを単純化。MSA による速度が実運用のレイテンシに効く。
- 巨大 MoE のサービングは GPU メモリとルーティング最適化が壁。手軽さ重視なら推論プロバイダ経由、主権・コスト重視なら自前ホスト。
- クローズド（Claude Opus 4.8 / GPT-5.5 / Gemini 3.x）との比較軸は「最高精度・運用の楽さ vs 自由度・コスト・データ主権」。ベンチ差（数点）と運用要件のどちらが効くかで選ぶ。ベンチ主張は自前タスクで裏取りを。

## 📖 用語解説（このトピックとのつながり）

- **オープンウェイト**: 重みを配布し自前運用できる形態。3モデルの共通土台で、MIT 等なら商用も自由。
- **MoE（混合エキスパート）**: 入力ごとに一部の専門家だけ起こす構造。大容量と低計算コストを両立し、オープン配布サイズを現実化。
- **疎注意（Sparse Attention）／MSA**: 注意計算を必要箇所に絞る手法。MiniMax M3 が長文脈の高速化に採用。
- **SWE-Bench Pro**: 実務的なコーディング課題ベンチ。GLM 5.2 のスコアがクローズド最前線に迫る根拠。

## 影響範囲 / 推奨アクション

- 自前運用検討: 用途別に GLM 5.2（コード・MIT）／Kimi K2.7（エージェント）／MiniMax M3（多モーダル＋長文脈）を評価。
- 推論基盤: 巨大 MoE 対応の SGLang/vLLM と GPU 資源を前提に検証。ベンチは自前タスクで裏取り。
- 調達戦略: クローズド一択から、オープン併用（コスト・主権軸）へ選択肢を広げる好機。
