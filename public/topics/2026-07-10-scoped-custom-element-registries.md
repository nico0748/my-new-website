# スコープ付きカスタム要素レジストリが Chrome 146 で提供 — Web Components の名前衝突を解消

> トピック: Scoped custom element registries（Chrome 146）｜出典: Chrome for Developers ブログ / MDN｜種別: Web Components 機能

## 一行サマリー

カスタム要素（独自 HTML タグ）の登録をグローバルではなく Shadow DOM のサブツリー単位に限定できる「スコープ付きカスタム要素レジストリ」が Chrome 146 で利用可能になった。これにより「同じタグ名を別バージョンで二重登録できない」という Web Components 最大級の実務的痛点が解消に向かう。

## 🔰 初学者向け（何が・なぜ重要か）

Web Components は「自分だけの HTML タグ」を作れる仕組みで、`<my-button>` のような部品を定義できる。ただし従来、この登録は「ページ全体でひとつ」の名簿（レジストリ）に対して行われた。つまりページ内で `<my-button>` という名前は一度しか使えない。

これが問題になるのは、たとえば古いバージョンの部品ライブラリと新しいバージョンを同じページで併用したいとき。両方が `<my-button>` を名乗ると衝突してエラーになる。マイクロフロントエンド（1画面を複数チーム/アプリが分担）でも同じ衝突が起きやすい。スコープ付きレジストリは「この Shadow DOM の中だけで通用する名簿」を持てるようにする仕組み。同じタグ名を、場所ごとに別の定義として共存させられる。「名前の取り合い」で設計が縛られていた Web Components が、ぐっと組み合わせやすくなる。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

従来カスタム要素はグローバルな `window.customElements`（`CustomElementRegistry`）にしか登録できず、同一タグ名の再定義は不可だった。スコープ付きレジストリは新規に `CustomElementRegistry` インスタンスを生成し、それを特定の `ShadowRoot` に紐づけて、そのツリー内でのみ有効なタグ定義を持てるようにする。Chrome for Developers の告知では Chrome 146 でスコープ付きカスタム要素レジストリが導入されたとされる（同世代ではスクロール駆動アニメーションや Sanitizer API も追加）。これにより同名タグの複数バージョン共存や、サブツリーローカルな要素定義が可能になる（正確な API 形状と他エンジンの対応は MDN で要確認）。

### 仕組み / 背景（なぜこう設計されたか）

Web Components の「グローバル単一レジストリ」は、当初はシンプルさを優先した設計だったが、実運用では「タグ名はグローバルに一意」という制約がライブラリ合成やバージョン共存の障害になっていた。とりわけ複数の独立デプロイ（マイクロフロントエンド）を1ページに束ねる構成では致命的で、名前空間の欠如が Web Components 採用を鈍らせる一因だった。スコープ付きレジストリは、Shadow DOM が提供する「カプセル化（スタイル・DOM の封じ込め）」の思想を要素定義にも拡張し、"定義もローカルに閉じる" ことで衝突を構造的に排除する。トレードオフは、レジストリを跨いだ要素の受け渡しやデバッグが複雑になりうること、ツール/ポリフィル・フレームワーク側の対応が揃うまで恩恵が限定される点。まず Chromium 系で先行し、他エンジンへ波及していく段階にある。

### 実務での使いどころ・移行の勘所

デザインシステムのバージョン移行期（旧新コンポーネントの併存）、複数チームのマイクロフロントエンド統合、サードパーティウィジェットの埋め込みで効く。移行の勘所は、単一エンジン先行のため当面はポリフィルや対応フレームワーク（スコープ付きレジストリを内部利用する Web Components ライブラリ）経由での採用が現実的なこと、クロスブラウザ要件がある場合は Baseline 到達を待つか段階採用に留めること。名前衝突を回避するための命名規約（プレフィックス付与）と併用し、恒久解決としてスコープ化を位置づけると移行が滑らかになる。

## 📖 用語解説（このトピックとのつながり）

- **カスタム要素 / CustomElementRegistry**: 独自 HTML タグとその名簿。従来はグローバル単一で、本機能がサブツリー単位を可能にした。
- **Shadow DOM**: コンポーネントの DOM/スタイルを外部から隔離するカプセル化機構。スコープ付きレジストリはこれに定義を紐づける。
- **マイクロフロントエンド**: 1つの画面を複数の独立アプリで構成する方式。タグ名衝突が起きやすく本機能の主要な受益者。
- **ポリフィル**: 未対応環境で新機能を補う実装。クロスブラウザ採用の橋渡しに使う。

## 影響範囲 / 推奨アクション

- Web Components のバージョン共存・合成のしやすさが向上し、マイクロフロントエンド構成の障壁が下がる。
- 現段階は Chromium 先行のため、ポリフィル/対応ライブラリ経由での採用を検討。クロスブラウザ要件では Baseline 到達を確認。
- 当面は命名規約（プレフィックス）と併用し、スコープ化を恒久的な衝突回避策として段階導入する。
