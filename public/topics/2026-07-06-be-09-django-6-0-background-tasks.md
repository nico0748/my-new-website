# Django 6.0 — Celery 不要の組み込みバックグラウンドタスク

> トピック: Django 6.0｜出典: Django 公式ドキュメント / InfoQ｜種別: メジャーリリース（2025年末、6.0.x 系が現行）

## 一行サマリー

Django 6.0 は、HTTP リクエスト/レスポンス外で処理を回す組み込みの「Tasks フレームワーク」を標準搭載し、`@task` デコレータと `.enqueue()` で Celery 等の外部ライブラリなしに非同期処理を定義・投入できるようにした。ただしワーカー実行機構は Django 側に含まれず、実行は外部インフラが担う。ネイティブ CSP 対応や Python 3.14 対応も加わる。

## 🔰 初学者向け（何が・なぜ重要か）

Web アプリでは「メール送信」「画像変換」「集計」など、時間のかかる処理をユーザーの待ち時間から切り離したい場面が多くあります。ユーザーの操作にはすぐ応答を返し、重い処理は裏側で後回しに実行する——これが「バックグラウンドタスク」です。これまで Django でこれをやるには、Celery のような外部ツールを別途組み込むのが定番でした。

Django 6.0 は、この仕組みを標準機能として取り込みました。関数に `@task` という目印を付け、`.enqueue()` で「後でやっといて」とキュー（順番待ちの列）に入れるだけ。外部パッケージを1つも足さずに、非同期の後回し処理を定義できます。

ただし1つ注意点があります。Django が提供するのは「タスクを定義して列に入れる」ところまでで、実際に列から取り出して働く「ワーカー（作業員）」は Django には含まれていません。ワーカーは別プロセスやサービスとして自分で用意する必要があります。つまり「共通の入口」を標準化した——エコシステムの分断を減らす、地味だが効く改善です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **組み込み Tasks フレームワーク**: `django.tasks` モジュールが `@task` デコレータと `.enqueue()` メソッドを提供。特定のキュー実装に依存しない統一 API でバックグラウンドタスクを定義・投入できる。
- **重要な但し書き**: Django はワーカー機構（キューからタスクを取り出して実行する仕組み）を提供しない。実行は別プロセス/サービスなど Django 外のインフラが担う。
- **ネイティブ CSP 対応**: Content Security Policy をフレームワーク標準でサポート。
- **その他**: Python 3.12/3.13/3.14 対応。`geom_type` ルックアップと `GeometryType()` 関数、全文検索の `Lexeme` 式など（GIS/検索系）。複合主キー関連の機能も話題（詳細は公式リリースノート参照）。
- LTS は後続の 6.2 が指定（リリース後3年以上のセキュリティ更新）。

### 仕組み / 背景（なぜこう設計されたか）

Django が「タスクの定義・投入 API」だけを標準化し、ワーカー実行を外部に委ねたのは意図的な設計だ。バックグラウンド処理のバックエンド（Redis、DB、クラウドキュー等）や実行モデルは用途で大きく異なり、フレームワークが特定実装を抱え込むと選択肢を狭める。そこで「アプリ側は共通 API で `@task`/`.enqueue()` を書き、バックエンドは差し替え可能」という抽象境界を引いた。これにより、Celery を使うプロジェクトも軽量な DB キューで済むプロジェクトも、同じアプリコードのまま実行基盤だけ選べる。標準化の狙いは、これまで各ライブラリでバラバラだった「タスクの書き方」を Django コアに統一し、学習コストと乗り換えコストを下げること。トレードオフは、ワーカーが同梱されないため「入れてすぐ動く」わけではなく、実行基盤の用意と運用（スケーリング、失敗時の再試行、監視）は依然として利用者の責務である点だ。

### 実務での使いどころ・移行の勘所

新規プロジェクトは、まず組み込み Tasks の統一 API でタスクを書き、実行基盤は要件に合わせて選ぶのが筋。小規模なら DB バックエンドで始め、負荷が上がれば専用キュー/ワーカーへ差し替えられる。既存の Celery 資産があるプロジェクトは、無理に置き換えず「新規タスクは標準 API で書く」ところから段階導入すると、将来のバックエンド差し替え自由度を確保できる。ネイティブ CSP 対応は、XSS 対策のヘッダ設定をミドルウェア標準で管理できるため、セキュリティ設定の一元化に有効。移行時は Python バージョン（3.14 まで対応）と、非同期実行に伴う冪等性・再試行設計を必ず詰めること。

## 📖 用語解説（このトピックとのつながり）

- **バックグラウンドタスク**: リクエスト応答の外で後回し実行する処理。Django 6.0 が標準 API を提供。
- **`@task` / `.enqueue()`**: タスクを定義し、キューへ投入する Django 6.0 の中核プリミティブ。
- **ワーカー**: キューからタスクを取り出して実行するプロセス。Django には含まれず外部で用意。
- **CSP（Content Security Policy）**: 読み込み可能なリソースを制限し XSS を緩和するヘッダ。6.0 で標準対応。
- **Celery**: 従来定番の外部タスクキュー。6.0 の標準 API で「必須ではない」選択肢に。

## 影響範囲 / 推奨アクション

- 対象: Django/Python でバックエンドを構築するチーム。
- アクション: 新規は組み込み Tasks の統一 API でタスクを書き、実行基盤は規模に応じて選定。既存 Celery 利用は新規タスクから段階導入。CSP は標準対応で一元管理。非同期処理は冪等性・再試行・監視を設計する。
