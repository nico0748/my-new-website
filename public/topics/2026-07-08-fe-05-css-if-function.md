# CSS `if()` 関数：プロパティ値の中に条件分岐を書く（2026 のブラウザ対応状況）

> トピック: CSS `if()` 関数｜出典: MDN / Chrome for Developers ブログ / caniuse｜種別: CSS 新機能（Chromium 先行実装）

## 一行サマリー

CSS の `if()` 関数は、プロパティ値の内側に条件分岐（`style()` / `media()` / `supports()` クエリ）を直接書けるインライン条件式。2026年半ば時点で Chrome 137+ / Edge / Opera（Chromium 系）に対応、Firefox は実装進行中、Safari は 2026〜2027 にロードマップ入り（要確認）。Baseline 未達のため、`@supports` ガード付きのプログレッシブエンハンスメントで使うのが定石。

## 🔰 初学者向け（何が・なぜ重要か）

これまで CSS で「もし〜なら色を変える」という条件分岐は、`@media`（画面幅など）や `@supports`（機能対応判定）といった「ルールのブロック単位」でしか書けなかった。1つのプロパティの値だけを条件で切り替えたいのに、ブロックを丸ごと複製する必要があり、冗長だった。

`if()` 関数は、プロパティの値そのものに条件を埋め込める。たとえば「あるカスタムプロパティが `dark` のときは背景を黒、それ以外は白」といった分岐を、`background: if(style(--theme: dark): black; else: white)` のように1行で書ける（構文はイメージ・詳細は仕様/MDN で要確認）。

デザインシステムでは、コンポーネントの見た目を CSS 変数で切り替えることが多い。`if()` があると「変数の値に応じてスタイルを分岐する」ロジックを CSS 内で完結でき、JavaScript でクラスを付け替える手間を減らせる。プログラミングでおなじみの三項演算子（`条件 ? A : B`）が CSS にやってきた、と捉えると分かりやすい。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- `if()` はプロパティ値内でインライン条件分岐を提供。条件には `style()`（カスタムプロパティ等の値判定）、`media()`（メディアクエリ相当）、`supports()`（機能サポート判定）が使える。
- 複数条件を並べ、最後に `else` フォールバックを置ける（順に評価し最初に真になった値を採用）。
- 対応状況（2026年半ば）: Chrome 137+ / Edge / Opera 対応、Firefox 実装進行中、Safari はロードマップ 2026〜2027（要確認）。
- **Baseline 未達**: 主要ブラウザの一部で未対応のため、Baseline の基準を満たさない。

### 仕組み / 背景（なぜこう設計されたか）

`if()` の狙いは「条件のためにセレクタやルールブロックを増やさない」こと。従来は 1 プロパティの分岐でも `@media` / `@supports` ブロックを複製し、同じセレクタを何度も書く必要があった。これは保守性を下げ、カスケードの見通しを悪くする。`if()` は分岐を「値のレベル」に閉じ込めることで、宣言の数を増やさずに条件を表現する。

特にカスタムプロパティ（CSS 変数）との相性が良い。デザイントークンやテーマ切り替えは変数に集約されがちで、その値に応じて別プロパティを変えたい要求が頻出する。`style()` クエリで変数値を直接読めると、JavaScript 側でクラスを操作せずに CSS だけでテーマ分岐が完結する。トレードオフは、条件が値の内側に隠れるため、複雑にしすぎると可読性が落ちる点だ。ネストしすぎず、デザイントークンの単純な分岐にとどめるのが健全。

Baseline 未達である以上、実務では「安全なデフォルト値を先に書き、その上に `if()` を `@supports` で重ねる」プログレッシブエンハンスメントが推奨される。未対応ブラウザではデフォルト値が使われ、対応ブラウザでだけ条件分岐が効く。

### 実務での使いどころ・移行の勘所

- デザインシステム / テーマ切り替えで、変数値に応じたスタイル分岐を CSS に寄せたいとき。
- 「JS でクラスを付け替えて色を変える」ロジックの一部を CSS 側へ移し、再レンダリングやフラッシュ（FOUC）を減らす。
- Chromium 系のみ対応のため、必ずフォールバック値を先に宣言し `@supports (...)` でガードする。
- 条件のネストは避け、単純な二分岐〜少数分岐に限定して可読性を保つ。

## 📖 用語解説（このトピックとのつながり）

- **カスタムプロパティ（CSS 変数）**: `--name` で定義する再利用可能な値。`if()` の `style()` クエリで値を判定でき、テーマ分岐の中心になる。
- **`@supports`**: 機能サポートを判定する CSS 規則。`if()` 未対応ブラウザ向けにフォールバックをガードするのに使う。
- **Baseline**: 主要ブラウザで安定利用できるかの指標。`if()` は現状 Baseline 未達。
- **プログレッシブエンハンスメント**: デフォルトを全ブラウザで保証し、対応ブラウザに機能を上乗せする手法。`if()` 導入の前提。
- **デザイントークン**: 色・余白などを変数化した設計単位。`if()` はトークン値に応じた分岐と好相性。

## 影響範囲 / 推奨アクション

- デザインシステム担当は `if()` を「変数値による分岐」の候補として評価する。
- 導入時は安全なデフォルト値を先に書き、`@supports` でガードしてプログレッシブに適用。
- Firefox / Safari の対応が進むまでは本番の必須挙動を `if()` に依存させない。
- 条件のネストを抑え、可読性とカスケードの見通しを維持する。
