# LiteLLM AIゲートウェイのコマンドインジェクション、認証回避と連鎖し未認証RCEへ（CVE-2026-42271, KEV・実環境で悪用）

> トピック: 脆弱性ニュース｜出典: The Hacker News / Horizon3.ai / Cloud Security Alliance / CISA KEV｜種別: コマンドインジェクション（CWE-78）＋認証回避（Starlette Host Header Bypass）の連鎖

## 一行サマリー
複数のLLMを束ねるAPIゲートウェイ LiteLLM に、有効なAPIキーを持つ利用者がホスト上で任意コマンドを実行できるコマンドインジェクション（CVE-2026-42271, CVSS 8.8）が存在。さらにStarletteのホストヘッダ検証回避（CVE-2026-48710）と連鎖すると認証を完全にすり抜けて未認証RCEに至り、すでにKEV登録・実環境で悪用されている。

## 🔰 初学者向け（何が・なぜ問題か）
LiteLLM は「いろいろなAI（OpenAIやAnthropicなど）への問い合わせ口を一本化する受付係（ゲートウェイ）」です。各AIのAPIキーを預かって取り次ぐため、ここを乗っ取られると預かり物のキーごと盗まれます。

弱点は2つあります。1つ目（CVE-2026-42271）は、受付係に「実行するプログラムの場所と引数」を指定できる機能があり、利用者がそこに任意のコマンドを書き込むと、受付係がサーバー上で素直に実行してしまうこと。本来は有効なAPIキーが必要ですが、2つ目（CVE-2026-48710）の「来訪者の身分証（ホストヘッダ）のチェックがザル」という欠陥と組み合わせると、身分証なし＝認証なしで1つ目の機能を呼び出せてしまいます。

結果、攻撃者は鍵もIDも持たずに受付係（サーバー）を乗っ取り、そこに保管された各AIのAPIキー（=お金がかかる資産）を盗んだり、サーバーを踏み台にできます。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）
- **CVE**: CVE-2026-42271（LiteLLMコマンドインジェクション）。連鎖相手は CVE-2026-48710（Starlette「BadHost」ホストヘッダ検証回避, CVSS 6.5）。
- **CVSS**: CVE-2026-42271 は CVSS v3.1 8.8 / v4.0 8.7（HIGH）。**CWE**: CWE-78（OSコマンドインジェクション）相当。
- **悪用状況**: CISA KEV 登録（2026-06-08）。Horizon3.ai が連鎖による未認証RCEを実証、実環境で悪用。
- **前提条件（単体）**: 有効なLiteLLM proxy APIキーを保持する認証済みユーザーが、任意の実行パスと引数配列を投入できる。**連鎖時**: ホストヘッダ検証回避により認証前提が崩れ、未認証で到達可能。
- **影響バージョン**: LiteLLM 1.74.2 〜 1.83.7未満。**修正版**: LiteLLM 1.83.7以降（加えてStarletteは1.0.1以降）。

### メカニズム（攻撃が成立する仕組み）
1. **危険なプリミティブの露出**: LiteLLMが内部処理で「実行ファイルのパス＋引数配列」をユーザー制御可能な形で受け取り、OSのコマンド実行に渡す経路を持つ。これは untrusted 入力→外部プロセス実行という典型的コマンドインジェクション面。
2. **入力検証/許可リストの欠如**: 実行対象パスや引数に対する厳格なバリデーション・許可リストが不足しているため、攻撃者は任意バイナリ/コマンドを指定できる。これにより「ホスト上での任意コマンド実行」というプリミティブを獲得。
3. **認証境界の回避（連鎖）**: 単体では有効APIキーが必要だが、配下のStarlette（ASGIフレームワーク）の**ホストヘッダ検証回避（BadHost）**を併用すると、信頼ホスト判定をすり抜け、本来認証/内部限定のはずのエンドポイントへ未認証で到達できる。認証という安全境界が、別レイヤの検証欠陥で無効化される。
4. **被害への橋渡し**: 未認証RCEが成立すると、攻撃者はLiteLLMが保持する各種**LLM APIキー/資格情報**の窃取、課金資源の悪用、ホストを踏み台にした横展開へ拡大できる。AIゲートウェイは多数の鍵を集約する“鍵束”であるため被害が増幅される。
   - 一般原理: 「危険プリミティブ（コマンド実行）＋境界回避（認証/ホスト検証バイパス）」の連鎖は、各単体では中程度でも、組み合わせで致命度が跳ね上がる典型。多層防御の各層が独立に堅牢である必要がある。
   - 本記述は概念レベルに留め、武器化手順・PoC・具体ペイロードは記載しない。

### 検知・暫定緩和・恒久対策
- **検知の着眼点**: LiteLLMプロセスから生成される予期しない子プロセス（シェル/任意バイナリ）、異常なHostヘッダを伴うリクエスト、APIキー列挙・大量利用、内部限定エンドポイントへの外部アクセス。
- **暫定緩和**: LiteLLMをインターネットから隔離、リバースプロキシで厳格なHost許可リスト/認証を前段に設置、proxyを非特権・egress制限下で実行、不要なコマンド実行系機能の無効化。
- **恒久対策**: LiteLLM 1.83.7以降＋Starlette 1.0.1以降へ更新。露出していた場合は保持していたLLM APIキーを失効・ローテーション。
- **運用示唆**: AIゲートウェイは「鍵の集約点」。侵害時の影響範囲（連携先全API）を前提に、キー分離・最小権限・ローテーション運用を整備。

## 📖 用語解説（このニュースとのつながり）
- **AIゲートウェイ（LiteLLM）**: 複数LLMへの呼び出しを一元化するプロキシ。各種APIキーを集約保持するため侵害時の被害が大きい。
- **コマンドインジェクション（CWE-78）**: ユーザー入力がOSコマンドとして実行される欠陥。本件CVE-2026-42271の中核。
- **ホストヘッダ検証回避（BadHost / CVE-2026-48710）**: リクエストのHostヘッダ検証をすり抜ける欠陥。認証/内部限定の前提を崩し、連鎖の起点となる。
- **Starlette / ASGI**: PythonのWeb基盤フレームワーク。LiteLLMが依存し、ここの検証欠陥が連鎖を可能にした。
- **KEV登録**: CISAが実悪用を確認した証左。連邦機関には期限付き対応義務、民間にも優先対応の指標。

## 影響範囲 / 推奨対応
LiteLLM 1.74.2〜1.83.7未満を運用する組織が対象。特にインターネット露出構成は未認証RCEの危険が高い。直ちにLiteLLMとStarletteを修正版へ更新し、前段に厳格なHost検証/認証を置く。露出していた場合は保持APIキーを全失効・ローテーションし、子プロセス生成や鍵の不正利用を侵害調査する。
