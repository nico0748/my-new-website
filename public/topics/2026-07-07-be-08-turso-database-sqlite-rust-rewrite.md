# Turso Database — SQLite を Rust で全面書き直し、MVCC・非同期 I/O・ベクトル検索を内蔵

> トピック: Turso Database（旧 Limbo）｜出典: Turso 公式サイト／GitHub｜種別: OSS データベースの大型プロジェクト（Beta）

## 一行サマリー

Turso が、組み込み DB の定番 SQLite を Rust でゼロから書き直した「Turso Database」を開発中（Beta）。SQLite の SQL 方言・ファイル形式・C API 互換を保ちつつ、同時書き込みのための MVCC、io_uring による非同期 I/O、拡張不要の**ベクトル検索を第一級機能**として内蔵、MCP サーバーモードまで備える。既存の本番向けには C フォークの libSQL が引き続き提供される。

## 🔰 初学者向け（何が・なぜ重要か）

SQLite は「サーバーを立てず、1つのファイルとしてアプリに埋め込めるデータベース」です。スマホアプリ、ブラウザ、組み込み機器など、世界で最も広く動いている DB の一つで、軽さと信頼性が武器です。一方、根っこが古く（C 製・単一書き込み前提）、「同時にたくさん書き込む」「最新のストレージ機能を使う」といった現代的な要求には限界がありました。

Turso Database は、その SQLite を **Rust という安全で高速な言語でイチから作り直した後継**です。SQLite のファイルや SQL がそのまま使える互換性は保ちつつ、(1) 複数の書き込みを同時にさばく仕組み（MVCC）、(2) OS の新しい高速 I/O（io_uring）、(3) AI で必須の**ベクトル検索を追加拡張なしで標準搭載**、といった現代機能を足しています。特にベクトル検索が DB に内蔵されると、「文章の意味の近さで検索する（RAG など）」処理を、外部の専用 DB を用意せず SQLite 級の手軽さで実現できます。ただしまだ Beta。今すぐ本番で使うなら、同社の実績ある C 版フォーク libSQL が推奨です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **SQLite の Rust 全面書き直し**（旧称 Limbo）。SQL 方言・ファイル形式・C API 互換を維持。
- **MVCC**（多版型同時実行制御）で同時書き込みに対応。従来 SQLite の単一ライタ制約を緩和。
- **非同期 I/O（io_uring）** をネイティブ採用し、ストレージ待ちを効率化。
- **ベクトル検索を第一級機能として内蔵**（拡張不要）。ベクトル列にコサイン／L2 距離で検索。近似検索用のベクトルインデックスはロードマップ（現状は厳密検索）。
- **MCP サーバーモード**を備え、AI エージェントから DB を直接操作させやすい。
- 2プロジェクト体制: **libSQL**（C 製の SQLite フォーク、本番実績あり・Turso Cloud を駆動）と **Turso Database**（Rust 全書き直し、Beta、長期方針）。

### 仕組み / 背景（なぜこう設計されたか）

SQLite の強みは「プロセス内・ファイルベース・ゼロ運用」だが、内部は単一ライタ（書き込みは1つずつ、WAL でも並行書き込みは1本）で、同期 I/O 前提の古い設計だった。エッジ／サーバーレスやローカルファースト（各デバイスにデータを持ち同期する構成）が広がると、「多数の同時書き込み」「AI 用のベクトル検索」「新しい OS I/O の活用」への要求が SQLite の設計限界にぶつかる。Turso は互換性を保ったまま内部を作り直し、MVCC で「読み手は書き手を待たない・書き手同士も版で分離」する並行性を得る。io_uring はシステムコールのバッチ化で I/O のオーバーヘッドを削り、非同期ランタイムと噛み合わせる。

ベクトル検索の内蔵は、RAG 時代の設計を「アプリ DB とベクトル DB の二重管理」から「1つの組み込み DB に統合」へ寄せる狙い。埋め込みを普通の列として保存し、SQL で近傍検索できれば、検索パイプラインが DB の中で完結する。トレードオフは、Rust 全書き直しはまだ Beta で、SQLite の膨大なエッジケース・互換性を完全再現するには時間が要る点。ゆえに Turso は「本番は libSQL（C フォーク、実績あり）、将来は Turso Database（Rust）」という二段構えで、互換性リスクを負わずに前進する現実解を取っている。Rust 採用はメモリ安全性と、非同期・並行を型で御しやすい点による。

### 実務での使いどころ・移行の勘所

- ローカルファースト／エッジ／組み込みで、SQLite の手軽さを保ちつつ同時書き込みやベクトル検索が欲しいケースに刺さる。
- 今日の本番採用は **libSQL** が現実解（SQLite 互換が必要な本番）。Turso Database（Rust）は Beta のため検証・将来投資として評価。
- RAG を軽量に組みたい小〜中規模用途では「アプリ DB＝ベクトル DB」の統合が運用を単純化。ただし近似インデックスはロードマップ段階で、大規模高速 ANN が要るなら専用ベクトル DB や pgvector も比較。
- MCP サーバーモードは、AI エージェントにローカル DB を安全に触らせる用途で注目。

## 📖 用語解説（このトピックとのつながり）

- **MVCC（多版型同時実行制御）**: 各書き込みをバージョンとして持ち、読み手を待たせず同時書き込みを可能にする方式。Turso が SQLite の単一ライタ制約を超える核。
- **io_uring**: Linux の高効率な非同期 I/O インターフェース。Turso の非同期 I/O 基盤。
- **ベクトル検索**: 埋め込み（数値ベクトル）の近さで検索する仕組み。RAG の中核で、Turso は拡張なしで内蔵。
- **libSQL / Turso Database**: 前者は C 製の実績あるフォーク（本番向け）、後者は Rust 全書き直し（Beta・長期方針）。

## 影響範囲 / 推奨アクション

- 本番で SQLite 互換が要る場合: libSQL を採用。Turso Database は検証・将来評価に。
- ローカルファースト／RAG 軽量用途: ベクトル内蔵の統合設計を評価。近似インデックスの成熟を注視。
- AI エージェント連携: MCP サーバーモードを試験。
