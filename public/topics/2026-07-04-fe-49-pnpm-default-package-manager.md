# pnpm が既定のパッケージマネージャに — 省容量・高速

> トピック: 開発トピック｜出典: thetshaped.dev｜種別: frontend

## 一行サマリー
pnpm はコンテンツアドレス方式のグローバルストアとハードリンクでディスクを大幅節約し、インストールも高速な次世代パッケージマネージャで、2026年のフロントエンド標準として npm/yarn を置き換えつつある。

## 🔰 初学者向け（何が・なぜ重要か）
JavaScript の開発では、外部ライブラリ（パッケージ）を大量にインストールします。従来の npm では、プロジェクトごとに同じライブラリのコピーが `node_modules` に何度も作られ、ディスクがみるみる埋まっていきました。10個のプロジェクトがあれば、同じ React が10個複製される、という無駄です。

pnpm はこの問題を「共有倉庫方式」で解決します。ライブラリの実体はパソコン内の1か所（グローバルストア）にだけ置き、各プロジェクトはそこへの「近道（ハードリンク）」を張るだけ。同じライブラリは何度使っても実体は1つなので、ディスクを劇的に節約でき、インストールも速くなります。

さらに pnpm は「厳格な依存管理」も特徴です。npm では「たまたま入っていた」パッケージを間違って使えてしまう緩さがありましたが、pnpm は宣言していないパッケージを使えないようにして、隠れた依存バグを防ぎます。省容量・高速・厳格の三拍子が支持の理由です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
pnpm はコンテンツアドレス方式のグローバルストアに各パッケージを一度だけ保存し、`node_modules` にはハードリンク/シンボリックリンクで構成する。これによりディスク使用量とインストール時間を大幅に削減。非フラットな `node_modules` 構造で phantom dependency（宣言していない依存の暗黙利用）を防ぐ。ワークスペースによるモノレポ対応も強力。具体的な速度・容量削減率は環境依存で要確認。

### 仕組み / 背景（なぜこう設計されたか）
npm/yarn のフラットな `node_modules` は重複コピーと依存の巻き上げ（hoisting）による曖昧さを生んだ。pnpm はストア共有で重複を排し、リンクベースの厳格な構造で依存の正しさを担保する。「速さ」と「正しさ」を同時に狙った設計。

### 実務での使いどころ・移行の勘所
モノレポや多数プロジェクトを持つ環境ほど効果が大きい。移行は `pnpm import` でロックファイル変換が可能だが、hoisting 前提のツールやパッケージが phantom dependency に依存していると壊れることがあり、`.npmrc` の `shamefully-hoist` 等で調整が要る場合がある。CI キャッシュ設定も見直すとよい。

## 📖 用語解説（このトピックとのつながり）
- **コンテンツアドレスストア**: 内容ハッシュでパッケージを一意管理する共有倉庫。省容量の核。
- **ハードリンク**: 実体を共有しつつ別パスから参照する仕組み。複製を避ける手段。
- **phantom dependency**: 宣言せず暗黙に使える依存。pnpm が厳格構造で防ぐ対象。
- **ワークスペース**: 複数パッケージをまとめて管理する機能。モノレポで重要。

## 影響範囲 / 推奨アクション
全 JavaScript 開発者、特にモノレポ運用チームに有効。CI 時間やディスクに悩む場合は pnpm 移行で改善が見込める。移行時は phantom dependency の破綻に注意すること。
