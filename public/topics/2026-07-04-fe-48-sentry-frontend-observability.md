# Sentry でフロントエンド可観測性 — エラー / パフォーマンス監視

> トピック: 開発トピック｜出典: thetshaped.dev｜種別: frontend

## 一行サマリー
Sentry は本番のフロントエンドで起きたエラーとパフォーマンス劣化を実ユーザー環境から収集・可視化するツールで、2026年の標準スタックに可観測性レイヤーとして組み込まれている。

## 🔰 初学者向け（何が・なぜ重要か）
アプリを公開すると、開発中には気づかなかったエラーがユーザーの環境で起きます。特定のブラウザ、特定の操作、特定の回線速度でしか出ないバグは、開発者の手元では再現できません。「ユーザーは困っているのに、こちらは何も気づかない」状態が最も怖いのです。

Sentry は、この「見えない不具合」を可視化する監視カメラのような存在です。ユーザーのブラウザでエラーが起きると、その内容・発生箇所（ソースマップで元コードまで特定）・操作の流れ（ブレッドクラム）を自動でこちらに送ってくれます。さらに「どのページが遅いか」といったパフォーマンスも計測します。

つまり Sentry は「本番で何が起きているか」を教えてくれる目です。バグ報告をユーザーからの問い合わせに頼るのではなく、先回りして検知・修正できるようになる。これが可観測性（オブザーバビリティ）の価値です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Sentry は JavaScript SDK でグローバルエラー・未処理 Promise 拒否を捕捉し、ソースマップでミニファイ後のスタックを元コードへ復元する。パフォーマンス面ではトランザクション/スパンで Web Vitals や遅延を計測、Session Replay で操作再現も可能。React/Next.js/Vite 等の統合を提供。プラン・機能詳細は要確認。

### 仕組み / 背景（なぜこう設計されたか）
フロントエンドは実行環境がユーザー側で多様なため、ローカル再現が困難。Sentry は実ユーザー環境（RUM 的発想）からテレメトリを集約することで「再現できないバグ」を可視化する。エラー監視と性能監視を一体化したのは、両者が UX 劣化の両輪だから。

### 実務での使いどころ・移行の勘所
本番リリースするアプリなら導入価値が高い。SDK 追加とソースマップアップロードの CI 設定が要点。注意点は個人情報の送信（PII マスキング設定）とイベント量によるコスト。sampling でトレース量を調整する。代替に LogRocket、Datadog RUM 等。

## 📖 用語解説（このトピックとのつながり）
- **可観測性（オブザーバビリティ）**: システムの内部状態を外部信号から把握する力。Sentry の目的。
- **ソースマップ**: ミニファイ後コードを元コードに対応づけるファイル。エラー箇所特定に必須。
- **ブレッドクラム**: エラー直前のユーザー操作履歴。原因調査の手がかり。
- **Session Replay**: 操作を録画的に再現する機能。再現困難なバグの調査に有効。

## 影響範囲 / 推奨アクション
本番運用するフロントエンドチーム全般に有効。エラーをユーザー報告頼みにしている場合、Sentry 導入で検知の先回りができる。PII マスキングとサンプリング設定を最初に固めること。
