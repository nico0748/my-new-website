# WASI 0.3 リリース — WebAssembly コンポーネントにネイティブ非同期、WASM 3.0 時代の基盤が整う

> トピック: WASI 0.3.0（2026-06-11）とコンポーネントモデルの非同期化｜出典: Bytecode Alliance / WebAssembly/WASI｜種別: WebAssembly 標準の重要更新

## 一行サマリー

サーバーやエッジで WebAssembly を動かすための標準インターフェース WASI が 0.3.0（2026年6月11日）に到達し、コンポーネントモデルへ**ネイティブな非同期（async func / stream / future）**を導入した。2025年9月に W3C 標準化された WebAssembly 3.0 と合わせ、WASM をブラウザ外の汎用ランタイムとして本格運用する基盤が整いつつある。

## 🔰 初学者向け（何が・なぜ重要か）

WebAssembly（WASM）は、もともと「ブラウザの中で高速に動く小さな実行形式」でした。近年はこれをブラウザの外——サーバーやエッジ——でも動かし、「言語を問わず、超軽量で安全に動く部品」として使おうという流れが強まっています。そのために必要なのが WASI という「WASM が外の世界（ファイル・ネットワーク等）とやり取りする共通ルール」です。

今回の WASI 0.3 の目玉は**非同期処理のネイティブ対応**。プログラムでは「ネットワークの返事を待つ間、他の仕事を進める」といった非同期の書き方が当たり前ですが、これまでの WASM 部品はこれを素直に扱えず、回りくどい仕組みが必要でした。0.3 はこれを言語レベルの標準機能として取り込み、複数の WASM 部品が非同期に協調しやすくなりました。派手なニュースではありませんが、「WASM をサーバーの本番で使う」ための土台が一段固まった、重要な一歩です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **WASI 0.3.0** が 2026年6月11日リリース。Wasmtime 43+ と jco で対応。
- コンポーネントモデルに**ネイティブ async** を追加。`async func`、`stream<T>`、`future<T>` をカノニカル ABI の基本要素として提供。
- WASI 0.2 で `wasi:io`（pollables / input-streams / output-streams）が担っていた処理が、コンポーネントモデルのカノニカル ABI へ吸収され、ネイティブに扱えるように。
- 後方互換: 実装は 0.2 と 0.3 を併存可能。0.2 を 0.3 上で仮想化（ポリフィル）することもできる。
- 前提の **WebAssembly 3.0** は 2025年9月に W3C 標準化済み。WasmGC・例外処理・末尾呼び出し・64bit メモリ・128bit SIMD など9機能を標準化。

### 仕組み / 背景（なぜこう設計されたか）

WASM をサーバーで使う際の最大の障壁は「非同期 I/O」だった。WASI 0.2 は「ストリーム」や「ポーラブル」といった低レベル部品で非同期を表現していたが、これは各言語のランタイムが自前でイベントループやポーリングを組む必要があり、コンポーネント間で非同期処理を跨ぐと相互運用が難しかった。コンポーネントモデルは「共有メモリを持たない部品同士を、型付きインターフェースで安全に合成する」思想だが、非同期がカノニカル ABI に無いと、この合成が同期呼び出しに縛られてしまう。

WASI 0.3 は `future<T>`（1回きりの結果）と `stream<T>`（連続する値）を ABI の一級市民にすることで、部品の境界を跨いだ非同期呼び出しを標準化した。これにより「非同期な HTTP クライアント部品」と「非同期な処理部品」を、言語もランタイムも異なるまま合成できる。トレードオフは移行期の複雑さ——0.2 と 0.3 が併存し、ツールチェーン（Wasmtime・jco・各言語バインディング）が追随するまでエコシステムがまだらになる。だが「共有メモリなし・型付き合成・ネイティブ非同期」という設計は、マイクロサービスやプラグイン機構を、コンテナより桁違いに軽く・安全に置き換えうる一般原理を体現している。

### 実務での使いどころ・移行の勘所

プラグイン機構（信頼できないコードを安全に実行）、エッジ関数、マルチテナントのサンドボックス、ポリグロット（多言語混在）なマイクロサービスで有望。現時点では「先行採用フェーズ」で、本番投入は Wasmtime 43+ や jco の対応状況、使う言語の WASI 0.3 バインディング成熟度を確認してから。既存の WASI 0.2 資産は仮想化で 0.3 上に載せられるため、段階移行が可能。コンテナからの全面置換ではなく、まず「軽量サンドボックスが効く部分」から適用するのが現実的。仕様・ツールの更新が速いため、依存バージョンは固定しつつ追随したい。

## 📖 用語解説（このトピックとのつながり）

- **WASM（WebAssembly）**: 言語非依存で高速・安全な実行形式。ブラウザ外運用が主題。
- **WASI**: WASM が OS 資源（ファイル・ネット）とやり取りする標準。0.3 で非同期対応。
- **コンポーネントモデル**: 共有メモリなしで WASM 部品を型付き合成する仕組み。async 追加の舞台。
- **future<T> / stream<T>**: 1回結果／連続値を表す非同期プリミティブ。0.3 の中核。
- **WasmGC / SIMD 等（WASM 3.0）**: 2025年標準化の9機能。上位言語の WASM 化を後押し。

## 影響範囲 / 推奨アクション

- 影響: WASM をサーバー/エッジ/プラグインに使う基盤層。将来のマイクロサービス設計に波及。
- 推奨: Wasmtime 43+・jco・各言語バインディングの成熟度を確認し、軽量サンドボックス用途から先行検証。0.2 資産は仮想化で段階移行。
