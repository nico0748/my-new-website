# Valkey 9.0 — クラスタでの複数論理DBとアトミックなスロット移行で超大規模へ

> トピック: Valkey 9.0 の新機能｜出典: Valkey 公式ブログ（valkey.io）｜種別: メジャーリリース

## 一行サマリー

Redis フォークの Valkey 9.0 は、クラスタモードでの複数論理データベース（multidatabase clustering）、アトミックなスロット移行、そして毎秒10億リクエスト級の超大規模ワークロードを狙う性能最適化を導入し、分散クラスタの高スループットと耐障害性を保ったまま論理DB分離を可能にした。

## 🔰 初学者向け（何が・なぜ重要か）

Valkey は、超高速なメモリ上のデータストア Redis から分かれて生まれた、オープンな後継プロジェクトです。キャッシュやセッション、順位表などに使われます。9.0 は「巨大な規模でも安全に分割して使う」ための機能が入りました。

ポイントは2つ。ひとつは「クラスタの中で複数の論理DBを持てる」こと。これまで、たくさんのサーバーをまとめた大規模クラスタでは「用途ごとにデータを論理的に分ける」使い方が制限されていました。9.0 では、高い性能を保ったまま用途別の仕切り（論理DB）を持てます。

もうひとつは「データの引っ越しが安全になった」こと。クラスタは負荷に応じてデータの担当範囲（スロット）をサーバー間で移動しますが、その移動を「途中半端な状態を作らない（アトミック）」形で行えるようになりました。移行中の不整合や取りこぼしのリスクが下がります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **multidatabase clustering**: クラスタモードで複数の論理データベースを運用可能に。分散クラスタのスループット・耐障害性を犠牲にせず論理分離を実現。
- **アトミックなスロット移行**: スロット（ハッシュスロット）の再配置を原子的に実行し、移行中の一貫性を強化。
- **大規模最適化**: 毎秒10億リクエストを超える規模のワークロードを想定した性能改善。
- リリースは Valkey 9.0（2026）。数値・機能は公式ブログに基づき、実効性能は構成・負荷依存で要検証。

### 仕組み / 背景（なぜこう設計されたか）

Redis 系のクラスタは、キー空間を 16384 個のハッシュスロットに分け、各ノードがスロットの部分集合を担当することで水平分散する。この設計は高スループットに強い一方、歴史的に「クラスタモードでは論理DB（`SELECT` で切り替える複数DB）が使えない」という制約があった。用途ごとにデータを分けたい運用は、別クラスタを立てるかキーの命名規約で擬似分離するしかなく、運用が煩雑だった。9.0 の multidatabase clustering は、この論理分離をクラスタの分散モデルと両立させることで、単一クラスタ内での用途分割を可能にする。

アトミックなスロット移行は、リシャーディング（担当範囲の再配分）時の一貫性課題への対処だ。従来のスロット移行はキー単位の移送と参照リダイレクトを組み合わせるため、移行の途中状態が観測されうる。原子的移行は、この中間状態を外部から見えないようにし、移行中のデータ取りこぼしや二重処理のリスクを減らす。トレードオフは、原子性と大規模論理分離を両立させるための内部複雑性の増加と、アップグレード時の互換性・運用手順の検証コストである。

### 実務での使いどころ・移行の勘所

「1つの大規模クラスタを複数チーム/用途で共用したいが、論理的には分けたい」現場に効く。これまで用途ごとに別クラスタを運用していた構成を集約でき、コストと運用面を単純化できる可能性がある。ただしメジャー更新のため、既存 Redis/Valkey からの移行はクライアントライブラリの対応状況、クラスタ設定の非互換、レプリケーション/永続化まわりの挙動を検証してから。毎秒10億級の性能主張は自社ワークロードで実測し、ネットワークやノード構成のボトルネックを併せて評価する。なお本 vault では後続の 9.1 も別記事で扱っている（クラウド既定インメモリストアとしての定着）。

## 📖 用語解説（このトピックとのつながり）

- **Valkey**: Redis から派生したオープンなインメモリデータストア。本トピックの主役。
- **クラスタモード**: 複数ノードでキー空間を分散する構成。高スループットの基盤。
- **ハッシュスロット**: キーを割り当てる 16384 個の区画。スロット移行の単位。
- **論理データベース（論理DB）**: 同一インスタンス内で用途を仕切る区画。9.0 でクラスタでも利用可能に。
- **リシャーディング**: スロット担当の再配分。アトミック移行で一貫性が向上。

## 影響範囲 / 推奨アクション

- 対象: キャッシュ/セッション/ランキング等で Valkey・Redis を使うバックエンド。
- 推奨: 用途別に別クラスタ運用していた構成の集約を検討。移行はクライアント対応と設定非互換を検証。
- 注意: 性能主張は自ワークロードで実測。メジャー更新の互換性・運用手順を事前確認。
