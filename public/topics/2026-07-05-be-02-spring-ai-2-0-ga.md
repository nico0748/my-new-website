# Spring AI 2.0 GA — Java の AI アプリ基盤が刷新、MCP は Streamable HTTP が既定に

> トピック: Spring AI 2.0.0 GA（2026年6月12日、Maven Central 提供開始）｜出典: Spring 公式ブログ｜種別: メジャーリリース（GA）

## 一行サマリー

Java の AI アプリ開発フレームワーク Spring AI が2026年6月12日に 2.0.0 GA へ到達。Spring Boot 4.0 を必須とし、`ChatClient` を中心 API に据え、MCP（Model Context Protocol）の WebMVC/WebFlux トランスポートを Spring AI 本体に取り込み、既定を非推奨の SSE から Streamable HTTP へ切り替えた。

## 🔰 初学者向け（何が・なぜ重要か）

いま多くのアプリが「AI に問い合わせる」機能を持ち始めています。Python にはそのためのライブラリが多い一方、企業システムの主力である Java でも「AI をきれいに組み込む標準的なやり方」が求められてきました。Spring AI は、Java で広く使われる Spring フレームワークの流儀（設定より規約、依存性注入）で AI 機能を扱えるようにする公式ライブラリです。

その 2.0 が正式版になりました。ポイントは、OpenAI・Anthropic・Google など各社の AI を「同じ書き方」で呼べること、そして AI エージェントが外部ツールに接続するための共通規格「MCP」を、Spring の中でスムーズに使えるよう整理したことです。

たとえるなら、メーカーごとにバラバラだったコンセントの形を、共通プラグに揃えてくれる変換アダプタが公式に用意されたようなもの。Java チームが、慣れた道具立てのまま AI アプリを本番品質で組みやすくなりました。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

Spring AI 2.0.0 GA は2026年6月12日に Maven Central で提供開始。主な変更点:

- **Spring Boot 4.0 が硬性依存**: Boot 3.x のプロジェクトは、まず Boot 4.0 へ移行しないと Spring AI 2.0 を採用できない。
- **API 再設計**: `ChatClient` を最も一般的なユーザー向け API に、`ChatModel` を低レベルの構成要素に位置づけ。
- **モデルサポート**: コアは OpenAI・Anthropic・Google の各ベンダー SDK を活用し、明確に定義されたチャットモデル群を標準サポート。
- **MCP 統合**: WebMVC/WebFlux トランスポート実装を MCP Java SDK から Spring AI 本体へ移設。**Streamable HTTP を既定**とし、非推奨の SSE トランスポートを置き換え。

同時期（6月）に Spring Data 2026.0.0 の GA など Spring エコシステム全体が動いている。

### 仕組み / 背景（なぜこう設計されたか）

Spring AI の設計思想は「AI 呼び出しを、Spring らしい抽象で覆う」ことにある。`ChatClient` は各ベンダー API の差異（認証、リクエスト形状、ストリーミング）を吸収するファサードで、アプリコードをベンダーロックインから守る。`ChatModel` を下位に置く二層構造は、通常は高レベル `ChatClient` を使いつつ、必要なら低レベルに降りて細かく制御する、という「良い既定＋逃げ道」の設計だ。

MCP まわりの変更が重要だ。MCP は AI エージェントがツールやデータ源に接続するためのオープン規格で、初期は SSE（Server-Sent Events）トランスポートが使われていたが、SSE は一方向・接続維持前提でスケールしづらい。Streamable HTTP は、通常の HTTP インフラ上でリクエスト/レスポンスとストリーミングを扱える方式で、ステートレスな水平スケールに向く。Spring AI が既定を切り替えたのは、この「HTTP コモディティ基盤で動く」方向への業界的移行に追随したもの。トレードオフは Boot 4.0 必須という移行コストで、レガシー資産を抱えるチームには重い。

### 実務での使いどころ・移行の勘所

Java 中心の組織で、チャットボット・RAG・ツール使用エージェントを本番品質で作るなら有力。マルチベンダー対応が要る場合、`ChatClient` 抽象でモデル差し替えが容易になる。移行の最大の壁は Spring Boot 4.0 への先行移行で、これは Java バージョンや他 Spring モジュールの互換性検証を伴う。MCP を使う既存実装は SSE から Streamable HTTP への切替影響（プロキシ/ロードバランサ設定、ステート管理）を確認したい。段階戦略として、まず新規のマイクロサービスで Boot 4 + Spring AI 2.0 を試し、知見を貯めてから中核へ広げるのが安全。

## 📖 用語解説（このトピックとのつながり）

- **Spring AI**: Java/Spring で AI 機能を扱う公式フレームワーク。本トピックの主役。
- **ChatClient / ChatModel**: 高レベルのファサード API と低レベルの構成要素。二層で抽象化。
- **MCP（Model Context Protocol）**: AI エージェントが外部ツール/データに接続する共通規格。
- **Streamable HTTP / SSE**: MCP のトランスポート方式。前者がステートレス・スケール向きで新既定、後者は非推奨化。
- **GA（General Availability）**: 正式版。本番採用が推奨される段階。

## 影響範囲 / 推奨アクション

Java で AI アプリを作る組織は採用検討の価値大。ただし Spring Boot 4.0 への先行移行が必須。MCP 利用時は Streamable HTTP への切替に伴うインフラ設定を確認。新規サービスで試行後に中核展開を推奨。
