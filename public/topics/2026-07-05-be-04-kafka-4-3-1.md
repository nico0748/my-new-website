# Apache Kafka 4.3.1 — Kafka Streams の RocksDB ネイティブメモリリークを修正

> トピック: Apache Kafka 4.3.1 リリース（2026年6月25日、バグ修正）｜出典: Apache Kafka 公式ブログ｜種別: パッチリリース（重要修正）

## 一行サマリー

分散ストリーミング基盤 Apache Kafka の 4.3.1 が2026年6月25日にバグ修正版として公開され、Kafka Streams の状態ストアが使う RocksDB のネイティブメモリリークをはじめとする重要な不具合を修正した。長時間稼働するストリーム処理でメモリが徐々に膨らむ問題を抱えていた運用者に影響する。

## 🔰 初学者向け（何が・なぜ重要か）

Apache Kafka は、大量のデータを「流れ（ストリーム）」として途切れなく処理・配送するための土台で、決済ログや行動履歴など「絶え間なく発生するイベント」を扱うシステムの心臓部として広く使われています。その上で動く Kafka Streams は、流れてくるデータをリアルタイムに集計・変換する仕組みです。

今回の 4.3.1 は「バグ修正版」です。目玉は、Kafka Streams が内部で使うデータ保管庫（RocksDB という埋め込みデータベース）で、メモリが少しずつ解放されずに溜まっていく「メモリリーク」の修正です。

たとえるなら、蛇口の栓がわずかに緩んでいて、放っておくとバケツ（メモリ）が少しずつ溢れてしまう状態。短時間なら気づきませんが、何日も動かし続けるサーバでは、いつか溢れて障害になります。4.3.1 はその緩みを締め直したアップデートです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

Apache Kafka 4.3.1 は2026年6月25日にリリースされたバグ修正版で、最も注目される修正が **Kafka Streams の RocksDB ネイティブメモリリーク**の解消。RocksDB は Kafka Streams のステートストア（集計状態・KTable などの保持先）に使われる C++ 実装の埋め込み KVS で、JVM ヒープ外の「ネイティブメモリ」を確保する。ここでの解放漏れは、JVM のヒープ監視では見えにくく、コンテナのメモリ上限（cgroup）に達して OOMKilled を招くタイプの厄介な問題だった。その他にも複数の重要修正が含まれる（詳細はリリースノート要確認）。

### 仕組み / 背景（なぜこう設計されたか）

Kafka Streams は「ステートフルなストリーム処理」を可能にするため、集計や結合の途中状態をローカルに保持する。この状態を JVM ヒープ上に置くと GC 圧やヒープサイズの制約を受けるため、RocksDB を使ってヒープ外（ネイティブメモリ／ディスク）に逃がす設計になっている。これはスループットと大規模状態の両立という利点をもたらす一方、「ネイティブメモリの管理が JVM の外で行われる」という代償を伴う。ネイティブ側のリソース（イテレータ、ブロックキャッシュ、書き込みバッファ等）が確実に解放されないと、JVM ヒープは正常なのにプロセス全体のメモリが膨らむ、という観測しづらいリークになる。

今回の修正は、この「JVM 外リソースのライフサイクル管理」の穴を塞ぐもので、長時間・高スループットで回すほど効いてくる。教訓として、埋め込みネイティブライブラリを使う設計では、リソースの明示的クローズと監視（RSS ベースの監視、RocksDB メトリクス）が不可欠だと分かる。

### 実務での使いどころ・移行の勘所

Kafka Streams を本番で長時間稼働させ、コンテナのメモリ使用量が時間とともに増えて再起動で誤魔化していた運用者は、4.3.1 への更新を優先検討したい。パッチリリースのため API 互換性の影響は小さいが、適用前にステージングで RSS（常駐メモリ）推移を計測し、修正効果を確認するのが定石。併せて RocksDB のブロックキャッシュ上限やイテレータのクローズ漏れ（アプリ側コード）も点検すると、ネイティブメモリ起因の不安定さを総合的に減らせる。Kafka 4.x 系を使うプロジェクトは、まずマイナーパッチ追随のポリシーを整えておくとよい。

## 📖 用語解説（このトピックとのつながり）

- **Apache Kafka**: 分散イベントストリーミング基盤。本トピックの土台。
- **Kafka Streams**: Kafka 上でステートフルなストリーム処理を行うライブラリ。修正の対象。
- **RocksDB**: Kafka Streams のステートストアに使う埋め込み KVS。ネイティブメモリを使う。
- **ネイティブメモリリーク**: JVM ヒープ外で解放漏れが起きる問題。ヒープ監視では検知しづらい。
- **OOMKilled**: メモリ上限超過でコンテナが強制終了される事象。ネイティブリークの典型的な帰結。

## 影響範囲 / 推奨アクション

Kafka Streams を長時間稼働させる運用者は 4.3.1 更新を優先検討。適用前にステージングで常駐メモリ推移を計測し効果を確認。アプリ側のイテレータクローズや RocksDB キャッシュ設定も併せて点検すると効果的。
