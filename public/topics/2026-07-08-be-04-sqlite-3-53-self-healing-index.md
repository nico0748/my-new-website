# SQLite 3.53 — 自己修復インデックスと制約変更可能なALTER TABLE（3.52は撤回）

> トピック: SQLite 3.53｜出典: sqlite.org 公式リリースログ / Simon Willison｜種別: マイナーメジャーリリース

## 一行サマリー

SQLite 3.52.0 は互換性の問題で撤回され、その機能は 3.53.0（2026-04-11 系）へ繰り越された。目玉は式インデックスの陳腐化問題を解く **自己修復インデックス**、NOT NULL/CHECK 制約を追加・削除できる **ALTER TABLE の拡張**、JSON 配列挿入関数、CLI の表描画改善（QRF）。最新パッチは 3.53.2（2026-06-03）。

## 🔰 初学者向け（何が・なぜ重要か）

SQLite は、サーバを立てずにファイル1つで動く超軽量なデータベースです。スマホアプリ、ブラウザ、組み込み機器、そして最近はエッジ環境まで、世界で最も多く「埋め込まれている」DB と言われます。

今回の話には二段構えがあります。まず 3.52 というバージョンが一度出そうになったものの、「古いバージョンと 100% 互換ではない挙動があった」ため撤回され、作り直して 3.53 として出し直されました。SQLite は「後方互換性を極端に大事にする」プロジェクトなので、少しでも過去を壊す恐れがあれば引っ込める、という慎重さがよく表れた出来事です。

機能面で分かりやすいのが「自己修復インデックス」。インデックスは検索を速くする索引ですが、「式インデックス（計算結果に張る索引）」は、計算のもとになる関数の挙動が変わると索引が古くなり、間違った結果を返す危険がありました。3.53 では SQLite がその陳腐化を自動で検知して直します。人が気づかないうちに壊れていた、という事故を防ぐ安全網です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **3.52.0 は撤回**: 一部の新機能・API が旧リリースと完全互換でなく、作り直しのため。予定機能は **3.53.0** に前倒し統合。
- **自己修復インデックス（self-healing index）**: 式インデックスの「陳腐化した式（stale expression index）」問題に対処。
- **ALTER TABLE 拡張**: **NOT NULL 制約と CHECK 制約の追加・削除**が可能に。
- **REINDEX EXPRESSIONS 文**: 式インデックスを再構築する新構文。
- **JSON 関数追加**: `json_array_insert()` / `jsonb_array_insert()` で JSON/JSONB 配列への値挿入をサポート。
- **CLI 改善（QRF）**: 新しい Query Result Formatter ライブラリで固定幅端末での表示を改善。Unicode 罫線文字で表を描画。
- **浮動小数点→テキスト変換**: 既定の丸め桁を **15 → 17 有効桁** に引き上げ（往復変換の精度向上）。
- **最新パッチ**: 3.53.2（2026-06-03）。

### 仕組み / 背景（なぜこう設計されたか）

**式インデックスの陳腐化**は、`CREATE INDEX ... ON t(f(x))` のように関数結果へ索引を張ると起こります。もし `f` の定義や依存する要因が変わると、索引に保存された値と実際の計算結果がズレ、クエリが誤った行を返しえます。SQLite は本来「同じ入力に同じ出力を返す決定的関数」しか式インデックスに許しませんが、現実には環境依存の関数などで陳腐化が生じ得ます。**自己修復インデックス**は、この不整合を検知したときに索引を無効化・再計算し、誤結果を防ぎます。`REINDEX EXPRESSIONS` は手動再構築の明示手段です。

**ALTER TABLE の制約追加/削除**は、従来 SQLite が苦手としてきた領域です。SQLite の ALTER TABLE は歴史的に機能が絞られ、制約を後から足すにはテーブル再作成が定石でした。NOT NULL/CHECK を直接いじれるようになったことで、マイグレーションの手間が減ります。

**3.52 撤回**は、SQLite の設計哲学「一度出したものは未来永劫読める」を守るためです。互換を壊すくらいなら番号ごと引っ込めて作り直す、という判断は、組み込み・長期保存用途で信頼される理由そのものです。

**17 有効桁化**は、IEEE 754 倍精度浮動小数点をテキストに変換して読み戻したとき、元の値を完全に復元できる最小桁数が 17 桁であることに基づく、正確な往復（round-trip）保証のための変更です。

### 実務での使いどころ・移行の勘所

- **式インデックスを使うアプリ**は自己修復により堅牢化。ただし再構築コストが発生しうるので、大規模データでは挙動を計測。
- **スキーマ移行**で NOT NULL/CHECK を後付けしたい場合、テーブル再作成の定型作業が簡略化できる。
- **浮動小数点のテキスト化に依存する処理**（ログ、シリアライズ、テスト期待値）は 15→17 桁で**出力文字列が変わる**点に注意。スナップショットテストが落ちる可能性。
- 3.52 系のビルドを取り込んでいた場合は **3.53.2 以降へ更新**すること。
- Turso（SQLite の Rust 全面書き換え）とは別系統。本記事は upstream C 実装 SQLite の話。

## 📖 用語解説（このトピックとのつながり）

- **式インデックス（expression index）**: 列そのものでなく式の計算結果に張る索引。陳腐化問題の当事者。
- **自己修復インデックス**: 陳腐化を検知して索引を再計算する 3.53 の新機構。
- **REINDEX EXPRESSIONS**: 式インデックスを明示的に再構築する新 SQL 文。
- **JSONB**: SQLite のバイナリ JSON 表現。`jsonb_array_insert()` が追加対象。
- **QRF（Query Result Formatter）**: CLI の表描画を担う新ライブラリ。Unicode 罫線で整形。
- **round-trip 精度**: 数値をテキスト化して読み戻したとき元値へ戻る保証。17 有効桁化の目的。

## 影響範囲 / 推奨アクション

- 3.52 を取り込んでいたら **3.53.2 以降へ更新**（3.52 は撤回済み）。
- 式インデックス利用箇所を洗い出し、自己修復の恩恵と再構築コストを評価。
- 浮動小数点テキスト出力に依存するテスト/シリアライズの**期待値を 17 桁基準で見直す**。
- スキーマ移行で NOT NULL/CHECK の後付けが必要なら新 ALTER TABLE を活用。
- JSON を多用するなら配列挿入関数でクエリを簡素化できるか検討。
