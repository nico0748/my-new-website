# GraphQL Federation v2 が成熟 — Apollo/Grafbase/WunderGraph が牽引

> トピック: 開発トピック｜出典: pockit.tools｜種別: backend

## 一行サマリー
GraphQL Federation v2 が成熟し、複数のサービスに分かれた GraphQL API を単一の統合グラフとして扱う手法が Apollo・Grafbase・WunderGraph 各社に支えられて実務標準になりつつある。

## 🔰 初学者向け（何が・なぜ重要か）
大きなサービスは、ユーザー管理・商品・注文…と機能ごとに別々のサーバー（マイクロサービス）に分かれていることが多いです。フロント側からすると「どこに何を聞けばいいか」が複雑になります。

GraphQL Federation は、これらバラバラの GraphQL API を裏でつなぎ合わせ、フロントには「1 つの大きな API」として見せる仕組みです。フロントは 1 回の問い合わせで、複数サービスにまたがるデータをまとめて取得できます。

たとえるなら、複数の専門店を回らずに済む「総合カウンター」です。裏では各専門店に取り次いでいますが、利用者には一箇所で完結して見えます。v2 で、この取り次ぎの設計がより堅牢で使いやすくなり、実運用に耐える成熟度に達しました。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Federation v2 は、各サブグラフを合成するスキーマ合成の柔軟性、共有型（`@shareable`）や参照解決（`@key`）の扱いが v1 より洗練されています。Apollo Router に加え、Grafbase・WunderGraph などのゲートウェイ実装が競合し、選択肢が広がりました（各実装の対応差は要確認）。

### 仕組み / 背景（なぜこう設計されたか）
モノリシックな単一 GraphQL サーバーは、組織が大きくなるとスキーマ所有権が衝突します。Federation は各チームが自分のサブグラフを独立して開発・デプロイでき、ゲートウェイが実行時（またはビルド時合成）に統合する設計です。これによりコンウェイの法則に沿った分業と、クライアントから見た統一 API を両立します。

### 実務での使いどころ・移行の勘所
複数チーム・複数ドメインを持つ中〜大規模組織に向きます。ゲートウェイが単一障害点・性能ボトルネックになり得るため、キャッシュや合成戦略の設計が重要です。小規模なら単一 GraphQL や REST/tRPC の方が単純で速い場合もあり、組織規模に見合った採用判断が要ります。

## 📖 用語解説（このトピックとのつながり）
- **サブグラフ**: Federation を構成する個別 GraphQL サービス。チーム単位で所有される。
- **ゲートウェイ / スーパーグラフ**: サブグラフを統合する層。Apollo Router 等が担う。
- **`@key` / `@shareable`**: 型の参照解決や共有を宣言するディレクティブ。合成の要。
- **スキーマ合成**: サブグラフを 1 つのグラフに束ねる処理。v2 の改善点の中心。

## 影響範囲 / 推奨アクション
マイクロサービスで GraphQL を使う中〜大規模組織のアーキテクトに影響します。チーム分業が実際の課題なら Federation v2 を検討し、ゲートウェイの可用性・キャッシュ設計を先に固め、小規模なら過剰投資を避けて単一スキーマを選ぶのが賢明です。
