# Go 1.26 — Green Tea GC がデフォルトに、多くのプログラムで GC 負荷10〜40%減

> トピック: Go 1.26 Green Tea GC｜出典: Go 公式（go.dev）｜種別: 言語ランタイム更新

## 一行サマリー

Go 1.26（2026年2月リリース）は、1.25 で実験導入された Green Tea ガベージコレクタを新しいデフォルトに昇格し、小さなオブジェクトのマーキング/スキャンをメモリ局所性と CPU スケーラビリティ改善で最適化することで、多くのプログラムの GC オーバーヘッドを10〜40%削減する。

## 🔰 初学者向け（何が・なぜ重要か）

プログラムは動くうちに「もう使わなくなったメモリ」を溜めていきます。これを自動で片付けてくれる掃除係が **ガベージコレクタ（GC）** です。Go は GC を内蔵していて、開発者がメモリ解放を手で書かなくて済むのが魅力です。ただし掃除には CPU 時間がかかり、掃除が重いとアプリの応答が一瞬もたつくことがあります。

Go 1.26 の目玉は、この掃除係を新型の **Green Tea GC** に入れ替えたことです。新型は「近くにある小さなゴミをまとめて効率よく片付ける」ように工夫されていて、掃除にかかる CPU を多くのプログラムで1〜4割ほど減らせます。しかも、開発者がコードを書き換える必要はありません。Go を 1.26 に上げるだけで、勝手に速くなる――そういう「タダで速くなる」タイプの改善です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **Go 1.25** で Green Tea GC を実験機能として導入（`GOEXPERIMENT=greenteagc` で有効化）。多くのワークロードで GC 時間が約10%減、一部で最大40%減。
- **Go 1.26（2026年2月）で Green Tea をデフォルト GC に昇格**。10〜40%の GC オーバーヘッド削減を「コード変更なし」で提供。
- 新しめの AMD64 系 CPU では、さらに約10%程度の GC オーバーヘッド改善が見込まれる。
- 当面 `GOEXPERIMENT=nogreenteagc` でオプトアウト可能だが、このオプトアウトは Go 1.27 で削除される見込み。
- 注: ワークロードによっては効果が出ない/退行するケースも報告されており（実測必須）、恩恵は「小さなオブジェクトの割り当てが多いプログラム」で顕著。

### 仕組み / 背景（なぜこう設計されたか）

Go の GC は並行マーク&スイープ方式で、到達可能なオブジェクトを「マーク」し、残りを回収する。マーク処理はヒープをたどってポインタを追う作業で、オブジェクトが小さく大量にあるとポインタ追跡が細切れになり、CPU キャッシュのミスが増える。従来 GC のボトルネックはこの「小オブジェクトのマーク/スキャンにおけるメモリ局所性の悪さ」と、マルチコアでのスケール限界だった。

Green Tea は、スキャン対象を「メモリ上で近いまとまり（スパン/ページ単位）」でグルーピングし、局所性を高めてマークする方向に再設計されている。近接オブジェクトをまとめて処理することでキャッシュ効率が上がり、CPU コア間の作業分配（ワークスティール等）もスケールしやすくなる。結果として「同じ量のゴミをより少ない CPU で片付ける」ことができる。トレードオフは、この最適化が効くのは主に小オブジェクト多発型のヒープであり、大オブジェクト中心や特殊なアロケーションパターンでは効果が限定的、あるいは僅かな退行が起きうる点。だからこそ Go チームは実験期間（1.25）を設けて広範なワークロードで検証し、十分な信頼が得られてから 1.26 でデフォルト化した。

### 実務での使いどころ・移行の勘所

Web サーバー、API ゲートウェイ、ストリーム処理など「多数の小さなオブジェクトを高頻度で割り当てる」Go サービスで恩恵が大きい。移行は Go 1.26 へ上げるだけだが、本番投入前に (1) GC 関連メトリクス（`GODEBUG=gctrace=1`、pprof、GC 停止時間・CPU 割合）で前後比較、(2) レイテンシの p99/p999 に退行がないか確認、を推奨。万一退行したワークロードは 1.26 の間は `nogreenteagc` で一時退避できるが、1.27 で消える前提なので原因調査を進めたい。アプリ側のコード変更は不要で、可観測性さえ整っていれば低リスクなアップグレード。

## 📖 用語解説（このトピックとのつながり）

- **ガベージコレクタ（GC）**: 不要メモリを自動回収する仕組み。本トピックの対象。
- **マーク&スイープ**: 到達可能オブジェクトを印付けし残りを回収する GC 方式。Go の基本方式。
- **メモリ局所性（locality）**: 近いメモリをまとめて触ると CPU キャッシュが効き速くなる性質。Green Tea の要点。
- **CPU キャッシュミス**: 必要データがキャッシュに無く主記憶へ取りに行く遅延。小オブジェクト多発時の GC を重くする要因。
- **GOEXPERIMENT**: Go の実験機能を切り替える環境変数。Green Tea の有効/無効化に使われた。

## 影響範囲 / 推奨アクション

- 影響: Go 製サービス全般、特に小オブジェクトを多用する高スループットサービス。
- 推奨: Go 1.26 へ更新し、GC メトリクスとレイテンシ p99/p999 を前後比較。退行時は原因調査（`nogreenteagc` は 1.27 で削除予定）。
