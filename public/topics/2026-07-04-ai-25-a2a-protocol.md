# A2A プロトコル — エージェント間通信の標準、150+組織が参加

> トピック: 開発トピック｜出典: Medium（AI Agent Frameworks 2026）｜種別: ai

## 一行サマリー
A2A（Agent-to-Agent）は、異なるベンダー・フレームワークで作られた AI エージェント同士が対話・協調するための通信標準で、150 を超える組織（要確認）が参加している。MCP が「エージェントとツール」を繋ぐのに対し、A2A は「エージェントとエージェント」を繋ぐ点で相補的だ。

## 🔰 初学者向け（何が・なぜ重要か）
一人の専門家が何でもこなすより、複数の専門家が役割分担して協力した方が、難しい仕事は早く正確に片づく。AI エージェントも同じで、「調査担当」「文章担当」「チェック担当」のように複数を組み合わせると強力になる。

問題は、それぞれのエージェントが別々の会社・別々の技術で作られていると、言葉が通じないことだ。A2A は、この「エージェント同士の共通語」を定める規格である。異なる出自のエージェントでも、A2A に対応していれば互いに仕事を依頼し合える。150+ の組織（要確認）が参加しているのは、「AI 同士が協調する未来」に多くのプレイヤーが賭けている証拠だ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
A2A はエージェントを「能力を公開するサービス」として扱い、能力記述（エージェントカード的なもの）・タスク委譲・進捗通知・成果物のやり取りを標準化する（要確認：詳細仕様）。これにより、あるフレームワークのオーケストレータが別フレームワークのエージェントをサブタスク実行者として呼び出せる。

### 仕組み / 背景（なぜこう設計されたか）
単一ベンダーのマルチエージェント機能に閉じると、他社の優れたエージェントを組み込めない。A2A は「エージェントの相互運用性」を狙い、MCP（ツール接続）と役割分担する形で設計された。MCP=縦（ツールへ降りる）、A2A=横（エージェント間を結ぶ）という補完関係で理解すると整理しやすい。

### 実務での使いどころ・移行の勘所
マルチエージェント構成を組む際、将来的に他社エージェントや外部サービスと連携する見込みがあるなら A2A 対応を前提に設計すると拡張が楽。ただし標準はまだ発展途上で、実装間の互換性やセキュリティ（他エージェントの信頼性・権限委譲）は要検証。まずは MCP でツール連携を固め、必要に応じて A2A を重ねる段階的採用が現実的。

## 📖 用語解説（このトピックとのつながり）
- **A2A（Agent-to-Agent）**: エージェント間通信の標準。本トピックの主役。
- **MCP**: エージェントとツールを繋ぐ標準。A2A と相補的。
- **マルチエージェント**: 複数エージェントの協調構成。A2A が横の連携を担う。
- **オーケストレータ**: サブタスクを各エージェントに割り振る司令塔役。
- **相互運用性**: 異なる実装同士が協調できる性質。A2A の中核目的。

## 影響範囲 / 推奨アクション
マルチエージェントシステムを設計する開発者・アーキテクトに影響する。他社・外部エージェントとの連携を見込むなら A2A 対応を設計初期に検討すべき。ただし仕様が成熟途上のため、権限委譲・信頼境界の設計は慎重に。当面は MCP でのツール連携を土台にし、A2A は拡張要件に応じて導入する。
