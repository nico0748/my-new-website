# Prisma「prisma-next」 — フル TypeScript 書き換えと拡張システム

> トピック: 開発トピック｜出典: prisma.io｜種別: backend

## 一行サマリー
Prisma が「prisma-next」としてクエリエンジンを含む中核をフル TypeScript へ書き換え、拡張システムを備えた次世代アーキテクチャへ刷新しつつある。

## 🔰 初学者向け（何が・なぜ重要か）
ORM（オブジェクト・リレーショナル・マッパー）とは、データベースの操作を SQL ではなく、プログラムの言葉（ここでは TypeScript）で書けるようにする道具です。Prisma はその代表格で、型安全で書きやすいことから広く使われてきました。

これまでの Prisma には弱点がありました。中核部分が Rust 製の別プロセス（クエリエンジン）として動いており、起動が重い・エッジ環境で動かしにくい・中身を拡張しづらい、といった問題があったのです。

「prisma-next」は、この Rust エンジン依存をやめて、全体を TypeScript で書き直す取り組みです。これにより起動が軽くなり、Cloudflare Workers のようなエッジ環境でも動きやすくなり、開発者が機能を拡張しやすくなります。エンジンの入れ替えという、道具の心臓部を作り替える大改修です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
prisma-next はネイティブ（Rust）クエリエンジンへの依存を排し、TS ネイティブなクエリ実行と拡張（extensions）システムへ移行します。これによりコールドスタート短縮、バンドルサイズ削減、サーバーレス／エッジ適合性の向上が見込まれます（正式版の対応範囲・バージョンは要確認）。

### 仕組み / 背景（なぜこう設計されたか）
従来 Prisma は、クエリ生成・接続管理を Rust バイナリに任せる二層構造でした。高性能だが、別バイナリの同梱・起動が必要で、サーバーレスやエッジのように「速く起動し軽くありたい」環境と噛み合いませんでした。TS 化はこの構造的制約を解き、JS ランタイム内で完結させる方向です。競合の Drizzle が軽量な TS ネイティブ ORM として支持を伸ばしたことも背景にあります。

### 実務での使いどころ・移行の勘所
サーバーレス・エッジ（Vercel、Cloudflare Workers）でのデータアクセス、軽量起動が要る環境に恩恵があります。既存 Prisma 利用者は、スキーマ定義や基本 API の互換性・移行手順を確認しつつ段階移行するのが安全です。軽量・型安全を重視するなら Drizzle との比較検討も有効で、拡張性やマイグレーション体験の差を評価すべきです。

## 📖 用語解説（このトピックとのつながり）
- **ORM**: DB 操作をプログラム言語で書く仕組み。Prisma の本体機能。
- **クエリエンジン**: SQL を生成・実行する中核。旧来は Rust 製で、prisma-next で TS 化される。
- **コールドスタート**: 未起動状態からの初回起動時間。エッジ／サーバーレス性能の鍵。
- **Drizzle**: 軽量な TS ネイティブ ORM。Prisma 刷新を促した競合。

## 影響範囲 / 推奨アクション
Node/TS でデータベースを扱う開発者、特にサーバーレス・エッジ運用者に影響します。新規はエッジ適合性を見て prisma-next か Drizzle を比較し、既存 Prisma 利用者は移行ガイドで互換性を確認して段階的に切り替えるのが望ましいです。
