# Chrome/Firefox が7月更新で多数のUse-After-Freeを修正——WebGLのサンドボックス脱出に警戒（CVE-2026-13028 ほか）

> トピック: 脆弱性ニュース ｜ 出典: SecurityWeek / Google Chrome Releases / Mozilla MFSA ｜ 種別: メモリ解放後使用（Use-After-Free）によるサンドボックス脱出／RCE

## 一行サマリー

Chrome と Firefox が7月の更新で多数の脆弱性を修正。Chrome は 147.0.7727.137/138 で 30 件を対応し、うち WebGL の Use-After-Free（CVE-2026-13028/13032）はサンドボックス脱出につながる高危険度。悪意あるWebページを開くだけで被害が成立しうるため、ブラウザの即時更新と再起動が必要。

## 🔰 初学者向け（何が・なぜ問題か）

ブラウザは、見知らぬサイトのプログラム（JavaScript や 3D 描画コード）を毎日大量に実行する“最前線”です。危険を封じ込めるため、ブラウザは「サンドボックス（砂場）」という隔離部屋の中でサイトのコードを動かし、たとえ悪意があっても部屋の外＝あなたのPC本体には手を出せないようにしています。

今回の穴の中心は「Use-After-Free（使用済みメモリの再利用）」という不具合です。例えるなら、ホテルでチェックアウトして返却したはずの部屋の鍵（メモリ）を、まだ有効なものとして使い続けてしまう状態です。フロント（プログラム）は「その部屋はもう空き」と思って別の客に割り当てるのに、古い鍵を持った人がまだ中を操作できてしまう。ここに攻撃者が細工をすると、本来触れないはずの領域を書き換え、砂場の壁を壊して外（PC本体）へ抜け出せてしまいます。これが「サンドボックス脱出」で、成功すれば悪意あるサイトを開いただけで PC 上で任意のコードが動く危険につながります。

特に危ないのが 3D 描画を担う「WebGL」という部品での Use-After-Free です。グラフィックス処理は高速化のため複雑な内部管理を行っており、解放タイミングのズレが生まれやすいためです。対策はシンプルで、ブラウザを最新版に更新し、必ず再起動して反映させることです。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **CWE**: CWE-416（Use-After-Free）が中心。多くがサンドボックス脱出可能と評価。
- **主な対象（Chrome）**: WebGL の UAF（CVE-2026-13028、CVE-2026-13032）＝細工HTMLでサンドボックス脱出。canvas の UAF（CVE-2026-7363、Linux/ChromeOS でサンドボックス内コード実行）。アクセシビリティ/Views UI の UAF（CVE-2026-7344、CVE-2026-7343、Windows でサンドボックス脱出）。
- **修正版（Chrome）**: 147.0.7727.137（Android/Linux）、147.0.7727.137/138（macOS/Windows）。計 30 件を修正。
- **Firefox**: 150.0.1、ESR 140.10.1 / 115.35.1 を公開。メモリ安全性の重大バグを修正（例: CVE-2026-7322、CVSS 7.3）。
- **前提条件**: 攻撃者が用意（または改ざん）した Web ページを被害者が開くこと。多くはユーザ操作＝「ページ閲覧」で成立し、追加権限は不要。
- **悪用状況**: 本バッチについて広範な実悪用の断定情報は現時点で確認できず（要確認）。ただしブラウザUAFは実環境ゼロデイに転用されやすく、公開後の逆算エクスプロイト（n-day）化が早い。

### メカニズム（攻撃が成立する仕組み）

ブラウザの多層防御（レンダラのサンドボックス）を突破するには「メモリ破壊で制御を奪い、隔離境界を越える」段取りが要る。UAF はその起点になりやすい。

1. **信頼境界＝サンドボックス**: ブラウザは信頼できないサイトコードをレンダラプロセスの砂場内で実行し、OS へのアクセスを厳しく制限する。攻撃者のゴールは、この砂場の内側から外（ブラウザ本体プロセスやOS）へ制御を伸ばすこと。

2. **どのコンポーネントの不備か**: WebGL/canvas/UI（Views・アクセシビリティ）といった、複雑なライフサイクル管理を持つ部品で、オブジェクトを解放した後も残った参照（ダングリングポインタ）を使ってしまう UAF が生じた。JavaScript から該当機能を巧妙に操作すると、解放と再利用の“隙間”を攻撃者が作り出せる。

3. **獲得プリミティブ**: 解放済み領域が別用途に再確保された状態で古い参照を通じて読み書きすると、攻撃者は「型の取り違え」を利用した限定的なメモリ読み書きプリミティブを得る。これを積み上げて任意アドレスの読み書き（AAR/AAW）へ発展させ、実行フローを掌握する。

4. **境界越えへの橋渡し**: WebGL/GPU 系や UI 系の UAF は、権限のより高いプロセスやサンドボックス外に近い層に触れられるため、レンダラ内 RCE から「サンドボックス脱出」への足場になりやすい。脱出後はブラウザ本体権限、さらに他の OS 脆弱性と連鎖してホスト権限へ拡大しうる。

5. **クラスの一般原理**: UAF はブラウザ・GPUドライバ・UIツールキットで反復して見つかる代表的メモリ安全性欠陥。近代ブラウザは MiraclePtr / PartitionAlloc / サイト分離等で緩和するが、GPU/描画系は緩和の適用が難しく残存しやすい。緩和は時間稼ぎであり、更新適用が本質的対策。

※具体的なヒープ整形手順・プリミティブ昇格の詳細・PoC は記載しない（概念レベル）。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: 端末のブラウザバージョンを資産管理で可視化し、未更新端末を洗い出す。ブラウザプロセスからの異常な子プロセス生成、レンダラの想定外クラッシュ多発、未知ドメインへの通信を EDR で監視。
- **暫定緩和**: 高リスク環境では WebGL/ハードウェアアクセラレーションの無効化、サイト分離の徹底、未知サイトのスクリプト制限を検討。ただし機能影響があるため本命はあくまで更新。
- **恒久対策**: Chrome を 147.0.7727.137/138 以降、Firefox を 150.0.1 / ESR 140.10.1・115.35.1 以降へ更新し、必ずブラウザを再起動して反映。組織は自動更新ポリシーを強制。
- **運用示唆**: ブラウザは日次で信頼できないコードを実行する最大の攻撃面。更新反映（再起動）まで含めた運用と、n-day 化の早さを前提とした短いパッチSLAを。

## 📖 用語解説（このニュースとのつながり）

- **Use-After-Free（CWE-416）**: 解放済みメモリを使い続ける欠陥。本バッチの中心で、制御奪取の起点。
- **サンドボックス脱出**: サイトコードの隔離部屋から抜け、ブラウザ/OS 権限へ到達すること。WebGL UAF が足場になりうる。
- **WebGL / canvas**: ブラウザ内の描画・3D 処理部品。複雑さゆえメモリ管理のミスが起きやすい。
- **レンダラプロセス / サイト分離**: サイトごとにプロセスを分け被害を封じ込める仕組み。突破されると影響が拡大。
- **n-day**: 修正公開後、パッチを逆算して作られる実用エクスプロイト。適用の遅れが致命傷になる。

## 影響範囲 / 推奨対応

Chrome/Firefox を利用する全デスクトップ・モバイル利用者が対象。悪意あるページの閲覧で成立しうるため、Chrome は 147.0.7727.137/138 以降、Firefox は 150.0.1／ESR 140.10.1・115.35.1 以降へ更新し、必ず再起動すること。組織では自動更新の強制と未更新端末の把握を行い、高リスク環境では WebGL 無効化等の追加緩和も検討する。
