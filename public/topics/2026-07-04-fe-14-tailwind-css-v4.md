# Tailwind CSS v4 ― CSS-first 設定と Lightning CSS でビルドが最大5倍高速に

> トピック: 開発トピック｜出典: fireup.pro｜種別: frontend

## 一行サマリー
Tailwind CSS v4 は、設定を JavaScript の `tailwind.config.js` から **CSS ファイル内（`@theme` など）に移す「CSS-first」設計**へ刷新し、Rust 製 **Lightning CSS** を採用してビルドを最大 5 倍高速化した（数値は要確認）。

## 🔰 初学者向け（何が・なぜ重要か）
Tailwind は、`text-blue-500` や `p-4` のような「クラス名」を HTML に書くだけでスタイルを当てられる人気の CSS フレームワークです。従来はテーマ色やフォントなどの設定を、CSS とは別の JavaScript ファイル（`tailwind.config.js`）に書く必要がありました。デザインの設定なのに CSS の外に書く、という少しちぐはぐな構造だったのです。

v4 ではこの設定を **CSS ファイルの中に直接書ける**ようになりました。色やサイズの定義が、それを使う CSS のすぐそばに置ける。家具の設計図と組み立て場所が同じ部屋にある、という感覚です。加えて内部エンジンを高速な Rust 製に載せ替え、ビルドが大幅に速くなりました。設定の場所が直感的になり、待ち時間も減る、二重の改善です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
- 設定が **CSS-first** に。`@theme` ディレクティブなどで CSS 変数として定義（`tailwind.config.js` は原則不要）。
- エンジンに **Lightning CSS**（Rust 製、Parcel チーム由来）を採用。パース・変換・ベンダプレフィクス付与を高速化。
- 公称ビルド速度は最大 **5 倍**（要確認）。ネイティブ CSS 変数を活用し、実行時のテーマ切替も柔軟。

### 仕組み / 背景（なぜこう設計されたか）
従来 Tailwind は独自のパースと PostCSS プラグイン群でスタイルを生成していました。v4 は Lightning CSS に処理を委ね、モダン CSS（ネスト、`@layer`、CSS 変数）をネイティブ前提で扱います。設定を CSS に寄せたのは、「デザイントークンは CSS 変数として持つべき」という近年の潮流に沿った判断で、フレームワーク固有の JS 設定から標準 CSS 機構への回帰でもあります。これにより他ツールとの相互運用も上がります。

### 実務での使いどころ・移行の勘所
- 新規プロジェクトは CSS-first 構成が既定。既存 v3 プロジェクトは `tailwind.config.js` の内容を `@theme` へ移す移行作業が必要。
- カスタムプラグインや複雑な JS 設定に依存している場合、互換性を要確認。
- ビルド高速化の恩恵は大規模プロジェクトほど大きい。デザイントークンを CSS 変数で持つ設計と相性が良い。

## 📖 用語解説（このトピックとのつながり）
- **CSS-first 設定**: テーマ等を CSS ファイル内で定義する v4 の中心思想。
- **`@theme`**: 色・サイズ等のトークンを CSS 変数として宣言する v4 のディレクティブ。
- **Lightning CSS**: Rust 製の高速 CSS 処理エンジン。v4 の高速化の源。
- **デザイントークン**: 色・間隔などデザインの最小単位。CSS 変数で持つのが潮流。
- **PostCSS**: 従来 Tailwind が依存した CSS 変換ツール。v4 で位置づけが変化。

## 影響範囲 / 推奨アクション
Tailwind 利用者全般に影響。既存 v3 プロジェクトは設定移行の工数を見積もること。新規はビルド速度と CSS 変数連携の恩恵が大きいため v4 を第一候補に。当ポートフォリオ（Tailwind 採用）も移行時は `@theme` への CSS 変数集約を検討する価値あり。
