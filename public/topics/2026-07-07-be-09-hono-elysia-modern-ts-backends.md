# Hono と Elysia — Express/Fastify を超える「次世代 TS バックエンド」二強の設計思想

> トピック: 次世代 TS バックエンド（Hono / Elysia）の台頭｜出典: 各公式サイト／比較記事｜種別: 設計論・エコシステム動向

## 一行サマリー

TypeScript バックエンドで Express/Fastify の後継として Hono と Elysia が二強に定着（2026年3月時点で Hono は週約180万 DL、Elysia は約50万 DL）。Hono はエッジ／マルチランタイム移植性を軸にした軽量設計、Elysia は Bun 特化と Eden Treaty による「サーバー定義からフロントの型が自動生成される」エンドツーエンド型安全を武器に、それぞれ異なる正解を提示する。

## 🔰 初学者向け（何が・なぜ重要か）

バックエンドフレームワークは「URL ごとにどんな処理を返すか（ルーティング）」を組み立てる土台です。Node.js では長年 Express が定番でしたが、型（TypeScript）との相性や速さ、そして「サーバーを動かす場所」が Node だけでなく Cloudflare Workers などの**エッジ（世界中の拠点で動く軽量な実行環境）**に広がったことで、新しい世代が求められました。

そこで出てきたのが Hono と Elysia です。**Hono** は「どこでも動く」を重視——Node でも、Bun でも、Deno でも、Cloudflare Workers でも同じコードが走り、起動が一瞬（コールドスタートが数ミリ秒）。世界中に配る API に向きます。**Elysia** は Bun（新しい高速な JavaScript ランタイム）に特化し、「サーバーで API を定義すると、フロント側がその入出力の型を自動で知る」仕掛け（Eden Treaty）で、フロントとバックの食い違いを型で潰します。どちらも Express より速く・型に強い、けれど**得意分野が違う**——というのが今の勢力図です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **採用規模**（npm、2026年3月）: Hono 週約180万 DL、Elysia 週約50万 DL。Express/Fastify からの移行先として二強化。
- **Hono**: エッジファースト設計。Cloudflare Workers / Deno / Bun / Node.js に横断移植でき、Workers でのコールドスタートは1桁 ms。軽量ルータと Web 標準（Fetch API）準拠のハンドラ。
- **Elysia**: Bun ネイティブ志向。ベンチで 200万 req/s 超を標榜。**Eden Treaty** でサーバーのルート定義から完全型付きクライアントを生成（コード生成・OpenAPI 不要でフロントが型を知る）。
- 両者とも「Express への訂正」と位置付けられ、Web 標準・型安全・高速を共通土台に、移植性（Hono）と型連携（Elysia）で差別化。

### 仕組み / 背景（なぜこう設計されたか）

Express は Node の `http` に密着し、ミドルウェアも型付けが弱く、エッジランタイム（Fetch API ベースで Node API が無い環境）では動かない。Hono は逆に **Web 標準の `Request`/`Response` を基盤**に据えることで、Node 固有 API に依存せず「Fetch 準拠ならどこでも動く」移植性を得た。ルータも軽量・高速な実装で、コールドスタートが支配的なエッジで有利。トレードオフは、抽象を薄く保つ分、Node 固有の重厚なエコシステム（既存 Express ミドルウェア資産）とは直接互換でない点。

Elysia の核は **型を"実行時スキーマ"と"静的型"の両方から導く**こと。ルート定義に付けたバリデーションスキーマから TypeScript 型を推論し、それを Eden Treaty がクライアントへ写す。結果、「サーバーの入出力の唯一の真実源」からフロントの型が導出され、API のリクエスト／レスポンス不整合をコンパイル時に検出できる（tRPC 的な体験を REST/標準的な形で得る）。Bun 特化はランタイム最適化を深く効かせ高スループットを狙う判断で、トレードオフは移植性（Node/エッジ横断）を Hono ほど重視しない点。つまり両者は「移植性を取るか、単一ランタイム最適化＋型連携を取るか」という設計上の分岐を体現している。

### 実務での使いどころ・移行の勘所

- **Hono 向き**: エッジファースト、マルチランタイム、移植性が最優先の API。Cloudflare Workers 等サーバーレス／エッジでの BFF や軽量 API。
- **Elysia 向き**: Bun 前提で、フロントと型を密結合させて最速の TS バックエンドを組みたい構成。モノレポで型を共有する自社開発に好適。
- 既存 Express 資産が厚い場合は、ミドルウェア互換の断絶を移行コストとして見積もる。段階的に BFF 層から差し替える手も。
- NestJS（重厚な DI／構造化）との比較軸は「軽量・関数的 vs フレームワーク的な構造化」。大規模チームの規約重視なら NestJS、身軽さ重視なら Hono/Elysia。

## 📖 用語解説（このトピックとのつながり）

- **エッジランタイム**: Cloudflare Workers など、世界中の拠点で動く軽量な実行環境。Fetch API ベースで Node API が無い。Hono の主戦場。
- **Eden Treaty**: Elysia のサーバー定義から完全型付きクライアントを生成する仕組み。フロント／バックの型不整合を防ぐ核。
- **コールドスタート**: 実行環境が初回起動する際の遅延。エッジで重要で、Hono は1桁 ms を実現。
- **Bun**: 高速な JavaScript ランタイム／ツールチェーン。Elysia が特化する基盤。

## 影響範囲 / 推奨アクション

- 新規 TS バックエンド: 移植性なら Hono、Bun 前提の型連携なら Elysia を第一候補に評価。
- 既存 Express/Fastify: 段階移行を計画。ミドルウェア互換の断絶をコスト計上。
- モノレポ: Elysia+Eden でフロント／バックの型共有を検討。
