# tRPC v11 — RSC ネイティブ対応で TanStack Start + tRPC がフルスタックの新定石に

> トピック: 開発トピック｜出典: pockit.tools｜種別: backend

## 一行サマリー
tRPC v11 が React Server Components にネイティブ対応し、TanStack Start と組み合わせるフルスタック TypeScript 構成が 2026 年の新しい定番として台頭している。

## 🔰 初学者向け（何が・なぜ重要か）
フロントエンド（画面側）とバックエンド（サーバー側）が通信するとき、普通は「どんなデータをやり取りするか」を両者で別々に定義します。ここがズレるとバグになります。tRPC は、サーバーで定義した関数の型をそのままフロントで使えるようにして、このズレを根本からなくす TypeScript 専用の仕組みです。

たとえるなら、サーバーとフロントが同じ「設計図」を共有している状態です。サーバー側の関数名や引数を変えると、フロントで即座に型エラーが出て気づけます。API 仕様書を別途書いてすり合わせる手間が消えます。

v11 では、React の新機能「Server Components（サーバー側で描画する部品）」と自然に統合できるようになりました。これにより、型安全なまま最新のフルスタック構成を組めるようになったのが今回の意義です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
v11 の目玉は RSC ネイティブ対応です。サーバーコンポーネント内から tRPC プロシージャを型安全に呼べ、TanStack Start のようなフルスタックフレームワークとシームレスに連携します。REST/GraphQL/gRPC と並ぶ選択肢として、TS 単一言語のプロジェクトで DX が突出します（具体 API は要確認）。

### 仕組み / 背景（なぜこう設計されたか）
tRPC はコード生成やスキーマ定義ファイルを使わず、TypeScript の型推論だけでクライアント・サーバー間の型を共有します。スキーマ言語（GraphQL SDL や Protobuf）を挟まないため、TS モノレポでは最短経路で型安全が得られます。RSC 対応は、サーバー描画とデータ取得を同一の型世界に統合する自然な進化です。

### 実務での使いどころ・移行の勘所
フロント・バックともに TypeScript の内製プロダクト、特にモノレポ構成に最適です。ただし多言語クライアントや公開 API には向きません（型は TS 前提のため）。その場合は GraphQL や gRPC が適します。TanStack Start + tRPC は少人数で速く作る新規開発の有力解ですが、外部連携要件があるなら REST/GraphQL 併用を検討します。

## 📖 用語解説（このトピックとのつながり）
- **RSC (React Server Components)**: サーバーで描画する React 部品。v11 が統合対象とした新機能。
- **型推論による型共有**: スキーマ無しでクライアント・サーバー型を一致させる tRPC の核心。
- **TanStack Start**: フルスタック TS フレームワーク。tRPC と組む新定番構成の相方。
- **モノレポ**: フロント・バックを 1 リポジトリで管理する構成。tRPC の利点が最大化される前提。

## 影響範囲 / 推奨アクション
TypeScript でフロント・バックを一貫開発するチームに影響します。TS モノレポの新規開発なら tRPC v11 + TanStack Start を有力候補にし、多言語・公開 API 要件があれば GraphQL/gRPC との使い分けを設計段階で決めるのが良いです。
