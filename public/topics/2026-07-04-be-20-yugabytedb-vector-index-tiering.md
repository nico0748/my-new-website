# YugabyteDB ベクトル索引のディスク階層化 — 10億超ベクトルを分散HNSW

> トピック: 開発トピック｜出典: medium（Google Cloud）｜種別: backend

## 一行サマリー
分散SQL DB の YugabyteDB が、ベクトル索引をディスクに階層化する仕組みを導入し、10億超のベクトルを分散 HNSW で扱えるようにした。メモリに載り切らない超大規模ベクトル検索を、分散DB上で現実的にする一手。

## 🔰 初学者向け（何が・なぜ重要か）
AI で意味検索を行うには、大量の「ベクトル（意味を数字で表したもの）」を保存し、その中から近いものを高速に探す必要があります。ベクトルが数百万個ならメモリに載りますが、10億個ともなると、すべてをメモリに置くのは費用的にも物理的にも困難です。

YugabyteDB は複数サーバーにデータを分散する「分散SQLデータベース」で、今回ベクトル索引を「メモリに全部載せず、一部をディスクに逃がす（階層化する）」仕組みを入れました。これで超大規模なベクトル検索が現実的なコストで可能になります。

たとえるなら、よく使う本は手元の机（メモリ）に、あまり使わない本は書庫（ディスク）に置き、必要なときだけ書庫から取り出す図書館運営のようなもの。全部を机に置かなくても、膨大な蔵書（10億ベクトル）を扱えるようになります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
YugabyteDB がベクトル索引（HNSW）のディスク階層化（tiering）をサポートし、メモリに収まらない規模のベクトルをディスクに退避しつつ、分散環境で HNSW 検索を成立させる。これにより10億超のベクトルを分散クラスタで扱える、とされる。索引パラメータや実測レイテンシは構成依存のため、採用時に一次情報で確認すること（要確認）。YugabyteDB は PostgreSQL 互換の分散SQL DB である点も特徴。

### 仕組み / 背景（なぜこう設計されたか）
HNSW はグラフ探索型の近似最近傍探索で高速だが、索引をメモリに保持する前提が多く、規模が上がるとメモリコストが跳ねる。ディスク階層化は「ホット（頻繁アクセス）はメモリ、コールドはディスク」という原則で、精度と速度を保ちつつコストを抑える。さらに分散化することで単一ノードのメモリ上限を超えた規模にスケールできる。

### 実務での使いどころ・移行の勘所
数億〜10億級のベクトルを扱うRAGやレコメンド、大規模セマンティック検索で有力。既に PostgreSQL 互換で運用しているなら、業務データとベクトルを同じ分散DBに統合できる利点がある。一方、中小規模なら pgvector や専用ベクトルDB（Qdrant, Milvus）の方が構築・運用が簡単なこともある。規模と運用体制で選択すること。

## 📖 用語解説（このトピックとのつながり）
- **分散SQL DB**: 複数ノードに水平分散するSQL DB。YugabyteDB が該当し、スケールの土台。
- **HNSW**: グラフ型の近似最近傍探索。ベクトル検索の主力アルゴリズム。
- **ディスク階層化（tiering）**: ホット/コールドをメモリとディスクに振り分ける手法。超大規模化の鍵。
- **pgvector**: PostgreSQL のベクトル拡張。中小規模での代替候補。

## 影響範囲 / 推奨アクション
数億〜10億級のベクトル検索を要するAI基盤担当が対象。PostgreSQL 互換で業務データと統合したい場合の候補になる。中小規模なら pgvector や専用ベクトルDBとの比較を行い、規模に見合った選択をすること。
