# A-RAG — エージェントに「階層的な検索インターフェイス」を渡す新しい RAG 設計

> トピック: A-RAG（Agentic RAG via Hierarchical Retrieval Interfaces、arXiv 2026年2月）｜出典: arXiv｜種別: RAG 手法（研究）

## 一行サマリー

A-RAG は、「一度検索して即答える」従来型 RAG ではなく、モデル（エージェント）に `keyword_search`・`semantic_search`・`chunk_read` という粒度の異なる検索ツールを直接渡し、自ら探索を繰り返して答えを組み立てさせる設計。長い調査的タスクで、検索の粒度をエージェントが適応的に選べるようにする研究提案（2026年2月の arXiv 論文）。

## 🔰 初学者向け（何が・なぜ重要か）

RAG（検索拡張生成）は、AI が答える前に「関連しそうな資料を検索して、それを読んでから答える」仕組み。社内文書 QA やサポートボットの土台になっている。ただ従来の RAG は「質問を1回検索して、出てきた数件を読んで答える」だけのことが多く、込み入った質問——複数の資料をまたいで調べたり、キーワードで絞ってから意味で探し直したり——には力不足だった。

A-RAG のアイデアは、AI に「検索の道具箱」を渡すこと。人間が調べ物をするとき、まずキーワードでざっくり絞り、次に「意味が近いもの」を探し、良さそうな文書を見つけたら全文を読む、と使い分ける。A-RAG はこの3つ（キーワード検索・意味検索・本文精読）をツールとしてエージェントに与え、AI 自身に「今はどれを使うべきか」を判断させながら、何度も検索を繰り返して答えを固めさせる。「一発検索」から「調べ物をするエージェント」への発想転換だ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

arXiv 論文（2602.03442、2026年2月）によれば、A-RAG は「階層的な検索インターフェイス」をモデルに直接公開する Agentic RAG フレームワーク。提供する3つのツールは、(1) `keyword_search`（語句一致で候補を絞る）、(2) `semantic_search`（埋め込みベースの意味的近さで探す）、(3) `chunk_read`（特定チャンクの本文を読む）。エージェントはこれらを複数回・適応的に呼び出し、粗い絞り込みから精読まで異なる粒度を行き来しながら証拠を蓄積して回答する。単発の「retrieve-then-generate」を、反復的な検索エージェントのループへ置き換える点が中核（ベンチマーク上の具体的な優位幅は原論文で要確認）。

### 仕組み / 背景（なぜこう設計されたか）

従来 RAG の弱点は「検索が固定・一発」であること。埋め込み類似度だけに頼ると、固有名詞や厳密な語句一致に弱く、逆にキーワード検索だけだと言い換え・意味的な関連を取りこぼす。さらに1回の検索で全証拠を得る前提だと、多段推論が必要な質問（複数文書の統合、条件の絞り込み）に対応できない。A-RAG は「検索を、エージェントが呼べる複数粒度のツール群として抽象化」することで、モデルの計画・推論能力を検索プロセスそのものに使わせる。これは近年の「大きなコンテキスト窓＋反復探索で長時間の調査をこなす」潮流（single-shot からマルチステップ検索エージェントへ）と一致する。トレードオフは、反復検索により LLM 呼び出し回数・レイテンシ・コストが増えること、そしてツール選択をモデルに委ねるぶん、無駄な探索や検索ループに陥らないためのガードレール（回数上限、停止条件、評価）が要ること。

### 実務での使いどころ・移行の勘所

込み入った社内ナレッジ調査、法務・技術文書の横断検索、リサーチアシスタントのような「深く調べる」用途に向く。既存のベクトル検索のみの RAG に、キーワード（BM25 等）検索と本文精読ツールを足し、エージェントに使い分けさせる形で段階導入できる。移行の勘所は、反復検索のコスト/レイテンシを抑えるため探索回数の上限と停止条件を設けること、ツール選択の質を観測・評価する仕組み（トレース、リトリーバル評価）を用意すること。単純な FAQ など一発検索で十分な用途にまで適用するとオーバーヘッドが勝つため、「深い調査が必要か」で従来型 RAG と使い分けるのが実務的。研究段階の提案である点を踏まえ、自データでの効果検証を前提に採り入れるとよい。

## 📖 用語解説（このトピックとのつながり）

- **Agentic RAG**: 検索を自律エージェントの反復行動として扱う RAG の発展形。A-RAG はその一手法。
- **keyword / semantic search**: 語句一致検索と意味（埋め込み）検索。A-RAG は両者を粒度違いのツールとして併用。
- **chunk（チャンク）**: 文書を分割した検索・読み取りの単位。`chunk_read` は該当チャンクの精読に使う。
- **retrieve-then-generate**: 「検索してから生成」の一発型 RAG。A-RAG が反復探索へ置き換える対象。

## 影響範囲 / 推奨アクション

- 深い調査を要する RAG では、単発検索から「複数粒度ツール＋反復探索」への再設計が有効な選択肢になる。
- 既存ベクトル RAG にキーワード検索・本文精読ツールを足し、エージェントに使い分けさせる形で段階検証する。
- 反復検索のコスト/レイテンシに上限・停止条件を設け、ツール選択の質を評価・トレースする。研究段階のため自データで効果検証する。
