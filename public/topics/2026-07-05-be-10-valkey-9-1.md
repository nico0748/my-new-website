# Valkey 9.1 — Redis フォークの新メジャー、クラウドの既定インメモリストアへ

> トピック: Valkey 9.1（2026年5月）と 8.1 保守継続（8.1.8 が2026年6月）｜出典: valkey-io GitHub / Redis 公式ブログ｜種別: インメモリDB（新メジャー）

## 一行サマリー

Redis のライセンス変更を機に Linux Foundation 傘下で分岐した OSS の Valkey が、2026年5月に 9.1 をリリースし、AWS ElastiCache / MemoryDB の既定として採用されるなど、クラウドのインメモリストアの標準的な選択肢に定着した。長期保守の 8.1 系も 8.1.8（2026年6月）が出るなど成熟したリリース運用を確立している。

## 🔰 初学者向け（何が・なぜ重要か）

Web サービスは、よくアクセスされるデータを高速な「メモリ上の倉庫」に置いて応答を速くします。その定番が長らく Redis でした。ところが Redis はライセンスを変更し「完全な自由ソフトではなくなった」ため、コミュニティが元のコードから枝分かれ（フォーク）して作った後継が Valkey です。

たとえるなら、みんなで使っていた公共の井戸が私有化されそうになったので、有志が同じ水源から新しい共同井戸（Valkey）を掘り直した、というイメージ。使い方や仕組みは Redis とほぼ同じで、多くの場合そのまま置き換えられます。

今回の 9.1 で Valkey はさらに成熟し、AWS の主要なマネージドサービスが標準採用するなど「迷ったらこれ」という位置づけに。ライセンスの自由さを保ちつつ、実績あるインメモリ技術を使い続けたいチームにとって重要な選択肢です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **Valkey 9.1（2026年5月）**: Redis 7.2.4 からフォークした Valkey の新メジャー系。安定した機能追加・性能改善の路線を継続。AWS ElastiCache / MemoryDB の既定として採用。
- **リリース運用の成熟**: 8.1 系を長期保守ブランチとして維持（8.1.8 が 2026年6月）。インフラソフトに期待される「長寿命ブランチ＋定期メンテ」を確立。
- **互換性**: Redis のプロトコル/コマンド互換を基本とし、既存クライアント・運用スクリプトの多くが流用可能。
- 参考として Redis 側は 8.2（2026年2月）を出しており、両系統が並存して進化している。

（各版の細目・追加機能は公式リリースノートで要確認。）

### 仕組み / 背景（なぜこう設計されたか）

Valkey 誕生の背景は技術というよりガバナンスにある。Redis がソース利用可能ライセンス（非 OSI）へ移行したことで、クラウド事業者やディストリビュータが従来通りに扱えなくなり、Linux Foundation の後ろ盾でコミュニティ主導のフォークが立ち上がった。設計上は Redis 互換を維持することが最優先で、これによって「既存資産をそのまま移せる」ことが最大の価値になっている。

技術面での進化の方向は、マルチスレッド化やメモリ効率、レプリケーション/クラスタの堅牢性など、大規模運用での実用性強化が中心。トレードオフとして、Redis 本体が独自に進める新機能（一部のモジュールや商用機能）とは差異が生じ得るため、特定機能に依存する場合は互換性の確認が要る。とはいえ、キャッシュ・セッション・レートリミット・pub/sub といった中核ユースケースでは Valkey と Redis の差は小さく、ライセンスとエコシステム（クラウド既定採用）が選定の主要因になっている。

### 実務での使いどころ・移行の勘所

キャッシュ、セッションストア、レートリミット、キュー、pub/sub など Redis の典型用途はそのまま Valkey に置き換え可能。新規はライセンスの自由度とクラウド既定採用を踏まえ Valkey を第一候補にしやすい。既存 Redis からの移行は、使用中のコマンド/モジュールが Valkey で提供されるか、クライアントライブラリの対応状況、マネージドサービスの移行手順を確認する。長期運用では 8.1 のような保守ブランチと最新メジャー（9.1）のどちらに乗るかを、安定性と新機能のバランスで決める。

## 📖 用語解説（このトピックとのつながり）

- **Valkey**: Redis 7.2 からフォークした OSS のインメモリストア。本トピックの主役。
- **Redis**: 元祖インメモリ KVS。ライセンス変更が Valkey 誕生の契機。
- **インメモリストア**: データをメモリ上に保持し高速応答する仕組み。キャッシュ等に使う。
- **ElastiCache / MemoryDB**: AWS のマネージドインメモリサービス。Valkey を既定採用。
- **保守ブランチ**: 長期にメンテされる安定版系列。8.1 系が該当。

## 影響範囲 / 推奨アクション

新規のキャッシュ/セッション基盤は Valkey を第一候補にしやすく、クラウド既定採用で導入も容易。既存 Redis はコマンド/モジュール互換とマネージド移行手順を確認のうえ移行検討。長期運用は保守ブランチと最新メジャーの選択を安定性重視で判断。
