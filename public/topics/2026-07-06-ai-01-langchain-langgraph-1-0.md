# LangChain / LangGraph 1.0 GA — Agent Middleware と Deep Agents で「エージェントの本番標準」へ

> トピック: LangChain / LangGraph 1.0｜出典: LangChain 公式ブログ｜種別: メジャー正式版（1.0 GA）

## 一行サマリー

LangChain と LangGraph が初のメジャー正式版 1.0 に到達し、`create_agent` を包む Agent Middleware（before/after 各種フック）、耐久状態と人間介在を第一級で扱う LangGraph ランタイム、そして計画・メモリ・委譲まで込みの harness である Deep Agents を軸に、エージェント開発の「安定した本番基盤」を打ち出した。

## 🔰 初学者向け（何が・なぜ重要か）

AI エージェントとは「目標を与えると、道具（検索・API・コード実行）を使いながら自分で手順を進める AI」です。LangChain と LangGraph はそれを作るための代表的な道具箱で、ついに「1.0（正式版）」になりました。1.0 は「これで本番に使ってよい」という区切りです。

大きな新しさは3つ。ひとつは Agent Middleware。エージェントの処理の「前後」に自分の処理を差し込める仕組みで、たとえば「モデルに送る前に個人情報を隠す」「道具を呼ぶ前に権限を確認する」といった横断的な処理を、本体を書き換えずに追加できます。

ふたつめは LangGraph の「耐久実行」。長い作業の途中で落ちても、保存した状態から再開できます。人間の承認を挟む（危ない操作の前に確認する）ことも標準でできます。

みっつめは Deep Agents。計画立て・記憶・仕事の分担（サブエージェントへの委譲）まで最初から入った「全部入りの土台」で、複雑なエージェントをゼロから組まずに済みます。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **LangChain 1.0**: コアをエージェントループに絞り、`create_agent` を中心に据えた。カスタマイズはサブクラス化ではなく **Agent Middleware**（`before_agent` / `before_model` / `wrap_model_call` / `wrap_tool_call` / `after_model` / `after_agent`）で行う。最新のコンテンツタイプにモデル統合を更新。
- **LangGraph 1.0**: 自動永続化による耐久状態、ワークフローの保存/再開、人間介在（HITL）を第一級 API で提供。
- **Deep Agents**: LangGraph 上に構築された本番向け harness。計画・コンテキスト管理・委譲を同梱。軽量に使いたいなら `create_agent`、フル装備なら Deep Agents、独自のグラフ形状が要るなら LangGraph へ降りる、という使い分け。
- **規模**: 月間 9,000 万 DL、Uber・JP Morgan・BlackRock・Cisco 等の本番採用。
- 1.0 GA は 2026 年（正確な日付は要確認）。数値は公式ブログに基づく。

### 仕組み / 背景（なぜこう設計されたか）

Middleware 設計の核心は「横断的関心事（ロギング、ガードレール、権限、コンテキスト整形）を、エージェント本体の制御フローから分離する」ことにある。従来はエージェントクラスを継承して挙動を上書きしていたが、これは組み合わせが効かず、複数のカスタムが衝突しやすい。Web フレームワークのミドルウェア（リクエスト前後にフックを挟む）と同じ発想で、`before_model`/`after_model` のような明確な差し込み点を用意すると、責務ごとに独立した部品を積み重ねられる。トレードオフは、フックの実行順序と副作用を理解する必要がある点だ。

LangGraph の耐久実行は、エージェントを「状態を持つ長時間プロセス」として扱う設計だ。LLM 呼び出しやツール実行は失敗・中断が前提で、各ステップ後に状態をチェックポイントとして永続化すれば、障害時に途中から再開でき、人間の承認待ちで一時停止することもできる。これは分散システムの Durable Execution（耐障害ワークフロー）の考え方をエージェントに持ち込んだものだ。Deep Agents はその上に「計画→委譲→統合」という定番パターンを標準実装として載せ、毎回ゼロから harness を書く無駄をなくす。3層（create_agent / Deep Agents / LangGraph）に分けたのは、単純な用途に重厚な抽象を強いないためで、必要な複雑さだけを選べるようにしている。

### 実務での使いどころ・移行の勘所

まず用途の複雑さで層を選ぶ——単発ツール呼び出し中心なら `create_agent`、計画と委譲が要る複雑タスクなら Deep Agents、独自のグラフ制御が要るなら LangGraph。0.x からの移行は、サブクラス化していたカスタムを Middleware フックへ移す作業が中心で、フック順序の設計が肝。長時間・多段のエージェントは LangGraph の永続化と HITL を使い、危険操作（送金・削除・外部書き込み）に人間承認境界を必ず置く。本番採用では、Middleware でガードレールと監査ログを一元化し、モデル/ツール呼び出しをコスト・レイテンシとともに計測する。

## 📖 用語解説（このトピックとのつながり）

- **エージェント**: 道具を使って自律的に手順を進める AI。本トピックの構築対象。
- **Agent Middleware**: エージェント処理の前後に差し込む横断処理。1.0 のカスタマイズ手段。
- **耐久実行（Durable Execution）**: 落ちても状態から再開できる実行方式。LangGraph の核。
- **HITL（人間介在）**: 危険操作前に人が承認する仕組み。第一級 API で提供。
- **Deep Agents**: 計画/記憶/委譲を同梱した本番向け harness。

## 影響範囲 / 推奨アクション

- 対象: LLM エージェントを開発/運用するチーム。
- 推奨: 用途の複雑さで create_agent / Deep Agents / LangGraph を選択。カスタムは Middleware へ集約。危険操作に HITL 境界を設置。
- 注意: 0.x からはサブクラス化を Middleware へ移行。フック順序と副作用を検証。GA 日付は要確認。
