# Vite + Rolldown + Oxc の統合ツールチェーン ― 開発から本番まで一貫した基盤へ

> トピック: 開発トピック｜出典: vite.dev｜種別: frontend

## 一行サマリー
Vite 8 は、バンドラ **Rolldown**（Rust）とパーサ/トランスフォーマ群 **Oxc**（Rust）を組み合わせ、開発サーバから本番ビルドまでを一貫した Rust 製ツールチェーンで統一する方向へ舵を切った。ツール間の挙動差を減らし、速度と一貫性を両立する狙い（詳細は要確認）。

## 🔰 初学者向け（何が・なぜ重要か）
これまでの Vite は「開発中は esbuild（Go 製）」「本番ビルドは Rollup（JS 製）」というように、**場面ごとに違う道具**を使い分けていました。台所で言えば、下ごしらえと本調理で別のコンロを使うようなもの。だいたいは問題ないのですが、「開発中は動くのに本番だけ壊れる」という嫌なバグが、この道具の違いから生まれることがありました。

Vite 8 は、下ごしらえも本調理も同じ「Rust 製の一体型コンロ」（Rolldown + Oxc）に統一していく方針です。パース（コードを読む）、変換（書き換える）、バンドル（束ねる）という一連の工程を、同じ思想・同じ実装で通します。これにより、速さだけでなく「開発と本番で結果が食い違わない」という安心感が得られます。地味ですが、実務では速度以上にありがたい変化です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
- **Oxc**（The Oxidation Compiler）: Rust 製のパーサ・リンタ（Oxlint）・フォーマッタ・トランスフォーマ群。
- **Rolldown**: Oxc を内部に用いる Rust 製バンドラ。Vite 8 の本番バンドラ。
- Vite・Rolldown・Oxc が同一エコシステム（VoidZero 系、要確認）で統合され、パース〜バンドルを共通基盤で処理。

### 仕組み / 背景（なぜこう設計されたか）
従来の JS ツールチェーンは、Babel・esbuild・Rollup・ESLint・Prettier などが**それぞれ独立にコードを解析**していました。同じソースを何度もパースする無駄と、ツール間の解釈差が課題でした。Oxc は「一度パースした AST（抽象構文木）を複数の工程で使い回す」統合設計で、この無駄と食い違いを同時に解消します。Rust 採用は速度のためだけでなく、単一の型定義・単一の実装で一貫性を担保するためでもあります。

### 実務での使いどころ・移行の勘所
- 大規模モノレポや CI 重視の現場で、ビルド一貫性とスピードの両取りが狙える。
- Oxlint / oxfmt（別項参照）と組み合わせると、リンタ・フォーマッタまで含めた高速統一環境が作れる。
- まだ発展途上の領域が残る可能性があり、既存 Rollup/ESLint プラグイン資産との互換性は要確認。段階移行が無難。

## 📖 用語解説（このトピックとのつながり）
- **Oxc**: Rust 製の統合ツールチェーン。パース・変換・リント・整形を担う基盤。
- **Rolldown**: Oxc を用いる Rust 製バンドラ。Vite 8 の本番バンドラ。
- **AST**: コードを木構造で表した中間表現。ツール共有の鍵。
- **統合ツールチェーン**: 別々のツールを共通基盤に束ねる考え方。無駄と食い違いを削る。
- **esbuild/Rollup**: 従来 Vite が使い分けていた別々のバンドラ。統一の対象。

## 影響範囲 / 推奨アクション
Vite ユーザ全般、とくにビルド一貫性で悩んできたチームに好影響。まずは開発/本番の成果物差を検証しつつ、Oxlint 等の周辺ツールも合わせて評価すると恩恵を最大化できる。
