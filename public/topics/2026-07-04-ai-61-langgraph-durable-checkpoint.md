# LangGraph の耐久実行（Durable Execution）― チェックポイントで“落ちても続きから”走るエージェント

> トピック: 開発トピック｜出典: changelog.langchain.com / github langchain-ai/langgraph｜種別: ai

## 一行サマリー
本番エージェント基盤の定番 LangGraph は、各ノード実行ごとに状態を自動保存する **チェックポイント**（保存先はメモリ / SQLite / PostgreSQL を選択可）により、サーバー再起動や障害があっても「中断した続きから」処理を再開できる耐久実行を提供する。Uber・LinkedIn・Klarna 等の本番運用で鍛えられ、長時間・信頼性重視のワークフローの土台になっている。

## 🔰 初学者向け（何が・なぜ重要か）
AI エージェントに「10 個の手順」を任せたとき、3 手順目でサーバーが落ちたらどうなるでしょう。普通は最初からやり直しで、時間もお金（API 課金）も無駄になります。LangGraph の **チェックポイント** は、各ステップが終わるたびに“しおり”を挟むように状態を保存します。だから落ちても、次に立ち上げたときに 4 手順目から再開できます。

長い出張の行程を、各都市に着くたびにメモしておくイメージです。途中でスマホが電池切れになっても、メモを見れば「次はどこへ行くか」がすぐ分かる。エージェントを本番で使うほど、途中で止まる・人間の承認を待つ・数時間走る、といった場面が増えます。そのとき「やり直しにならない」ことは、信頼性とコストの両面でとても重要です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
- LangGraph は **1.0（2025-10 GA）** で耐久実行を柱に据え、**各ノード遷移を自動チェックポイント**する永続化層を提供。保存先は **in-memory / SQLite / PostgreSQL** から選択（開発〜本番でスケール）。
- 「10 手順中 3 手順目でサーバー再起動 → 続きから再開」といった復帰が可能。**Human-in-the-loop（HITL）** も interrupt により、途中で人の承認を挟んで再開できる。
- Uber・LinkedIn・Klarna など本番採用で成熟。2026 年時点で 1.x 系がプロダクション標準として広く使われる（1.2 等の続くマイナー更新あり、詳細は公式 changelog 参照、要確認）。

### 仕組み / 背景（なぜこう設計されたか）
LangGraph の設計思想は「暗黙のチェーンを、明示的なグラフに置き換える」ことです。エージェントの処理をノード（処理単位）とエッジ（遷移）で表し、各ノード実行後に **状態（state）をチェックポイントとして永続化**します。これにより、実行はステートフルかつ再開可能になります。再起動時は最後のチェックポイントを読み込み、そこから決定的に処理を継続できる——これが「耐久実行（durable execution）」です。

なぜ重要か。LLM エージェントは非決定的で、外部 API 呼び出し・長時間タスク・人間の承認待ちを含み、途中で失敗・中断しやすい。チェックポイントがないと、失敗＝全やり直しになり、信頼性・監査性・コストが破綻します。保存先を選べる設計は、開発時は手軽な in-memory、軽量運用は SQLite、本番は PostgreSQL、と段階的にスケールさせるためです。トレードオフ／注意点として、チェックポイントは「状態のスナップショット再開」であり、外部副作用（すでに送信したメール等）まで巻き戻せるわけではない点、そして厳密な "durable execution"（Temporal のようなワークフローエンジン）と比べると保証モデルが異なる、という議論もあります。用途に応じて、冪等性（同じ処理を再実行しても安全にする設計）を併用することが実務では重要です。

### 実務での使いどころ・移行の勘所
- 長時間・多ステップ・人間承認を挟むエージェント（審査、オンボーディング、複雑な業務自動化）で信頼性を担保できる。
- まず in-memory/SQLite で PoC、本番は PostgreSQL チェックポイントに切替、という段階移行がしやすい。
- 外部副作用を伴うノードは冪等に設計し、再開時の二重実行を防ぐ。厳密な耐久保証が要る領域は専用ワークフローエンジンとの比較検討を。

## 📖 用語解説（このトピックとのつながり）
- **LangGraph**: グラフでエージェントを構築する LangChain 系の本番向け基盤。本トピックの主役。
- **チェックポイント**: 各ノード実行後に状態を保存する仕組み。再開の要。
- **耐久実行 (durable execution)**: 障害・再起動をまたいで処理を継続できる実行モデル。
- **Human-in-the-loop (HITL)**: 途中で人の承認・介入を挟む仕組み。interrupt で実現。
- **冪等性 (idempotency)**: 同じ処理を再実行しても結果が壊れない設計。再開時の二重実行対策。

## 影響範囲 / 推奨アクション
本番でエージェントを運用する AI 開発者に直結する。長時間・多ステップ・承認待ちを含むワークフローでは、チェックポイントによる再開性が信頼性とコストを大きく改善する。PoC は in-memory/SQLite、本番は PostgreSQL に切替え、副作用のあるノードは冪等に設計するのが推奨。厳密な耐久保証が必要なら専用エンジンと比較する。
