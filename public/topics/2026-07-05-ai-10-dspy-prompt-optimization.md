# DSPy — プロンプトを「書く」から「最適化する」へ、宣言的 LLM プログラミング

> トピック: DSPy（宣言的 LLM プログラミング / 自動プロンプト最適化）2.x・MIPROv2｜出典: DSPy 公式 / 研究論文｜種別: プロンプト技術・フレームワーク

## 一行サマリー

Stanford 発の DSPy は、LLM パイプラインを「手書きのプロンプト」ではなく「signature（何をするか）」として宣言し、オプティマイザ（MIPROv2 等）が学習データを使ってプロンプトと few-shot 例を自動最適化する枠組み。手動プロンプトに対し構造化タスクで10〜40%の品質改善が報告され、2026年も 2.x API とオプティマイザ改良で実運用パターンが整いつつある。

## 🔰 初学者向け（何が・なぜ重要か）

LLM に良い仕事をさせるには「プロンプト（指示文）」の書き方が重要ですが、良い文面を手探りで探すのは職人技で、モデルが変わるとまたやり直し、という消耗がありました。

DSPy はこの発想を変えます。プロンプトの文面を人が磨くのではなく、「入力はこれ、出力はこれ」という**やりたいことの仕様（signature）だけを宣言**し、あとは自動の最適化装置に「一番成績の良いプロンプトと例文」を探させます。たとえるなら、料理のレシピを一言一句手直しする代わりに、「材料と完成イメージ」を決めておけば、試作を繰り返して最適な手順を自動で見つけてくれる調理ロボットのようなもの。

嬉しいのは、手作業のプロンプト調整に比べて品質が上がりやすく、モデルを乗り換えても仕様はそのままに再最適化できること。プロンプトエンジニアリングを「勘と根性」から「測って改善する工学」に近づける道具です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **宣言的プログラミング**: DSPy（Declarative Self-improving Python）は LLM パイプラインを「プロンプトテンプレート」ではなく「ML プログラム」として扱う。各モジュールは signature（入力→出力の仕様）を持ち、「何をするか」と「どう言うか（プロンプト）」を分離する。
- **オプティマイザ（Teleprompters）**: MIPROv2 などが、ラベル付き開発セット上の指標を最大化するように、プロンプト文面と few-shot 例を自動探索・最適化する。
- **効果**: 構造化タスクで手動プロンプト比10〜40%の品質改善、研究では事実正確性で最大30〜45%改善・幻覚（hallucination）約25%削減の報告。
- 2026年3月時点で 2.x API と最新オプティマイザ改良、実運用（デプロイ）パターンを整備。

### 仕組み / 背景（なぜこう設計されたか）

手書きプロンプトの問題は「再現性」と「移植性」にある。文面のわずかな違いで結果が変わり、良い文面を見つけても別モデルでは通用しないことが多い。DSPy の中核思想は、機械学習の「モデルを直接いじらず、目的関数に対してパラメータを最適化する」考え方をプロンプトに適用することだ。signature が「関数の型」、モジュールが「実装」、オプティマイザが「学習器」に相当し、開発セット上の指標（正解率など）を最大化するようにプロンプト＝可変パラメータを探索する。

これにより、プロンプトは人間が磨く成果物から、データ駆動で導出される中間表現へと変わる。トレードオフは、最適化に評価用データと指標（メトリック）の用意が要る点、探索に計算/API コストがかかる点、そして「なぜこの最適プロンプトになったか」の解釈性が下がる点。一方で、モデル更新時に signature を保ったまま再最適化できる移植性、そしてパイプライン全体（複数モジュール）を一貫して最適化できる合成性は、複雑な LLM アプリで大きな利点になる。

### 実務での使いどころ・移行の勘所

分類・抽出・構造化生成・多段推論など「正解や評価指標を定義しやすいタスク」で効果が高い。導入には、(1) タスクを signature として分解、(2) 評価メトリックと開発セットを用意、(3) オプティマイザで最適化、の流れを整える。最適化コスト（API 呼び出し）と評価データ整備が初期の壁なので、まず高頻度・高価値なモジュールから適用する。プロンプトが自動生成されるため、本番前に出力の検証・ガードレールを併設する。エージェント SDK や MCP と併用し、「ツール選択や推論の質」を DSPy で最適化する構成も相性が良い。

## 📖 用語解説（このトピックとのつながり）

- **DSPy**: 宣言的に LLM パイプラインを書き、プロンプトを自動最適化する枠組み。本トピックの主役。
- **signature**: モジュールの入力→出力仕様。「何をするか」を宣言する部分。
- **オプティマイザ / MIPROv2**: 指標を最大化するようプロンプトと few-shot 例を探索する仕組み。
- **few-shot 例**: プロンプトに含める手本の例。最適化の対象になる。
- **メトリック / 開発セット**: 最適化の目的関数と評価用データ。DSPy 運用の前提。

## 影響範囲 / 推奨アクション

評価指標を定義できる LLM タスクは、DSPy で品質と移植性を高められる。導入は signature 分解・メトリック整備・段階最適化の順で、高価値モジュールから着手。最適化コストと出力検証（ガードレール）を織り込み、エージェント/MCP 構成との併用も検討する。
