# SurrealDB ― ドキュメント＋グラフ＋KV＋リレーショナルを1つに、AIエージェント時代に再注目

> トピック: 開発トピック｜出典: surrealdb.com｜種別: backend

## 一行サマリー
SurrealDB は、ドキュメント・グラフ・キーバリュー・リレーショナルの複数モデルを1つのエンジンで扱い、ライブクエリ（変更の即時購読）も備えるマルチモデルDB。普及は緩やかだが、2026 年の DB 動向では「AI エージェントがこの抽象化を扱いやすい」として再評価が進んでいる。エージェントが多様なデータ形を横断的に触る用途で、モデルを跨ぐ柔軟さが強みになる。

## 🔰 初学者向け（何が・なぜ重要か）
データベースには「表形式（リレーショナル）」「文書のまま（ドキュメント）」「点と線のつながり（グラフ）」「単純な鍵と値（KV）」など、いくつも“流派”があります。普通はどれか1つを選び、足りなければ複数の DB を組み合わせます。SurrealDB は、それらを**1つのデータベースの中でまとめて**扱えるのが特徴です。用途ごとに道具箱を分けず、万能ナイフ1本で済ませるイメージです。

なぜ今注目かというと、AI エージェント（自律的にタスクをこなす AI）が絡むからです。エージェントは「ユーザー情報（表）」「会話ログ（文書）」「関係性（グラフ）」など、性質の違うデータを次々に触ります。DB が複数モデルを1つで扱えると、エージェントにとって“覚えることが1つで済む”ので相性が良い、というわけです。さらに「データが変わったら即座に知らせる（ライブクエリ）」機能も、リアルタイムに反応するエージェントに向いています。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
- SurrealDB は **マルチモデル**（document + graph + key-value + relational）を単一エンジンで提供し、**ライブクエリ**（変更ストリームの購読）をサポート。
- 2026 年の DB 動向まとめでは「採用は緩やかだが、AI エージェントが抽象化を扱いやすいため再注目されている」と評価（具体的な最新バージョンの機能差分は公式 release notes 参照、要確認）。
- 単一のクエリ言語（SurrealQL）で、テーブル・グラフ関係・ネストしたドキュメントを横断的に操作できる設計。

### 仕組み / 背景（なぜこう設計されたか）
マルチモデル DB の狙いは「ポリグロット・パーシステンス（用途ごとに別 DB を並べる）」の運用負担を減らすことです。表・文書・グラフ・KV をそれぞれ別製品で持つと、整合性の担保・運用・学習コストが分散します。SurrealDB は1エンジン・1クエリ言語に統合することで、この分散を1か所に集約します。ライブクエリは、DB 側が変更を購読者へ push する仕組みで、アプリがポーリング（定期的に問い合わせる）しなくてよくなります。

AI エージェント文脈での適合性は、エージェントの「メモリ」要件と重なります。エージェントは短期・長期の記憶、関係グラフ、埋め込み（ベクトル）などを扱い、これらを1つの一貫した抽象で読み書きできると実装が単純になります。トレードオフは、汎用性ゆえに各モデル専用 DB（ネイティブなグラフ DB や成熟した RDB）ほど各領域を深く最適化しきれない可能性と、採用実績・エコシステムがまだ発展途上である点です。ミッションクリティカルな大規模運用では、成熟度の評価が欠かせません。

### 実務での使いどころ・移行の勘所
- AI エージェントのメモリ層、プロトタイプ、複数モデルを横断するアプリで「DB を1つに集約したい」場合に有効。
- 変更をリアルタイム反映したいアプリ（コラボ、通知、エージェントの状態監視）でライブクエリが効く。
- 大規模・高可用が最優先の基幹系では、成熟した専用 DB との比較検証を行い、SurrealDB は新規・実験用途から。

## 📖 用語解説（このトピックとのつながり）
- **SurrealDB**: 複数データモデルを1エンジンで扱うマルチモデル DB。本トピックの主役。
- **マルチモデル DB**: document/graph/KV/relational を単一製品で扱う DB カテゴリ。
- **ライブクエリ**: データ変更を購読者へ即時 push する仕組み。リアルタイム用途に有効。
- **ポリグロット・パーシステンス**: 用途ごとに複数 DB を使い分ける方針。マルチモデルはこれを集約する。
- **エージェントのメモリ**: AI エージェントが扱う短期/長期記憶・関係・ベクトル等。マルチモデルと相性が良い。

## 影響範囲 / 推奨アクション
AI エージェントや複数データモデルを扱うバックエンド開発者に参考になる。既存の基幹 DB を急いで置き換える話ではないが、エージェントのメモリ層や新規プロトタイプで「DB 集約」を試す価値がある。本番採用は各モデルの成熟度と運用実績を評価してから。
