# PydanticAI が台頭 — 型安全でネイティブに MCP/A2A/永続実行を備えるエージェント基盤

> トピック: 開発トピック｜出典: alicelabs.ai｜種別: ai

## 一行サマリー
Python のデータ検証ライブラリ Pydantic の作者チームが手がける AI エージェントフレームワーク PydanticAI が、型安全・MCP/A2A ネイティブ対応・永続実行を武器に 2026 年の有力候補として台頭している。

## 🔰 初学者向け（何が・なぜ重要か）
AI エージェントとは「LLM に道具（ツール）を持たせ、目的に向かって自分で手順を判断させる仕組み」です。ただし LLM の出力は自由な文章なので、そのままではプログラムに渡すときに形が崩れたり、想定外の値が返ってきたりします。

PydanticAI はここに「型」という設計図を持ち込みます。Pydantic は Python で「このデータはこういう形でなければならない」を宣言し、外れた入力を弾く定番ライブラリです。その仕組みをエージェントの入出力に効かせることで、LLM の気まぐれな出力を安全な構造化データに落とし込めるのが強みです。例えるなら、フリーハンドで書かれた住所を、必ず郵便番号・都道府県・番地の枠に収める用紙を用意するようなものです。

「Pydantic をすでに使っているなら学習コストが低い」「FastAPI と同じ設計思想」という点が、Python 開発者にとって採用しやすい理由になっています。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
PydanticAI は、エージェントの依存性注入・ツール定義・構造化出力を型付きで扱えるフレームワーク。出典によれば、MCP（Model Context Protocol）と A2A（Agent-to-Agent）をネイティブにサポートし、さらに「永続実行（durable execution）」を備えるとされる。永続実行は長時間・複数ステップのワークフローを途中で中断・再開できる仕組みで、Temporal などのワークフローエンジンと連携する設計思想に近い。具体的な対応バージョン・連携先の詳細は要確認。

### 仕組み / 背景（なぜこう設計されたか）
従来のエージェントフレームワーク（LangChain 等）は抽象化が厚く、内部で何が起きているか追いにくいという批判があった。PydanticAI は「FastAPI がバックエンド開発にもたらした体験を、GenAI 開発にも」という発想で、薄いレイヤーと型安全を重視する。型で出力を縛ることで、LLM のハルシネーションや形式崩れをコード側で早期に検出でき、テストや保守がしやすくなる。トレードオフは、厳密な型定義を書く初期コストがかかる点。

### 実務での使いどころ・移行の勘所
Python 中心で、出力の正確さ・監査可能性が重要な業務（データ抽出、社内ツール連携、RAG バックエンド）に向く。MCP ネイティブなので既存の MCP サーバ資産をそのまま道具として使える。LangChain/LlamaIndex からの移行では、チェーンや大量の抽象化に依存したコードの再設計が必要。軽量な Agent SDK 系（項目33参照）との比較検討も推奨。

## 📖 用語解説（このトピックとのつながり）
- **Pydantic**: Python のデータ検証・設定管理ライブラリ。PydanticAI の型安全の土台。
- **MCP（Model Context Protocol）**: LLM と外部ツール/データを繋ぐ標準規格。PydanticAI がネイティブ対応することで道具連携が容易になる。
- **A2A（Agent-to-Agent）**: エージェント同士が通信する標準プロトコル。複数エージェント協調に使う。
- **永続実行（durable execution）**: 長時間ワークフローを中断・再開できる実行モデル。エージェントの信頼性を高める。
- **構造化出力**: LLM の出力を決まった型のデータに整える仕組み。PydanticAI の中核。

## 影響範囲 / 推奨アクション
Python でエージェントを組む開発者・チームに影響。出力の正確さと保守性を重視するなら評価に値する。既存 LangChain 資産がある場合は段階移行を検討し、まず小さなツール連携で型安全の効果を試すとよい。
