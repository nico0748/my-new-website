# Windows HTTP.sys「HTTP/2 Bomb」CVE-2026-49160（公開済みゼロデイ・メモリ枯渇DoS）

> トピック: 脆弱性ニュース ｜ 出典: SOCRadar / BleepingComputer / Tenable / Talos ｜ 種別: サービス拒否（CWE-770 リソース消費の無制限）

## 一行サマリー

Windows のカーネルドライバ HTTP.sys に、HTTP/2 のヘッダ圧縮とフロー制御を悪用してサーバのメモリを急速に枯渇させる DoS 脆弱性 CVE-2026-49160（CVSS 7.5）が判明。少量のデータで巨大なメモリ確保を強い、解放を阻止できる。パッチ前に公開され「悪用される可能性が高い」と評価。IIS・Exchange OWA・RD ゲートウェイなど HTTP.sys を使う広範なサービスが影響を受ける。

## 🔰 初学者向け（何が・なぜ問題か）

Windows サーバの「Web の受付窓口」を担う中核部品が HTTP.sys です。IIS（Web サーバ）や Exchange のメール画面、リモート接続のゲートウェイなど、たくさんの機能がこの同じ受付窓口の上で動いています。だからここが倒れると、その上の多くのサービスが一斉に止まります。

今回の「HTTP/2 Bomb」は、最近の通信規格 HTTP/2 の仕組みを逆手に取ります。HTTP/2 はデータを圧縮して効率よく送れるのですが、攻撃者は**ごく小さなリクエストを送るだけで、サーバ側に大量のメモリを確保させる**ことができます。さらに「私の受信窓口は今いっぱいなので送らないで」と嘘の合図を出し続けることで、サーバが確保したメモリを解放できないように縛ります。

たとえるなら、小さな注文書を1枚渡すだけで、店側に巨大な倉庫を用意させ、しかも「まだ引き取れない」と言い張って倉庫を占有し続けるようなものです。これを繰り返すとメモリが尽き、サーバが落ちます（サービス拒否＝DoS）。検証ではわずか45秒で64GBのメモリを使い切った例もあり、可用性への脅威は大きいです。なお、これは「乗っ取り（コード実行）」ではなく「サービス停止」を狙う攻撃です。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **CVE**: CVE-2026-49160（通称「HTTP/2 Bomb」）
- **CVSS**: 7.5（High）。Microsoft 評価は「Exploitation More Likely」。パッチ提供前に公開済み（公開済みゼロデイ）。
- **種別**: サービス拒否 / リソース消費の無制限（CWE-770 Allocation of Resources Without Limits or Throttling）。コード実行ではなく可用性影響。
- **影響範囲**: Windows の HTTP プロトコルスタック HTTP.sys。これを利用する IIS、Exchange OWA、リモートデスクトップ ゲートウェイ、Web Application Proxy、WSUS、RD Web Access、.NET の HttpListener を使う独自/サードパーティサービス。
- **前提条件**: HTTP/2 を有効にして待ち受けるエンドポイントに、ネットワークから到達できること。**認証不要**。
- **修正版**: 2026年6月10日（Patch Tuesday）で修正。
- **暫定緩和の目安**: レジストリ `MaxRequestBytes` を既定の16,384バイト（16KB）相当に保つ構成は影響を受けない（65,534バイト以下に抑える運用が緩和に寄与）。**自環境での適用前検証は要確認**。

### メカニズム（攻撃が成立する仕組み）

増幅（amplification）型 DoS の一種で、「圧縮による展開比」と「フロー制御による解放阻止」を組み合わせて成立する。概念レベルで分解する。

1. **HPACK による増幅**: HTTP/2 はヘッダを HPACK で圧縮する。攻撃者は、受信側が展開・テーブル登録する過程で**1リクエストあたり多数の小さなメモリ確保**を誘発する細工リクエストを送る。送信量は小さいのに、サーバ側の確保量は不釣り合いに大きくなる（非対称な増幅）。
2. **フロー制御による解放阻止**: HTTP/2 にはフロー制御があり、受信ウィンドウを0にすると相手に「今は送らないで」と伝えられる。攻撃者は自分の受信ウィンドウを0に保ち、**サーバが既に確保したメモリを送出・解放できない状態**に固定する。サーバは「送る予定のデータ」を抱えたまま待たされる。
3. **被害への橋渡し**: 1と2を多数コネクションで並行・反復すると、確保されたまま解放されないメモリが積み上がり、RAM を食い尽くしてプロセス（ひいてはサービス）がクラッシュする。検証では IIS で約45秒・64GB、Envoy で10秒・32GBの消費が報告された。
4. **影響の波及**: HTTP.sys が共通基盤のため、単一の穴が IIS / OWA / RD ゲートウェイ等を横断的に巻き込む。「一点を倒すと多面が止まる」構造的リスク。

一般原理として、本件は「外部入力に応じた確保に上限・スロットリングを設けていない」資源管理欠陥（CWE-770）と、プロトコル機能（フロー制御）の正当な仕様を悪用した解放阻止の合わせ技。武器化に必要な具体的パラメータ等は記載しない。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: 受信ウィンドウ0を維持する HTTP/2 コネクションの多発、確立済みコネクション数とプロセスメモリの急増、HTTP.sys/IIS のメモリ使用量の急上昇、短時間でのワーカープロセス再起動。
- **暫定緩和（パッチ前）**: `MaxRequestBytes` を過大にしない（既定16KB相当を維持。65,534バイト超にしない）。前段の WAF/ロードバランサで HTTP/2 のヘッダ量・コネクションあたりのリソースを制限。必要に応じ一時的に HTTP/2 を無効化（性能影響と要相談）。**適用前検証は要確認**。
- **恒久対策**: 2026年6月の月例更新を適用。HTTP.sys を共有する全サービス（IIS/Exchange/RDゲートウェイ等）の更新状況を横断確認。
- **運用示唆**: 可用性監視（メモリ・コネクション数）にしきい値アラートを設定。境界での HTTP/2 リソース制限をベースラインに。

## 📖 用語解説（このニュースとのつながり）

- **HTTP.sys**: Windows カーネル内の HTTP 受付ドライバ。IIS など多数のサービスの共通土台で、本件はここが落ちると上位サービスも止まる点が重大。
- **HTTP/2 / HPACK**: 効率的な通信のための規格と、そのヘッダ圧縮方式。本件は HPACK の展開コストを増幅に悪用する。
- **フロー制御（受信ウィンドウ）**: 送受信ペースを調整する仕組み。ウィンドウ0を悪用して、確保済みメモリの解放を妨げるのが本攻撃の肝。
- **サービス拒否（DoS）/ CWE-770**: 資源を枯渇させて正規利用を妨げる攻撃と、その原因となる「無制限な資源確保」。本件は乗っ取りではなく可用性を狙う。
- **増幅（amplification）**: 小さな入力で大きな負荷を生む非対称性。本件の「小さなリクエストで巨大なメモリ確保」がこれにあたる。

## 影響範囲 / 推奨対応

HTTP/2 を有効にした Windows の HTTP.sys 系サービス（IIS、Exchange OWA、RD ゲートウェイ、WAP、WSUS、RD Web、HttpListener 利用アプリ）が対象。可用性重視の公開サービスは特に注意。2026年6月月例更新の適用を最優先とし、未適用の間は `MaxRequestBytes` の妥当化・前段でのリソース制限・監視強化で凌ぐ。
