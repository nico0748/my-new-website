# CockroachDB v26.1 ― 分散SQLの代表格が LTS 併走で本番運用を安定化

> トピック: 開発トピック｜出典: cockroachlabs.com（release notes）｜種別: backend

## 一行サマリー
地理分散・水平スケールする分散 SQL データベース CockroachDB は、最新安定版が **v26.1.3（2026-04-20 リリース）**、並行して **v25.x の LTS 系列**もサポートされている。PostgreSQL 互換のインターフェースを保ちつつ、複数リージョンにまたがる強整合トランザクションを提供する「分散 SQL」枠の代表格として、運用の安定性を重視するアップデートが続く。

## 🔰 初学者向け（何が・なぜ重要か）
普通のデータベースは1台のサーバーに載っていて、アクセスが増えたり地震で1台止まったりすると困ります。CockroachDB は、データを複数台・複数地域に自動で分散して置き、1台や1地域が落ちても止まらず、しかも「SQL」という慣れ親しんだ言語でそのまま使える、という“頑丈で分散した”データベースです（名前のゴキブリ＝しぶとく生き残る、が由来）。

今回のニュースは、その最新版（v26.1）が出つつ、企業が長く安心して使える「LTS（長期サポート版）」も並行提供されている、という運用寄りの話です。攻めの新機能より「止まらない・長く保守できる」ことを重視する現場にとって、こうした安定運用の整備は重要です。世界中に利用者がいるサービスで「どの地域からでも速く、落ちない」を実現したいときの選択肢になります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
- 最新安定版は **v26.1.3（2026-04-20 リリース）**。並行して **v25.x LTS** 系列がサポート継続（正確な各機能差分は公式 release notes 参照、要確認）。
- PostgreSQL 互換の wire protocol / SQL 方言を維持しつつ、マルチリージョンでの強整合トランザクションを提供。
- LTS（長期サポート）系列と通常リリース系列を併走させ、更新頻度と安定性のニーズを両立する運用体制。

### 仕組み / 背景（なぜこう設計されたか）
CockroachDB は、データを「レンジ」という単位に分割し、各レンジを **Raft** による合意（レプリケーション）で複数ノードに複製します。書き込みは過半数ノードの合意で確定するため、少数ノードや1リージョンの障害でも整合性を保ったまま生き残れます。分散環境での強整合トランザクションは、**ハイブリッド論理クロック（HLC）** による時刻順序づけと MVCC（多版数同時実行制御）で実現しており、地理分散していても「まるで単一の SQL DB」のように見せます。

なぜ PostgreSQL 互換なのか。既存の Postgres エコシステム（ドライバ・ツール・SQL 知識）をそのまま活かせるため、分散 SQL への移行障壁を下げられるからです。トレードオフは、合意ベースの書き込みがリージョン間のネットワーク遅延の影響を受けやすく、レイテンシ設計（データの地理配置＝どのデータをどの地域に置くか）が性能を左右する点です。LTS 併走は、頻繁なアップグレードを避けたいミッションクリティカル運用のために、安定版を長く保守できるようにする設計判断です。

### 実務での使いどころ・移行の勘所
- 複数リージョンでの高可用・強整合が要件（グローバル SaaS、金融系、在庫・決済）で真価を発揮。
- 単一リージョン・小規模用途では、素の PostgreSQL の方が運用が単純でコスト効率が良い場合も多い。
- 導入時はデータの地理配置とレイテンシ要件を設計の中心に据える。長期運用は LTS 系列の採用でアップグレード頻度を抑えられる。

## 📖 用語解説（このトピックとのつながり）
- **CockroachDB**: PostgreSQL 互換の分散 SQL データベース。本トピックの主役。
- **分散 SQL (distributed SQL)**: 水平スケールと強整合トランザクションを両立する DB カテゴリ。
- **Raft**: 複数ノードで状態を合意・複製する分散合意アルゴリズム。CockroachDB の耐障害性の核。
- **LTS（長期サポート）**: 安定版を長期間保守する提供形態。頻繁な更新を避けたい運用向け。
- **MVCC / HLC**: 多版数同時実行制御とハイブリッド論理クロック。分散下の整合性を支える。

## 影響範囲 / 推奨アクション
グローバル規模で高可用・強整合が必要なバックエンドチームに関係する。単一リージョン用途なら通常の PostgreSQL で十分なことも多い。採用検討時は、マルチリージョン要件の有無とレイテンシ設計を見極め、長期運用なら LTS 系列を選ぶのが安全。
