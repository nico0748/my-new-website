# React 19.2 — `<Activity>` / useEffectEvent / cacheSignal で「非表示の状態保持」と「非リアクティブ処理」が標準に

> トピック: React 19.2 の新API群｜出典: React 公式ブログ（react.dev）｜種別: マイナーメジャー機能追加

## 一行サマリー

React 19.2 は、非表示にしても状態を保てる `<Activity>` コンポーネント、Effect から「反応させたくない処理」を切り出す `useEffectEvent` フック、非同期処理の後片付けを担う `cacheSignal`、そして Chrome DevTools と連動する Performance Tracks を追加し、これまで自前のワークアラウンドで対処していたパターンを公式APIに置き換えた。

## 🔰 初学者向け（何が・なぜ重要か）

React はページの見た目（UI）を「部品（コンポーネント）」の組み合わせで作るライブラリです。19.2 では、部品を扱ううえで長年もやもやしていた3つの問題に、公式の道具が用意されました。

ひとつめは「一時的に隠したいけど、隠すと中身が消えてしまう」問題です。たとえばタブを切り替えると、前のタブに入力した内容やスクロール位置が消えてしまうことがあります。新しい `<Activity>` は「表示はしないけど、記憶は残しておく」という指示を出せる箱です。裏側に片付けておいて、戻ってきたら続きから使えるイメージです。

ふたつめは「更新のたびに勝手にやり直されて困る処理」問題です。React には「何かが変わったら実行する」仕組み（Effect）がありますが、変わってほしくない値まで巻き込まれて余計に再実行されることがありました。`useEffectEvent` は「この部分は最新の値を見るけど、これがきっかけで再実行はしないでね」と線を引ける道具です。

みっつめは開発中の可視化です。Performance Tracks を使うと、ブラウザの開発者ツール上に「React が今どの処理にどれだけ時間をかけたか」が専用のレーンで表示され、遅い原因を目で追えるようになりました。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **`<Activity>` コンポーネント**: `mode="visible" | "hidden"` を受け取り、`hidden` のときは子ツリーをアンマウントせずに描画だけ止める。Effect は破棄されるがコンポーネントの state と DOM 状態（フォーム入力・スクロール位置）は保持される。将来の事前レンダリング（隠れたタブを裏で用意しておく）にもつながる基盤。
- **`useEffectEvent`**: Effect Event を作るフック。中で参照する props/state は常に最新値を読むが、その値は Effect の依存配列に含まれず再実行のトリガーにならない。「最新値は欲しいが再購読はしたくない」ケース（アナリティクス送信、直近の onChange 参照など）を安全に書ける。
- **`cacheSignal`**: React の `cache()` と組み合わせ、キャッシュのライフサイクルに紐づく `AbortSignal` を提供。キャッシュが無効化された時点で進行中の fetch を中断でき、サーバーコンポーネントでのリソースリークを防ぐ。
- **`useId` の接頭辞変更**: 生成IDが `:r:` 系の特殊文字から `_r_` に変更。`view-transition-name` や XML 1.0 命名規則に適合させ、View Transitions との併用を可能にした。
- **Performance Tracks / SSR のバッチ処理改善**: Chrome DevTools のパフォーマンスパネルに React 専用トラックを追加。加えて Suspense 境界のストリーミング SSR でフラッシュをまとめるなどの最適化。
- リリースは 2025-10-01（React 19.2）。数値・仕様は公式ブログに基づく。

### 仕組み / 背景（なぜこう設計されたか）

`<Activity>` の核心は「アンマウント＝状態破棄」という React の従来モデルへの穴埋めだ。これまで開発者は、隠したいだけの UI を `display:none` で消したり、state を親に持ち上げて保存したりしていたが、前者は Effect が動き続け、後者はボイラープレートが増える。`<Activity>` は「描画を止めつつ state ツリーは温存」という中間状態を第一級で表現し、React の並行レンダリングと組み合わせて、隠れた領域を低優先度で保つ設計になっている。トレードオフは、隠れていてもツリーが生きているためメモリを消費する点で、大量の `<Activity>` を積むと逆効果になりうる。

`useEffectEvent` は Effect の「反応性（reactivity）」を細かく制御するための設計だ。React の Effect は依存配列に載せた値が変わるたび再実行されるが、実務では「最新値は読みたいが、それを購読トリガーにはしたくない」処理が混在する。従来は依存配列から意図的に値を外す（lint 警告を無視する）というアンチパターンで凌いでいた。`useEffectEvent` はこの「非リアクティブな読み取り」を型と lint が理解できる形で公式化し、クロージャの stale 値問題を回避する。

Performance Tracks と SSR バッチ処理は、19系で進めてきた「並行レンダリングの可観測性と本番性能」の延長線上にある。

### 実務での使いどころ・移行の勘所

タブ UI・ウィザード・モーダルなど「戻ったときに状態を残したい」箇所は `<Activity>` の第一候補。ただし隠れツリーもメモリに残るため、無制限に使わず「戻る確率が高い数枚」に限定する。`useEffectEvent` は「Effect が余計に再実行される」既存バグの修正に効く。まず依存配列から外して誤魔化していた箇所を洗い出すと移行対象が見つかる。19.2 はマイナーバージョンで後方互換を重視しているため、`useId` の接頭辞変更が CSS セレクタや E2E のハードコード ID に影響しないかだけ確認すればよい。

## 📖 用語解説（このトピックとのつながり）

- **Effect（副作用）**: 描画以外の処理（データ取得・購読・DOM操作）。本トピックの `useEffectEvent` は Effect の再実行条件を絞る道具。
- **依存配列（dependency array）**: Effect を再実行する条件のリスト。`useEffectEvent` はここに載らない値の参照を可能にする。
- **アンマウント**: コンポーネントをツリーから取り除き state を破棄すること。`<Activity hidden>` はこれを避けつつ描画を止める。
- **AbortSignal**: 進行中の非同期処理を中断する標準API。`cacheSignal` はキャッシュ無効化に連動した Signal を返す。
- **View Transitions**: ページ/要素遷移をアニメ化するブラウザAPI。`useId` の命名変更はこれとの互換のため。

## 影響範囲 / 推奨アクション

- 対象: React を使うフロントエンド全般（Web / SSR / RSC 構成）。
- 推奨: タブ・ウィザードの状態消失バグは `<Activity>` を検討。Effect の過剰再実行は `useEffectEvent` で解消。`useId` を CSS/E2E でハードコードしていないか点検。
- 注意: `<Activity hidden>` はメモリを消費する。多用しない。
