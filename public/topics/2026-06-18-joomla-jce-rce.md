# Joomla Content Editor（JCE）の最大深刻度RCE（CVE-2026-48907）

> トピック: 脆弱性ニュース ｜ 出典: The Hacker News / BleepingComputer / CISA ｜ 種別: ニュース解説（3レベル）

## 一行サマリー
CMS「Joomla」の人気エディタ拡張 **JCE（Joomla Content Editor）** に、**CVSSスコア満点10.0**の脆弱性。ログイン不要（認証不要）で攻撃者が**PHPコードをアップロード・実行**でき、Webサーバを乗っ取れる。攻撃コードは公開済みで自動化攻撃が進行中。CISAが2026-06-16にKEV（悪用既知脆弱性カタログ）へ登録し、米政府機関に6/19までの修正を命令。

---

## 🔰 初学者向け（何が・なぜ問題か）
**Joomla**はWebサイトを作るためのCMS（コンテンツ管理システム。WordPressの仲間）です。**JCE**はその中で「記事を見た目よく編集する」ための人気の追加機能（拡張/プラグイン）です。

今回の問題は、このJCEに「**ログインしていない赤の他人でも、サーバに自分のプログラム（PHPファイル）を置いて動かせてしまう**」という致命的な穴があったこと。Webサーバ上で自由にプログラムを動かせるということは、サイトの中身を書き換える・訪問者を別サイトに飛ばす・サーバに保存された個人情報を盗む・他の攻撃の踏み台にする、など事実上「**サイトの乗っ取り**」を意味します。

深刻度は「**CVSS 10.0**」。これは脆弱性の危険度を0〜10で表す国際スコアの**満点（最悪）**で、めったに付きません。さらに「**認証不要**（ログイン不要）」で「攻撃コードが公開済み・自動化されている」ため、**該当サイトは無差別にスキャン・攻撃されている**状態です。心当たりがあれば、いますぐの更新が必要です。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）
本脆弱性 CVE-2026-48907 は **CWE: Improper Access Control（不適切なアクセス制御）**に分類される **CVSS 10.0** の **未認証リモートコード実行（Unauthenticated RCE）**です。

攻撃の流れはこうです。攻撃者はJCEの**インポート用エンドポイント**へ細工したリクエストを送り、本来必要な権限チェックを回避して**新しいエディタプロファイルを作成**します。そのプロファイルを経由して**悪意あるPHPファイル（Webシェル）をアップロード**し、ブラウザからそのファイルにアクセスして**サーバ上でコードを実行**します。結果として、攻撃者は永続的な裏口（**バックドア/Webシェル**）を獲得します。

- **影響バージョン**: JCE 1.0.0 〜 2.9.99.4
- **修正バージョン**: 2.9.99.5（2026-06-03 リリース）
- **状況**: 実用的なエクスプロイトコードが公開済み、攻撃は自動化、CISA KEV登録（2026-06-16）

「Webシェル」とは、攻撃者がサーバ上に置く小さなスクリプトで、HTTPリクエスト経由で任意コマンドを実行できる遠隔操作窓口です。一度設置されると、**パッチを当てた後も裏口が残る**ため、侵害有無の調査と駆除が不可欠です。

### メカニズム（攻撃が成立する仕組み）
「未認証なのに、なぜ任意PHPまで実行できるのか」を、データの流れで分解する。

**1) 鍵は“エディタプロファイル”という設定オブジェクト**
JCEの**エディタプロファイル**は、単なる見た目設定ではなく「**どの拡張子のファイルをどのディレクトリにアップロード可能か**」を決める**権限ポリシーの実体**です。つまりプロファイルを操作できる＝**アップロードの許可ルールを操作できる**、という設計になっています。ここを攻撃者に握られると話が一気に深刻化します。

**2) アクセス制御の欠落（入口の検問が無い）**
本来、プロファイルの**インポート（取り込み）機能**は管理者専用で、実行前に「**この操作をする権限があるか**（認証＋認可トークン/CSRFチェック）」を検問するはずです。CVE-2026-48907の核心は、この**インポート用エンドポイントの認可チェックが欠落/不十分**な点。攻撃者は細工したHTTPリクエストを直接そのエンドポイントへ送るだけで、ログインせずにプロファイル作成処理へ到達できます（＝**CWE: Improper Access Control**）。

**3) 悪性プロファイルの注入 → アップロードポリシーの書き換え**
攻撃者はインポートで「**.php を許可拡張子に含め、Webルート配下の書き込み可能ディレクトリを保存先に指定した**」プロファイルを作ります。Joomlaは通常、画像など安全な拡張子だけを許可してPHPを弾きますが、この悪性プロファイルが**安全装置（拡張子ホワイトリスト）そのものを無効化**します。

**4) Webシェルの設置 → 実行**
緩んだポリシー下でJCEのファイルブラウザ/アップロード機能を使い、`shell.php` のような**Webシェル**をWebルート配下に置きます。あとはブラウザから `https://victim/path/shell.php?cmd=...` のようにアクセスするだけ。**サーバ上のPHPエンジンがそのファイルを“プログラムとして”解釈・実行**するため、攻撃者のコマンドがWebサーバ権限で走ります。これが**未認証RCE**の完成形です。

**5) なぜ“パッチだけでは不十分”なのか**
4の時点で攻撃者は**サーバ上に永続的な裏口（Webシェル）**を確保済み。プラグインを2.9.99.5へ更新しても、**すでに設置されたWebシェルや、攻撃者が作った不正管理プロファイル/ユーザは残る**ため、別途の捜索・駆除が必須になります。

> 本質は「**設定オブジェクト（プロファイル）が安全装置を兼ねているのに、その変更入口に認可が無かった**」こと。攻撃連鎖は〈認可欠落→ポリシー書換→任意ファイル設置→PHP実行〉という4段の橋渡しで成立します。

### 検知・暫定緩和・恒久対策
- **Patch ≠ Cleanup**: 自動化攻撃＋公開PoCの組み合わせは「パッチ前にすでに侵入されている」前提で動くべき事案。更新後も `images/` や `tmp/`、書き込み可能ディレクトリ配下に設置された未知のPHP（Webシェル）を捜索する。
- **侵害の痕跡（IoC）の着眼**: インポート系エンドポイントへの異常POST、短時間での新規エディタプロファイル作成、`.php` の新規/改変ファイル、外部からのコマンド実行を示すアクセスログ（不自然なクエリ文字列・base64ペイロード）。
- **対応手順**: ①即時 2.9.99.5 以降へ更新 → ②Web rootの整合性チェック（ファイル改ざん検知） → ③不審PHPの隔離・削除 → ④DB内の不正管理ユーザ/プロファイル確認 → ⑤資格情報ローテーション → ⑥WAFで該当エンドポイントへの不正パターンを暫定遮断。
- **運用示唆**: CMS拡張は攻撃面が広い。使っていない拡張は無効化・削除し、棚卸し（インベントリ管理）と自動更新を徹底する。

## 📖 用語解説（このニュースとのつながり）
- **CMS / Joomla**: Webサイト構築・運用システム。JoomlaはWordPressに次ぐ代表的CMS。
- **JCE（Joomla Content Editor）**: Joomla向けの人気WYSIWYGエディタ拡張。本件の対象。
- **RCE（Remote Code Execution）**: 遠隔の攻撃者が標的サーバ上で任意コードを実行できる、最も危険な部類の脆弱性。
- **未認証（Unauthenticated）**: ログイン情報なしで悪用可能。攻撃ハードルが極めて低い。
- **Webシェル / バックドア**: サーバに残す遠隔操作用スクリプト。パッチ後も残存しうる。
- **CISA KEV**: 米CISAが「実際に悪用が確認された脆弱性」をまとめたカタログ。登録＝今まさに狙われている、の強いシグナル。
- **CVSS 10.0**: 深刻度スコアの満点。最優先対応対象。

## 影響範囲 / 推奨対応
- 影響: JCE 1.0.0〜2.9.99.4 を使う全Joomlaサイト（Linux Webサーバ中心）。
- 対応: 直ちに 2.9.99.5 以降へ更新。更新後も侵害調査（Webシェル捜索）と資格情報ローテーションを実施。
