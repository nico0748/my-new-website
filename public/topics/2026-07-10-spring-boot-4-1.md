# Spring Boot 4.1 リリース — gRPC 自動構成・SSRF 対策・Kotlin 2.3 対応

> トピック: Spring Boot 4.1.0（2026年6月10日）｜出典: Spring 公式ブログ / InfoQ｜種別: フレームワーク メジャーリリース

## 一行サマリー

Spring Boot 4.1.0 が 2026年6月10日にリリースされ、gRPC の自動構成、HTTP クライアントの SSRF 対策（InetAddressFilter）、Kotlin 2.3 対応、遅延データソース接続などが加わった。Spring Framework 7 を土台に、Java 17 最小・最大 Java 26 をサポートする Java エコシステムの基幹更新。

## 🔰 初学者向け（何が・なぜ重要か）

Spring Boot は、Java でサーバーアプリ（Web API や業務システム）を作るときの定番の土台。「面倒な初期設定を自動でやってくれる（auto-configuration）」のが最大の売りで、企業の基幹システムで広く使われている。

今回の 4.1 で目立つのは3つ。ひとつめは **gRPC の自動構成**。gRPC は「サービス同士が高速に会話する」ための通信方式で、これを設定なしで使い始められるようになった。ふたつめは **SSRF 対策**。SSRF は「アプリを騙して、本来アクセスできない社内サーバーへ代わりにアクセスさせる」攻撃で、その入口になりがちな HTTP クライアントに防御フィルタが入った。みっつめは **Kotlin 2.3 対応**。Kotlin は Java と一緒に使われる人気言語で、その最新版に追随した。全体として「安全に・速く・最新言語で作れる」方向の順当な強化だ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

InfoQ / Spring 公式によれば、Spring Boot 4.1 の主な追加は、(1) Spring gRPC のサポートと auto-configuration、(2) HTTP クライアントの SSRF 緩和として `InetAddressFilter` を提供、(3) 遅延データソース接続（lazy datasource connections）、(4) `@Async` メソッドの非同期コンテキスト伝播、(5) OpenTelemetry サポート更新を含む可観測性の改善、(6) Log4j のファイルローテーション対応、(7) Jackson 設定プロパティの更新など。土台は Spring Framework 7 で、Java 17 が最小要件、4.1 では Java 26 までサポート。関連プロジェクトも同時期に更新され、Spring AI 2.0、Spring Security 7.1.0、Spring Data 2026.0.0、Spring for Apache Kafka 4.1.0 などが揃ってリリースされた。

### 仕組み / 背景（なぜこう設計されたか）

Spring Boot の中核思想は「規約による自動構成」で、クラスパスに存在する依存を検出して合理的な既定設定を注入する。gRPC 自動構成もこの延長で、gRPC スターターを入れるだけでサーバー/チャネルの定型設定が組み上がる。SSRF 対策を「HTTP クライアント層のフィルタ」として提供したのは、アプリ側のバリデーション漏れに依存せず、送信先アドレス（内部レンジ等）をフレームワーク境界で遮断できるようにする防御的設計。`InetAddressFilter` は解決後の IP を検査して内部ネットワークへの到達を抑止する発想だ。トレードオフは、フィルタの許可/拒否リスト運用が必要で、正当な内部通信まで塞がないよう構成の手当てが要ること。遅延データソース接続や `@Async` のコンテキスト伝播は、起動時間・スレッド跨ぎのトレース継続という運用課題への対応で、可観測性（OTel）強化と合わせ「本番運用の解像度を上げる」意図が一貫している。

### 実務での使いどころ・移行の勘所

Java/Kotlin でマイクロサービスを組む組織で、gRPC 導入のハードルが下がるのが最大の実益。SSRF フィルタは、外部 URL を取得する機能（Webhook 検証、画像フェッチ、プロキシ）を持つサービスで有効化を検討する価値が高い。移行の勘所は、4.x は Spring Framework 7・Java 17 以上が前提で、3.x からの移行は非互換（設定プロパティ・パッケージ再編）の確認が必須なこと。SSRF フィルタは既存の正当な内部通信を誤って遮断しないよう、許可レンジの設計とテストをセットで行う。関連プロジェクト（Security 7.1、Data 2026.0）も足並みで上がるため、依存バージョンの整合を BOM で一括管理するのが安全。

## 📖 用語解説（このトピックとのつながり）

- **auto-configuration（自動構成）**: 依存の存在を検出して既定設定を注入する Spring Boot の中核機能。gRPC もこれで設定レス化。
- **SSRF（Server-Side Request Forgery）**: サーバーを踏み台に内部リソースへアクセスさせる攻撃。4.1 の `InetAddressFilter` が緩和策。
- **gRPC**: Protocol Buffers ベースの高速な RPC 方式。サービス間通信で採用が広がる。
- **Spring Framework 7 / BOM**: Boot 4.x の土台となるコアと、依存バージョンを束ねる管理表。移行時の整合管理に使う。

## 影響範囲 / 推奨アクション

- Java/Kotlin バックエンドで gRPC・可観測性・セキュリティの標準装備が前進。新規サービスは 4.1 系を基準に検討。
- 外部 URL を扱うサービスは SSRF フィルタの有効化と許可レンジ設計を評価する。
- 3.x からの移行は Java 17+・Spring Framework 7 前提の非互換確認と、関連プロジェクトの BOM 整合が必須。
