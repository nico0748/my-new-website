# RAG × ベクトル検索（Atlas Vector Search in LangChain）— ハイブリッド / GraphRAG

> トピック: 開発トピック｜出典: langchain｜種別: ai

## 一行サマリー
LangChain と MongoDB の提携により、Atlas Vector Search を RAG のバックエンドとして統合しやすくなり、キーワード＋ベクトルのハイブリッド検索やグラフ構造を活かす GraphRAG が実務の定番手法として成熟している（詳細仕様は要確認）。

## 🔰 初学者向け（何が・なぜ重要か）
LLM は学習した範囲しか知らず、社内文書や最新情報は答えられません。RAG（検索拡張生成）は、質問に関連する文書をまず検索し、その内容を LLM に渡してから答えさせる仕組みです。試験に「持ち込み可の資料」を用意するようなものです。

その「検索」の中核がベクトル検索で、文章を意味の座標（ベクトル）に変換し、意味の近い文書を探します。MongoDB Atlas はこのベクトル検索を、既存のデータベースにそのまま追加できる点が魅力です。さらにハイブリッド検索（キーワード検索とベクトル検索の併用）や GraphRAG（文書同士の関係グラフを辿る手法）を使うと、検索の精度が上がり、LLM の回答がより正確になります。使い慣れた DB の上で高度な RAG が組めるのが今回の要点です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
LangChain × MongoDB 提携で、Atlas Vector Search をエージェント/RAG のストレージ兼検索層として統合するインテグレーションが整備された。ベクトル検索・全文検索・メタデータフィルタを同一 DB で扱え、ハイブリッド検索や GraphRAG パターンを構築しやすい。API・バージョン詳細は要確認。

### 仕組み / 背景（なぜこう設計されたか）
純粋なベクトル検索は意味的類似に強い一方、固有名詞・型番などの厳密一致に弱い。ハイブリッド検索はキーワード（BM25 等）とベクトルのスコアを融合してこの弱点を補う。GraphRAG は文書・エンティティの関係グラフを併用し、単純な類似検索では拾えない多段の関連情報を辿る。既存 DB 上で完結させることで、専用ベクトル DB を増やさず運用・データ整合を単純化できる。

### 実務での使いどころ・移行の勘所
既に MongoDB を使う組織は、専用ベクトル DB を追加せず RAG を構築できる利点が大きい。設計では、チャンク分割・埋め込みモデル選定・ハイブリッド重み付け・再ランキングが精度を左右する。GraphRAG は関係性が重要なドメイン（規程・組織・依存関係）で効くが、グラフ構築コストと保守を要する。まずハイブリッド検索で基準線を作り、必要に応じ GraphRAG を足す。

## 📖 用語解説（このトピックとのつながり）
- **RAG（検索拡張生成）**: 検索結果を LLM に渡して回答する枠組み。本トピックの主題。
- **ベクトル検索**: 意味の近さで文書を探す検索。RAG の中核。
- **ハイブリッド検索**: キーワード＋ベクトルの併用。厳密一致の弱点を補う。
- **GraphRAG**: 関係グラフを辿る RAG。多段関連の取得に有効。

## 影響範囲 / 推奨アクション
社内文書 QA・エージェントを作る開発者に影響。MongoDB 利用者は Atlas Vector Search で RAG を集約検討し、まずハイブリッド検索で基準を作ってから GraphRAG を段階導入するとよい。
