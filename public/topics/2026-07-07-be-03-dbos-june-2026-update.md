# DBOS 2026年6月更新 — プラガブルデータソース・Go の SQLite 対応・Google ADK 統合で「Postgres だけの耐障害実行」が横に広がる

> トピック: DBOS Transact の6月アップデート｜出典: DBOS 公式ブログ｜種別: OSS ライブラリの機能拡張（既報 07-05 の DBOS 記事からの新展開）

## 一行サマリー

Durable Execution ライブラリ DBOS が2026年6月の更新でデータソースのプラガブル化を進め、Go 版に SQLite バックエンドを追加して Python/TypeScript との機能同等性を高め、さらに Google ADK 統合により AI エージェントの実行をデータベース裏付きの再開可能な処理にできるようになった。

## 🔰 初学者向け（何が・なぜ重要か）

「Durable Execution（耐久実行）」とは、長い処理の途中経過をデータベースに記録しながら進めることで、サーバーが再起動しても「続きから」やり直せる仕組みです。宅配便の配送システムが「集荷済み→輸送中→配達中」と各段階を記録しているから、担当者が交代しても配送が迷子にならないのと同じ発想です。

この分野は従来、Temporal のような「専用の司令塔サーバー」を立てる方式が主流でしたが、DBOS は「あなたのアプリに図書館（ライブラリ）として組み込み、記録先はいつもの Postgres でいい」という軽量路線で注目されてきました。

6月の更新のポイントは「間口が広がった」こと。記録先として Postgres 以外も選べる方向に進み、Go 版では SQLite（ファイル1つで動く軽量データベース）でも動くようになりました。つまり「本番は Postgres、手元の開発やテストは SQLite」という使い分けができます。さらに Google のエージェント開発キット（ADK）との連携で、「AI エージェントが途中で落ちても続きから再開」という組み合わせも公式にサポートされました。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **プラガブルデータソース**: ワークフロー状態の保存先を差し替え可能にする抽象化を拡張（対応バックエンドの全リストは要確認）。
- **Go 版の強化**: DBOS Transact for Go が Python/TypeScript 版との機能同等性を改善し、**SQLite バックエンド**を追加。v1.0 に向けた大きな一歩と位置付け。
- **Google ADK 統合**: ADK で構築したエージェントの実行を DBOS の耐久実行で裏付けるインテグレーションを公開。エージェントのステップがデータベースにチェックポイントされ、クラッシュ後の再開・再実行制御が可能に。
- 6月時点のその他の細かい改善（バージョン番号の詳細は要確認）。

### 仕組み / 背景（なぜこう設計されたか）

Durable Execution の本質は「ワークフローの各ステップの完了を永続ストアに記録し、再起動時はログを再生して未完了ステップから続行する」こと。Temporal/Cadence 系はこれを外部オーケストレータ＋ワーカーの分散システムとして実装するが、運用物が増える。DBOS は「アプリのプロセス内でステップ境界をライブラリがフックし、記録は普通の DB トランザクションで行う」ことで、追加インフラをゼロにする設計を選んだ。

今回のプラガブル化は、この「普通の DB」の定義を広げる動きである。SQLite 対応の意味は3つ:(1) ローカル開発・CI で Postgres コンテナ不要、(2) エッジ/組み込み的な単一ノード用途への進出、(3)「耐久実行＝重装備」という心理障壁の撤去。トレードオフとして、SQLite は単一ライタ制約があるため高並行ワークフローには不向きで、水平スケールや複数ワーカー協調はやはり Postgres 系が前提になる。

ADK 統合は「エージェントは本質的に長時間・不確実・途中失敗が多いワークフローである」という認識の帰結。LLM 呼び出し・ツール実行の各ステップをチェックポイント化すれば、失敗時に「最初からやり直して同じトークン代を二重に払う」ことを避け、副作用のあるツール実行の二重実行（例: 二重送金）も冪等性管理で防げる。LangGraph がチェックポインタを内蔵する路線に対し、DBOS は「実行基盤を汎用の耐久実行層として提供し、エージェントFWは上に乗せる」水平分業を狙う。

### 実務での使いどころ・移行の勘所

- 決済・在庫・プロビジョニングなど「途中で死ぬと不整合が残る」多段処理を、キュー＋自前状態機械で組んでいるなら置き換え候補。既に Postgres があるなら導入コストは小さい。
- AI エージェントの本番運用で「再試行のたびに最初から」問題に当たっているチームは、ADK + DBOS の組み合わせ（または LangGraph のチェックポイント）を比較検討。選定軸は「エージェントFWに寄せるか、実行基盤を分離するか」。
- Go チームは SQLite バックエンドで PoC → Postgres で本番、という段階導入が可能に。v1.0 前なので API 変動リスクは織り込むこと。
- Temporal との比較: 組織横断の大規模ワークフロー・多言語ワーカー混在なら Temporal、単一アプリ内の耐障害性なら DBOS が簡潔、が現時点の目安。

## 📖 用語解説（このトピックとのつながり）

- **Durable Execution**: ステップ完了を永続化し再開可能にする実行モデル。本更新の主題。
- **チェックポイント**: 進捗の保存点。エージェントの文脈では「どのツールをどの引数で実行済みか」の記録。
- **冪等性（idempotency）**: 同じ操作を2回行っても結果が変わらない性質。再実行時の二重副作用防止の鍵。
- **Google ADK（Agent Development Kit）**: Google のエージェント構築キット。今回 DBOS が実行の耐久化を提供する統合を出した。
- **SQLite**: サーバーレスの組み込みDB。開発体験向上と軽量用途への足がかりとして採用された。
