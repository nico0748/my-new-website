# Axum 1.0 リリース — Rust 最有力 Web フレームワークが安定版に

> トピック: 開発トピック｜出典: rustify.rs｜種別: backend

## 一行サマリー
Tokio チームが開発する Rust の Web フレームワーク Axum が 1.0 に到達し、非同期ランタイム Tokio を核とした Rust バックエンドの事実上の標準として安定版 API を確定させた。

## 🔰 初学者向け（何が・なぜ重要か）
Web サーバーを作るとき、生の HTTP を自分で組み立てるのは大変です。そこで「ルーティング（URL と処理の対応付け）」「リクエストの受け取り」「レスポンスの返却」をまとめて面倒見てくれる道具が Web フレームワークで、Axum はその Rust 版です。料理でいえば、包丁からコンロまで一式そろったキッチンのような存在です。

Axum を作っているのは Tokio チームです。Tokio は Rust で「たくさんの通信を同時にさばく」ための土台（非同期ランタイム）で、Rust の非同期処理のほぼ標準になっています。その本家チームが作るフレームワークなので、土台との相性が非常に良いのが最大の強みです。

「1.0」という数字は、外から見える使い方（API）を今後むやみに壊さないという約束です。これまでのバージョンは改良のたびに書き方が変わることがあり、業務投入をためらう理由になっていました。1.0 到達はその不安が解消され、安心して長期採用できる合図になります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Axum 1.0 の中心は、`tower` / `tower-http` のミドルウェア資産をそのまま活かせる設計と、型による抽出（Extractor）の安定化です。ハンドラの引数に `Json<T>`、`Query<T>`、`State<S>` などを並べるだけで、リクエストから必要な部分を型安全に取り出せます。1.0 では中核 API が凍結され、破壊的変更の頻度が抑えられます（詳細な変更点は一次ソースで要確認）。

### 仕組み / 背景（なぜこう設計されたか）
Axum はマクロを極力使わず、Rust の型システムと `tower::Service` トレイトの合成で機能を組み立てる設計です。ミドルウェアを Tower のレイヤーとして重ねられるため、タイムアウト・レート制限・トレーシングなどを再利用できます。従来の Rust Web FW はエコシステムが分断されがちでしたが、Tower を共通土台に据えることで相互運用性を確保しています。

### 実務での使いどころ・移行の勘所
高スループットと低レイテンシが要る API サーバー、gRPC ゲートウェイ、エッジ寄りのサービスに向きます。Actix Web が実行性能で並び立つ競合ですが、Tokio エコシステムとの一体感では Axum が優位です。1.0 化により、これまで Actix を選んでいたチームも移行候補に入れやすくなりました。既存の 0.x コードは Extractor まわりの API 変更に追従が必要な点に注意します。

## 📖 用語解説（このトピックとのつながり）
- **Tokio**: Rust の非同期ランタイム。Axum が動く土台であり、両者は同じチーム製で密結合。
- **Tower**: サービスとミドルウェアを合成する抽象。Axum の拡張性はここに依存する。
- **Extractor**: リクエストから型安全に値を取り出す Axum の仕組み。ハンドラ引数の型がそのまま抽出指示になる。
- **非同期ランタイム**: 多数の I/O を少ないスレッドで同時処理する実行基盤。高並行 API の性能の鍵。

## 影響範囲 / 推奨アクション
Rust でバックエンドを書くチーム、特に新規 API サービスの技術選定者に影響します。新規採用なら 1.0 を前提に設計し、既存 0.x プロジェクトは 1.0 移行ガイドで Extractor / ミドルウェア API の差分を確認して段階移行するのが安全です。
