# Redis 8 — Vector Sets とモジュール統合、AI時代のインメモリDB

> トピック: Redis 8 GA（Vector Sets 新データ型・モジュール統合・名称変更）｜ 出典: Redis 公式ブログ / Docs ｜ 種別: メジャーバージョンアップ

## 一行サマリー

定番のインメモリDB Redis が 8 系で、Search・JSON・Time Series・確率的データ構造をコア（Redis Open Source）に統合し、作者 antirez による新データ型「Vector Sets」でベクトル類似検索を標準搭載。30以上の性能改善でクエリ処理を大幅強化し、AI アプリのセマンティック検索・推薦の基盤として再定義された。

## 🔰 初学者向け（何が・なぜ重要か）

Redis は「超高速なメモリ上のデータ置き場」だ。ディスクではなくメモリにデータを持つため読み書きが桁違いに速く、キャッシュ（よく使うデータの一時保管）、セッション管理、順位表、簡易メッセージ配信などに広く使われる。

今回の **Redis 8** で大きいのは 2 点。ひとつは **Vector Sets という新しいデータの入れ物**。これは AI 時代のキーワード「ベクトル検索」に対応するものだ。AI はテキストや画像を「ベクトル」という数値の並びに変換して意味を表す。「意味が近いものを探す」——たとえば「この質問に似た過去のQ&Aは？」——にはベクトル同士の近さを計算する必要がある。Vector Sets はその"意味の近いもの探し"を Redis の中で直接できるようにする。RAG（検索して答える AI）や推薦システムの心臓部だ。

もうひとつは **機能の一本化**。以前は全文検索や JSON 保存、時系列データなどは「別売りの拡張モジュール」だったが、Redis 8 ではそれらがオープンソースの本体に取り込まれた。「各種オプションが最初から全部入りになった」イメージで、追加インストールなしに高機能が使える。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **Vector Sets（新データ型・ベータ）**: ベクトル類似検索専用の新データ構造。Redis 作者 Salvatore Sanfilippo（antirez）が Redis のコア原則に沿って開発。セマンティック検索・推薦向け。
- **モジュールのコア統合**: Redis Search（ベクトル検索・全文検索・地理空間・集計を担うクエリエンジン）、JSON、Time Series、確率的データ構造（Bloom filter、Cuckoo filter、Count-Min Sketch、Top-K、t-digest）をオープンソース本体に統合。従来モジュールだった機能が全ユーザーに開放。
- **性能改善30以上**: 一部コマンドで最大87%高速化、スループット最大2倍、レプリケーション最大18%高速化、Redis Query Engine のクエリ処理能力を最大16倍に。
- **名称変更**: Redis Community Edition → **Redis Open Source** に改称し、統一された近代的ディストリビューションへ。
- **データ構造追加**: Vector Set（ベータ）含む 8 種のデータ構造をコアに追加。

### 仕組み / 背景（なぜこう設計されたか）

**なぜベクトル検索を本体に**。生成 AI アプリの多くは「埋め込み（embedding）で意味をベクトル化 → 近傍を検索 → その文脈で回答」という RAG パターンで動く。この近傍検索を専用ベクトル DB に外出しすると構成が複雑になる。Redis はもともと低レイテンシのインメモリ基盤であり、キャッシュやセッションと同じ場所でベクトル近傍検索も担えれば、アーキテクチャを単純化できる。Vector Sets はこの「AI ワークロードを既存の Redis に載せる」狙いの中核だ。近傍探索は HNSW 等の近似最近傍（ANN）アルゴリズムで、厳密検索よりメモリと速度を優先する設計思想（厳密さと速度のトレードオフ）に立つ。

**なぜモジュールを統合したか**。かつて Search/JSON 等は別ライセンスのモジュールとして提供され、機能の分断とライセンスの複雑さを生んでいた。8 系でこれらをオープンソースのコアに取り込むことで、「追加なしで多機能」「ライセンスの見通し改善」を実現。名称の Redis Open Source 化はその象徴だ。背景には、フォークである Valkey の台頭に対し、オープンソース版の価値を再提示する狙いも読み取れる。

**トレードオフ**。全機能をメモリ常駐で持つため、大規模ベクトルや大量インデックスはメモリコストが高くつく。Vector Sets はベータであり、本番採用は評価と容量設計が前提。

### 実務での使いどころ・移行の勘所

RAG のベクトルストア、リアルタイム推薦、セマンティックキャッシュ、全文＋ベクトルのハイブリッド検索に向く。既にキャッシュやセッションで Redis を使うシステムなら、別のベクトル DB を導入せず Redis 8 に寄せて構成を単純化できる。移行の勘所は容量計画——ベクトルはメモリを食うため、次元数・件数からメモリ試算を行う。Vector Sets はベータなので、SLA が厳しい本番では専用ベクトル DB（Qdrant 等）との比較評価を。ライセンス面は、Valkey（BSD 系フォーク）との選択も含め、組織のポリシーに照らして判断する。

## 📖 用語解説（このトピックとのつながり）

- **インメモリDB**: データをメモリ上に保持する DB。低レイテンシが特徴。Redis の本質。
- **ベクトル検索 / 埋め込み（embedding）**: テキスト等を数値ベクトル化し、意味の近さで検索する技術。Vector Sets が担う。RAG の基盤。
- **ANN（近似最近傍）/ HNSW**: 厳密でなく高速に近いものを探すアルゴリズム群。ベクトル検索の実装土台。
- **RAG（検索拡張生成）**: 外部データを検索して LLM の回答に使うパターン。ベクトルストアが要る。
- **確率的データ構造**: Bloom filter 等、少ないメモリで近似的に集合判定・カウントする構造。コアに統合。
- **Valkey**: Redis のオープンソース・フォーク。8 系の統合・改称の背景にある競合。

## 影響範囲 / 推奨アクション

- 既に Redis を使うシステムは、別ベクトル DB を足す前に Redis 8 への集約で構成単純化を検討。
- Vector Sets はベータ——本番採用前に容量設計と専用ベクトル DB との比較評価を。
- ライセンス／運用ポリシーに応じ Redis Open Source と Valkey を比較検討。
