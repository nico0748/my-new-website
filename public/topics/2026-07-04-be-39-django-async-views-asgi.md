# Django 5.x の async views + ASGI — ただしORMは依然同期

> トピック: 開発トピック｜出典: developersvoice｜種別: backend

## 一行サマリー
Django 5.x は非同期ビューと ASGI をサポートし、I/O 待ちの多い処理で並行性を高められる一方、ORM（データベース操作層）は依然として同期中心で、非同期化の恩恵を最大化するには設計上の注意が必要である。

## 🔰 初学者向け（何が・なぜ重要か）
Django は Python の老舗 Web フレームワークで、管理画面や認証などが最初から揃った「全部入り」が魅力です。従来は「1つのリクエストを最後まで順番に処理する」同期モデルが基本でした。

近年の Django 5.x では「非同期（async）ビュー」に対応し、外部 API 呼び出しやファイル読み込みなど「待ち」が多い処理の間に、他のリクエストを進められるようになりました。レストランでいえば、1人の店員が注文→調理待ち→次の客、と待ち時間を有効活用するイメージです。

ただし注意点があります。Django の心臓部であるデータベース操作（ORM）はまだ同期が中心で、ここで待ちが発生すると非同期の利点が削がれます。つまり「非同期対応は進んだが、まだ全体が非同期になったわけではない」段階です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Django は `async def` ビューと ASGI サーバ（uvicorn/daphne 等）に対応し、非同期ミドルウェアやビューを扱えます。しかし ORM のクエリは基本的に同期実装で、非同期コンテキストからは `sync_to_async` などのラッパー経由で呼ぶ必要があります（完全な async ORM は未成熟、対応範囲は要確認）。このため、DB 主体のアプリでは非同期化の効果が限定的になりがちです。

### 仕組み / 背景（なぜこう設計されたか）
Django は長年 WSGI（同期）を前提に巨大なエコシステムを築いており、ORM を含む中核を一気に非同期化するのは互換性リスクが大きい。そこで「ビューと入口（ASGI）から段階的に非同期対応する」漸進戦略が採られています。トレードオフは、非同期と同期が混在し、どこで待ちが発生するかを開発者が意識する必要がある点です。

### 実務での使いどころ・移行の勘所
外部 API 集約、WebSocket、ストリーミングなど I/O 待ちが支配的な機能で非同期ビューが効きます。逆に、DB クエリが処理の大半を占めるアプリでは効果が限定的で、無理に async 化すると複雑さだけ増えることがあります。ネイティブ非同期を最優先するなら FastAPI 等が選択肢に入ります。既存 Django 資産を活かすなら、非同期は「効く箇所」に限定投入するのが勘所です。

## 📖 用語解説（このトピックとのつながり）
- **ASGI**：非同期対応の Python Web サーバ標準。Django 非同期化の入口。
- **WSGI**：従来の同期 Web サーバ標準。Django の伝統的基盤。
- **ORM**：DB をオブジェクトとして扱う層。Django では依然同期中心が課題。
- **sync_to_async**：同期処理を非同期文脈で呼ぶための橋渡し。ORM 併用で必要。

## 影響範囲 / 推奨アクション
Django を使うバックエンド開発者に影響します。I/O 待ちが多い機能では非同期ビューを検討し、DB 主体の処理では効果を見極めること。ネイティブ非同期が要件の中心なら FastAPI との比較検討が推奨されます。
