# Go の encoding/json/v2 — 標準 JSON の大改訂で速度と正しさを両取り

> トピック: Go の `encoding/json/v2`（実験的・Go 1.25 で導入、以降改良）｜出典: Go 公式リリースノート｜種別: 標準ライブラリの大改訂

## 一行サマリー

Go の定番 JSON ライブラリ `encoding/json` の全面改訂版として、実験的な `encoding/json/v2` が Go 1.25 で導入され、以降のリリースで改良が続いている。ストリーミング処理、より厳密な仕様準拠、大幅な性能改善を狙った再設計で、Go の JSON 処理の「次の標準」を目指す（正確な安定化バージョンは要確認）。

## 🔰 初学者向け（何が・なぜ重要か）

JSON は、Web API がデータをやり取りするときの共通フォーマット（`{"name": "...", "age": 20}` のような形式）。Go 言語には最初から `encoding/json` という標準の変換ライブラリがあり、ほぼすべての Go 製サーバーが使っている。

ただ、この初代ライブラリは10年以上前の設計で、「速度」「大きなデータの扱い」「JSON 仕様の細かな正しさ」の面で今の要求に追いつかない部分が出てきた。そこで Go チームは、互換性を壊さずに済むよう **別の新しいパッケージ（v2）** として全面的に作り直しを進めている。既存コードはそのまま動かしつつ、新しく書く部分や性能が欲しい部分では v2 を選べる、という移行のしやすさが特徴。JSON を大量に処理するサーバーにとって、地味だが効いてくる基盤の更新だ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

Go 公式リリースノートによれば、`encoding/json/v2` は初代 `encoding/json` の大幅な改訂版として実験的に提供が始まった（Go 1.25 で導入、GOEXPERIMENT フラグ下での有効化を想定）。主眼は、(1) ストリーミング（大きな JSON を全読み込みせず逐次処理）を第一級に扱うトークン/デコーダ API、(2) RFC 準拠の厳密化（重複キーや不正な UTF-8 の扱いなど）、(3) マーシャリング/アンマーシャリングの性能改善とオプションによる挙動の細かな制御。初代 API との後方互換を保つため v1 は据え置き、v2 は新パッケージとして並存させる設計。正式な安定化と既定化のバージョン・タイミングは進行中で、確定情報はリリースノートで要確認。

### 仕組み / 背景（なぜこう設計されたか）

初代 `encoding/json` はリフレクション主体で、API が「バイト列⇔構造体」の一括変換に最適化されていた。これは書きやすい反面、巨大 JSON をメモリに載せきる必要があったり、仕様の曖昧な入力に対する挙動が緩かったり、ホットパスでの割り当てが多かったりした。v2 は「トークン列としての JSON」を中核に据え、逐次読み書き（ストリーミング）と厳密なバリデーションを両立させる方向へ再設計している。互換破壊を避けるために別モジュールパスで並存させ、既存資産を壊さずに新旧を選択できるようにしたのが移行戦略上の要点。トレードオフは、当面は実験扱いで API が固まりきっていない点、そして v1/v2 が二重に存在することでエコシステム（ライブラリ・ツール）の対応が揃うまで過渡期の複雑さが残る点。

### 実務での使いどころ・移行の勘所

大量の JSON をやり取りする API ゲートウェイ、ログ/イベントのストリーム処理、巨大レスポンスを扱うデータ連携などで恩恵が大きい。移行の勘所は、実験段階のうちは本番全面採用を避け、性能クリティカルな箇所やストリーミングが必要な箇所から限定的に試すこと、GOEXPERIMENT 等の有効化条件とビルド影響を把握すること。v1 と v2 で厳密性（重複キー・不正 UTF-8 の扱い）が変わりうるため、既存 API の入出力互換をテストで担保するのが重要。安定化・既定化の時期は公式情報で追い、確定するまではフォールバックとして v1 を残す構成が安全。

## 📖 用語解説（このトピックとのつながり）

- **encoding/json**: Go 標準の JSON 変換ライブラリ。v2 はその大改訂版で、v1 は互換のため据え置き。
- **ストリーミング処理**: 全体を一括で読み込まず、逐次的にトークン単位で処理する方式。巨大 JSON でメモリを節約。
- **GOEXPERIMENT**: Go の実験的機能を有効化するビルドフラグ。v2 の利用に関わる。
- **RFC 準拠の厳密化**: 重複キーや不正な文字列など、曖昧な入力に対する挙動を仕様どおり厳しくすること。

## 影響範囲 / 推奨アクション

- Go の JSON 処理の性能・正しさの基盤が更新される。大量 JSON を扱うサービスは中期的な移行対象として注視。
- 実験段階のため本番全面採用は避け、性能クリティカル／ストリーミング箇所から限定検証する。
- v1/v2 で厳密性が変わりうる点をテストで担保し、安定化時期は公式リリースノートで確認する。
