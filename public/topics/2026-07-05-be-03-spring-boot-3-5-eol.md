# Spring Boot 3.5 が OSS EOL（6/30）— Boot 4.0 系への移行が現実的な締切に

> トピック: Spring Boot 3.5 OSS EOL（2026年6月30日、最終 OSS リリース 3.5.16 は6/25）｜出典: Spring 公式／HeroDevs｜種別: ライフサイクル（EOL）

## 一行サマリー

Java の主力フレームワーク Spring Boot の 3.5 系が2026年6月30日でオープンソース版のサポート終了（EOL）を迎え、最終 OSS リリース 3.5.16 が6月25日に出荷された。無償のセキュリティ修正が止まるため、多くのチームにとって Boot 4.0 系（最新 4.0.7）への移行が実質的な締切となった。

## 🔰 初学者向け（何が・なぜ重要か）

ソフトウェアには「いつまで面倒を見てもらえるか」という期限があります。これを EOL（End of Life、サポート終了）と呼びます。EOL を過ぎると、たとえ重大な脆弱性が見つかっても無償の修正は提供されなくなり、古いまま使い続けるのはセキュリティ上のリスクになります。

Spring Boot は Java でサーバアプリを作るときの定番の土台で、その 3.5 系が2026年6月末で無償サポートを終えました。つまり「今後は自分たちでリスクを抱えるか、有償の延命サポートを買うか、新しい 4.0 系に引っ越すか」の判断が迫られます。

たとえるなら、住んでいるアパートの管理会社が「この棟の管理は今月で終了します」と告知したようなもの。住み続けることはできますが、水漏れ（脆弱性）が起きても直してもらえません。だから多くの人は、期限までに新しい棟（Boot 4.0）へ引っ越すことを考えます。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **Spring Boot 3.5 OSS EOL**: 2026年6月30日。最終 OSS リリースは **3.5.16**（6月25日）。以降のセキュリティ修正は無償では提供されず、延命には商用の拡張サポート（VMware/HeroDevs 等）が必要。
- **Boot 4.0 系**: 最新パッチは **4.0.7**（6月10日）。6月上旬〜中旬には Spring Security・Session・Integration・Modulith・AMQP・Vault などエコシステム全体の点リリースも相次いだ。
- 併せて Spring AI 2.0 GA・Spring Data 2026.0.0 GA が同時期に到達し、Boot 4.0 前提の新スタックが揃った。

正確な各モジュールの EOL 日は公式サポートマトリクスで要確認。

### 仕組み / 背景（なぜこう設計されたか）

Spring の商用/OSS 二段階サポートは、無償利用者には一定期間の OSS メンテを、長期安定を求める企業には有償の延命を提供する設計だ。OSS EOL は「そのマイナー系列への無償コミットを止める」区切りで、開発リソースを最新系列へ集中させる狙いがある。

Boot 4.0 系は、Spring Framework 7・新しい Jakarta EE 基盤・より深い AOT/GraalVM ネイティブ対応・AI 統合（Spring AI 2.0）を前提とする世代で、`javax`→`jakarta` 移行や設定・依存の刷新を伴う。EOL がこのタイミングに置かれるのは、エコシステム（Data・Security・AI 等）が 4.0 前提で GA を出し揃えた「移行先が整った」ことと連動している。トレードオフは、4.0 への移行が単なるバージョン上げでなく、依存ライブラリの Jakarta 対応・非推奨 API の置換・ビルド設定変更を伴う点で、規模の大きいアプリほどコストが重い。

### 実務での使いどころ・移行の勘所

まず自組織のアプリが 3.5 以下かを棚卸しし、EOL 済みなら (1) 4.0 系へ移行、(2) 商用拡張サポートで延命、(3) リスク受容、のいずれかを意思決定する。移行時の勘所は、Jakarta 名前空間・非推奨 API・サードパーティ依存の 4.0 互換性確認。Spring AI 2.0 や Data 2026.0.0 を使いたいなら 4.0 移行は不可避なので、AI 機能導入計画と移行を一体で進めると投資対効果が高い。CI に依存性脆弱性スキャンを組み込み、EOL 系列の放置を可視化しておくと再発を防げる。

## 📖 用語解説（このトピックとのつながり）

- **EOL（End of Life）**: サポート終了。無償のセキュリティ修正が止まる区切り。本トピックの主役。
- **OSS サポート / 商用拡張サポート**: 無償メンテ期間と、EOL 後に有償で延命するサービスの二段構え。
- **Jakarta EE 移行**: `javax`→`jakarta` 名前空間変更。Boot 4 系移行の主要作業のひとつ。
- **サポートマトリクス**: 各バージョンの対応期間を示す公式表。移行計画の根拠にする。

## 影響範囲 / 推奨アクション

Spring Boot 3.5 以下を本番運用中の組織は、EOL 済みのため移行・延命・リスク受容の判断が必要。Boot 4.0 移行は Jakarta 対応と依存互換性の検証を伴う。Spring AI 2.0 導入計画と移行を一体化すると効率的。
