# Storybook 10 — ESM 専用化で install 29%減、sb.mock と Vitest 4 連携

> トピック: Storybook 10 リリース｜出典: Storybook 公式ブログ｜種別: メジャーバージョンリリース

## 一行サマリー

UI コンポーネントの開発・カタログ化ツール Storybook のメジャー版 10 は ESM 専用化を実施し、配布からの CommonJS 排除で install サイズを（9 の削減に加え）さらに約29%削減、dist を非圧縮化してデバッグを容易にした。Vitest チームとの協業による新しいモジュールモック `sb.mock`、Next 16 と Vitest 4 のフルサポート、CSF Factories の Preview 昇格が入る。

## 🔰 初学者向け（何が・なぜ重要か）

Storybook は「アプリ本体から切り離して、UI 部品（ボタン、カード、フォームなど）を単独で表示・確認できる作業場」です。部品ごとに「こういう状態のときはこう見える」という見本（ストーリー）を並べておけば、デザイナーやチームで見た目を確認したり、そのままテストに使ったりできます。

10 の目玉は主に「軽く・素直に」なったことです。まず配布物を **ESM（JavaScript 標準のモジュール形式）専用**にし、古い形式（CommonJS）を捨てたことで、インストールが約29%軽くなりました。さらに配布コードを圧縮せずに配ることで、不具合を追うとき中身が読めて調べやすくなります。加えて、テスト時に「この部品が使う外部データだけ差し替える（モックする）」操作が `sb.mock` という分かりやすい仕組みで書けるようになりました。掃除と整理でカバンが軽くなり、中身も探しやすくなった、というイメージです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **ESM 専用（ESM-only）**: 公開コードから CommonJS を排除。install サイズを（Storybook 9 の削減の上に）約29%削減。dist を非圧縮化しデバッグ容易化。
- **`sb.mock`**: Vitest チームとの協業による新しいモジュールモック。Vitest の `vi.mock` に着想を得つつ、より単純で堅牢な API。コンポーネント分離時の依存差し替えを改善。
- **Next 16 / Vitest 4 フルサポート**。旧バージョンのサポートも当面維持。
- **CSF Factories** が Experimental → **Preview** に昇格。Storybook 11（翌春予定）で既定化を見据え、大きな仕様変更はない見込み。
- リリースは 2025年10月（最新は 10.4 系、次期 10.5 系が alpha）。

### 仕組み / 背景（なぜこう設計されたか）

ESM 専用化の狙いは、デュアルパッケージ（CJS/ESM 両出し）の維持コストとバグ源を断つこと。両形式を出すと、`require`/`import` の初期化順の食い違いや「同一モジュールが二重に読まれる」問題（dual package hazard）が起き、ツールの複雑性が増す。Vite・Vitest・Next といった周辺が ESM 前提に最適化されている今、CJS 互換レイヤーを切り捨てることでバンドル削減とデバッグ性（非圧縮 dist）を同時に得られる。トレードオフは、CJS で Storybook 設定や addon を読み込むレガシー構成の移行が必要になる点。

`sb.mock` は「コンポーネントを本当に隔離して見せる／テストする」ためのモジュール置換機構。Storybook はアプリ本体から部品を切り出す性質上、部品が依存する外部（API クライアント、日時、乱数など）を制御下に置かないと見本の再現性が保てない。Vitest のモック機構と設計を揃えることで、Storybook の見本と Vitest のテストで**同じモック定義を共有**でき、「見本＝テスト」の一体運用（Storybook Test）が滑らかになる。CSF Factories は、ストーリー定義をより型安全・宣言的にするための新フォーマットで、段階的昇格（Experimental→Preview→Default）で破壊を抑えつつ移行する定石を踏む。

### 実務での使いどころ・移行の勘所

- デザインシステム／共通 UI ライブラリのカタログとリグレッション基盤として引き続き中核。Vitest 4 の Browser Mode／ビジュアル回帰と組み合わせると「見本がそのままテスト」になる。
- 移行時はまず ESM 前提へ設定・addon を寄せる。CJS 依存の独自 addon や古い設定ファイルは要対応。
- `sb.mock` へ既存のモック手法を寄せると、Storybook とユニットテストでモック定義を一本化できる。
- CSF Factories は今のうちに Preview として試し、11 での既定化に備える。

## 📖 用語解説（このトピックとのつながり）

- **ESM 専用（ESM-only）**: CommonJS を排し JS 標準モジュールのみで配布する形態。10 の install 削減・デバッグ性向上の根拠。
- **モジュールモック**: テスト時に依存モジュールを差し替える手法。`sb.mock` がこれを担い、Vitest と定義を共有できる。
- **CSF（Component Story Format）／CSF Factories**: ストーリー記述の標準フォーマットと、その型安全な新形。10 で Preview に昇格。
- **dual package hazard**: CJS/ESM 両出しで同一モジュールが二重ロードされる罠。ESM 専用化で回避する。

## 影響範囲 / 推奨アクション

- Storybook 利用チーム: ESM 前提へ設定・addon を移行し、10 系へ更新。install 削減とデバッグ性を享受。
- テスト運用: `sb.mock` と Vitest 4 でモックを一本化、ビジュアル回帰を主要 UI に導入。
- 将来: CSF Factories を Preview で先行採用し、Storybook 11 の既定化に備える。
