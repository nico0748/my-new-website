# ZenStack v3 — アクセス制御を組み込んだTypeScript ORM

> トピック: 開発トピック｜出典: dev.to｜種別: backend

## 一行サマリー
ZenStack v3 は ORM のスキーマ層に「誰がどのデータを読み書きできるか」というアクセス制御ポリシーを直接記述できる TypeScript ORM で、認可ロジックの重複や漏れを防ぐアプローチとして注目されている。

## 🔰 初学者向け（何が・なぜ重要か）
Web アプリでは「このユーザーは自分の投稿しか編集できない」といったルール（認可）が必ず必要です。通常こうしたルールは、API の各所に if 文で散らばって書かれ、書き忘れると情報漏洩につながります。

ZenStack は、このルールをデータベースのスキーマ（設計図）のすぐ隣に一元的に書けるようにします。建物でいえば、各部屋のドアごとに鍵の設定を貼り紙するのではなく、建物の設計図に「この部屋は所有者のみ入室可」と最初から書き込んでおくイメージです。設計図に書いてあれば、どの入口から来ても同じルールが自動で効きます。

これにより、認可の抜け漏れが減り、フロントエンドから安全に呼べる自動生成 API まで一気通貫で得られます。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
ZenStack はもともと Prisma の上に「アクセスポリシー」層を重ねる拡張として登場しましたが、v3 では設計が刷新され、より独立した TypeScript ORM としての性格を強めています（v3 の Prisma 依存度・内部構成の詳細は要確認）。スキーマに `@@allow` / `@@deny` のようなポリシー宣言を書くと、クエリ実行時にそのルールが自動適用され、加えて CRUD API や型安全なクライアントフックの自動生成が可能です。

### 仕組み / 背景（なぜこう設計されたか）
従来はアプリ層のミドルウェアやサービス層で認可を実装するのが一般的でしたが、ルールが分散し監査が難しくなる問題がありました。ZenStack は「データにルールを密結合させる」ことで、どの経路から来ても一貫した認可を保証します。トレードオフは、ポリシーをスキーマに寄せる分、複雑な業務ルールが宣言的に表現しづらい場合がある点です。

### 実務での使いどころ・移行の勘所
マルチテナントSaaSや、行レベルの権限が重要なアプリに向きます。Prisma を既に使っているチームは比較的移行しやすい一方、Drizzle のような低レベル志向の ORM とは思想が異なるため、認可の一元化をどこまで重視するかで選択が分かれます。

## 📖 用語解説（このトピックとのつながり）
- **認可（Authorization）**：認証済みユーザーが何をできるかを決める仕組み。ZenStack の中心テーマ。
- **アクセスポリシー**：ZenStack がスキーマに書けるルール宣言。行/列単位で適用される。
- **マルチテナント**：複数顧客のデータを1つのDBで扱う構成。厳密な認可が必須で ZenStack の得意領域。
- **Prisma**：ZenStack が影響を受けた ORM。v3 での関係性は要確認。

## 影響範囲 / 推奨アクション
認可ロジックの複雑なアプリを開発するチームに影響します。認可を「後付けのミドルウェア」から「データ層の宣言」へ移したい場合、ZenStack を評価対象に入れる価値があります。既存の認可実装が安定しているなら急いだ移行は不要です。
