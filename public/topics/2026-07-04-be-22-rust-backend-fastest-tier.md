# Rust バックエンドが最速帯 — Axum/Actix が Go・ASP.NET を上回る

> トピック: 開発トピック｜出典: pockit.tools｜種別: backend

## 一行サマリー
2026 年の API フレームワーク性能比較で、Rust 製の Axum / Actix が最速帯を占め、Go(Gin)、ASP.NET Core がそれに続く序列が示された。

## 🔰 初学者向け（何が・なぜ重要か）
Web サービスの「速さ」には、1 秒あたりにさばけるリクエスト数（スループット）と、1 件あたりの応答時間（レイテンシ）という 2 つの尺度があります。同じ機能でも、どの言語・フレームワークで書くかによってこの数字が大きく変わります。

Rust は、ガベージコレクション（不要メモリを自動回収する仕組み）を持たず、メモリ管理をコンパイル時に厳密に決める言語です。実行中に「掃除で一瞬止まる」時間が生まれにくいため、応答が安定して速くなります。これが Axum / Actix が最速帯にいる理由です。

ただし「最速＝常に最適」ではありません。Rust は学習コストが高く開発速度が落ちやすい一方、Go は書きやすさと十分な速さのバランスが良く、ASP.NET Core は企業向けの機能が豊富です。速さは選定要素の一つにすぎない、という視点が大切です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
比較記事の序列は概ね「Axom/Actix(Rust) > Go(Gin) > ASP.NET Core」。Rust 勢はゼロコスト抽象と GC 無しにより、高負荷時の p99 レイテンシが安定する傾向があります。具体的な RPS 数値はベンチ条件（ペイロード、並行数、DB 有無）で大きく振れるため、実測値は一次ソースで要確認です。

### 仕組み / 背景（なぜこう設計されたか）
GC を持つランタイム（Go、.NET、JVM 系）は、回収タイミングで短い停止（ストップ・ザ・ワールド）が入り得ます。Go は GC を高度に最適化して停止を短く抑えていますが、原理的にはゼロにできません。Rust は所有権モデルで解放時点を静的に決めるため、この停止自体が発生しません。高並行・低レイテンシ要件で Rust が有利になるのはこの構造差によります。

### 実務での使いどころ・移行の勘所
超低レイテンシが売上に直結する領域（広告入札、決済ゲートウェイ、リアルタイム配信）では Rust の採用価値が高いです。一方、開発速度と人材確保を重視する一般的な業務システムでは Go や ASP.NET Core が現実解になります。ベンチ数値だけで移行を決めず、運用・採用・保守コストを含めた総所有コストで判断すべきです。

## 📖 用語解説（このトピックとのつながり）
- **スループット (RPS)**: 秒間処理リクエスト数。フレームワーク比較の主要指標。
- **p99 レイテンシ**: 応答時間の 99 パーセンタイル。最悪ケースの体感速度を表す。
- **ガベージコレクション (GC)**: 不要メモリの自動回収。停止時間がレイテンシ安定性に影響する。
- **Gin**: Go の高速 Web フレームワーク。Rust 勢に次ぐ性能帯の代表格。

## 影響範囲 / 推奨アクション
性能要件が厳しいサービスの技術選定者に影響します。まず自社ワークロードで小規模なベンチを取り、Rust 採用の性能利得が開発コスト増を上回るかを見極めた上で決定するのが賢明です。
