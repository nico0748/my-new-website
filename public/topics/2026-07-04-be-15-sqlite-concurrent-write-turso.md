# SQLite 同時書き込みプレビュー（Turso）— Rust 書き換えとMVCCでACID維持

> トピック: 開発トピック｜出典: devnewsletter（State of Databases 2026）｜種別: backend

## 一行サマリー
Turso による SQLite の Rust 書き換え版で、SQLite の長年の弱点だった「同時書き込み」がプレビュー提供された。MVCC で複数の書き込みを並行させつつ ACID を維持する設計が要点。

## 🔰 初学者向け（何が・なぜ重要か）
SQLite は、サーバーを立てずにファイル1つで完結する超軽量なデータベースで、スマホアプリやブラウザ、組み込み機器など至る所で使われています。手軽さが最大の魅力ですが、伝統的に「一度に一人しか書き込めない（書き込みはロックされる）」という制約がありました。

Turso はこの SQLite を Rust という安全性の高い言語で書き直し、「複数の書き込みを同時に行える」ようにするプレビューを出しました。これにより、書き込みが多いアプリでも SQLite が使いやすくなります。

たとえるなら、これまで一冊の帳簿に一人ずつしか記入できなかったのを、複数人が同時に別ページへ記入でき、しかも記録の整合性（誰かの途中書きが混ざらない）は保たれる、という改善です。SQLite の手軽さを保ちながらスケールしやすくなります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Turso（旧 libSQL 系譜）は SQLite 互換エンジンを Rust で再実装し、MVCC（多版型同時実行制御）ベースで同時書き込みを実現するプレビューを公開した。従来 SQLite は書き込みをファイル単位で直列化していたが、MVCC 版は各トランザクションが独自のスナップショットで動くため、書き込み同士のブロックを減らせる。ACID（原子性・一貫性・独立性・永続性）は維持するとされる。実装は継続開発中で、本番採用は安定版の到来を待つ必要がある（要確認）。

### 仕組み / 背景（なぜこう設計されたか）
MVCC は PostgreSQL などでも使われる仕組みで、「データを上書きせず新しい版として書き足し、読み手には整合的な過去スナップショットを見せる」ことで読み書きの衝突を減らす。SQLite の単一ライタ制約はエッジ/分散環境でボトルネックになりやすく、Turso はこれを解いてエッジDB（世界各地にレプリカを置くDB）としての価値を高めようとしている。

### 実務での使いどころ・移行の勘所
書き込みが多い、あるいは分散配置したいユースケースで従来 SQLite を諦めていた場合の有力候補。ただしプレビュー段階のため、まずは非クリティカルな用途や検証環境から。既存 SQLite との互換性・移行手順、そして本番SLAが必要な場合は安定版リリースまで待つのが堅実。

## 📖 用語解説（このトピックとのつながり）
- **SQLite**: ファイル1つで動く組み込みDB。同時書き込み制約が今回の改善対象。
- **MVCC（多版型同時実行制御）**: データを版管理して読み書き衝突を減らす手法。同時書き込みの土台。
- **ACID**: トランザクションの信頼性を保証する4性質。MVCC 化後も維持がうたわれる。
- **Turso / libSQL**: SQLite 互換のエッジDB系譜。今回の Rust 書き換えの担い手。

## 影響範囲 / 推奨アクション
SQLite を書き込み集約・分散用途で使いたい開発者が対象。プレビュー段階のため検証環境で評価し、互換性と安定性を確認してから本番採用を判断すること。
