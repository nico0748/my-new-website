# TypeScript 7.0 RC — コンパイラを Go で書き直した「10倍速」の型検査が現実に

> トピック: TypeScript 7.0 RC（"Project Corsa"）｜出典: Microsoft TypeScript 公式ブログ｜種別: メジャーバージョン RC

## 一行サマリー

TypeScript のコンパイラと言語サービスを Go で完全に書き直した TypeScript 7.0 が 2026年6月18日に RC（リリース候補）へ到達し、正式版が目前に迫った。VS Code 級（約150万行）のコードベースで型検査が約77秒→7.5秒と約10倍に短縮され、大規模プロジェクトのエディタ応答性とビルド時間が根本的に改善する。

## 🔰 初学者向け（何が・なぜ重要か）

TypeScript は「JavaScript に型（この変数は数値、この関数は文字列を返す、といった約束事）を付けて、書いている最中にミスを見つけてくれる」言語です。エディタで赤い波線が出るのも、`tsc` でビルド時にエラーが出るのも、すべて TypeScript コンパイラという「翻訳＆点検係」が働いているおかげです。

ところがこの点検係、これまでは TypeScript 自身（＝ JavaScript）で書かれていました。自分で自分を翻訳しているようなもので、プロジェクトが大きくなるほど点検に時間がかかり、「保存するたびに数秒固まる」「CI のビルドが遅い」という悩みの種でした。

7.0 では、この点検係を **Go というコンパイル型の言語で丸ごと作り直しました**。Go は機械語に近い速さで動き、複数の CPU コアを同時に使えます。結果、点検が桁違いに速くなります。料理でいえば、これまで一人で全部の下ごしらえをしていたのを、複数の調理台に分担させて一気に進められるようになったイメージです。しかも「書き方」や「型のルール」自体は変わらないので、私たちのコードはそのままで速さだけを受け取れます。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- コンパイラ／言語サービスを TypeScript（JS）実装から **Go ネイティブ実装**へ全面移植（コードネーム "Project Corsa"）。7.0 はこの Go 実装が土台。
- 2026年6月18日に **RC 到達**、正式版は直後を予定（正確な GA 日は要確認）。
- 直前の **TypeScript 6.0**（2026年3月23日）は JS ベース最後のメジャーで、非推奨・既定値変更・移行フラグを入れて 7.0 への橋渡しを担った。
- 性能: VS Code（約150万行）で型検査が約77秒→7.5秒（約10倍）。Playwright・TypeORM 等でも 8〜13倍の高速化を報告。
- メモリ使用量も削減、共有メモリのマルチスレッド化でスケーラビリティが向上。
- 言語仕様・型システムの意味論は原則そのまま（挙動互換を最優先）。差異が出る箇所は移行ガイドで明示。

### 仕組み / 背景（なぜこう設計されたか）

なぜ「別言語での書き直し」という大手術に踏み切ったのか。JS 実装の tsc は、シングルスレッドかつ GC を持つランタイム上で膨大な型グラフを走査する構造で、プロジェクト規模に対して型検査コストが超線形に伸びやすかった。エディタの補完・ホバー・リネームはこの型検査結果に依存するため、大規模モノレポでは「1操作＝数百 ms〜数秒」の待ちが積み重なる。

Go を選んだのは、(1) 手動 GC 言語（Rust 等）ほど所有権モデルに縛られず既存の可変・循環参照だらけの AST/型構造を移植しやすい、(2) それでいてネイティブコンパイル＋軽量ゴルーチンで真の並列化ができる、という「移植しやすさ×並列性能」のバランスによる。型検査はファイル／プログラム単位で並列化の余地が大きく、共有メモリ・マルチスレッドが素直に効く領域である。

トレードオフは、TypeScript がこれまで享受してきた「JS エコシステムへの容易な組み込み（ブラウザや任意の JS ランタイムで tsc を動かす）」から、プラットフォーム別ネイティブバイナリの配布へ移る点。Web 上で tsc を直接動かす一部ツールや、コンパイラ内部 API を叩くプラグイン群は移行対応が要る。互換性維持のため公式は当面 JS 版（6.x 系）も並存させる方針。

### 実務での使いどころ・移行の勘所

- 大規模モノレポ・大量の型パズルを抱えるプロジェクトほど恩恵が大きい。CI の tsc フェーズ短縮とエディタ体感の改善が直結する。
- まず **6.0 へ上げて非推奨・移行フラグの警告を潰す**のが公式推奨の順路。7.0 は挙動互換だが、6.0 で炙り出される古い書き方を先に整理しておく。
- コンパイラ内部 API（`typescript` パッケージの型チェッカ API 等）に依存するツール（一部の型駆動コード生成・独自 Lint・ts-morph 系）は 7.0 対応状況を確認。API は再設計されており移植期間中は両対応が現実的。
- ビルドは高速化するが、型システムの厳密さは不変。「速くなった＝ゆるくなった」ではない点に注意。

## 📖 用語解説（このトピックとのつながり）

- **Project Corsa**: 今回の Go 書き直しプロジェクトの旧コードネーム。7.0 の正体。
- **言語サービス（Language Service）**: 補完・ホバー・リネーム等をエディタへ供給する API 層。型検査結果に依存するため、7.0 の並列化はここの体感速度を直接押し上げる。
- **共有メモリ・マルチスレッド**: 複数スレッドが同じメモリ上の型グラフを並行処理する方式。JS 実装では困難だった並列化を Go で実現した核心。
- **移行フラグ**: 6.0 が導入した、7.0 で変わる挙動を先に警告・切替するオプション。段階的移行の要。

## 影響範囲 / 推奨アクション

- アプリ開発者: 6.0 へ更新して警告を解消 → 7.0 RC を CI で試験導入し、型検査時間とエラー差分を計測。
- ツール／ライブラリ作者: コンパイラ API 依存箇所の 7.0 対応を確認・追随。
- CI: `tsc` ジョブの所要時間ベースラインを取り直し、キャッシュ戦略を見直す好機。
