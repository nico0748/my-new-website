# Drizzle ORM — SQLに近い次世代TypeScript ORM

> トピック: 開発トピック｜出典: Drizzle 公式｜種別: backend

## 一行サマリー
Drizzle ORM は「SQL をそのまま TypeScript で書く」思想の軽量 ORM で、スキーマ定義から GraphQL サーバをほぼ1行で生成できるなど、Prisma に代わる選択肢として2026年に採用が広がっている。

## 🔰 初学者向け（何が・なぜ重要か）
ORM（オブジェクト関係マッピング）は、データベースの表を JavaScript/TypeScript のオブジェクトとして扱えるようにする道具です。SQL を直接書かずに `user.findMany()` のように操作できるので便利ですが、便利すぎるとどんな SQL が発行されているのか分からなくなり、性能問題の原因になりがちです。

Drizzle は「SQL を隠さない」という逆張りの発想で作られています。料理でいえば、出来合いのソースを使う代わりに、材料と手順が見える自炊に近い。書き味は SQL とほぼ同じなので、SQL を知っている人ほど直感的に使え、生成されるクエリも予測しやすいのが特徴です。

さらに Drizzle はブラウザや Cloudflare Workers のようなエッジ環境でも動く軽さを持ち、余計な実行時レイヤ（別プロセスのクエリエンジン）を必要としない点も評価されています。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Drizzle は TypeScript で書いたスキーマ定義がそのまま型情報になり、クエリの結果型が自動で推論されます。マイグレーションは `drizzle-kit` が担当し、スキーマ差分から SQL マイグレーションを生成します。PostgreSQL / MySQL / SQLite に加え、各種サーバレス DB ドライバへ対応します。スキーマから GraphQL API を生成する連携機能も提供されています（具体的な対応バージョン・生成物の詳細は要確認）。

### 仕組み / 背景（なぜこう設計されたか）
従来の Prisma は独自スキーマ言語（`schema.prisma`）と、実行時にネイティブのクエリエンジンを介する構成でした。これは強力な反面、バンドルサイズやコールドスタート、エッジ環境との相性で不利になる場面がありました。Drizzle は「TypeScript とゼロ依存の薄い層だけ」で完結させることで、この課題を解消しようとしています。トレードオフとして、SQL を意識する分だけ抽象度は低く、開発者に SQL の知識を求めます。

### 実務での使いどころ・移行の勘所
エッジ/サーバレス（Cloudflare Workers、Vercel Edge 等）でのデプロイや、発行 SQL の透明性を重視するチームに向きます。Prisma からの移行はスキーマとクエリの書き換えが必要で、Prisma が提供する一部の高レベル機能（リレーション自動解決など）は自前で書く必要が出ます。既存の Prisma 資産が大きい場合は段階移行が現実的です。

## 📖 用語解説（このトピックとのつながり）
- **ORM**：DB のレコードとプログラムのオブジェクトを対応づける仕組み。Drizzle はその中でも「SQL 寄り」の設計。
- **drizzle-kit**：スキーマからマイグレーション SQL を生成する CLI。Drizzle 運用の中核。
- **型推論**：クエリ結果の型を自動導出する仕組み。Drizzle の安全性の源泉。
- **エッジランタイム**：ユーザー近傍で動く軽量実行環境。Drizzle の軽さが活きる場所。

## 影響範囲 / 推奨アクション
TypeScript でバックエンドや API を書く開発者に影響します。新規プロジェクトでエッジ配備や SQL の可視性を重視するなら Drizzle を有力候補に。既存 Prisma プロジェクトは急いで移行せず、性能やエッジ要件が顕在化した時点で比較検討するのが妥当です。
