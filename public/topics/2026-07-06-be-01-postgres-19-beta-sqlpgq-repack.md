# PostgreSQL 19 Beta 1 — SQL/PGQ グラフクエリ・REPACK CONCURRENTLY・外部キー検査 最大2倍

> トピック: PostgreSQL 19 Beta 1 の新機能｜出典: PostgreSQL 公式ニュース｜種別: メジャーリリースのベータ

## 一行サマリー

2026-06-04 リリースの PostgreSQL 19 Beta 1 は、SQL 標準のプロパティグラフ問い合わせ（SQL/PGQ）、テーブルを止めずに再編成する組み込み REPACK CONCURRENTLY、外部キー制約チェックが最大2倍速くなる最適化など 200 超の変更を含み、正式版は 2026 年秋頃の見込み。

## 🔰 初学者向け（何が・なぜ重要か）

PostgreSQL（ポスグレ）は、世界で最も広く使われている高機能なデータベースの1つです。19 のベータ版が出て、正式版に向けた新機能が試せるようになりました。今回の目玉は3つあります。

1つめは「グラフ問い合わせ」です。SNSの友達関係のような「つながり」を辿る検索を、これまでは複雑な SQL や専用のグラフDBで書いていました。19 では SQL 標準の書き方（SQL/PGQ）で「AさんとBさんは何人を介してつながっているか」といった問いを、ポスグレの中で素直に書けるようになります。

2つめは「止めずにお掃除」です。データを長く使うとファイルに無駄な隙間が溜まって遅くなります。従来は掃除の間テーブルを一時的にロック（使用停止）する必要がありましたが、19 の REPACK CONCURRENTLY は稼働させたまま再編成できます。

3つめは単純な高速化で、テーブル間の整合性チェック（外部キー）が最大2倍速くなります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **SQL/PGQ（プロパティグラフクエリ）**: SQL 標準のグラフパターンマッチング構文をサポート。リレーショナルなテーブルをグラフとして扱い、パス探索的な問い合わせを SQL で記述できる。
- **REPACK CONCURRENTLY（組み込み）**: テーブル/インデックスの膨張（bloat）を、書き込みを止めずに再編成。従来 `pg_repack` 拡張で外部的に行っていた作業を本体に取り込んだ位置づけ。
- **外部キー検査の高速化**: 制約検証アルゴリズムの最適化により FK チェックが最大約2倍。
- **論理レプリケーション改善**ほか、性能・DX・セキュリティ・監視・クエリ機能にまたがる 200 超の変更。
- 位置づけは Beta 1（2026-06-04）。正式版は 2026 年 9〜10 月頃見込み。数値は公式ニュース/リリースノートに基づき、細部は正式版で要確認。

### 仕組み / 背景（なぜこう設計されたか）

SQL/PGQ 採用の背景は「多くのグラフ的問い合わせのために専用DBを別途運用するのは重い」という現場の実感だ。ポスグレは既に JSON・全文検索・ベクトル（pgvector）などを取り込み「1つで済ませる」路線を強めており、グラフもその延長にある。プロパティグラフはノード（頂点）とエッジ（辺）にプロパティを持たせるモデルで、SQL/PGQ はそれを既存テーブル上の「ビュー」として定義し、宣言的にパターンを書けるようにする。トレードオフは、専用グラフDB（Neo4j 等）ほど深いグラフ探索最適化は期待しにくい点で、超大規模・多ホップ探索が主用途なら専用DBが優位な場面は残る。

REPACK CONCURRENTLY は MVCC（多版型同時実行制御）に由来する宿命的な課題への対処だ。ポスグレは更新/削除で「死んだ行（dead tuple）」を残し、VACUUM で回収するが、物理的な膨張はオンラインでは縮まない。従来の VACUUM FULL は排他ロックで運用を止めてしまう。REPACK CONCURRENTLY は新しい物理レイアウトを裏で構築し、追いつき差分を反映して切り替えることでロック時間を最小化する。代償として、再編成中は追加のディスク容量と I/O を要する。

### 実務での使いどころ・移行の勘所

グラフ的な問い合わせ（権限の到達可能性、依存関係、レコメンド初段）を既にポスグレ内で無理な自己結合で書いているなら SQL/PGQ は有力。ただし本番採用は正式版待ちが無難で、Beta では構文・計画性の検証に留める。REPACK CONCURRENTLY は、これまで `pg_repack` を cron 運用していた現場の標準化・簡素化に効く。FK 高速化は書き込み多発系で無償の恩恵。移行時は 200 超の変更のうち非互換項目（設定・型・関数挙動）をリリースノートで洗い、まずステージングで回帰テストする。

## 📖 用語解説（このトピックとのつながり）

- **SQL/PGQ**: SQL 標準のプロパティグラフ問い合わせ。19 でグラフ探索を SQL 化する中核。
- **プロパティグラフ**: ノードとエッジにプロパティを持たせるデータモデル。SQL/PGQ の対象。
- **bloat（膨張）**: 更新/削除で残る無駄領域。REPACK が解消する対象。
- **MVCC**: 同時実行のための多版管理。膨張が起きる根本理由。
- **外部キー（FK）**: テーブル間整合性の制約。19 で検査が最大2倍高速化。

## 影響範囲 / 推奨アクション

- 対象: PostgreSQL を使うバックエンド全般。
- 推奨: Beta で SQL/PGQ・REPACK を検証。書き込み多発系は FK 高速化を評価。
- 注意: 本番採用は正式版（秋頃）まで待つのが無難。非互換項目をリリースノートで確認。
