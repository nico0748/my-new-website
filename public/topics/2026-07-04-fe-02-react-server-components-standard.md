# React Server Components が本番標準へ成熟

> トピック: 開発トピック｜出典: nucamp｜種別: frontend

## 一行サマリー
React Server Components（RSC）が実験段階を脱し、本番運用の標準的な選択肢として成熟した。サーバー側でコンポーネントを描画し、クライアントに送る JavaScript を減らすことで、初期表示の速さとデータ取得の簡潔さを両立する。

## 🔰 初学者向け（何が・なぜ重要か）
これまでの React は、基本的にブラウザ（クライアント）側で JavaScript を実行して画面を組み立てていました。そのため、アプリが大きくなるほどブラウザに届く JavaScript が増え、最初の表示が重くなりがちでした。

React Server Components は、一部のコンポーネントを「サーバー側で完成させてから」ブラウザに送るという発想です。たとえば記事一覧のように、ユーザー操作でぐりぐり動く必要がない部分はサーバーで作ってしまい、ブラウザには完成品だけを渡す。ボタンやフォームなど動きが必要な部分（Client Components）だけ従来どおりブラウザで動かす、という役割分担です。

これにより、ブラウザに送る JavaScript が減って表示が速くなり、データベースへの問い合わせもサーバーコンポーネント内で直接書けるため、API を経由する手間が減ります。「重い調理場はサーバーに置き、盛り付けだけブラウザで」というイメージが近いでしょう。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
出典によれば RSC は production-standard に成熟したとされる。Next.js の App Router をはじめ主要フレームワークが RSC を前提とした設計へ移行しており、サーバーコンポーネントを既定、クライアントコンポーネントを `"use client"` で明示する構成が定着している（各フレームワークの対応状況は個別に要確認）。

### 仕組み / 背景（なぜこう設計されたか）
RSC はコンポーネントを「サーバーでのみ実行され JS を送らないもの」と「クライアントで対話性を持つもの」に分離する。サーバーコンポーネントの出力は特殊なシリアライズ形式でストリーミング配信され、クライアントで React ツリーに統合される。狙いは JS バンドル削減とデータ取得の局所化。トレードオフとして、サーバー/クライアント境界（どこで何が実行されるか）を意識する必要があり、状態やイベントハンドラはクライアント側にしか置けないなどの制約が生じる。

### 実務での使いどころ・移行の勘所
コンテンツ主体でSEOと初期表示速度が重要なサイトに有効。移行時は「対話性が必要な葉（leaf）だけをクライアント化する」設計が肝で、境界を上位に置きすぎると RSC の利点が薄れる。既存の CSR 前提ライブラリはサーバーコンポーネント内で使えないことがあり、依存関係の棚卸しが必要。

## 📖 用語解説（このトピックとのつながり）
- **RSC（React Server Components）**: サーバーで描画しJSを送らないコンポーネント。本トピックの主役。
- **Client Component**: `"use client"` を付けてブラウザで動く従来型コンポーネント。対話性を担う。
- **ストリーミング**: 描画結果を部分ごとに逐次送る配信方式。初期表示を早める。
- **App Router**: RSC を前提とした Next.js のルーティング機構。
- **サーバー/クライアント境界**: どのコードがどちら側で実行されるかの分かれ目。設計上の要点。

## 影響範囲 / 推奨アクション
React/Next.js 系の新規・既存プロジェクトに広く影響。新規はRSC前提の設計を、既存はクライアント境界を意識した段階移行を推奨。CSR依存ライブラリの互換性確認を先に行うとよい。
