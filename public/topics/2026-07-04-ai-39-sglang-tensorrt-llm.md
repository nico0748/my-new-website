# SGLang / TensorRT-LLM — 大規模 LLM サービングの二大選択肢

> トピック: 開発トピック｜出典: bizon-tech｜種別: ai

## 一行サマリー
大規模 LLM サービングでは、プログラマブルな高速エンジン SGLang と、NVIDIA GPU に最適化した TensorRT-LLM が有力な選択肢として並び立ち、vLLM と合わせて用途別に使い分ける構図になっている。

## 🔰 初学者向け（何が・なぜ重要か）
LLM を大量のユーザーに提供するとき、どの「推論エンジン」を使うかで速度もコストも大きく変わります。ここで注目されているのが SGLang と TensorRT-LLM です。

SGLang は「構造化された生成」を得意とするエンジンです。例えば「JSON 形式で必ず返す」「複数の分岐を並行して処理する」といった複雑な生成パターンを、プログラムのように記述して高速に実行できます。RadixAttention という仕組みで、共通する文脈（プロンプトの前半など）を賢く使い回し、無駄な再計算を減らします。

TensorRT-LLM は NVIDIA が自社 GPU 向けに徹底的にチューニングしたライブラリです。GPU の性能を限界まで引き出すため、NVIDIA ハードを使う環境では最速クラスの性能が出ます。汎用性の SGLang、NVIDIA 特化の TensorRT-LLM、という住み分けです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
出典は大規模サービングの選択肢として SGLang と TensorRT-LLM を挙げる。SGLang は RadixAttention によるプレフィックスキャッシュ共有、制約付きデコード、並行実行を特徴とし、複雑な生成ワークフローで高スループットを出す。TensorRT-LLM は NVIDIA GPU 向けのカーネル最適化・量子化（FP8 等）・インフライトバッチングで最高峰の性能を狙う。両者の具体的バージョン・ベンチは要確認。

### 仕組み / 背景（なぜこう設計されたか）
LLM サービングの効率化には「重複計算をいかに削るか」が重要。SGLang の RadixAttention は、複数リクエストで共通するプロンプト接頭辞の KV キャッシュを木構造で共有し、再計算を回避する。エージェントや RAG のように似た文脈が繰り返される用途で特に効く。TensorRT-LLM はハードとソフトを垂直統合し、NVIDIA GPU のテンソルコアや低精度演算を極限まで使う。トレードオフは、TensorRT-LLM が NVIDIA 依存でモデル変換（エンジンビルド）の手間がある点。

### 実務での使いどころ・移行の勘所
構造化出力・エージェント・多分岐生成が多いなら SGLang が有利。NVIDIA GPU で純粋な最高性能を求めるなら TensorRT-LLM。汎用的な導入しやすさなら vLLM（項目34）。三者はベンチマークで比較検討するのが定石で、Triton（項目38）のバックエンドとして TensorRT-LLM を組み合わせる構成も一般的。

## 📖 用語解説（このトピックとのつながり）
- **SGLang**: 構造化生成に強いプログラマブル推論エンジン。本トピックの一方の主役。
- **RadixAttention**: プロンプト接頭辞の KV キャッシュを木構造で共有する SGLang の技法。
- **TensorRT-LLM**: NVIDIA GPU 向け LLM 最適化ライブラリ。もう一方の主役。
- **制約付きデコード**: 出力を JSON など指定形式に強制する生成手法。
- **FP8 量子化**: 8bit 浮動小数点でモデルを軽量・高速化する技術。

## 影響範囲 / 推奨アクション
大規模に LLM を提供する MLOps・インフラ担当に影響。構造化出力やエージェント負荷が多いなら SGLang、NVIDIA GPU で最高性能なら TensorRT-LLM を評価対象に。vLLM を含む三者でベンチマークを取り、ワークロード特性で選ぶとよい。
