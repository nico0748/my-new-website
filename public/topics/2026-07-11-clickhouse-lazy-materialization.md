# ClickHouse の遅延マテリアライゼーション — 列を必要になるまで読まない最適化

> トピック: ClickHouse の Lazy Materialization（遅延実体化）｜ 出典: ClickHouse 公式ブログ / Docs ｜ 種別: クエリ実行エンジンの最適化

## 一行サマリー

列指向の分析DB ClickHouse が導入した「遅延マテリアライゼーション」は、列データを実際に必要になる瞬間まで読み込みを先送りする最適化。25.4（2026-03）で既定有効化され、上位N件を並べ替えて絞り込む Top-N クエリなどで桁違いの高速化を実現。1.5億行の例では同一クエリが0.139秒・ピークメモリ3.8MiBで完了した。

## 🔰 初学者向け（何が・なぜ重要か）

ClickHouse は「巨大なデータを SQL で猛烈に速く集計する」ことに特化した分析用データベースだ。ログ解析やダッシュボード、リアルタイム集計などで使われる。

今回の **遅延マテリアライゼーション（lazy materialization）** は、その速さをさらに引き上げる賢い工夫だ。名前は難しいが、発想はシンプル——「**要らない列は、要ると分かるまで読まない**」。

たとえば「レビュー1.5億件から、評価の高い上位10件を取り出して、その本文を表示したい」というクエリを考える。素直にやると全件の"本文"まで読み込んでから絞り込むことになり、大量の無駄が出る。遅延マテリアライゼーションは違う。まず並べ替えに必要な"評価"の列だけを読んで上位10件を確定し、**重い"本文"の列は最後に、その10件分だけ読む**。図書館で「人気トップ10の本文コピー」が欲しいとき、全部の本をコピーするのではなく、まず貸出ランキングだけ見て、トップ10の本文だけコピーするのと同じだ。

結果、読むデータ量とメモリが激減し、クエリが劇的に速くなる。実測で1.5億行のクエリが0.139秒・メモリわずか3.8MiB。分析基盤の応答が速く・安く済むのは、ダッシュボードを使う全員に効いてくる。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **遅延マテリアライゼーション**: 列データの読み込みを実際に必要になるまで遅延させるクエリ最適化。ClickHouse **25.4（2026-03リリース）**で導入され、`PREWHERE` とともに**既定で有効**。
- **効き所**: 大規模データを `ORDER BY` して `LIMIT` する Top-N クエリで特に効果大。並べ替え・絞り込みに使う列だけ先に読み、重い列は最終的に残った行の分だけ読む。
- **実測例**: 1.5億行の Amazon レビューで、同一クエリが遅延マテリアライゼーション有効時 **0.139秒**、処理 1.81GB、ピークメモリ **3.80MiB** で完了。
- **後続改善**: 25.12 で遅延読み込みが **join スタイルの実行モデル**に切り替わり、ベクトル化・並列実行の恩恵を受けて大きな `LIMIT` でも効率的に。過去1年で相関サブクエリ・遅延マテリアライゼーション・ランタイムフィルタ・自動 JOIN 並べ替えにより **JOIN 性能が6倍超**改善。

### 仕組み / 背景（なぜこう設計されたか）

**列指向 × I/O 最小化**。ClickHouse は列ごとにデータを格納するため、クエリで使う列だけを読める（行指向より有利）。しかし従来は、クエリに登場する列を早い段階でまとめて読み出す（early materialization）ことがあり、最終的に少数行しか返さない Top-N でも、フィルタ前に重い列を全件読む無駄が生じていた。遅延マテリアライゼーションは「フィルタ・ソートに必要な軽い列 → 行の絞り込み → 残った行だけ重い列を読む」と段階化し、読み込みを必要最小限にする。データベース教科書でいう late materialization の実装で、`PREWHERE`（条件を先に評価して行を間引く仕組み）と相補的に働く。

**なぜ join スタイル実行へ**。素朴な遅延読み込みは「絞り込んだ行 ID を使って重い列を後から引く」ため、`LIMIT` が大きいと後追い読み出しが逐次的でボトルネックになりうる。25.12 では、この後追いを JOIN と同じベクトル化・並列実行パイプラインに載せ、大きな `LIMIT` でもスループットを保つ。ClickHouse の実行エンジンが一貫して「ベクトル化された列演算」を志向していることの現れだ。

**トレードオフ**。遅延化は「後から重い列を引く」オーバーヘッドを伴うため、返す行が多い（=絞り込みが効かない）クエリでは必ずしも得にならない。ClickHouse はコストベースで適用可否を判断し、既定 ON でも不利なケースは避ける設計。ユーザーが意識せず恩恵を受けられる"透過的最適化"を狙っている。

### 実務での使いどころ・移行の勘所

ログ／イベント分析、BI ダッシュボード、ランキング・上位抽出系のクエリで効果が大きい。25.4 以降にアップグレードするだけで既定有効になるため、多くの場合コード変更なしで速くなる。特に「広いテーブルから少数の上位行＋その詳細列」を返すパターンで恩恵が顕著。移行の勘所は、既存クエリのプロファイル（`EXPLAIN`／実行統計）で読み込みバイト数・メモリの改善を確認すること。絞り込みが効かない全件走査系では効果が薄いので、そこは別途 `PREWHERE` やインデックス（スキップインデックス）設計で最適化する。

## 📖 用語解説（このトピックとのつながり）

- **マテリアライゼーション（実体化）**: クエリ実行中に列データを実際にメモリへ読み出すこと。「いつ読むか」が性能を左右する。
- **early / late materialization**: 早期＝先に列を読む、遅延＝必要になるまで読まない。本トピックは後者の実装。
- **PREWHERE**: 条件を早期評価して行を間引く ClickHouse の仕組み。遅延実体化と相補的。
- **Top-N クエリ**: `ORDER BY ... LIMIT n` で上位 n 件を返すクエリ。遅延実体化が最も効く形。
- **ベクトル化実行**: 値をまとめて処理する実行方式。25.12 の join スタイル遅延読み込みが活用。
- **スキップインデックス**: 不要なデータブロックを読み飛ばす ClickHouse のインデックス。全件走査系の補完策。

## 影響範囲 / 推奨アクション

- ClickHouse を BI・ログ分析で使うなら 25.4 以降へ上げるだけで Top-N 系が高速化（既定有効）。
- 既存クエリを `EXPLAIN`／実行統計でプロファイルし、読み込みバイト・メモリ改善を確認。
- 絞り込みが効かない全件走査は効果が薄いため、`PREWHERE`・スキップインデックス設計で補完。
