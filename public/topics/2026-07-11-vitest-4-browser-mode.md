# Vitest 4.0 — Browser Mode 安定化とビジュアルリグレッションテスト

> トピック: Vitest 4.0（Browser Mode 安定化）｜ 出典: Vitest 公式ブログ / VoidZero / InfoQ ｜ 種別: メジャーバージョンアップ

## 一行サマリー

Vite ネイティブのテストランナー Vitest が 4.0 を公開し、実ブラウザで単体テストを走らせる Browser Mode を正式に安定化。スクリーンショット比較によるビジュアルリグレッションテスト（`toMatchScreenshot`）、`toBeInViewport` マッチャ、Playwright トレース連携を備え、jsdom 擬似環境に頼らない「本物のブラウザでのテスト」を実用段階に押し上げた。

## 🔰 初学者向け（何が・なぜ重要か）

フロントエンドの自動テストでは長らく、**本物のブラウザではなく「ブラウザのふり」をする擬似環境（jsdom など）**でコンポーネントを動かして検証していた。速くて手軽な反面、実ブラウザとの微妙な挙動差（レイアウト計算や一部 API）でテストが通るのに本番で壊れる、逆も起きる、という落とし穴があった。

**Vitest 4.0 の Browser Mode は、テストを実際の Chrome や Firefox の中で走らせる仕組みで、これが"実験的"から"安定版"になった**。擬似環境ではなく本物のブラウザで動かすので、見た目や挙動が本番に近い。

さらに **ビジュアルリグレッションテスト**が加わった。これは「基準となる画面のスクショ」を保存しておき、コード変更後に撮り直したスクショと自動比較して、意図しない見た目の崩れを検出する仕組み。ボタンの色がずれた、余白が変わった——こうした"目で見ないと気づかない不具合"を機械が拾ってくれる。写真の「間違い探し」を自動でやってくれる番人だと思えばいい。

Vitest は Vite（人気のビルドツール）と一体で速く、React/Vue/Svelte など幅広く使えるため、この安定化は多くのプロジェクトに効く。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

**Vitest 4.0（2025-12リリース）の主眼**:

- **Browser Mode 安定化**: 実験ステータスを外し、設定形（config shape）と API 面を確定。Playwright / WebdriverIO などのプロバイダ経由で実ブラウザ実行。
- **ビジュアルリグレッションテスト**: `toMatchScreenshot` アサーションで UI のスクショを撮り、基準画像と差分比較して意図しない変化を検出。
- **`toBeInViewport` マッチャ**: `IntersectionObserver` を用い、要素が現在ビューポート内にあるかを検証。
- **Playwright トレース対応**: `browser.trace`（`on` / `off` / `on-first-retry` / `on-all-retries` / `retain-on-failure`）でトレースを生成。レポータの注釈として Playwright Trace Viewer で確認可能。
- **VSCode 拡張**: ブラウザテストに Debug Test ボタン対応。
- 後続の 4.1 でさらに機能追加（詳細は「要確認」）。

### 仕組み / 背景（なぜこう設計されたか）

**なぜ実ブラウザなのか**。従来の単体テストは jsdom（Node 上で DOM を模した実装）で高速に回せるが、jsdom はレイアウトエンジンを持たず、`getBoundingClientRect` などの実測系や一部 Web API が実ブラウザと乖離する。結果、「テストは緑なのに本番で崩れる」偽陰性・偽陽性が生じる。Browser Mode は Playwright 等でヘッドレス／実ブラウザに接続し、テストコードをブラウザ内コンテキストで実行することでこのギャップを埋める。E2E（Playwright/Cypress）ほど重くなく、単体テストの粒度で"本物の描画"を得られる中間点を狙う設計だ。

**ビジュアル比較の原理**。`toMatchScreenshot` は決定的にレンダリングした要素を撮影し、ピクセル／知覚差分でベースラインと比較する。フォントやアンチエイリアスの環境差でフレーク（不安定）になりやすいため、しきい値やレンダリング固定が要る——これが安定運用の勘所。Playwright トレースは失敗時の DOM・ネットワーク・スクショを時系列で残し、CI での再現デバッグを容易にする。

**トレードオフ**。実ブラウザ実行は jsdom より遅く、CI にブラウザ実行基盤（Playwright ランタイム）を要する。全テストをブラウザ化するのではなく、レイアウト・可視性・視覚回帰が重要な部分に選択適用するのが現実的。

### 実務での使いどころ・移行の勘所

デザインシステムや UI コンポーネントライブラリで威力を発揮する——視覚回帰で意図しないスタイル崩れを CI で検出できる。既存の jsdom ベーステストはそのまま残しつつ、可視性・レイアウト・スクショ検証が要るテストだけ Browser Mode に振り分けるハイブリッド運用が無難。スクショ比較はベースライン画像の管理（環境差の固定、しきい値設定）が肝心で、CI 実行環境とローカルのレンダリング差でフレークしないよう固定化する。Jest からの移行を検討中なら、Vite 統合の速さと合わせ 4.0 は乗り換えの後押しになる。

## 📖 用語解説（このトピックとのつながり）

- **テストランナー**: テストを実行・集計するツール。Vitest はその一種で Vite と統合。
- **jsdom**: Node 上で DOM を模す擬似環境。高速だが実ブラウザと差がある。Browser Mode が置き換える対象。
- **Browser Mode**: 実ブラウザ内でテストを走らせる Vitest の機能。今回安定化。
- **ビジュアルリグレッションテスト**: 画面のスクショを基準と比較し視覚的変化を検出する手法。`toMatchScreenshot` が担う。
- **Playwright トレース**: 失敗時の操作・DOM・ネットワークを記録した再現用データ。デバッグを容易にする。
- **フレーク（flaky test）**: 実行のたびに結果が揺れる不安定なテスト。スクショ比較で起きやすく、固定化が必要。

## 影響範囲 / 推奨アクション

- UI コンポーネント／デザインシステムを持つチームは、視覚回帰テストの導入を検討。
- 全面ブラウザ化ではなく、可視性・レイアウト検証に選択適用するハイブリッドが現実的。
- スクショ比較はベースライン管理と環境固定を徹底し、フレークを抑制。CI にブラウザ実行基盤が要る点に留意。
