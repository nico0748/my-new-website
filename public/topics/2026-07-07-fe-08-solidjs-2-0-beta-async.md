# SolidJS 2.0 Beta — 非同期を「第一級」に、Suspense 再設計と決定的バッチング

> トピック: SolidJS 2.0 Beta リリース｜出典: solidjs/solid GitHub Discussion｜種別: メジャーバージョン Beta

## 一行サマリー

きめ細かいリアクティビティで知られる SolidJS のメジャー版 2.0 が Beta（npm の `next` タグ）に到達。非同期処理をフレームワークの「第一級市民」として扱えるようになり、Promise を直接シグナルの世界で扱えるほか、Suspense の再設計・ミューテーション用の新プリミティブ・決定的なバッチングが入る。仮想 DOM を持たない設計思想は維持しつつ、大きな破壊的変更を伴う。

## 🔰 初学者向け（何が・なぜ重要か）

SolidJS は React に似た書き味でありながら、「変わった値だけを、変わった場所だけ更新する」ことに徹した UI ライブラリです。React が「変化があったら画面全体を一度作り直して差分を取る（仮想 DOM）」のに対し、Solid は「この値が変わったらこの一箇所だけ書き換える配線」をあらかじめ引いておく方式で、無駄が少なく速いのが売りです。

2.0 の目玉は **非同期（データの取得待ちなど）の扱いが一段素直になったこと**です。Web アプリでは「サーバーからデータが返るのを待つ」処理が避けられませんが、これまでは待ち時間の扱いに独特の作法が要りました。2.0 では Promise（「あとで結果が来る約束」を表すオブジェクト）をそのままリアクティブな仕組みに載せられ、「読み込み中の表示（Suspense）」も作り直されて分かりやすくなります。料理でいえば、下ごしらえの完成を待つ工程が、レシピの中に自然に書けるようになった、というイメージです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **非同期の第一級化**: Promise をリアクティブグラフに直接組み込めるようになり、`async` を軸にした新しいデータフローを提供。
- **Suspense の再設計**: 読み込み境界の挙動を作り直し、待ち状態の伝播をより予測可能に。
- **ミューテーション用の新プリミティブ**: 書き込み／更新系の状態遷移を扱う API を追加。
- **決定的バッチング（deterministic batching）**: 複数の状態更新のまとまり方・反映順序を決定的にし、レースや順序依存の不具合を減らす。
- 仮想 DOM を持たないきめ細かいリアクティビティは維持。多数の破壊的変更を伴い、移行ガイドとエコシステム整備が進行中。
- Beta は 2026年3月3日に npm `next` へ、以後 Beta を反復（InfoQ が2026年5月に解説）。SolidStart 2.0 への統合と、TanStack Start 側の Solid 2.0 対応も進む。

### 仕組み / 背景（なぜこう設計されたか）

Solid の核は「シグナル（値）→ 計算 → エフェクト」を配線するプッシュ型のきめ細かいリアクティビティ。ここに非同期を後付けで載せると、「まだ解決していない値」をどう伝播し、いつ再計算し、どこで待たせるかが曖昧になり、Suspense の挙動が読みにくくなっていた。2.0 は非同期を後付けではなく**リアクティビティモデルの基盤に統合**することでこれを解く。Promise 自体を追跡可能な依存として扱えば、「待ち」の伝播はシグナル伝播の一般則に還元でき、Suspense 境界も特殊機構ではなくグラフ上の一貫した振る舞いになる。

決定的バッチングは、複数更新がマイクロタスク境界でどうまとまるかを規定し、「更新順が実行環境で揺れる」問題を排除する設計。トレードオフは、これらの中核変更が広範な破壊的変更を招き、既存の 1.x コード・ライブラリの移行コストが発生する点。仮想 DOM を捨てた高速性という強みを保ったまま、非同期という現代 UI の必須要件を後付けの継ぎ足しではなく土台から作り直した、というのが 2.0 の本質である。

### 実務での使いどころ・移行の勘所

- データ取得が多い SPA／SSR で、ローディング状態の設計を素直に書きたいケースに向く。React の use()/Suspense と同じ悩みに、仮想 DOM なしで応える選択肢。
- Beta 段階のため本番採用は慎重に。1.x からの移行は破壊的変更が多く、周辺ライブラリ（ルーター・状態管理）の 2.0 対応状況を確認する。
- SolidStart 2.0 / TanStack Start と組み合わせるフルスタック構成で真価。型安全ルーティング志向なら TanStack との相性を評価。
- React からの移行検討軸は「きめ細かい更新による性能 vs エコシステムの厚み」。2.0 で非同期体験が近づいた分、比較の焦点は性能とバンドルサイズに寄る。

## 📖 用語解説（このトピックとのつながり）

- **きめ細かいリアクティビティ（fine-grained reactivity）**: 変化した値に紐づく最小箇所だけを更新する仕組み。Solid の核で、2.0 でも維持される。
- **Suspense**: データ待ちの間フォールバック（読み込み中表示）を出す境界。2.0 で再設計された中心機能。
- **決定的バッチング**: 複数の状態更新のまとめ方・反映順を一意に定める仕組み。順序依存バグを防ぐ。
- **プリミティブ（primitive）**: `createSignal` 等、Solid が提供する最小の反応単位。2.0 でミューテーション向けが追加。

## 影響範囲 / 推奨アクション

- 既存 1.x プロジェクト: Beta を検証環境で試し、破壊的変更と依存ライブラリの対応を洗い出す。本番移行は安定版待ちが無難。
- 新規: 性能重視・非同期多めの UI なら 2.0 系を評価対象に。SolidStart 2.0 の整備状況を注視。
- ライブラリ作者: 2.0 の新リアクティビティ／Suspense 前提での動作確認を進める。
