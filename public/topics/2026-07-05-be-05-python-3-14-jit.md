# Python 3.14 の copy-and-patch JIT — 標準 CPython が機械語生成を持つ時代へ

> トピック: Python 3.14（2025年10月7日リリース）の copy-and-patch 方式 JIT（PEP 744）｜出典: Python 公式 What's New｜種別: 言語ランタイム／性能機能

## 一行サマリー

2025年10月に登場した Python 3.14 は、実行時に「熱い」バイトコードを機械語へ変換する copy-and-patch 方式の JIT コンパイラ（PEP 744）を標準 CPython に導入した。計算重めのコードで 10〜30% の高速化が報告される一方、環境変数などで明示的に有効化する実験的機能で、フリースレッドビルドとは併用できない。

## 🔰 初学者向け（何が・なぜ重要か）

Python は書きやすい反面、実行速度が遅いとよく言われます。理由のひとつは、Python が「プログラムを1行ずつ解釈しながら動かす（インタプリタ）」方式だからです。JIT（Just-In-Time コンパイル）は、この弱点を和らげる技術で、「よく通る道（ホットな部分）だけを、その場で高速な機械語に翻訳して実行」します。

たとえるなら、毎回辞書を引きながら外国語を読むのがインタプリタ、よく出てくる文だけ暗記して一瞬で読めるようにするのが JIT です。Python 3.14 は、この JIT を標準の Python 本体（CPython）に、外部ツールなしで載せた最初の世代のひとつです。

重要なのは「Python 公式が、速度改善に本腰を入れ始めた」という方向性です。まだ実験段階で、有効化しないと動かず、常に速くなるわけでもありませんが、Python が「遅い言語」という評価を書き換えにいく大きな一歩です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

Python 3.14 は2025年10月7日にリリース。PEP 744 により **copy-and-patch 方式の JIT** を導入した。PyPy のようなトレーシング JIT とは異なり、CPython の手法は個々のホットなバイトコードを、事前コンパイル済みの「テンプレート（stencil）」を使って機械語へと組み立てる。

- 効果: ループ・算術・関数呼び出しなど計算集約コードで 10〜30% の高速化報告。
- 有効化: 既定では無効に近い扱いで、`PYTHON_JIT=1` 環境変数やコマンドラインフラグで有効化する実験的機能。
- 制約: **フリースレッド（GIL 無効）ビルドでは JIT 非対応**。ワークロード依存で効果は一定しない。

同じ 3.14 では、GIL を任意で無効化するフリースレッドが PEP 779 で正式サポート段階に進んだ（単一スレッド性能のオーバーヘッドは 3.13 の約40%から 5〜10% に改善）。

### 仕組み / 背景（なぜこう設計されたか）

一般的な JIT には「トレーシング型（実行経路を追って最適化した機械語を生成）」と「メソッド型」があるが、いずれも本格的なコンパイラをランタイムに組み込む必要があり、実装・保守・移植のコストが大きい。copy-and-patch は、コンパイル時に各バイトコード操作の機械語テンプレート（stencil）を用意しておき、実行時はそれらを連結して定数だけ「patch（穴埋め）」する方式。フルコンパイラを持たずに機械語生成の恩恵を得られるため、CPython のような移植性重視のプロジェクトに適する。

トレードオフは、生成コードの最適化度合いが本格 JIT より控えめで、効果がワークロードに強く依存する点。さらに JIT とフリースレッドが排他なのは、両者がインタプリタ内部（参照カウントやメモリモデル）に異なる前提を置くためで、当面は「多コアを活かすフリースレッド」か「単一スレッド高速化の JIT」かを選ぶことになる。この二本立ては、Python の性能改善が「単一の銀の弾丸」ではなく、複数の直交する最適化（Faster CPython プロジェクト）の積み上げで進むことを示している。

### 実務での使いどころ・移行の勘所

数値計算・データ変換・アルゴリズム的なループが多い CPU バウンドのコードで JIT を試す価値がある。逆に I/O バウンドや外部ライブラリ（NumPy 等、既に C 実装）が支配的な処理では効果は薄い。実験的機能ゆえ、本番投入前に自分のワークロードでベンチマークし、回帰がないか確認するのが必須。多コア並列が主目的ならフリースレッドビルドを選び、その場合は JIT を諦める、という切り分けになる。3.14 全体としては、性能改善がオプトインの実験機能として段階提供される移行期であり、まずは非クリティカルなバッチ処理などで計測から始めるのが安全。

## 📖 用語解説（このトピックとのつながり）

- **JIT（Just-In-Time コンパイル）**: 実行時にホットなコードを機械語へ変換する高速化技術。本トピックの主役。
- **copy-and-patch**: 事前用意した機械語テンプレートを連結・穴埋めして機械語を生成する方式。CPython が採用。
- **CPython**: 標準の Python 実装。3.14 で JIT を内蔵。
- **フリースレッド（GIL 無効）**: 多コアを活かすためのビルド。3.14 で正式サポートだが JIT とは排他。
- **PEP 744 / PEP 779**: それぞれ JIT とフリースレッド正式化を定めた Python 拡張提案。

## 影響範囲 / 推奨アクション

CPU バウンドで計算集約的な Python コードは JIT で改善の余地あり。実験的機能のため本番前に自ワークロードでベンチマーク必須。多コア並列が主目的ならフリースレッドを選び JIT とは切り分ける。
