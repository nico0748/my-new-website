# DuckDB 1.5.0 — 組み込み分析DBの新世代、暗号化とMERGEのLTS路線

> トピック: DuckDB 1.5.0 リリース（1.4 LTS の後続）｜ 出典: DuckDB 公式ブログ ｜ 種別: マイナーバージョンアップ

## 一行サマリー

「SQLite の分析版」とも呼ばれる組み込み型の列指向 DB、DuckDB が 1.5.0 を公開。1.4 LTS（Andium）で入った AES-256 のデータベース暗号化・`MERGE` 文・Iceberg 書き込み・CTE のデフォルト実体化を土台に、さらなる機能改善を重ねた。サーバー不要で手元のデータをSQLで高速集計できる強みはそのまま。

## 🔰 初学者向け（何が・なぜ重要か）

DuckDB は「インストール不要・サーバー不要で、CSV や Parquet のような大きめのデータファイルを SQL でサクサク分析できる」データベースだ。よく「分析版の SQLite」と説明される。SQLite が 1 ファイルの軽量 DB としてアプリに組み込まれるのと同じく、DuckDB もアプリや Python スクリプトの中に埋め込んで動く。違いは中身の設計で、DuckDB は**列指向**——「行を1件ずつ」ではなく「列（カラム）ごと」にデータを並べるため、「売上の合計は？」のような集計が桁違いに速い。

今回の 1.5.0 と、その基礎になった 1.4 LTS では、実務で待たれていた機能が入った。ひとつは **データベースの暗号化**。DuckDB のファイルそのものを AES 方式で暗号化できるようになり、機微データを手元ファイルに置く用途で安心感が増した。もうひとつが **`MERGE` 文**。これは「あれば更新、なければ追加（UPSERT）」をまとめて書ける SQL で、日次で入ってくるデータの取り込みが素直に書ける。DuckDB は「小さくて速い分析エンジン」として、ローカル分析やデータパイプラインの部品として広く使われている。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

DuckDB のリリース系譜（一次情報で確認できる範囲）:

- **1.4.0 LTS「Andium」（2025-09）**: 長期サポート版。主要新機能に、
  - **データベース暗号化**: ファイルを AES-256（既定 GCM モード）で暗号化。長年の要望に対応。
  - **`MERGE` 文**: UPSERT 相当をまとめて記述。
  - **Iceberg 書き込み**: Apache Iceberg テーブルへの書き出し対応。
  - **CTE のデフォルト実体化**: 共通テーブル式（WITH 句）をインライン展開せず実体化する既定に変更。性能向上と一部の正しさ問題を解消。
  - **CLI プログレスバー（ETA 付き）**、**FILL ウィンドウ関数**、**Teradata コネクタ**。
- **1.5.0（2026-03-09）**: 1.4 LTS の後続として機能・性能を追加（具体項目は公式リリースノートで要確認）。
- LTS 系（1.4「Andium」）はパッチ（1.4.3〜1.4.5 など）で保守され、EOL は 2026年9月予定。

### 仕組み / 背景（なぜこう設計されたか）

**列指向 × ベクトル化実行**。DuckDB は列ごとにデータを格納し、演算を「一度に大量の値をまとめて処理する（ベクトル化）」方式で行う。集計・スキャン主体の分析（OLAP）では、必要な列だけ読み、CPU キャッシュとSIMDを活かせるため、行指向の汎用 DB より圧倒的に速い。反面、1行の頻繁な更新（OLTP）には向かない——用途を分析に絞ることでこの速度を得るトレードオフだ。

**なぜ CTE 実体化を既定にしたか**。従来は WITH 句を参照箇所へインライン展開していたが、同じ CTE を複数回参照すると計算が重複し、非決定的な副作用や性能劣化を招くことがあった。実体化（一度計算して結果を保持）を既定化することで、計算重複の回避と結果の一貫性を得る。代償はメモリ使用の増加だが、多くの分析クエリでは総合的に有利。

**暗号化と Iceberg の意味**。ローカルファイル DB は「盗まれたら中身が見える」リスクがある。ファイル暗号化はこれを塞ぎ、規制業界やノートPC上の機微データ分析に道を開く。Iceberg 書き込みは、レイクハウス（オブジェクトストレージ上のテーブル形式）への出力を可能にし、DuckDB を「手元で加工してレイクへ書き戻す」パイプラインの一部として組み込める。

### 実務での使いどころ・移行の勘所

ローカルでの探索的データ分析、ETL の中間処理、Parquet/CSV への SQL クエリ、ノートブックやアプリ組み込みの分析エンジンとして最適。長期運用や社内標準化を狙うなら、機能追加が速い最新（1.5系）より **1.4 LTS** を基盤に据える選択が堅実——LTS は1年のコミュニティサポートがあり、パッチで安定運用できる。暗号化を使う場合は鍵管理の運用設計が別途必要。大量の逐次更新（OLTP）用途には不向きなので、その場合は行指向 DB と役割分担する。

## 📖 用語解説（このトピックとのつながり）

- **組み込み型データベース**: アプリやスクリプトに埋め込んで動く、サーバー不要の DB。DuckDB / SQLite が代表。
- **列指向（columnar）**: データを列ごとに格納する方式。集計・分析（OLAP）に強い。DuckDB の中核。
- **OLAP / OLTP**: 前者は集計・分析主体、後者は少量レコードの頻繁な更新主体。DuckDB は OLAP 特化。
- **MERGE / UPSERT**: 「あれば更新、なければ挿入」をまとめて行う SQL。日次取り込みに有用。
- **CTE（共通テーブル式）**: WITH 句で定義する一時的な名前付きサブクエリ。1.4 で実体化が既定に。
- **Apache Iceberg**: オブジェクトストレージ上のテーブル形式（レイクハウス）。DuckDB が書き込み対応。
- **LTS（長期サポート）**: 長期間の保守が約束された版。1.4「Andium」が該当。

## 影響範囲 / 推奨アクション

- ローカル分析・ETL・組み込み分析エンジン用途の DuckDB 利用者は、新機能取り込みを検討。
- 長期運用・社内標準化は 1.4 LTS を基盤にすると安定（EOL 2026年9月予定、要確認）。
- 機微データを扱うなら暗号化＋鍵管理の運用を設計。OLTP 用途には別 DB と役割分担。
