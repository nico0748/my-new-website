# nginx-ui の MCP 認証バイパス CVE-2026-33032「MCPwn」（実環境悪用・サーバ完全乗っ取り）

> トピック: 脆弱性ニュース ｜ 出典: The Hacker News / Infosecurity Magazine / eSentire ｜ 種別: 認証バイパス（CWE-306 認証の欠如）

## 一行サマリー

nginx の Web 管理ツール「nginx-ui」(≤2.3.5) の MCP エンドポイント `/mcp_message` が認証を要求しないため、ネットワーク到達可能な攻撃者が認証なしで Nginx 設定の書き換え・リロードまで実行でき、サーバを完全乗っ取りできる。CVSS 9.8、2026年3月以降 実環境で悪用が観測されている。

## 🔰 初学者向け（何が・なぜ問題か）

「nginx-ui」は、Web サーバ Nginx をブラウザの画面からポチポチ操作するための管理ツールです。本来この管理画面は、ログイン（パスワードや鍵による本人確認）を通った人だけが触れるべきものです。

ところが nginx-ui には、AI エージェントなどから機械的に操作するための新しい「窓口」（MCP = Model Context Protocol のエンドポイント）が増設されていました。問題は、玄関（通常のログイン画面）には鍵がかかっていたのに、**この裏口の窓口には鍵がかかっていなかった**ことです。

つまり、インターネットからこの裏口を見つけた攻撃者は、ログインを一切せずに「Nginx の設定ファイルを書き換えろ」「設定を再読み込みしろ」といった命令を直接投げ込めてしまいます。Nginx はサイトの入口を司る門番なので、その設定を握られると、偽サイトへの誘導、内部システムへの中継、不正なプログラムの実行など、サーバまるごとの乗っ取りに直結します。インターネット上に約2,689台の露出インスタンスが確認され、すでに悪用が始まっているため危険度は高い状態です。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **CVE**: CVE-2026-33032（コードネーム「MCPwn」, 命名は Pluto Security）
- **CVSS**: 9.8（Critical）
- **種別**: 認証の欠如 / 認証バイパス（CWE-306 Missing Authentication for Critical Function）
- **影響範囲**: nginx-ui バージョン 2.3.5 以前。MCP 連携機能が有効な構成。
- **前提条件**: 攻撃者が `/mcp_message` エンドポイントへネットワーク到達できること。**認証情報・利用者操作は不要**（unauthenticated・no user interaction）。
- **修正版**: ベンダ案内に不整合があり「2.3.4 へ更新」との記述と「2.3.5 が影響」との記述が混在。**最新の固定版番号は要確認**（公式リポジトリのリリースノートで確認のうえ MCP 無効化と併用が安全）。
- **悪用状況**: 実環境で悪用。観測は2026年3月以降、Recorded Future Insikt Group がリスクスコア94/100の高影響脆弱性として警告。

### メカニズム（攻撃が成立する仕組み）

本件は「認証境界の設計欠陥」に分類される。攻撃が成立する流れを概念レベルで分解する。

1. **信頼境界の置き場所のミス**: nginx-ui は対話 UI / API には認証ミドルウェアを通していたが、後付けの MCP 統合エンドポイント `/mcp_message` は同じ認証ゲートの内側に配置されなかった。結果、本来「認証済み」であるべき特権操作が、認証チェックを通らない経路から到達可能になった。
2. **攻撃者が握るプリミティブ**: このエンドポイントは nginx-ui が公開する 12 個の MCP ツールをすべて呼び出せる。中には「Nginx 設定ファイルの読み取り・書き換え」「設定の自動リロード」といった、サーバ挙動を直接左右する機能が含まれる。すなわち攻撃者は **任意設定書き込み + 反映トリガ** という強力なプリミティブを認証なしで得る。
3. **被害への橋渡し**: 設定書き換えプリミティブは、`proxy_pass` の差し替えによる内部サービスへの中継（SSRF/ピボット）、`root`/`alias` 改変によるファイル開示、`lua`/外部コマンド実行系ディレクティブを伴う構成での任意コード実行などに昇格しうる。門番たる Nginx の構成支配は、そのままホスト・背後ネットワークの掌握へ繋がる。

一般原理として、本件は「機能を後から追加した際に、新経路を既存の認証・認可境界の内側に確実に組み込まなかった」典型的なアクセス制御欠落（broken access control / missing authentication）である。とくに **自動化用 API（MCP/エージェント連携）は人間用 UI と別系統で実装されがちで、認証の網から漏れやすい**点が今回の教訓。武器化の具体的手順・ペイロードは記載しない（概念説明に留める）。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: アクセスログ上の `/mcp_message`（および MCP 関連パス）への未認証 POST、短時間での設定リロード多発、`nginx.conf` 系ファイルの想定外更新時刻、見覚えのない `proxy_pass`/`server` ブロックの追加。設定ファイルのハッシュ監視（FIM）で改竄を検知。
- **暫定緩和（パッチ前）**: MCP 機能を無効化する。管理ポート/エンドポイントをファイアウォール・リバースプロキシの ACL で社内・VPN 経由に限定し、インターネット露出をなくす。
- **恒久対策**: 公式の固定版へ更新（**正確な固定版番号は要確認**）。更新後も管理面はネットワーク分離を維持。
- **運用示唆**: 「自動化用エンドポイントが既存認証の内側にあるか」を構成変更時のレビュー項目にする。Shodan 等で自組織の nginx-ui 露出を棚卸し。

## 📖 用語解説（このニュースとのつながり）

- **MCP（Model Context Protocol）**: AI エージェントなどがツール/サービスを機械的に呼び出すためのプロトコル。本件では nginx-ui がこの窓口を追加した結果、その窓口に認証がかかっておらず侵入口になった。
- **認証バイパス / 認証の欠如（CWE-306）**: 本来ログイン等で守られるべき機能が、認証チェックを経ずに使えてしまう欠陥。本件の根本原因。
- **特権操作プリミティブ**: 攻撃者が手にする「できること」の最小単位。本件では「Nginx 設定の任意書き換え＋リロード」がそれにあたり、これが乗っ取りの起点になる。
- **リバースプロキシ（Nginx の役割）**: 外部からの通信を受けて内部のサービスへ振り分ける門番。設定を握られると通信経路ごと操作されるため、本件の被害が大きい理由。

## 影響範囲 / 推奨対応

インターネット公開された nginx-ui（≤2.3.5、MCP 有効）が最も危険。中国・米国・インドネシア・ドイツ・香港に露出が多いが、国内でも自前運用の Nginx 管理に nginx-ui を使う環境は対象になりうる。直ちに (1) MCP 機能の無効化、(2) 管理エンドポイントのネットワーク制限、(3) 公式固定版への更新（版番号は要確認）、(4) 既存改竄（不正な設定・管理者・Webシェル）の痕跡調査を実施する。
