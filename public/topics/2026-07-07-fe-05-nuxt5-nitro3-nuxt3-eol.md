# Nuxt 5 と Nitro v3 が目前 — Tasks API・TypeScript 厳格化、そして Nuxt 3 は 7/31 に EOL

> トピック: Nuxt 5 / Nitro v3 の準備状況と Nuxt 3 の EOL（2026-07-31）｜出典: Nuxt 公式ロードマップ・ブログ｜種別: ロードマップ/EOL 告知（状況整理）

## 一行サマリー

Vue 系メタフレームワーク Nuxt の次期メジャー v5 は、サーバーエンジン Nitro v3（Tasks API 搭載）の安定化を待って公開される段階にあり、一方で Nuxt 3 は 2026年7月31日にサポート終了（EOL）を迎えるため、v3 利用者は今月中の v4 移行判断が現実的な締切になっている。

## 🔰 初学者向け（何が・なぜ重要か）

Nuxt は Vue でウェブアプリを作るときの「全部入りの骨組み」（メタフレームワーク）です。React でいう Next.js の Vue 版と考えると分かりやすいです。その心臓部には Nitro というサーバーエンジンがあり、どのクラウドにも同じコードを配備できる役割を担っています。

いま起きていることは2つです。1つめは次のメジャー版 Nuxt 5 の準備が最終段階に入っていること。目玉の一つが Nitro v3 の「Tasks API」で、これは「毎晩0時にデータを集計する」「要求に応じて重い処理を裏で走らせる」といった裏方仕事（バッチ処理）を、フレームワーク公式の仕組みとして書けるようにするものです。これまでは cron 設定や別サービスを自前で用意する必要がありました。

2つめは締切の話です。ひとつ前の世代の Nuxt 3 は 2026年7月31日でサポートが終わります。サポート終了後はセキュリティ修正が出なくなるため、「まだ Nuxt 3」のプロジェクトはこの夏が移行のリミットです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **Nuxt 5**: Nitro v3 の安定化後にリリースされる方針（公式ロードマップ）。当初 2025 Q4 目標だったが後ろ倒しになっており、正式リリース日は要確認。予告されている方向性は Nitro 3 統合、より厳格な TypeScript 境界（server/client 間の型分離）、SSR 最適化、アクセシビリティの組み込み強化など。
- **Nitro v3**: 最大の追加が **Tasks API**（オンデマンド/スケジュール実行のサーバータスク定義）。
- **Nuxt 3 EOL**: 2026-07-31 でメンテナンス終了。Nuxt 4 は「Nuxt 5 リリース後も最低6か月」サポートが明言されている。
- 現行安定版は Nuxt 4.3 系。v4 は v3 からの段階的移行（互換レイヤーあり）が可能。

### 仕組み / 背景（なぜこう設計されたか）

Nitro は「デプロイ先（Node、エッジ、サーバーレス各社）を抽象化するサーバーツールキット」として設計されており、Nuxt 本体から独立して汎用化してきた。Tasks API がここに入る意味は大きい。従来、SSR フレームワークは「HTTP リクエスト応答」だけを責務とし、バッチや遅延ジョブは外部（cron、キュー、Cloudflare Queues や Vercel Cron などベンダー機能）に押し出されてきた。その結果、同じアプリのロジックがデプロイ先ごとに別の書き方になる「ロックインの隙間」が生じていた。Tasks API はこの裏方仕事もプロバイダ非依存の抽象に取り込む試みで、Rails が Active Job で、Django 6 が組み込みタスクで進めた「フレームワークがジョブ実行まで面倒を見る」潮流の JS 版といえる。

トレードオフ: 抽象化はエッジ環境の制約（実行時間上限、常駐プロセス不可）に対して漏れやすく、どのプロバイダでどこまで動くかの対応表が実用上の鍵になる（詳細は Nitro v3 安定版のドキュメントで要確認）。

TypeScript 境界の厳格化は、server ディレクトリと app ディレクトリで別々の tsconfig/型コンテキストを持たせ、サーバー専用コードのクライアント混入をコンパイル時に防ぐ方向。Next.js の `server-only` パッケージが実行時・規約ベースなのに対し、型システムで静的に線を引くアプローチ。

### 実務での使いどころ・移行の勘所

- **Nuxt 3 プロジェクトは今月が判断点**。v4 への移行は codemod と互換フラグで比較的漸進的に行える。EOL 後の塩漬け運用はフロントエンド（=攻撃面がブラウザに露出する層）では特にリスクが高い。
- Nuxt 5 を待って v3→v5 を一気に跳ぶ戦略は、EOL が先に来るため非推奨。v4 を経由するのが公式想定ルート。
- cron やジョブランナーを自前運用している Nuxt サーバー構成は、Nitro v3 Tasks への集約を設計候補に入れておく（ベンダー機能への直接依存を減らせる）。

## 📖 用語解説（このトピックとのつながり）

- **メタフレームワーク**: ルーティング・SSR・ビルドを統合した上位フレームワーク。Nuxt は Vue 版の代表格。
- **Nitro**: Nuxt のサーバーエンジン。デプロイ先を抽象化し、v3 で Tasks API を得る。
- **EOL（End of Life）**: セキュリティ修正を含む全サポートの終了。Nuxt 3 は 2026-07-31。
- **Tasks API**: オンデマンド/定期のサーバータスクを定義する Nitro v3 の新機構。外部 cron の内製化に相当。
- **codemod**: 移行用の自動コード書き換えツール。v3→v4 移行の作業量を下げる。

## 影響範囲 / 推奨アクション

- Nuxt 3 利用プロジェクト: 7/31 までに v4 移行計画を確定（少なくとも依存モジュールの v4 対応状況の棚卸しを今週実施）。
- Nuxt 4 利用者: v5 は「Nitro 3 安定化待ち」のため慌てる必要なし。リリース後も6か月の v4 サポート猶予がある。
