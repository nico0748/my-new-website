# 「2026 は Redis 不要論」：Postgres 18 をキュー/キャッシュに使う議論

> トピック: 開発トピック｜出典: Medium｜種別: backend

## 一行サマリー
Postgres 18 の I/O 高速化や機能強化を背景に「多くの現場では Redis を別途持たず、Postgres だけでキュー・キャッシュを賄えるのでは」という議論が広がっている。運用簡素化の観点で注目される。

## 🔰 初学者向け（何が・なぜ重要か）
多くの Web サービスは、メインのデータベース（多くは Postgres）に加えて、Redis という「超高速の一時記憶」を併用します。Redis は「よく使うデータの手元メモ」や「順番待ちの行列（キュー）」を高速にさばくために使われてきました。

この「Redis 不要論」は、「Postgres 18 が十分速くなったので、規模がそれほど大きくない現場なら、わざわざ Redis を別に用意して 2 つのシステムを運用しなくても、Postgres 1 つで足りるのでは」という主張です。道具箱を 2 つ持ち歩くより、1 つで済めば管理も学習も楽になります。ただしこれは「万人向けの結論」ではなく、規模や要件によっては依然 Redis が正解です。過度な単純化に注意しながら、自分の規模で本当に必要かを問い直すきっかけとして重要なニュースです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
主張の核は、Postgres 18 の I/O 改善・`SKIP LOCKED` を用いたジョブキュー、`UNLOGGED` テーブルや `LISTEN/NOTIFY`、マテリアライズドビュー等の既存機能で、Redis が担ってきたキュー・キャッシュ・Pub/Sub の一部を代替できる、という点。新機能というより「既存機能＋性能向上で単一 DB 化しやすくなった」という設計論。

### 仕組み / 背景（なぜこう設計されたか）
コンポーネントを増やすほど運用コスト・障害点・一貫性管理（DB と Redis の二重書き込み問題）が増える。単一データストアに寄せれば、トランザクション境界内でキューとデータを同時に更新でき、整合性が保ちやすい。Postgres の性能向上が、この「単純化のトレードオフ」を成立しやすくした。

### 実務での使いどころ・移行の勘所
中小規模・秒間数百〜数千程度のキュー／キャッシュ要件なら Postgres 単一化は有力。一方、超高スループットのレート制限、揮発キャッシュ、ミリ秒未満のレイテンシ、専用データ構造（ソート済みセット等）が要る場合は Redis/Valkey が依然優位。「不要論」を鵜呑みにせず、負荷特性と運用体制で判断すること。

## 📖 用語解説（このトピックとのつながり）
- **SKIP LOCKED**：行ロックを飛ばして次を取る句。Postgres でのジョブキュー実装の要。
- **LISTEN/NOTIFY**：Postgres 組み込みの簡易 Pub/Sub。Redis 代替候補。
- **UNLOGGED テーブル**：WAL を書かない高速テーブル。キャッシュ用途に検討される。
- **二重書き込み問題**：DB と Redis の整合性課題。単一化で回避できる論点。

## 影響範囲 / 推奨アクション
アーキテクチャ選定者・小〜中規模サービス開発者に影響。新規や小規模では Postgres 単一化を検討し運用を簡素化。高スループット・低レイテンシ要件では Redis/Valkey 継続を前提に、負荷試験で見極めること。
