# Apache Kafka 4.2 で Queues（Share Groups）が本番対応 — Kafka がジョブキューにも

> トピック: 開発トピック｜出典: Apache Kafka 公式ブログ｜種別: backend

## 一行サマリー
Apache Kafka 4.2（2026-02）で、キュー的な消費を可能にする「Kafka Queues（Share Groups）」が本番対応（production-ready）になった。従来はパーティション数に縛られていた並列消費の制約が外れ、Kafka を従来型メッセージキュー（RabbitMQ 等）の用途にも使いやすくなる。4.0 での ZooKeeper 廃止（KRaft 化）に続く大きな機能拡張。

## 🔰 初学者向け（何が・なぜ重要か）
Kafka は「イベントの記録帳」として広く使われる仕組みです。注文・クリック・センサー値などの出来事を順番に書き溜め、複数のサービスが好きなタイミングで読みに来ます。ただ Kafka には「パーティション」という区画があり、「1区画は1人の消費者しか同時に読めない」という制約がありました。つまり、10人で分担して処理したければ、区画を10個以上用意しておく必要があったのです。区画数は後から気軽に増やしにくく、これが並列処理の足かせでした。

Kafka Queues（Share Groups）は、この制約をゆるめ、「1つの区画を複数の作業員で分け合って処理する」ことを可能にします。まさに従来の「ジョブキュー（仕事の行列を皆で捌く）」の使い方です。1件ごとに「これは私が処理した」と確認応答でき、失敗したものは別の作業員に回せます。

これにより、Kafka を「イベント記録帳」としてだけでなく「タスクを皆で分担する行列」としても使えるようになりました。別途 RabbitMQ のようなキュー専用ツールを立てなくて済む場面が増える、という意味で実務的なインパクトがあります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Kafka 4.2（2026-02）で Share Groups（Kafka Queues）が production-ready に到達。従来の consumer group は「パーティション単位で消費者に排他割り当て」だったため、消費並列度はパーティション数が上限だった。Share Groups は同一パーティションを複数コンシューマで共有消費でき、レコード単位の acknowledgement（ACCEPT / RELEASE / REJECT 等）と、処理延長のための RENEW 応答型を備える。前提として Kafka 4.0（2025-03）で ZooKeeper を完全廃止し KRaft モードが既定化、KIP-848 の新コンシューマグループプロトコルでリバランス性能が改善済み。4.3.1（2026-06）は Kafka Streams の RocksDB ネイティブメモリリーク修正などを含むバグfix。各バージョンの厳密な機能状態は公式リリースノートで要確認。

### 仕組み / 背景（なぜこう設計されたか）
Kafka のパーティション=並列単位という設計は、順序保証とスケールを両立する強力なモデルだが、「順序は要らないから件数を皆で捌きたい」ジョブキュー用途とは相性が悪かった。Share Groups は、パーティションのログ構造（追記型・オフセット管理）を保ちつつ、消費側にレコード単位の状態（未処理・取得中・完了・拒否）を持たせることで、キュー的セマンティクスを後付けする。これにより「イベントストリーミング（順序・再生）」と「ワークキュー（分担・個別 ACK）」の両モデルを1つの基盤で扱える。トレードオフは、レコード単位の状態管理が増える分の複雑性とオーバーヘッド、そして厳密な順序保証はキュー用途では犠牲になる点。KRaft 化はメタデータ管理を Kafka 自身の Raft に統合し、ZooKeeper という別クラスタの運用負担を消した。

### 実務での使いどころ・移行の勘所
既に Kafka を運用していて、別途 RabbitMQ/SQS 的なジョブキューも抱えている組織は、Share Groups で統合できる可能性がある。順序が不要でスループット重視のタスク分配（通知送信、画像処理ジョブ等）が候補。移行の勘所は、(1) 4.x へのアップグレードでは KRaft 前提とクライアント/ブローカの Java 要件（Clients は Java 11、Brokers 等は Java 17）を満たすこと、(2) 順序保証が要る処理は従来の consumer group を維持し、キュー用途だけ Share Groups にすること、(3) 4.3.1 のような修正パッチを適用して Streams のメモリ問題等を回避すること。

## 📖 用語解説（このトピックとのつながり）
- **パーティション**: トピックを分割した並列単位。従来は消費並列度の上限だった。
- **consumer group / Share Group**: 排他割り当ての従来方式と、パーティションを共有消費する新方式（キュー的）。
- **acknowledgement（ACK）**: レコード処理結果の応答。Share Groups はレコード単位で ACCEPT/RELEASE/REJECT/RENEW を扱う。
- **KRaft**: ZooKeeper を置き換える Kafka 内蔵の Raft ベースのメタデータ管理。4.0 で既定化。
- **KIP-848**: リバランス性能を改善する新コンシューマグループプロトコル。ダウンタイム短縮に寄与。

## 影響範囲 / 推奨アクション
Kafka を運用する組織、特にジョブキューを併用している構成に関係。順序不要のタスク分配を Share Groups へ集約できるか検討。4.x 移行では KRaft と Java バージョン要件を確認し、順序保証が必要な処理は従来方式を維持。修正パッチ（4.3.1 等）の適用も忘れずに。
