# Podman 6.0リリース — cgroups v1/CNI/slirp4netns/BoltDBを一掃した破壊的メジャー

> トピック: Podman 6.0｜出典: Podman GitHub Releases / Linuxiac / Fedora Project Wiki｜種別: メジャーリリース（破壊的変更）

## 一行サマリー

Docker 互換のデーモンレスコンテナエンジン Podman 6.0 が 2026-06-24 にリリースされ、cgroups v1・iptables・CNI・slirp4netns・BoltDB という5つのレガシー層を削除。Netavark/Pasta/SQLite への一本化と AMD GPU サポート、Docker API 互換の改善が入った。アップグレード前に監査が必須の破壊的メジャー。

## 🔰 初学者向け（何が・なぜ重要か）

Podman は Docker の代わりに使えるコンテナ実行ツールです。最大の違いは「常駐する管理プロセス（デーモン）を持たない」こと。Docker は root 権限で動くデーモンが中心にいますが、Podman は各コマンドが直接コンテナを起動するので、権限まわりが安全でシンプルという特徴があります。

今回の 6.0 は、長年の互換維持のために抱えていた「古い部品」を一気に捨てた大掃除リリースです。例えるなら、家の配管や配線を新方式にまるごと入れ替える大規模リフォーム。新しい家（新環境）では快適ですが、古い設備前提で動いていたものは動かなくなります。だからこそ「アップグレード前に何が壊れるか点検（監査）してから移行する」ことが強く求められます。

削除された「cgroups v1」は Linux がコンテナのリソース（CPU やメモリ）を制限する古い仕組み、「CNI」「slirp4netns」はネットワークの古い方式、「BoltDB」は内部データの保存方式です。いずれも後継（cgroups v2 / Netavark / Pasta / SQLite）があり、Podman はそちらに完全移行しました。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **リリース日**: 2026-06-24。
- **削除された5つのレガシー層**:
  - **cgroups v1**: 非サポート化。cgroups v2 が必須。
  - **iptables**: ネットワークバックエンドから除去。
  - **CNI**: 削除。ネットワークは **Netavark** に一本化。
  - **slirp4netns**: 削除。rootless ネットワークは **Pasta** に置き換え。
  - **BoltDB**: 完全削除。DB バックエンドは **SQLite のみ**。
- **新機能**: **AMD GPU サポート**、Docker API 互換の改善、実験的な **Pesto** rootless ポートフォワーディング。`--net` に `ip=` を複数回渡すことで、コンテナへ**複数の静的 IP** を割り当て可能に。
- **安全な移行経路**: いったん **Podman 5.8 に上げて再起動** → BoltDB→SQLite の自動マイグレーションを走らせる → その後 **6.0 へ**。

### 仕組み / 背景（なぜこう設計されたか）

破壊的削除は、後継技術がすでに実用段階に達し「二重メンテナンスのコストが便益を上回った」ことの表明です。

- **cgroups v2 への一本化**: v2 は統一階層（unified hierarchy）でリソース制御を整理し、rootless コンテナや圧力ストール情報（PSI）など v1 にない機能を提供します。主要ディストロは既定を v2 に移しており、v1 を残す意味が薄れました。
- **Netavark + Aardvark-dns**: CNI プラグイン群に代わる Rust 製ネットワークスタック。Podman 専用に設計され、設定と DNS 解決を統合。CNI の汎用性を捨てる代わりに、Podman ワークロードでの一貫性と保守性を得ています。
- **Pasta（passt）**: slirp4netns に代わる rootless ネットワーク。TCP/UDP をユーザー空間で扱いつつ、より高いスループットと素直な IP 割り当てを提供。
- **SQLite 一本化**: BoltDB（Go 製 key-value）を廃し、SQLite に統一。トランザクション・並行性・可搬性で優れ、状態管理を単純化。

これらは「汎用プラグイン方式で何でも差せる」柔軟性より、「Podman が最適に動く決め打ちスタック」を選ぶ方向転換です。トレードオフとして、CNI 前提の既存構成や v1 依存の古いホストは切り捨てられます。

### 実務での使いどころ・移行の勘所

- **監査が最優先**: cgroups v1 のみのホスト、CNI 設定ファイル、slirp4netns 前提のスクリプト、BoltDB を直接触る運用がないか棚卸し。
- **段階的アップグレード**: いきなり 6.0 に飛ばず、5.8 → 再起動 → 自動 DB マイグレーション → 6.0 の順を守る。飛び越えるとデータ移行に失敗しうる。
- **CI/CD・K8s ノード**: ホスト OS が cgroups v2 を有効化しているか確認。古いエンタープライズ Linux は要注意。
- **AMD GPU** を使う ML/推論ワークロードでは、NVIDIA 一辺倒だった GPU コンテナ運用の選択肢が広がる。
- Docker からの移行検討組にとって、API 互換改善は `docker-compose` 系ツールとの相性向上を意味する（ただし実挙動は要検証）。

## 📖 用語解説（このトピックとのつながり）

- **cgroups v2**: Linux のリソース制御機構の第2世代。6.0 で必須化。
- **Netavark**: CNI に代わる Podman 専用の Rust 製ネットワークバックエンド。
- **Pasta（passt）**: slirp4netns に代わる rootless ユーザー空間ネットワーク。
- **BoltDB → SQLite**: 内部状態の保存先。6.0 で SQLite に一本化。
- **rootless コンテナ**: root 権限なしでコンテナを動かす方式。Podman の看板機能で、ネットワーク層の刷新はここに直結。
- **Pesto**: 6.0 で実験導入された rootless ポートフォワーディング。

## 影響範囲 / 推奨アクション

- 本番へ入れる前に **cgroups v1 / CNI / slirp4netns / BoltDB 依存の監査**を実施。
- アップグレードは **5.8 経由**の段階移行を徹底し、DB 自動マイグレーションを確認。
- ホスト OS の cgroups v2 有効化と、ネットワーク（Netavark/Pasta）の動作確認を CI で先行検証。
- AMD GPU ワークロードは新サポートの活用可否を評価。
- 既存 CNI 構成に強く依存する環境は、移行計画または旧系列の据え置きを判断。
