# OpenAI GPT-5.6 ファミリー（Sol/Terra/Luna）が限定プレビュー

> トピック: GPT-5.6 / Codex｜出典: OpenAI Developers（Codex Changelog）/ 各種報道｜種別: フロンティアモデル（ゲート付きプレビュー）

## 一行サマリー

OpenAI は2026年6月26日に GPT-5.6 ファミリー（Sol/Terra/Luna 階層）を発表し、Codex への組み込みを開始。Codex 特化の Sol はセキュリティ評価で高スコアを出し、Preparedness Framework の「High」リスク区分を越えたとされる。あわせて最も強力な agentic コーディングモデル GPT-5.3-Codex も登場。GA前のゲート付きプレビュー段階。

## 🔰 初学者向け（何が・なぜ重要か）

OpenAI の GPT シリーズに、新世代 GPT-5.6 が加わりつつあります。今回は用途別に Sol / Terra / Luna という階層（ティア）で提供され、なかでも Sol は「コードを書く AI エージェント」Codex に組み込まれる、プログラミング特化の位置づけです。AI コーディングエージェントとは、指示を受けて自分でファイルを編集し、コマンドを実行し、テストまで回す“自律的な開発アシスタント”のこと。その頭脳が新モデルで強化される、という話です。

ここで注目すべきは「安全性の扱い」です。GPT-5.6 の Codex 版 Sol は、セキュリティ関連の内部評価で高い成績を出し、OpenAI 自身の安全基準（Preparedness Framework）で「High（高リスク）」区分に達したと報じられています。能力が上がるほど悪用リスクも上がるため、OpenAI は誰でも即使えるようにはせず、「ゲート付きプレビュー（審査を通した相手に段階公開）」という慎重な出し方をしています。

初学者向けに言えば、「AI がコードを書く能力は着実に上がり続けているが、強くなるほど公開は慎重になっている」という、いまの AI 開発の空気をよく表すニュースです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **GPT-5.6 ファミリー発表（2026-06-26）**: Sol / Terra / Luna のティア構成。Codex へのロールインを開始。
- **GPT-5.6 Sol（Codex 特化）**: OpenAI 内部の Capture the Flag（CTF、セキュリティ攻防）評価で 96.7% を記録し、Preparedness Framework の「High」リスク区分を越えたとされる。
- **GPT-5.3-Codex**: 複雑・実世界のソフトウェア開発向けに「これまでで最も高性能な agentic コーディングモデル」として提供。
- **提供形態**: GPT-5.6 は GA 前のゲート付きプレビュー（一般提供ではない）。
- 数値・区分は報道・公式チェンジログ由来で、最終仕様は要確認。

### 仕組み / 背景（なぜこう設計されたか）

agentic コーディングモデルは、単発のコード補完ではなく「計画→編集→実行→検証」を自律ループで回すことを前提に訓練・評価される。ここで鍵になるのが、long-horizon（多段・長時間）のタスクを破綻なく進める能力と、ツール（シェル、ファイル編集、テスト実行）を正しく使う信頼性だ。CTF のようなセキュリティ攻防評価は、こうした自律実行能力の“諸刃”を測る指標でもある——脆弱性を見つけ悪用するのも、防御・修正するのも同じ推論能力に依存するため、高スコアは「攻撃にも転用しうる」ことを意味する。OpenAI の Preparedness Framework は、モデル能力をリスク区分で評価し、閾値を越えたらデプロイに追加の緩和策・審査を課す枠組み。GPT-5.6 Sol が「High」に達したためゲート付きプレビューに留めるのは、この枠組みに沿った判断だ。トレードオフは明快で、能力を最大化するほど誰でも即利用させにくくなる。ティア構成（Sol/Terra/Luna）は、用途とコスト・安全プロファイルを分けて提供するための設計と読める。

### 実務での使いどころ・移行の勘所

agentic コーディングを本番ワークフローに組み込むなら、GPT-5.3-Codex のような現行提供モデルが実利用の対象。GPT-5.6 系はゲート付きプレビューのため、アクセスが得られても本番前提の可用性・価格・レート制限は未確定と考える。導入時は、(1) エージェントに与える権限（ファイル/シェル/ネットワーク）を最小化する、(2) 変更はレビュー/CI を必ず通す、(3) セキュリティ能力の高いモデルほど誤用・過剰実行のガードレール（サンドボックス、承認フロー）を厚くする、が要点。ベンチ値（CTF 96.7% 等）は特定評価上の数値で、自分のコードベースでの有効性は別途検証すること。

## 📖 用語解説（このトピックとのつながり）

- **agentic コーディングモデル**: 計画・編集・実行・検証を自律ループで回すコード生成モデル。Codex の頭脳。
- **Codex**: OpenAI の AI コーディングエージェント。GPT-5.6 Sol などを組み込む。
- **Preparedness Framework**: OpenAI の能力リスク評価枠組み。閾値超えでデプロイに追加緩和を課す。
- **ゲート付きプレビュー**: 審査を通した相手にのみ段階公開する提供形態。高能力モデルの慎重な出し方。
- **CTF（Capture the Flag）**: セキュリティ攻防の競技/評価。自律実行能力の指標にもなる。

## 影響範囲 / 推奨アクション

- 対象: AI コーディングエージェントを開発フローに組み込むチーム。
- アクション: 実運用は現行提供モデル（GPT-5.3-Codex 等）を対象に。エージェント権限は最小化し、変更は必ずレビュー/CI とサンドボックスを通す。GPT-5.6 系は可用性・価格未確定として扱い、ベンチ値は自コードベースで検証。
