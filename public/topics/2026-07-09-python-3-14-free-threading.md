# Python 3.14 — フリースレッド（no-GIL）が正式サポート段階へ（PEP 779）

> トピック: Python 3.14 フリースレッドビルド｜出典: Python 公式ドキュメント｜種別: 言語ランタイムの重大更新

## 一行サマリー

Python 3.14 は PEP 779 の受理により、GIL を持たない「フリースレッドビルド」が実験段階（フェーズ1）から正式サポート（フェーズ2）へ移行し、真のマルチコア並列がオプション扱いながら公式に支えられるようになった一方、コード実行には環境により約1〜8%のオーバーヘッドが伴う。

## 🔰 初学者向け（何が・なぜ重要か）

Python には長年「GIL（ジル、グローバルインタプリタロック）」という仕組みがありました。これは「一度に1つのスレッドしか Python のコードを実行できない」という交通整理の門番です。安全のためですが、CPU が何コアあっても Python の計算は基本1コア分しか使えない、という制約になっていました。重い計算を並列でこなしたいのに、コアを遊ばせてしまうわけです。

Python 3.14 では、この門番を外した **フリースレッド版（no-GIL）** が「実験」から「正式にサポートする」段階へ進みました。これで、複数のスレッドが同時に本当に別々のコアで計算できるようになります。データ処理や AI 関連のように計算量の多い用途で、マシンの性能を余さず使える可能性が広がります。

ただし良いことばかりではありません。門番を外すと安全確保のための別の仕組みが要り、その分だけ普通のコードは少し遅くなります（数%程度）。また、外部ライブラリが no-GIL に対応していないと使えない場合もあります。だから「標準の GIL 版」と「no-GIL 版」の2種類が当面併存し、用途で選ぶ形になります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **PEP 779 が Python 3.14 で受理**され、フリースレッドビルドがフェーズ2（=完全・公式サポート）へ。ただしビルドは依然オプション扱い。
- 通常の Python 3.14 は GIL 有効のまま動作。フリースレッド版は別バイナリとして配布/ビルドする。3.13 以降、公式 macOS/Windows インストーラでフリースレッド版のインストールが選択可能。
- フリースレッドビルドは GIL 有効版に比べ Python コード実行に追加オーバーヘッド。ワークロード・ハードウェア依存で、macOS aarch64 で約1%、x86-64 Linux で約8%程度。
- 3.14 では実験的 JIT の話題も併走（本記事はフリースレッドに焦点。JIT 詳細は要確認）。サードパーティの非対応リストは小さく、リリースごとに縮小中。

### 仕組み / 背景（なぜこう設計されたか）

GIL は CPython の参照カウント方式メモリ管理を「単純かつ高速に」保つための解だった。オブジェクトごとの参照カウントを複数スレッドが同時に増減するとデータ競合が起きるが、GIL があれば一度に1スレッドしか走らないので、カウント操作をロックなしで行える。つまり GIL は「シングルスレッド性能」と「実装の単純さ」を、マルチコア並列と引き換えに買っていた。

フリースレッド化は、この参照カウントをスレッド安全にする必要がある。手法としてバイアス付き参照カウント（オブジェクト所有スレッドは高速パス、他スレッドはアトミック操作）や、内部データ構造の細粒度ロック化などを導入する。代償が「約1〜8%のオーバーヘッド」で、これはアトミック操作やロックのコスト、キャッシュ整合の負荷に由来する。つまり **並列でスケールする利益と、単一スレッド実行がわずかに遅くなる代償のトレードオフ**だ。二本立て配布にしているのは、CPU バウンドで真に並列化したい用途にだけ no-GIL を選ばせ、大多数の I/O バウンド/単一スレッド用途には従来性能を維持するため。C 拡張は参照カウント前提で書かれていることが多く、no-GIL 下での正しさ（スレッド安全性）を各ライブラリが保証し直す必要があるのが移行の実際的なハードルとなる。

### 実務での使いどころ・移行の勘所

科学計算・データ処理・特徴量生成・推論前処理など、Python レベルで CPU バウンドかつ並列化余地が大きい処理で恩恵が出る。逆に I/O バウンド中心のアプリは `asyncio` で十分なことが多く、no-GIL の旨みは小さい上にオーバーヘッド分だけ損をしうる。移行前に (1) 依存ライブラリ（特に C 拡張: NumPy 系, DB ドライバ, ネイティブ拡張）の no-GIL 対応状況を確認、(2) 自前のグローバル状態・共有ミュータブルのスレッド安全性を点検、(3) ベンチマークで実効スケールとオーバーヘッドを実測、という順が安全。当面は「本番は GIL 版、性能検証で no-GIL 版」という併用が現実的。

## 📖 用語解説（このトピックとのつながり）

- **GIL（グローバルインタプリタロック）**: 一度に1スレッドしか Python コードを実行させない機構。本トピックが外そうとする対象。
- **フリースレッドビルド（no-GIL）**: GIL を除いた CPython ビルド。真のマルチコア並列を可能にする。
- **参照カウント**: オブジェクトの参照数でメモリ管理する方式。GIL 撤廃で競合対策が必要になる中核要因。
- **CPU バウンド / I/O バウンド**: 計算律速か入出力律速か。no-GIL が効くのは前者。
- **C 拡張**: C で書かれた Python 拡張。参照カウント前提が多く、no-GIL 対応が移行の鍵。

## 影響範囲 / 推奨アクション

- 影響: CPU バウンド並列処理を持つ Python プロジェクト、および C 拡張を提供するライブラリ作者。
- 推奨: 依存ライブラリの no-GIL 対応を確認し、CPU バウンド処理でベンチ。共有状態のスレッド安全性を点検。当面は GIL 版本番＋no-GIL 版検証の併用。
