# Joomla拡張 Page Builder CK に認証不要のファイルアップロードRCE（CVE-2026-56290）— 修正数時間後にWebシェル設置

> トピック: 脆弱性ニュース ｜ 出典: CCB Belgium / mySites.guru / SentinelOne / CISA KEV ｜ 種別: 危険なファイルの無制限アップロード（CWE-434 / CWE-284）

## 一行サマリー

Joomla向けページ作成拡張「Page Builder CK（Joomlack製）」に、認証なしで任意のファイルを好きなフォルダにアップロードでき、そのままサーバ乗っ取り（RCE）に至る欠陥（CVE-2026-56290, CVSS 10.0）が見つかった。修正公開の数時間後には実際にWebシェルが設置され、CISAは7月7日にKEV登録した。3.6.0への即時更新が必須。

## 🔰 初学者向け（何が・なぜ問題か）

Joomla は、プログラムを書かなくてもWebサイトを作れる人気のCMS（コンテンツ管理システム）です。「Page Builder CK」は、そのJoomlaでページを部品を並べて作るための拡張機能です。

今回の問題は、この拡張が持つ「ファイルをアップロードする窓口」が、**訪問者が誰かを一切確認していなかった**ことです。ログインしていない見ず知らずの人でも、その窓口に細工したファイルを送りつけるだけで、サーバ上の好きな場所にファイルを置けてしまいました。しかも置く場所（フォルダ）まで攻撃者が選べたのです。

ここで攻撃者が「プログラムとして実行できるファイル（PHPファイル＝サーバ上で動く命令書）」を公開フォルダに置くと、あとはそのファイルをブラウザで開くだけで、サーバに好きな命令を実行させられます。これが「Webシェル」で、いわば侵入者が居座るための"裏口の操作パネル"です。

たとえるなら、宅配ボックスが「誰でも・中身を確認せず・置き場所も自由に」荷物を受け取ってしまい、その荷物が実は"家の鍵を勝手に作る装置"だった、というような状況です。パスワードもIDも要らないため、世界中のスキャナから一斉に狙われました。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **脆弱性種別**: 危険な種類のファイルの無制限アップロード（CWE-434）＋不適切なアクセス制御（CWE-284）。
- **CVSS**: 10.0（Critical）。2026年6月29日公開、発見者は mySites.guru の Phil Taylor 氏。
- **前提条件**: **認証不要（pre-auth）**。フロントエンド（公開側）のアップロードエンドポイントが、JoomlaのアンチCSRFトークンでのみ体裁上ガードされているが、このトークンはサイトのページから誰でも読み取れるため実質的な防御にならない。
- **影響バージョン**: Page Builder CK（PageBuilder CK）3.5.10 以前を含む同系列。
- **修正版**: PageBuilder CK 3.6.0（2026年6月27日リリース）。
- **悪用状況**: 修正リリースの数時間後には実環境で悪用され、対象サイトにWebシェルが設置された。CISAが7月7日にKEV登録。

### メカニズム（攻撃が成立する仕組み）

以下は「型（クラス）の説明」であり、武器化できる手順・PoC・正確なパラメータは示さない概念レベルの解説である。

1. **信頼境界**: 「未認証の公開フロントエンド」→「サーバのファイルシステムへの書き込み」が最初の信頼境界。本来ここは"誰が・何を・どこに"を厳格に検査すべき境界だが、その検査が欠落している。
2. **不備のあるコンポーネント**: フロントエンドのアップロード処理が、(a)呼び出し元の認証・権限を確認しない、(b)アップロード可能なファイル種別を制限しない、(c)保存先フォルダを呼び出し側の指定に委ねる、という三重の不備を持つ。唯一のガードであるアンチCSRFトークンは公開ページから取得可能で意味を成さない。
3. **攻撃者が握るプリミティブ**: 「任意ファイルを任意フォルダに書き込む（arbitrary file write）」プリミティブ。特に**Webからアクセス可能かつ実行可能な場所にサーバ実行ファイル（PHP）を配置できる**点が致命的。
4. **橋渡し（RCEへ）**: 配置したPHPファイル（Webシェル）にブラウザからアクセスすると、Webサーバがそれをスクリプトとして解釈・実行する。これで「データの書き込み」が「コードの実行」へ変換され、任意コマンド実行＝サーバ掌握に至る。
5. **最終被害**: Webシェルによる持続的アクセス、サイト改ざん、データ窃取、他サーバへの横展開の足場化。
- **一般原理**: アップロード機能の安全性は「認証・認可」「ファイル種別の許可リスト」「保存先の固定と実行権限の剥奪」の3点セットで担保される。どれか1つでも欠けると、任意書き込み→RCEの王道パターンが成立する。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: 拡張のアップロード用エンドポイントへの未認証POST、公開ディレクトリに突如出現した `.php` 等の実行可能ファイル、それらへの外部からのアクセス、Webサーバプロセスからの想定外プロセス生成・外向き通信。ファイル整合性監視（FIM）で新規ファイルを検知。
- **暫定緩和（パッチ前）**: 拡張を無効化、または該当エンドポイントをWAF/リバースプロキシで遮断。アップロード先ディレクトリでのスクリプト実行を無効化（`.htaccess`/サーバ設定で実行権限を剥奪）。管理面をIP制限。
- **恒久対策**: PageBuilder CK 3.6.0 以降へ更新。更新後も、既に設置されたWebシェルはパッチでは消えないため、Webルート配下の不審ファイルを棚卸しして駆除し、認証情報・DB資格情報をローテーション、侵害範囲を精査する。
- **運用示唆**: CMS拡張は攻撃者の常設標的。フロントエンドの書き込み系機能は"未認証で叩ける前提"で監査し、アップロード先は実行不可の隔離領域に固定する。

## 📖 用語解説（このニュースとのつながり）

- **CMS / Joomla**: プログラミングなしでサイトを構築・運用する仕組み。拡張が多く、拡張側の欠陥がサイト全体の侵害口になる（本件の舞台）。
- **Webシェル**: サーバ上に置かれ、Web経由で命令を実行できる悪性スクリプト。侵入者が居座る"操作パネル"で、本件で実際に設置された。
- **無制限ファイルアップロード（CWE-434）**: 種別や保存先を検査せずファイルを受け入れる欠陥。実行可能ファイルを置ければRCEへ直結する（本件の中核）。
- **アンチCSRFトークン**: 本来はリクエスト偽造対策の仕組み。だが認証・認可の代わりにはならず、公開ページから読めるトークンは防御にならない（本件の"見せかけの防御"）。
- **pre-auth（認証不要）**: ログイン前に成立する攻撃。敷居が最も低く、スキャナによる一括悪用を受けやすい（本件が数時間で悪用された理由）。

## 影響範囲 / 推奨対応

- **影響範囲**: PageBuilder CK 3.5.10 以前を導入したJoomlaサイト全般。インターネット公開サイトは即時リスク。CMS共有ホスティングでは他サイトへの波及も懸念。
- **推奨対応**: (1) 直ちに 3.6.0 以降へ更新、(2) 更新前は拡張無効化またはWAF遮断＋アップロード先の実行権限剥奪、(3) Webルート配下の新規・不審ファイル（特に `.php`）を全点検し駆除、(4) 資格情報ローテーションと侵害調査、(5) 6/27以降のアクセスログを遡及調査。
