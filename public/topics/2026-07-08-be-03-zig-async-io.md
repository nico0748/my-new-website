# Zigの新しい非同期I/O — std.Ioインターフェースが「関数の色付け」を解消する

> トピック: Zig 非同期I/O（std.Io）｜出典: ziglang.org / Loris Cro's Blog｜種別: 言語設計・機能追加

## 一行サマリー

Zig は async/await を「言語キーワード」ではなく標準ライブラリの **std.Io インターフェース**として提供する設計を確定させ、スレッド実行・イベント駆動（Linux は io_uring、macOS は GCD）を同じコードで切り替え可能にした。これにより多くの言語を悩ませてきた「関数の色付け（function coloring）」問題を回避する。async I/O は 0.16.0 サイクルの二大テーマの一つ。

## 🔰 初学者向け（何が・なぜ重要か）

プログラムが「ファイルを読む」「ネットワークを待つ」といった I/O を行うとき、結果が返るまで待たされます。その待ち時間に別の仕事を進めるのが「非同期（async）」です。多くの言語では、非同期の関数には `async` という印を付け、それを呼ぶ側も `await` を付けなければならず、しかも「async な関数」と「普通の関数」が混ざると相性が悪い、という問題があります。これを俗に「関数の色付け（function coloring）」と呼びます。青い関数（async）と赤い関数（同期）が簡単には混ざらない、という比喩です。

Zig は「速くてシンプルなシステムプログラミング言語」を目指す新興言語で、この色付け問題への答えを出しました。async/await を言語の予約語にせず、標準ライブラリの `std.Io` という「I/O のやり方を渡すインターフェース」として実装するのです。コードは「I/O の実体（バックエンド）」を外から受け取るので、同じ関数がスレッドでも、イベント駆動（io_uring など）でも動きます。呼び方を色分けする必要がなくなります。

これは「非同期にするかどうかを関数の書き方に埋め込むのではなく、実行時に注入する道具として扱う」という設計思想で、Zig らしい「明示的で仕込みの少ない」哲学に沿っています。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **方針確定**: 言語に `async`/`await` キーワードは入れない。非同期は **標準ライブラリの `std.Io` インターフェース**として提供する。
- **バックエンド切替**: `std.Io` はスレッドベースとイベント駆動ベースの両方を実装可能。**Linux は io_uring**、**macOS は Grand Central Dispatch（GCD）** を利用。
- **低レベル基盤**: suspend/resume 等のスタックレスコルーチン機構は「低レベルプリミティブ」として残る可能性がある（"stackless coroutines as low-level primitives" 提案。詳細は要確認）。
- **リリース位置づけ**: 非同期 I/O と aarch64 バックエンドが **0.16.0 サイクルの二大テーマ**。0.16.0 は 2026-04 リリース（要確認）。0.15 系はその地ならしを担った。

### 仕組み / 背景（なぜこう設計されたか）

「関数の色付け」は、async を型システムやシグネチャに刻む言語（Rust の `async fn`、JS の `async function` 等）で不可避に生じます。async 関数は async コンテキストからしか自然に呼べず、ライブラリ作者は同期版と非同期版を二重に用意しがちです。

Zig の解法は **依存性注入**に近い発想です。I/O を行う関数は `std.Io` を引数として受け取り、その `Io` が「どう待つか（スレッドで待つ／io_uring に投げる）」を決めます。関数自身は自分が同期か非同期かを知らなくてよく、呼び出し側が渡す `Io` 実装で挙動が決まります。結果として:

- 同じ関数がテスト時はスレッド実装、本番は io_uring 実装、というように差し替えられる。
- 「色」がシグネチャに現れないので、同期/非同期の壁でライブラリを二重化する必要が減る。

トレードオフとして、`Io` を引き回す明示性が求められ、暗黙のグローバルランタイム（Tokio のような）に慣れた開発者には冗長に映ります。しかし Zig は「隠れた制御フローを持たない」ことを美徳とする言語なので、この明示性はむしろ設計上の一貫性です。io_uring / GCD という OS ネイティブの非同期基盤に直接乗ることで、ランタイムのオーバーヘッドを抑えつつ高スループットを狙えます。

### 実務での使いどころ・移行の勘所

- **高スループットなネットワークサーバ / システムツール**を Zig で書く場合、io_uring バックエンドは syscall コストを削減でき有効。
- ライブラリ作者にとって、同期/非同期の二重 API を書かずに済む可能性は保守負担の軽減につながる。
- ただし **API は 0.16 サイクルで安定化途上**。1.0 前の Zig は破壊的変更が頻繁なので、本番採用は変更追従の覚悟が要る。
- Tokio（Rust）のような「暗黙のグローバルランタイム」を期待すると設計が合わない。`Io` を明示的に配線する前提で設計すること。
- macOS では GCD、Linux では io_uring とバックエンドが異なるため、**プラットフォーム間の性能・挙動差**をベンチで確認するとよい。

## 📖 用語解説（このトピックとのつながり）

- **関数の色付け（function coloring）**: async/sync がシグネチャに現れ混在しにくくなる問題。Zig の std.Io はこれを回避。
- **std.Io インターフェース**: I/O のやり方（バックエンド）を注入する Zig 標準ライブラリの抽象。async の中核。
- **io_uring**: Linux の高速非同期 I/O 機構（共有リングバッファで syscall を削減）。Zig の Linux バックエンド。
- **GCD（Grand Central Dispatch）**: macOS/Apple の並行処理基盤。Zig の macOS バックエンド。
- **スタックレスコルーチン**: 各タスクが独自スタックを持たない軽量な中断・再開機構。低レベルプリミティブとして検討中。
- **aarch64 バックエンド**: ARM64 向けコード生成。async I/O と並ぶ 0.16 の重点テーマ。

## 影響範囲 / 推奨アクション

- Zig でネットワーク/システム系を書くなら、std.Io ベースの非同期設計を前提に学習・設計する。
- io_uring 前提の性能を狙う場合、対象 Linux カーネルの io_uring 対応（5.1+）を確認。
- ライブラリを公開するなら、`Io` を引数で受ける API 設計にして色付け二重化を避ける。
- 0.16 系は API 変動が続くため、依存を固定しつつ変更ログを追う運用を。
- macOS/Linux 両対応が要る場合、バックエンド差の性能検証を CI に組み込む。
