# Next.js Partial Prerendering（PPR）— 静的と動的のハイブリッド描画

> トピック: 開発トピック｜出典: nucamp｜種別: frontend

## 一行サマリー
Partial Prerendering（PPR）は、1ページの中で「静的にあらかじめ作れる部分」と「リクエストごとに動的に作る部分」を混在させる描画方式。静的な速さと動的な鮮度を1ページで両立できる。

## 🔰 初学者向け（何が・なぜ重要か）
Webページの作り方には大きく二種類あります。ひとつは「静的」で、あらかじめ完成品を用意しておき誰にでも同じものを瞬時に返す方式。速い代わりに内容は固定です。もうひとつは「動的」で、アクセスのたびにその人向けに作る方式。最新・個別化できる代わりに毎回作るぶん遅くなりがちです。

現実のページは、この両方が混ざっています。たとえば商品ページなら、商品名や説明は皆に共通（静的向き）ですが、「あなたのカート」「在庫状況」はその都度変わる（動的向き）。従来はページ全体をどちらかに寄せる必要があり、無駄が生じていました。

PPR は、1ページの中で「共通部分は静的に先に配り、個別部分だけ後から動的に埋める」ことを可能にします。骨組みを瞬時に表示しつつ、パーソナルな部分を追って流し込むイメージです。速さと鮮度のいいとこ取りができるのが嬉しい点です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
出典によれば Next.js は PPR により静的/動的ハイブリッド描画を提供する。静的シェルを事前生成し、動的な穴（dynamic holes）を Suspense 境界としてストリーミングで埋める仕組み。PPR の安定/実験ステータスやデフォルト有無はバージョンにより異なるため公式で要確認。

### 仕組み / 背景（なぜこう設計されたか）
従来は SSG（完全静的）か SSR（完全動的）かの二者択一で、少しでも動的要素があるとページ全体が動的扱いになりキャッシュ効率が落ちた。PPR は静的な部分を CDN から即返しつつ、`Suspense` で囲った動的部分だけをリクエスト時に解決する。これにより TTFB を抑えつつ個別化を実現する。トレードオフは、境界設計（どこを静的/動的に切るか）の判断が必要になり、動的部分のデータ取得遅延がユーザー体験に直結する点。

### 実務での使いどころ・移行の勘所
ECの商品ページ、ダッシュボード、記事＋パーソナライズ枠のような「共通骨格＋個別要素」構成に最適。移行では、まずページを静的な骨組みと動的な島に分解し、動的島を `Suspense` で包む。動的部分のフォールバックUI（ローディング表示）の設計が体験の質を左右する。

## 📖 用語解説（このトピックとのつながり）
- **PPR（Partial Prerendering）**: 静的と動的を1ページで混在させる描画方式。本トピックの主役。
- **SSG / SSR**: 完全静的生成／サーバー側動的描画。PPR はこの中間を埋める。
- **Suspense**: 非同期部分を待つ React の境界。動的な穴を表現する。
- **dynamic holes**: 静的シェル内の動的に埋める領域。PPR の要。
- **TTFB**: 最初のバイトが返るまでの時間。静的シェルで短縮できる指標。

## 影響範囲 / 推奨アクション
Next.js で個別化と速度を両立したいチームに影響。共通骨格＋個別要素のページは PPR 化を検討。安定ステータスとデフォルト設定を公式で確認し、動的島の分割とフォールバック設計を先に固めるのが推奨。
