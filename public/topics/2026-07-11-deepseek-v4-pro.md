# DeepSeek V4 Pro — MIT ライセンスの 1.6T MoE、圧縮アテンションで 1M 文脈

> トピック: DeepSeek V4 Pro のオープンウェイト公開 ｜ 出典: Hugging Face（deepseek-ai）/ DeepSeek API Docs ｜ 種別: 新モデルリリース

## 一行サマリー

DeepSeek が V4 Pro を MIT ライセンスの完全オープンウェイトで公開。1.6兆パラメータ（うち約49B が活性）の MoE で 100万トークンの文脈長を持ち、圧縮アテンション（CSA/HCA）により V3.2 比で 1M 文脈時の推論 FLOPs を約27%・KVキャッシュを約10%に削減。SWE-bench Verified で 80.6% と、オープンウェイト最高水準を記録した。

## 🔰 初学者向け（何が・なぜ重要か）

大規模言語モデル（LLM）には「重み（パラメータ）が公開されていて誰でもダウンロードして自分のマシンで動かせる」オープンウェイト型と、API 越しにしか使えないクローズド型がある。DeepSeek V4 Pro は前者で、しかも **MIT ライセンス**——商用利用も再配布もほぼ自由という、最も緩い部類の条件で出された。企業が自社サーバーで動かしたり改造したりしやすい。

技術面のキーワードは 3 つ。**MoE（混合エキスパート）** は「巨大な脳を持ちつつ、質問ごとに関係する一部の専門家だけを働かせる」仕組みで、1.6兆という莫大な知識量を持ちながら、実際に動く部分（約490億）は小さく抑えて効率を稼ぐ。**100万トークンの文脈長**は、本何冊分もの情報を一度に読ませられる長さ。**圧縮アテンション**は、その長文処理でかさむ計算とメモリを大幅に削る新工夫だ。

加えて **SWE-bench Verified 80.6%** という数字は、「実際の GitHub のバグ修正課題をどれだけ自力で解けるか」の指標で、オープンウェイトとして最高クラス。つまり「タダで手に入って、改造でき、コーディング能力も一級」——AI を自前で使いたい開発者・企業にとって非常に大きい。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **公開**: 2026-04-22、完全オープンウェイト、**MIT ライセンス**（コード・重みとも）。従来の DeepSeek Model License と異なり商用・再配布が明快。
- **アーキテクチャ**: V4-Pro は **1.6T 総パラメータ / 約49B 活性**の MoE、**1M トークン文脈**。
- **圧縮アテンション**: V4 系は **Compressed Sparse Attention（CSA）** と **Heavily Compressed Attention（HCA）** のハイブリッド。1M 文脈設定で、V3.2 比で単一トークン推論 FLOPs は約27%、KVキャッシュは約10%に削減。
- **性能**: DeepSeek-V4-Pro-Max が **SWE-bench Verified 80.6%**（オープンウェイト最高、Gemini 3.1 Pro と同水準との報）。
- **配布**: Hugging Face `deepseek-ai/DeepSeek-V4-Pro`。重みは FP4+FP8 混合精度で約865GB。V4-Flash（284B 総／13B 活性、1M 文脈）も MIT で併存。
- ※ ベンチ値・詳細仕様は提供元公表に基づき、独立検証状況は「要確認」。

### 仕組み / 背景（なぜこう設計されたか）

**MoE の狙い**。密（dense）モデルは全パラメータを毎回使うため、規模を上げると推論コストが線形以上に膨らむ。MoE は多数の「エキスパート」を持ち、ルータが入力ごとに少数だけを活性化する。これにより「総容量は巨大／実効計算は小さい」を両立し、1.6T の知識量を約49B 活性のコストで扱える。代償はメモリ——全エキスパートを保持するため VRAM 需要が大きく、865GB 級の重みは複数 GPU が前提になる。

**なぜ圧縮アテンション**。Transformer の自己注意は文脈長の二乗でコストが増え、長文では KVキャッシュ（過去トークンのキー・値の保持）がメモリを圧迫する。CSA/HCA は注意計算とキャッシュを圧縮・スパース化し、1M 文脈という極端な長さでも FLOPs とメモリを実用域に収める。長文脈をエージェント用途（大量のツール履歴・コードベース全体の読み込み）で使う流れに合致した設計だ。トレードオフとして、圧縮は情報のロスを伴い得るため、極端な長距離依存での精度は評価が要る。

**MIT 化の意味**。重みまで MIT で出すことは、企業が法務リスクを気にせず商用製品に組み込み・微調整・再配布できることを意味する。クローズド API 勢に対し「自前で回せる高性能モデル」を武器に市場を取りに行く戦略で、オープンウェイト陣営の到達点を一段押し上げた。

### 実務での使いどころ・移行の勘所

自社データを外に出さずに動かしたい企業の内製 LLM、長大なコードベースや文書群を一括で読ませるエージェント、コーディング支援に向く。ただし 1.6T MoE は推論インフラが重く、865GB 級の重みを載せる複数 GPU（および高速な相互接続）が要る——自前運用は GPU コストと運用能力が前提。手軽さ優先なら V4-Flash（13B 活性）や量子化版、あるいはホスティング API の利用が現実的。MIT ライセンスは商用に有利だが、微調整データや出力の取り扱いは自社ポリシーで別途管理する。ベンチ値はベンダ公表ベースのものを含むため、自社タスクでの実測評価を推奨。

## 📖 用語解説（このトピックとのつながり）

- **オープンウェイト**: モデルの重みが公開され、手元で実行・改造できる形態。V4 Pro は MIT で公開。
- **MoE（Mixture of Experts）**: 多数のエキスパートから入力ごとに一部だけ活性化する構造。総容量と実効計算を分離。
- **活性パラメータ**: 1トークン処理で実際に使うパラメータ数。V4-Pro は約49B（総1.6T）。
- **文脈長（context length）**: 一度に扱える入力トークン数。V4 は 1M。
- **KVキャッシュ**: 生成時に過去トークンのキー・値を保持する領域。長文でメモリを圧迫。圧縮アテンションが削減。
- **SWE-bench Verified**: 実際の GitHub 課題を解けるかを測るコーディングベンチ。80.6% はオープンウェイト最高水準。
- **MIT ライセンス**: 商用・再配布に寛容な緩いライセンス。企業採用の障壁を下げる。

## 影響範囲 / 推奨アクション

- 内製 LLM・長文脈エージェント・コーディング支援を自前で回したい組織に有力。
- 1.6T MoE は複数 GPU 前提——手軽さ優先なら V4-Flash・量子化版・ホスティング API を検討。
- 公表ベンチは自社タスクで実測評価を。長距離依存の精度は圧縮アテンションの特性を踏まえ検証。
