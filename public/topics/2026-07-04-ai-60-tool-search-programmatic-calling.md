# Tool Search と Programmatic Tool Calling ― 数千MCPツールを捌く本番エージェントのスケーリング術

> トピック: 開発トピック｜出典: anthropic.com / releasebot（Anthropic）｜種別: ai

## 一行サマリー
Anthropic が API に **Tool Search**（大量ツールから必要なものを検索して絞り込む）と **Programmatic Tool Calling**（ツール呼び出しをコードで束ねて実行）を導入。MCP が普及して1エージェントが数千のツールに接続しうる時代に、全ツールを毎回モデルへ提示する非効率を避け、レイテンシとコンテキスト消費を抑えて本番規模のエージェントを成立させる仕組み。

## 🔰 初学者向け（何が・なぜ重要か）
AI エージェントに「使える道具（ツール）」をたくさん持たせると賢く動けますが、道具が数千個になると別の問題が起きます。毎回「これ全部が使えますよ」と AI に一覧を見せると、説明だけで頭（コンテキスト＝一度に読める量）がいっぱいになり、遅く・高コストになるのです。

**Tool Search** は、その巨大な道具箱から「今の仕事に関係する道具だけ」を先に検索して手渡す仕組みです。工具棚から必要な工具だけ出す、というイメージ。**Programmatic Tool Calling** は、複数の道具の使い方を AI が一つ一つ言葉でやり取りする代わりに、まとめてコードとして実行させる仕組みです。手順書を毎回口頭で確認せず、スクリプトで一気に流す感じ。どちらも「道具が増えても実用的な速さ・コストで動かす」ための工夫で、エージェントを本番で使うほど効いてきます。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
- Anthropic が API に **Tool Search** と **Programmatic Tool Calling** を追加（本番規模の MCP 展開で、数千ツールを効率的に扱いレイテンシを削減、要確認で最新の一次情報を参照）。
- 背景に MCP のほぼ普遍的な普及（Claude のディレクトリは 75+ コネクタ、公開 MCP サーバは1万超が本番稼働）。1エージェントが接続しうるツール数が爆発。
- 関連整備として、CLI をエージェントが直接駆動できるよう **vault の環境変数対応**（API キー名と到達可能ドメインを登録し、認証付きリクエストを実行）も拡張。

### 仕組み / 背景（なぜこう設計されたか）
LLM エージェントは、利用可能なツールの定義（名前・引数スキーマ・説明）をプロンプト（コンテキスト）に載せて動きます。ツールが数個なら問題ありませんが、数百〜数千になると、その定義だけでコンテキストを食い潰し、推論が遅く・高コストになり、モデルが「どれを使うべきか」を選ぶ精度も落ちます。**Tool Search** は、この問題を「関連ツールだけを検索・注入する」動的な絞り込みで解決します。全件を常時提示するのではなく、タスクに応じて必要な部分集合だけをモデルに見せる——RAG（検索拡張）をツール選択に応用した発想です。

**Programmatic Tool Calling** は、ツール実行の“オーケストレーション”の効率化です。従来は「モデルが1ツール呼ぶ→結果を受け取る→次を考える」という会話往復を繰り返し、往復ごとにトークンとレイテンシを消費しました。呼び出しをコードとして表現・実行できれば、複数ツールの連鎖や制御フロー（分岐・反復）を1回のプログラム実行にまとめられ、モデルとの往復回数を減らせます。トレードオフは、検索で絞る以上「関連ツールを取りこぼす」リスクの管理が必要な点、そしてプログラム実行を挟むぶん、実行環境のサンドボックス化や権限・監査といった運用設計が重要になる点です。

### 実務での使いどころ・移行の勘所
- 多数の MCP サーバ／ツールを束ねる本番エージェント（社内システム横断、業務自動化）で、コストとレイテンシを実用域に収められる。
- ツール定義が肥大化してコンテキスト圧迫や選択精度低下が起きている場合、Tool Search で改善余地が大きい。
- Programmatic Tool Calling を使う際は、実行のサンドボックス化・権限スコープ・監査ログを設計に組み込む。取りこぼし対策として検索対象のメタデータ整備が効く。

## 📖 用語解説（このトピックとのつながり）
- **Tool Search**: 大量ツールから関連するものだけを検索・注入する仕組み。本トピックの主役の一つ。
- **Programmatic Tool Calling**: ツール呼び出しをコードで束ねて実行し、往復回数を減らす仕組み。
- **MCP (Model Context Protocol)**: エージェントのツール接続標準。ツール数爆発の前提。
- **コンテキスト（ウィンドウ）**: モデルが一度に読める量。ツール定義の肥大が圧迫する。
- **オーケストレーション**: 複数ツール実行の制御・連鎖。プログラム化で効率化する対象。

## 影響範囲 / 推奨アクション
多数のツールを扱う本番エージェントを構築する AI 開発者に直接関係する。ツール定義でコンテキストが逼迫している、または呼び出し往復のコストが問題化しているなら、Tool Search と Programmatic Tool Calling の導入で改善できる。導入時は検索メタデータの整備と、プログラム実行の権限・サンドボックス設計を重視する。
