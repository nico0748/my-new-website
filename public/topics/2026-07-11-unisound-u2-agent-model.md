# Unisound U2 — 100ステップ超を自律で解く「実行特化」ネイティブ・エージェントモデル

> トピック: Unisound U2 のリリース ｜ 出典: PR Newswire / Unisound Token Hub / llm-stats ｜ 種別: 新モデルリリース

## 一行サマリー

中国の Unisound が 2026-06 に公開した U2 は、266B 総 / 10B 活性の MoE で「実行のために作られた」ネイティブ・エージェントモデル。複雑な実世界ワークフローを 100 ステップ超に自律分解して完遂することを掲げ、トークン消費は兆規模の密モデルの約25%。IFBench・Agent 評価・GPQA で少ない活性パラメータながら競争力を主張する。

## 🔰 初学者向け（何が・なぜ重要か）

多くの大規模言語モデルは「賢く答える」ことを主目的に作られ、エージェント（自分で道具を使って作業を進める使い方）は後付けで乗せることが多い。U2 は発想が逆で、**最初から"作業を実行しきる"ことを目的に設計されたモデル**だ。開発元は「複雑な現実の業務を 100 以上のステップに自分で分解し、最後までやり切れる」ことを売りにしている。

ポイントは効率の考え方だ。U2 は「**高い知能密度・高いトークン価値**」を掲げる。噛み砕くと、「少ない計算（活性10B）で、できるだけ結果に直結する応答を出す」——1回1回の呼び出しを"成果物に近づける"設計だという。実際、同じ仕事をこなすのに使うトークン量が、超巨大な密モデルの4分の1ほどで済むとされる。エージェントは何十回も AI を呼び出すため、この"1呼び出しあたりの無駄の少なさ"はコストに直結する。

比喩でいえば、饒舌だが遠回りする相談役ではなく、「短い言葉で的確に次の一手を出し、作業を前に進める実務家」を目指したモデルだ。エージェント運用のコスト効率を重視する流れの、中国勢からの一手といえる。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **公開**: 2026-06（PR は 6/8 前後）。Unisound Token Hub で個人・開発者・組織向けに提供。
- **アーキテクチャ**: スパース MoE、**266B 総 / 10B 活性**。トークン使用量は兆規模の密モデルの約25%。ナレッジカットオフは 2026年1月。
- **位置づけ**: 「ネイティブ・エージェント大規模モデル」。複雑な実世界ワークフローを 100 ステップ超に自律分解・完遂する能力を掲げる。設計思想は "high intelligence density / high token value"（少ない活性資源でより強い能力、各呼び出しを成果物に近づける）。
- **ベンチ**: 指示追従の IFBench、Claw 系 Agent 評価、難問推論の GPQA で、少ない活性パラメータながら競争力を主張。
- **価格**: 入力 $0.150 / 出力 $0.300（100万トークンあたり、Unisound 経由）。
- ※ ベンチ・主張はベンダ公表ベース、独立検証は「要確認」。オープンウェイト配布の有無も要確認（Token Hub 経由 API 提供が確認できる範囲）。

### 仕組み / 背景（なぜこう設計されたか）

**"ネイティブ・エージェント"とは**。従来は汎用 LLM にプロンプトやスキャフォールドを被せてエージェント化していた。U2 は学習段階から「タスク分解・ツール使用・長い実行系列の完遂」を主目的に据えたと主張する。エージェント運用で効くのは、単発の賢さより「多段実行を通じて破綻せず前進する」安定性であり、そこを直接最適化する狙いだ。

**トークン効率という指標**。エージェントは 1 タスクで数十〜数百回モデルを呼ぶため、総コストは「呼び出し回数 × 1回あたりトークン量 × 単価」で決まる。U2 の "high token value" は、冗長な思考の垂れ流しを抑え、各出力を行動・成果に直結させることで総トークンを圧縮する発想。10B 活性の MoE により 1 呼び出しの計算コストも低い。両者を掛け合わせ、「エージェントを回し続けても安い」を狙う。トレードオフは、効率を優先するほど"じっくり長考"型タスクでの深さが犠牲になり得る点で、深い推論が要る問題での実力は評価が要る。

**MoE の一般原理**。266B の容量を持ちつつ 10B だけ活性化することで、密な 266B より遥かに安く動く。専門知識を局所的に呼び出すエージェント作業と相性がよい。メモリには全エキスパートを載せる必要があるため、自前運用時は相応の GPU が要る（API 利用なら関係しない）。

### 実務での使いどころ・移行の勘所

多段の業務自動化（フォーム操作、社内システム横断の処理、繰り返しの実務フロー）で、コスト効率よくエージェントを回したい用途に向く。評価の勘所は、ベンダ公表のベンチではなく**自社の実タスクでの完遂率とトークン消費**を実測すること——"100ステップ完遂"や"25%トークン"は宣伝上の主張で、ワークフロー次第で結果は変わる。中国系モデルゆえ、データの取り扱い・提供経路・コンプライアンス要件（機微データを Token Hub API に送ってよいか）を組織ポリシーで確認する。深い推論が主眼のタスクでは、推論特化モデルとの比較も併せて行いたい。

## 📖 用語解説（このトピックとのつながり）

- **ネイティブ・エージェントモデル**: 学習段階からエージェント実行（分解・ツール使用・多段完遂）を主目的に設計したモデル。U2 の位置づけ。
- **MoE / 活性パラメータ**: 一部エキスパートだけ起こす構造と実効計算量。266B 総 / 10B 活性。
- **トークン価値（token value）/ 知能密度**: 1 出力・1 活性資源あたりどれだけ成果に近づくかという効率指標。U2 の設計軸。
- **IFBench / GPQA / Claw**: それぞれ指示追従・難問推論・エージェント評価のベンチマーク。
- **ナレッジカットオフ**: 学習データの知識が及ぶ最終時点。U2 は 2026年1月。
- **スキャフォールド**: 汎用モデルをエージェント化する実行の足場。ネイティブ設計はこれへの依存を減らす狙い。

## 影響範囲 / 推奨アクション

- 多段業務自動化をコスト効率よく回したい用途に候補。ただし主張は自社実タスクで実測評価を。
- "100ステップ完遂"・"トークン25%"はベンダ主張——完遂率とトークン消費を自前ワークフローで検証。
- 中国系モデルのため、データ経路・コンプライアンス要件を組織ポリシーで確認。深い推論用途は推論特化モデルと比較。
