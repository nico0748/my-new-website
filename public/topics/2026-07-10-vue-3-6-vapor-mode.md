# Vue 3.6 Vapor Mode が機能完成 — 仮想 DOM を捨てて Solid 級の描画速度へ

> トピック: Vue 3.6 Vapor Mode（beta・機能完成）｜出典: Vue 公式（X/リリースノート）ほか（安定時期は要確認）｜種別: レンダリング方式の刷新

## 一行サマリー

Vue 3.6 の目玉「Vapor Mode」が 2026年4月に機能完成（feature-complete）に達し、単一ファイルコンポーネントを仮想 DOM を経由せず直接 DOM 操作へコンパイルする描画方式が、既存の仮想 DOM モードとほぼ同等の機能パリティを持つに至った。ただし正式安定は 3.6 系のリリース進行に依存し、既定化はさらに先の見込み（時期は要確認）。

## 🔰 初学者向け（何が・なぜ重要か）

Vue や React は「仮想 DOM」という仕組みで画面を更新してきた。これは「本物の画面（DOM）」のコピーをメモリ上に作り、変更前後を比較して差分だけを本物に反映する方式。賢いが、コピーを作って比較する手間そのものがコスト（速度・メモリ）になる。

Vapor Mode は、この「コピーして比較」を丸ごとやめる新モード。コンポーネントを事前に解析して「この値が変わったら、この DOM のここだけ書き換える」という命令に直接コンパイルしてしまう。結果、Solid.js や Svelte のような「細かいところだけをピンポイントで更新する」高速さに近づく。しかも Vue の書き方（テンプレートやリアクティビティ）はそのままなので、学び直しは最小限。「速いフレームワークに乗り換える」のではなく「今の Vue が速くなる」のがうれしいところだ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

Vapor Mode は SFC をコンパイル時に仮想 DOM ノードではなく、実 DOM を直接生成・更新する命令列へ変換する。2025年12月に beta 投入、2026年4月に「Suspense を除き仮想 DOM モードの安定機能とパリティ」との告知が出た（3.6.0-beta 系）。ベンチでは極端なケースで最大約97%高速という主張もあるが、これは理想条件の数値で、一般的なアプリでの実効値は用途依存（要確認）。有効化はコンポーネント/ビルド単位でオプトイン可能な設計で、既存の仮想 DOM モードと共存する。安定リリース時期は 3.6 系の進行に依存し、既定モード化は 2027 見込みという情報もある（要確認）。

### 仕組み / 背景（なぜこう設計されたか）

仮想 DOM は「宣言的に書けて、差分計算はランタイムが面倒みる」代わりに、更新のたびにツリー生成・diff・パッチのオーバーヘッドを払う。近年 Solid/Svelte が示したのは「コンパイラがリアクティブ依存を静的に解析できれば、更新箇所は事前に確定でき、diff は不要」という方向。Vue はもともと `ref`/`reactive` による細粒度リアクティビティを持つため、テンプレートのコンパイル出力を「シグナル→特定 DOM 更新」の直結コードに変えるだけで、既存の作者体験を保ったまま仮想 DOM を外せる。トレードオフは、コンパイラが解析しづらい動的すぎるテンプレート表現や、仮想 DOM 前提のエコシステム（一部のレンダラ・ツール）との互換で、だからこそ全面切替ではなくオプトイン共存という慎重な設計を取っている。

### 実務での使いどころ・移行の勘所

大量のリストや高頻度更新（ダッシュボード、リアルタイム表示、データグリッド）を持つ画面で恩恵が大きい。既存 Vue 3 プロジェクトなら書き換えはほぼ不要で、ホットパスのコンポーネント/ルートから段階的にオプトインするのが現実的。注意点は、正式安定前は本番全面採用を避け検証にとどめること、Suspense など未パリティ機能や仮想 DOM 前提のサードパーティ描画ロジックとの併用可否を確認すること。代替として、いま速度が必要で新規なら Solid、既存 Vue 資産を活かすなら Vapor という住み分けになる。

## 📖 用語解説（このトピックとのつながり）

- **仮想 DOM（Virtual DOM）**: DOM のメモリ上コピーを差分比較して更新する方式。Vapor Mode はこれを廃し直接 DOM 操作にする。
- **細粒度リアクティビティ**: 変更があった値に紐づく最小の描画だけを更新する仕組み。Solid/Svelte の速さの源で、Vapor もこの路線。
- **feature-complete（機能完成）**: 予定機能が出揃った状態。ただし安定（stable）とは別で、まだ破壊的変更や不具合修正の余地がある。
- **オプトイン**: 既定では無効で、明示的に有効化して使う方式。Vapor は既存モードと共存させるためこの形を取る。

## 影響範囲 / 推奨アクション

- Vue エコシステム全体の性能上限が引き上がる。パフォーマンス理由で Solid/Svelte を検討していた Vue チームは再評価の価値あり。
- 現時点では検証・PoC 段階。ホットパスのコンポーネントでベンチを取り、正式安定を待って本番導入する。
- ライブラリ作者は仮想 DOM 前提の実装が Vapor 下でも動くか互換確認を進める。
