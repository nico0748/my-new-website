# Rails 8.1 の Active Job Continuations — 「中断しても続きから」が AI ワークフロー時代に再注目

> トピック: Active Job Continuations（Rails 8.1）の設計と2026年の再評価｜出典: Ruby on Rails 公式ブログ｜種別: 設計論・ベストプラクティス（リリース自体は2025年10月、AI バッチ処理の文脈で議論が再燃）

## 一行サマリー

Rails 8.1 で導入された Active Job Continuations は、長時間ジョブを「ステップ」に分割して進捗を保存し、デプロイや再起動で中断されても最初からでなく続きから再開できる仕組みで、LLM 呼び出しを含む長時間バッチが増えた2026年、軽量な耐久実行パターンとして改めて注目されている。

## 🔰 初学者向け（何が・なぜ重要か）

ウェブサービスの裏側では「バックグラウンドジョブ」と呼ばれる裏方の処理が動いています。数万人へのメール送信、大量データの集計、動画の変換などです。困るのは、こうした長い処理の途中でサーバーが再起動（デプロイのたびに起きます）されると、処理が最初からやり直しになることでした。90分の映画を見ている途中で停電し、続きからでなく冒頭から見直しになるようなものです。

Rails 8.1 の「Continuations（継続）」は、ジョブを「ステップ」という章立てに分けて、「第3章まで完了」としおりを挟みながら進む仕組みです。中断されても、しおりの位置から再開されます。繰り返し処理の途中位置（カーソル）も保存できるので、「10万件中 62,431 件目まで処理済み」といった細かい単位でも続きから走れます。

いまこの機能が再注目されているのは、AI 関連の処理と相性が良いからです。LLM への問い合わせを何千回も繰り返すバッチは時間もお金もかかるため、「やり直しのたびに同じ問い合わせ料金を二重に払う」のを防げる価値が大きいのです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- `ActiveJob::Continuable` を include し、`step` メソッドでジョブを離散ステップに分割。ステップはブロック（インラインロジック）、カーソル付き（コレクション処理用）、メソッド参照の3形式で定義可能。
- ステップ間の進捗、およびカーソルによるステップ内の反復位置を Rails が自動保存。中断後はちょうど中断地点から再開する。
- 背景: Kamal でのデプロイはジョブ実行コンテナに既定30秒しか猶予を与えないため、30秒を超えるジョブは「中断される前提」で設計する必要がある。
- Rails 8.1 にはほかに構造化イベント（structured events)、ローカル CI などが同梱（本記事は Continuations に焦点）。
- コスト面の試算例として、ワーカー時間単価 $0.10/時のクラウドで1ジョブ種あたり月 $33.60、20〜30 種で月 $500〜1,000 の節約になるとの解説記事がある（前提条件依存の概算であり要確認）。

### 仕組み / 背景（なぜこう設計されたか）

Continuations は Durable Execution（Temporal、DBOS 等）の「軽量版・Rails 内蔵版」と読むのが正確である。フル装備の耐久実行がステップの入出力・実行履歴を完全に永続化して決定論的リプレイを行うのに対し、Continuations は「どのステップまで終わったか＋カーソル」だけを保存する簡素な設計を選んだ。得られるのは導入の軽さ（include して step で括るだけ、追加インフラなし）で、失うのは厳密性——各ステップは「少なくとも1回」実行されるため、**ステップ単位の冪等性は開発者の責務**として残る。

この割り切りは Rails/37signals の「conceptual compression」哲学の典型で、Solid Queue（DB ベースのキュー）と同様、「Redis や外部オーケストレータを増やさず、まず Postgres/MySQL と Rails だけで運用を成立させる」路線にある。Kamal の30秒制約という自社の運用現実から必要性が逆算されている点も 37signals 流。

2026年の再評価は、AI ワークフローの構造がまさに「高価で時間のかかる外部呼び出しの長い列」であるため。jetrockets らによる「ActiveJob::Continuable で resilient な AI バックグラウンドジョブを作る」系の実践記事が続き、LLM バッチ（埋め込み生成、一括分類、レポート生成）の標準パターンとして定着しつつある。

### 実務での使いどころ・移行の勘所

- 適用すべき: 5秒超・反復型・再実行コストが高いジョブ（大量メール、埋め込み生成、外部 API 同期、データ移行）。
- 適用すべきでない: 5秒未満の高速ジョブ（オーバーヘッド損）、単一トランザクションで原子的に終えるべき処理、単純なジョブ。公式もこの線引きを明示している。
- 設計の勘所: ステップ境界＝再実行され得る境界。各ステップ冒頭で「既に処理済みか」を確認できるようにする（unique 制約、処理済みフラグ、upsert）。カーソルは「ソート安定なキー」で進めること（オフセットは途中挿入でズレる）。
- Sidekiq ユーザーは Sidekiq Iteration（同種の機能）との比較、より厳密な保証が必要なら Temporal/DBOS への段階的移行が選択肢。

## 📖 用語解説（このトピックとのつながり）

- **Active Job**: Rails のジョブ抽象層。Continuations はこの層の機能なのでキュー実装（Solid Queue/Sidekiq 等）を問わず使える。
- **カーソル**: 反復処理の現在位置の記録。中断時に保存され、再開時にそこから続行する。
- **冪等性**: 再実行しても結果が変わらない性質。「少なくとも1回」実行の世界では必須の設計要件。
- **Kamal**: Rails 公式のコンテナデプロイツール。既定30秒の停止猶予が「中断前提設計」の直接動機。
- **Durable Execution**: 進捗永続化による再開可能実行の一般概念。Continuations はその最小実装、Temporal/DBOS は完全実装。

## 影響範囲 / 推奨アクション

- Rails 8.1+ のプロジェクト: 30秒超のジョブを棚卸しし、Continuable 化の候補リストを作る（特に LLM/外部 API 呼び出しを含むもの）。
- 8.0 以前: この機能はアップグレード動機の一つになる。移行前は自前のチェックポイント（DB フラグ等）で代替。
