# Windows HTTP.sys に CVSS 9.8〜10.0 の未認証RCE（CVE-2026-47291）— 整数オーバーフローがカーネル権限の任意実行へ

> トピック: 脆弱性ニュース ｜ 出典: Microsoft MSRC / Zero Day Initiative / Talos / NVD ｜ 種別: 整数オーバーフロー由来のヒープバッファオーバーフローによる未認証リモートコード実行（カーネル権限）

## 一行サマリー

IIS や WinRM など多くのWindowsサービスの土台で動く中核ドライバ HTTP.sys に、約65,535バイトを超える細工HTTPリクエストで整数オーバーフローを誘発し、カーネル権限で任意コードを実行できる CVSS 9.8〜10.0 の未認証RCE（CVE-2026-47291）が、6月のPatch Tuesdayで修正された。Webサービスを公開する全Windows環境が対象で、即時パッチ適用が必要。

## 🔰 初学者向け（何が・なぜ問題か）

HTTP.sys は、Windowsが「Webの通信（HTTP）」を受け取るときに最初に処理する“入口の仕分け係”です。普段は表に出ませんが、Windowsのウェブサーバ（IIS）やリモート管理（WinRM）など、HTTPを使う多くの機能がこの係に支えられています。しかもこの係は、OSの最も深い場所（カーネル）で働く特別な存在です。

今回の問題は、この仕分け係が「リクエストの大きさ」を数えるときに、小さな計算用メモ（16ビットの整数＝65,535までしか数えられない器）を使っていたことにあります。攻撃者が65,535バイトを超える非常に大きなリクエストを送ると、数える器があふれて、計算結果が“ぐるっと一周して小さな数”になってしまいます。

その結果、「小さい荷物が来た」と勘違いして小さな置き場所しか用意しないのに、実際には巨大な荷物が運び込まれ、置き場所からあふれてしまいます（バッファオーバーフロー）。このあふれを攻撃者が巧みに使うと、Windowsの最も深い部分（カーネル）で好きなプログラムを動かせてしまいます。

なぜ深刻かというと、(1)ログイン不要で攻撃できる、(2)操作を誘う必要もない（利用者が何もしなくても成立）、(3)奪われる権限がカーネル＝OSの全権、という三拍子が揃っているからです。インターネットにWebサービスを公開しているWindowsは、特に危険です。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **CVSS**: CVSS v3.1 で 10.0 とする情報源と 9.8 とする情報源があり、いずれにせよ Critical（正確な値は**要確認**）。**未認証・ネットワーク経由・低複雑度・ユーザー操作不要**。
- **CWE分類**: CWE-190（Integer Overflow or Wraparound）を起点とし、CWE-787（Out-of-bounds Write）/ ヒープバッファオーバーフローへ連鎖。
- **対象コンポーネント**: HTTP.sys（カーネルモードのHTTPプロトコルスタックドライバ）。IIS・WCF・WinRM など HTTP を扱う各種Windowsサービスの基盤。
- **前提条件**: HTTP.sys 経由でHTTPを受け付ける構成。**既定の `MaxRequestBytes` レジストリ値を使用している場合は影響を受けない**とされ、非既定の大きな値に変更している環境が危険。
- **悪用条件**: ヘッダブロックが65,535バイト以上に達する細工HTTPリクエストで16ビット整数演算をオーバーフローさせる。
- **修正・緩和**: 6月のPatch Tuesday（2026-06-09公開）で修正。パッチ適用前の暫定策として `MaxRequestBytes` のレジストリ調整（MicrosoftがPowerShellスクリプトを提供）。
- **悪用状況**: 公開時点で広範な実環境悪用の確証は**要確認**だが、未認証・カーネルRCEという性質上、優先度は最高。

### メカニズム（攻撃が成立する仕組み）

以下は概念レベルの説明であり、武器化可能な手順・PoC・オフセット等は含めない。

1. **信頼境界の所在**: HTTP.sys はネットワークからの「未認証の生入力」を最初に解析するカーネルコンポーネント。ここでの境界＝「外部の任意バイト列」と「カーネル内部の信頼された処理」の間にある。
2. **整数演算の不備（根因）**: リクエスト全体／ヘッダブロックのサイズ計算に16ビット幅の整数演算が用いられている。サイズが 65,535 に達すると値がラップアラウンド（一周して小さな値に）する。
3. **過小確保のプリミティブ**: ラップした「小さな計算結果」に基づきヒープ領域が確保される一方、実際に書き込まれるデータは本来の（大きな）サイズ。確保サイズ＜書込サイズの不整合が生じ、ヒープ上で範囲外書き込み（out-of-bounds write）が発生する。
4. **メモリ破壊から制御奪取へ**: 攻撃者は書き込み内容とヒープ配置を制御することで、隣接するカーネルオブジェクトやメタデータを上書きし、制御フローを掌握する足場を作る。over-write プリミティブが最終的にカーネル権限での任意コード実行へ橋渡しされる。
5. **権限と被害の規模**: HTTP.sys はカーネルモードで動作するため、成立した実行はそのままカーネル権限。OSの分離境界の上位にあたるため、被害は機器全体の完全掌握に至る。

脆弱性クラスとしては、(a) サイズ計算における整数オーバーフロー（CWE-190）、(b) それに起因するヒープ範囲外書き込み（CWE-787）の連鎖という、メモリ安全性に関する古典的かつ高危険度のパターン。カーネル境界で認証前に成立する点が危険度を最大化している。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: 65,535バイト超の異常に大きなHTTPヘッダ／リクエスト、HTTP.sys/`http`ドライバ関連のクラッシュやブルースクリーン（攻撃の試行で発生しうる）、IIS/WinRMプロセスの異常終了、システムログのカーネル例外。境界のWAF/IDSでヘッダサイズ閾値超過のリクエストを監視・遮断。
- **暫定緩和（パッチ前）**: `MaxRequestBytes` を既定値（または安全な値）に戻す（Microsoft提供のPowerShellスクリプト／手順に従う）。不要なHTTPエンドポイントの停止、WinRMのインターネット露出排除。
- **恒久対策**: 6月Patch Tuesdayの更新を即時適用。適用後はレジストリ暫定策の要否を見直す。
- **運用示唆**: HTTP.sys は「多数のサービスが共有する単一の土台」。一つの欠陥が IIS/WCF/WinRM など広範囲に波及するため、Webを受けるWindowsは公開・非公開を問わず最優先で更新する。

## 📖 用語解説（このニュースとのつながり）

- **HTTP.sys**: WindowsがHTTP通信を受け取る最初の処理を担うカーネルモードドライバ。意味＝Web通信の「入口の仕分け係」。本件＝この共有基盤の欠陥が広範囲に波及。
- **整数オーバーフロー（CWE-190）**: 計算結果が型の最大値を超えて一周し小さな値になる現象。意味＝「数える器があふれる」。本件＝過小確保の引き金。
- **ヒープバッファオーバーフロー（CWE-787）**: 確保領域を超えてメモリへ書き込む欠陥。意味＝「置き場所からあふれる」。本件＝制御奪取の足場。
- **カーネルモード**: OSの最も特権の高い実行領域。本件＝奪われる権限がOS全権であることを意味する。
- **MaxRequestBytes**: 受け付けるHTTPリクエストの最大サイズを定めるレジストリ値。本件＝既定値なら影響回避、非既定の大値が危険。暫定緩和の鍵。
- **未認証RCE**: ログイン不要で任意コードを実行できる脆弱性。本件＝攻撃の前提が極めて低い。

## 影響範囲 / 推奨対応

影響を受けるのは、HTTP.sys 経由でHTTPを受け付ける全Windows環境（IIS・WinRM・WCF等を含む）。とりわけ非既定の `MaxRequestBytes` を設定し、Webサービスをインターネット公開している環境は最優先。推奨対応は、(1) 6月Patch Tuesday更新の即時適用、(2) パッチ前は `MaxRequestBytes` を既定／安全値へ調整（Microsoft手順）、(3) 境界で巨大ヘッダ・異常リクエストの監視と遮断、(4) WinRM等の管理用HTTPの露出排除、(5) HTTP.sys関連クラッシュの監視による試行検知。
