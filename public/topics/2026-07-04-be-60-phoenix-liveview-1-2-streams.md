# Phoenix LiveView 1.2 ― JavaScript を書かずにリアルタイム UI、Streams で大規模リストも軽量に

> トピック: 開発トピック｜出典: phoenixframework.org / hex.pm｜種別: backend

## 一行サマリー
Elixir の Web フレームワーク Phoenix の目玉機能 LiveView が **1.2** をリリース。サーバー主導でリアルタイムな画面更新を実現しつつ、**Streams API** により大規模・動的なリスト（リアルタイムフィードやチャット）をメモリ効率よく扱えるようになった。「JavaScript をほとんど書かずに双方向 UI を作る」路線がさらに実用度を増した。

## 🔰 初学者向け（何が・なぜ重要か）
チャットや通知フィードのような「相手の操作が自分の画面に即反映される」UI は、普通フロント側で大量の JavaScript を書いて作ります。Phoenix LiveView は、その双方向のやり取りを**サーバー側**で処理し、差分だけをブラウザに送る仕組みです。開発者は JS をほとんど書かず、サーバーのコードだけでリアルタイム画面を組めます。

今回の 1.2 で強化された「Streams」は、長くて絶えず増減するリスト（例：流れ続けるチャットログ）を効率よく扱う仕組みです。従来は「リスト全体をサーバーの記憶に持ち続ける」必要があり、件数が増えるとメモリを圧迫しました。Streams は「全部を抱え込まずに、変化した項目だけを差し込む／消す」ようにでき、大量データでも軽く動きます。掲示板の全投稿を毎回積み直すのではなく、新着だけを貼り足すイメージです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
- **Phoenix LiveView 1.2** をリリース（2026 年時点の最新系列。ブラウザ対応要件など細部は公式ブログ参照、要確認）。
- **`Phoenix.LiveView.Stream` API** が、大規模・動的リストの扱いを改善。サーバー側にリスト全体を保持せず、追加・更新・削除を差分としてクライアントへ反映。
- リアルタイムフィード、チャット、無限スクロール等、項目数が大きく変動する UI での省メモリ運用が主眼。

### 仕組み / 背景（なぜこう設計されたか）
LiveView の基本は「サーバー側にステートフルなプロセス（各接続に対応）を持ち、状態が変わると HTML の差分を計算して WebSocket でブラウザへ送る」モデルです。ブラウザは受け取った差分を当てるだけなので、フロントに複雑なロジックが不要になります。ここでの弱点が「リストの状態管理」でした。従来はテンプレートに束ねる全項目をサーバーのメモリ（assigns）に保持する必要があり、件数に比例してメモリと差分計算コストが膨らみました。

Streams はこの前提を覆し、「サーバーは全項目を保持せず、DOM 上の項目集合への操作（insert/update/delete）だけを宣言的に送る」設計にしました。これにより、無限に増えるフィードでもサーバーメモリが一定に保てます。トレードオフは、サーバーが全リストを“知らない”ため、並べ替えや全件再計算のようにリスト全体を前提とする操作はやや扱いに工夫が要る点です。BEAM の軽量プロセスモデル（接続ごとに安価にプロセスを持てる）があるからこそ、このサーバー主導 UI が現実的に成立します。

### 実務での使いどころ・移行の勘所
- チャット、ライブ通知、ダッシュボード、コメント欄など「増え続けるリスト＋リアルタイム更新」の UI に最適。
- フロントに重い SPA を持たず、Elixir/Phoenix のサーバー技術に集中したいチームで生産性が高い。
- 既存 LiveView アプリでリスト系画面を Streams へ移行すると、メモリと差分計算の負荷を下げられる。全件操作の要否を設計時に見極める。

## 📖 用語解説（このトピックとのつながり）
- **Phoenix / LiveView**: Elixir の Web フレームワークと、そのサーバー主導リアルタイム UI 機能。本トピックの主役。
- **Streams API**: リストの追加・更新・削除を差分で扱い、全件保持を避ける仕組み。
- **assigns**: LiveView がサーバー側で保持するテンプレート用の状態。従来はここにリスト全体を保持していた。
- **WebSocket**: サーバーとブラウザの双方向通信路。差分配信の土台。
- **BEAM / 軽量プロセス**: 接続ごとに安価な並行プロセスを持てる VM 特性。サーバー主導 UI を支える。

## 影響範囲 / 推奨アクション
Elixir/Phoenix でリアルタイム UI を作るバックエンド開発者に直接の恩恵。フロントの JS 量を抑えつつ双方向 UI を作りたい場合の有力手段。増え続けるリストを扱う画面は Streams への移行で省メモリ化でき、まずは負荷の高いフィード系画面から適用するとよい。
