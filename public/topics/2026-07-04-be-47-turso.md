# Turso — SQLite ベースの分散エッジデータベース

> トピック: 開発トピック｜出典: devnewsletter｜種別: backend

## 一行サマリー
Turso は SQLite（および Rust による書き換え版）を土台に、エッジで低レイテンシに動く分散データベースを提供し、SQLite を「単一ファイルの組み込み DB」からクラウド規模へ拡張する。

## 🔰 初学者向け（何が・なぜ重要か）
SQLite は世界で最も広く使われるデータベースだが、伝統的に「1つのファイルにデータを収める、単体で動く小さな DB」だった。スマホアプリやブラウザに組み込まれているのがその典型だ。速くて手軽な反面、複数のサーバーで共有したり世界中に分散させたりするのは苦手だった。

Turso はこの SQLite を、世界中のエッジ拠点に複製して置き、ユーザーの近くから読み書きできる「分散データベース」に仕立てる。近所のコンビニに商品が置いてあれば遠くの倉庫まで取りに行かなくて済むように、データがユーザーの近くにあれば反応が速い。

さらに Turso は SQLite を Rust で書き直し、従来 SQLite が苦手だった「同時書き込み」を MVCC（複数バージョン管理）で改善しようとしている。手軽さはそのままに、本格的なアプリでも使える DB を目指しているのが要点だ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Turso は libSQL（SQLite フォーク）を基盤に、エッジ複製・組み込み replica・サーバーレス接続を提供。加えて SQLite を Rust で全面書き換えるプロジェクトを進め、同時書き込みのプレビュー（MVCC で ACID を維持）を打ち出している。エッジ拠点への読み取りレプリカ配置で読み取りレイテンシを下げる。具体的バージョン・GA 状況は要確認。

### 仕組み / 背景（なぜこう設計されたか）
従来 SQLite はライタが1つだけ（書き込みがファイル全体をロック）で、Web バックエンドの並行書き込みに不向きだった。Turso/libSQL は組み込み replica を各アプリインスタンスに置き、読み取りをローカル化しつつ書き込みをプライマリへ集約。Rust 書き換え版の MVCC は、読み取りと書き込みが互いをブロックしない設計を狙い、SQLite の弱点だった並行性を克服しようとしている。

### 実務での使いどころ・移行の勘所
読み取りが多くレイテンシに敏感なエッジアプリ、マルチテナントで「テナントごとに1 DB」を大量に持ちたい用途に向く。SQL 方言は SQLite 互換なので既存 SQLite 資産の移行が容易。ただし高い書き込み集中や複雑な分散トランザクションは Postgres 等が依然有利。同時書き込み機能はプレビュー段階のため本番採用は成熟度を確認。

## 📖 用語解説（このトピックとのつながり）
- **libSQL**: SQLite のオープンなフォーク。Turso の技術的基盤。
- **組み込み replica**: アプリ内にローカル複製を置く方式。読み取りのエッジ化を実現。
- **MVCC**: 複数バージョンで並行制御する方式。SQLite の同時書き込み改善の鍵。
- **マルチテナント（DB-per-tenant）**: テナント毎に DB を分ける設計。Turso の得意分野。

## 影響範囲 / 推奨アクション
エッジ / サーバーレスアプリ開発者が対象。読み取り主体・低レイテンシ要件なら Turso を評価し、書き込み並行性が高い用途は Postgres 系と比較検討するのが妥当。
