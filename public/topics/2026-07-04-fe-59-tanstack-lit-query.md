# TanStack Query が Lit 公式対応 ― @tanstack/lit-query でフレームワーク非依存が一歩前進

> トピック: 開発トピック｜出典: GitHub TanStack/query releases｜種別: frontend

## 一行サマリー
非同期状態管理の定番ライブラリ TanStack Query が、2026-06 に Web Components 向けの `@tanstack/lit-query` を追加した（0.2.0 で devtools やテスト設定も整備）。React/Vue/Svelte/Solid に続き Lit（Web Components）を公式サポートし、「フレームワークに依存しないデータ取得層」という TanStack の思想が一段進んだ。

## 🔰 初学者向け（何が・なぜ重要か）
アプリがサーバーからデータを取ってくるとき、「読み込み中」「エラー」「キャッシュ（再取得を減らす一時保存）」などを毎回自前で書くのは大変です。TanStack Query は、その面倒を肩代わりしてくれる“データ取得の執事”のようなライブラリです。

これまでこの執事は React や Vue などの主要フレームワークで雇えましたが、今回「Lit」という Web Components（ブラウザ標準の部品化技術）向けの雇用形態が新たに用意されました。つまり、特定のフレームワークに縛られず、標準寄りの作り方をしている人でも同じ執事を使えるようになった、ということです。同じ道具を色々な現場で使い回せると、学び直しが減って開発が楽になります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
- 2026-06 のリリースで **`@tanstack/lit-query`**（Lit / Web Components アダプタ）を初期追加。
- **0.2.0** で devtools のテストセットアップと TypeScript 設定の修正を実施（要確認、詳細は GitHub Releases 参照）。
- TanStack Query のコア（キャッシュ・再取得・無効化ロジック）はフレームワーク非依存で、各アダプタ（react-query / vue-query / svelte-query / solid-query / lit-query）が薄い橋渡しを担う構造。

### 仕組み / 背景（なぜこう設計されたか）
TanStack Query は「サーバー状態（server state）」を、クライアント状態とは別物として扱う設計です。サーバー状態は本質的に非同期で、いつ古くなるか分からない（stale）。そこでコアは、キー単位のキャッシュ、バックグラウンド再取得、`staleTime`/`gcTime` による鮮度管理、無効化（invalidation）といった普遍的ロジックを持ちます。この中核はフレームワークに依存しないため、各フレームワーク向けには「そのフレームワークのリアクティビティにコアを接続する薄いアダプタ」を用意するだけで済みます。

Lit 対応の意義は、この薄いアダプタ方式が Web Components という“標準寄りの土俵”にも広がったことです。Web Components は特定フレームワークに縛られず再利用できる部品化技術で、そこにデータ取得層の定番を持ち込めると、フレームワーク間の資産共有が進みます。トレードオフは、Lit アダプタが新しく（0.x 台）、成熟度・情報量が React 版に比べてまだ薄い点です。

### 実務での使いどころ・移行の勘所
- Lit / Web Components でアプリや共有部品を作るチームは、独自のデータ取得実装を `@tanstack/lit-query` に置き換えられる。
- マイクロフロントエンドや複数フレームワーク混在環境で、データ取得の作法を統一したい場合に有効。
- 0.x のため API 変更を織り込み、まずは非クリティカルな画面から導入するのが安全。

## 📖 用語解説（このトピックとのつながり）
- **TanStack Query**: 非同期・サーバー状態を管理する定番ライブラリ。本トピックの主役。
- **Lit**: 軽量に Web Components を書くための Google 製ライブラリ。今回の新アダプタ対象。
- **Web Components**: ブラウザ標準の部品化技術。フレームワーク非依存で再利用できる。
- **サーバー状態 (server state)**: サーバー由来で古くなりうる非同期データ。Query が専門に扱う。
- **アダプタ**: フレームワーク固有のリアクティビティにコアを接続する薄い橋渡し層。

## 影響範囲 / 推奨アクション
Lit / Web Components を採用するフロントエンド開発者に直接の恩恵。React 等の既存利用者への影響はないが、複数フレームワークをまたぐ組織ではデータ取得層を統一する好機。導入時は 0.x のバージョン動向を追い、devtools でキャッシュ挙動を確認しながら広げるとよい。
