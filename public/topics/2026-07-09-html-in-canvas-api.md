# HTML-in-Canvas API — Canvas の中に本物の HTML/CSS を描画、Chrome I/O 2026 で発表

> トピック: HTML-in-Canvas API｜出典: Chrome for Developers（Google I/O 2026）｜種別: 新ブラウザ API

## 一行サマリー

Chrome が Google I/O 2026 で発表した HTML-in-Canvas API は、HTML 要素と CSS スタイルを Canvas の描画コンテキストに高忠実度で合成できる新機能で、これまで Canvas 内でゼロから描くしかなかったリッチなテキスト・レイアウトを、実 DOM の表現力そのままで描けるようにする。

## 🔰 初学者向け（何が・なぜ重要か）

Web の画面には大きく2つの世界があります。ひとつは普段の HTML/CSS の世界（文章やボタンを「宣言」すると、ブラウザがきれいに配置してくれる）。もうひとつが **Canvas** の世界です。Canvas は「真っ白なキャンバスにプログラムで絵を描く」領域で、ゲームやグラフ、地図、画像編集などに使われます。自由度が高い代わりに、文字やレイアウトも全部プログラムで座標指定して描く必要があり、HTML のような楽さはありませんでした。

**HTML-in-Canvas API** は、この2つの世界の橋渡しです。Canvas の絵の中に、本物の HTML/CSS で作った部品（整形済みのテキストやカードなど）をそのまま貼り込めるようになります。たとえばゲーム画面（Canvas）の中に、CSS でデザインした UI パネルを美しく重ねる、といったことが簡単になります。「Canvas の自由さ」と「HTML/CSS の作りやすさ」の良いとこ取りができる、というのが嬉しさです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- Google I/O 2026 で Chrome が **HTML-in-Canvas API** を発表。HTML 要素と CSS スタイルを Canvas の描画へ高忠実度で合成し、多次元的（レイヤー的）な創作レイアウト・エフェクトを可能にする、と公式が説明。
- 同 I/O では、SPA 向けに Core Web Vitals 計測をもたらす **Soft Navigations API** の Chrome 近リリース投入も告知。「エージェント Web」を支える一連の機能群の一部として位置づけ。
- 正式な API 形状（メソッド名、対象要素の制限、セキュリティ制約など）は Chrome のドキュメントで要確認。実験フラグ/オリジントライアルの段階かどうかも要確認。

### 仕組み / 背景（なぜこう設計されたか）

Canvas は「イミディエイトモード（即時描画）」のグラフィックス API だ。`fillText` や `drawImage` で命令を出すと即座にピクセルが塗られ、要素という概念は残らない。これは高性能・高自由度な反面、テキスト整形（行分割、双方向テキスト、フォント合成）やレイアウト（Flexbox/Grid）といった、ブラウザが本来得意とする複雑処理を全部自前で再実装する羽目になる。ゲーム UI やデータ可視化のラベルなどで、開発者が「HTML なら一瞬なのに」と感じてきた根本原因がこれだ。

HTML-in-Canvas は、レンダリングエンジンが持つ「レイアウト＋ペイント」の成果物を、Canvas の合成対象として取り込む発想だ。DOM ノードをブラウザが通常どおりレイアウト・スタイル解決し、その描画結果を Canvas のフレームに合成する。これにより、テキスト整形や CSS の表現力を Canvas に「借りて」こられる。設計上の難所はセキュリティとタイミングだ。任意 HTML の合成は、クロスオリジン画像やフォントの読み取り（キャンバスの「汚染」）や、レイアウト計測を通じた情報漏えいの経路になりうるため、対象要素や取得できるピクセル情報に制約が課される可能性が高い（詳細は要確認）。また DOM のレイアウトは非同期・可変なので、フレーム単位の合成タイミングをどう保証するかが実装上の鍵になる。

### 実務での使いどころ・移行の勘所

データ可視化（グラフのリッチなラベル・凡例）、ゲーム/インタラクティブ体験の UI オーバーレイ、画像・動画編集ツールのテキストレイヤー、クリエイティブコーディングなどで有効。従来 Canvas 内でテキストを泥臭く描いていた箇所を、HTML/CSS 由来の高品質表現に置き換えられる。ただし発表直後の新 API であり、当面はオリジントライアルや実験フラグ前提の可能性が高いため、本番採用は対応ブラウザ・フォールバック（従来の Canvas テキスト描画や DOM オーバーレイ）を前提に設計したい。セキュリティ制約（キャンバス汚染、クロスオリジン素材）に触れる用途では、素材の同一オリジン化や CORS 設定を先に整理しておくのが安全。

## 📖 用語解説（このトピックとのつながり）

- **Canvas / 描画コンテキスト**: プログラムでピクセルを描く領域。本 API はここに HTML を合成する。
- **イミディエイトモード**: 命令を出すと即描画され要素が残らない方式。Canvas の自由さと不便さの源。
- **レイアウト＋ペイント**: ブラウザが DOM を配置し描画する工程。本 API はその成果を Canvas に取り込む。
- **キャンバス汚染（tainting）**: クロスオリジン素材の取り込みでピクセル読み取りが制限される安全機構。本 API の制約要因。
- **Soft Navigations API**: SPA の擬似遷移を計測可能にする API。同 I/O で発表された関連機能。

## 影響範囲 / 推奨アクション

- 影響: データ可視化・ゲーム・画像編集・クリエイティブ系のフロントエンド。Canvas UI の表現力に直結。
- 推奨: 対応状況（オリジントライアル/フラグ）を Chrome ドキュメントで確認し、実験導入。セキュリティ制約と非対応環境向けフォールバックを前提に設計。
