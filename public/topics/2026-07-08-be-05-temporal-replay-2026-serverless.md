# Temporal Replay 2026 — Serverless WorkersとWorkflow Streamsでエージェント時代の永続実行へ

> トピック: Temporal Replay 2026｜出典: The New Stack / temporal.io｜種別: 製品カンファレンス発表

## 一行サマリー

Durable Execution（永続実行）基盤の Temporal が Replay 2026 で **Serverless Workers**・**Standalone Activities**・**Workflow Streams** を発表し、Google ADK と OpenAI Agents SDK 連携を追加。長時間・失敗耐性が要る AI エージェントのオーケストレーションに永続実行を持ち込む方向を鮮明にした。

## 🔰 初学者向け（何が・なぜ重要か）

「永続実行（Durable Execution）」とは、途中でサーバが落ちても、プログラムが**中断した地点から正確に再開できる**仕組みです。普通のプログラムはプロセスが死ぬと進捗が消えますが、Temporal はワークフローの各ステップの状態を自動で記録し、障害後にその続きから走らせます。例えるなら、ゲームのオートセーブが全行に入っているようなもので、電源が落ちても失った進行がありません。

なぜ今これが重要かというと、AI エージェント（自律的に複数ステップの作業をこなす AI）が増えているからです。エージェントは「検索して→要約して→API を叩いて→結果を待って…」と長時間・多段階で動き、途中で失敗しがちです。ここに永続実行を敷くと、失敗しても最初からやり直さず、途中から復帰できます。Temporal は Replay 2026 で、まさにこのエージェント用途に舵を切りました。

新機能の「Serverless Workers」は、ワークフローを実行する側（Worker）を自前で常時立てなくても動かせるようにするもの。運用の重さを下げ、より手軽に永続実行を始められます。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

Replay 2026（Temporal の年次カンファレンス）での主な発表:

- **Serverless Workers**: Worker を自前運用せずに動かせるサーバレス実行オプション。
- **Standalone Activities**: ワークフローに包まず、Activity（副作用を持つ処理単位）を単体で durable に実行する仕組み。
- **Workflow Streams**: ワークフローからのストリーミング出力。長時間実行や AI 生成の逐次結果を扱う用途を想定。
- **AI 連携**: **Google ADK**（Agent Development Kit）および **OpenAI Agents SDK** との統合。
- 登壇: CEO Samar Abbas、CTO Maxim Fateev。

### 仕組み / 背景（なぜこう設計されたか）

Temporal の中核は **イベントソーシング + 決定論的リプレイ**です。ワークフローコードの実行は、外部との相互作用（Activity 呼び出し、タイマー、シグナル）をすべてイベント履歴として永続化します。障害後は、その履歴を「リプレイ」してワークフロー関数を再実行し、記録済みイベントに達したら実際の副作用を再実行せず履歴の結果を返す——これで「一度だけ・完了まで」実行される錯覚を作ります。だからワークフローコードは**決定論的**でなければなりません（乱数や時刻の直接使用は禁止で、Temporal API 経由で扱う）。

なぜエージェント指向なのか。AI エージェントの実行は、(1) 長時間（分〜時間）、(2) 外部 API・ツール呼び出しが多く失敗しやすい、(3) 途中結果を逐次返したい、という特徴を持ちます。これは永続実行が最も価値を出す領域です。**Standalone Activities** は、エージェントの「ツール呼び出し1回」を独立して durable にし、**Workflow Streams** はトークンや中間結果のストリーミングを扱えるようにします。**Serverless Workers** は、エージェントを立ち上げるたびに Worker フリートを用意する運用負担を下げます。

競合として **Restate**（HTTP ネイティブで、ワークフローを特別な SDK 付き HTTP ハンドラとして書く）や **DBOS**（DB 中心の軽量な永続実行）があります。「Temporal は個々のワークフローを durable にし、Restate はシステム全体を durable にする」という対比がよく語られます。Temporal は専用サーバ（Cluster）とイベント履歴を核に据える重厚な設計で、複雑・長期のオーケストレーションに強い一方、学習コストと運用が重いのがトレードオフでした。今回の Serverless 化はその運用の重さを緩和する動きです。

### 実務での使いどころ・移行の勘所

- **長時間・多段・失敗耐性が要るオーケストレーション**（決済フロー、注文処理、データパイプライン、そして AI エージェント）に最適。
- ワークフローコードは**決定論的制約**を守る必要がある（時刻・乱数・外部呼び出しは Temporal API 経由）。ここを外すとリプレイが壊れる。
- Serverless Workers は運用負担を下げるが、レイテンシ・コスト・コールドスタート特性は自前 Worker と異なるため、ワークロードで評価。
- 既存の軽量用途なら Restate/DBOS の方が導入が軽い場合がある。**「システム全体 durable」志向なら Restate、複雑な長期ワークフロー志向なら Temporal**、という住み分けで比較検討を。
- Google ADK / OpenAI Agents SDK を使うエージェント基盤に、失敗耐性の土台として組み込む価値が高い。

## 📖 用語解説（このトピックとのつながり）

- **Durable Execution（永続実行）**: 障害後も中断地点から再開できる実行モデル。Temporal の核。
- **決定論的リプレイ**: イベント履歴を再生してワークフローを復元する仕組み。決定論コードが前提。
- **Activity**: 副作用（API 呼び出し等）を担う処理単位。Standalone Activities で単体 durable 化。
- **Worker**: ワークフロー/Activity を実行するプロセス。Serverless Workers で自前運用不要に。
- **Workflow Streams**: ワークフローからの逐次ストリーミング出力。AI 生成に有用。
- **Restate / DBOS**: 永続実行の対抗製品。設計思想（HTTP ネイティブ / DB 中心）が異なる。

## 影響範囲 / 推奨アクション

- AI エージェント基盤を作るなら、失敗耐性の土台として Temporal の新機能群を評価。
- 既存 Temporal 利用者は Serverless Workers での運用簡素化とコスト影響を試算。
- ワークフローの**決定論制約**を再点検（時刻・乱数・非決定的 I/O の直呼び出しがないか）。
- 軽量用途は Restate/DBOS との比較で過剰投資を避ける。
- Workflow Streams でエージェントの中間結果をユーザーへ逐次返す UX を検討。
