# Vite 8 が tsconfig paths と emitDecoratorMetadata を内蔵 ― 設定ファイルが減る

> トピック: 開発トピック｜出典: vite.dev｜種別: frontend

## 一行サマリー
Vite 8 は、これまで追加プラグインや手動設定が必要だった **TypeScript の `paths` エイリアス解決**と **`emitDecoratorMetadata`（デコレータのメタデータ出力）**を標準サポートし、初期セットアップの手間を削減した（詳細挙動は要確認）。

## 🔰 初学者向け（何が・なぜ重要か）
プロジェクトでは、深い階層のファイルを `../../../utils/foo` のように書く代わりに、`@/utils/foo` という短い別名（エイリアス）を使いたくなります。これを実現するには、これまで TypeScript 側（`tsconfig.json`）とビルドツール側（Vite の設定）の**両方に同じ設定を二重に書く**必要があり、片方だけ直して壊す、という事故がよく起きていました。

Vite 8 では、この別名の設定を `tsconfig.json` の記述から自動で読み取ってくれます。つまり「同じことを二か所に書く」煩わしさが消えます。さらに、NestJS などが使う「デコレータ」という文法（`@Injectable()` のような目印）に必要な裏方の設定も内蔵され、追加プラグインなしで動くようになりました。設定ファイルが減ることは、初心者にとって「つまずきポイント」が減ることを意味します。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
- `tsconfig.json` の `compilerOptions.paths` を Vite が自動解決（従来は `vite-tsconfig-paths` 等のプラグインが必要）。
- `emitDecoratorMetadata` を内蔵サポート（従来は SWC/Babel プラグインで補っていた、要確認）。
- これらは Vite 8 の統合ツールチェーン（Oxc ベース）の恩恵として提供される（詳細は要確認）。

### 仕組み / 背景（なぜこう設計されたか）
従来、Vite の裏側は esbuild で、esbuild は設計思想上デコレータのメタデータ出力を完全サポートしていませんでした。そのため NestJS 系プロジェクトは別トランスパイラを噛ませる必要がありました。Vite 8 が Rust 製 **Oxc** をトランスフォーム基盤に採用したことで、TypeScript の型設定を直接理解し、デコレータのメタデータも扱えるようになった、という構図です。「設定を一元化する」設計は、ツール間の設定ドリフト（食い違い）を根絶する狙いがあります。

### 実務での使いどころ・移行の勘所
- パスエイリアスを多用するモノレポで、Vite 設定の重複記述を削れる。
- NestJS + Vite や、`class-validator`／`TypeORM` などデコレータ依存の資産を持つプロジェクトで恩恵大。
- 従来プラグインを外す際は、挙動差（特にメタデータ形式）を要確認。段階的に移行するのが安全。

## 📖 用語解説（このトピックとのつながり）
- **paths エイリアス**: 長い相対パスを短い別名にする TS の機能。二重設定の温床だった。
- **emitDecoratorMetadata**: デコレータの型情報を実行時に残す TS オプション。DI 系ライブラリが依存。
- **デコレータ**: `@Something()` の形でクラス/メソッドに機能を付与する構文。
- **Oxc**: Rust 製の JS/TS ツールチェーン。Vite 8 の変換基盤。
- **設定ドリフト**: 同じ内容を複数箇所に書いた結果、片方だけ古くなる現象。

## 影響範囲 / 推奨アクション
Vite + TypeScript 利用者、とくにパスエイリアスやデコレータを使うプロジェクトに好影響。移行時は既存の `vite-tsconfig-paths` 等の重複プラグインを外し、ビルド成果物の差分を確認すること。
