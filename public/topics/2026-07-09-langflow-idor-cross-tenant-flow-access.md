# AIエージェント基盤 Langflow に認証済みIDOR（CVE-2026-55255）— UUIDだけで他人のフローを乗っ取り、RCEへ連鎖

> トピック: 脆弱性ニュース ｜ 出典: BleepingComputer / The Hacker News / Sysdig TRT / GitHub Advisory ｜ 種別: 認可回避（IDOR / CWE-284）

## 一行サマリー

AIエージェント構築フレームワーク Langflow に、ログインさえしていればフローのUUIDを指定するだけで他ユーザーのフローを実行できる欠陥（CVE-2026-55255, CVSS 9.9）が見つかり、6月25日から実際の悪用が確認された。攻撃者はこれを足がかりに別のRCE欠陥へ連鎖し、LLMやAWSの鍵を盗んでいる。1.9.2への更新が急務。

## 🔰 初学者向け（何が・なぜ問題か）

Langflow は、AIエージェント（自動で仕事をこなすAIの仕組み）を、部品をつなぐ図（フロー）としてマウス操作で組み立てられるツールです。複数のユーザーが同じLangflowサーバをSaaSのように共有して使うこともあります。

今回の問題は、その共有サーバで「自分のフローだけが見えるはず」という仕切りが壊れていたことです。攻撃者は、まず何らかの一般ユーザーとしてログインします（正規登録でも構いません）。そのうえで、他人のフローに割り振られた**整理番号（UUID）**をリクエストに書いて送ると、サーバは「これはあなたのものですか？」を確認せずに、その他人のフローを動かしてしまいます。

たとえるなら、コインロッカーで「ロッカー番号さえ言えば、鍵の持ち主か確認せず開けてくれる管理人」のようなものです。番号は総当たりや推測で見つけられるため、他人の作業や、その中に保存された秘密（AIサービスのAPIキーなど）を勝手に取り出せてしまいます。

さらに悪いことに、攻撃者はこの「他人のフローを動かせる」入り口から、別の弱点（コードを直接実行させる欠陥）へと乗り継ぎ、サーバ上で好きなプログラムを走らせ、クラウドの鍵まで盗み出していました。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **脆弱性種別**: IDOR（Insecure Direct Object Reference）= 直接オブジェクト参照の不備。CISA KEV では「User-Controlled Key による認可回避（CWE-284 / CWE-639 相当）」と分類。
- **CVSS**: 9.9（Critical）で報告。
- **前提条件**: 攻撃者が**任意の認証済みアカウント**を保有していること（自己登録可能な構成や、テナントを共有するSaaS型で特に危険）。対象エンドポイントは `/api/v1/responses`。
- **影響バージョン**: 1.9.2 より前のすべてのバージョン。
- **修正版**: Langflow 1.9.2（`/api/v1/responses` に所有者・認可チェックを追加）。
- **悪用状況**: Sysdig Threat Research Team が2026年6月25日に初の実環境悪用を観測。CISA が7月7日にKEV登録し連邦機関に優先パッチを指示。攻撃者は本IDOR→別RCE（CVE-2026-33017）へ連鎖し、LLMプロバイダ鍵・AWS鍵を窃取。

### メカニズム（攻撃が成立する仕組み）

以下は「型（クラス）の説明」であり、武器化できる手順・PoC・正確なペイロードは示さない概念レベルの解説である。

1. **信頼境界**: 「認証済みユーザー」→「そのユーザーが所有するリソースのみ操作可」というテナント分離が本来の境界。IDORはこの"認証の後段にある認可"を破る欠陥である。認証（誰であるか）は通っていても、認可（それを操作してよいか）が抜けている。
2. **不備のあるコンポーネント**: `/api/v1/responses` エンドポイントがフローをUUID（`flow_id`）で解決する際、DBへの問い合わせに**呼び出し元ユーザーの所有権（user_id）チェックが無い**。結果、UUIDさえ知っていれば誰のフローでも解決・実行できる。
3. **攻撃者が握るプリミティブ**: 「任意の他ユーザーのフローを列挙・実行できる」という横断的アクセス（cross-tenant access）。フローには接続先の認証情報（APIキー等）が組み込まれることが多く、実行＝それらの秘密を使わせられる。
4. **橋渡し（RCEへの連鎖）**: 実際の攻撃では、偵察（アプリ/認証の探索）→フロー列挙→本IDORで他人フローへ到達→別の公開ビルドエンドポイントの欠陥（CVE-2026-33017）を用いた任意コード実行、という段階を踏んだ。IDORは"入口"を、RCEは"最終的な実行"を担う。
5. **最終被害**: LLMプロバイダ鍵・AWS鍵の窃取、サーバ上での外向き通信の反復（外部C2への接続試行）に至る。鍵の悪用はクラウド課金の乗っ取りや二次侵入につながる。
- **一般原理**: IDORは「認証は実装したが、リソース単位の認可（オブジェクトレベル認可）を実装し忘れる」典型例。CVSSが最高でなくとも、自己登録可能・マルチテナント・秘密情報同梱という条件が重なると"最も悪用されやすい"部類になる。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: `/api/v1/responses` への、呼び出し元と所有者が一致しないフロー実行。短時間に多数の異なる `flow_id`（UUID）へアクセスする列挙的な挙動。フロー実行に伴う想定外の外向き通信、認証情報（APIキー）の異常利用。
- **暫定緩和（パッチ前）**: Langflow をインターネットから隔離し、信頼できるネットワーク/VPN経由に限定。自己登録（オープンサインアップ）を無効化。テナント間で秘密情報を共有しない構成に見直す。
- **恒久対策**: 1.9.2 以降へ更新。更新後は、露出期間に使われた可能性のあるLLM/クラウド鍵を**全ローテーション**し、不正なフロー・アカウント・外向き通信を点検。
- **運用示唆**: AI基盤は"秘密情報の集積地"であり、認可の抜けが即座に鍵漏洩に直結する。オブジェクトレベル認可（所有権チェック）をAPI設計の必須項目とし、鍵は基盤に埋め込まず短命トークン/ボールト経由にする。

## 📖 用語解説（このニュースとのつながり）

- **IDOR（直接オブジェクト参照の不備）**: 識別子（ここではUUID）を差し替えるだけで他人の資源にアクセスできる欠陥。本件の中核。認証は通っていても認可が抜けている点が肝。
- **認可（Authorization）と認証（Authentication）**: 認証は「誰か」を確かめる工程、認可は「それをして良いか」を確かめる工程。本件は認証OK・認可NGの状態が悪用された。
- **マルチテナント / クロステナントアクセス**: 複数利用者が1つの基盤を論理的に区切って共有する形態。区切りが破れると他テナントのデータに横断アクセスできる（本件の被害拡大要因）。
- **UUID（フローID）**: リソースに割り振る識別子。推測困難とされがちだが、列挙・漏えいで入手されると"番号を知る＝アクセスできる"設計は破綻する。
- **KEV（Known Exploited Vulnerabilities）**: CISAが「実際に悪用が確認された」脆弱性を掲載する目録。掲載＝優先対応の強い根拠。

## 影響範囲 / 推奨対応

- **影響範囲**: 1.9.2 未満の Langflow 全般。特に自己登録可能・マルチテナント・インターネット露出の構成は即時リスク。AIエージェント開発でLLM/クラウド鍵を保持している環境は被害が大きい。
- **推奨対応**: (1) 直ちに 1.9.2 以降へ更新、(2) 更新前はインターネット遮断・自己登録無効化、(3) 6/25以降のフロー実行ログを遡って横断アクセスの痕跡を調査、(4) LLM/AWS等の鍵を全ローテーション、(5) 連鎖悪用（CVE-2026-33017）を想定しRCE痕跡・外向き通信も点検。
