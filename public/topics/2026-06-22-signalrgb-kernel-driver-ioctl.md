# SignalRGB カーネルドライバに不適切なアクセス制御とIOCTLの脆弱性（CVE-2026-8049 / CVE-2026-8050）

> トピック: 脆弱性ニュース ｜ 出典: JVNVU#95564871 / CERT/CC VU#380058 ｜ 種別: 不適切なアクセス制御（CWE-284）／ NULLポインタ参照（CWE-476）

## 一行サマリー
RGBライティング制御ソフト SignalRGB のカーネルドライバ `SignalIo.sys` に、誰でも特権IOCTLを発行できる不適切なアクセス制御（CVE-2026-8049）と、空入力でカーネルをクラッシュさせるNULLポインタ参照（CVE-2026-8050）が存在。前者はローカル権限昇格やBYOVD攻撃の足場になり得る。ドライバ 1.3.7.0 で修正。

## 🔰 初学者向け（何が・なぜ問題か）
**SignalRGB** は、PCのキーボードやファンなどの"光り方（RGBライティング）"をまとめて操作する人気ソフトです。こうしたハードを細かく制御するため、Windows の最深部=**カーネル**で動く小さな部品「**ドライバ**」（`SignalIo.sys`）を入れます。カーネルは家でいう"分電盤と元栓が集まる地下室"で、ここを触れる権限は本来ごく限られた人だけに渡すべきものです。

ところが今回のドライバは、その地下室のドアに**鍵をかけ忘れていました**。本来は SignalRGB 本体だけが使える"特権操作の窓口（IOCTL）"を、PCにログインしている普通のユーザー（あるいはそのPCで動く一般プログラム）なら誰でも開けて使えてしまいます（CVE-2026-8049）。窓口の先には「メモリの読み書き」のような強力な操作が並んでいるため、悪用されると一般ユーザーが管理者（カーネル＝最高権限）に成り上がる足場になります。

もう一つ（CVE-2026-8050）は、窓口に"中身が空っぽの依頼書"を出すと係員が混乱して倒れる＝**PCがブルースクリーンで落ちる**という不具合です。こちらは乗っ取りより"妨害（クラッシュ）"の問題です。

さらに怖いのは **BYOVD（自前で持ち込む脆弱なドライバ）** という手口。攻撃者がこの"鍵のかかっていない正規ドライバ"を別のPCに持ち込んでインストールし、それを踏み台にセキュリティ対策を無力化する――という悪用が現実に広がっています。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）
- **CVE**: CVE-2026-8049（不適切なアクセス制御 / CWE-284）、CVE-2026-8050（NULLポインタ参照 / CWE-476）
- **対象**: SignalRGB カーネルドライバ `SignalIo.sys`。CVE-2026-8050 は SignalRGB 1.3.7.0 より前。
- **CVSS**: 要確認（公開アドバイザリ参照）。影響の質はCVE-2026-8049が高（LPE/BYOVD）、CVE-2026-8050は可用性（DoS）。
- **悪用状況**: 個別の実環境悪用報告は要確認。ただしBYOVD（脆弱な正規ドライバの持ち込み）という攻撃クラス自体は活発で、同種の署名済みドライバが武器化される事例が継続。CERT/CCは「複数のサードパーティ製カーネルドライバがIOCTLの不適切なアクセス制御に脆弱」と横断的に注意喚起（JVNTA#90371415）。
- **修正版**: ドライバ revision 1.3.6 で第一弾、1.3.7.0 で両件を修正。
- **前提条件**: CVE-2026-8049 はローカルの認証ユーザー（そのマシンでコード実行できること）。リモート単独では成立しない。

### メカニズム（攻撃が成立する仕組み）
1. **デバイスオブジェクトの保護不足（CVE-2026-8049の根本）**: ドライバが作成する `\\.\SignalIo` デバイスオブジェクトに、明示的なSDDL（セキュリティ記述子）も `FILE_DEVICE_SECURE_OPEN` フラグも設定されていなかった。結果として既定のアクセス制御が過度に緩く、**任意の認証ローカルユーザーがハンドルを取得**できる。
2. **特権IOCTLの開放（握るプリミティブ）**: ハンドルを得たプロセスは、本来カーネル特権を要する操作（実装によってはメモリ操作等）を行うIOCTLを発行できる。すなわち「カーネル空間での特権操作の呼び出し」というプリミティブを一般ユーザーが握る。
3. **LPE/BYOVDへの橋渡し**: カーネル特権の操作プリミティブは、ローカル権限昇格（一般ユーザー→SYSTEM/カーネル）や、セキュリティ製品の無効化（カーネルからの改ざん）に橋渡しされる。署名済みの正規ドライバであるため、攻撃者が別環境へ**持ち込んで悪用（BYOVD）**しても署名検証を通過しやすい。
4. **修正の考え方**: 1.3.6では `IRP_MJ_CREATE` 時に二段階の呼出元検証ゲートを導入――(1)呼出プロセスのイメージ名が許可リスト（`SignalRgb.exe` / `SignalRgbService.exe`）と一致するか、(2)EV証明書でAuthenticode署名され、そのSHA-1サムプリントがドライバに埋め込んだ値と一致するか。いずれか失敗で I/O 実行前に `STATUS_ACCESS_DENIED` を返す。
5. **CVE-2026-8050（NULL参照DoS）**: 複数のIOCTLハンドラが入力バッファの検証を行わず、**空の入力バッファ**で呼ばれた際にNULLポインタを参照。カーネルクラッシュ（BSoD）＝可用性喪失に至る。信頼境界（ユーザー→カーネルのIOCTL入力）での入力検証欠落が原因。

> ※ 一般原理: カーネルドライバの脆弱性クラスは「(a) デバイスオブジェクトのアクセス制御不備で誰でもIOCTLを叩ける」「(b) IOCTL入力（バッファ長/ポインタ）の検証不足」が典型。前者はBYOVDの主因、後者はDoS/メモリ破壊の主因。武器化手順・具体IOCTLコードは記載しない（概念レベル）。

### 検知・暫定緩和・恒久対策
- **恒久対策**: SignalRGB ドライバを **1.3.7.0 以降**へ更新。SignalRGB 本体を最新化。
- **暫定緩和**: 不要なら SignalRGB/ドライバをアンインストール。Microsoft の**脆弱ドライバブロックリスト（WDAC / Vulnerable Driver Blocklist）**を有効化し、既知の脆弱ドライバのロードを抑止。管理者権限の付与とドライバ導入を統制。
- **検知の着眼点**: 一般ユーザープロセスによる `\\.\SignalIo` デバイスへのハンドルオープン、想定外プロセスからのIOCTL発行、`SignalIo.sys` の旧版ロード、相次ぐカーネルクラッシュ（CVE-2026-8050のDoS痕跡）。EDRのドライバロード/IOCTL監視、Sysmon等でドライバ署名・ハッシュを照合。
- **運用示唆**: BYOVD対策として「署名されていても脆弱な正規ドライバ」をブロックリスト管理する運用が重要。エンドポイントのドライバインベントリ化を推奨。

## 📖 用語解説（このニュースとのつながり）
- **カーネルドライバ / IOCTL**: OS最深部で動く部品と、その部品にユーザー空間から命令を送る窓口。窓口の鍵（アクセス制御）が甘いと特権操作を悪用される。
- **不適切なアクセス制御（CWE-284）**: 本来制限すべき操作を誰でも実行できてしまう欠陥。本件はデバイスオブジェクトの保護設定漏れ。
- **FILE_DEVICE_SECURE_OPEN / SDDL**: デバイスへのアクセス権を厳格化する仕組み。設定漏れが今回の根本原因。
- **BYOVD（Bring Your Own Vulnerable Driver）**: 攻撃者が脆弱な"正規署名済み"ドライバを持ち込み、カーネル特権やセキュリティ無効化に悪用する手口。本件のような署名ドライバが標的になりやすい。
- **NULLポインタ参照（CWE-476）**: 未初期化/空入力で無効アドレスを参照し、カーネルではクラッシュ（DoS）を招く。

## 影響範囲 / 推奨対応
SignalRGB を導入した Windows 端末（ゲーミングPC/自作PC環境等）が対象。**ドライバ 1.3.7.0 以降への更新**、不要環境ではアンインストール、組織では**脆弱ドライバブロックリスト**の有効化が要。BYOVD観点で、署名済みでも脆弱な既知ドライバの持ち込みを統制する。
