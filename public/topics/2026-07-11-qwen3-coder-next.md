# Qwen3-Coder-Next — 80B総/3B活性、ローカルで動く最安級コーディングエージェント

> トピック: Qwen3-Coder-Next（小型ハイブリッド MoE コーディングモデル）｜ 出典: Qwen 公式ブログ / Hugging Face ｜ 種別: 新モデルリリース

## 一行サマリー

Alibaba の Qwen チームが公開した Qwen3-Coder-Next は、80B 総パラメータのうち活性はわずか 3B というハイブリッド MoE。256K（YaRN で 1M まで）文脈、358 言語対応、Apache 2.0 で、SWE-bench Verified 70.6% を達成しつつ、24GB GPU＋システムRAM でローカル実行でき、API も $0.11/Mトークンと"最安級の実用コーディングエージェント"を実現した。

## 🔰 初学者向け（何が・なぜ重要か）

「AI にコードを書かせる」用途では、賢さと同じくらい「安く・手元で動くか」が重要になる。四六時中コードを生成させると API 料金がかさむし、社外にコードを送りたくない企業も多いからだ。

**Qwen3-Coder-Next は、この"安く・手元で"を極めたコーディング特化モデルだ**。仕掛けは MoE（混合エキスパート）——全体では 800億パラメータの大きな脳を持ちつつ、1回の処理で実際に働くのはたった 30億分だけ。だから賢さは大型並みなのに、動かす負荷は小型並み。結果、ゲーミング PC クラスの 24GB GPU にシステムメモリを足す程度で動き、API 経由でも 100万トークンあたり約 $0.11 という破格の安さになる。

実力の指標 **SWE-bench Verified 70.6%** は「実際の GitHub のバグ修正を7割方こなせる」水準で、この価格・サイズ帯としては非常に高い。しかも **Apache 2.0** という自由なライセンスで、商用でも改造でも使える。「安い・速い・自由・そこそこ賢い」が揃うため、コーディングエージェントを自前で回したい個人・企業にとって現実的な選択肢になった。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **公開**: 2026-02-04、Qwen チーム。**Apache 2.0** ライセンス、Hugging Face `Qwen/Qwen3-Coder-Next`。
- **アーキテクチャ**: 80B 総 / **3B 活性**の MoE。ベースは Qwen3-Next-80B-A3B-Base（ハイブリッドアテンション＋MoE の新アーキテクチャ）。10〜20倍の活性パラメータを持つモデルに匹敵する性能を主張。
- **文脈長**: 256K ネイティブ（YaRN で 1M まで拡張可）。358 のプログラミング言語をカバー。
- **学習**: 80万件の実行可能タスクによる環境相互作用＋強化学習で"エージェント的"に大規模学習。多様なスキャフォールド（CLI/IDE テンプレート）に適応。
- **性能**: SWE-Agent スキャフォールドで **SWE-bench Verified 70.6%**。
- **コスト/ハードウェア**: API $0.11/M 入力トークンで"最安級の実用コーディングエージェント"。ローカルでは量子化して単一 24GB GPU＋システムRAM オフロードで動作（46GB RAM 級での実行報告あり）。
- ※ ベンチ・コストは提供元／各記事ベース、独立検証状況は「要確認」。

### 仕組み / 背景（なぜこう設計されたか）

**超スパース MoE の狙い**。3B 活性というのは MoE の中でも極端にスパース（活性率が低い）。ルータが 80B の中から入力ごとにごく一部のエキスパートだけを起こすことで、推論を 3B 級のコストに抑える。コーディングは「特定言語・特定パターンの専門知識を局所的に呼び出す」タスクが多く、専門家を選んで使う MoE と相性がよい。代償は、全 80B をメモリに載せる必要がある点——だからローカルでは量子化＋RAM オフロード（一部を CPU 側メモリに置く）が前提になる。

**ハイブリッドアテンション**。ベースの Qwen3-Next は、計算の重いフルアテンションと軽量な線形/スパース系アテンションを組み合わせ、長文脈でのコストを抑える設計。256K〜1M の長文脈を扱いつつ推論を実用速度に保つための工夫だ。

**エージェント学習**。近年のコーディングモデルは、静的なコード補完から「ツールを使い、テストを実行し、失敗から直す」エージェント運用へ軸足が移った。Qwen3-Coder-Next は 80万件の"実行可能タスク"で環境と相互作用しながら強化学習しており、SWE-bench のような「リポジトリを直す」タスクに直接効く設計。トレードオフとして、スキャフォールド（実行の足場）に性能が左右されやすく、使う CLI/IDE 連携の作り込みが結果を左右する。

### 実務での使いどころ・移行の勘所

社内コードを外に出せない環境でのローカルコーディングエージェント、大量呼び出しでコストを抑えたい CI/自動修正、個人開発の常時アシスタントに向く。24GB GPU クラスで動くため、フロンティア API を常用するより桁違いに安く回せる。移行の勘所は 2 つ——(1) 80B をメモリに載せる必要があるので、量子化とオフロード構成の検証が要る、(2) 性能はスキャフォールド依存なので、SWE-Agent 等の実行足場やツール連携を整えると本領を発揮する。最難関タスクではフロンティアモデルに一歩譲る場面もあり、"難所は大型、日常は Next"のような使い分けが現実的。

## 📖 用語解説（このトピックとのつながり）

- **MoE（混合エキスパート）/ 活性パラメータ**: 一部エキスパートだけを起こす構造と、実際に使うパラメータ数。3B 活性が低コストの鍵。
- **ハイブリッドアテンション**: フルと線形/スパースの注意を組み合わせ長文脈コストを抑える方式。
- **YaRN**: 学習後に文脈長を延長する手法。256K を 1M へ拡張。
- **SWE-bench Verified**: 実 GitHub 課題の解決率を測るベンチ。70.6% はこの価格帯で高い。
- **スキャフォールド（scaffold）**: エージェントがツール実行・反復修正を行う実行の足場。結果を左右。
- **量子化 / オフロード**: 重みを低ビット化し、一部を CPU メモリに逃がしてローカル実行を可能にする技術。
- **Apache 2.0**: 商用・改変に寛容なライセンス。企業採用しやすい。

## 影響範囲 / 推奨アクション

- ローカル・低コストのコーディングエージェントを求める個人／企業に有力な選択肢。
- ローカル運用は量子化＋RAM オフロード構成を検証（80B をメモリに保持）。
- 性能はスキャフォールド依存——SWE-Agent 等の足場・ツール連携を整備。最難関は大型モデルと使い分け。
