# Hugging Face TGI が保守モード→アーカイブ — 推論エンジンは vLLM / SGLang / llama.cpp に収斂

> トピック: TGI 保守モードと推論エンジンの収斂｜出典: Hugging Face（GitHub / docs）｜種別: プロジェクト方針転換

## 一行サマリー

Hugging Face の LLM 推論サーバー TGI（Text Generation Inference）は 2025-12 に保守モード入りし 2026-03-21 にリポジトリがアーカイブされ、公式ドキュメントも下流の vLLM・SGLang（およびローカルの llama.cpp・MLX）を推奨、LLM サービングは少数の主要エンジンへ実質的に収斂した。

## 🔰 初学者向け（何が・なぜ重要か）

学習済みの大規模言語モデルを「実際にサーバーで動かして応答を返す」部分を推論（inference）と呼び、その専用サーバーがいくつかあります。Hugging Face（AIモデルの巨大な配布拠点）が作っていた TGI は、その定番の一つでした。

そのTGIが「もう積極的に開発しません（保守モード）」となり、リポジトリも凍結（アーカイブ）されました。Hugging Face 自身が「これからは vLLM や SGLang を使ってください」と案内しています。

これは「どの推論エンジンを選ぶべきか迷う時代」が終わりに近づいた、という合図です。選択肢が乱立していた状況から、本番は vLLM を軸に、共有文脈が多い用途は SGLang、手元で動かすなら llama.cpp や（Apple 向け）MLX、という具合に役割分担が明確になりました。新しくシステムを組む人は、この収斂を前提に選べば大きく外しません。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **TGI の位置づけ変更**: 2025-12-11 頃に保守モード（軽微なバグ修正・ドキュメント改善のみ受付）。2026-03-21 にリポジトリをアーカイブ。
- **公式の推奨移行先**: Inference Endpoints でも vLLM / SGLang を推奨。ローカルは llama.cpp / MLX。既存 TGI エンドポイントはエンジン変更不可のため、移行は新規エンドポイント作成で対応。
- **エコシステムの流れ**: TGI が牽引していた「最適化推論エンジンが transformers に依存する」動きを引き継ぐ形で、コミュニティは vLLM/SGLang へ集約。
- **主要エンジンの特性**（参考）: vLLM は PagedAttention でメモリ効率が高く本番既定。SGLang は RadixAttention で共有文脈（チャット/RAG/エージェント）に強くスループット優位。llama.cpp/MLX はローカル/オンデバイス。
- 日付・推奨は Hugging Face の公式記述に基づく。

### 仕組み / 背景（なぜこう設計されたか）

推論エンジンの価値は「同じGPUでいかに多くのリクエストを、低レイテンシで捌くか」に集約される。ここでの技術的な鍵は KV キャッシュ（生成済みトークンの中間状態）の管理だ。素朴な実装は各リクエストに固定長の連続メモリを確保し、大量の未使用スロットを無駄にする。vLLM の PagedAttention は、OS の仮想メモリのページングのように KV キャッシュを小さなブロックに分割・非連続配置し、無駄を大幅に削って同時実行数を上げる。SGLang の RadixAttention は、複数リクエストが共有する接頭辞（システムプロンプトや会話履歴）の計算・キャッシュを木構造（radix tree）で再利用し、再計算を避ける。共有文脈が多いチャットやRAG、エージェントで効く。

TGI 撤退の背景は、こうした最適化競争でコミュニティの労力が分散することの非効率だ。同種の高難度最適化（連続バッチング、量子化カーネル、分散サービング）を複数プロジェクトが並行実装するより、少数のエンジンに集約する方が進化が速い。Hugging Face が自前サーバーを畳んで下流エンジンを推奨に回したのは、この「選択と集中」の判断といえる。トレードオフは、寡占化により特定エンジンの設計思想やハードウェア対応（NVIDIA 中心か、AMD/TPU/Gaudi 対応か）に生態系全体が引きずられる点だ。

### 実務での使いどころ・移行の勘所

新規のセルフホスト推論は vLLM を第一候補にし、システムプロンプトや会話履歴の共有が多いチャット/RAG/エージェント基盤なら SGLang を比較検討する。ローカル開発・オンデバイスや GGUF 資産があるなら llama.cpp、Apple Silicon は MLX。既存 TGI 運用は「保守モード＝将来のモデル/機能追従が止まる」前提で、移行計画を前倒しに。TGI エンドポイントはエンジン変更ができないため、新エンドポイントを立てて切り替える。移行時はトークナイザ差異、量子化形式（AWQ/GPTQ/FP8 等）の対応、API 互換（OpenAI 互換エンドポイントの有無）を確認する。

## 📖 用語解説（このトピックとのつながり）

- **推論（inference）**: 学習済みモデルで応答を生成する処理。TGI はその専用サーバーだった。
- **KV キャッシュ**: 生成済みトークンの中間状態。効率が同時実行数を左右する。
- **PagedAttention**: vLLM の KV キャッシュ最適化。メモリ無駄を削る。
- **RadixAttention**: SGLang の共有文脈再利用。チャット/RAG で有利。
- **保守モード / アーカイブ**: 積極開発の停止と凍結。TGI が該当。

## 影響範囲 / 推奨アクション

- 対象: LLM を自前運用する AI 基盤チーム。
- 推奨: 新規は vLLM 既定、共有文脈多用は SGLang、ローカルは llama.cpp/MLX。既存 TGI は移行を前倒し。
- 注意: TGI はエンジン変更不可のため新エンドポイントで移行。量子化形式・API 互換を確認。
