# OpenTelemetry GenAI セマンティック規約：AI エージェント可観測性の事実上標準になりつつある

> トピック: OpenTelemetry GenAI Semantic Conventions｜出典: Uptrace / OpenTelemetry / 各ベンダー｜種別: 可観測性標準

## 一行サマリー

OpenTelemetry の GenAI セマンティック規約が、LLM 呼び出し・エージェントの推論ステップ・ツール実行・トークン使用量・コストを標準スパンとして捕捉する共通言語になりつつある。Datadog・New Relic・Dynatrace がネイティブ対応し、OTel 計装済みコードは SDK 変更なしで各バックエンドに送れる。ただし多くは依然 experimental で、出力評価・安全性スコアはカバー外。

## 🔰 初学者向け（何が・なぜ重要か）

AI エージェントが本番で「なぜかおかしな動きをする」とき、原因を突き止めるのは難しい。エージェントは「モデルに質問→ツールを呼ぶ→結果を見て再び考える」を何度も繰り返すので、どのステップで何が起きたのかを追う必要がある。これを支えるのが「可観測性（observability）」だ。

従来のソフトウェアでは OpenTelemetry（OTel）という業界標準がこの役割を担ってきた。処理を「スパン（span＝ある操作の記録）」の連なり（トレース）として記録し、どこが遅い/失敗したかを可視化する。問題は、これまで OTel には「LLM 呼び出しやエージェントのツール使用をどう記録するか」の共通ルールがなかったこと。各ツールが独自形式で記録していたので、後から別のツールに乗り換えると計装をやり直す羽目になっていた。

GenAI セマンティック規約は、この「AI 特有の記録形式」を標準化するものだ。「モデル名はこの属性名で」「トークン使用量はこのメトリクスで」と共通の型を決めることで、一度計装すればどのバックエンド（監視サービス）でも読める。ベンダーロックインを避けられるのが最大の利点だ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **規約の対象範囲**: v1.41 時点で agent / workflow / tool / model の各スパン、レイテンシとトークン使用量の必須メトリクスを定義。カバー領域は大きく4つ ―― LLM クライアントスパン、エージェントスパン、イベント（プロンプト/補完内容の捕捉）、メトリクス。
- **エージェント向け属性**: LLM 呼び出し・ツール呼び出し・エージェント推論ステップ・トークン使用量・コストを属性として捕捉し、互換バックエンドへエクスポート。
- **ベンダー対応**: Datadog・New Relic・Dynatrace が GenAI 規約をネイティブサポート。OTel 計装済みコードは SDK 変更なしでこれらに送信可能。
- **ステータス**: 2026年3月時点で多くの GenAI 規約は experimental（API 未安定）。
- **限界**: 規約はモデル属性・トークン使用量・レイテンシを標準化するが、出力評価・安全性スコア・コンテンツ品質評価はカバーしない。

### 仕組み / 背景（なぜこう設計されたか）

セマンティック規約とは要するに「属性名とその意味の辞書」だ。たとえば「使ったモデル名」を `gen_ai.request.model` という決まった属性名で記録する、と全員が合意すれば、後段のダッシュボードやアラートは属性名だけ見れば意味が分かる。ここが揃わないと、Datadog 用と New Relic 用で別々の計装が要り、移行のたびに書き直しになる。規約はこの断片化を解消するために存在する。

エージェント特有の難しさは「スパンのネスト構造」にある。1つのエージェント実行の中に、複数のモデル呼び出しと複数のツール実行が入れ子で走る。規約は agent スパンの下に model スパンや tool スパンをぶら下げる階層を定義し、「エージェントがどの順で何を呼び、どこで時間を食ったか」を1本のトレースで俯瞰できるようにする。MCP のツール呼び出しもこのツールスパンとして記録される。

重要な設計上の割り切りは、規約が「テレメトリの形」だけを決め、「良し悪しの判断」はしない点だ。トークン数やレイテンシは客観的に測れるが、「出力が正しいか」「安全か」はドメイン依存で自動判定できない。だから規約は評価・安全性スコアを意図的にスコープ外に置き、そこは別の評価ツール（LLM-as-judge、ガードレール等）に委ねる。

### 実務での使いどころ・移行の勘所

エージェントを本番投入するなら OTel GenAI 規約での計装は早めに入れておくべき。ベンダー中立なので、監視 SaaS の乗り換え自由度を確保できる。既に Datadog/New Relic/Dynatrace を使っているなら、ネイティブ対応済みなので導入障壁は低い。ただし規約が experimental な属性を含む点は注意 ―― 属性名が将来変わりうるので、ダッシュボードやアラートが規約バージョンに依存しすぎないよう設計する。出力品質・安全性の評価は規約の外なので、評価パイプライン（オフライン eval、オンラインガードレール）は別途用意が必要。

## 📖 用語解説（このトピックとのつながり）

- **OpenTelemetry (OTel)**: 可観測性データ（トレース・メトリクス・ログ）の収集標準。ベンダー中立。
- **セマンティック規約 (semantic conventions)**: 属性名とその意味を統一する辞書。断片化を防ぐ核。
- **スパン / トレース**: 個々の操作の記録(スパン)と、その連なり(トレース)。エージェントは入れ子構造になる。
- **experimental ステータス**: API が未安定で将来変更されうる段階。GenAI 規約の多くが現状これ。
- **LLM-as-judge**: 出力品質を別の LLM で評価する手法。規約がカバーしない評価領域を補う。
- **トークン使用量メトリクス**: 入出力トークン数の計測。コスト監視の基礎で規約の必須メトリクス。

## 影響範囲 / 推奨アクション

- 本番エージェントには OTel GenAI 規約での計装を早期導入し、監視 SaaS のロックインを回避する。
- Datadog/New Relic/Dynatrace 利用中ならネイティブ対応済み。SDK 変更なしで送信可能か確認。
- 規約は experimental を含むため、ダッシュボード/アラートを規約バージョンに強く依存させない設計に。
- 出力品質・安全性評価は規約外。評価パイプライン（eval・ガードレール）を別途整備する。
