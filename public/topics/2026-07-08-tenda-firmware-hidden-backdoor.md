# Tenda ルーターに隠し認証バックドア（CVE-2026-11405）— パッチ無し、家庭・SOHOに影響

> トピック: 脆弱性ニュース ｜ 出典: CERT/CC (VU#213560) / The Hacker News / BleepingComputer ｜ 種別: 認証バックドア（意図的隠し機能）

## 一行サマリー

Tenda 製ルーターの複数機種のファームウェアに、正規パスワード認証を回避して管理者権限を得られる**未文書の隠し認証経路（バックドア）**が発見された（CVE-2026-11405）。ベンダーと連絡が取れずパッチ未提供のため、リモート管理の無効化など緩和策で凌ぐしかない。家庭・SOHO で使われる機種のため国内利用者にも影響しうる。

## 🔰 初学者向け（何が・なぜ問題か）

家庭や小規模オフィスでインターネットに繋ぐための機器が「Wi-Fi ルーター」です。今回問題になった Tenda 製ルーターには、**表向きの正規の入り口（パスワード認証）とは別に、こっそり用意された裏口（バックドア）**が仕込まれていました。

普通は、管理画面に入るには正しいパスワードが必要です。ところがこのルーターは、パスワード認証に**わざと失敗させる**と、裏口の判定に切り替わり、正しいパスワードを知らなくても管理者として入れてしまいます。マンションの正面玄関の暗証番号を知らなくても、"わざと間違えると開く勝手口"がある、というイメージです。

管理者権限を奪われると、通信内容の盗み見、偽サイトへの誘導（DNS の書き換え）、他の家電への攻撃の踏み台化などが可能になります。さらに厄介なのは、**メーカーと連絡が取れず修正プログラムが出ていない**ことです。そのため「裏口に外から手が届かないようにする」自衛策が中心になります。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **脆弱性種別**: 未文書の認証バックドア（CWE-912: Hidden Functionality / CWE-288: 代替経路による認証回避）。設計・実装に埋め込まれた意図的な隠し機能。
- **CVSS**: 「要確認」（公開時点で機関により評価差。CERT/CC VU#213560 を一次情報とする）。深刻度は管理者権限奪取相当で高い。
- **前提条件**: 管理用 Web インターフェイス（`/bin/httpd`）に到達できること。リモート管理が有効、または LAN 内からのアクセスが可能な状況。
- **影響機種**: Tenda FH1201, W15E, AC10, AC5, AC6 V2 など複数（機種・版により異なる。全容は要確認）。
- **修正版**: 提供なし（ベンダーと連絡不能）。緩和策のみ。
- **悪用状況**: 隠し認証経路として悪用容易。CERT/CC が7月7日に警告、実悪用リスクを指摘。

### メカニズム（攻撃が成立する仕組み）

以下は概念レベルの説明であり、具体的なトリガ値やPoCは示さない。

1. **信頼境界**: 利用者／攻撃者 → ルーター管理 Web（`/bin/httpd` の `login()`）。ここが「正規パスワードを持つ者だけを通す」べき認証境界。
2. **不備の所在**: `login()` は当初 MD5 ベースのパスワード検証という**正規経路**をたどる。しかし検証に失敗すると、そのまま拒否せず**代替コードパス（隠し経路）へ分岐**する実装になっている。
3. **攻撃者が握るプリミティブ**: 攻撃者は正規パスワードを知らなくてよい。むしろ**認証に失敗させること**が隠し経路の起動条件。これにより「正しい資格情報なしで認証成功状態を得る」プリミティブを握る。
4. **橋渡し（認証の否定を肯定へ）**: 本来「失敗＝拒否」で終わるべきロジックが「失敗＝別経路で許可」へと反転している。認証ロジックの**フェイルセキュア（失敗時は安全側）**原則が破られ、失敗が管理者昇格へ橋渡しされる。
5. **最終被害**: 管理者権限奪取 → 設定改ざん（DNS/ファイアウォール）、通信傍受、ボットネット化、内部ネットワークへの侵入。
- **一般原理**: 認証は「明示的に成功した場合のみ通す（デフォルト拒否）」で設計すべき。今回のように"失敗時に別経路が生きている"のは、代替経路認証（alternate channel）とハードコード資格情報の典型的アンチパターン。IoT/組込みでのサプライチェーン起因の隠し機能問題でもある。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: 管理画面への外部からのアクセス、認証失敗直後に成立する管理セッション、設定（DNS・管理者アカウント・ファーム）改ざんの痕跡、想定外のアウトバウンド通信。
- **暫定緩和（パッチが無いため中心策）**: (1) **リモート管理（WAN 側管理）を無効化**、(2) 既定 LAN IP を変更して到達性を下げる、(3) 管理画面へのアクセス元を信頼セグメントに限定、(4) 可能なら該当機種の**リプレース（別ベンダー・サポート継続機種へ）**を検討。
- **恒久対策**: ベンダー修正が出ない前提で、サポート切れ／連絡不能ベンダーの機器は計画的に置き換える。
- **運用示唆**: 家庭・SOHO ルーターは"設置後放置"になりがち。棚卸しと EOL 管理、リモート管理の既定無効化を運用ルール化する。

## 📖 用語解説（このニュースとのつながり）

- **バックドア（裏口）**: 正規認証を迂回する隠し経路。本件は `login()` 内に実装され、パスワードを知らなくても管理者になれる。
- **代替経路認証（Alternate Path Authentication）**: 正規の認証とは別の判定経路。失敗時にこちらへ分岐する設計が今回の致命傷。
- **フェイルセキュア**: 「失敗したら安全側（拒否）に倒す」原則。本件はこれが破れ、失敗が許可に化けている。
- **リモート管理（WAN 側管理）**: インターネット側から管理画面に入れる機能。有効だと世界中から裏口を試される。無効化が第一の自衛策。
- **EOL / ベンダー連絡不能**: 修正提供が見込めない状態。緩和策と機器リプレースが唯一の現実解になる。

## 影響範囲 / 推奨対応

- **影響範囲**: Tenda FH1201・W15E・AC10・AC5・AC6 V2 等の該当機種（詳細は要確認）。家庭・SOHO 利用が中心で国内でも流通。
- **推奨対応**: (1) リモート管理を直ちに無効化、(2) 既定 LAN IP を変更、(3) 管理アクセスを信頼セグメントに限定、(4) パッチが出ない前提でサポート継続機種への置き換えを計画。
