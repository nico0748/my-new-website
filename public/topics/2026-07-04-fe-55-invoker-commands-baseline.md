# HTML Invoker Commands が Baseline 入り — JSなしでポップオーバー/ダイアログ操作

> トピック: 開発トピック｜出典: InfoQ｜種別: frontend

## 一行サマリー
`command` / `commandfor` 属性による HTML Invoker Commands が主要ブラウザ全てで対応し Baseline に到達した。Popover API と組み合わせると、ダイアログやポップオーバーの開閉を JavaScript を書かずにボタン属性だけで宣言的に制御できる。

## 🔰 初学者向け（何が・なぜ重要か）
これまで「ボタンを押したらメニューやモーダルを開く」といったよくある操作は、たいてい JavaScript を書いて実現していました。ボタンにクリックの反応を仕込み、対象の要素を探して、表示/非表示を切り替える——短いコードでも、積み重なると保守の手間になります。

Invoker Commands は、この「ボタンを押したら、あの要素にこの操作をする」という関係を、ボタンの属性2つだけで書けるようにする仕組みです。`commandfor` に操作したい相手の ID を、`command` に「開く」「閉じる」などの動作を書くだけ。配線を HTML に直接書き込むイメージで、JavaScript は不要になります。

嬉しいのは、標準化されているのでブラウザが挙動を保証してくれる点です。キーボード操作やアクセシビリティ（支援技術対応）もブラウザ側が面倒を見てくれるため、自前実装にありがちな「マウスでは動くがキーボードで閉じられない」といった抜け漏れが減ります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Invoker Commands は `<button>` に `commandfor`（操作対象要素の id）と `command`（実行するアクション）の2属性を追加する。Safari 26.2 の対応で Chrome 135 / Firefox 144 に続き主要ブラウザが出揃い、Baseline に到達した（各ブラウザの正確なバージョン・提供時期は公式ドキュメントで要確認）。組み込みコマンドとして `show-popover` / `hide-popover` / `toggle-popover`、`<dialog>` 向けの `show-modal` / `close` などが用意され、Popover API（`popover` 属性）と組み合わせて宣言的な UI 制御が完結する。カスタムコマンド（`--` 接頭辞）で独自イベントを飛ばすこともできる。

### 仕組み / 背景（なぜこう設計されたか）
従来はアクセシブルなポップオーバー/モーダルを作るのに、フォーカストラップ、ESC での閉じ、`aria-*` 属性、外側クリック検知などを自前実装する必要があり、Floating UI などのライブラリや大量のボイラープレートに頼っていた。Popover API と Invoker Commands は、これらの「暗黙のUX契約」をブラウザ側の宣言的プリミティブに引き上げる設計だ。トップレイヤー（top layer）でのレンダリング、light-dismiss（外側クリックで閉じる）、フォーカス管理を標準が担うため、実装差やバグが減る。トレードオフは、複雑な条件分岐や動的生成を伴うUIでは依然 JS が要ること、そして古いブラウザ向けにはプログレッシブ・エンハンスメント（未対応時は JS フォールバック）の設計が必要な点。

### 実務での使いどころ・移行の勘所
メニュー、ツールチップ、確認ダイアログ、通知トーストなど「開閉するだけ」の UI は Invoker Commands + Popover API に置き換える価値が高い。既存のモーダルライブラリを段階的に減らせ、バンドルサイズとアクセシビリティ双方に効く。移行時は、まず単純な確認ダイアログから置換し、`command` の対応アクション表を確認。未対応ブラウザを切る判断ができない場合は、属性ベースを基本にしつつ JS で機能検出してフォールバックを添える。

## 📖 用語解説（このトピックとのつながり）
- **Invoker Commands**: `command` / `commandfor` 属性で要素操作を宣言的に指定する HTML 機能。Baseline 入り。
- **Popover API**: `popover` 属性でポップオーバーを宣言する仕組み。Invoker Commands と組で JS 不要に。
- **Baseline**: 主要ブラウザで安心して使える機能セットの目印。Invoker Commands が到達。
- **top layer**: 他要素より前面に描画される専用レイヤー。ポップオーバー/モーダルが z-index 競合を避けられる。
- **プログレッシブ・エンハンスメント**: 対応環境で強化し、非対応でも壊れないようにする設計方針。移行時の指針。

## 影響範囲 / 推奨アクション
フロントエンド全般に関係。開閉するだけの UI から段階的に置換すると、コード量とアクセシビリティ品質を同時に改善できる。対応ブラウザ範囲を確認し、必要に応じて JS フォールバックを併用。既存モーダルライブラリの縮小を検討する好機。
