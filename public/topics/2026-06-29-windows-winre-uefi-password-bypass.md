# Windows回復環境(WinRE)でUEFI/BIOSパスワードを回避（CVE-2026-45585）

> トピック: 脆弱性ニュース ｜ 出典: GBHackers / CERT/CC VU#226679 / Windows News ｜ 種別: 認証(ポリシー)強制の回避・物理アクセス悪用

## 一行サマリー
Windows回復環境(WinRE)の不備（CVE-2026-45585）により、`BootNext` などのブート変数を操作してUEFI/BIOSパスワードの強制を回避できる。物理または管理者アクセスを持つ攻撃者がファームウェア保護を迂回し、設定改ざんやBitLocker復号、永続マルウェア設置に至る恐れがある。

## 🔰 初学者向け（何が・なぜ問題か）
パソコンには電源を入れた直後に動く「土台のソフト（UEFI/BIOS）」があり、ここに**起動時のパスワード**をかけて勝手な起動や設定変更を防ぐ仕組みがあります。これは盗難や物理的ないたずらへの最後の砦です。

今回の問題は、Windowsの「修復モード（WinRE＝調子が悪い時に立ち上がる回復用の環境）」を悪用すると、この**起動パスワードのチェックをすり抜けられる**というものです。攻撃者は「次はこれを起動して」という指示（BootNextという起動メモのようなもの）を書き換えることで、パスワードを問われない裏ルートからの起動を成立させます。

たとえるなら、金庫に暗証番号をかけていても、「点検モード」という別の扉から番号なしで中に入れてしまうようなものです。その結果、ディスクの暗号化(BitLocker)が解かれてデータを読まれたり、消えないマルウェアを仕込まれたりする危険があります。物理的に鍵をかけた端末でも被害が起きうる点が深刻です。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）
- **CVE / 参照**: CVE-2026-45585、CERT/CC VU#226679（"Microsoft WinRE allows for bypass of UEFI/BIOS password enforcement"）。深刻度はHigh（CVSS数値は要確認）。
- **種別（CWE）**: 認証／ポリシー強制の回避（Authentication/Policy Bypass）。ブート構成変数の信頼性に依存する設計上の欠陥。
- **前提条件**: 攻撃者が **物理アクセス**もしくは**管理者権限**を保有。特定のファームウェア実装下でWinREの代替ブート経路が認証を一貫して強制しないことが条件。
- **影響範囲**: WinREを利用する Windows 10 / Windows 11（要確認: ファームウェア実装に依存）。
- **悪用状況**: 研究者により実証。BitLocker復号・永続化への波及リスクが指摘される。

### メカニズム（攻撃が成立する仕組み）
ファームウェアのパスワード強制と、OS側の回復環境の起動経路の間にある「信頼の隙間」を突く設計上の欠陥である（概念レベルで説明）。

1. **信頼境界**: UEFI/BIOSパスワードは「正規の起動経路では必ず認証を要求する」前提で保護を提供する。だが起動経路は一つではなく、`BootNext` 等のNVRAM変数で次回起動先を指定できる。
2. **不備の所在**: 一部ファームウェア実装では、WinREへ遷移する代替ブート経路でパスワード強制が一貫して適用されない。`BootNext` の書き換えで、この「認証が緩い経路」を意図的に選ばせられる。
3. **獲得プリミティブ**: 攻撃者は「パスワード認証を回避した特権的な回復環境の実行」というプリミティブを得る。回復環境はディスクやブート設定に強い権限を持つ。
4. **被害の成立**: ブート設定の改ざん、BitLockerで保護されたボリュームの復号・アクセス、ブート段階での永続マルウェア設置へと橋渡しされる。物理保護を前提にした端末でも保護が無効化される。

一般原理として、保護機構（ここではファームウェアパスワード）は **すべての到達経路で一貫して強制**されなければ意味を成さない。1つでも認証を緩める経路（代替ブート）が残ると、保護全体が回避可能になる。

### 検知・暫定緩和・恒久対策
- **検知の着眼点**: `BootNext`/ブート順の予期しない変更、WinREへの想定外遷移、BitLocker回復イベントの多発、物理アクセスの形跡。
- **暫定緩和**: 物理セキュリティの強化（筐体施錠・持ち出し管理）、不要な外部起動の無効化、可能なら未使用時のWinRE無効化を検討。ファームウェア設定の保護強化。
- **恒久対策**: Microsoft/OEMの修正適用、ファームウェア(UEFI)更新、BitLockerはTPM＋PIN等の多要素構成を採用。

## 📖 用語解説（このニュースとのつながり）
- **WinRE（Windows回復環境）**: 障害時に起動する回復用ミニOS。今回の回避に悪用される経路。
- **UEFI/BIOSパスワード**: 起動時に要求するファームウェアレベルの認証。本件で回避される保護。
- **BootNext / ブート変数**: 次回の起動先を指定するNVRAM変数。書き換えにより緩い経路を選ばせる。
- **BitLocker**: Windowsのディスク暗号化。回避が成立すると復号・データ閲覧の波及先になる。
- **VU#226679**: CERT/CCの脆弱性ノート番号。本件の一次情報源の一つ。

## 影響範囲 / 推奨対応
WinREを利用するWindows端末、とくに**盗難・持ち出しリスクの高いノートPC**で影響が大きい。物理アクセス前提の保護（ファームウェアパスワード＋BitLocker）に依存する組織は、ベンダ修正の適用、UEFI更新、BitLockerの多要素化、物理管理の強化を進める。
