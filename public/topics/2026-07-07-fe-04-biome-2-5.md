# Biome v2.5 — 500 ルール到達・クロスファイルリント・watch モードで「ESLint 置き換え」が完成度を増す

> トピック: Biome v2.5 リリース｜出典: Biome 公式ブログ｜種別: ツールチェーンのマイナーリリース

## 一行サマリー

Rust 製リンタ/フォーマッタの Biome が v2.5 で lint ルール 500 個を突破し、Vue 対応・型認識ルールを含む70超のルールを安定化、モジュールグラフを使うクロスファイルリント（未使用 CSS クラス検出など）、プラグインによるコード修正、`--watch` オプションを追加した。

## 🔰 初学者向け（何が・なぜ重要か）

リンタとは「コードの誤字脱字チェッカー」のようなもので、バグの芽や書き方の乱れを自動で指摘してくれる道具です。JavaScript の世界では長年 ESLint（チェック）と Prettier（整形）の2本立てが定番でしたが、どちらも JavaScript 製で、大きなプロジェクトでは動作が遅いのが悩みでした。

Biome は「チェックと整形を1つの道具に統合し、Rust という高速な言語で作り直す」プロジェクトです。今回の v2.5 で、指摘できるルールの数が 500 種類を超え、定番 ESLint からの乗り換えで「あのルールがない」と困る場面がかなり減りました。

もう一つの目玉は「ファイルをまたいだチェック」です。従来のリンタは基本的に1ファイルずつしか見ないため、「この CSS クラス、実はどこからも使われていない」といった全体を見ないと分からない問題は苦手でした。Biome はプロジェクト全体の地図（モジュールグラフ）を持っているので、こうした「横断的なゴミ検出」ができます。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **lint ルール 500 超**: v2.5 で70以上のルールを nursery（実験枠）から昇格。Vue フレームワーク対応ルール、型認識（type-aware）ルールを含む。
- **クロスファイルリント**: モジュールグラフを利用する新 nursery ルール `noUndeclaredClasses` / `noUnusedClasses`。後者はプロジェクト内の全 CSS クラス定義を解析し、HTML 系ファイルや JSX から一度も参照されないクラスを検出する。
- **プラグインによるコード修正**: プラグインが診断だけでなく fix（自動修正）を提供可能に。
- **`linter.rules.preset` オプション**: 既定で有効にするルール群を制御。`all` プリセットで nursery 以外の全ルールを一括有効化できる。
- **watch モード**: `lint` / `format` / `check` に `--watch` を追加。ファイル変更を監視し、入力中に診断を出し続ける常駐動作。

### 仕組み / 背景（なぜこう設計されたか）

ESLint の設計は「1ファイル＝1解析単位」を基本とし、型情報やプロジェクト横断情報が必要なルール（typescript-eslint の type-aware ルール等）は TypeScript コンパイラを別プロセスで回すため極端に遅くなる。Biome のアプローチは、パーサ・セマンティックモデル・モジュールグラフを単一の Rust プロセス内に常駐させ、インクリメンタルに更新することでクロスファイル解析のコストを償却する、というもの。watch モードはこのアーキテクチャの自然な帰結で、「毎回ゼロから解析」ではなく「差分だけ再解析」が前提にある。

型認識ルールについては、TypeScript コンパイラを使わず Biome 自身が型推論を部分実装する路線を取っており、精度と網羅性は tsc に及ばない場面がある（どのルールがどこまで推論できるかは要確認）。これは「完全な型検査は tsc/tsgo に任せ、リンタは高速な近似で日常の 9 割を拾う」という役割分担のトレードオフである。

背景として、2026 年のフロントエンドツールチェーンは Rust/Go 化が主戦場（Rolldown、Oxc、tsgo）で、Biome は「lint + format」の枠を超えて Oxlint と競合しつつ、フレームワーク対応（Vue、今後 Svelte 等）の広さで差別化を図っている。

### 実務での使いどころ・移行の勘所

- ESLint + Prettier からの移行は「フォーマッタ→リンタ」の2段階が定石。`biome migrate eslint` 系の移行支援でルールマッピングを確認し、対応のないカスタムルールはプラグイン（GritQL ベース）で補えるか評価する。
- `noUnusedClasses` はデザインシステムの負債掃除（使われていないユーティリティ/レガシークラスの棚卸し）に即効性がある。ただし動的クラス名（テンプレート文字列で組み立てる等）は検出できず偽陽性が出るため、まず警告レベルで導入を。
- monorepo では単一プロセスで全パッケージを見られる利点が大きい一方、エディタ統合（LSP）のメモリ使用は監視対象に。
- type-aware ルールを本格運用する場合も、CI での `tsc --noEmit`（または tsgo）は引き続き必須。リンタの型推論は代替ではなく補完。

## 📖 用語解説（このトピックとのつながり）

- **nursery ルール**: 安定化前の実験的ルール枠。v2.5 では70超が安定版へ昇格し、新しいクロスファイルルールが nursery に入った。
- **モジュールグラフ**: import/export や参照関係でファイル同士を結んだグラフ。クロスファイルリントの基盤データ構造。
- **type-aware リント**: 型情報を使う静的検査。`noFloatingPromises` のように型が分からないと判定できないルール群。
- **GritQL**: Biome プラグインで使われるコードパターン記述言語。v2.5 で「検出」に加え「修正」も書けるようになった。
- **Oxlint**: Oxc プロジェクトの高速リンタ。Biome の主要な競合で、Vite/Rolldown 陣営との関係が対比軸になる。

## 影響範囲 / 推奨アクション

- ESLint の実行時間が CI ボトルネックのプロジェクトは v2.5 を再評価する好機（ルール数の壁が大きく下がった）。
- 既存 Biome ユーザーは `preset` 設定の導入と、CSS 系プロジェクトでの `noUnusedClasses` 試験導入を推奨。
