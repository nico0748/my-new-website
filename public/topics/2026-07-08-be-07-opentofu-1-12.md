# OpenTofu 1.12 — dynamic prevent_destroyとdestroy=falseで壊れやすさを設計する

> トピック: OpenTofu 1.12｜出典: OpenTofu Release Notes / Releasebot｜種別: マイナーリリース

## 一行サマリー

Terraform のオープンソースフォーク OpenTofu が 1.12 をリリース（2026-06-23 更新）。**動的な prevent_destroy**、リソース単位の **destroy = false**、`-json-into` 出力フラグ、プロバイダロックへの**チェックサム自動補完**など、ライフサイクル制御と init の使い勝手を強化した。

## 🔰 初学者向け（何が・なぜ重要か）

インフラを「コード」で定義して自動構築するのが IaC（Infrastructure as Code）で、その代表格が Terraform です。OpenTofu はその Terraform が有償ライセンスに移行したのを受けて分岐した、Linux Foundation 傘下の**完全オープンソース版**です。設定ファイル（HCL）で「こういうサーバ・DB・ネットワークが欲しい」と書くと、クラウドにその通りの状態を作ってくれます。

IaC で怖いのは「うっかり大事なリソースを消してしまう」事故です。本番データベースをコードのちょっとした変更で破棄する、といった惨事が起こり得ます。1.12 はこの「壊れやすさ」を賢く制御する機能を強化しました。

例えば `prevent_destroy`（破棄禁止）を**動的に**切り替えられるようになり、「本番環境では消せない、検証環境では消せる」といった条件付きの安全弁を1つのコードで表現できます。`destroy = false` はリソースを丸ごと破棄対象から外す指定です。人為ミスによる破壊を、設定として明示的に防げるようになります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **リリース**: 1.12（2026-06-23 更新）。
- **dynamic prevent_destroy**: これまで `lifecycle { prevent_destroy = true }` は**静的なリテラルのみ**だったが、式・変数で動的に評価できるように。環境やフラグに応じた破棄防止が可能。
- **destroy = false**: リソースを破棄対象から除外する新しいライフサイクルオプション。
- **-json-into フラグ**: 出力を指定先へ JSON で書き出す新オプション。CI/自動化での結果取り込みを容易に。
- **provider install 高速化**: プロバイダインストールの高速化と init 挙動の改善。
- **チェックサム自動補完（checksum backfill）**: プロバイダロックファイルの不足チェックサムを自動補完し、ロックをきれいに保つ。

### 仕組み / 背景（なぜこう設計されたか）

**なぜ prevent_destroy を動的化するのか**。従来 `prevent_destroy` は安全のため**定数リテラル**しか受け付けませんでした。しかし現実の運用では「本番だけ守り、他は自由に壊したい」「特定フラグが立っているときだけ守りたい」という条件分岐が欲しくなります。静的制約はそれを module の分岐や重複コードで回避させ、ミスの温床でした。動的評価を許すことで、**1つの構成に環境依存の安全弁を織り込める**ようになります。トレードオフは「動的にした結果、うっかり `false` に評価されて守られない」リスクで、式の設計は慎重にすべきです。

**destroy = false** は、`prevent_destroy`（破棄を試みるとエラーで止める）とは異なり、そもそもそのリソースを破棄計画から**除外**します。データストアや共有基盤など「Terraform/OpenTofu の管理下にあるが、ライフサイクルとしては別扱いしたい」対象に有効です。

**チェックサム自動補完**は、チーム開発で `.terraform.lock.hcl` のチェックサムがプラットフォーム間で欠けることによる init 失敗を減らします。ロックファイルはサプライチェーン安全性（改ざん検知）の要なので、欠損を静かに埋めて整合を保つのは実務での摩擦削減につながります。

OpenTofu はフォーク後、Terraform との差別化として state 暗号化やこうしたライフサイクル制御など**コミュニティ主導の機能**を積み上げており、1.12 もその路線上にあります。

### 実務での使いどころ・移行の勘所

- **本番の破壊防止**を環境変数やワークスペースに応じて動的に切り替えたいチームに直接効く。`prevent_destroy` を条件式化して安全弁を一元管理。
- `destroy = false` は、DB やストレージバケットなど「絶対に破棄計画へ入れたくない」リソースの保護に。
- `-json-into` は CI パイプラインで plan/apply 結果を機械可読に取り込む用途に有用。
- init 高速化とチェックサム補完は、大規模モノレポや多プロバイダ構成での**日常的な待ち時間と摩擦**を軽減。
- Terraform からの移行を検討中なら、OpenTofu 独自のライフサイクル制御が移行の後押しになりうる。ただし HCL 互換性と state の取り扱いは事前検証を。

## 📖 用語解説（このトピックとのつながり）

- **IaC（Infrastructure as Code）**: インフラをコードで宣言し自動構築する手法。OpenTofu はその実装。
- **prevent_destroy**: リソース破棄を阻止する lifecycle 設定。1.12 で動的評価に対応。
- **destroy = false**: リソースを破棄計画から除外する新オプション。prevent_destroy とは別機構。
- **lock ファイル（.terraform.lock.hcl）**: プロバイダのバージョン・チェックサムを固定する依存ロック。改ざん検知の要。
- **provider**: クラウド等を操作するプラグイン。init 時にインストールされる。
- **HCL**: OpenTofu/Terraform の設定記述言語。

## 影響範囲 / 推奨アクション

- 環境別の破壊防止を求めるなら **dynamic prevent_destroy** で安全弁を条件式化。
- DB/ストレージなど死守すべきリソースに **destroy = false** の適用を検討。
- CI で plan/apply 結果を機械処理するなら **-json-into** を採用。
- 大規模構成では init 高速化とチェックサム自動補完の恩恵を計測。
- 動的化した prevent_destroy が誤って false に評価されないよう、式のレビューを厳格に。
