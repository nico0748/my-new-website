# TypeScript 7.0 が GA — Go で書き直した「ネイティブ・コンパイラ」で型検査が約10倍速く

> トピック: TypeScript 7.0 GA（Go ネイティブ移植「Project Corsa」）｜出典: microsoft/typescript-go・The Register・Visual Studio Magazine｜種別: 言語 / ツールチェーン

## 一行サマリー

Microsoft は 2026 年 7 月 8 日、コンパイラ本体を JavaScript から Go へ移植した TypeScript 7.0 を正式版（GA）としてリリースした。型検査は多くのプロジェクトで 8〜12 倍速くなり、VS Code のコードベースでは 125.7 秒→10.6 秒（約11.9倍）に短縮された。ただし安定した公開 API はまだ無く、Vue・Angular・typescript-eslint など API 依存ツールの本格対応は 7.1 以降を待つ段階。

## 🔰 初学者向け（何が・なぜ重要か）

TypeScript は「JavaScript に型（この変数は数値、これは文字列…という約束）を足して、書き間違いを実行前に見つけてくれる」言語。その型チェックを行うのが「コンパイラ」で、これまでは TypeScript 自身（JavaScript）で書かれていた。

今回の 7.0 は、そのコンパイラを **Go というより速い言語で書き直した**のが最大の変化。たとえるなら、これまで自転車で配達していた荷物を、同じ配達ルート・同じ荷物のまま、性能のいいバイクに載せ替えたようなもの。ルート（型チェックの判断）は一切変えていないので結果は同じだが、到着（チェック完了）が桁違いに速くなる。

なぜ嬉しいか。大きなプロジェクトでは型チェックに数分かかり、エディタの補完やエラー表示がもたつくことがあった。それが数秒で終わるようになると、書いた瞬間に赤線が出て、直して、また書く…という開発のリズムが一気に軽くなる。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **GA 日**: 2026 年 7 月 8 日。RC は 6 月 18 日に公開され、「RC からおよそ1か月で安定版」という予告どおりに着地した。
- **性能**: フルビルドで概ね 8〜12 倍。VS Code の型検査は既定設定で 125.7 秒 → 10.6 秒（約11.9倍）。ネイティブコード速度・共有メモリのマルチスレッド化・各種最適化による。
- **「移植（port）」であって「書き直し（rewrite）」ではない**: 既存コードベースをファイル単位で Go へ移し、アルゴリズムとデータ構造を保存。型検査ロジックは 6.0 と構造的に同一で、型判定のセマンティクスは変わらない。
- **配布**: 通常の `latest` タグで配布。`npm install -D typescript` で 7.0 が入る。
- **注意点（要確認扱いにすべき点）**: 7.0 は**安定した programmatic API を持たない**。新 API は 7.1 で提供予定とされ、そのため Vue、Angular のテンプレート型検査、Svelte、Astro、MDX、typescript-eslint 等の API 依存ツールは即座には切り替えられない。

### 仕組み / 背景（なぜこう設計されたか）

もともと TypeScript コンパイラは TypeScript（＝JavaScript）で書かれ、シングルスレッドの JS ランタイム上で動いていた。型検査は「巨大な依存グラフを何度も走査し、型の互換性を判定する」CPU 集約的な処理で、JS のシングルスレッド実行とガベージコレクションが速度上限になっていた。

Go を選んだのは、(1) ネイティブコンパイルで実行が速い、(2) goroutine による並行処理と共有メモリで型検査を並列化できる、(3) 構造体・ポインタなどのデータモデルが既存 TS コンパイラの内部表現と写しやすい、という三点が理由とされる。あえて「rewrite」ではなく「port」にしたのは、**挙動を1ミリも変えずに速度だけを得る**ため。型システムを作り直すと膨大な既存コードの型付けが変わってしまうリスクがあるが、ロジックを保存した移植ならそれを回避できる。

トレードオフは公開 API の空白期間だ。エコシステムの多くのツールは TS のコンパイラ API を内部から呼び出しており、Go 実装ではその API 表面がまだ確定していない。結果として「単体の型検査・ビルドは今すぐ速くなるが、フレームワーク統合ツールは追随に時間がかかる」という過渡期が生じる。

### 実務での使いどころ・移行の勘所

型検査が律速になっている大規模モノレポ、CI の型チェック時間が長いプロジェクト、エディタ補完がもたつく現場で効果が大きい。移行の勘所は、(1) まず CI やローカルの `tsc --noEmit` を 7.0 に切り替えて速度を測る、(2) ただし typescript-eslint や Vue/Angular/Svelte のテンプレート型検査など API 依存ツールは 6.x を併用し、7.1 の新 API 提供まで無理に一本化しない、(3) セマンティクスは同一なので型エラーの内容自体は変わらない前提で、差分が出たら報告する、という順序。段階導入が現実的。

## 📖 用語解説（このトピックとのつながり）

- **コンパイラ / 型検査**: ソースを解析し型の整合性を検証する処理。7.0 はこれを Go で高速化した。
- **port（移植）と rewrite（書き直し）**: 前者はロジックを保存したまま別言語へ移す。7.0 は port なので挙動不変。
- **programmatic API**: 他ツールがコンパイラ機能を呼ぶための API。7.0 では未安定で、対応ツールの追随を左右する。
- **typescript-eslint / テンプレート型検査**: TS の API を内部利用するツール群。7.1 の API 提供が本格対応の前提。
- **共有メモリ・マルチスレッド**: 複数スレッドで同じデータを触りながら並列処理する方式。型検査の並列化に効く。

## 影響範囲 / 推奨アクション

- 大規模 TS プロジェクトはビルド/型検査時間の大幅短縮が見込める。まず CI とローカルで速度計測を。
- API 依存ツール（eslint・各フレームワークの型検査）は当面 6.x 併用が無難。7.1 の新 API を待つ。
- セマンティクス不変が原則。移行で型エラーの挙動差が出たら upstream に報告。
