# GNU Inetutils telnetd の認証バイパスで遠隔からroot（CVE-2026-24061, PoC公開）

> トピック: 脆弱性ニュース ｜ 出典: SafeBreach Labs / Kudelski Security / NVD ｜ 種別: 認証バイパス（USER環境変数経由の信頼境界誤り）

## 一行サマリー

GNU Inetutils の telnetd（〜2.7）に、接続時の `USER` 環境変数へ `-f root` を渡すことでパスワード入力なしにrootとしてログインできる認証バイパス（CVE-2026-24061、CVSS 9.8）。公開PoCが存在し、Telnetを外部公開する機器は即時の初期侵入リスクにさらされる。

## 🔰 初学者向け（何が・なぜ問題か）

「Telnet（テルネット）」は、ネットワーク越しに別のコンピュータへログインして操作する**古くからの仕組み**です。今は安全なSSHが主流ですが、古いネットワーク機器・産業機器・IoT機器ではいまだにTelnetが動いています。

たとえるなら、建物の受付に「お名前を教えてください」と聞かれて名前を答えると、本人確認の前に**名前と一緒にこっそり『私は管理者です』というメモを渡せてしまう**ような状態です。受付係（telnetd）は、本人確認（パスワード確認）を始める前に、利用者が自己申告した情報を信じて、その後のログイン処理の進め方を変えてしまいます。

具体的には、接続時に渡せる「ユーザー情報」の欄に特殊な指定（`-f root`）を仕込むと、telnetdが「この人はもうroot（管理者）として確認済み」と勘違いし、**パスワードを一切確認せずに最高権限でログインさせてしまう**のです。Telnetを外部に開けている機器なら、攻撃者は接続して特殊な値を送るだけで、いきなり管理者として侵入できます。すでに攻撃手順（PoC）が公開されているため危険度が高い状態です。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **CVE**: CVE-2026-24061
- **種別**: 認証バイパス（信頼境界の誤配置 / pre-auth状態の悪用）。CWEは概ねCWE-287/CWE-302系（表記は出典により**要確認**）。
- **CVSS**: 9.8（Critical）
- **前提条件**: 対象がGNU Inetutils telnetdを起動しTCP/23等を公開していること。**認証不要・遠隔から**。
- **影響範囲**: GNU Inetutils の telnetd 2.7 以前。組込みLinux/ネットワーク機器・IoTで採用例あり。
- **修正版**: Inetutilsの修正版へ更新（ディストリ提供パッチ含む。具体版数は環境ごとに**要確認**）。
- **悪用状況**: SafeBreach Labsが公開PoCを提供、検知ルール（Splunk/Elastic）も公開済み。Telnetは常時スキャン対象（Mirai系ボットネット）であり、武器化が容易。

### メカニズム（攻撃が成立する仕組み）

1. **信頼境界の誤配置**: telnetdはTelnetプロトコルのオプション交換段階で、クライアントから環境変数（`USER` など）を受け取り保持する。本来これは「認証**前**の、攻撃者が完全制御できる入力」であり信用してはならない。ところがtelnetdはこのpre-auth状態を、後段のログイン処理（`login`プログラムの起動）に引き渡してしまう。
2. **login起動への引数注入**: telnetdは受け取った`USER`値を、システムのログイン処理を開始する際の入力として使う。ここで攻撃者が`-f root`という値を仕込むと、`-f`（多くのlogin実装で「事前認証済みとみなしパスワードを省略する」オプション）と`root`（対象ユーザー）が、認証フローに渡ってしまう。
3. **認証フローの省略**: `-f` 相当の指定により、loginは「このユーザーは既に認証済み」と解釈し、パスワード検証ステップを丸ごとスキップしてroot権限のセッションを確立する。
4. **プリミティブから最終被害へ**: 攻撃者が握るのは「認証前にlogin起動パラメータへ任意値を注入する」プリミティブ。これが認証スキップを介して**遠隔・事前認証なしのroot取得（完全な初期侵入）**へ直結する。

一般原理として、これは**「認証前にユーザー制御データを特権処理へ渡してはならない」**という鉄則（pre-auth入力の隔離）を破った典型例。引数インジェクション（argument injection）の一種でもある。具体的なパケット列・PoCコードは記載しない。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: telnetd接続で`USER`環境変数に`-`（ハイフン）始まりや`-f`を含む値、認証ログを伴わないrootセッション確立、TCP/23への不審な接続。公開されたSplunk/Elastic検知ルールの適用。
- **暫定緩和**: **Telnetを停止しSSHへ移行**（最優先）。やむを得ず使う場合は接続元をIP制限・VPN内に限定し、外部公開を遮断。
- **恒久対策**: Inetutils/ディストリの修正版telnetdへ更新。そもそもTelnetはクリアテキストで本質的に危険なため、廃止が望ましい。
- **運用示唆**: IoT/OT/古い機器の棚卸しでTCP/23の露出を点検。Mirai系の自動探索対象であり、放置=自動侵害のリスク。

## 📖 用語解説（このニュースとのつながり）

- **telnetd**: Telnet接続を受け付けるサーバ常駐プログラム。今回の脆弱な対象。
- **認証バイパス**: 本来必要な本人確認を回避して権限を得る欠陥。今回の本体。
- **引数インジェクション**: ユーザー入力をコマンド/プログラムの引数として注入する攻撃。`-f`オプション悪用がこれ。
- **pre-auth状態**: 認証完了前に保持される接続情報。攻撃者制御下で、特権処理に渡してはならない。
- **`-f`（loginオプション）**: 「事前認証済み」を示し、多くの実装でパスワード検証を省略する。今回悪用された挙動。

## 影響範囲 / 推奨対応

GNU Inetutils telnetd 2.7以前を稼働し、Telnet（TCP/23等）を公開する全システム・ネットワーク機器・IoTが対象。**Telnetの即時停止とSSH移行**を最優先に、停止できない場合は接続元制限と修正版telnetdへの更新を行うこと。PoC公開・自動スキャン対象のため放置は危険。
