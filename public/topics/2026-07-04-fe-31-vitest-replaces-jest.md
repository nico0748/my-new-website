# Vitest が Jest を置き換える — Vite ネイティブな高速テスト基盤

> トピック: 開発トピック｜出典: vitest.dev｜種別: frontend

## 一行サマリー
Vite をそのままテスト実行基盤に使う Vitest が、長年の定番だった Jest を置き換える標準になりつつある。ビルド設定を共有できるため、コールドスタートが速く、ESM/TypeScript/JSX をそのまま扱える。

## 🔰 初学者向け（何が・なぜ重要か）
テストとは「書いたコードが期待通り動くかを自動で確認する仕組み」です。従来は Jest というツールがその王座に長く座っていました。ただ Jest は、アプリを動かすための道具（バンドラ）とテスト用の道具が別々で、しかも最近主流の ES Modules や TypeScript を動かすのに変換設定を何重にも重ねる必要があり、遅く・複雑になりがちでした。

Vitest は「アプリのビルドに使う Vite を、そのままテストにも使い回す」という発想の道具です。料理でたとえるなら、これまでは調理用と試食用でキッチンを2つ持っていたのを、1つの高性能キッチンで両方こなすようなもの。設定は1か所で済み、起動も速く、開発中に保存するたびテストが即座に走る「watch モード」も軽快です。Vite が今のフロントエンドの標準ツールになった流れに乗って、テストも Vitest に一本化する、という自然な合流が起きています。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Vitest は Jest 互換の API（`describe` / `it` / `expect` / モック関数など）を提供しつつ、Vite の変換パイプライン（esbuild / SWC 相当）を共有する。ESM をネイティブに扱い、`vite.config.ts` の resolve・plugin・alias 設定をテストでもそのまま利用できる。近年は Browser Mode（Playwright/WebdriverKit を裏で使い、実ブラウザ内でコンポーネントをテスト）が強化され、jsdom エミュレーションでは再現しづらい CSS・レイアウト・実 DOM 挙動を検証できる。具体的な速度倍率は環境依存のため「要確認」だが、コールドスタートと HMR 連動の watch は Jest より体感で速いという報告が多い。

### 仕組み / 背景（なぜこう設計されたか）
Jest は独自のトランスフォーム層（babel-jest 等）とモジュール解決を持つため、Vite/webpack 側の設定と二重管理になりがちだった。ESM 化が進む中でこの二重構造が摩擦を生む。Vitest は「テストもアプリと同じビルドグラフで動かす」ことで設定の重複と変換のズレを排除する設計。トレードオフとして Vite への依存が強まるが、フロント側が既に Vite 前提なら実質デメリットは小さい。

### 実務での使いどころ・移行の勘所
新規プロジェクトは最初から Vitest を選ぶのが定石。既存 Jest からの移行は API 互換性が高く比較的容易だが、`jest.mock` の巻き上げ挙動や一部の設定キー（`moduleNameMapper` → `resolve.alias`）は読み替えが必要。jsdom 依存の重いテストは Browser Mode への置換を検討する価値がある。

## 📖 用語解説（このトピックとのつながり）
- **Vite**: 高速な開発サーバー兼ビルドツール。Vitest はこの上に構築される。
- **Jest**: Meta 発の従来標準テストランナー。Vitest の置換対象。
- **Browser Mode**: 実ブラウザでテストを走らせる Vitest の機能。jsdom の限界を補う。
- **jsdom**: Node 上で DOM を擬似再現するライブラリ。軽いが実描画は再現しない。
- **watch モード**: ファイル変更を検知し関連テストだけ即再実行する仕組み。

## 影響範囲 / 推奨アクション
Vite ベースのフロント開発者全般が対象。新規は Vitest 一択でよい。既存 Jest プロジェクトは、CI 時間短縮や設定簡素化のメリットが大きければ段階移行を検討。コンポーネントの見た目・DOM 挙動まで確かめたいチームは Browser Mode の導入を評価する。
