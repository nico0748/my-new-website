# ClickHouse ネイティブJSON（25.3）— JSONBench で MongoDB の2,500倍集計

> トピック: 開発トピック｜出典: earezki（PR-by-PR analysis）｜種別: backend

## 一行サマリー
分析DBの ClickHouse がバージョン 25.3 でネイティブ JSON 型を導入し、JSONBench の集計クエリで MongoDB 比 約2,500倍という桁違いの速度を示した（数値は二次情報のため要確認）。半構造化データの分析が列指向DBで一気に高速化する。

## 🔰 初学者向け（何が・なぜ重要か）
JSON は `{"name": "foo", "age": 20}` のような、キーと値の入れ子で表せる柔軟なデータ形式です。ログやイベントデータはこの形で大量に生まれます。従来こうした「かっちり列が決まっていない」データの集計は苦手とするDBが多くありました。

ClickHouse は「列指向」の分析DBで、大量データの集計が非常に速いのが売りです。今回、JSON をネイティブに（文字列として扱うのではなく専用の型として）保存・検索できるようになり、JSON データの集計まで爆速になりました。

たとえるなら、これまで「箱に雑多に放り込んだ書類（JSON文字列）」を集計のたびに開いて中身を読んでいたのを、書類の項目を最初から棚に整理して置くようにした、というイメージ。必要な項目だけを一気に取り出せるので、集計が桁違いに速くなります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
ClickHouse 25.3 でネイティブ JSON 型が実装され、JSON の各パス（フィールド）を内部的に列としてサブカラム化する。これにより、JSON 内の特定フィールドへの集計を列指向の圧縮・ベクトル化処理で高速に実行できる。JSONBench では MongoDB の約2,500倍という集計性能が報告されたが、ベンチ条件依存のため実データでの検証が必要（要確認）。

### 仕組み / 背景（なぜこう設計されたか）
列指向DBは「同じ列の値をまとめて格納」することで圧縮率とスキャン速度を稼ぐ。JSON をただの文字列で持つとこの利点が使えないが、パスごとにサブカラムへ展開すれば列指向の恩恵を JSON にも適用できる。スキーマ非依存で受け取りつつ、内部では列として最適化する、という「柔軟さと速度の両立」を狙った設計だ。

### 実務での使いどころ・移行の勘所
ログ分析、イベントトラッキング、可変スキーマのテレメトリなど、JSON を大量に集計する用途で威力を発揮する。ドキュメントDB（MongoDB 等）で集計が重い場合の移行先候補。ただし ClickHouse はトランザクション処理（OLTP）向きではないため、更新頻度が高い業務データには不向き。分析（OLAP）と業務処理を分けて考えること。

## 📖 用語解説（このトピックとのつながり）
- **列指向DB（OLAP）**: 列単位で格納する分析特化DB。ネイティブJSONがこの利点をJSONに広げる。
- **サブカラム化**: JSON の各パスを内部列に展開する処理。集計高速化の中核。
- **JSONBench**: JSON 集計性能のベンチマーク。MongoDB 比2,500倍の根拠。
- **OLTP / OLAP**: 業務処理と分析処理の区別。ClickHouse は OLAP 側で、用途の見極めが重要。

## 影響範囲 / 推奨アクション
JSON ログ/イベントを大量に分析するデータ基盤担当が対象。MongoDB 等で集計が重いなら ClickHouse への分析基盤移設を検証する価値がある。ただし更新頻度の高い業務データは別DBに保つこと。
