# ChatGPT Work — GPT-5.6 搭載、成果物を仕上げる自律エージェント

> トピック: OpenAI ChatGPT Work のローンチ ｜ 出典: OpenAI / VentureBeat / Axios ｜ 種別: 新プロダクト・AIエージェント

## 一行サマリー

OpenAI が 2026-07-09、GPT-5.6 を頭脳とする業務エージェント「ChatGPT Work」を公開。目標を伝えると、Slack・Teams・Google Drive・メール・カレンダー・CRM など接続アプリから文脈を集め、タスクを分解し、数時間かけて自律実行して、チャットの返信ではなく完成したスプレッドシート・スライド・ドキュメント・Web アプリを返す。接続は MCP ベースのプラグインで行う。

## 🔰 初学者向け（何が・なぜ重要か）

これまでの ChatGPT は基本的に「質問すると文章で答える」ツールだった。ChatGPT Work はその一歩先で、「**成果物そのものを仕上げて渡してくれる**」ことを狙ったエージェントだ。

たとえば「先月の営業実績をまとめた資料を作って」と頼むと、ChatGPT Work は自分から Slack やメール、Google Drive、CRM といった**あなたが接続したアプリを回って必要な情報を集め**、作業を細かいステップに分け、それを何時間もかけて自分でこなし、最後に**完成したスライドや表計算**を返してくる。返ってくるのは「こう作ればいいですよ」というアドバイスではなく、実際のファイルだ。

比喩でいえば、これまでの ChatGPT が「質問に答えてくれる相談相手」だったのに対し、ChatGPT Work は「仕事を渡すと最後まで仕上げてくれる新人アシスタント」に近い。アプリとの接続には **MCP（Model Context Protocol）** という共通規格が使われる。これは AI と外部サービスをつなぐ"共通のコンセント形状"のようなもので、Gmail・カレンダー・Slack・GitHub などに標準的な方法でつながる。AI が「答える」から「働く」へ移る流れを象徴するプロダクトだ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **ローンチ**: 2026-07-09、Pro / Enterprise / Edu 向けに Web・モバイルで展開開始。
- **エンジン**: **GPT-5.6** と Codex 技術を組み込んだエージェント層。GPT-5.6 は同日、ChatGPT・ChatGPT Work・Codex・API で提供開始（Sol/Terra/Luna のティア、各 1.05M 文脈・最大128K 出力との報）。
- **動作**: アウトカム（達成したい結果）を受け取り、接続アプリから情報収集 → ジョブを小ステップに分解 → 自律実行。複雑案件では数時間"張り付いて"作業を継続。出力は会話ではなく**スプレッドシート・スライド・ドキュメント・対話的 Web アプリ**などの完成物。
- **接続先**: Slack、Microsoft Teams、Google Drive、SharePoint、メール、カレンダー、CRM、プロジェクト管理ツールなど。外部連携は **MCP ベースのプラグイン**（Gmail、Google Calendar、Slack、GitHub 等）。
- **統合**: Codex アプリが新しい ChatGPT デスクトップアプリに統合され、Chat・Work・Codex を全プラン（Free 含む）で提供する方向。
- ※ ティア仕様・ロールアウト範囲は公表ベース、詳細は「要確認」。

### 仕組み / 背景（なぜこう設計されたか）

**"答える"から"仕上げる"へ**。従来の LLM 製品は 1 回の応答で完結する対話が中心だった。業務の多くは「複数ソースから情報を集め、順序立てて作業し、成果物を作る」多段プロセスであり、単発応答では埋められない。ChatGPT Work は、計画（タスク分解）・ツール実行・長時間の状態維持を組み合わせる「エージェント・ループ」を製品化したもの。長い文脈（1M 級）と、外部ツールを叩く能力、そして途中経過を保持しながら数時間走り続ける実行基盤が要になる。

**なぜ MCP か**。エージェントが実務で価値を出すには、社内外の多様な SaaS に安全につながる必要がある。ベンダごとに独自連携を作るのは非現実的なので、OpenAI は業界標準化が進む **Model Context Protocol** をプラグイン接続の土台に採用した。ツールの発見・呼び出しを標準化することで、接続先を横断的に増やせる。エージェント設計の一般原理でいえば、「推論（計画）」と「行動（ツール実行）」を交互に回す ReAct 的ループに、標準化されたツール層（MCP）と長期文脈を組み合わせた形だ。

**トレードオフ／論点**。(1) 自律実行はミスの波及や意図しない操作のリスクを伴い、権限設計・人間の承認ゲート・監査が不可欠。(2) 接続アプリに広範なアクセスを与えるため、データガバナンスとプロンプトインジェクション対策が重い課題。(3) 数時間の実行はコストと予測可能性の管理を要する。エージェントの"できること"が増えるほど、"任せてよいか"のガードレール設計が製品価値を左右する。

### 実務での使いどころ・移行の勘所

定型レポート作成、資料の下ごしらえ、部門横断の情報集約、繰り返しの事務作業に向く。導入の勘所は権限とガバナンス——接続アプリごとに最小権限を設定し、機微データへのアクセス範囲を絞り、成果物は人間がレビューする前提で運用する。プロンプトインジェクション（外部データに仕込まれた不正指示）への耐性はエージェント全般の弱点なので、信頼できないソースを読ませる場面では特に注意。まずは低リスクな社内タスクで試し、監査ログと承認フローを整えてから業務クリティカルへ広げるのが安全。競合（Anthropic の Claude 系エージェント、各社の業務エージェント）との比較評価も並行したい。

## 📖 用語解説（このトピックとのつながり）

- **AI エージェント**: 目標を受け、計画・ツール実行・反復を自律的に行う AI。ChatGPT Work はその業務版。
- **MCP（Model Context Protocol）**: AI と外部ツール／データを標準的につなぐプロトコル。接続プラグインの土台。
- **エージェント・ループ / ReAct**: 「考える→道具を使う→結果を見てまた考える」を反復する実行様式。
- **Codex**: OpenAI のコード実行・生成技術。ChatGPT Work のエンジンに組み込み。
- **プロンプトインジェクション**: 外部データに紛れた指示で AI を乗っ取る攻撃。接続アプリ運用の主要リスク。
- **データガバナンス**: どのデータに誰／何がアクセスできるかの統制。エージェント導入で重要度が増す。

## 影響範囲 / 推奨アクション

- レポート作成・情報集約・定型事務の自動化を狙う組織に有力だが、権限・ガバナンス設計が前提。
- 接続アプリは最小権限に絞り、成果物は人間レビュー、監査ログと承認フローを整備。
- プロンプトインジェクション対策を講じ、低リスク社内タスクから段階導入。競合エージェントと比較評価。
