# Python 3.14 で free-threading（No-GIL）が正式サポート — 真のマルチコア並列へ

> トピック: 開発トピック｜出典: Python 公式ドキュメント｜種別: backend

## 一行サマリー
Python 3.14（2025-10-07）で、GIL を持たない free-threaded ビルドが「実験的」から「正式サポート」に格上げされた。加えて template string literals、アノテーションの遅延評価、標準ライブラリでの subinterpreters 対応も入り、Python の並列処理と型まわりが大きく前進した。

## 🔰 初学者向け（何が・なぜ重要か）
Python にはこれまで「GIL（グローバル・インタプリタ・ロック）」という交通整理役がいました。これは「一度に Python のコードを実行できるスレッドは1つだけ」というルールです。おかげで内部の安全は保たれますが、CPU にコアが8個あっても、Python の計算処理は基本1コア分しか使えない、という制約がありました。料理人が8人いるのに、まな板は1つしか使わせない、という状態です。

free-threading（No-GIL）は、この「まな板1つ」制限を外し、複数のスレッドが本当に同時に計算できるようにする仕組みです。3.14 でこれが「実験扱い」から「正式に対応」へ昇格しました。CPU をフルに使える道が開けた、という節目です。

ただし、まな板を増やすと「同じ材料を2人が同時に触ってぶつかる」危険も出てきます。だからライブラリ側の対応やデータ競合への配慮が必要で、すぐ全員が恩恵を受けるわけではありません。それでも「Python は並列が苦手」という長年の弱点に正面から手を入れた、重要な一歩です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Python 3.14 は 2025-10-07 リリース。free-threaded（no-GIL）ビルドが公式サポート対象になった（従来は 3.13 で実験導入）。ベンチでは 3.13 の free-threaded インタプリタが標準比 約2.2倍、3.14 では 約3.1倍の並列性能向上が報告される（数値はワークロード依存・要確認）。他に、実行時に文字列を組み立てる template string literals（t-strings）、`from __future__ import annotations` に依らないアノテーションの遅延評価、標準ライブラリからの subinterpreters（インタプリタごとに独立した状態を持つ並行実行）サポートが追加された。

### 仕組み / 背景（なぜこう設計されたか）
CPython の GIL は、参照カウントによるメモリ管理を単純かつ高速に保つための単一ロックだった。撤廃には、参照カウントを競合安全にする（バイアス付き参照カウントや不変オブジェクトの特別扱い）、内部データ構造をスレッドセーフ化する、といった広範な作り替えが要る。3.14 の正式サポートは、この作り替えが一定の完成度に達したことを意味する。トレードオフは明確で、(1) シングルスレッド性能が僅かに低下しうる、(2) C 拡張が free-threaded 対応でないと安全に使えない、(3) 従来 GIL に暗黙依存していたコードでデータ競合が顕在化する。subinterpreters は「メモリを共有しない並行」で競合を避ける別解として併存する。

### 実務での使いどころ・移行の勘所
CPU バウンドな並列処理（数値計算、画像/データ処理、並列スクレイピングの解析部など）で恩恵が大きい。まず free-threaded ビルドを別環境で用意し、依存 C 拡張（NumPy 等）が free-threaded 対応済みかを確認する。既存コードは共有可変状態を洗い出し、ロックや不変データで守る。I/O バウンドな用途は従来通り `asyncio` で十分なことが多く、無理に切り替えない。当面は「標準ビルド + free-threaded ビルドの二本立て検証」で移行判断するのが現実的。

## 📖 用語解説（このトピックとのつながり）
- **GIL（グローバル・インタプリタ・ロック）**: 一度に1スレッドしか Python コードを実行させない仕組み。free-threading で撤廃可能に。
- **free-threaded / No-GIL ビルド**: GIL 無しでマルチコア並列を可能にする CPython 構成。3.14 で正式サポート。
- **subinterpreters**: メモリを共有しない独立インタプリタを並行実行する仕組み。競合を避ける並列の別解。
- **template string literals（t-strings）**: 実行時に安全に文字列を組み立てる新リテラル。SQL/HTML 生成の安全性向上に寄与。
- **C 拡張**: C で書かれた高速モジュール。free-threaded 対応でないと no-GIL 下で使えない点が移行の鍵。

## 影響範囲 / 推奨アクション
CPU バウンド処理を持つ Python バックエンド/データ処理に大きく関係。free-threaded ビルドを別環境で検証し、依存 C 拡張の対応状況を確認。共有可変状態の保護を前提に段階導入。I/O 中心の用途は asyncio 継続で問題ない。
