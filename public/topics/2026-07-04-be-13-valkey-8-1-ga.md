# Valkey 8.1 GA — 新ハッシュテーブルで反復3.5倍・Redis比3倍スループット

> トピック: 開発トピック｜出典: zilliz｜種別: backend

## 一行サマリー
Redis からフォークしたOSS「Valkey」の 8.1 が GA となり、新しいハッシュテーブル実装で反復処理が約3.5倍、Redis 比で最大3倍のスループットを謳う（数値は二次情報のため要確認）。ライセンス変更を嫌う層の受け皿として存在感を増している。

## 🔰 初学者向け（何が・なぜ重要か）
Valkey は、Redis が採用ライセンスを変更した際に「これまで通り完全オープンな Redis 互換が欲しい」という声から生まれた、Redis のフォーク（分岐版）です。Linux Foundation の下で開発され、多くのクラウド事業者が支援しています。

「フォーク」とは、あるソフトのソースコードを丸ごとコピーして別プロジェクトとして育てることです。Valkey は Redis と高い互換性を保ちつつ、独自に性能改善を進めています。8.1 はその性能改善が大きく進んだ版です。

たとえるなら、人気レストランのレシピを引き継いだ別の店が、同じ味を出しながら厨房の動線を改善してより速く料理を出せるようになった、というイメージ。既存の Redis 用アプリをほぼそのまま動かしつつ、速度と自由なライセンスを得られるのが魅力です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
8.1 の目玉は新ハッシュテーブル実装で、キー空間の反復（スキャン系操作）が約3.5倍高速、条件次第で Redis 比3倍のスループットを主張する。マルチスレッド化やメモリ効率の改善も継続している。ただし具体的な数値はベンチ条件依存で、まとめ記事由来のため一次ソースでの裏取りが必要（要確認）。

### 仕組み / 背景（なぜこう設計されたか）
ハッシュテーブルはキー・バリューストアの心臓部で、ここの効率がスループットとメモリ消費を大きく左右する。Valkey は Redis 互換を保ちながらこの中核を再設計することで、互換性を犠牲にせず性能を伸ばす戦略を採る。OSS ガバナンスが Linux Foundation にあることで、特定企業のライセンス方針に縛られない点が採用の安心材料になっている。

### 実務での使いどころ・移行の勘所
Redis のライセンス変更を懸念する組織、あるいは純粋に性能を求める組織にとって、Valkey は現実的な移行先。Redis プロトコル互換なので、多くのケースでクライアントコードの変更は最小限で済む。ただしモジュールエコシステム（RedisJSON, RediSearch 等）の互換範囲は要確認で、Redis 固有機能に依存している場合は移行コストが跳ねる可能性がある。

## 📖 用語解説（このトピックとのつながり）
- **フォーク**: 既存OSSを分岐させて別プロジェクト化すること。Valkey は Redis のフォーク。
- **ハッシュテーブル**: キーから値を高速に引くデータ構造。8.1 の性能改善の中心。
- **Linux Foundation**: 中立的なOSS財団。Valkey のガバナンスを担い、ライセンスの安心感を提供。
- **プロトコル互換**: Redis 用クライアントがそのまま通信できる性質。移行コストを下げる要因。

## 影響範囲 / 推奨アクション
Redis 利用者全般、特にライセンス方針を気にする組織が対象。移行時は使用中の Redis モジュール・機能の Valkey 互換性を確認し、ステージングでベンチと動作検証を行うこと。
