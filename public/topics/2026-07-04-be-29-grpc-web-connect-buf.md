# gRPC-Web 安定化と Connect・Buf 普及 — gRPC の開発体験が大幅改善

> トピック: 開発トピック｜出典: pockit.tools｜種別: backend

## 一行サマリー
gRPC-Web が安定化し、Connect や Buf といったツール群の普及によって、これまで扱いづらかった gRPC の開発体験（DX）が大幅に改善している。

## 🔰 初学者向け（何が・なぜ重要か）
サービス同士が通信する方法にはいくつか流派があります。gRPC は Google 発の高速な通信方式で、データを効率の良いバイナリ形式で送るため速いのですが、ブラウザから直接使いにくいという弱点がありました。

gRPC-Web は、その gRPC をブラウザからも使えるようにする橋渡しです。これが安定版になったことで、Web フロントから直接 gRPC サービスを呼びやすくなりました。

さらに Connect と Buf という道具が普及しています。Connect は gRPC と普通の HTTP の両方を 1 つのサーバーで扱える柔軟な仕組み、Buf は通信の設計図（Protocol Buffers）を安全に管理・生成するツールです。これらにより、速いけれど難しかった gRPC が「速くて扱いやすい」ものに変わってきました。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
gRPC-Web の安定化に加え、Connect（connect-go / connect-es 等）はブラウザ・gRPC・HTTP/JSON の複数プロトコルを単一エンドポイントで提供できます。Buf は proto の lint・破壊的変更検知・コード生成・スキーマレジストリ（BSR）を提供し、従来 `protoc` まわりで煩雑だった作業を大幅に簡素化しました（詳細は要確認）。

### 仕組み / 背景（なぜこう設計されたか）
素の gRPC は HTTP/2 のトレーラーやバイナリフレーミングに依存し、ブラウザやプロキシとの相性問題がありました。Connect はこれを緩和し、curl でも叩ける HTTP/JSON フォールバックを持たせています。Buf は proto 管理の属人性・破壊的変更事故を防ぐガードレールを提供し、スキーマ駆動開発を実務的にしました。

### 実務での使いどころ・移行の勘所
サービス間通信（マイクロサービス）や、型安全かつ高速なブラウザ↔サーバー通信に向きます。REST より高速・厳密で、GraphQL より単純な RPC が欲しい場合の中間解です。既存 gRPC 資産があるなら Connect / Buf の導入で DX を底上げでき、新規なら Buf を最初から採用すると proto 運用が安定します。

## 📖 用語解説（このトピックとのつながり）
- **gRPC-Web**: ブラウザから gRPC を使う橋渡し層。安定化が今回の要点。
- **Connect**: gRPC・HTTP/JSON を単一実装で扱える RPC フレームワーク。DX 改善の中心。
- **Buf**: Protocol Buffers の lint・生成・レジストリを担うツール。破壊的変更検知が強み。
- **Protocol Buffers (proto)**: 通信スキーマの定義形式。RPC の設計図にあたる。

## 影響範囲 / 推奨アクション
マイクロサービスや高速 API を設計する開発者に影響します。gRPC を検討する際は素の実装より Connect + Buf を前提にすると学習・運用コストが下がり、既存 gRPC プロジェクトも Buf 導入でスキーマ管理を安定化させるのが推奨です。
