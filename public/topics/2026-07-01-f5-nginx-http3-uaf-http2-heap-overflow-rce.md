# F5、NGINXの緊急パッチを臨時公開：HTTP/3のUse-After-Free等で無認証RCE/DoS（CVE-2026-42530, CVE-2026-42055）

> トピック: 脆弱性ニュース ｜ 出典: BleepingComputer / F5 K000161614 / SecurityWeek ｜ 種別: メモリ破壊（UAF / ヒープオーバーフロー）

## 一行サマリー

世界中のWebサーバ／リバースプロキシで使われるNGINXに、認証なしでワーカープロセスをクラッシュ（DoS）させ、条件次第でコード実行（RCE）に至る重大な欠陥2件（いずれもCVSS 9.2）が見つかり、F5が定例外の緊急パッチを公開した。HTTP/3とgRPC/HTTP2プロキシのメモリ管理不備が原因。

## 🔰 初学者向け（何が・なぜ問題か）

NGINX（エンジンエックス）は、Webサイトへのアクセスを最前線でさばく“受付係”のようなソフトです。世界中の膨大なサイトやサービスの入口に立っています。受付係が倒れると、その奥にあるサービス全体が止まってしまいます。

今回の弱点は、この受付係の「メモ帳の使い方のミス」に例えられます。受付係は来客の情報をメモ帳（メモリ）に書いて処理しますが、(1) **もう捨てたはずのメモをもう一度読みに行ってしまう**ミス（Use-After-Free＝解放済みメモリの再利用）と、(2) **メモ欄からはみ出して隣の欄まで書き込んでしまう**ミス（ヒープオーバーフロー）がありました。

攻撃者は、細工したリクエスト（来客）を送りつけるだけで――しかもログインも不要で――この混乱を意図的に引き起こせます。軽ければ受付係が気絶（サービス停止＝DoS）、悪くするとメモ帳の混乱に乗じて受付係を操り、サーバ上で攻撃者の命令を実行（RCE）させられる恐れがあります。入口が無防備だと中身全部が危ない、というのが怖いところです。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **CVE-2026-42530**（CVSS 9.2 / Critical）: `ngx_http_v3_module`（HTTP/3）における **Use-After-Free（CWE-416）**。無認証のリモート攻撃者がDoS、ASLR無効/回避可能な環境ではRCEに至りうる。
- **CVE-2026-42055**（CVSS 9.2 / Critical）: `ngx_http_proxy_v2_module` および `ngx_http_grpc_module` における **ヒープベースのバッファオーバーフロー（CWE-122）**。ワーカープロセス内でのヒープ破壊によりDoS／同条件下でRCE。
- **前提条件**: 該当モジュールが有効であること（HTTP/3を有効化、またはgRPC/HTTP2バックエンドへのプロキシ構成）。RCE成立にはASLR（アドレス空間配置のランダム化）が無効、または回避可能であることが条件で、ハードルは相応に高い一方、DoSは比較的容易。
- **影響製品**: NGINX Plus, NGINX Open Source, NGINX Gateway Fabric, NGINX Instance Manager, NGINX Ingress Controller。
- **影響バージョン（例）**: CVE-2026-42530 は NGINX OSS 1.31.0–1.31.1、NGINX Instance Manager 2.17.0–2.22.0、Gateway Fabric 1.3.0–1.6.2 / 2.0.0–2.6.3、Ingress Controller 3.x/4.x/5.x系。（詳細マトリクスはF5 K000161614 を要確認）
- **修正版**: NGINX OSS は mainline 1.31.2 / stable 1.30.3（1.30.3はCVE-2026-42055のみ修正、HTTP/3 UAFは1.31.x導入のため1.31.2で対応）。NGINX Plus は R37 P2（37.0.2.1）／ R36 P6。
- **追加の高深刻度**: CVE-2026-11311, CVE-2026-50107（Gateway Fabricで認証済み攻撃者が任意のNGINX設定ディレクティブを注入しうる）も同時修正。
- **悪用状況**: 公開時点で実環境での悪用報告は確認されていない（要継続確認）。臨時パッチ（out-of-band）であること自体が深刻度の高さを示す。

### メカニズム（攻撃が成立する仕組み）

※脆弱性クラスの一般原理を概念レベルで説明する。武器化手順・PoC・正確なオフセットは記載しない。

1. **入力の信頼境界**: NGINXのワーカープロセスは、外部から届く生のプロトコルデータ（HTTP/3のQUICストリーム、gRPC/HTTP2フレーム）を解析する最前線。ここがインターネットとの信頼境界であり、解析器の不備は致命的になりやすい。
2. **Use-After-Free（CVE-2026-42530）の原理**: HTTP/3処理では、ストリームやコネクションに紐づくメモリ構造体が動的に確保・解放される。あるオブジェクトを解放した後も、別の処理パスがそのポインタを保持・参照し続ける（ライフタイム管理の不整合）と、解放済み領域への読み書きが発生する。攻撃者が解放と再確保のタイミングを誘導すれば、解放領域に攻撃者制御のデータを再配置でき、本来とは異なる関数ポインタ/制御データを参照させる「制御フロー奪取」のプリミティブが得られうる。
3. **ヒープオーバーフロー（CVE-2026-42055）の原理**: プロキシ/gRPC処理で、入力長と確保済みバッファサイズの検証が不整合だと、ヒープ上のバッファ境界を越えた書き込みが起きる。隣接するヒープメタデータや別オブジェクトを上書きでき、これも制御データの改ざんプリミティブになりうる。
4. **DoS→RCEへの橋渡し**: いずれもまずはワーカープロセスのクラッシュ（DoS）として観測される。さらに攻撃者がヒープレイアウトを“整地”（heap grooming）し、ASLRを無効化/回避できる場合に限り、破壊したポインタを攻撃者が用意したコードへ向け、任意コード実行へ昇格する。ASLRが有効に効いている環境では難度が大きく上がる。
5. **被害の最終形**: 入口のNGINXでのRCEは、背後のアプリ・内部ネットワークへの侵入起点になる。プロキシ/Ingressとして使われる構成では特に横展開リスクが高い。

クラスとしては「プロトコル解析器におけるメモリ安全性の欠如（UAF / バッファオーバーフロー）」であり、C実装の高性能サーバが宿命的に抱えるリスク領域である。

### 検知・暫定緩和・恒久対策

- **恒久対策**: 上記修正版へアップデート。OSSは1.31.2（または1.30.3）、Plusは該当パッチ適用。
- **暫定緩和（パッチ前）**: (1) 不要なら**HTTP/3（QUIC, UDP 443）を無効化**しCVE-2026-42530の攻撃面を除去、(2) gRPC/HTTP2プロキシが不要な箇所は無効化、(3) ASLRが有効であることを確認（RCE難度を上げる）、(4) WAF/レート制限で異常なQUIC/HTTP2トラフィックを抑制。
- **検知の着眼点**: ワーカープロセスの異常終了・再起動の多発、コアダンプの発生、HTTP/3エンドポイントへの異常な接続数・異常フレーム、プロキシ処理での想定外のメモリ使用急増。

## 📖 用語解説（このニュースとのつながり）

- **NGINX**: Webサーバ／リバースプロキシ／ロードバランサ。インターネットの“入口”として広く使われ、今回の影響範囲が極めて広い理由。
- **Use-After-Free（UAF）**: 解放済みメモリを再び使ってしまう不具合。CVE-2026-42530の本体で、制御奪取に悪用されうる。
- **ヒープオーバーフロー**: 確保領域を越えた書き込み。CVE-2026-42055の本体。
- **HTTP/3 / QUIC**: UDP上で動く新しいHTTP。高速化の反面、新しい解析コードに新しいバグが入りやすい。本件はそのHTTP/3モジュールが舞台。
- **ASLR**: メモリ配置をランダム化し悪用を難しくする防御。今回RCE成立の可否を左右する鍵。
- **out-of-band（定例外）パッチ**: 通常サイクルを待たずに緊急公開されるパッチ。重大さの証左。

## 影響範囲 / 推奨対応

HTTP/3を有効化、またはgRPC/HTTP2バックエンドへプロキシしているNGINX（OSS/Plus/Gateway Fabric/Instance Manager/Ingress Controller）が対象。修正版へ更新する。即時更新が難しい場合は、不要なHTTP/3・gRPCモジュールを無効化し、ASLR有効化とトラフィック監視で攻撃面とリスクを下げる。エッジ機器のため優先度は高い。
