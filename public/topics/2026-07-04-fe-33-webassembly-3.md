# WebAssembly 3.0 — ブラウザで JS の数倍速い計算基盤へ

> トピック: 開発トピック｜出典: platform.uno｜種別: frontend

## 一行サマリー
WebAssembly（Wasm）3.0 が 2025-09 に公開され、Rust などを Wasm へコンパイルすると、計算負荷の高い処理でブラウザ上でも JavaScript の 8〜10 倍という報告のある高速実行が可能になった（倍率は処理内容依存・要確認）。

## 🔰 初学者向け（何が・なぜ重要か）
ブラウザで動く言語といえば長らく JavaScript 一択でした。JavaScript は柔軟で便利ですが、画像処理・暗号計算・物理シミュレーションのような「ひたすら数値を回す」重い処理は苦手です。

WebAssembly は「ブラウザが直接理解できる、機械語に近いコンパクトな命令セット」です。Rust や C++ で書いたコードをこの形式に変換すると、ブラウザ内でネイティブアプリに近い速度で走ります。たとえるなら、これまで通訳を介して喋っていた（JS）のを、共通の速記言語で直接やり取りする（Wasm）ようなもの。3.0 ではこの仕組みがさらに整理・拡張され、より多くの言語機能や大きなメモリを扱えるようになりました。ブラウザが「軽いページ表示装置」から「本格的な計算プラットフォーム」へ進化する流れの中心にあります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Wasm 3.0（2025-09 公開）は仕様として GC（ガベージコレクション）、参照型、例外処理、複数メモリ、64bit メモリ（memory64）などの拡張を取り込む方向で成熟してきた（各機能の Baseline 状況は「要確認」）。これにより GC を持つ言語（Java/Kotlin/C# など）を Wasm へ載せやすくなり、Rust→Wasm の計算集約処理は JS 比で数倍〜十倍規模の報告がある。ただし DOM 操作を伴う処理は JS 境界のコストが乗るため、常に速いわけではない。

### 仕組み / 背景（なぜこう設計されたか）
JS は動的型付け・JIT 最適化に依存するため、重い数値計算では最適化が読みにくく揺らぐ。Wasm は事前コンパイル済みの静的なバイトコードで、予測可能な高性能を出せる。GC 対応が入ったのは、これまで線形メモリを自前管理する言語しか快適に載らなかった制約を外し、対応言語を広げるため。トレードオフは、DOM/Web API へのアクセスは JS 経由で境界越えコストが発生する点。

### 実務での使いどころ・移行の勘所
画像・動画処理、暗号、ゲーム物理、CAD、そして後述の Transformers.js のようなブラウザ内 ML など「重い計算をクライアントで完結させたい」用途が本命。UI 全体を Wasm 化する必要はなく、ホットスポットだけを Rust/Wasm に切り出すのが現実的。ビルド成果物のサイズと初回ロードコストは要計測。

## 📖 用語解説（このトピックとのつながり）
- **WebAssembly（Wasm）**: ブラウザ・エッジで動く低レベルバイナリ命令形式。本トピックの主役。
- **Rust→Wasm**: メモリ安全な Rust を Wasm へコンパイルする定番ルート。高速実行の代表例。
- **memory64**: 4GB超のメモリを扱えるようにする拡張。大規模データ処理を可能に。
- **GC 提案**: Wasm に組み込みガベージコレクションを持たせる仕様。対応言語を拡大。
- **JS 境界コスト**: Wasm と JS/DOM を跨ぐ際に生じるオーバーヘッド。速度に影響。

## 影響範囲 / 推奨アクション
計算集約な機能を持つフロント開発者・ML をクライアントで動かしたいチームが対象。まず重い処理をプロファイルで特定し、その部分だけ Rust/Wasm 化を PoC。バンドルサイズと初回ロードのトレードオフを計測してから本採用する。
