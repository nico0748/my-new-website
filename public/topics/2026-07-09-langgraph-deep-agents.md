# LangGraph が LangChain エージェントの標準ランタイムに — Deep Agents と DeltaChannel

> トピック: LangGraph / Deep Agents｜出典: LangChain 公式｜種別: エージェント基盤の更新

## 一行サマリー

LangGraph は 2025年末に v1.0 到達後 LangChain 全エージェントの既定ランタイムとなり、2026 Q2 でノード単位タイムアウト・DeltaChannel（チェックポイント保存を大幅圧縮）・v2 ストリーミングを追加、さらに計画やサブエージェント管理を標準装備する上位フレームワーク Deep Agents を提供する。

## 🔰 初学者向け（何が・なぜ重要か）

AI エージェントとは「目標を渡すと、自分で手順を考え、道具を使いながら作業を進める AI」です。たとえば「この資料を調べて要約して」と頼むと、検索し、読み、整理して答えを返す――そういう一連の流れを自動でこなします。

問題は、この「一連の流れ」を安定して作るのが難しいこと。途中で脱線したり、失敗したときにやり直せなかったり、どこまで進んだか分からなくなったりします。**LangGraph** は、この作業手順を「グラフ（作業の地図）」として明示的に組み立てる道具です。各ステップをノード（作業マス）として置き、条件分岐やループ、途中経過の保存（チェックポイント）を設計できます。地図があるので、途中で止めて再開したり、失敗した所まで巻き戻したりできるのが強みです。

今回の話題は、この LangGraph が LangChain（AI アプリ作りの定番ライブラリ）の「標準の実行エンジン」になったこと、そして初心者でも扱いやすいよう計画やサブ作業の管理を最初から備えた **Deep Agents** が登場したことです。要するに「本格的な AI エージェントを、より安全に・作りやすく」する流れが進んでいます。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- LangGraph は 2025 年末に **v1.0** 到達、LangChain 全エージェントの既定ランタイムに。2026年6月2日に 1.2.4 をリリースするなど活発。
- **2026 Q2**: ノード単位タイムアウト、**DeltaChannel**、v2 ストリーミングを追加。DeltaChannel はスケール時のチェックポイント保存量を最大73,000分の1に圧縮すると報告。
- 2026 前半に GitHub スター数で CrewAI を上回るなど、エンタープライズ採用が牽引。
- **Deep Agents**: LangGraph 上に構築された上位フレーム。計画（planning）、サブエージェント管理、ファイルシステム機能を標準提供し、低レベル制御が不要な開発者にも扱いやすく。
- 位置づけ: LangChain は単純パイプライン/試作、LangGraph は永続メモリ・条件分岐・複雑なマルチエージェント連携が要る本番向け。

### 仕組み / 背景（なぜこう設計されたか）

素朴なエージェントは「1つの while ループ（考える→ツールを呼ぶ→結果を見る→繰り返す）」で実装される。試作は簡単だが、本番要件（監査ログ、途中再開、ロールバック、並列サブタスク、人間の承認割り込み）を満たそうとすると、ループ内に状態管理と分岐が絡まり、保守不能になりやすい。LangGraph はこれを「状態機械／有向グラフ」として外在化する。ノード＝処理単位、エッジ＝遷移条件、共有状態＝チャンネルとして定義し、実行系がステップごとに状態をチェックポイントとして永続化する。これにより「任意ステップから再開」「失敗ノードのリトライ」「監査のための実行履歴」が構造的に手に入る。

DeltaChannel はこの永続化のコスト問題への解だ。全状態をステップごとに丸ごと保存すると、長い実行・大きな状態でストレージが爆発する。DeltaChannel は状態の「差分（デルタ）」だけを記録することで保存量を桁違いに圧縮する（＝バージョン管理の差分コミットに近い発想）。ノード単位タイムアウトは、外部ツールの遅延で全体が固まるのを防ぐための粒度の細かい制御。Deep Agents は逆方向の抽象化で、「計画→サブエージェントへ委譲→作業用ファイルシステムで中間成果を管理」という頻出パターンをテンプレ化し、グラフを手書きせずとも堅牢なエージェントを組めるようにする。トレードオフは、グラフ設計の学習コストと、抽象化（Deep Agents）に乗るほど内部制御が見えにくくなる点。

### 実務での使いどころ・移行の勘所

長時間・多段の自律タスク（リサーチ、コード生成の反復、業務プロセス自動化）、監査証跡や再開性が要る本番エージェント、複数エージェントの協調が必要な場面で LangGraph が適する。素の LangChain で試作 → 本番要件（再開・監査・分岐）が見えた段階で LangGraph へ移行、という段階設計が現実的。導入時は (1) チェックポイントの保存先（DB）と DeltaChannel 設定、(2) ノードタイムアウトと失敗時リトライ方針、(3) 人間承認の割り込み点の設計を先に決めたい。低レベル制御が不要なら Deep Agents で素早く立ち上げ、要件が尖ったら生の LangGraph へ降りる。競合（CrewAI, AutoGen, Semantic Kernel）との比較では、グラフによる明示的制御と本番運用機能が LangGraph の差別化点。

## 📖 用語解説（このトピックとのつながり）

- **AI エージェント**: ツールを使い自律的に手順を進める AI。LangGraph はその実行基盤。
- **オーケストレーション**: 複数ステップ/エージェントの流れを制御すること。LangGraph の役割。
- **チェックポイント**: 実行途中の状態を保存する仕組み。再開・巻き戻し・監査を可能にする。
- **DeltaChannel**: 状態の差分だけ保存して容量を圧縮する機構。長時間実行のコストを下げる。
- **Deep Agents**: 計画・サブエージェント・作業ファイルを標準装備した上位フレーム。導入のハードルを下げる。

## 影響範囲 / 推奨アクション

- 影響: LangChain/LangGraph で本番エージェントを組むチーム、複雑な自律ワークフローを扱うプロダクト。
- 推奨: 試作は LangChain、本番要件が見えたら LangGraph へ。チェックポイント保存先・DeltaChannel・ノードタイムアウト・人間承認点を設計。素早い立ち上げは Deep Agents を活用。
