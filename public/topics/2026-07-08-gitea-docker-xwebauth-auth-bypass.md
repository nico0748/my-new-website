# Gitea Docker イメージの初期設定で認証バイパス（CVE-2026-20896）— ヘッダ1つで管理者になりすまし

> トピック: 脆弱性ニュース ｜ 出典: The Hacker News / SecurityWeek / Sysdig ｜ 種別: 認証バイパス（なりすまし）

## 一行サマリー

セルフホスト型Git基盤 Gitea の Docker イメージが危険な初期値でリバースプロキシ認証を信頼していたため、攻撃者が `X-WEBAUTH-USER` ヘッダを付けるだけで任意ユーザー（管理者含む）になりすませる欠陥（CVE-2026-20896, CVSS 9.8）。公開13日後から実際の探索・悪用が観測されている。

## 🔰 初学者向け（何が・なぜ問題か）

Gitea は、会社やチームが自前で運用する「ソースコードの保管庫（GitHub の自前版）」です。多くの場合、前段に"受付係（リバースプロキシ）"を置き、そこでログイン認証を済ませてから「この人は○○さんです」と Gitea に伝える構成をとります。

このとき受付係は、`X-WEBAUTH-USER` という**付箋（HTTPヘッダ）**に「利用者は○○」と書いて Gitea に渡します。Gitea 側は「受付係が確認済みなら信じてよい」という前提で、その付箋を鵜呑みにします。

問題は、Gitea の Docker 版が**「どの受付係からの付箋でも信じる」という危険な初期設定**（`REVERSE_PROXY_TRUSTED_PROXIES = *`）で出荷されていたことです。本来は「玄関ロビーの受付係（localhost）からの付箋だけ信じる」べきなのに、"世界中の誰が書いた付箋でも信じる"状態でした。

結果、攻撃者は自分で「利用者は admin」と書いた偽の付箋を貼るだけで、パスワードもトークンも無しに管理者になりすませます。受付を通さず、偽の名札を自分で書いて堂々と入るイメージです。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **脆弱性種別**: 認証バイパス（CWE-290: なりすましによる認証回避 / 信頼できない入力への過信）。安全でないデフォルト設定（CWE-1188）が根本要因。
- **CVSS**: 9.8（Critical）。
- **前提条件**: Gitea 側で**リバースプロキシ認証（reverse proxy auth）が有効**であること。Docker 既定の `REVERSE_PROXY_TRUSTED_PROXIES = *` のまま運用していると成立。
- **影響バージョン**: Gitea Docker イメージ v1.26.2 以前。
- **修正版**: v1.26.3 以降。加えて `REVERSE_PROXY_TRUSTED_PROXIES` を `127.0.0.0/8,::1/128` に明示。
- **悪用状況**: Sysdig が公開13日後に実環境での悪用試行を初観測。ProtonVPN 出口ノード由来の自動スキャン（ポート探索＋ヘッダ注入による認証回避試行）。

### メカニズム（攻撃が成立する仕組み）

以下は概念レベルの説明であり、武器化手順やPoCは示さない。

1. **信頼境界の誤設定**: 本来の信頼境界は「リバースプロキシ（localhost）→ Gitea」。ここでは"プロキシは信頼できる主体"という前提で、プロキシが付与するユーザー識別ヘッダを認証済みとみなす。
2. **不備の所在**: 「どの送信元をプロキシとして信頼するか」を決める `REVERSE_PROXY_TRUSTED_PROXIES` が Docker 既定で `*`（全許可）。つまり**信頼境界が事実上消滅**し、インターネット上の任意クライアントが"プロキシのふり"をできる。
3. **攻撃者が握るプリミティブ**: 攻撃者は自作の HTTP リクエストに `X-WEBAUTH-USER: gitea_admin`（任意ユーザー名）を付与できる。Gitea はこのヘッダを"信頼できるプロキシからの確定情報"として扱う。
4. **橋渡し（認証ロジックの短絡）**: パスワード・トークン・多要素のいずれも検証されないまま、ヘッダ値＝ログインユーザーとして確定。認証という関門が"入力を信じるだけ"の処理に短絡される。
5. **最終被害**: 管理者アカウントへの即時なりすまし。リポジトリの窃取・改ざん、CI/CD の乗っ取り、サプライチェーン汚染へ発展しうる。
- **一般原理**: 「ネットワーク位置（送信元）で信頼を判断する設計」は、送信元を偽れる環境では破綻する。ヘッダ由来のIDは**必ず信頼できる経路からのみ**受理し、既定は"最小信頼（localhostのみ）"であるべき、という設計教訓。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: 外部送信元から到達した `X-WEBAUTH-USER` を含むリクエスト、リバースプロキシを経由しない直アクセスでの認証成功、`admin`/`gitea_admin` の想定外ログイン。設定値 `REVERSE_PROXY_TRUSTED_PROXIES` の実値監査。
- **暫定緩和（パッチ前）**: `REVERSE_PROXY_TRUSTED_PROXIES` を `127.0.0.0/8,::1/128` に明示。前段プロキシで受信 `X-WEBAUTH-USER` を必ず削除・再設定。Gitea を外部へ直接公開しない。
- **恒久対策**: v1.26.3 以降へ更新し、上記の信頼プロキシ設定を明示。既存アカウント・アクセストークンを棚卸しし不審分を失効。
- **運用示唆**: "コンテナの初期値は安全とは限らない"。特に認証関連の環境変数は導入時に必ず明示設定し、既定値依存を避ける。

## 📖 用語解説（このニュースとのつながり）

- **リバースプロキシ認証**: 前段のプロキシで認証を済ませ、その結果をヘッダでアプリに渡す方式。便利だが「誰の付箋を信じるか」を厳密に絞らないと本件のようになりすましを許す。
- **X-WEBAUTH-USER**: Gitea がプロキシ認証済みユーザー名を受け取るヘッダ。本来は内部専用だが、外部から偽装可能だとそのまま認証回避の武器になる。
- **REVERSE_PROXY_TRUSTED_PROXIES**: 「どの送信元をプロキシとして信頼するか」を決める設定。既定 `*`（全許可）が事故の主因で、安全値は `127.0.0.0/8,::1/128`。
- **安全でないデフォルト（Insecure Default）**: 導入直後の初期値が危険な状態。利用者が明示的に締めないと脆弱なまま、という本件の構造的問題。
- **サプライチェーン汚染**: コード保管庫やCI/CDを乗っ取り、正規の成果物に悪性コードを混ぜる攻撃。Gitea 管理者を奪われた場合の最悪シナリオ。

## 影響範囲 / 推奨対応

- **影響範囲**: リバースプロキシ認証を有効化した Gitea Docker v1.26.2 以前。特に既定設定のままインターネット公開している個体。
- **推奨対応**: (1) v1.26.3 以降へ更新、(2) `REVERSE_PROXY_TRUSTED_PROXIES` を localhost に限定、(3) 前段プロキシで外部由来の `X-WEBAUTH-USER` を除去、(4) 管理者・トークンの棚卸しと不審ログイン調査。
