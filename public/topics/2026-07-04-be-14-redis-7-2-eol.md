# Redis 7.2 が EOL（2026-02-28）— 移行判断を迫るサポート終了

> トピック: 開発トピック｜出典: devnewsletter（State of Databases 2026）｜種別: backend

## 一行サマリー
Redis 7.2 が 2026-02-28 にサポート終了（EOL）を迎え、セキュリティ修正が受けられなくなった。7.2 を本番で使う組織は 8.x 系または Valkey への移行を計画する必要がある。

## 🔰 初学者向け（何が・なぜ重要か）
ソフトウェアには「サポート期間」があり、その期間が終わることを EOL（End of Life、寿命の終わり）と呼びます。EOL を過ぎると、たとえ重大なセキュリティ欠陥が見つかっても公式の修正パッチが出ません。つまり、使い続けるほど危険が積み上がります。

Redis 7.2 は 2026-02-28 にこの EOL を迎えました。動作自体は続きますが、新しい脆弱性が放置されるため、本番環境で使い続けるのはリスクです。

たとえるなら、メーカー保証が切れた車にそのまま乗り続けるようなもの。普段は問題なく走っても、いざ壊れたときに正規の修理（=セキュリティ修正）が受けられません。だからこそ、保証のある新しいバージョンへの乗り換えを早めに計画すべきです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Redis 7.2 系への公式サポート（バグ・セキュリティ修正）が 2026-02-28 で終了した。以降は 8.x 系が現行サポート対象で、8.2 では実行速度・スループット・メモリ効率の改善も入っている。正確なEOL日程やサポートポリシーは公式のライフサイクル表で最終確認すること（要確認）。

### 仕組み / 背景（なぜこう設計されたか）
DB や言語ランタイムには明確なライフサイクルが設定され、EOL 後は修正を打ち切ることで開発リソースを新版に集中させる。利用者側はこのスケジュールを前提に、計画的なアップグレードを織り込む必要がある。Redis のライセンス変更が並行して起きたため、単純な版上げに加え「Redis 8.x に上げるか、Valkey へ移るか」という二択の判断も同時に発生している。

### 実務での使いどころ・移行の勘所
まず自組織で 7.2 系が動いていないか棚卸しする。稼働中なら、（1）Redis 8.x への版上げ、（2）Valkey への移行、のどちらかを選ぶ。互換性は高いが、モジュール依存やライセンス方針が判断材料になる。移行前にステージングで機能・性能を検証し、レプリケーションを使った無停止に近い切替手順を用意すると安全。

## 📖 用語解説（このトピックとのつながり）
- **EOL（End of Life）**: サポート終了。7.2 が該当し、セキュリティ修正が止まる。
- **セキュリティパッチ**: 脆弱性を塞ぐ修正。EOL 後は提供されないため放置リスクが増す。
- **ライフサイクル表**: 各版のサポート期限一覧。移行計画の根拠になる。
- **Valkey**: Redis フォークのOSS。8.x への版上げと並ぶ移行先候補。

## 影響範囲 / 推奨アクション
Redis 7.2 を本番で使う全組織が対象。至急バージョンを棚卸しし、8.x 系か Valkey への移行計画を立てて実行すること。修正が止まった状態で放置しないのが最優先。
