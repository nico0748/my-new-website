# Exim メールサーバの UAF 脆弱性 "Dead.Letter"(CVE-2026-45185)で認証不要 RCE

> トピック: 脆弱性ニュース ｜ 出典: BleepingComputer / Exim ｜ 種別: 解放後使用(Use-After-Free, CWE-416) による RCE

## 一行サマリー

Exim の GnuTLS ビルドで STARTTLS と CHUNKING を有効にした構成に、認証不要でリモートコード実行できる Use-After-Free 脆弱性 CVE-2026-45185("Dead.Letter", CVSS 9.8)が判明。TLS 切断時に解放済みのバッファへ書き込む不具合で、4.99.3 で修正済み。

## 🔰 初学者向け（何が・なぜ問題か）

Exim は、インターネット上でメールを配送する「郵便局の仕分け係」にあたるソフト(MTA=メール転送エージェント)で、世界のメールサーバーの大きなシェアを占めます。表からアクセスできる位置に置かれることが多く、ここが破られると被害が大きくなります。

今回の問題は「メモリの後片付けミス」です。プログラムは作業用のメモリを借りて使い、終わったら返却(解放)します。イメージとしては、貸しロッカーを使い終えて鍵を返したのに、返したことを忘れて後からまたそのロッカーを開けて物を出し入れするようなものです。すでに他人に貸し出されたかもしれないロッカーを触るので、中身がぐちゃぐちゃになったり、悪意ある人がそのロッカーに罠を仕込んで乗っ取ったりできます。この「解放したのに使い続ける」バグを Use-After-Free(解放後使用)と呼びます。

CVE-2026-45185 は、暗号化通信(TLS)を終える瞬間に、Exim が転送用のメモリを解放したのに、その後も古い参照を使って書き込んでしまう点が原因です。攻撃者は**ログイン不要**で、細工したメール通信を送るだけでこれを突けるため、成功するとサーバー上で任意のコマンドが実行され、メール本体の窃取や社内への侵入の踏み台化に至ります。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **CVE**: CVE-2026-45185(通称 "Dead.Letter") / **CVSS**: 9.8(要確認)
- **種別(CWE)**: CWE-416 Use-After-Free(解放後使用)
- **前提条件**: **GnuTLS でコンパイルされた** Exim ビルドで、STARTTLS と CHUNKING(BDAT) の両方がアドバタイズされている構成。**OpenSSL ビルドは影響を受けない**。認証は不要(unauthenticated)。
- **影響バージョン**: Exim 4.97 〜 4.99.2(GnuTLS ビルド)。
- **修正版**: Exim 4.99.3。
- **悪用状況**: 発見・報告は XBOW の研究者 Federico Kirschbaum。本記事時点で広範な実環境悪用の断定情報はなく、CVSS の高さと認証不要性から**優先度は最高クラス**。実悪用の有無は**要確認**。

### メカニズム（攻撃が成立する仕組み）

TLS 終了処理と BDAT チャンク処理が交差する箇所での UAF、という公表性質に基づき、成立の流れを概念レベルで説明する(武器化手順・オフセット・PoC は記載しない)。

1. **前提のプロトコル状態**: SMTP の CHUNKING 拡張(BDAT)は、メール本文をサイズ指定の「チャンク」で送る方式。STARTTLS により以降の通信は TLS で暗号化される。攻撃者はこの 2 機能が有効な構成へ接続する。
2. **バッファのライフサイクルの取り違え(信頼境界ではなく状態管理の不備)**: TLS シャットダウン時に、Exim は BDAT のデータ転送に使っていた TLS 転送バッファを解放する。ところがコールバック/参照側はそのバッファがまだ生きている前提で処理を続ける。ここに「解放」と「使用」の順序矛盾(ライフサイクル不整合)が生じる。
3. **攻撃者が握るプリミティブ**: 解放済み領域への書き込み(dangling ポインタ経由の write)。UAF では、解放後にその領域が別用途で再確保(reallocation)されると、攻撃者データで解放済みオブジェクトの中身(関数ポインタや構造体フィールド)を上書きできる可能性が生まれる。
4. **制御奪取への橋渡し**: 上書きされた参照が後段で使用される際、攻撃者の意図した先へ制御が移る。ヒープの状態を送信データ量やタイミングで整える(ヒープグルーミングの一般原理)ことで、解放→再確保→使用の窓を安定させ、任意コード実行へつなげうる。
5. **クラスとしての位置づけ**: 本件はアクセス制御やロジックの欠陥ではなく、**メモリ安全性(temporal safety)** の破れ。TLS 終了という「正常系のイベント」が引き金になる点が厄介で、特別な認証や権限を要さない。

> 注: 上記は UAF の一般原理に基づく概念説明。CVE-2026-45185 固有の内部データ構造・悪用可能性の詳細は公表が限られるため、具体的な武器化可否は断定しない。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: Exim の panic/main ログでの異常終了・再起動の増加、TLS ハンドシェイク/シャットダウン周辺のクラッシュ、`exim` プロセスの想定外の子プロセス生成。コアダンプが出る環境ではクラッシュの再現性を確認。
- **暫定緩和(パッチ前)**: 直ちに更新できない場合、GnuTLS ビルドでの CHUNKING(BDAT) のアドバタイズを無効化する、あるいは OpenSSL ビルドへ切り替える構成変更を検討(自環境の互換性確認の上で)。境界での送信元制限も有効。
- **恒久対策**: Exim 4.99.3 以降へアップデート。ディストリのバックポート修正が出ている場合はそれを適用。
- **運用示唆**: 自ビルドの Exim がどの TLS ライブラリでコンパイルされているか(`exim -bV` の出力等)を把握しておくと、影響判定が速い。

## 📖 用語解説（このニュースとのつながり）

- **MTA(メール転送エージェント)**: メールを別サーバーへ配送する中継役。Exim は代表的な MTA で、今回の被害面。
- **Use-After-Free(UAF)**: 解放済みメモリを使い続けるバグ。今回の根本原因で、任意書き込み/制御奪取の起点。
- **STARTTLS / CHUNKING(BDAT)**: それぞれ通信の暗号化開始と本文分割送信の SMTP 拡張。両方有効な構成が発火条件。
- **GnuTLS / OpenSSL**: TLS を実装するライブラリ。今回は GnuTLS ビルドのみ影響し、ライブラリ選択が影響有無を分ける。
- **ヒープグルーミング**: ヒープメモリの配置を意図的に整え、UAF の再確保を狙い通りにする手法。悪用安定化の一般原理。

## 影響範囲 / 推奨対応

GnuTLS でビルドされ、STARTTLS と CHUNKING を有効化した Exim 4.97〜4.99.2 を運用する組織が対象。認証不要かつ外部到達性の高いサービスであるため、優先度は最上位。まずビルドの TLS ライブラリと有効拡張を確認し、4.99.3(またはディストリのバックポート)へ更新。即時更新が難しければ CHUNKING の無効化や OpenSSL ビルドへの切替を暫定策とし、ログ監視を強化する。
