# 分散システムの可観測性 — Kafka × OpenTelemetry によるトレース連携

> トピック: 開発トピック｜出典: oneuptime｜種別: backend

## 一行サマリー
非同期メッセージング（Apache Kafka）を含む分散システムでも、OpenTelemetry でトレースコンテキストを伝播させ、プロデューサからコンシューマまでを1本のトレースとしてつなぐことが可観測性の要になっている。

## 🔰 初学者向け（何が・なぜ重要か）
Kafka は、サービス同士が直接会話せず「メッセージを箱に入れて後で取り出す」ことで疎結合を実現する仕組みだ。手紙をポストに投函し、相手が都合の良いときに受け取るのに似ている。便利だが、送り手と受け手が時間的に切り離されるため、「どのリクエストがどのメッセージを生み、どこで処理されたか」を追いにくい。

同期的な API 呼び出しなら追跡は比較的簡単だが、Kafka を挟むと「手紙をポストに入れた時点で追跡が途切れる」問題が起きる。可観測性の観点では、この非同期の切れ目をどうつなぐかが最大の課題になる。

OpenTelemetry は、メッセージのヘッダーに「追跡番号（trace context）」を埋め込むことで、投函〜受け取り〜後続処理までを1本の追跡ラインとして復元する。手紙の封筒に追跡番号を書いておき、受け取った側がそれを引き継ぐイメージだ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
OTel の messaging セマンティック規約に沿い、プロデューサ側で trace context（W3C traceparent 等）を Kafka メッセージヘッダーに注入し、コンシューマ側で抽出して span をリンクする。これによりプロデュース→コンシューム→下流処理をまたぐエンドツーエンドのトレースが得られる。対応ライブラリ・自動計測の範囲は言語・クライアントにより異なり要確認。

### 仕組み / 背景（なぜこう設計されたか）
分散トレースは通常「親 span → 子 span」の親子関係で構築されるが、非同期メッセージングでは呼び出しの時間的連続性が切れる。そこで OTel は span link（因果はあるが親子ではない関係）や context の明示的伝播を用意した。メッセージヘッダーはこの伝播の運搬手段として理想的で、ブローカを改変せずにコンテキストを運べる。

### 実務での使いどころ・移行の勘所
イベント駆動アーキテクチャや CQRS/Saga を採る場合は必須級。プロデューサ・コンシューマ双方で計測ライブラリのバージョンとヘッダー伝播の互換を揃えること。バッチ消費やファンアウトでは span link を使い、1メッセージ=1トレースの単純化に固執しない。サンプリング判断はメッセージ生成元で行い、下流に伝播させると一貫性が保てる。

## 📖 用語解説（このトピックとのつながり）
- **Apache Kafka**: 分散イベントストリーミング基盤。非同期連携の代表格。
- **trace context（traceparent）**: トレースを識別・伝播する情報。非同期の切れ目をつなぐ鍵。
- **span link**: 親子でない因果関係の表現。ファンアウト/バッチ処理に有効。
- **messaging セマンティック規約**: メッセージング計測の OTel 標準。相互運用の前提。

## 影響範囲 / 推奨アクション
イベント駆動 / ストリーミングを扱うバックエンド開発者・SRE が対象。Kafka の前後でトレースが途切れていないか点検し、ヘッダー伝播と span link を導入して一貫トレースを確立すべき。
