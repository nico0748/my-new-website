# React Compiler 1.0 が安定化 — 自動メモ化が標準に

> トピック: 開発トピック｜出典: dev.to｜種別: frontend

## 一行サマリー
React Compiler が 1.0 として安定化し、コンポーネントの再描画最適化を自動で行うようになった。これにより `useMemo` / `useCallback` を手書きする必要が原則なくなり、パフォーマンスチューニングのコストが大きく下がる。

## 🔰 初学者向け（何が・なぜ重要か）
React は「状態（データ）が変わると画面を描き直す」仕組みで動いています。ただし、親が描き直されると子も巻き添えで描き直されるため、不要な再描画がパフォーマンス低下を招くことがありました。これを防ぐために、開発者は「この値は変わっていないから計算し直さないで」と手動で印を付ける道具、`useMemo` や `useCallback` を使ってきました。

しかしこれは面倒で、付け忘れれば遅くなり、付けすぎればコードが読みにくくなる、いわば「手作業の交通整理」でした。React Compiler は、この交通整理をビルド時に機械が自動で行ってくれる仕組みです。書いたコードを解析し、「どこを覚えておけば無駄がなくなるか」を自動判定してメモ化コードを挿入します。

つまり、開発者は最適化を意識せず素直にコンポーネントを書くだけでよく、速さはコンパイラが担保してくれる、というのが今回の大きな変化です。手動チューニングという「Reactの通過儀礼」が一つ消えたと考えると分かりやすいでしょう。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
まとめ記事によれば React Compiler は 2025-10 に 1.0 相当で安定化したとされる（正確なリリース時期・バージョン番号は公式リリースノートで要確認）。ビルド時に JSX/関数コンポーネントを静的解析し、再計算が不要な値・関数・要素に対して自動でメモ化を適用する。結果として `useMemo` / `useCallback` の手書きは原則不要になる。Next.js など主要フレームワークもコンパイラ統合を進めている。

### 仕組み / 背景（なぜこう設計されたか）
従来のメモ化は「参照の同一性」を人間が管理する必要があり、依存配列の書き間違いがバグ・性能劣化の温床だった。React Compiler は React の規約（Rules of React、副作用の純粋性など）を前提に依存関係を推論し、安全にメモ化を差し込む。トレードオフとして、規約から外れた「不純」なコンポーネント（レンダー中の副作用など）ではコンパイラが最適化を諦める（bail out する）ことがあり、その挙動を理解しないと期待した高速化が得られない場合がある。

### 実務での使いどころ・移行の勘所
新規プロジェクトでは最初から有効化するのが素直。既存プロジェクトでは、まず ESLint プラグイン（React の規約違反検出）でコードを規約準拠にしてから導入すると効果が出やすい。手書きの `useMemo` を一斉削除する必要はなく、段階的に外していける。ライブラリ作者は、コンパイラ有効・無効の両環境で動くことを確認する必要がある。

## 📖 用語解説（このトピックとのつながり）
- **メモ化（memoization）**: 計算結果を覚えておき、入力が同じなら再計算を省く手法。React Compiler はこれを自動化する。
- **`useMemo` / `useCallback`**: 値・関数の参照を保持する React フック。コンパイラ導入後は手書きが原則不要になる。
- **再描画（re-render）**: 状態変化に応じてコンポーネント関数が再実行されること。無駄な再描画の抑制が最適化の主眼。
- **Rules of React（規約）**: レンダーの純粋性など、コンパイラが安全に最適化するための前提条件。
- **bail out**: コンパイラが最適化を適用できないと判断して見送ること。

## 影響範囲 / 推奨アクション
React を使う全開発者に影響する。新規は有効化を前提に、既存は ESLint 準拠→段階導入が推奨。手動メモ化の削減でコードが簡潔になる一方、規約違反コードは効果が出にくいため、規約チェックの整備を先に行うとよい。
