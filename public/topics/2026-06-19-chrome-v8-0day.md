# Chrome V8 ゼロデイ CVE-2026-11645 — Web閲覧だけでコード実行、実環境で悪用中

> トピック: 脆弱性ニュース｜出典: The Hacker News / Google Chrome Releases / CISA KEV｜種別: ブラウザ・ゼロデイ（OOB読み書き）

## 一行サマリー

Google Chrome の JavaScript エンジン「V8」に境界外読み書き（CVSS 8.8）のゼロデイ CVE-2026-11645 が見つかり、悪意あるWebページを開くだけで攻撃が成立する。すでに実環境で悪用が確認され、CISA KEV にも登録済みのため、Chrome を最新版（149.0.7827.102/.103 以降）へ即時更新すべき。

## 🔰 初学者向け（何が・なぜ問題か）

ブラウザの中には「V8（ブイエイト）」という、Webページの JavaScript（ページを動かすプログラム）を高速で動かすエンジンが入っています。料理に例えると、V8 はレシピ（JavaScript）を受け取って猛スピードで調理する優秀なシェフのようなものです。

今回の問題は、このシェフが「冷蔵庫のこの棚まで」と決められた範囲を超えて、隣の家の冷蔵庫にまで手を伸ばして食材を取ったり置いたりできてしまう欠陥（=境界外読み書き、メモリの本来触ってはいけない領域への読み書き）です。攻撃者はこの「はみ出し」を利用してブラウザの内部データを書き換え、最終的に自分の用意したプログラムを動かせてしまいます。

怖いのは、被害者がやることは「攻撃者が用意したWebページを開くだけ」という点です。怪しいファイルをダウンロードしたり、パスワードを入力したりする必要はありません。リンクをクリックして開いた瞬間に攻撃が始まります。すでに現実の攻撃で使われている「ゼロデイ（修正が出回る前から悪用されている穴）」なので、放置は危険です。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **CVE**: CVE-2026-11645 / **CVSS**: 8.8（High、要確認: ベンダーによりスコア表記が前後する場合あり）
- **CWE**: CWE-787（Out-of-bounds Write）/ CWE-125（Out-of-bounds Read）相当
- **影響コンポーネント**: Chromium V8 エンジンの JIT コンパイラ「TurboFan」の最適化処理
- **根本原因**: JIT コンパイル時の「bounds-check elimination（境界チェック削除最適化）」が誤って働き、本来残すべき配列境界チェックを除去してしまう。結果としてレンダラプロセス内で境界外の読み取り・書き込みプリミティブが得られる。
- **悪用状況**: Google が「exploit が実環境に存在する」と認定。CISA KEV 登録済み（2026-06-09 追加分）。報告者 "303f06e3" が 2026-04-27 に報告、$55,000 の報奨。
- **前提条件**: 被害者が攻撃者の制御下にあるWebページを Chrome（脆弱版）で開くこと。認証不要・ユーザー操作はページ閲覧のみ。
- **修正版**: Chrome 149.0.7827.102/.103（Windows/macOS）、149.0.7827.102（Linux）以降。Chromium ベースの Edge / Brave / Vivaldi 等も追従更新が必要。

### メカニズム（攻撃が成立する仕組み）

※以下は脆弱性クラスの一般原理に基づく概念説明であり、武器化手順・PoC・正確なオフセットは記載しない。

1. **JIT最適化の前提が崩れる**: TurboFan は実行時の型情報をもとに「この配列アクセスは常に範囲内」と推論し、高速化のため境界チェックを削除する。この推論の前提（信頼境界）が、特定の操作列によって後から覆ると、チェックなしのアクセスだけが残る。
2. **境界外プリミティブの獲得**: チェックを失った配列アクセスを通じ、攻撃者は「狙った相対位置のメモリを読む／書く」操作（OOB read/write プリミティブ）を JavaScript から間接的に行えるようになる。これは任意アドレス読み書きへ発展しうる強力な原始操作。
3. **内部構造の破壊と制御奪取**: 隣接する V8 内部オブジェクト（配列のlength やオブジェクトのメタデータ）を書き換え、メモリレイアウトを攻撃者有利に整える。これにより「任意アドレスの読み書き」を安定化し、最終的に実行フローをハイジャックしてレンダラプロセス内でコード実行に到達する。
4. **サンドボックス突破は別途必要**: 得られるのはサンドボックス化されたレンダラ内の実行権限まで。完全なシステム侵害には、OSカーネルの欠陥やブラウザIPCの脆弱性など「二次脆弱性」との連鎖（チェイン）が前提となる。
   - 脆弱性クラスの一般原理: JITコンパイラ最適化の不正な前提（型推論・境界チェック削除）に起因するメモリ破壊は、ブラウザエンジンで繰り返し現れる典型パターン。最適化器が「安全」と判断した根拠を実行時に覆せること（型混乱・最適化の取りこぼし）が共通の核。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: クライアント側のクラッシュレポート（renderer プロセスの異常終了）、EDR による Chrome 子プロセスからの不審なプロセス生成、未更新バージョンの資産棚卸し（`chrome://version` のバージョン値）。
- **暫定緩和**: パッチ適用が即時にできない場合、サイト分離（Site Isolation）は既定で有効を維持。高リスク端末では JIT を制限する運用（エンタープライズポリシーでの JavaScript 最適化制限）も検討余地。ただし恒久策はあくまで更新。
- **恒久対策**: Chrome および Chromium 派生ブラウザを修正版へ更新し、自動更新を有効化。組織では更新の強制配布（管理ポリシー）でバージョン下限を強制。

## 📖 用語解説（このニュースとのつながり）

- **V8 / TurboFan**: V8 は Chrome の JavaScript/WebAssembly 実行エンジン、TurboFan はその最適化コンパイラ。今回はこの最適化処理の誤りが穴になった。
- **境界外読み書き（OOB read/write）**: 配列など割り当てられた範囲の外のメモリを読む・書く不具合。本件の心臓部で、メモリ破壊→コード実行の起点。
- **bounds-check elimination**: 「この配列アクセスは安全」と判断して境界チェックを省く高速化。前提が崩れると無防備なアクセスが残る。
- **ゼロデイ / KEV**: 修正前から悪用されている脆弱性がゼロデイ。CISA KEV は「実際に悪用中」と政府機関が認定した一覧で、本件は登録済み＝優先対応対象。
- **サンドボックス**: レンダラを隔離する仕組み。本件単体ではこの壁の内側までで、突破には別の脆弱性が必要。

## 影響範囲 / 推奨対応

影響は Chrome を使う全プラットフォーム（Windows / macOS / Linux）と、Chromium ベースのブラウザ（Edge, Brave, Vivaldi, Opera 等）に及ぶ。一般利用者・企業端末いずれも対象。推奨対応は、Chrome を 149.0.7827.102/.103 以降へ即時更新し、再起動で適用を確定すること。組織では管理ポリシーで最低バージョンを強制し、未更新端末を資産管理で洗い出す。Chromium 派生ブラウザのベンダー更新も追跡する。
