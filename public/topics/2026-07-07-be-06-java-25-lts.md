# Java 25 LTS — 仮想スレッドの pinning 解消・Scoped Values 正式化で「軽い並行」が完成へ

> トピック: Java 25 LTS リリース｜出典: Oracle 公式発表｜種別: LTS メジャーリリース

## 一行サマリー

2025年9月16日にリリースされた Java 25 は、Java 21 に続く LTS（長期サポート、Oracle は最低8年）で、計18個の JEP を含む。目玉は Java 21 で導入された仮想スレッドの弱点（synchronized ブロックでの pinning）の解消、Scoped Values の正式化、Structured Concurrency の継続改善で、「大量の軽量スレッドで書くサーバーサイド並行処理」が実用として一段成熟した。

## 🔰 初学者向け（何が・なぜ重要か）

サーバーは「同時にたくさんのリクエストをさばく」のが仕事です。従来の Java では 1 リクエスト＝ 1 スレッドで処理していましたが、OS のスレッドは重く、数千・数万を同時に立てるとメモリと切り替えコストで詰まりました。そこで Java 21 が導入したのが **仮想スレッド**——「OS スレッドの上で自由に載せ替えできる、非常に軽いスレッド」です。数百万立てても平気で、「1 リクエスト 1 スレッド」の素直な書き方のまま高い同時実行数を出せます。

ただ 21 の仮想スレッドには弱点がありました。`synchronized`（複数スレッドが同じ資源を奪い合わないよう順番待ちさせる仕組み）の中で待つと、下敷きの OS スレッドを離せず「詰まって」しまう（pinning）現象です。Java 25 ではこれが解消され、待っている間ちゃんと OS スレッドを他へ譲れるようになりました。さらに「スレッド間で安全にデータを共有する」新しい仕組み（Scoped Values）も正式機能になりました。軽くて速い並行処理が、落とし穴を埋めて「安心して使える段階」に来た、というのがこの LTS の意味です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **LTS リリース**（2025年9月16日）。18 JEP を収録、うち9つが性能・ランタイム関連、7つが preview/incubation を経てファイナライズ。
- **仮想スレッドの pinning 解消**: `synchronized` ブロック内でブロックしても下層のプラットフォームスレッドを解放できるように改善。従来は pin されスケーラビリティを損ねていた。
- **Scoped Values 正式化**: スレッド内・スレッド間で不変データを共有する API。ThreadLocal より軽量・見通しがよく、仮想スレッド＋構造化並行と好相性。
- **Structured Concurrency 継続改善**: タスクを子タスクに分割・並列実行し、ライフサイクルとエラー伝播をまとめて管理する枠組み。
- そのほか性能・起動・フットプリント系の改善（コンパクトなオブジェクトヘッダ等、ランタイム最適化）。正確な JEP 番号・細目は一次ソースで要確認。

### 仕組み / 背景（なぜこう設計されたか）

仮想スレッドの核心は「ブロッキング I/O を待つ間、キャリア（OS）スレッドを手放して他の仮想スレッドに貸す」こと。これにより「同期的で読みやすいコード」と「ノンブロッキング並の同時実行数」を両立する。ところが 21 では JVM 内部の実装制約で `synchronized` 中の待機時にキャリアを手放せず（pinning）、モニタを多用する既存ライブラリが仮想スレッドの利点を削いでいた。25 はモニタ実装を見直してこの制約を外し、「既存の synchronized コードでもスケールする」状態にした。これは仮想スレッドを"新規コード専用の実験"から"既存資産にそのまま効く標準機能"へ引き上げる転換点である。

Scoped Values は、ThreadLocal の問題（可変・後始末忘れ・継承コスト、仮想スレッド大量生成時のメモリ増）への回答。値を不変にし、スコープ（実行区間）に束ねて自動で片付けることで、リクエストスコープのコンテキスト（認証主体、トレース ID 等）を安全・低コストに配れる。Structured Concurrency と組み合わせると、「親タスクの寿命＝子タスク群の寿命」「1つ失敗したら兄弟をキャンセル」といったキャンセル・エラー伝播が構造的に保証され、goroutine 的な"投げっぱなし"に伴うリークやゾンビタスクを防げる。トレードオフは、これらを活かすにはコードを新パターンへ寄せる必要がある点と、CPU バウンド処理には仮想スレッドが効かない（そこはプラットフォームスレッド／並列ストリームの領域）点。

### 実務での使いどころ・移行の勘所

- I/O バウンドなサーバー（Web API、DB/外部 API 呼び出しの多いバックエンド）で、リアクティブの複雑さを避けつつ高い同時実行数を得たいケースに最適。
- 21 で仮想スレッドを試して synchronized の pinning に悩んだプロジェクトは、25 で素直に効くようになる。既存の同期ベースのライブラリもスケールしやすい。
- LTS のため本番採用のアンカーに好適。Spring Boot 4 / 3.x、Quarkus、Helidon 等のフレームワークが仮想スレッド・構造化並行の対応を進めており、フレームワーク側の対応状況とセットで移行を計画。
- Scoped Values は ThreadLocal ベースのコンテキスト伝播（MDC ログ、セキュリティコンテキスト）の置き換え候補。段階移行を推奨。

## 📖 用語解説（このトピックとのつながり）

- **仮想スレッド（Virtual Threads）**: OS スレッドから切り離せる超軽量スレッド。25 で pinning が解消され実用性が向上。
- **pinning**: 仮想スレッドが下層 OS スレッドを手放せず固定される現象。synchronized 内で起きていたが 25 で解消。
- **Scoped Values**: スコープに束ねた不変データをスレッド内外で共有する API。ThreadLocal の後継的存在として 25 で正式化。
- **Structured Concurrency**: タスクを親子構造で管理し、キャンセル・エラー伝播を構造的に扱う枠組み。仮想スレッドと組み合わせて真価を発揮。

## 影響範囲 / 推奨アクション

- サーバーサイド Java: 21→25（LTS→LTS）移行を計画。仮想スレッド活用箇所は pinning 改善の恩恵大。
- フレームワーク利用者: Spring Boot 4 等の仮想スレッド／構造化並行対応と合わせて採用判断。
- コンテキスト伝播: ThreadLocal から Scoped Values への段階移行を検討。
