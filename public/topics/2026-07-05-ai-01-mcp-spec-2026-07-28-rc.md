# MCP 仕様 2026-07-28 リリース候補 — プロトコルがステートレス化、Tasks と MCP Apps を追加

> トピック: Model Context Protocol 次期仕様 2026-07-28 のリリース候補｜出典: MCP 公式ブログ｜種別: プロトコル仕様（RC）

## 一行サマリー

AI エージェントと外部ツールをつなぐ標準規格 MCP の次期仕様（2026-07-28）のリリース候補が公開され、プロトコルコアを**ステートレス化**、機能を独立して拡張できる Extensions フレームワーク、非同期実行の Tasks、サンドボックス化 UI を配信する MCP Apps、認可の強化、正式な非推奨ポリシーを導入する。最終版は7月28日出荷予定。

## 🔰 初学者向け（何が・なぜ重要か）

AI エージェントが「天気を調べる」「社内データベースを引く」といった作業をこなすには、外部の道具（ツール）につながる必要があります。MCP は、その「AI と道具の接続方法」を各社共通の規格にしたもの。USB がどんな機器も同じ差し込み口でつなげるように、MCP はどんな AI ・どんなツールも共通の作法で会話できるようにします。

今回はその MCP の「次のバージョンの試作版（リリース候補）」が出ました。目玉は3つ。ひとつめは、接続を「毎回状態を覚えておく」方式から「その都度きれいに完結する（ステートレス）」方式へ変えたこと。これによりクラウドで大規模に動かしやすくなります。ふたつめは、時間のかかる作業を「後で結果を取りに行く」形で扱える Tasks。みっつめは、ツールが対話用の画面（UI）を安全な枠の中で表示できる MCP Apps です。

たとえるなら、共通プラグの規格が「電源だけ」から「電源＋画面＋長時間ジョブの受付票」まで扱える、より本格的な業務用コネクタへ進化した、というイメージです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

MCP 2026-07-28 リリース候補の主な要素:

- **ステートレスなプロトコルコア**: 基盤の JSON-RPC 通信をステートレス化し、コモディティな HTTP インフラ上で水平スケールしやすくする。
- **Extensions フレームワーク**: Tasks や MCP Apps などの能力を、コアとは独立したタイムラインで進化させられる拡張機構。
- **Tasks 拡張**: `tools/call` に対しサーバがタスクハンドルを返し、クライアントが `tasks/get` / `tasks/update` / `tasks/cancel` で駆動。タスク生成はサーバ主導。
- **MCP Apps**: サーバが対話的な HTML UI を配信し、ホストがサンドボックス化 iframe で描画。ツールが UI テンプレートを事前宣言することで、ホストが prefetch・キャッシュ・セキュリティ審査を先行できる。UI は同じ JSON-RPC 基盤でホストと通信。
- **認可の強化**と**正式な非推奨ポリシー**（ライフサイクル管理）。

10週間の RC 期間で SDK メンテナ・クライアント実装者が実ワークロード検証を行い、Tier 1 SDK は期間内の対応が期待される。最終版は7月28日出荷予定。

### 仕組み / 背景（なぜこう設計されたか）

初期 MCP は接続にステートを持ち、SSE ベースの長時間接続を前提としていた。これはローカルのデスクトップ用途では素直だが、クラウドで多数のクライアントを捌く際に「どのサーバインスタンスがどのセッション状態を持つか」を管理する必要が生じ、スケールとフェイルオーバーを難しくしていた。ステートレス化は、各リクエストが自己完結することで、任意の HTTP ロードバランサ配下に平置きできるようにする設計転換だ。代償として、長時間ジョブや進行状態のような「本質的にステートフルな作業」を別レイヤで扱う必要が生じ、それに応えるのが Tasks 拡張。ステートを「プロトコルコア」から「タスクという明示的な資源」へ移し、クライアントがポーリング的に駆動する形にした。

MCP Apps は「ツールは文字列だけ返す」制約を越え、リッチな対話 UI を安全に配信するための仕組み。UI テンプレートを事前宣言させ、ホストがサンドボックス iframe で審査・描画するのは、任意 HTML を実行させるセキュリティリスク（XSS、データ流出）を抑えるための設計。Extensions フレームワークと非推奨ポリシーは「コアは安定・小さく保ち、機能は独立進化させる」というモジュラリティ志向で、プロトコルの長期的な後方互換性を担保する狙いがある。

### 実務での使いどころ・移行の勘所

サーバ側 MCP をクラウドでスケールさせたいプロバイダには、ステートレスコアが直接効く。長時間の分析・生成ジョブを扱うツールは Tasks でノンブロッキング化でき、タイムアウトや接続切断の問題を回避しやすい。MCP Apps は、承認フローや対話的な設定 UI をツール側から提供したいケースで有用。移行の勘所は、これはまだ RC であり最終仕様（7/28）まで変更余地がある点。既存の SSE ベース実装は、Streamable HTTP／ステートレス前提への移行影響（プロキシ、認証、状態管理）を検証する必要がある。まずは非本番環境で RC SDK ベータを試し、Tasks/Apps の採用は最終版確定後に本格化するのが安全。

## 📖 用語解説（このトピックとのつながり）

- **MCP（Model Context Protocol）**: AI エージェントとツール/データ源を接続する標準規格。本トピックの主役。
- **ステートレス**: 各リクエストが自己完結し、サーバがセッション状態を保持しない方式。水平スケールに有利。
- **Tasks 拡張**: 長時間処理をハンドル経由でノンブロッキングに扱う仕組み。
- **MCP Apps**: サーバが配信する対話 UI をサンドボックス iframe で安全に描画する機能。
- **Extensions フレームワーク / 非推奨ポリシー**: コアと独立して機能を進化させ、後方互換を守る枠組み。

## 影響範囲 / 推奨アクション

MCP サーバをクラウド運用するプロバイダやエージェント基盤開発者に大きく影響。RC 段階のため最終版（7/28）まで仕様変更の余地あり。非本番で RC SDK ベータを検証し、Tasks/Apps の本採用は最終確定後を推奨。
