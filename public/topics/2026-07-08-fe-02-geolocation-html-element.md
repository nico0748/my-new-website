# 宣言的な位置情報取得：新しい HTML `<geolocation>` 要素（Chrome 144）

> トピック: HTML `<geolocation>` 要素｜出典: Chrome for Developers ブログ / MDN｜種別: 新 HTML 要素（Chrome 144 で搭載）

## 一行サマリー

Chrome 144 から、ユーザー操作を起点に位置情報の許可フローと取得をまとめて扱う宣言的な HTML 要素 `<geolocation>` が使えるようになった。従来 JavaScript の `navigator.geolocation` で書いていた許可ハンドリングのボイラープレートを大幅に減らし、ユーザーの明確な意思表示（ユーザー起動）を許可プロンプトに直結させる。

## 🔰 初学者向け（何が・なぜ重要か）

これまで Web サイトが「現在地を教えてほしい」ときは、JavaScript から `navigator.geolocation.getCurrentPosition()` を呼び、ブラウザが許可ダイアログを出す、という流れだった。問題は、ページを開いた瞬間に脈絡なく許可を求めるサイトが多く、ユーザーがうんざりして「拒否」を押しがちだったこと。ブラウザ側も、乱発される許可要求を「静かにブロック」する介入を入れるようになり、開発者にとっても扱いづらかった。

新しい `<geolocation>` 要素は、この流れをボタンのような「見える部品」に変える。ユーザーがその要素をクリックしたときにだけ許可フローが走り、位置情報が返ってくる。「ユーザーが自分で押した」という明確な意思があるので、ブラウザは安心して許可を出せるし、ユーザーも文脈を理解した上で押せる。

たとえるなら、これまでは「家に入った瞬間、見知らぬ人にいきなり住所を聞かれる」状態だったのが、「配達フォームの中に置かれた『現在地を使う』ボタンを自分で押す」状態になったイメージだ。誰が・なぜ・いつ位置情報を求めているかが明確になる。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- Chrome 144 で `<geolocation>` HTML 要素が利用可能に。
- 位置情報の**許可フロー**と**データ取得**を要素自身が引き受け、多くの場合 JavaScript API 呼び出しを不要にする。
- 許可状態やエラーのハンドリングに必要なコードを削減。
- 「ユーザー起動（user-activated）」を強制するため、ページ読み込み時の無差別な許可要求（Chrome のブラウザ介入・quiet block の対象）を回避できる。
- これは「Page-Embedded Permission Control」構想の最新形。当初は `type` 属性を持つ汎用 `<permission>` 要素として提案され、Chrome 126〜143 でオリジントライアルが行われた。

### 仕組み / 背景（なぜこう設計されたか）

最重要なのは「汎用 `<permission>` から、機能ごとの専用要素（`<geolocation>`）へ舵を切った」という設計判断だ。汎用要素は一見エレガントだが、Apple（Safari/WebKit）や Mozilla（Firefox）からのフィードバックで「機能ごとに許可の意味・挙動が違いすぎて、one-size-fits-all は複雑すぎる」という指摘があった。カメラ・マイク・位置情報では、必要な UI・プライバシー上の配慮・許可の粒度がまったく異なる。

そこで「汎用の許可コントロール」から「機能特化型の許可コントロール」へ転換し、まず位置情報だけを `<geolocation>` として切り出した。トレードオフは、機能ごとに要素が増えていく（将来 `<camera>` 等が別途必要になり得る）冗長さと引き換えに、各要素が自分の機能に最適な UX とプライバシー設計を持てる点だ。

宣言的アプローチの本質的な利点は「意思のシグナルが強くなる」ことにある。スクリプトが勝手に呼ぶ許可要求は、ユーザーの意図が読めないためブラウザが疑う。要素をクリックするという明示的操作は、ユーザーの同意を強く示すため、ブラウザ側の介入（静かなブロック）を避けられる。

### 実務での使いどころ・移行の勘所

- 「近くの店舗を探す」「配送先に現在地を使う」など、ユーザーが能動的に位置を使いたい場面に最適。
- ページ読み込み直後の位置取得（Lighthouse の geolocation-on-start 警告に該当する使い方）を置き換える良い機会。
- 現状は Chrome 系先行。Safari / Firefox 未対応のため、`navigator.geolocation` へのフォールバックを併用し、プログレッシブエンハンスメントで導入する（クロスブラウザ状況は要確認）。
- スタイルの自由度・要素のカスタマイズ範囲には制約があり得るため、デザイン要件と照らして検証する。

## 📖 用語解説（このトピックとのつながり）

- **`navigator.geolocation`**: 従来の位置情報取得 JavaScript API。`<geolocation>` 要素はこれの宣言的代替であり、フォールバック先でもある。
- **オリジントライアル**: 実験的機能を本番サイトで期間限定試用できる Chrome の仕組み。汎用 `<permission>` 要素が Chrome 126〜143 で試された。
- **ユーザー起動（user activation）**: クリック等の明示操作があった状態。多くの権限要求はこれを前提に許可されやすくなる。
- **ブラウザ介入（quiet block）**: 乱発される許可要求をブラウザが静かに抑制する挙動。宣言的要素で回避できる。
- **Page-Embedded Permission Control**: 許可コントロールをページ内の要素として埋め込む構想。`<geolocation>` はその一実装。

## 影響範囲 / 推奨アクション

- 位置情報を使う導線を「ユーザーがクリックする要素」に設計し直す。
- 読み込み時の無差別な許可要求をやめ、`<geolocation>` へ移行する。
- Chrome 系のみ対応のため、`navigator.geolocation` フォールバックを必ず用意する。
- Safari / Firefox の対応状況を継続ウォッチ（機能特化型許可要素の標準化動向を追う）。
