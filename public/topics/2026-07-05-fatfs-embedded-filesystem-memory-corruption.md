# FatFs 組み込みファイルシステムに複数脆弱性（CVE-2026-6682 ほか）— USB/SDの細工で数百万の組み込み機器がメモリ破壊、上流修正なし

> トピック: 脆弱性ニュース ｜ 出典: The Hacker News / runZero / JPCERT/CC（調整） ｜ 種別: 整数オーバーフロー／バッファオーバーフロー（メモリ破壊・上流未修正）

## 一行サマリー

多数の組み込み機器のファームウェアに同梱される小型ファイルシステム「FatFs」に7件の脆弱性が公開され、代表例のFAT32マウント処理の整数オーバーフロー（CVE-2026-6682, CVSS 7.6）は細工したUSB/SD/更新ファイルでメモリ破壊・コード実行に至り得る。監視カメラ・ドローン・産業機器・ハードウェアウォレット等が対象で、上流（開発者1名）に修正がなく、runZeroはJPCERT/CCを通じた調整も試みたが応答が得られていない。

## 🔰 初学者向け（何が・なぜ問題か）

「**FatFs**」は、USBメモリやSDカードでおなじみの保存形式（FAT／exFAT）を、小さな機器でも読み書きできるようにする“部品ソフト”です。監視カメラ、ドローン、工場の制御装置、暗号資産のハードウェアウォレットなど、パソコン以外の**組み込み機器のファームウェア**に広く組み込まれています。

今回の問題は、この部品ソフトが**USBやSDカードの中身を読み込むときの計算を誤り**、想定より大きな値やラベルを渡されると、用意した箱（メモリ）からデータがあふれてしまうことです。あふれたデータが機器の“頭脳”の別の場所を壊し、攻撃者の仕込んだ命令を実行させられます。

たとえるなら、受付が来客カードの記入欄の文字数を数え間違え、長すぎる名前を書かれると隣の書類まで上書きしてしまう——そんな状態です。攻撃者は**細工したUSBメモリやSDカード、更新ファイルを機器に読み込ませるだけ**で、機器を乗っ取れる可能性があります。多くの組み込み機器はパソコンのような防御機能が乏しく、「物理的に触れれば乗っ取り（jailbreak）につながる」と警告されています。

さらに深刻なのは、この部品ソフトが**たった一人の開発者**によって細々と保守されており、連絡も取りづらく、**根本を直す公式修正が出ていない**点です。日本のJPCERT/CCを通じた調整も試みられましたが応答が得られず、これを組み込んだ多数の製品が“自分が影響を受けていることすら気づけない”状況にあります。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **CVE**: CVE-2026-6682（代表）ほか計7件。CVE-2026-6687、CVE-2026-6688 など。
- **CVSS**: いずれも 7.6（High）近辺。
  - **CVE-2026-6682**: FAT32ボリュームのマウント処理における**整数オーバーフロー**。実機ではメモリ破壊・コード実行に発展し得る。
  - **CVE-2026-6687**: exFATのボリュームラベル項目が小さなバッファを溢れさせる（メモリ破壊の足場）。
  - **CVE-2026-6688**: 長いファイル名が、多くのプロジェクトがFatFs周りに置くラッパーコード（例: `fno.fname` を固定長バッファへ `strcpy`）を溢れさせる。
- **影響対象**: FatFsを同梱するRTOSベースの組み込み機器全般（監視カメラ、ドローン、産業用コントローラ、ハードウェアウォレット等）。「数百万台」規模。
- **前提条件**: 攻撃者が**細工したFAT/exFATメディア（USB/SDカード）や更新ファイル**を機器に読み込ませられること。多くは物理アクセスまたは悪性メディアの供給が起点。
- **悪用状況**: runZeroの2026年7月1日開示時点で実攻撃報告なし。ただし**PoCディスクイメージ・テストハーネス・QEMUベースの動作例が公開済み**で、悪用材料は既に入手可能。
- **修正状況**: **上流に恒久修正なし**。セキュリティ連絡先やメーリングリストも整備されておらず、影響製品の把握が困難。国内関連: runZeroがJPCERT/CCへ調整を依頼（応答は得られず）。

### メカニズム（攻撃が成立する仕組み）

本件は「信頼できない外部メディアのメタデータを、境界検証なしにサイズ計算・コピーへ渡す」古典的メモリ破壊である。概念レベルで整理する（武器化手順・PoCは記載しない）。

1. **信頼境界の所在**: ファイルシステムのマウント／解析処理は、**外部から差し込まれたメディアの構造（ボリューム情報・ラベル・ファイル名）を解釈する**入口であり、そこは本質的に“信頼できない入力”の境界にあたる。
2. **計算の誤り（整数オーバーフロー）**: FAT32マウント時、メディアのパラメータからバッファ長やオフセットを算出する計算が**桁あふれ**を起こす。結果として、本来より小さい／不正なサイズが確保・使用され、後続のコピーで境界を越える。
3. **境界外書き込みプリミティブの獲得**: exFATラベルや長いファイル名（ラッパーの `strcpy`）でも、**入力長を検証しないコピー**により固定長バッファを溢れさせる。攻撃者は隣接メモリを上書きする**境界外書き込み（メモリ破壊）**プリミティブを得る。
4. **コード実行への橋渡し**: 組み込み機器は**メモリ保護（DEP/ASLR等）が乏しい**ことが多く、破壊したメモリ上の制御データ（戻りアドレス・関数ポインタ・構造体）を書き換えることで、比較的容易にコード実行へ昇格する。RTOS上では権限分離も薄く、機器全体の掌握に至りやすい。

一般原理として、これは**入力サイズの検証欠如＋安全でないコピー（unbounded copy）＋整数演算の桁あふれ**が重なった典型的な組み込みメモリ破壊で、パーサが信頼できないメディアを扱う以上、境界チェックの徹底が唯一の防波堤になる。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: 機器がメディア挿入・更新読込直後にクラッシュ／リブート／異常挙動を示す、想定外のメディア構造（極端なボリューム情報・長大ラベル／ファイル名）、供給元不明のUSB/SD・更新ファイルの使用。多くの組み込み機器はログが乏しく、物理・運用管理での予防が中心。
- **暫定緩和**: 信頼できないUSB/SDメディアの挿入を運用・物理的に禁止。ファーム更新は正規署名済みファイルのみ許可し供給経路を管理。可能なら外部メディア読取機能を無効化・制限。機器を物理的に露出させない。
- **恒久対策**: 上流修正がないため、**各製品ベンダーが自製ファームでFatFs該当箇所を修正・再ビルド**する必要がある。SBOM（部品表）でFatFs同梱有無を棚卸しし、ベンダーへ対応を要請。境界チェック・安全なコピー関数への置換を検証。

## 📖 用語解説（このニュースとのつながり）

- **FatFs**: FAT/exFATを扱う軽量な組み込み向けファイルシステムライブラリ。多数の機器に“部品”として組み込まれ、影響範囲を広げている。
- **整数オーバーフロー**: 数値計算が扱える上限を超えて破綻し、誤ったサイズ・オフセットを生む欠陥。今回のFAT32マウント処理が該当。
- **境界外書き込み（バッファオーバーフロー）**: 確保領域を超えてメモリを書き換える欠陥。exFATラベルや長いファイル名のコピーで発生し、コード実行の足場になる。
- **上流未修正（no upstream fix）**: 部品ソフト本体に公式修正が存在しない状態。各製品ベンダーが個別に直すしかなく、対応の抜けが生じやすい。
- **SBOM**: ソフトウェア部品表。どの製品がFatFsを含むか把握するために有効。

## 影響範囲 / 推奨対応

FatFsを同梱する組み込み機器（監視カメラ・ドローン・産業機器・ハードウェアウォレット等、数百万規模）が対象。上流恒久修正がなく、PoCも公開済み。信頼できないメディア／更新ファイルの排除、正規署名済み更新の徹底、SBOMによる同梱棚卸しとベンダーへの修正要請が現実的な対策。物理アクセスやサプライチェーン経由の悪性メディアに特に注意。
