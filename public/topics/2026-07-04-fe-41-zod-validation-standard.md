# Zod がバリデーションの標準に — 型安全なスキーマ検証

> トピック: 開発トピック｜出典: thetshaped.dev｜種別: frontend

## 一行サマリー
Zod は TypeScript の型とランタイムのバリデーションを1つのスキーマ定義から両立させるライブラリで、2026年のフロントエンド標準スタックの定番として定着している。

## 🔰 初学者向け（何が・なぜ重要か）
TypeScript の型は「開発中の約束事」に過ぎず、ビルドされた後（実行時）には消えてしまいます。つまりフォーム入力や API のレスポンスのように、外部からやってくる「実際のデータ」が本当に想定した形なのかは、TypeScript だけでは保証できません。ここに大きな穴があります。

Zod は、この穴をふさぐ「入り口のチェック係」です。たとえば「名前は文字列」「年齢は0以上の数値」といったルールを1回書けば、そのルールで実行時にデータを検査でき、しかも同じ定義から TypeScript の型も自動生成してくれます。1つの設計図で「実行時の門番」と「開発時の型」の両方をまかなえるのが強みです。

これは「二重管理をなくす」という発想です。従来は型定義とバリデーション処理を別々に書き、片方だけ更新して食い違うバグが頻発していました。Zod なら「真実の源」が1つになり、ズレが起きません。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Zod はスキーマから `z.infer<typeof schema>` で静的型を導出できるのが核心。`parse()` は失敗時に例外を投げ、`safeParse()` は `{ success, data | error }` を返す。React Hook Form の `@hookform/resolvers/zod` や TanStack Router のバリデーションと組み合わせる構成が定番。バージョンや細かな API 差分は要確認だが、tree-shaking や起動速度を意識した軽量版（Zod Mini 系）の動向も出ている点は要確認。

### 仕組み / 背景（なぜこう設計されたか）
TypeScript の型は消去（type erasure）されるため、境界（フォーム・API・環境変数）でのランタイム検証が別途必要になる。Zod は「スキーマ＝型の源」という設計で、この境界問題を単一定義に集約した。従来の Yup 等と比べ TypeScript 推論との親和性が高いのが採用理由。

### 実務での使いどころ・移行の勘所
API レスポンス、フォーム入力、環境変数（`process.env`）の検証に最適。既存プロジェクトへは境界部分から段階導入できる。トレードオフはバンドルサイズと、複雑なスキーマでの推論コスト。代替として Valibot（軽量）や ArkType も検討対象。

## 📖 用語解説（このトピックとのつながり）
- **スキーマ**: データの形のルール定義。Zod ではこれが型とバリデーションの共通の源になる。
- **ランタイムバリデーション**: 実行時のデータ検査。TypeScript の型が届かない領域を補う。
- **型推論（infer）**: スキーマから静的型を自動生成する仕組み。二重管理を防ぐ核心。
- **safeParse**: 例外を投げず結果オブジェクトを返す検証 API。UI でのエラー表示に向く。

## 影響範囲 / 推奨アクション
フォームや API 連携を扱う全フロントエンド開発者に影響。新規プロジェクトでは境界バリデーションの標準として Zod を採用し、既存では外部入力の受け口から段階導入するとよい。
