# pgvector 0.8 — 反復スキャンと並列 HNSW 構築で Postgres のベクトル検索が実用強化

> トピック: pgvector 0.8 系（iterative scan / 並列 HNSW build）｜出典: pgvector 公式リポジトリ / PostgreSQL News｜種別: DB 拡張（実用強化）

## 一行サマリー

Postgres にベクトル類似検索を足す拡張 pgvector の 0.8 系は、絞り込み条件付きクエリで結果が不足する問題を解く「反復スキャン（iterative scan）」と、HNSW インデックスの並列構築・ストリーミング挿入の高速化を導入した。専用ベクトル DB を増やさずに、既存の Postgres 上で RAG・推薦・意味検索を実用運用しやすくなった。

## 🔰 初学者向け（何が・なぜ重要か）

AI アプリで「意味が近い文書を探す」には、文章を数値の並び（ベクトル）に変換して「近さ」で検索します。この検索を、専用のベクトルデータベースを新たに立てずに、普段使う Postgres でやれるようにするのが pgvector という拡張です。

これまでの pgvector には弱点がありました。「近いものを速く探す索引（HNSW）」を使いつつ「カテゴリ＝Aだけ」のような絞り込みを重ねると、索引が返した候補が絞り込みで消えてしまい、必要な件数に足りない、という取りこぼしが起きたのです。0.8 の「反復スキャン」は、たとえるなら**足りなければ棚を追加で漁って必要数が揃うまで探し続ける**動きで、この取りこぼしを防ぎます。加えて索引を作る作業も並列化されて速くなりました。

要するに、「Postgres だけで、絞り込み付きのベクトル検索がちゃんと使える」段階に近づいた、というのが今回の意味です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **iterative scan（反復スキャン）**: フィルタ付きクエリで、索引が返した近傍候補が WHERE 条件で除去され結果件数が不足する古典的問題に対処。十分なフィルタ通過結果が集まるまで索引探索を継続する。HNSW / IVFFlat 双方で利用可能。
- **並列 HNSW 構築**: HNSW インデックスのビルドを並列化し、大規模データでの索引作成時間を短縮。ストリーミング挿入も改善。
- 0.8 系は 0.7 の量子化（halfvec 等）路線に続く「品質・運用性」寄りの改善で、iterative scan は「pgvector 史上最大級の使い勝手改善」と評される。
- 本稿執筆時点の最新安定は 0.8 系（0.9 は未確認）。正確な最新版・細目は公式リポジトリで要確認。

### 仕組み / 背景（なぜこう設計されたか）

近似最近傍（ANN）索引である HNSW / IVFFlat は「全走査せずに近いものを速く返す」ために、探索の打ち切り（HNSW の `ef_search`、IVF の `probes`）で候補数を制限する。この打ち切りは「フィルタなし」を前提に効率化されているため、`WHERE category = 'A'` のような後段フィルタと組み合わせると、返した候補の多くが捨てられ、要求 `LIMIT` に届かない——いわゆる「post-filtering の欠落」問題が生じる。

iterative scan は、この状況で索引探索を一度きりで終えず、フィルタ後の有効件数が満たされるまで反復的に候補を追加取得することで解決する。設計上のトレードオフは、フィルタ選択率が極端に低い（該当が非常に少ない）場合に探索が伸び、レイテンシが増える点。並列 HNSW 構築は、索引作成という CPU バウンドな処理をコア並列で捌くことで、数百万〜のベクトルでも現実的な時間で索引を張れるようにする一方、ビルド時のメモリ/CPU 消費は増える。全体として、pgvector は「専用ベクトル DB でなくても RAG 実運用に耐える」ための実用面を詰めている。

### 実務での使いどころ・移行の勘所

RAG の文書検索、推薦、意味検索を Postgres に統合したいケースで有効（データとベクトルを1つの DB で管理でき、トランザクションや JOIN と併用できる）。フィルタ付き検索を多用するなら iterative scan を有効化し、選択率が低いクエリはレイテンシを計測してパラメータ（`ef_search` 等）を調整する。大規模索引は並列ビルドでダウンタイムを圧縮できるが、ビルド時のリソース確保を計画する。専用ベクトル DB との比較では、規模・QPS 要件が非常に高い場合は専用製品が有利な領域が残る点を踏まえて選定する。

## 📖 用語解説（このトピックとのつながり）

- **pgvector**: Postgres にベクトル型と類似検索を足す拡張。本トピックの主役。
- **HNSW / IVFFlat**: 近似最近傍（ANN）検索の索引方式。iterative scan と並列構築の対象。
- **iterative scan（反復スキャン）**: フィルタで結果が減る問題を、必要数まで探索継続で補う機能。
- **post-filtering 問題**: 索引が返した候補が後段フィルタで欠落し件数不足になる現象。
- **RAG**: 検索で集めた文脈を LLM に与える手法。pgvector が検索基盤になる。

## 影響範囲 / 推奨アクション

Postgres を中核に RAG/意味検索を運用するチームは、専用ベクトル DB を増やさずに実用強化を受けられる。フィルタ併用が多い場合は iterative scan を有効化し、低選択率クエリのレイテンシを計測してチューニング。超大規模・高 QPS 要件は専用製品との比較を継続する。
