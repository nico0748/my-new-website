# SvelteKit 大型更新 — 設定を vite.config へ統合、明示的 env 変数がプレビュー（破壊的変更あり）

> トピック: SvelteKit 2026年7月の破壊的リリース｜出典: Svelte 公式ブログ｜種別: フレームワークのメジャー更新

## 一行サマリー

SvelteKit が破壊的変更を含む大型リリースを公開。`svelte.config.js` を廃して設定を `vite.config.js` に統合できるようになり、`$env/*` を置き換える「明示的な環境変数」がプレビュー導入された。あわせてフォームエラーの型付け、`$app/stores`、リモート関数のファイル規則、トレーシングなど広範な仕様が更新され、Vite 8.0.12+（安定版 Rolldown 同梱）が必須になった。

## 🔰 初学者向け（何が・なぜ重要か）

SvelteKit は、Svelte で本格的な Web アプリ（ページ遷移・サーバー処理・フォームなど）を作るための土台フレームワークです。今回の更新は「使い勝手を整理する代わりに、いくつかの書き方を変える必要がある」大きめのアップデートです。

象徴的なのが**設定ファイルの一本化**。これまで SvelteKit の設定は専用ファイル（`svelte.config.js`）に書き、ビルドツール Vite の設定は別ファイルに書いていました。2つに分かれていて分かりにくかったのを、Vite の設定ファイル1つにまとめられるようになりました。また、アプリが使う「環境変数（APIキーなどの外部設定値）」の扱い方も、より安全で分かりやすい新方式が試験導入されました。破壊的変更を含むので、既存プロジェクトはアップデート時に少し書き換えが要りますが、全体としては「散らかっていた部分を整頓する」方向の進化です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **設定の統合**: SvelteKit の設定を `vite.config.js` 内に直接定義でき、`svelte.config.js` を省略可能に。ツールチェーンの設定を一元化。
- **明示的 env 変数（プレビュー）**: 将来 SvelteKit 3 で `$env/*` モジュールを置き換える新しい環境変数の扱いがプレビュー導入。
- **破壊的変更（主なもの）**: フォームエラーの型付け、トレーシング、`$app/stores`、リモート関数のファイル配置規則、param ファイルの扱いが変更。レスポンス処理、server-only ディレクトリ、origin 設定、ナビゲーション、エラーステータスコード、リダイレクト周りにも破壊的更新。
- **追加機能**: アダプタの Vite プラグイン対応、モジュールベースの Service Worker 登録、フォームフィールドのヘルパー、`cookies.parse`、`preloadData`/`goto` 挙動の調整、リモートフォーム対応。
- **前提**: Vite 8.0.12+（安定版 Rolldown 1.0.0 を同梱する最初の Vite 8 系）が必須。ツールチェーン（language tools・`sv` CLI）は Svelte の新しい `{const ...}` 宣言タグに追随済み。

### 仕組み / 背景（なぜこう設計されたか）

SvelteKit は「Svelte 用の Vite プラグイン＋規約」という構造で、設定が SvelteKit 側（`svelte.config.js`）と Vite 側（`vite.config.js`）に二重化していた。これは初学者の混乱源であり、プラグイン起動順やエイリアス解決の重複も招いていた。設定を Vite 側に寄せることで「ビルドツールが単一の真実（single source of truth）を持つ」という一般原則に近づけ、設定の重複と読み込み順の曖昧さを減らす狙いがある。

環境変数の刷新も同じ思想だ。`$env/static` と `$env/dynamic`、public/private の区別は強力だが暗黙のルールが多く、どの値がクライアントに漏れるかが分かりにくかった。「明示的な env 変数」は、どの変数を公開し、いつ静的注入・動的読み込みするかを明示させることで、機密漏洩リスクと環境差バグを設計段階で潰す。トレードオフは移行コスト——破壊的変更を伴うため、既存の型付けやファイル配置に手を入れる必要がある。だからこそ新方式は当面プレビューに留め、`$env/*` と併存させながら SvelteKit 3 での正式置換に向けて慣らす段階的移行が採られている。

### 実務での使いどころ・移行の勘所

新規 SvelteKit プロジェクトは設定統合の恩恵をすぐ受けられる。既存プロジェクトのアップグレードでは、(1) Vite を 8.0.12+ に上げる（Rolldown への切替差分も同時に検証）、(2) リモート関数のファイル配置規則とフォームエラーの型付け変更を修正、(3) `$app/stores` 利用箇所を新 API へ、(4) env は当面 `$env/*` を維持しつつプレビュー機能を検証、という順が安全。破壊的変更が広範なため、CI とテスト（Vitest 設定も更新対象）を通して段階的に上げること。Vite 8 必須ゆえ、エコシステム（アダプタ・コミュニティプラグイン）の対応状況も確認したい。

## 📖 用語解説（このトピックとのつながり）

- **SvelteKit**: Svelte 用のアプリフレームワーク（ルーティング・SSR・フォーム）。本トピックの主役。
- **svelte.config.js / vite.config.js**: 従来分かれていた設定ファイル。今回 Vite 側へ統合可能に。
- **$env/*（環境変数モジュール）**: SvelteKit の env アクセス手段。明示的 env 変数が将来これを置換。
- **リモート関数（remote functions）**: サーバー側処理をクライアントから型安全に呼ぶ仕組み。ファイル規則が変更。
- **Rolldown / Vite 8**: Rust 製バンドラと必須のビルド基盤。8.0.12+ が前提。

## 影響範囲 / 推奨アクション

- 影響: SvelteKit を使う全プロジェクト。アップグレードに破壊的変更対応が必要。
- 推奨: Vite 8.0.12+ へ更新 → リモート関数配置・フォーム型・`$app/stores` を修正 → env はプレビューを検証しつつ既存維持。CI/テストで段階的に。
