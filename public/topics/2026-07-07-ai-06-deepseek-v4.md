# DeepSeek V4 — 1.6兆パラメータ MoE・1M コンテキストの MIT 公開重みが最強クラスへ

> トピック: DeepSeek V4 リリース｜出典: DeepSeek（Hugging Face／公式）｜種別: 新モデルリリース（オープンウェイト）

## 一行サマリー

DeepSeek が2026年4月24日に V4 を公開。Pro と Flash の2構成で、いずれも MoE（混合エキスパート）と既定1Mトークンのコンテキストを備える。V4-Pro は総1.6兆・活性49億パラメータの最大級オープンウェイトで、SWE-bench Verified 80.6% とオープン重み最上位（Gemini 3.1 Pro と同水準）。MIT ライセンスで重みを配布し、クローズド最前線との差を「3〜6か月」に縮めたと主張する。

## 🔰 初学者向け（何が・なぜ重要か）

大規模言語モデル（LLM）には、OpenAI や Anthropic のように「中身（重み）は非公開でAPI越しに使う」クローズド型と、重みを配って誰でもダウンロード・自前運用できる「オープンウェイト型」があります。DeepSeek V4 は後者の代表格で、しかも**性能がクローズド最前線に肉薄**しているのが衝撃です。

キーワードの一つが **MoE（混合エキスパート）**。巨大な脳を丸ごと毎回使うのではなく、「質問に応じて必要な専門家（エキスパート）だけを起こす」仕組みです。V4-Pro は総パラメータ1.6兆と巨大ですが、1トークンあたり実際に働くのは49億だけ——だから「賢いのに、動かすコストは見た目より軽い」わけです。もう一つが **1Mトークンのコンテキスト**。一度に読める文章量が桁違いで、長大なコード全体や書籍レベルの資料をまるごと文脈に入れて考えられます。加えて MIT ライセンスなので、商用でも自由に使える。「高性能・自前運用可・自由なライセンス」が揃った、開発者にとって非常に実用的な選択肢です。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **リリース**: 2026年4月24日。**V4-Pro** と **V4-Flash** の2構成。両者 MoE・既定1Mトークンコンテキスト。
- **規模**: V4-Pro は総1.6兆・活性49億パラメータ。現行オープンウェイト最大級。
- **ベンチ**: V4-Pro は SWE-bench Verified 80.6%（オープン重み最上位、Gemini 3.1 Pro と同水準の主張）。
- **ライセンス**: MIT。Hugging Face / ModelScope から重みを直接ダウンロード可能。
- **価格（API）**: V4-Pro で入力 約$0.435/M・出力 約$0.87/M トークン（コスト競争力を強調）。
- DeepSeek は「クローズド最前線との差は3〜6か月、コストは競合の数分の一」と主張（ベンチ・主張は一次ソースで要確認）。

### 仕組み / 背景（なぜこう設計されたか）

DeepSeek 系が一貫して追うのは「**スパースな計算で密なモデル並みの知能を、桁違いに安く出す**」路線。MoE は総パラメータ（知識容量）と活性パラメータ（1トークンあたりの計算コスト）を分離できる。1.6兆の容量を持ちつつ49億しか毎回動かさないことで、密（dense）な巨大モデルに対し推論コスト・メモリ帯域を大幅に抑える。ルータ（どのエキスパートを起こすか決める部品）の学習が肝で、専門家の負荷が偏らないよう分散させる工夫が性能とコストを左右する。トレードオフは、MoE 特有のサービング難度——多数のエキスパートを配置し、トークンごとに動的ルーティングしながら高スループットを保つには、専用の推論スタック（SGLang/vLLM 等の MoE 最適化）とそれなりの GPU メモリが要る点。

1Mコンテキストは、長文脈での注意機構コスト（素朴には長さの二乗）を抑える工夫（スパース／効率化された注意や位置表現の拡張）で成立する。長文脈は「巨大コードベース全体を文脈に入れるエージェント」「大量資料を一括参照する RAG 縮約」に効く一方、実効的に長距離の情報を使えるか（"読めている"か）はベンチと別で、実タスクでの検証が要る。MIT 公開の狙いは、エコシステム（微調整・蒸留・派生モデル・ツール統合）を最大化し、クローズド勢に対する"オープンの土俵"を広げること。オープン最上位モデルが SWE-bench でクローズド最前線に並ぶ意味は、「自前運用でフロンティア級のコーディング支援が組める」現実性が出たことにある。

### 実務での使いどころ・移行の勘所

- データを外に出せない／コストを厳密に管理したい組織で、自前運用のフロンティア級モデルとして有力。エージェント・コード生成用途に特に向く（SWE-bench 上位）。
- 自前ホストは MoE サービングの難度と GPU メモリ要件が壁。SGLang/vLLM 等の MoE 対応推論スタックが前提。手軽さ重視なら API か推論プロバイダ経由。
- 1Mコンテキストは長文脈エージェント／大規模コード理解に効くが、コスト・レイテンシと"実効的に読めているか"をタスクで検証。全部を毎回詰め込むより RAG との併用が実務的な場合も多い。
- クローズド（Claude/GPT/Gemini）との比較軸は「最高精度・運用の楽さ vs 自前運用の自由度・コスト・データ主権」。V4 はオープン側の天秤を大きく傾けた。

## 📖 用語解説（このトピックとのつながり）

- **オープンウェイト**: モデルの重みを配布し自前運用できる形態。V4 は MIT で配布され、商用含め自由度が高い。
- **MoE（混合エキスパート）**: 入力ごとに一部の専門家サブネットだけを起こす構造。総容量と計算コストを分離し、V4-Pro の「1.6兆容量・49億活性」を成立させる。
- **活性パラメータ**: 1トークン推論で実際に使うパラメータ数。推論コストの実効指標で、MoE では総数より小さい。
- **SWE-bench Verified**: 実 GitHub の課題を解けるかを測るコーディングベンチ。V4-Pro のスコアがクローズド最前線級であることの根拠。

## 影響範囲 / 推奨アクション

- 自前運用志向の組織: V4-Pro をコード／エージェント用途で評価。MoE サービング基盤（SGLang/vLLM）とセットで検証。
- コスト最適化: API 価格の競争力を踏まえ、クローズド最前線とコスト対効果を比較。
- 長文脈用途: 1M コンテキストの実効性をタスクで検証、RAG 併用も検討。
