# OpenTelemetry eBPF Instrumentation（OBI）— コード改変ゼロでカーネルから分散トレースを取得

> トピック: OpenTelemetry eBPF Instrumentation（OBI）｜出典: Cloud Native Now（KubeCon EU 2026）｜種別: 可観測性ツールの発表

## 一行サマリー

KubeCon + CloudNativeCon Europe 2026 で発表された OpenTelemetry eBPF Instrumentation（OBI）は、Linux カーネルの eBPF を使い、アプリのコードを一切変更せずに分散トレースと RED メトリクスを取得するゼロコード可観測性の仕組み。SDK 埋め込みやサービス再起動なしに、既存サービスの telemetry を横から取得できる。

## 🔰 初学者向け（何が・なぜ重要か）

システムが遅い・エラーが出るとき、原因を突き止めるには「どのリクエストがどこで何ミリ秒かかったか」を記録する必要があります（これを可観測性・オブザーバビリティと呼びます）。従来はこのために、アプリのコードに計測用の部品（SDK）を埋め込み、ビルドし直して再起動する必要がありました。手間がかかり、古いアプリには入れにくいのが悩みでした。

OBI は、この計測を**アプリの外側（Linux の心臓部＝カーネル）**から行います。eBPF という技術で、OS がやり取りする通信を安全に覗き見し、「どのサービスがどのサービスを呼び、どれだけかかったか」を自動で記録します。アプリのコードを1行も変えず、再起動もいりません。しかも、記録の形式は OpenTelemetry という業界標準なので、既存の監視ツールにそのまま繋げられます。「触りたくない・触れない既存システムでも、外から計測できる」——これが嬉しさです。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- Splunk が KubeCon EU 2026 で **OpenTelemetry eBPF Instrumentation（OBI）**と Kubernetes Operator を発表。
- eBPF でカーネルから直接 telemetry を取得し、**分散トレース**と **RED メトリクス**（Rate/Errors/Duration）を生成。
- **ゼロコード**: コード改変・SDK 埋め込み・サービス再起動なしで計測可能。
- 出力は OpenTelemetry 準拠のため、既存の OTel コレクタ／バックエンドへそのまま流せる。
- 背景として eBPF はすでにメインストリーム化——AWS EKS が Cilium（eBPF ベース CNI）を既定採用、Cilium は sidecar 不要のサービスメッシュ機能を提供。

### 仕組み / 背景（なぜこう設計されたか）

従来の可観測性は「アプリ内計測（in-process instrumentation）」が主流だった。SDK を各サービスに埋め、リクエスト境界でスパンを開始・終了する。精度は高いが、(1) 全サービスへの SDK 導入・保守コスト、(2) 言語ごとに実装が分裂、(3) 既存/レガシー/サードパーティ製バイナリには入れられない、という限界があった。eBPF はこの前提を覆す。カーネル内に安全な小プログラムを差し込み（検証器が無限ループやクラッシュを防ぐ）、システムコールやソケット I/O のイベントをフックする。アプリから見れば完全に透過的で、どの言語で書かれていようと、ネットワーク境界での「呼び出しと応答時間」を横から観測できる。

トレードオフは可視性の粒度だ。eBPF はネットワーク/カーネル境界を見るため、「サービス間の関係と所要時間」は取れても、アプリ内部の関数レベルの詳細やビジネス文脈（どのユーザーの何の操作か）は、アプリ内計測ほど細かくは分からない。また暗号化トラフィックの中身やアプリ固有の属性付与には限界がある。ゆえに実務では「eBPF で全サービスの土台マップと RED を自動取得し、重要経路だけ従来の SDK 計測で深掘りする」ハイブリッドが定石になりつつある。OBI が OTel 標準で出力することは、この2層を同じパイプラインへ統合できる点で意義が大きい。

### 実務での使いどころ・移行の勘所

多言語混在・レガシー混在のマイクロサービス群、SDK を全面導入するコストが見合わない環境、まず全体像（サービスマップと遅延）を素早く把握したい場面で有効。導入は Kubernetes Operator でクラスタに展開し、既存 OTel コレクタへ接続する形が素直。注意点は、カーネルバージョン・eBPF 権限（特権や CAP_BPF）の要件、暗号化トラフィックの扱い、そして粒度の限界を理解して「深掘りは従来計測で補う」設計にすること。オーバーヘッドは低いとされるが、対象ノードで実測して確認するのが安全。

## 📖 用語解説（このトピックとのつながり）

- **eBPF**: Linux カーネルに安全なプログラムを差し込む技術。アプリ透過の計測を可能に。本トピックの中核。
- **OpenTelemetry（OTel）**: telemetry の業界標準。OBI の出力形式で、既存ツールと接続できる鍵。
- **RED メトリクス**: Rate/Errors/Duration。サービス健全性の基本指標。
- **分散トレース**: 1リクエストが複数サービスを跨ぐ経路と所要時間の記録。
- **Cilium / CNI**: eBPF ベースのネットワーク基盤。eBPF メインストリーム化の象徴。

## 影響範囲 / 推奨アクション

- 影響: マイクロサービス運用・SRE 全般。計測導入コストの削減とレガシー可視化に直結。
- 推奨: Operator でクラスタ展開し既存 OTel パイプラインへ接続。粒度の限界は従来計測で補完。カーネル要件・オーバーヘッドを実測。
