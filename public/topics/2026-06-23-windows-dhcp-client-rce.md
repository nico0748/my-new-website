# Windows DHCP Client に CVSS 9.8 の未認証RCE（CVE-2026-44815）— 細工DHCP応答でスタックBOF

> トピック: 脆弱性ニュース ｜ 出典: Microsoft MSRC / NVD / Tenable ｜ 種別: スタックベースのバッファオーバーフロー（CWE-121）に起因する未認証リモートコード実行（RCE）

## 一行サマリー

ほぼ全てのWindows端末が動かしている「DHCPクライアント」（IPアドレスを自動取得する機能）に、悪意あるDHCP応答を送り込むだけで認証もユーザー操作も不要で任意コードを実行できる CVSS 9.8 の脆弱性 CVE-2026-44815 が見つかった。同一ネットワークに攻撃者がいれば端末を乗っ取られうるため、6月の月例パッチを最優先で適用する必要がある。

## 🔰 初学者向け（何が・なぜ問題か）

パソコンをWi-Fiや有線LANにつなぐと、何も設定しなくても自動的にIPアドレス（ネット上の住所）が割り当てられます。この「住所の自動受け取り」を担うのが **DHCPクライアント** という、Windowsに最初から入っている小さな仕組みです。仕組みとしては、パソコンが「住所をください」と呼びかけ、ネットワーク上の **DHCPサーバー**（住所を配る係）が「この住所を使ってね」と返事をする、というやり取りをしています。

今回の問題は、その「返事」の中身をパソコン側が**鵜呑みにして処理してしまう**点にあります。返事の手紙には、住所のほかにも「ご近所さんの一覧」「ゴミ出しのルール」など、いろいろな付帯情報を書き込める欄（オプション欄）があります。攻撃者がこの欄に、本来の枠をはるかに超える長すぎる文字列を詰め込んだニセの返事を送ると、パソコンが用意していた小さなメモ用紙（メモリ領域）からあふれ出し、隣の大事なメモ（プログラムの次の行き先を記したメモ）まで書き換えてしまいます。

たとえるなら、受付係が来客カードを所定の枠に書き写すとき、枠を無視した超長文を渡されてはみ出させ、その先にある「次に何をするかの指示書」まで攻撃者の好きな内容に上書きされてしまうイメージです。指示書を奪われた受付係は、攻撃者の命令どおりに動く（＝端末が乗っ取られる）ことになります。

なぜ深刻かというと、攻撃に **ログインも、利用者のクリックも一切不要**だからです。攻撃者が同じネットワーク（社内LAN、カフェや空港の公共Wi-Fi、ホテルの回線など）にいて細工した応答を流すだけで成立します。ネットにつなぐという誰もが毎日する操作が、そのまま侵入の入口になりうる点が問題です。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **CVSS**: 9.8（Critical）。**未認証・ネットワーク経由・低複雑度・ユーザー操作不要**で悪用可能。
- **CWE分類**: CWE-121（Stack-based Buffer Overflow）。DHCPサーバー応答のオプション欄処理における境界チェック不備。
- **前提条件**: 攻撃者が標的と**同一L2セグメント（隣接ネットワーク）**、または応答が到達するルーティング経路上にいること。標的がDHCPで構成を取得する通常運用であれば成立条件を満たす。
- **影響バージョン**: Windows 10（1607〜22H2）、Windows 11（23H2〜26H1）、Windows Server 2012〜2025 までの広範なサポート対象（正確な対象ビルドは MSRC の更新ガイドで要確認）。
- **修正版・対応**: Microsoft が2026年6月の月例セキュリティ更新で修正済み。Windows Update / WSUS / 資産管理ツール経由で該当KBを適用。
- **悪用状況**: 本稿時点で**公開PoCや実環境悪用の確認情報はなし（要確認）**。ただしDHCPは攻撃面が極めて広く、パッチ解析からの武器化リスクが高い。

### メカニズム（攻撃が成立する仕組み）

以下は概念レベルの説明であり、武器化可能な手順・PoC・正確なオフセット等は含めない。

1. **信頼境界の所在**: DHCPクライアントは「ネットワークから来る未認証のサーバー応答」を、起動直後・ネットワーク接続直後という**最も早い段階**で解析する。この時点では応答の真正性を検証する手段がほぼなく、外部入力を信頼して処理せざるを得ない危険地帯となる。
2. **TLV解析の不備**: DHCPの応答（DHCPOFFER / DHCPACK）は、各オプションを「型（Type）・長さ（Length）・値（Value）」の3つ組（TLV）で表現する。クライアントは宣言された長さに従って値を固定長のスタックバッファへコピーするが、**宣言長と実バッファサイズの突き合わせ検証が欠如**しているため、長さを過大に偽ったオプションで範囲外書き込み（オーバーフロー）が起きる。
3. **制御フローの奪取**: スタック上で隣接する**保存された戻りアドレス（return address）や例外ハンドラ（SEH）レコード**が攻撃者データで上書きされる。これにより、関数からの復帰時や例外発生時に制御が攻撃者の意図したアドレスへ移り、任意コード実行（RCE）に至る。
4. **未認証で完結する致命性**: 上記が認証前・操作前の経路で成立するため、攻撃者は資格情報もユーザーの協力も不要。隣接ネットワークにいる中間者やならず者DHCPサーバーが、応答を送るだけで端末制御を奪える。

脆弱性クラスとしては、「未検証の外部入力を固定長スタックバッファへコピーする古典的スタックBOF」に該当する。ネットワークプロトコルパーサにおける長さフィールド検証の欠如は、最も再発の多い実装欠陥の一つである。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: 同一セグメント上に**正規でないDHCPサーバー（rogue DHCP）**が出現していないかの監視、DHCP応答パケットの異常な長大オプション、端末でのDHCPクライアントサービス（`dhcp`）の異常終了・再起動・予期しない子プロセス生成。スイッチの **DHCP Snooping** ログや IDS/IPS のDHCP関連シグネチャを活用。
- **暫定緩和（パッチ前）**: スイッチで **DHCP Snooping** を有効化し、信頼ポート以外からのDHCPサーバー応答を遮断。重要端末・サーバーで可能なら静的IP運用を検討。公共Wi-Fiなど信頼できないネットワークへの接続を制限。
- **恒久対策**: 2026年6月の月例更新を即時適用。クライアントだけでなく**Windows Server全般**も対象である点に注意（DCやインフラサーバーも該当）。
- **運用示唆**: DHCPは「最初にネットワークから受け取る未認証データ」であり、境界VPNと並ぶ"認証前コード実行"の温床。パッチ適用とL2セグメント設計（Snooping・ポートセキュリティ）を恒常原則とする。

## 📖 用語解説（このニュースとのつながり）

- **DHCP / DHCPクライアント**: IPアドレス等のネットワーク設定を自動配布・受領する仕組み。意味＝「住所の自動受け取り係」。本件＝この受領処理に欠陥があり乗っ取りの入口になる。
- **TLV（Type-Length-Value）**: 可変長データを「型・長さ・値」で表す汎用エンコード。本件＝長さフィールドを偽られてオーバーフローが起きる。
- **スタックバッファオーバーフロー（CWE-121）**: 関数のローカル領域（スタック）の固定枠を超えて書き込む欠陥。本件＝戻りアドレス上書き→制御奪取の起点。
- **戻りアドレス / SEH**: 関数復帰先や例外処理先を指す制御データ。本件＝これを上書きされるとコード実行に直結。
- **rogue DHCP / DHCP Snooping**: 不正なDHCPサーバーと、それを遮断するスイッチ機能。本件＝暫定緩和と検知の主役。

## 影響範囲 / 推奨対応

影響を受けるのは、DHCPで構成を取得する**ほぼ全てのサポート対象Windowsクライアント／サーバー**。社内LAN、VDI、公共Wi-Fiを使う端末すべてが潜在的標的となる。推奨対応は、(1) 2026年6月月例更新の即時適用（クライアント・サーバー双方）、(2) スイッチでのDHCP Snooping有効化とrogue DHCP監視、(3) 信頼できないネットワーク接続の制限、(4) DHCPクライアントサービスの異常終了・クラッシュの監視。
