# PWA が主流化 — オフライン・プッシュ・バックグラウンド同期が標準に

> トピック: 開発トピック｜出典: sencha（blog）｜種別: frontend

## 一行サマリー
PWA（Progressive Web App）が主流化し、オフライン動作・プッシュ通知・バックグラウンド同期といった「ネイティブアプリ的な機能」を、ウェブアプリで標準的に提供できるようになった。

## 🔰 初学者向け（何が・なぜ重要か）
普通のウェブサイトは「ネットに繋がっていないと開けない」「通知が来ない」「ホーム画面にアプリとして置けない」といった制約がありました。一方でスマホのネイティブアプリはそれができる。この差を埋めるのが PWA です。

PWA は「アプリのように振る舞えるウェブサイト」だと思ってください。裏で動く小さな常駐プログラム（Service Worker）がページの部品を端末に保存しておくので、電波が悪い場所でも開ける。ホーム画面にアイコンを置ける。通知も送れる。たとえるなら、ブラウザで見る普通のページに「オフラインでも動く・通知が来る」という魔法をかけたもの。ストア審査なしで配布でき、更新も即時。だから多くのサービスが、専用アプリを作る前にまず PWA 化を選ぶようになっています。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
PWA の中核は Service Worker（ネットワークを介在してキャッシュ制御するプロキシ）、Web App Manifest（インストール情報）、および Push API / Background Sync といった Web API。近年これらのブラウザ対応が成熟し（iOS Safari のプッシュ対応など）、実用の壁が下がった。Vite の `vite-plugin-pwa`（Workbox ベース）などでキャッシュ戦略・SW 生成・manifest を自動化でき、フレームワーク側の導入コストも低い。

### 仕組み / 背景（なぜこう設計されたか）
Service Worker はページとネットワークの間に立ち、リクエストを横取りしてキャッシュから返す/更新するを制御する。これによりオフライン・高速再訪問・バックグラウンド処理が可能になる。設計思想は「プログレッシブ（段階的）」で、対応環境では強化機能を、非対応でも通常のサイトとして動く。トレードオフは、キャッシュ戦略を誤ると古い内容が残る・更新が反映されないといった SW 特有の落とし穴。

### 実務での使いどころ・移行の勘所
オフライン閲覧・再訪問の高速化・プッシュ再訪促進が効くサービス（メディア、EC、業務ツール）に好適。`vite-plugin-pwa` の `runtimeCaching` でリソース別にキャッシュ戦略を設計し、SW 更新時のバージョン反映（skipWaiting/更新通知 UX）を丁寧に扱う。当プロジェクト（ポートフォリオサイト）でも同プラグインで SW・webmanifest を自動生成している。

## 📖 用語解説（このトピックとのつながり）
- **Service Worker**: ネットワークを仲介しキャッシュ制御する常駐スクリプト。PWA の心臓部。
- **Web App Manifest**: アイコン・名称・表示モードを定義しインストールを可能にする設定。
- **Push API**: ウェブからプッシュ通知を送る仕組み。再訪問を促す。
- **Background Sync**: オフライン時の操作を復帰後に同期する API。
- **Workbox / vite-plugin-pwa**: SW とキャッシュ戦略を自動生成する定番ツール。

## 影響範囲 / 推奨アクション
再訪問・オフライン・通知が価値になるウェブサービスの開発者が対象。`vite-plugin-pwa` 等で PWA 化を導入し、キャッシュ戦略と SW 更新 UX を設計する。古いキャッシュが残る事故を防ぐため、更新反映フローの検証を必ず行う。
