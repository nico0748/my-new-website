# Biome v2.4 — 埋め込みCSS/GraphQLの整形とHTMLアクセシビリティ対応

> トピック: 開発トピック｜出典: Biome 公式ブログ｜種別: frontend

## 一行サマリー
Rust 製オールインワン・ツールチェーン Biome の v2.4 が 2026 年最初のマイナーとして公開され、JS/TS 内のテンプレートリテラルに埋め込まれた CSS・GraphQL の整形とリント、HTML のアクセシビリティ検査、フレームワーク（Qwik 等）向けルールを追加した。ESLint + Prettier を一本化する選択肢がさらに実用的になった。

## 🔰 初学者向け（何が・なぜ重要か）
コードを書くとき、私たちは2種類の「お掃除係」を使ってきました。一つは Prettier のような「整形係」で、インデントや改行を揃えて見た目を整えます。もう一つは ESLint のような「検査係」で、バグの芽や危険な書き方を指摘します。従来はこの2つを別々に入れ、設定も別々に管理していました。

Biome は、この整形と検査を1本にまとめた道具です。しかも Rust で書かれているため非常に速い。v2.4 では、JavaScript の中に文字列として埋め込んだ CSS や GraphQL（例：`css\`...\`` のような書き方）も、外側と同じように整形・検査できるようになりました。これまで「文字列の中身」は掃除の対象外で見過ごされがちでしたが、そこにも目が届くようになった、という改善です。

さらに HTML の「アクセシビリティ（障害のある人にも使えるか）」の検査や、Qwik のような特定フレームワークの正しい使い方チェックも追加され、1つの道具でカバーできる範囲が広がりました。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Biome v2.4 は 2026 年最初のマイナーリリース。主な追加は (1) JS/TS のテンプレートリテラル内に埋め込まれた CSS・GraphQL スニペットの整形・リント（自動最適化を含む）、(2) HTML のアクセシビリティ関連ルール、(3) フレームワーク特化ルール（Qwik 向けの `useQwikMethodUsage`・`useQwikValidLexicalScope` など）。Biome は単一バイナリでフォーマッタとリンタを兼ね、Prettier 互換の整形と ESLint 相当のルール群を提供する。正確なルール一覧・既定の有効/無効は公式チェンジログで要確認。

### 仕組み / 背景（なぜこう設計されたか）
ESLint（JS）+ Prettier（整形）+ stylelint（CSS）… とツールが分かれると、パーサを重複して持ち、設定・プラグイン・実行時間が増える。Biome は Rust 製の高速パーサと共通の抽象構文木（AST）基盤上に整形・リント・（部分的に）型を意識した解析を集約し、単一ツールで多言語を扱う。埋め込みスニペット対応は、テンプレートリテラルを言語境界として認識し、内側を該当言語の AST として再解析することで実現している。トレードオフは、ESLint の巨大なプラグイン資産をそのまま使えない点で、独自ルールに依存するチームは移行時に代替を要する。速度と一体運用の利益と、既存プラグイン資産の断念を天秤にかける形になる。

### 実務での使いどころ・移行の勘所
CI のリント・整形時間がボトルネックの大規模モノレポで効果が大きい。新規は最初から Biome 単体運用が素直。既存は、まず整形を Prettier から Biome へ切り替え（差分を1コミットで吸収）、次にリントを ESLint と並走させながら段階的に移すのが安全。ESLint 固有プラグインに強く依存する場合は、対応ルールの有無を先に確認する。`css` / `graphql` タグ付きテンプレートを多用する CSS-in-JS 構成では v2.4 の埋め込み対応が直接効く。

## 📖 用語解説（このトピックとのつながり）
- **リンタ / フォーマッタ**: コードの問題を検出する道具と、体裁を整える道具。Biome は両者を1本化。
- **オールインワン・ツールチェーン**: 整形・リント等を単一ツールで提供する構成。設定と実行時間を削減。
- **テンプレートリテラル埋め込み**: JS 文字列内に CSS/GraphQL を書く手法。v2.4 で整形・検査対象に。
- **アクセシビリティ（a11y）ルール**: 支援技術で使えるかを検査するルール。HTML 対応が v2.4 で追加。
- **AST（抽象構文木）**: コードを木構造で表した内部表現。Biome の解析基盤で、言語横断の要。

## 影響範囲 / 推奨アクション
フロントエンドの品質・CI 運用に関わる全チームに関係。整形の一本化から段階導入するとリスクが低い。CSS-in-JS や GraphQL を多用するプロジェクトは v2.4 の埋め込み対応で恩恵が大きい。ESLint 固有プラグイン依存の有無を移行前に棚卸しすること。
