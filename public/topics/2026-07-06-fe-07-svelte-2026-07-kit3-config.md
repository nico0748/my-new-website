# SvelteKit、設定を vite.config へ集約 — Kit 3 への布石（2026年7月）

> トピック: What's new in Svelte 2026年7月｜出典: Svelte 公式ブログ｜種別: マイナーリリース群（Kit 3 プレビュー）

## 一行サマリー

2026年7月の SvelteKit（2.62.0〜2.66.0）は、`svelte.config.js` を廃し設定を `vite.config.js` に集約する方向を打ち出し、`$env/*` を置き換える明示的環境変数の初プレビュー、リモート関数の File 受け取りやクエリ間リフレッシュなどを追加した。いずれも次期メジャー「SvelteKit 3」への地ならしとなる変更。

## 🔰 初学者向け（何が・なぜ重要か）

Web アプリを作ると、設定ファイルがいくつも増えていきます。SvelteKit ではこれまで「フレームワーク用の `svelte.config.js`」と「ビルドツール Vite 用の `vite.config.js`」の2つを持つのが普通でした。今回のアップデートは、この2つを1つ（`vite.config.js`）にまとめられるようにする、という「散らかった机を片づける」系の改善です。設定の置き場所が一本化されると、初学者が「どっちに何を書くんだっけ？」と迷う場面が減ります。

もう一つ大事なのが「環境変数」の扱いの見直しです。環境変数とは、APIキーや接続先URLのように「開発環境と本番環境で中身を差し替えたい値」のこと。SvelteKit は従来 `$env/...` という特別な入口で読み込ませていましたが、これを「どの変数を使うか明示的に宣言する」方式へ変えるプレビューが始まりました。何をどこで使っているかがハッキリするので、秘密情報の取り違えや、思わぬ変数の混入を防ぎやすくなります。

これらは単体では地味ですが、まとめて「SvelteKit 3 でこうなる」という将来像の先出しです。破壊的変更は事前に小出しにして慣れさせる——という、利用者に優しい進め方といえます。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

7月の主な変更点は次の通り（バージョンは公式ブログ準拠）。

- **設定の集約（2.62.0）**: SvelteKit の設定を `vite.config.js` 内に直接定義でき、`svelte.config.js` を省略可能に。これは Kit 3 が設定を `vite.config.js` に置くことを要求する方向のプレビュー。
- **明示的環境変数のプレビュー**: `$env/*` モジュールを将来置き換える、環境変数を明示宣言する仕組みの初プレビュー（SvelteKit 3 で本格化予定）。
- **リモート関数の強化**: リモート関数コマンドが `File` オブジェクトを直接受け取れる（2.64.0）、リモートクエリが他のクエリをリフレッシュできる（2.65.0）。
- **ビルド/検証まわり**: プリレンダリングした `.md`/`.mdx` を HTML・JS・CSS と同様に事前圧縮（2.66.0）、リモートフォームスキーマの boolean フィールドが optional 未指定の場合に警告（2.66.0）。
- **ツールチェーン同期**: language tools と `sv` CLI が Svelte の新しい `{const ...}` 宣言タグに追随し、エコシステム全体が整合。

### 仕組み / 背景（なぜこう設計されたか）

SvelteKit は Vite 上に構築されるメタフレームワークで、これまでフレームワーク固有設定を `svelte.config.js` に分離していた。しかし設定が二重化すると、プラグイン解決・環境ごとの挙動・型推論の起点が分かれ、認知負荷とバグの温床になる。Vite 側が Environment API などで「1つの設定から本番同等のランタイムを再現する」方向へ進んだこともあり、設定の単一情報源（single source of truth）を Vite 側へ寄せるのが理にかなう。環境変数の明示化も同じ思想で、「暗黙のマジックモジュール」から「宣言された依存」へ移すことで、静的解析・秘密情報の追跡・ツリーシェイクを効かせやすくする。代償は移行コスト（設定の書き換え）だが、破壊的変更をマイナーで予告してから本番投入する段階的リリースで緩和している。

### 実務での使いどころ・移行の勘所

既存プロジェクトは今すぐ移行する必要はないが、Kit 3 を見据えるなら「設定を `vite.config.js` に寄せる」「環境変数の使用箇所を棚卸しする」ことを前倒しで始めると移行が滑らか。リモート関数で `File` を直接扱えるようになったため、アップロード系の実装を簡素化できる。`.md`/`.mdx` の事前圧縮はコンテンツ系サイトで転送量削減に効く。boolean の optional 警告は既存フォームで大量に出る可能性があるので、CI を割る前にスキーマを見直しておくとよい。

## 📖 用語解説（このトピックとのつながり）

- **メタフレームワーク**: ルーティング・SSR・ビルドをまとめて提供する上位フレームワーク。SvelteKit は Vite 上のそれ。
- **リモート関数（remote functions）**: サーバー側関数をクライアントから型安全に呼ぶ SvelteKit の仕組み。今回 File 受け取りとクエリ間リフレッシュが強化。
- **`$env/*`**: 環境変数を読み込む SvelteKit の仮想モジュール。明示宣言方式へ置き換えるプレビューが今回登場。
- **事前圧縮（precompress）**: 配信ファイルをビルド時に gzip/brotli 化しておく最適化。今回 `.md`/`.mdx` も対象に。

## 影響範囲 / 推奨アクション

- 対象: SvelteKit を使う全プロジェクト、特に Kit 3 移行を見据えるチーム。
- アクション: 新規は `vite.config.js` 集約で始める。既存は環境変数の使用箇所を棚卸しし、boolean スキーマの optional 警告に対応。破壊的変更はマイナーでの予告に沿って段階的に取り込む。
