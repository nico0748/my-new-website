# Apache HTTP Server 2.4.68 公開 — mod_proxy系を中心に12件の脆弱性を修正（JVNVU#99913823）

> トピック: 脆弱性ニュース ｜ 出典: JVN（JPCERT/CC・IPA）JVNVU#99913823 / Apache httpd Security Advisory ｜ 種別: 複数脆弱性（バッファオーバーフロー・DoS・XSS等）

## 一行サマリー

世界で最も使われるWebサーバの一つ Apache HTTP Server に多数の脆弱性を修正した 2.4.68 が公開された（JPCERT/CCも告知）。プロキシ機能（mod_proxy系）のバッファオーバーフロー・無限ループDoS・XSSなどを含み、2.4.0〜2.4.67 が対象。深刻度は低〜中だが利用範囲が広く、計画的な更新が必要。

## 🔰 初学者向け（何が・なぜ問題か）

Apache HTTP Server（通称 httpd）は、Webサイトを世界中の閲覧者に届ける“配膳係”のソフトです。ブラウザが「このページを見せて」と頼むと、httpdが該当ページを取ってきて返します。インターネット上の膨大なサイトがこのソフトで動いており、まさに縁の下の力持ちです。

今回、その httpd に複数の不具合が見つかり、修正版 2.4.68 が公開されました。特に多いのが「プロキシ機能」まわりの問題です。プロキシとは、httpd が“仲介役”として別のサーバ（バックエンド）に取り次ぐ働きのこと。受付が、奥の調理場に注文を伝えて料理を運んでくる、あの仲介役を想像してください。

今回の問題の多くは、その「奥の調理場」（バックエンドサーバやFTPサーバ）が悪意を持っていたり乗っ取られていたりした場合に、仲介役の httpd 側で不具合が起きる、というものです。たとえば、

- 仲介中にデータの置き場（バッファ）からあふれてしまう（バッファオーバーフロー）
- 仲介がぐるぐる止まらなくなりサービスが固まる（無限ループによるサービス停止）
- 仲介して表示する一覧ページに悪意あるスクリプトが紛れ込み、閲覧者のブラウザで実行される（XSS）

いずれも今回は「深刻度は低〜中」とされ、攻撃には特定の条件（プロキシ構成や悪意あるバックエンド）が必要です。とはいえ httpd は使われている数が桁違いに多いので、油断せず計画的に更新することが大切です。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

Apache Software Foundation は 2026-06-08 に httpd 2.4.68 をリリースし、12件の脆弱性（低〜中深刻度）を修正。対象は **2.4.0〜2.4.67** の全デプロイ。代表的なもの：

- **CVE-2026-34355（Moderate）**: `mod_proxy_html` のバッファオーバーフロー。前提＝**非信頼なバックエンドサーバ**が応答を返す構成で悪用可能。
- **CVE-2026-44186（Moderate）**: `mod_proxy_ftp` のハンドラで無限ループを誘発するDoS。前提＝**攻撃者が制御するバックエンドFTPサーバ**。
- **CVE-2026-29170（XSS）**: `mod_proxy_ftp` がFTPディレクトリ一覧を動的HTML生成する際の出力未サニタイズ（CWE-79）。前提＝forward/reverseプロキシでFTP内容を取り次ぐ構成。
- その他、WebDAV、HTTP/2、SSL 関連を含む広範なモジュールにまたがる修正。各CVEのCVSS厳密値・全リストは Apache 公式アドバイザリ参照（「要確認」）。
- **悪用状況**: 公開時点で実環境での広範な悪用報告は確認されず（「要確認」）。深刻度は低〜中で、前提条件依存。

### メカニズム（攻撃が成立する仕組み）

以下は概念レベルの説明であり、武器化可能な手順は含めない。

1. **信頼境界の所在**: プロキシ構成では「httpd（仲介）↔ バックエンド（被仲介サーバ）」の間に信頼境界がある。多くの実装はバックエンド応答を暗黙に信頼しがちだが、本来バックエンド応答も**非信頼入力**として扱うべき。今回の不具合群はこの前提のズレに起因する。
2. **応答処理のメモリ安全欠陥（34355）**: `mod_proxy_html` が悪意あるバックエンド応答（想定外サイズ／構造のHTML）を変換処理する際、境界チェック不足でバッファ境界を越える。メモリ破壊はクラッシュや、条件次第で挙動改変につながりうる。
3. **状態機械の停止条件欠落（44186）**: `mod_proxy_ftp` のハンドラが、攻撃者制御のFTP応答に対し終了条件を満たせず無限ループに陥る。これがワーカ枯渇＝サービス妨害（DoS）の橋渡しになる。
4. **出力エンコード不備（29170）**: プロキシ経由のFTPディレクトリ一覧を動的HTMLに埋め込む際、ファイル名等の出力を文脈別エスケープしないため、悪意ある名前がブラウザでスクリプトとして解釈される（反射型に近いXSS）。被害は閲覧者セッションのスクリプト実行（クッキー窃取・操作偽装）。
5. **被害の橋渡し**: いずれも「非信頼バックエンド／非信頼応答」を信頼してしまう設計上のギャップが核。メモリ安全・状態機械・出力エンコードという別レイヤの不備が、それぞれクラッシュ／DoS／XSSという異なる最終被害に至る。

脆弱性クラスとしては、(a) 非信頼入力（バックエンド応答）に対する境界検証不足、(b) 状態機械の終了条件欠落によるDoS、(c) 動的HTML生成時の出力エスケープ漏れ（XSS）という、Web基盤で繰り返し現れる古典クラス。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: httpd ワーカの異常クラッシュ・CPU張り付き（DoS兆候）、プロキシ経由ページでの不審スクリプト、mod_proxy_ftp 経由アクセスの異常。エラーログ（segfault/child process exit）とアクセスログを突合。
- **暫定緩和（パッチ前）**: 不要な `mod_proxy_html` / `mod_proxy_ftp` を無効化。FTPプロキシの利用を停止・制限。プロキシ先（バックエンド）を信頼できるものに限定し、外部の不審バックエンドへの取り次ぎを遮断。WAFでプロキシ応答由来のスクリプト混入を低減。
- **恒久対策**: 2.4.68 へ計画的にアップグレード。広域に展開している場合は段階適用＋回帰テストを併用。
- **運用示唆**: 利用範囲が極めて広いミドルウェアは、深刻度が中程度でも「適用対象の母数」が大きく総リスクが高い。資産棚卸しで自組織のhttpdバージョンと有効モジュールを把握し、定期更新の運用に組み込む。

## 📖 用語解説（このニュースとのつながり）

- **Apache HTTP Server（httpd）**: 代表的なオープンソースWebサーバ。本件＝広く使われる基盤に複数修正が入った。
- **mod_proxy（プロキシモジュール）**: httpd が他サーバへ要求を取り次ぐ機能群。本件＝修正の中心。
- **バッファオーバーフロー（CWE-120系）**: 確保領域を超えてデータを書き込む欠陥。本件＝CVE-2026-34355。
- **DoS（サービス妨害）**: サービスを使用不能にする攻撃。本件＝無限ループ型のCVE-2026-44186。
- **XSS（クロスサイトスクリプティング, CWE-79）**: 出力未サニタイズで他者ブラウザにスクリプトを実行させる欠陥。本件＝CVE-2026-29170。
- **バックエンドサーバ**: プロキシの取り次ぎ先。本件＝多くの不具合は「悪意あるバックエンド」を前提に成立する点が重要。

## 影響範囲 / 推奨対応

影響を受けるのは Apache HTTP Server 2.4.0〜2.4.67 を運用する全環境。特にプロキシ機能（mod_proxy_html / mod_proxy_ftp）を有効化し、信頼できないバックエンドへ取り次ぐ構成はリスクが相対的に高い。推奨対応は、(1) 2.4.68 への計画的アップグレード、(2) 不要なプロキシ系モジュールの無効化、(3) プロキシ先の信頼境界の見直し、(4) クラッシュ・CPU異常・不審スクリプトの監視。
