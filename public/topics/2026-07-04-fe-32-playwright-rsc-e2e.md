# Playwright で async Server Components を E2E 検証する

> トピック: 開発トピック｜出典: medium（securestartkit）｜種別: frontend

## 一行サマリー
非同期な React Server Components（RSC）はユニットテストで単体検証しづらいため、Playwright による E2E（実ブラウザで実際の画面遷移を辿る）テストが、特に認証フローの検証で標準的な手段になっている。

## 🔰 初学者向け（何が・なぜ重要か）
最近の Next.js では、画面の一部を「サーバー側で組み立ててから届ける部品（Server Components）」として書けます。しかもデータ取得を待つ非同期（async）な部品も普通です。便利な反面、これらは「サーバーで動く」ため、ブラウザだけを模した従来のユニットテストでは中身をそのまま呼び出して確かめにくいという難しさがあります。

そこで登場するのが Playwright です。これは「本物のブラウザを自動で操作するロボット」だと思ってください。人間の代わりにページを開き、ログインし、ボタンを押し、画面に正しい文字が出たかを確認します。サーバーで組み立てられた結果が最終的にユーザーの画面へどう届くかを、丸ごと通しで検証できるのです。特にログイン・認可のように「サーバーとブラウザが行き来する流れ」は、通し（E2E）で見ないと安心できません。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
2026 の Next.js テスト構成は「Vitest（ユニット・コンポーネント）＋ Playwright（E2E）」の二層が定番。async Server Components は React 公式もユニットテストを現状非推奨扱いとしており、実行環境全体を起動して外側から検証する E2E が現実解となる。Playwright は複数ブラウザエンジン（Chromium/Firefox/WebKit）対応・自動待機（auto-waiting）・トレースビューアを備え、認証済みセッションの `storageState` 再利用でログインフローの高速化も可能。

### 仕組み / 背景（なぜこう設計されたか）
RSC はサーバーでレンダリングされ、クライアントへはシリアライズされた結果が流れる。この境界をまたぐ挙動（Suspense のストリーミング、Cookie/セッションに依存する分岐など）は、モックで再現すると実態とズレやすい。実ブラウザ＋実サーバーを通す E2E なら境界を意識せず「ユーザーが見る結果」を直接アサートできる。トレードオフは実行が重く遅いこと。ゆえに全部を E2E にせず、テストピラミッド（ユニット多・E2E 少）で配分する。

### 実務での使いどころ・移行の勘所
ログイン・権限・決済など「壊れると致命的なフロー」を E2E で薄く固める。UI ロジックや純粋関数は Vitest 側に寄せる。CI では Playwright のブラウザバイナリキャッシュと並列シャーディングで時間を抑える。認証は毎回ログインせず `storageState` を使い回すのが勘所。

## 📖 用語解説（このトピックとのつながり）
- **E2E テスト**: ユーザー操作を端から端まで通す検証。RSC の境界越えを丸ごと確認できる。
- **RSC（React Server Components）**: サーバーで描画される React 部品。ユニット検証が難しく E2E と補完関係。
- **auto-waiting**: 要素が操作可能になるまで自動で待つ Playwright の機能。フレークを減らす。
- **storageState**: ログイン済みのブラウザ状態を保存・再利用する仕組み。認証テストを高速化。
- **テストピラミッド**: ユニット多め・E2E 少なめが理想という配分の考え方。

## 影響範囲 / 推奨アクション
Next.js（App Router / RSC）を使うチームが対象。認証・課金など重要フローに Playwright E2E を最小限導入し、それ以外は Vitest に寄せる。CI では並列化とセッション再利用で実行時間を管理する。
