# Prefill/Decode 分離サービング — vLLM の大規模推論を「速度の光」に近づける設計

> トピック: Prefill/Decode 分離（vLLM 大規模サービング）｜出典: vLLM ロードマップ/ブログ・vLLM Docs｜種別: 推論基盤 / MLOps

## 一行サマリー

LLM 推論の「プロンプトを一気に処理する prefill」と「1 トークンずつ生成する decode」を別々の vLLM インスタンスに分離する Prefill/Decode 分離が、大規模サービングの標準設計になりつつある。TTFT（最初の 1 トークンまで）と ITL（トークン間遅延）を独立にチューニングでき、GB200/B200 クラスタ上で NVFP4/FP8 カーネルや Wide-EP と組み合わせて H200 1 枚あたり毎秒 2.2k トークン級の MoE サービングを狙う。

## 🔰 初学者向け（何が・なぜ重要か）

LLM が文章を生成するとき、処理は大きく2段階に分かれる。まず入力プロンプト全体を一気に読み込む「prefill（前処理）」、次に答えを 1 文字ずつ紡ぐ「decode（生成）」だ。この2つは**性質がまるで違う**。prefill は「大量の入力をまとめて計算する（計算パワー勝負）」、decode は「1 トークンずつ、過去の文脈を参照しながら小刻みに計算する（メモリ帯域勝負）」。

これまでは同じ GPU で両方をこなしていたが、性質が違うため互いに足を引っ張り合う。長いプロンプトの prefill が走っている間、他のリクエストの decode が待たされる、といった具合だ。たとえるなら、レストランで「大量の仕込み」と「一皿ずつの盛り付け」を同じ調理台でやると渋滞する。

Prefill/Decode 分離は、この2工程を**別の GPU（別チーム）に分ける**発想。仕込み専門と盛り付け専門に分ければ、それぞれを最適な体制に組め、渋滞が減って全体が速くなる。大規模サービスで応答速度とスループットを両立するための鍵になっている。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **仕組み**: prefill と decode を別の vLLM インスタンスに配置。各段に異なる並列戦略を割り当て、TTFT を ITL に影響させずに調整（またはその逆）できる。
- **KV 転送**: prefill が生成した KV キャッシュを decode 側へ受け渡す。分散 KV ストア/多層キャッシュ（例: FlexKV）で超大規模の KV 管理を行う。
- **大規模最適化**: NVIDIA GB200 上で NVFP4/FP8 カーネル、Wide Expert Parallelism（Wide-EP）、カーネル融合、weight offloading、チャンク化オーバーヘッド削減により DeepSeek 型 MoE サービングを高速化。
- **到達性能（要確認）**: Wide-EP・非同期スケジューリング・dual-batch overlap・分離サービング・CUDA graphs・DeepGEMM・エキスパート負荷分散の併用で、H200 1 枚あたり毎秒 2.2k トークンの MoE サービングに到達。
- **ロードマップ（Q2 2026）**: 過剰なプリエンプション回避、prefill の Head-of-Line ブロッキング、PP/DP バランスなどスケジューラ課題に注力。

### 仕組み / 背景（なぜこう設計されたか）

LLM 推論は2つのフェーズで計算特性が対照的だ。prefill は入力トークン全体を並列に処理する **compute-bound**（演算律速）、decode は自己回帰で 1 トークンずつ進む **memory-bound**（メモリ帯域律速）。両者を同一インスタンスで混在させると、(1) 長い prefill が decode の応答を止める Head-of-Line ブロッキング、(2) それぞれに最適な並列度（テンソル並列・パイプライン並列・エキスパート並列）を同時に満たせない、という矛盾が生じる。

分離サービングは「フェーズごとに独立したリソースプールと並列戦略を持たせる」ことでこれを解く。prefill 側は演算スループット重視、decode 側はメモリ帯域と低遅延重視に振れる。両者をつなぐのが **KV キャッシュの受け渡し**で、prefill が計算した文脈状態（KV）を decode へ転送する必要があり、ここのネットワーク帯域と KV ストア設計が全体性能を左右する。トレードオフは、(1) KV 転送のコストとレイテンシが新たなボトルネックになり得る、(2) インスタンス間の負荷比（prefill 対 decode の比率）を負荷変動に合わせて調整する運用が要る、(3) 実装・スケジューラが複雑化する点。だからこそ vLLM のロードマップはプリエンプションやバランス調整といったスケジューラ課題に注力している。

### 実務での使いどころ・移行の勘所

大規模 MoE や長プロンプトを高スループット・低 TTFT でサービングしたい本番推論に効く。移行の勘所は、(1) まず単一インスタンスでボトルネック（TTFT か ITL か）を計測し、分離で改善する余地を見極める、(2) prefill:decode のインスタンス比を負荷特性に合わせて調整、(3) KV 転送帯域が律速にならないよう高速インターコネクト（NVLink/InfiniBand 等）と KV ストア設計を用意、(4) NVFP4/FP8 や Wide-EP と併用して MoE の実効効率を上げる。小〜中規模では単一インスタンスの方が簡潔で有利なことも多く、分離は大規模・高負荷が前提。性能値は構成依存が大きいので自環境で実測（要確認）。

## 📖 用語解説（このトピックとのつながり）

- **prefill / decode**: 入力一括処理と逐次生成の2段階。前者は演算律速、後者はメモリ帯域律速。
- **TTFT / ITL**: 最初の 1 トークンまでの時間と、トークン間遅延。分離で独立に調整可能。
- **KV キャッシュ**: 既処理トークンの文脈状態。prefill→decode で転送が必要。
- **Wide-EP（Wide Expert Parallelism）**: MoE のエキスパートを広く分散配置する並列戦略。
- **Head-of-Line ブロッキング**: 先頭の重い処理が後続を止める現象。混在サービングの弊害。

## 影響範囲 / 推奨アクション

- 大規模 MoE/長プロンプトの本番推論で TTFT と ITL を両立でき、GPU 実効効率が上がる。
- まず単一構成でボトルネックを計測し、prefill:decode 比・KV 転送帯域を設計。
- 小〜中規模は単一構成が簡潔で有利な場合も。性能はハード/構成依存のため実測を（要確認）。
