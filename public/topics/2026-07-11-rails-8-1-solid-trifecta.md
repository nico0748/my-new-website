# Rails 8.1 と Solid トリオ — Redis に頼らないジョブ・キャッシュ・WebSocket

> トピック: Rails 8.1 系と Solid Queue / Solid Cache / Solid Cable ｜ 出典: Ruby on Rails 公式 / 各種解説 ｜ 種別: フレームワークの構成思想

## 一行サマリー

Ruby on Rails 8 系が標準採用した「Solid トリオ」（Solid Queue＝ジョブ、Solid Cache＝キャッシュ、Solid Cable＝WebSocket）は、いずれも既存のデータベースを裏側に使い、Redis や Memcached といった外部ミドルウェアなしで動く。Rails 8.1.3 が 2026-03 に出て 8.1 系は同年10月まで保守され、「1台のサーバーとDBだけ」でのデプロイを現実化した。

## 🔰 初学者向け（何が・なぜ重要か）

Web アプリを本番運用するとき、これまでは「本体のデータベース」以外にも、いくつかの脇役サーバーが必要だった。重い処理を後回しにする**ジョブ用**、よく使うデータを一時保存する**キャッシュ用**、チャットのようなリアルタイム通信をさばく**WebSocket 用**——それぞれに Redis などの別サーバーを立てるのが定番だった。これらは便利だが、「立てる・監視する・障害に備える」運用コストがかさむ。

**Rails 8 の「Solid トリオ」は、この脇役たちを本体のデータベース1つで肩代わりする仕組みだ**。ジョブもキャッシュもリアルタイム通信も、専用サーバーを立てず、いつもの DB（PostgreSQL/MySQL/SQLite）に相乗りさせる。結果、構成がぐっとシンプルになり、極端には「アプリ 1 台＋DB 1 個」でまともに動くサービスが作れる。

なぜ今これが現実的かというと、**SSD が速くなった**からだ。かつては「DB は遅いからキャッシュは別に Redis で」という常識だったが、現代の SSD なら DB でも十分速い。個人開発や小〜中規模サービスでは、この「脇役を減らす」発想はコストと運用の両面で効く。実際 Basecamp は Solid Cache で 10TB 規模のキャッシュを運用している。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

Rails 8 系はデフォルトで 3 つの DB バックエンド部品を採用する。

- **Solid Queue**: DB を裏に使う Active Job バックエンド。ジョブを PostgreSQL/MySQL/SQLite に直接格納。Sidekiq/Resque/Delayed Job と Redis の組み合わせを不要にし、**トランザクション保証**と組み込みの同時実行制御を提供。
- **Solid Cache**: DB バックエンドの `ActiveSupport::Cache::Store`。現代 SSD の性能を活かし、Redis/Memcached を既定キャッシュとして置換。Basecamp は 60日保持で 10TB を格納しレンダリング時間を大幅短縮。
- **Solid Cable**: Action Cable（WebSocket）の pub/sub を DB で実現。受信メッセージを DB テーブルに書き、各 Action Cable インスタンスが既定 100ms 間隔でポーリングして配信。外部 pub/sub を不要に。

バージョンは Rails 8.1.3 が 2026-03-24 リリースで、8.1 系はバグ修正を 2026年10月まで継続。

### 仕組み / 背景（なぜこう設計されたか）

**No-Redis / "one person framework" 思想**。Rails 作者 DHH（37signals）は「少人数でも本格アプリを作れる」ことを重視し、依存ミドルウェアを削る方向を推進してきた（Kamal によるデプロイ簡素化と同じ流れ）。Solid トリオはその中核で、運用対象を減らすことで小規模チームの負担を下げる。

**なぜ DB で足りるのか**。従来 Redis が選ばれた最大の理由は「メモリ上で速い」こと。しかし NVMe SSD の普及で DB の I/O が劇的に速くなり、キャッシュやジョブキューの多くのワークロードでは DB でも実用十分になった。Solid Cache は SSD 前提の設計で大容量・低コストのキャッシュを狙い、Solid Queue は「ジョブとビジネスデータを同一トランザクションで扱える」という DB ならではの利点（ジョブ登録とデータ更新の原子性）を得る。

**トレードオフ**。(1) Solid Cable のポーリング（既定100ms）は、Redis の即時 pub/sub と比べ僅かな遅延がある——超低遅延が必須のリアルタイム用途では Redis が優位。(2) すべてを DB に集約するため、DB が単一障害点・性能ボトルネックになりやすく、負荷が高い規模では DB のスケーリング設計が要る。(3) 大規模・高スループット環境では専用ミドルウェアの方が総合効率で勝る場合もある。要は「規模とのトレードオフ」で、小〜中規模で最も輝く。

### 実務での使いどころ・移行の勘所

個人開発、スタートアップ、社内ツール、小〜中規模 SaaS で、インフラ構成の単純化に大きく効く。新規 Rails 8 アプリは既定で Solid トリオが入るためそのまま使える。既存アプリからの移行は部品単位で可能——まず Solid Queue や Solid Cache から段階導入し、負荷特性を測る。超低遅延の WebSocket や極端な高スループットが要る場合は、その部分だけ Redis を残すハイブリッドが妥当。DB を全機能の受け皿にする以上、DB のサイジングと監視（テーブル肥大・ポーリング負荷）は重点的に見る。

## 📖 用語解説（このトピックとのつながり）

- **Active Job**: Rails のジョブ抽象レイヤ。Solid Queue はその DB バックエンド実装。
- **pub/sub**: 発行者と購読者を仲介するメッセージ配信方式。WebSocket 配信の基盤で、Solid Cable が DB で実現。
- **Action Cable**: Rails の WebSocket フレームワーク。Solid Cable がその pub/sub を担う。
- **トランザクション保証**: 複数操作を全部成功か全部失敗にまとめる DB の性質。ジョブ登録とデータ更新の原子性に効く。
- **Kamal**: 37signals 製のデプロイツール。少数運用・脱クラウド依存の思想を Solid トリオと共有。
- **単一障害点（SPOF）**: そこが落ちると全体が止まる箇所。DB 集約はここに注意が要る。

## 影響範囲 / 推奨アクション

- 小〜中規模 Rails アプリはインフラ簡素化とコスト削減の観点で Solid トリオ採用を検討。
- 移行は部品単位（Queue → Cache → Cable）で段階的に。負荷特性を計測しながら進める。
- 超低遅延 WebSocket・超高スループットが要る箇所は Redis 併用のハイブリッドを。DB のサイジング・監視を重点化。
