# WordPress プラグイン Avada Builder の認証不要 任意ファイル削除 CVE-2026-8713（約100万サイト）

> トピック: 脆弱性ニュース ｜ 出典: Cybersecurity News / SecurityOnline / Wordfence ｜ 種別: 不適切なパス検証による認証不要の任意ファイル削除（CWE-22/パストラバーサル系）

## 一行サマリー

人気の WordPress テーマビルダー「Avada（Fusion）Builder」に、認証不要で任意ファイルを削除できる脆弱性 CVE-2026-8713（CVSS 9.1）があり、wp-config.php 等の重要ファイル削除を通じてサイト乗っ取りに繋がりうる。約100万サイトが影響し、v3.15.4 への即時更新が必要（DBにエントリ保存するフォーム公開が前提条件）。

## 🔰 初学者向け（何が・なぜ問題か）

「Avada」は WordPress で大人気の“見た目を組み立てるツール（テーマ／ビルダー）”で、世界で約100万サイトが使っています。その中の「フォーム機能」に、危険な弱点が見つかりました。

Avada のフォームには、古くなった応募データに紐づくアップロードファイルを“自動でお掃除（削除）する”機能があります。ところがこの掃除係が「どのファイルを消すか」をきちんと確認せず、外から指定された通りに消してしまいます。たとえるなら、掃除業者に「この紙を捨てて」と紙の場所を伝えると、家中どこの部屋でも言われた場所の物を確認せず捨ててしまう状態です。ログイン不要で誰でも頼めてしまうのが問題です。

攻撃者がサイトの“心臓部”である設定ファイル（wp-config.php）を消すよう仕向けると、WordPress が初期セットアップ状態に陥り、そこを突いてサイトを乗っ取られる恐れがあります。実行条件は「DBに保存する公開フォームがあること」。該当サイトは直ちに更新が必要です。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- 種別: 不適切なファイルパス検証による認証不要の任意ファイル削除（パストラバーサル／アップロードディレクトリ封じ込め欠如、CWE-22 系）。
- CVSS: 9.1（Critical、要確認）。攻撃元はネットワーク、認証不要・管理者操作不要。
- 前提条件: 対象サイトに「エントリをデータベースへ保存する設定の公開済み Avada フォーム」が存在すること（これが攻撃成立の必須条件）。
- 影響: Avada（Fusion）Builder v3.15.3 以前。脆弱箇所はプライバシークリーンアップ処理内の `maybe_delete_files()` 関数（要確認）。realpath 解決やアップロードディレクトリ封じ込めが欠如。
- 修正版: v3.15.4（2026-06-02 公開、要確認）で解消。
- 悪用状況: Wordfence 経由で 2026-05-13 報告、5/19 ベンダ修正。公開時点で広範な実環境悪用の確証は乏しい（要確認）が、CVSS 高く前提条件も満たされやすいため危険。

### メカニズム（攻撃が成立する仕組み）

注: 概念レベルにとどめ、武器化手順・パストラバーサル文字列・PoC は記載しない。

1. **削除対象パス＝信頼できない入力**: フォームの自動クリーンアップは「削除すべきアップロードファイルのパス」を扱う。この値が外部入力（攻撃者制御）に由来し得るにもかかわらず、削除前の検証が削除操作の信頼境界となる。
2. **パス封じ込めの欠落**: `maybe_delete_files()` 相当の処理が、realpath による正規化や「アップロードディレクトリ配下に限定する」チェックを行わない。このため `../` 等での上位ディレクトリ脱出（パストラバーサル）を許す。
3. **任意ファイル削除プリミティブ**: 結果として攻撃者は、Web サーバ権限で到達できる任意ファイルを削除できるプリミティブを得る。認証不要・管理者操作不要で発火する点が脅威。
4. **サイト乗っ取りへの橋渡し**: `wp-config.php` を削除すると WordPress は未設定状態になり、攻撃者が再セットアップ画面を悪用して自分の DB に接続し管理権限を確立する等、サイト乗っ取りの古典的経路へ繋がる。他の重要ファイル削除はサービス停止（DoS）にもなる。
5. **一般原理**: 「ファイル操作対象のパスを正規化・封じ込めせず外部入力で決める」設計は、任意ファイル削除／読取／書込に共通する典型クラス。削除は一見地味だが、設定ファイル破壊を介して RCE 相当の被害へ昇華しうる。

### 検知・暫定緩和・恒久対策

- 検知の着眼点: `wp-config.php` 等の重要ファイル消失・サイトのセットアップ画面化、Avada フォーム関連エンドポイントへの異常リクエスト（パストラバーサル様パターン）、Web サーバによる予期せぬファイル削除イベント。
- 暫定緩和: 即時更新が難しい場合は「DB保存設定の公開フォーム」を一時停止、WAF でパストラバーサル遮断、wp-config.php 等のファイル権限厳格化と改ざん/削除監視、バックアップの確実な取得。
- 恒久対策: Avada（Fusion）Builder を v3.15.4 以降へ更新。一般論としてファイル操作は realpath 正規化＋許可ディレクトリ封じ込めを必須化。
- 運用示唆: 「任意ファイル削除」は過小評価されがちだが設定ファイル経由で致命傷化する。プラグイン更新と整合性監視・即時復旧体制を併用する。

## 📖 用語解説（このニュースとのつながり）

- **パストラバーサル（CWE-22）**: `../` 等で想定外のディレクトリに到達する攻撃。本件の根本原因。
- **realpath 正規化 / ディレクトリ封じ込め**: パスを実体解決し許可範囲内に限定する防御。これが欠けていた。
- **任意ファイル削除**: 攻撃者が選んだファイルを消せる欠陥。設定破壊→乗っ取りに発展。
- **wp-config.php**: WordPress の中核設定（DB接続等）。削除されると再セットアップ悪用の的になる。
- **前提条件（公開フォーム）**: 本脆弱性は「DB保存の公開フォーム」があると成立。攻撃面の有無を左右する。

## 影響範囲 / 推奨対応

Avada（Fusion）Builder v3.15.3 以前を使い、DBにエントリ保存する公開フォームを持つ WordPress サイト（最大約100万）が対象。直ちに v3.15.4 以降へ更新し、更新までは該当フォームの停止・WAF 遮断・重要ファイルの監視と権限強化で露出を抑える。万一の設定ファイル消失に備えバックアップと復旧手順を確認する。実環境悪用の確証は要確認。
