# CSS ネイティブカルーセル（::scroll-button / ::scroll-marker）— JavaScript なしでスライダーを組む

> トピック: CSS Overflow Level 5 の `::scroll-button()` / `::scroll-marker` 擬似要素｜出典: Chrome for Developers / MDN｜種別: Web 標準（実装進行中）

## 一行サマリー

CSS Overflow Level 5 が導入した `::scroll-button()` と `::scroll-marker` 擬似要素により、前後ボタンとドット型インジケータを備えたカルーセル（スライダー）を Swiper や Slick などの JavaScript ライブラリなしで組めるようになった。Chrome が先行実装し、スナップ・ボタン制御・現在位置表示をブラウザ側が肩代わりする。ただし2026年7月時点では全ブラウザ対応には至らず Baseline 未達で、本番採用は対応状況の確認が必要。

## 🔰 初学者向け（何が・なぜ重要か）

「横に並んだ画像を矢印ボタンやドットで切り替えるスライダー」は、Web でよく見る部品です。これまではこの動きを作るのに Swiper や Slick といった JavaScript のライブラリを読み込むのが定番で、数百行のコードやライブラリのサイズ、更新の手間がついて回りました。

新しい CSS 機能は、この「よくある部品」をブラウザ自身が作れるようにするものです。たとえるなら、これまで毎回手作りしていた「矢印ボタン」と「今どこ？を示す丸ぽち」を、ブラウザに備え付けの標準パーツとして呼び出せるようになった、というイメージです。`::scroll-button()` が前後の送りボタン、`::scroll-marker` が位置を示すドットに対応します。

嬉しいのは、余計なライブラリを読み込まなくてよいので**ページが軽くなり、表示も速くなる**こと。そしてキーボード操作やスクリーンリーダー対応といったアクセシビリティも、ブラウザ標準の実装に乗るぶん外れにくくなります。ただし現時点では一部ブラウザ限定なので、「使える人には使わせ、そうでない人には普通のスクロールを見せる」といった配慮が要ります。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

CSS Overflow Level 5 が、スクロールコンテナに対して次の生成擬似要素を定義した。

- `::scroll-button(<direction>)`: スクロールコンテナに最大4つ（inline/block × start/end）の送りボタンを生成する。`content` プロパティでラベルを与え、押下でコンテナを1ページ分スクロールする。到達端では自動的に `:disabled` 状態になる。
- `::scroll-marker` と `::scroll-marker-group`: 各子要素に対応するマーカー（ドット）を生成し、`::scroll-marker-group` でまとめて配置する。マーカーはアンカー的に振る舞い、クリックで対応要素へスクロール、現在表示中のものは `:target-current` で装飾できる。

これらは CSS Scroll Snap（`scroll-snap-type` / `scroll-snap-align`）や scroll-driven animations と組み合わせて使う。MDN の互換性表では2026年7月時点で **experimental 扱い・Baseline 未達**（Chrome 系が先行、他エンジンは実装途上）で、`@supports selector(::scroll-marker)` での能力検出とフォールバックが前提になる。

### 仕組み / 背景（なぜこう設計されたか）

従来の JS カルーセルは、(1) スクロール位置の監視、(2) 現在インデックスの算出、(3) ボタン活性/非活性の管理、(4) マーカーの同期、(5) キーボード/フォーカス制御、を全て手続き的に実装していた。これらは本質的に「スクロールコンテナの状態」から導ける情報であり、DOM とスクロール状態を最もよく知るのはブラウザ自身、という発想が今回の設計の核にある。

擬似要素として提供することで、追加の DOM ノードを著者が書かずに済み、状態同期（現在位置→マーカー強調、端到達→ボタン無効化）をレンダリングエンジンが宣言的に担う。トレードオフは柔軟性で、複雑なトランジションや無限ループ、可変幅スライドの高度な制御は依然として JS 併用が要る場面がある。また現段階では**クロスブラウザの穴**が最大の代償で、Baseline 未達のうちはプログレッシブ・エンハンスメント（未対応環境では素のスクロールスナップにフォールバック）が現実解となる。

### 実務での使いどころ・移行の勘所

画像ギャラリー、商品スライダー、カード横スクロールなど「スナップする横並び＋前後ボタン＋ドット」の定番 UI に最適。まずスクロールスナップで土台を作り、その上に `::scroll-button` / `::scroll-marker` を重ねる二層構成にすると、未対応環境でも基本操作（スワイプ/スクロール）が成立する。既存の Swiper 依存を段階的に外す場合、装飾要件が軽いコンポーネントから置換すると効果と安全性のバランスが良い。`@supports` による分岐と、`prefers-reduced-motion` への配慮を忘れないこと。

## 📖 用語解説（このトピックとのつながり）

- **::scroll-button()**: スクロールコンテナに生成される送りボタン擬似要素。本トピックの前後ナビを担う。
- **::scroll-marker / ::scroll-marker-group**: 位置インジケータ（ドット）を生成・グループ化する擬似要素。現在位置表示に使う。
- **Scroll Snap**: スクロールを特定位置に吸着させる CSS 機能。カルーセルの土台。
- **CSS Overflow Level 5**: これらの擬似要素を定義する仕様。
- **プログレッシブ・エンハンスメント**: 未対応環境でも基本機能が成立するよう段階的に強化する設計方針。Baseline 未達の今の必須作法。

## 影響範囲 / 推奨アクション

カルーセル系ライブラリへの依存を減らせる可能性が高く、バンドルサイズとアクセシビリティの改善が見込める。ただし Baseline 未達のため、当面は `@supports` 検出＋スクロールスナップのフォールバック前提で導入する。まずは非クリティカルな装飾スライダーから試験採用を推奨。
