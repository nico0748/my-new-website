# Oxlint / oxfmt — Rust 製リンタ・フォーマッタ（Oxc）の採用増

> トピック: 開発トピック｜出典: thetshaped.dev｜種別: frontend

## 一行サマリー
Oxc プロジェクトの Rust 製リンタ Oxlint とフォーマッタ oxfmt は、ESLint / Prettier を桁違いの速度で置き換える選択肢として2026年に採用が広がっている。

## 🔰 初学者向け（何が・なぜ重要か）
コードを書くとき、裏で「文法や書き方のミスを指摘する係（リンタ）」と「見た目を整える係（フォーマッタ）」が働いています。従来この役目は JavaScript 製の ESLint と Prettier が担ってきましたが、プロジェクトが大きくなると、これらのチェックに数十秒〜数分かかることも珍しくありませんでした。

Oxc は、これらを JavaScript ではなく Rust という高速な言語で作り直したツール群です。同じ「校正作業」を、より高性能なエンジンでこなすイメージです。結果として、保存のたびに走るチェックが体感で一瞬になり、開発のテンポが大きく変わります。

「待ち時間の削減」は地味ですが、1日に何百回も走るツールだからこそ、積み重なると生産性に大きく効きます。CI（自動テスト環境）の実行時間短縮にも直結します。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Oxlint は ESLint 比で大幅に高速（数十倍規模の高速化が謳われるが具体倍率は環境依存・要確認）で、設定不要ですぐ動くのが特徴。oxfmt は Prettier 互換に近いフォーマットを目指す。Oxc は同一の Rust 製パーサ基盤を共有し、Vite 8 系の Rolldown / Oxc 統合ツールチェーンとも連続性がある。プラグイン互換性やカスタムルール対応度は要確認。

### 仕組み / 背景（なぜこう設計されたか）
JavaScript 製ツールは AST 解析やファイル走査でボトルネックになりやすい。Rust のネイティブ実行と並列処理、単一のパーサ共有によって速度を稼ぐ設計。「ツールチェーン全体を1つの高速基盤で統一する」という潮流の一部。

### 実務での使いどころ・移行の勘所
まず Oxlint を ESLint と併用し、対応ルールを Oxlint に寄せて段階移行するのが安全。ESLint の豊富なエコシステム（型認識ルールやフレームワーク固有プラグイン）に依存している場合は完全置換が難しく、共存構成が現実的。既存の設定資産をどこまで引き継げるかを事前検証すること。

## 📖 用語解説（このトピックとのつながり）
- **Oxc**: Rust 製の JavaScript ツールチェーン基盤。Oxlint / oxfmt の共通の土台。
- **リンタ**: コードの問題を静的に検出するツール。Oxlint がこれにあたる。
- **フォーマッタ**: コードの見た目を統一するツール。oxfmt がこれにあたる。
- **AST（抽象構文木）**: コードを木構造で表した内部表現。解析速度がツールの性能を左右する。

## 影響範囲 / 推奨アクション
大規模リポジトリや CI 時間に悩むチームに特に有効。まず Oxlint を試験導入し速度と検出精度を確認、問題なければ ESLint ルールを段階移行するのがよい。
