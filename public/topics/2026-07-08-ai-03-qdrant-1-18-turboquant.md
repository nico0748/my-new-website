# Qdrant 1.18 が TurboQuant を実装：同じ recall でメモリ半分、io_uring でマルチベクトルも高速化

> トピック: Qdrant 1.18 / TurboQuant｜出典: Qdrant 公式ブログ / GitHub Releases｜種別: ベクトルDB更新

## 一行サマリー

ベクトルデータベース Qdrant の 1.18 系（v1.18.0 5月11日 / v1.18.1 5月22日 GA）が、Google Research 発の量子化手法 TurboQuant を実装。スカラー量子化と同程度の recall を保ったままメモリを半減し、さらに Linux 上のマルチベクトル量子化スコアリングを io_uring で最適化した。

## 🔰 初学者向け（何が・なぜ重要か）

RAG（検索拡張生成）では、文書を「ベクトル（数百〜数千次元の数値の並び）」に変換して保存し、質問文のベクトルと近いものを探す。この保存・検索を担うのが Qdrant のようなベクトルデータベースだ。

問題は、ベクトルが巨大なメモリを食うこと。1000万件の文書を 1024 次元の float32 で持つと、それだけで数十 GB になる。そこで「量子化（quantization）」という圧縮技術を使う。各次元の数値を粗く丸めて（例：32ビット→8ビットや1ビット）、メモリを削る。ただし丸めすぎると検索精度（recall＝本来ヒットすべきものを取りこぼさない割合）が落ちる。

今回 Qdrant が取り入れた **TurboQuant** は、この「メモリ削減と精度のトレードオフ」を従来より良い地点に押し上げる新手法。ざっくり言えば「同じ精度なら半分のメモリで済む」。ベクトル検索のコストの大半はメモリ（RAM）なので、これは運用費に直結する嬉しい話だ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **TurboQuant 実装**: Google Research 発の回転ベース量子化アルゴリズム（ICLR 2026）。Qdrant はこれを実際の埋め込みで機能するよう拡張し、RaBitQ のアイデアも取り入れた。スカラー量子化(SQ)と同程度の recall を、2倍少ないメモリで実現。2-bit / 1-bit は BQ（バイナリ量子化）と同じストレージ予算で一貫して高い recall を出す。
- **dynamic named vectors**: 名前付きベクトルを動的に扱える機能を追加。
- **io_uring マルチベクトル最適化**: v1.18.1（5月22日）で、Linux 上のマルチベクトル量子化スコアリングを io_uring で最適化。
- **リリース**: v1.18.0（2026年5月11日）＋ v1.18.1（5月22日）GA。

### 仕組み / 背景（なぜこう設計されたか）

TurboQuant の核心は「量子化前に高速アダマール回転（fast Hadamard transform）でベクトルを回す」ことにある。通常の埋め込みは次元ごとに値の分布が偏っている（ある次元だけ大きな値が集中する等）。この偏りがあると、単純なスカラー量子化は「値が集中する次元」に量子化誤差が偏り、精度を損なう。

アダマール回転をかけると、値が全座標にほぼ均等に再分配される。数学的には、回転後の座標が集中した Beta 分布に従い、高次元では各座標がほぼ独立になる。この性質のおかげで「各座標に最適なスカラー量子化を独立に適用する」だけで良い結果が得られる。しかもこの回転はデータに依存しない（data-oblivious）ため、どんな埋め込みモデルにも学習なしで効く。回転は O(d log d) で計算できるので前処理コストも軽い。

io_uring 最適化は別軸の話で、Linux カーネルの非同期 I/O インタフェースを使い、量子化済みマルチベクトルのスコアリング時のディスク/メモリ I/O を効率化するもの。ColBERT 系のマルチベクトル検索（1文書に複数ベクトルを持たせる）で効いてくる。

### 実務での使いどころ・移行の勘所

大規模 RAG でメモリコストが支配的なら TurboQuant への切り替えは有力。特に「1-bit / 2-bit まで攻めたいが recall を落としたくない」ケースで、従来の SQ/BQ より良いトレードオフが得られる。埋め込みモデル非依存なので、既存のインデックスの再構築だけで導入できるのも移行負担が軽い。マルチベクトル（ColBERT / late interaction）を Linux で運用しているなら io_uring 最適化も効く。他 DB との比較では、Pinecone は運用の手軽さ、Weaviate はネイティブ MCP サーバ、Qdrant は量子化性能とセルフホスト性能に強みを持つ（同日の別記事「ベクトルDB 2026 勢力図」参照）。

## 📖 用語解説（このトピックとのつながり）

- **量子化 (quantization)**: ベクトルの各次元を粗い数値に丸めてメモリを削減する技術。recall とのトレードオフがある。
- **TurboQuant**: アダマール回転＋独立スカラー量子化による回転ベース量子化。Google Research 発、ICLR 2026。
- **recall**: 本来ヒットすべき近傍を取りこぼさない割合。量子化で犠牲になりやすい指標。
- **アダマール変換 (Hadamard transform)**: 値を全座標に均等再分配する高速な直交変換。TurboQuant の前処理の要。
- **io_uring**: Linux の高効率な非同期 I/O インタフェース。マルチベクトルスコアリングの高速化に利用。
- **マルチベクトル (multi-vector)**: 1文書を複数ベクトルで表す方式（ColBERT 等）。細粒度な検索が可能。

## 影響範囲 / 推奨アクション

- 大規模 RAG でメモリコストが課題なら TurboQuant の 1-bit/2-bit を評価。SQ/BQ との recall 比較を実測する。
- 埋め込みモデル非依存なので、既存インデックスの再構築だけで試せる。まず一部コレクションで検証。
- Linux でマルチベクトル（ColBERT 等）を運用中なら io_uring 最適化の恩恵を確認。
- ベクトル DB 選定中なら Pinecone/Weaviate との比較軸（運用・MCP・量子化）を整理。
