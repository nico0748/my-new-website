# Axum が Rust Web フレームワークの本命に — Tokio 製・Tower ミドルウェアで高性能 API

> トピック: Axum（Rust Web フレームワーク）｜出典: Axum 公式リポジトリ｜種別: フレームワーク動向

## 一行サマリー

Axum は Rust の主要非同期ランタイム Tokio チームが開発する Web フレームワークで、2026年1月に 0.8.8 をリリース、Tower ミドルウェア資産とエルゴノミックな API を両立し、2026 年のベンチで単一スレッド約1.7〜1.8万 req/s を出すなど Rust バックエンドの事実上の第一選択となりつつある。

## 🔰 初学者向け（何が・なぜ重要か）

**Rust（ラスト）** は「C 言語なみに速いのに、メモリ関連のバグが起きにくい」ことで人気の言語です。サーバー用途でも「速くて落ちにくい」バックエンドを作れるため、近年採用が増えています。

その Rust で Web サーバー（API）を作るときの土台が「Web フレームワーク」です。いくつか候補がありますが、いま最も勢いがあるのが **Axum（アクサム）**。Rust の非同期処理の心臓部である「Tokio」を作ったチーム自身が開発しているので、相性と信頼性が高いのが特徴です。

Axum の良さは2つ。ひとつは書き味の良さ（型を活かして、素直に・安全にルーティングやハンドラを書ける）。もうひとつは「Tower」という共通部品群の資産を使えること。ログ・認証・タイムアウト・レート制限といった「どの API にも要る機能」を、既存の部品を組み合わせて足せます。結果として「速い・安全・作りやすい」が揃い、Rust でサーバーを書くなら Axum、という空気になっています。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **Axum 0.8.8 が 2026年1月にリリース**。Tokio チームが開発し、Rust で最も人気の Web フレームワークとされる。
- Tower/Tower-HTTP ミドルウェアエコシステムをフル活用でき、エルゴノミックな API 設計が特徴。
- 2026 年ベンチで単一スレッドマシン約 **17,000〜18,000 req/s**。io_uring 採用の ntex にはわずかに劣るが Actix を上回る、との比較結果。
- 比較対象: Actix Web 4.12.1（2025年11月）、Rocket 0.5.1（2024年5月）など。用途・ベンチ条件で優劣は変わるため要確認。

### 仕組み / 背景（なぜこう設計されたか）

Axum の設計の芯は「Tower の `Service` トレイトへの全面依存」だ。Tower は「リクエストを受けてレスポンスを返す非同期関数」を `Service` という共通抽象で表す。Axum のルーター・ハンドラ・ミドルウェアはすべてこの `Service` として合成できるため、ロギング・認証・圧縮・タイムアウト・同時実行制限といった横断的関心事を、フレームワーク独自機構ではなく汎用ミドルウェアとして付け外しできる。これが「他の Tower ベース資産（tonic gRPC 等）と部品を共有できる」強みを生む。

もう一つの核が「エクストラクタ（extractor）」パターンだ。ハンドラの引数の型（`Json<T>`, `Path<T>`, `State<S>` 等）を見て、Axum が自動でリクエストから必要な値を取り出し型付きで渡す。これは Rust の型システムとトレイト解決を使った宣言的な依存注入で、ボイラープレートを減らしつつコンパイル時に不整合を弾く。Tokio 上で動くため I/O は非同期・ノンブロッキングで、少ないスレッドで大量接続を捌ける（イベント駆動＋タスクスケジューリング）。トレードオフは、Rust 特有の学習コスト（所有権・ライフタイム・非同期の理解）と、エクストラクタやミドルウェアの型が絡んだ際のコンパイルエラーの難解さ。生の最速を狙うなら io_uring ベースの実装に一歩譲る場面もあるが、エコシステムの広さと保守性で Axum が総合力を持つ。

### 実務での使いどころ・移行の勘所

高スループット・低レイテンシが要る API、レイテンシ予算が厳しいマイクロサービス、既存 Tokio/Tower スタック（tonic gRPC 等）と部品を共有したいシステムに向く。Node/Go からの移行では、Rust の学習コストとビルド時間がチームの初速に効くため、まず1サービスを Axum で書いて運用知見（ビルド/デプロイ、可観測性、エラーハンドリング）を貯めるのが現実的。フレームワーク選定では「生ベンチの最速」より「ミドルウェア資産・ドキュメント・保守体制」を重視すると Axum が有力。逆に、既に Actix で回っていて不満がないなら、無理な乗り換えは不要。

## 📖 用語解説（このトピックとのつながり）

- **Tokio**: Rust の主要非同期ランタイム。Axum の実行基盤で、開発チームも同じ。
- **Tower / Service トレイト**: 非同期処理を共通抽象化するミドルウェア基盤。Axum の合成性の源。
- **エクストラクタ（extractor）**: 引数の型からリクエスト値を型付きで取り出す仕組み。宣言的な依存注入。
- **非同期ノンブロッキング I/O**: 少数スレッドで多数接続を捌く方式。高スループットの土台。
- **io_uring**: Linux の高速非同期 I/O 機構。ntex 等が採用し生ベンチで優位に立つ要因。

## 影響範囲 / 推奨アクション

- 影響: Rust でバックエンド/API を構築する（または検討する）チーム。
- 推奨: 新規 Rust サービスは Axum を第一候補に。移行は1サービスから運用知見を蓄積。選定はベンチ最速より資産・保守性で判断。
