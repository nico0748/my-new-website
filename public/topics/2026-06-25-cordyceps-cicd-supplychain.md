# 「Cordyceps」CI/CDワークフローの体系的欠陥 — 匿名PRでMicrosoft/Google/Apache等300超リポジトリがサプライチェーン侵害可能に

> トピック: 脆弱性ニュース ｜ 出典: Novee Security / The Hacker News / Dark Reading / SecurityWeek ｜ 種別: CI/CD設定不備（過剰権限のワークフロー）に起因するサプライチェーン攻撃（コマンドインジェクション／認証ロジック欠陥／成果物汚染／権限昇格の複合）

## 一行サマリー

Novee SecurityがGitHub ActionsのCI/CDワークフローに体系的な悪用可能パターン「Cordyceps」を公表した。約30,000の重要リポジトリを調査し、**300超が完全悪用可能**——匿名・無権限のユーザーが悪意あるプルリクエスト（PR）でビルドを乗っ取り、トークンを窃取し、サプライチェーンを汚染できる。Microsoft・Google・Apache・Cloudflare・Python Software Foundationなど大手で修正が確認された。CVE番号は付かないが、根本は「ワークフローをコードでなく設定として扱う」運用文化にある。広範な実環境悪用の証拠は現時点でない。

## 🔰 初学者向け（何が・なぜ問題か）

現代のソフト開発では、コードを保存すると**自動で検査・ビルド・配布する仕組み（CI/CD）**が動きます。GitHub Actionsはその代表で、「ワークフロー」と呼ばれる手順書に従って、サーバーが自動でコマンドを実行します。便利な反面、この自動実行サーバーは**プロジェクトの秘密の鍵（トークン）**を握り、公式の成果物（配布物）を作る権限を持っています。

「Cordyceps（コルディセプス＝宿主を乗っ取る寄生菌の名）」は、この自動実行の仕組みに潜む**設計の隙**を体系化したものです。問題の核心は、**誰でも送れる「プルリクエスト（修正提案）」に、本来与えるべきでない強い権限が与えられている**ことにあります。

たとえるなら、工場の見学者（＝外部の匿名ユーザー）が「改善提案カード」を投函しただけで、その提案に紛れた指示を工場のロボットが実行し、**金庫の鍵を持ち出したり、出荷する製品に細工したりできてしまう**ようなものです。攻撃者は無料アカウント一つあれば、組織のメンバーである必要すらありません。実際、MicrosoftのAzure Sentinelでは、PRへのコメントだけで匿名の攻撃コードがMicrosoftのCI上で動き、失効しない鍵を盗める例が見つかりました。月1.3億回インストールされるPythonの整形ツール「Black」でも、どのPRからでもボットのトークンを盗め、公式Dockerイメージの汚染に至る経路が確認されています。1つのプロジェクトが汚染されると、それを使う**無数の下流利用者まで巻き込む**のがサプライチェーン攻撃の怖さです。

## 🧠 専門的解説（技術詳細・メカニズム・検知対応）

### 技術的詳細（種別・前提条件）

- **分類**: 単一CVEではなく、**CI/CDワークフローの悪用可能パターンの体系（クラス）**。コマンドインジェクション、認証ロジックの破綻、成果物汚染チェーン、GitHub Actionsワークフローの権限昇格を含む。
- **規模**: 約30,000の高インパクトリポジトリを走査、数百の完全悪用可能チェーンを検証。単一スキャンで**654リポジトリを検出、300超が完全悪用可能**と確認。
- **前提条件**: 攻撃者は**匿名・無権限**でよい。組織メンバーシップや特権は不要、無料アカウントで足りる。承認の偽造・コード push・資格情報窃取が可能。
- **影響確認組織**: Microsoft、Google、Apache、Cloudflare、Python Software Foundation等で修正確認。
- **根本原因**: ワークフローを「コード」でなく「設定」として扱い、アプリケーションコードと同等のセキュリティレビューを欠く文化。PRに過剰な権限を付与する弱いCI/CD構成。
- **悪用状況**: Noveeは**現時点で広範な実環境悪用の証拠はない**としている。

### メカニズム（攻撃が成立する仕組み）

以下は概念レベルの説明であり、武器化可能な手順・PoC等は含めない。

1. **信頼境界の所在**: CI/CDでは「外部から提案されるコード（PR）」と「内部の信頼されたビルド環境・秘密情報」の間が信頼境界。PRの内容は本来"未検証の外部入力"として扱うべきもの。
2. **過剰権限トリガの誤用**: 一部のワークフローは、PRに対して**書き込み権限や秘密情報へのアクセスを伴う文脈**で実行される（例: フォークからのPRやPRコメントを契機に特権ジョブが走る構成）。ここで外部入力が特権コンテキストに持ち込まれる。
3. **入力のコード化（インジェクション）**: ワークフローはシェルコマンドやスクリプトを実行する。PR本文・コメント・ブランチ名・成果物などの外部由来データが、**式展開や文字列連結を通じてコマンドとして解釈**され、攻撃者制御のコード実行が成立する。
4. **トークン窃取と認証ロジック回避**: 実行コンテキストが保持する**ビルド用トークン／アプリ鍵**（時に失効しない長寿命鍵）を奪取。さらにPR承認の判定ロジックの不備を突き、**承認偽造**で本来不可能なマージ・公開を成立させる。
5. **成果物汚染チェーンと下流伝播**: 奪った権限で公式リリースやコンテナイメージを**汚染**し、それを利用する全下流（依存パッケージ・Docker利用者）へ悪性コードを伝播させる。これがサプライチェーン攻撃の最終段で、被害が指数的に拡大する。

脆弱性クラスとしては、「ワークフローを設定とみなしレビュー対象外とする運用」に起因する**過剰権限＋外部入力のコード混入**であり、`pull_request_target`相当の特権トリガとシークレット露出が組み合わさる近年頻発のCI/CD固有パターンに該当する。

### 検知・暫定緩和・恒久対策

- **検知の着眼点**: フォークPR／PRコメントを契機に**シークレットへアクセスする特権ジョブ**の有無、ワークフローログでの外部由来文字列のコマンド展開、CIからの不審なアウトバウンド（トークン送信）、リリース／イメージ公開の異常イベント。
- **暫定緩和**: 信頼境界を跨ぐトリガ（フォークPRで特権実行する構成）を見直し、**外部PRには最小権限**を付与。シークレットを外部PR文脈で露出させない。式展開へのユーザー入力直挿しを排し、環境変数経由＋入力検証に変更。承認・マージ保護ルールを強化。
- **恒久対策**: **ワークフローを"コード"として扱い**、アプリと同等のレビュー・権限最小化・依存ピン留め（アクションのSHA固定）を徹底。長寿命トークンを廃し短命・スコープ限定の資格情報へ。OIDC連携等で鍵の常駐を排除。
- **運用示唆**: サプライチェーンの起点はCI/CD設定。DevSecOpsとして権限最小化と外部入力の不信を恒常原則に。

## 📖 用語解説（このニュースとのつながり）

- **CI/CD / GitHub Actions**: コードを自動検査・ビルド・配布する仕組み。意味＝「自動の製造ライン」。本件＝ここが乗っ取られ下流まで汚染。
- **プルリクエスト（PR）**: 外部からのコード修正提案。本件＝匿名PRが攻撃の入口。
- **特権トリガ（pull_request_target相当）**: 外部PRを特権文脈で動かす設定。本件＝外部入力が秘密情報に触れる原因。
- **トークン窃取**: ビルド用の鍵を盗む行為。本件＝失効しない鍵の奪取が深刻。
- **サプライチェーン攻撃**: 上流の汚染が下流利用者へ波及する攻撃。本件＝Cordycepsの最終被害。

## 影響範囲 / 推奨対応

影響を受けるのは、GitHub Actions等のCI/CDで**外部PRを特権文脈で実行する構成**を持つ全プロジェクト（OSS・商用問わず）。とりわけ広く使われるライブラリ／イメージは下流被害が甚大。推奨対応は、(1) 外部PRを特権実行するワークフローの棚卸しと権限最小化、(2) シークレットの外部PR文脈での露出排除と式展開へのユーザー入力直挿し禁止、(3) アクションのSHA固定・長寿命トークン廃止・OIDC化、(4) リリース／イメージ公開イベントとCIアウトバウンドの監視。
