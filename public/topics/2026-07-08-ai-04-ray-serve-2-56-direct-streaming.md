# Ray Serve 2.56 が Direct Streaming Mode を導入：LLM サービングのレイテンシを最大8分の1に

> トピック: Ray Serve 2.56 / LLM Serving｜出典: Anyscale 公式ブログ / Ray Docs｜種別: 推論インフラ更新

## 一行サマリー

分散モデルサービングフレームワーク Ray Serve の 2.56 が「Direct Streaming Mode」を導入し、ルーティング制御プレーンとトークンストリーミングのデータプレーンを分離。HAProxy がターゲットレプリカと直接 HTTP 接続してトークンを返すことで、最大 5倍のスループット・8倍のレイテンシ改善を実現した。

## 🔰 初学者向け（何が・なぜ重要か）

LLM を本番で動かすには「モデルサービング」の仕組みがいる。1台の GPU では捌ききれない大量のリクエストを、複数の GPU レプリカに振り分け（ロードバランス）、生成されたトークンをユーザーに逐次返す（ストリーミング）役割だ。Ray Serve は、この分散サービングを担うフレームワークの代表格で、vLLM などの高速推論エンジンと組み合わせて使う。

これまでの Ray Serve には「中継地点がボトルネックになる」問題があった。リクエストを正しいレプリカに振り分ける「ルーター」が、生成トークンの往復も全部中継していたのだ。宅配で言えば、荷物（トークン）が必ず一度中央仕分けセンターを経由してから届く状態で、そこが混むと全体が遅くなる。

2.56 の Direct Streaming Mode は「振り分けの指示」と「実際の荷物の流れ」を分離した。ルーターは「どのレプリカに送るか」だけを決め、実際のトークンの流れはロードバランサ（HAProxy）とレプリカが直接やり取りする。中央仕分けセンターを経由せず産地直送になったので、速くなった、という話だ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **Direct Streaming Mode**: Ray Serve LLM のリクエストルーティング制御プレーンと、リクエスト/レスポンスのストリーミングデータプレーンを分離。HAProxy が選択したターゲットレプリカと直接 HTTP 接続を確立し、トークンを直接クライアントへストリームする。従来は中間のルーティングデプロイがレスポンストークンの転送を担っていたボトルネックを解消した。
- **性能**: Ray Serve 2.56 は最大 5倍のスループットと 8倍のレイテンシ改善を実現（Anyscale 計測・ワークロード依存）。
- **HAProxy 統合**: HAProxy は pip でインストールでき、2.56 ではフラグで有効化。Ray 2.57 では自動的に組み込まれる予定。
- **推奨イメージ**: `rayproject/ray-llm:2.56-py312-cu130` など。
- **ワークロード横断の一貫性**: prefill 偏重・decode 偏重・エージェント型マルチターンのいずれでも、Ray の耐障害性・可観測性・異種ハードウェア対応を保ちつつ、vllm-router と同等の集計スループットに到達。

### 仕組み / 背景（なぜこう設計されたか）

LLM サービングの難所は「1リクエストが長時間ストリーミングを続ける」点にある。通常の Web リクエストは短命だが、LLM はトークンを1個ずつ長ければ数十秒〜数分にわたり返す。この間、もし全トークンが中間ルーターを経由すると、ルーターは同時に大量のストリームを保持し続けねばならず、CPU とネットワークの両面でボトルネックになる。

そこで「制御プレーン（どこに送るか＝短命な決定）」と「データプレーン（トークンの流れ＝長命なストリーム）」を分ける、という古典的なネットワーク設計思想を持ち込んだ。ルーターはルーティング判断だけを高速に行い、実データは HAProxy とレプリカの直結パスに逃がす。これにより中間ホップが1つ減り、レイテンシとスループットの両方が改善する。分散システムで「コントロールパスとデータパスを分離する」のは定石で、それを LLM ストリーミングに適用した形だ。

vLLM などの推論エンジンは「1台の GPU 上で KV キャッシュを効率管理し高スループットを出す」層を担い、Ray Serve は「複数レプリカへの振り分けと耐障害性」を担う。両者は役割が分かれており、今回の改善は後者のオーケストレーション層のオーバーヘッド削減にあたる。

### 実務での使いどころ・移行の勘所

Ray Serve で vLLM を回している既存ユーザーは、2.56 へ上げて HAProxy フラグを有効化するだけでレイテンシ改善が見込める（実効果はワークロード依存なので必ず計測）。2.57 では自動組み込みになるため、いずれ標準になる機能だ。KServe との比較では、KServe は Kubernetes ネイティブで CNCF Incubating、traffic 管理・可観測性が強い一方、Ray Serve は Ray エコシステム統合とステートフルなマルチアクター処理に強い。両者は排他ではなく、KServe の InferenceGraph から外部 Ray Serve エンドポイントへルーティングする併用構成も可能。

## 📖 用語解説（このトピックとのつながり）

- **Ray Serve**: Ray 上の分散モデルサービングフレームワーク。複数レプリカへの振り分けと耐障害性を担う。
- **制御プレーン / データプレーン**: 「どこへ送るか」の判断と「実データの流れ」を分ける設計。Direct Streaming の核。
- **HAProxy**: 高性能なロードバランサ。2.56 でレプリカと直結しトークンを直接ストリームする。
- **prefill / decode**: LLM 推論の2フェーズ。入力処理(prefill)と1トークンずつの生成(decode)で負荷特性が異なる。
- **vLLM**: 高スループットな LLM 推論エンジン。KV キャッシュ管理に優れ、Ray Serve と組み合わせて使う。
- **KServe**: Kubernetes ネイティブのモデルサービング。CNCF Incubating。Ray Serve と併用可能。

## 影響範囲 / 推奨アクション

- Ray Serve + vLLM 運用中なら 2.56 へ更新し HAProxy を有効化、自ワークロードでレイテンシ/スループットを実測。
- 2.57 で HAProxy が自動組み込みになるため、早めに検証しておくと移行が滑らか。
- 新規に LLM サービング基盤を選ぶなら KServe（K8s ネイティブ）との比較軸を整理。併用構成も選択肢。
- prefill/decode の負荷特性が偏るワークロードでは効果が出やすいので優先的に検証。
