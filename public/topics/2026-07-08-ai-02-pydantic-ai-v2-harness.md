# Pydantic AI V2 が安定版に：Capabilities と Harness で「小さいコア＋速い運用層」へ

> トピック: Pydantic AI V2｜出典: Pydantic 公式ブログ / GitHub Releases｜種別: フレームワーク更新

## 一行サマリー

Pydantic AI が7回のベータを経て 2026年6月23日に v2.0.0 安定版へ到達。「Capabilities」という単一の合成可能な単位でツール・フック・指示・モデル設定を束ね、コアを小さく保ちつつ運用機能は別パッケージ「Harness」に切り出す設計へ再構成された。

## 🔰 初学者向け（何が・なぜ重要か）

Pydantic AI は、Python の定番バリデーションライブラリ Pydantic を作ったチームによる AI エージェントフレームワーク。型付きで堅牢にエージェントを書ける点が売りで、FastAPI 的な「型で守る」思想を LLM の世界に持ち込んだものだ。

V2 で一番変わったのは「設計思想」だ。エージェントには「ツール（道具）」「フック（処理の割り込み点）」「指示（システムプロンプト）」「モデル設定」など複数の関心事がバラバラに存在していた。V2 はこれらを **Capability（ケイパビリティ）** という1つの箱にまとめられるようにした。「メモリ機能」「ガードレール機能」といった塊を Capability として付け外しできるので、レゴブロックのように機能を組み合わせられる。

もう1つの変化は、機能の置き場所を2つに分けたこと。頻繁には変わらない基幹部分（ループ処理・プロバイダ接続）は「コア」に小さくまとめ、どんどん進化させたい実用機能（メモリ・ファイルアクセス・コードモード等）は「Harness」という別パッケージに逃がした。基礎工事は堅く固め、内装は自由に模様替えできる家、というイメージだ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）

### 技術的詳細（何が変わったか）

- **リリース**: 7本のベータを経て 2026年6月23日に v2.0.0 安定版。以降 v2.5.1 系まで更新が続いている（要確認）。
- **Capabilities をコア・プリミティブに昇格**: エージェントのツール・フック・指示・モデル設定を束ねる単一の合成可能な単位。エージェントのあらゆる層に1つの概念で到達できる。
- **Harness ファースト設計**: 一部の Capability は Pydantic AI 本体に同梱するが、より多くはファーストパーティの「Pydantic AI Harness」から供給される。メモリ、ガードレール、コンテキスト管理、ファイルシステムアクセス、コードモードなどが Harness 側にある。
- **コアの縮小**: コアはループ、プロバイダ、Capability/フック API、そしてすべてのエージェントに不可欠なごく一部の Capability だけを持つ。

### 仕組み / 背景（なぜこう設計されたか）

この「小さいコア＋速い運用層」というアーキテクチャは、フレームワークの安定性と進化速度のトレードオフを解く狙いがある。エージェント技術は 2026年時点でも激しく変化しており、メモリやガードレールの「正解」はまだ固まっていない。もしこれらをコアに直接埋め込むと、実験的機能の変更が破壊的変更としてコア全体のバージョンに波及してしまう。

そこで「Harness」に運用機能を追い出し、そこでは高速にイテレーションしつつ、広く必須と証明された Capability だけをコアに「昇格」させる仕組みにした。安定を求めるユーザーはコアだけを使い、最新機能を追いたいユーザーは Harness を取り込む、という選択が可能になる。これは Microsoft Agent Framework の Agent Harness とほぼ同じ発想であり、2026年のエージェントフレームワーク設計の収束点だと言える。

Capability を「合成可能な単一単位」にした背景には、従来フレームワークで「ツールの登録」「フックの登録」「プロンプトの設定」が別々の API に散らばり、再利用が難しかった問題がある。1つの Capability にまとめれば、たとえば「社内ナレッジ検索」機能一式を1オブジェクトとして配布・再利用できる。

### 実務での使いどころ・移行の勘所

型安全なエージェントを Python で書きたいチームには依然として第一候補。特に既存の Pydantic / FastAPI 資産があるプロジェクトと相性が良い。V1 からの移行は Capability への概念統合が伴うため、公式のアップグレードガイド（changelog）を必ず参照すること。運用機能を Harness に頼る設計なので、コアだけを固定バージョンで使いつつ Harness を追従させる、という運用が可能。LangGraph（グラフでの明示的制御）や Microsoft Agent Framework（.NET 併用・Azure 統合）との比較では、「Python 単一・型安全・軽量」が Pydantic AI の差別化点。

## 📖 用語解説（このトピックとのつながり）

- **Capability**: ツール・フック・指示・モデル設定を束ねた合成可能な単位。V2 のコア・プリミティブ。
- **Harness**: 運用機能（メモリ・ガードレール・ファイルアクセス等）を収めた別パッケージ。速く進化させる領域。
- **フック (hooks)**: エージェント実行の特定タイミングに処理を差し込む機構。Capability の構成要素。
- **コードモード (code mode)**: エージェントにコードを書かせて実行させる動作。CodeAct 系の発想に近い機能で Harness 側に存在。
- **昇格 (graduation)**: 広く必須と証明された Capability を Harness からコアへ移す運用ポリシー。

## 影響範囲 / 推奨アクション

- V1 利用チームは changelog のアップグレードガイドを確認し、Capability への移行を計画する。
- コアの安定性を重視するなら、コアはバージョン固定し Harness だけ追従させる運用を検討。
- 型安全性を重視する Python 中心チームは新規採用の第一候補として評価。
- メモリ・ガードレール等の実用機能は Harness にあるため、依存に Harness を含めるか要判断。
