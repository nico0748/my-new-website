# Serverless 2.0 vs Kubernetes — WebAssembly が問う実行基盤論争

> トピック: 開発トピック｜出典: tentoftech｜種別: backend

## 一行サマリー
WebAssembly（Wasm）を実行単位とする「Serverless 2.0」が、コンテナ+Kubernetes 中心の実行基盤に対して、高速起動・軽量・高密度という利点で挑戦を仕掛けているという議論。

## 🔰 初学者向け（何が・なぜ重要か）
サーバーレスは「サーバーの管理を意識せず、関数を置くだけで動く」仕組みだ。ただ従来のサーバーレス（AWS Lambda など）は内部でコンテナを起動しており、初回起動が遅い「コールドスタート」問題を抱えていた。冷えた車のエンジンをかけるのに時間がかかるのに似ている。

Wasm は、もともとブラウザ向けに作られた「軽量で安全な実行形式」だ。コンテナより桁違いに速く起動でき（ミリ秒未満も可能）、メモリも小さい。これを実行単位にすれば、コールドスタート問題がほぼ消え、1台のサーバーに大量の関数を詰め込める。エンジンをかけっぱなしにしなくても、瞬時に始動できる電動モーターのようなイメージだ。

「Serverless 2.0」はこの Wasm ベースの新世代サーバーレスを指す。Kubernetes を完全に置き換えるかは議論があるが、実行基盤の選択肢が広がった点が重要だ。

## 🧠 専門的解説（技術詳細・設計・使いどころ）
### 技術的詳細（何が変わったか）
Wasm ランタイム（Wasmtime / WasmEdge 等）と WASI（WebAssembly System Interface）の成熟により、サーバーサイドで Wasm モジュールを安全にサンドボックス実行できるようになった。コールドスタートはコンテナのミリ秒〜秒に対しマイクロ秒〜ミリ秒級。Spin や Fermyon、Cloudflare Workers などが Wasm ベースの実行基盤を提供。具体的な性能値・採用状況は要確認。

### 仕組み / 背景（なぜこう設計されたか）
コンテナは OS プロセス+名前空間で隔離するため起動が重く、イメージも大きい。Wasm は言語非依存の中間表現をランタイムが JIT/AOT で実行し、能力ベースのサンドボックス（デフォルト無権限、明示的に付与）で安全性を確保する。この「軽さ×安全×移植性」が高密度・高速起動を可能にする。ただしシステムコールや既存ライブラリの制約、成熟度の差が残る。

### 実務での使いどころ・移行の勘所
イベント駆動・短命・多数の小さな関数（エッジ処理、プラグイン実行、マルチテナント）で Wasm の利点が際立つ。逆に長寿命・重い依存・既存コンテナ資産が多いワークロードは Kubernetes が依然有利。現実解は「Kubernetes 上で Wasm ワークロードも動かす（SpinKube 等）」ハイブリッド。全面移行より適材適所が2026年の落としどころ。

## 📖 用語解説（このトピックとのつながり）
- **Wasm（WebAssembly）**: 軽量・移植可能な実行形式。Serverless 2.0 の実行単位。
- **WASI**: Wasm にシステム機能を与える標準。サーバーサイド実行の前提。
- **コールドスタート**: 未起動状態からの初回遅延。Wasm が解決を狙う課題。
- **能力ベースのサンドボックス**: 明示付与のみ許す隔離モデル。Wasm の安全性の根拠。

## 影響範囲 / 推奨アクション
バックエンド・プラットフォームエンジニアが対象。「Kubernetes を捨てる」より、エッジ・短命・高密度な処理から Wasm を試験導入し、既存基盤と併存させる評価を勧める。
